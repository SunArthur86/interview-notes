# 【拼多多 Java服务端】JVM内存区域如何划分的？

> 来源：拼多多复活赛一面面经（小红书）

## 一、费曼类比

```
JVM = 一栋写字楼:

┌──────────────────────────────────────────────────────┐
│                   JVM 内存布局                        │
│                                                      │
│  ┌─── 线程共享区 ────────────────────────────────┐   │
│  │                                                │   │
│  │   堆 (Heap) = 公共仓库                          │   │
│  │   ┌──────────────┬────────────────────────┐   │   │
│  │   │   新生代      │      老年代             │   │   │
```

## 二、第一性原理分析

**为什么分这么多区域？因为数据特性不同：**

```
数据类型             生命周期          共享性      →    放在哪里
─────────────────────────────────────────────────────────────
方法局部变量          方法调用期间       线程私有    →    虚拟机栈
对象实例             GC决定             线程共享    →    堆
类元信息             几乎永久           线程共享    →    方法区/元空间
执行位置             实时变化           线程私有    →    程序计数器
```

### 3.2 各区域详解

| 区域 | 线程 | 存储内容 | 异常类型 | 大小控制 |
|------|------|---------|---------|---------|
| 程序计数器 | 私有 | 当前线程执行的字节码行号 | **不会OOM** | 固定 |
| 虚拟机栈 | 私有 | 栈帧(局部变量表、操作数栈等) | StackOverflowError / OOM | -Xss |
| 本地方法栈 | 私有 | native方法的栈帧 | StackOverflowError / OOM | -Xss |
| 堆 | 共享 | 对象实例、数组 | OutOfMemoryError: Java heap space | -Xms/-Xmx |
| 方法区/元空间 | 共享 | 类信息、常量池、静态变量 | OOM: Metaspace | -XX:MaxMetaspaceSize |

### 3.4 JDK8 永久代→元空间

```
JDK <8 (永久代 PermGen):
  - 位于JVM堆内存中，有固定上限(-XX:MaxPermSize)
  - 容易OOM: java.lang.OutOfMemoryError: PermGen space
  - 字符串常量池在永久代中
  - GC效率低，Full GC才能清理

JDK 8+ (元空间 Metaspace):
  - 使用本地内存(Native Memory)，不受JVM堆大小限制
  - 默认无上限(可设-XX:MaxMetaspaceSize)
  - 字符串常量池移到堆中
  - 更难OOM，但仍需监控

为什么要换？
  1. 永久代大小难调优，容易OOM
  2. 元空间用本地内存，大小随需增长
  3. 合并JRockit和HotSpot的代码
```

## 四、对象在堆中的流转

```
new Object()
     │
     ↓
┌──────────┐     │ ┌──────────┐
│   Eden   │ ──→ │ │ Survivor │
│  (出生)   │     │ │  From/To │
└──────────┘     │ └────┬─────┘
  Minor GC ──────┘      │
  (Eden满触发)          │ 存活
                        ↓
                 ┌──────────────┐
                 │   老年代      │
                 │ (Old Gen)    │
                 └──────────────┘
                   ↑ 年龄≥15 或 大对象
                   
对象晋升老年代条件:
  1. 年龄达到 -XX:MaxTenuringThreshold (默认15)
  2. 大对象直接进入(-XX:PretenureSizeThreshold)
  3. 动态年龄判断(Survivor中相同年龄对象>Survivor空间50%)
  4. 空间分配担保(Survivor不够时)
```

## 五、OOM排查

| OOM信息 | 区域 | 原因 | 排查 |
|---------|------|------|------|
| Java heap space | 堆 | 对象太多/内存泄漏 | jmap dump → MAT分析 |
| Metaspace | 元空间 | 类加载过多(动态生成类) | -XX:MaxMetaspaceSize + 类加载日志 |
| unable to create new native thread | 堆外 | 线程数太多 | 减少线程数/调整Xss |
| Direct buffer memory | 直接内存 | NIO Buffer泄漏 | 检查ByteBuffer分配 |

## 六、苏格拉底式面试提问

1. **"你说程序计数器不会OOM，为什么？"** — 引出PC只存一个行号，大小固定，且JVM规范明确保证
2. **"如果栈深度超过限制了，是StackOverflowError还是OOM？"** — 递归过深是StackOverflowError；无法扩展栈空间才是OOM
3. **"元空间用本地内存，是不是就不会OOM了？"** — 仍然会OOM（本地内存也有上限），只是阈值更高
4. **"字符串常量池到底在哪？"** — JDK6在永久代，JDK7+在堆中（不是元空间）
5. **"如何用命令查看JVM各区域使用情况？"** — jmap -heap, jstat -gc, JConsole/Arthas

## 结构化回答

**30 秒电梯演讲：** JVM内存分为线程共享区（堆、方法区）和线程私有区（虚拟机栈、本地方法栈、程序计数器），不同区域存放不同类型的数据。

**展开框架：**
1. **线程私有** — PC寄存器、虚拟机栈、本地方法栈
2. **线程共享** — 堆(新生代Eden:S0:S1=8:1:1 + 老年代)、方法区(元空间)
3. **JDK8** — 永久代→元空间(本地内存，不在堆中)
