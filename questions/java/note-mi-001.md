# HashMap JDK8 底层原理是什么？

## 数据结构

```
HashMap内部 = Node<K,V>[] table (数组)

每个桶(table[i])可能是:
  null                    → 空桶
  Node → Node → Node     → 链表
  TreeNode (红黑树)       → 树化后的结构

┌────┬────┬────┬────┬────┬────┬────┬────┐
│ 0  │ 1  │ 2  │ 3  │ 4  │ 5  │ 6  │ 7  │
└─┬──┴────┴─┬──┴────┴─┬──┴────┴────┴────┘
  │        │        │
  ▼        ▼        ▼
 Node     Node     TreeNode (红黑树, 链表≥8时)
  │        │        │
  ▼        ▼        ▼
 Node     null     TreeNode
  │                 │
  ▼                 ▼
 null              TreeNode
```

## 核心流程

### put(K, V) 流程

```java
public V put(K key, V value) {
    // 1. 计算hash: (h = key.hashCode()) ^ (h >>> 16) — 扰动函数
    int hash = hash(key);

    // 2. 定位桶: hash & (capacity - 1)  — 位运算替代取模
    int index = hash & (table.length - 1);

    // 3. 桶为空 → 直接放入
    if (table[index] == null) {
        table[index] = new Node(hash, key, value, null);
    }
    // 4. 桶非空 → 遍历链表/树
    else {
        Node<K,V> node = table[index];
        // 4a. key相同 → 覆盖value
        // 4b. key不同 → 尾插法追加
        // 4c. 链表长度≥8 → 树化(treeifyBin)
    }

    // 5. 检查是否需要扩容
    if (++size > threshold) {  // threshold = capacity * 0.75
        resize();  // 扩容2倍
    }
}
```

### 树化条件 (重要!)

```java
final void treeifyBin(Node<K,V>[] tab, int hash) {
    // ⚠️ 两个条件都必须满足:
    // 1. 链表长度 ≥ 8 (TREEIFY_THRESHOLD)
    // 2. 数组容量 ≥ 64 (MIN_TREEIFY_CAPACITY)
    //    如果容量 < 64 → 不树化，而是扩容!
    if (tab.length < MIN_TREEIFY_CAPACITY) {
        resize();  // 优先扩容而非树化
    } else {
        // 转红黑树
    }
}
```

### 扩容机制 (JDK8 优化)

```java
// JDK7: 每个node重新计算 index = hash & (newCapacity - 1)
// JDK8: 利用高位bit判断，不用重新计算hash!

// 扩容后，元素的新位置只有两种可能:
// 原位置 或 原位置 + oldCapacity

// 判断方法: hash & oldCapacity
//   = 0 → 留在原位置
//   = 1 → 移到 原位置 + oldCapacity

// 示例: capacity 8→16
// hash=9 (二进制1001), oldCap=8 (二进制1000)
// 1001 & 1000 = 1000 ≠ 0 → 新位置 = 1 + 8 = 9
```

## 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 初始容量 | 16 | `1 << 4` |
| 加载因子 | 0.75 | 容量达75%触发扩容 |
| 树化阈值 | 8 | 链表长度≥8触发树化 |
| 退化阈值 | 6 | 树节点≤6退回链表 |
| 最小树化容量 | 64 | 容量<64时优先扩容 |

### 为什么树化阈值是8？

```
基于泊松分布: 在加载因子0.75下，
桶中节点数达到8的概率 ≈ 0.00000006 (千万分之一)

这意味着: 正常情况下几乎不会树化
树化是极端情况的保护机制，不是常规路径

退化阈值6(不是8): 避免频繁在树↔链表之间切换
  8→树, 6→链表, 留2的缓冲区间
```

### 为什么加载因子是0.75？

```
空间 vs 时间 的平衡:
  0.5 → 空间浪费大(一半空着)
  1.0 → 冲突率高(几乎每个桶都有冲突)
  0.75 → 泊松分布下冲突率最低的甜点
```

## JDK7 vs JDK8 区别

| 维度 | JDK7 | JDK8 |
|------|------|------|
| 数据结构 | 数组+链表 | 数组+链表+红黑树 |
| 插入方式 | 头插法 | 尾插法 |
| 头插法问题 | 扩容时可能形成环 → 死循环 | ✅ 解决 |
| 扩容计算 | 重新hash定位 | hash & oldCap位运算 |
| 最坏查找 | O(n) | O(logn) |

## 记忆要点

- 数据结构：JDK8为数组+链表+红黑树，因防哈希冲突查询退化，故链表超阈值转树
- 核心参数：初始容量16，加载因子0.75，树化需链表长≥8且数组容量≥64
- 树化双条件：链表≥8是触发点，但容量<64时只扩容不树化，退化阈值为6防频繁抖动
- 扩容与插入：JDK8采用尾插法，扩容2倍后通过位运算将元素留原位或移至原索引+旧容量


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：HashMap JDK 8 引入红黑树，你说为了"防哈希冲突查询退化"，但为什么不直接用平衡树替代数组+链表（像 TreeMap 那样）？**

红黑树是"链表退化时的兜底"，不是"默认结构"。HashMap 的主结构是"数组 + 链表"，因为大多数 bucket 的链表很短（哈希均匀时平均长度 = load_factor，约 0.75），O(1) 查询极快。只有哈希严重冲突时（如恶意构造的 key 或 hashCode 实现差）某 bucket 链表变长，退化到 O(N)，红黑树把这个 bucket 转成 O(logN)。如果默认就用红黑树（如 TreeMap），每次 put/get 都要 O(logN)，比"数组+短链表"的 O(1) 慢。所以设计是"快路径 + 慢路径兜底"——绝大多数 bucket 用 O(1) 链表，极少数冲突严重的 bucket 升级为 O(logN) 树。这是"为常见情况优化 + 为极端情况兜底"的工程思路，不是追求单一最优。

### 第二层：证据与定位

**Q：树化的双条件你说是"链表≥8 且数组容量≥64"，为什么要两个条件？只看链表长度不行吗？**

只看链表长度会导致"小容量时频繁树化/退树化抖动"。场景：初始容量 16，put 8 个冲突 key，链表达 8，触发树化；再 put 几个不冲突的 key，触发扩容（容量翻倍），冲突 key 被重新散列，链表变短，退化回链表。频繁树化-退树化浪费 CPU（树化要构建红黑树节点，退树化要拆）。加"容量≥64"的条件后，小容量时即使链表达 8 也不树化，而是触发扩容（扩容让冲突 key 分散到更多 bucket）。只有容量够大（≥64）且仍冲突（说明是哈希函数或 key 的真问题）才树化。这是"用扩容解决哈希冲突 > 用树化解决"的优先级——扩容是治本（让冲突分散），树化是治标（在冲突 bucket 内加速查询）。8 这个阈值的依据是泊松分布——理想哈希下，bucket 链表长度达 8 的概率是 0.00000006（千万分之一），几乎不会触发，只在恶意或极端哈希冲突时生效。

### 第三层：根因深挖

**Q：JDK 8 的扩容你说是"位运算判断留原位还是移到原索引+旧容量"，这个位运算是什么？为什么能省掉重新哈希？**

扩容时容量从 oldCap 翻倍到 2×oldCap。元素的 bucket 索引是 `hash & (newCap - 1)`，由于 newCap 是 2 的幂，newCap-1 是"低位全 1"的掩码。对比 `hash & (oldCap - 1)` 和 `hash & (newCap - 1)`，差别只在多出来的那一位（oldCap 对应的 bit 位）。如果该 bit 是 0，新索引 = 旧索引（留原位）；如果是 1，新索引 = 旧索引 + oldCap（移到原索引 + 旧容量）。所以不用重新算 hash（hash 值不变），只要检查 `hash & oldCap` 是否为 0，就能确定元素去向。这把扩容的复杂度从"每个元素重新哈希"优化到"每个元素一次位运算判断"。链表节点还可以批量拆分（原链表按 bit=0/bit=1 拆成两条，分别留原位/移新位），不用逐节点处理。这是 JDK 8 相比 JDK 7 的扩容优化，性能提升明显。

**Q：那为什么 JDK 7 的扩容在多线程下会死循环（链表成环），JDK 8 不会？根因是什么？**

JDK 7 扩容用"头插法"转移链表——遍历原链表，每个节点用头插法插入新 bucket。多线程扩容时，线程 A 和 B 都在转移，头插法会反转链表顺序，两个线程交替执行可能让节点互相指向形成环（如 A.next = B 且 B.next = A），后续 get 遍历到环就死循环。JDK 8 改用"尾插法"+ 批量拆分——保持链表原顺序（不反转），且把一条链表按 bit 拆成两条，分别整体移动。尾插法不改变链表顺序，不会形成环。根因是"头插法反转链表"导致的顺序变化，尾插法保持顺序消除了这个隐患。但 JDK 8 的 HashMap 仍不是线程安全的——多线程 put 可能数据覆盖（两个线程同时写同一 bucket 的链表头，CAS 失败的那个被覆盖）、size 不准（size++ 非原子）。要线程安全必须用 ConcurrentHashMap。

### 第四层：方案权衡

**Q：HashMap 的 load_factor 默认 0.75，为什么不调到 0.5（更省冲突）或 1.0（更省空间）？**

0.75 是"时间 vs 空间"的平衡点。load_factor = size / capacity，决定何时扩容。一、0.5——扩容早（size 达 capacity 一半就扩），冲突少（链表短）查询快，但空间利用率低（一半 bucket 空着），频繁扩容浪费 CPU；二、1.0——扩容晚（size 等于 capacity 才扩），空间利用率高，但冲突多（平均链表长度 1，实际方差大，部分 bucket 链表长）查询慢；三、0.75——平均链表长度 0.75（冲突少），空间利用率 75%（不太浪费），扩容频率适中。0.75 是 JDK 工程师的经验值，基于"泊松分布下 bucket 链表长度的期望"测算，兼顾时间和空间。除非有明确需求（如内存极紧张调高到 0.9 接受冲突、查询极频繁调低到 0.5 省冲突），不要改默认值。

**Q：为什么不用 ConcurrentHashMap 替代所有 HashMap（反正功能一样且线程安全）？**

ConcurrentHashMap 有额外开销：一、put 要 CAS + 分段锁/synchronized（JDK 8 锁 bucket），单线程场景比 HashMap 的"直接数组写"慢；二、size() 是估算的（baseCount + CounterCell 数组累加），不精确；三、迭代是弱一致（迭代期间不反映并发修改），不如 HashMap 强一致。对单线程或线程封闭的场景（如方法内局部变量），HashMap 更快更简单。ConcurrentHashMap 适合"多线程共享"场景（如全局缓存）。选型：单线程用 HashMap、多线程用 ConcurrentHashMap，不要无脑用 ConcurrentHashMap（多此一举的锁开销）。我的实践：方法内局部 Map 用 HashMap、全局共享缓存用 ConcurrentHashMap。

### 第五层：验证与沉淀

**Q：你怎么验证 HashMap 的扩容和树化在特定场景下真的触发？**

写测试用例触发：一、扩容——new HashMap<>(16)，put 13 个不同 bucket 的 key（13 > 16×0.75=12，触发扩容），用反射看 threshold 字段变化；二、树化——构造 9 个哈希冲突的 key（重写 hashCode 都返回同一值），put 后看 bucket 是否是 TreeNode（用反射读 table[i] 的类型）；三、退树化——树化后 remove 到链表长度 ≤6，验证退化回链表。生产监控：HashMap 是局部对象不直接监控，但如果是全局缓存建议用 ConcurrentHashMap（暴露 size、分段锁竞争等指标）。验证哈希冲突：用 `HashMap.entrySet()` 遍历看各 bucket 链表长度分布，如果某些 bucket 异常长说明 hashCode 实现差（如取模不均匀）。

**Q：这道题做完，你沉淀出了什么可复用的 HashMap 使用经验？**

四条经验：一、初始化指定容量——如果已知大小（如 put 1000 个），`new HashMap<>((int)(1000/0.75)+1)` 避免多次扩容（默认 16 会扩容 6 次）；二、key 的 hashCode 要分布均匀——自定义 key 类要正确实现 hashCode/equals，且 hashCode 分布均匀（避免冲突集中在某 bucket）；三、多线程用 ConcurrentHashMap——HashMap 不是线程安全，多线程 put 会丢数据/死循环；四、避免大对象做 key——key 的 hashCode 和 equals 会被频繁调用，大对象（如大 String）影响性能。这套经验用于所有 Map 使用，核心是"理解底层结构 → 合理初始化 → 正确实现 key → 按并发性选 Map 类型"。
