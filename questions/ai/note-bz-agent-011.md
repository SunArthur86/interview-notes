# Self-Ask 框架是什么？

## 一、Self-Ask 核心思想

**Self-Ask** = 让 LLM **自问自答**，把复杂问题分解为一串子问题，逐个检索后组合答案。

```
用户问题: "马斯克创立的太空公司的火箭首次回收成功是在哪一年？"

Self-Ask 过程:
  Q: 马斯克创立的太空公司是？
    → 检索 → A: SpaceX
  
  Q: SpaceX的火箭首次回收成功是哪一次？
    → 检索 → A: 猎鹰9号，2015年12月
  
  最终答案: 2015年
```

## 二、Self-Ask 的 Prompt 模板

```python
SELF_ASK_PROMPT = """
回答问题时，如果需要，先分解成子问题逐个查找：

Question: {用户问题}

Follow up questions:
  Q: {子问题1}
  A: {检索得到的答案1}

  Q: {基于上一答案的子问题2}
  A: {检索得到的答案2}

  ...

So the final answer is: {综合所有子答案}

规则：
- 如果问题简单可直接回答，不必分解
- 子问题要能通过检索得到事实答案
- 后续子问题可依赖前面子问题的答案
"""
```

## 三、Self-Ask 执行流程

```
┌──────────────────────────────────────────────┐
│ 1. LLM判断：问题是否需要分解？                   │
│    - 简单 → 直接回答                            │
│    - 复杂 → 进入分解流程                        │
├──────────────────────────────────────────────┤
│ 2. 生成分问题（Follow-up）                      │
│    LLM: "为了回答X，我需要先知道Y"               │
├──────────────────────────────────────────────┤
│ 3. 检索子问题答案                                │
│    搜索(Y) → 得到答案A1                         │
├──────────────────────────────────────────────┤
│ 4. 基于A1生成下一个子问题                        │
│    LLM: "现在知道Y=A1，还需知道Z"               │
├──────────────────────────────────────────────┤
│ 5. 重复2-4，直到信息充分                         │
├──────────────────────────────────────────────┤
│ 6. 综合所有子答案，输出最终答案                   │
└──────────────────────────────────────────────┘
```

## 四、代码实现

```python
class SelfAskAgent:
    def __init__(self, llm, retriever):
        self.llm = llm
        self.retriever = retriever
    
    def run(self, question):
        qa_pairs = []
        current_q = question
        
        for _ in range(MAX_SUBQUESTIONS):
            # LLM决定是否需要追问 + 生成子问题
            sub_q = self.llm.generate_subquestion(
                original=question,
                history=qa_pairs
            )
            
            if sub_q == "NO_MORE":  # 信息充分
                break
            
            # 检索子问题
            docs = self.retriever.search(sub_q)
            sub_a = self.llm.extract_answer(sub_q, docs)
            
            qa_pairs.append((sub_q, sub_a))
        
        # 综合答案
        return self.llm.synthesize(question, qa_pairs)
```

## 五、Self-Ask vs ReAct vs Plan-Execute

| 框架 | 决策方式 | 检索 | 适用 |
|------|---------|------|------|
| **Self-Ask** | 固定流程（自问自答） | 每子问题检索 | 多跳事实问答 |
| **ReAct** | 动态（Thought+Act） | 灵活决策检索 | 通用Agent任务 |
| **Plan-Execute** | 全局规划 | 执行时检索 | 复杂多步任务 |

```
Self-Ask 特点：
  + 流程固定，简单可靠
  + 专攻多跳检索
  - 不够灵活（不能动态换策略）
  - 只做检索，不做其他工具

ReAct 特点：
  + 灵活，支持任意工具
  + 动态决策
  - Token消耗大
  - 可能方向跑偏
```

## 六、典型适用场景：多跳问答

```
适合Self-Ask的问题特征：
  - 答案需要串联多个事实
  - 每个事实可独立检索
  - 子问题间有依赖关系

例子（HotpotQA风格）：
  Q: 《盗梦空间》的导演的其他科幻电影获过什么奖？
  
  分解:
    Q1: 《盗梦空间》的导演是谁？ → 诺兰
    Q2: 诺兰的其他科幻电影有哪些？ → 《星际穿越》《信条》
    Q3: 《星际穿越》获过什么奖？ → 奥斯卡最佳视觉效果
  
  答案: 奥斯卡最佳视觉效果奖
```

## 七、面试加分点

1. **定位清晰**：Self-Ask 是 ReAct 的简化版，专攻多跳检索问答，流程固定更可靠
2. **强调"链式"**：子问题答案会喂给下一个子问题，形成信息链
3. **知道局限**：不如 ReAct 灵活，分解质量依赖 LLM，分解错就全错

## 记忆要点

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Self-Ask 把复杂问题拆成子问题链，这和 ReAct 的多步推理有什么本质区别？不都是"分解+检索"吗？**

Self-Ask 的分解是"问题维度的分解"（把"谁获得了2022诺贝尔物理学奖"拆成子问题"2022诺贝尔物理学奖获得者是谁"），ReAct 是"行动维度的分解"（Thought→Action→Observation 循环）。区别在于 Self-Ask 的子问题链是"线性追查"（answer1→用 answer1 构造 question2→answer2），每一步都是明确的"子问答"，检索目标清晰；ReAct 的每步是"推理+行动"的混合，行动可能不是检索（可能是计算/API 调用）。Self-Ask 更适合"多跳事实问答"（每跳是一个子问题），ReAct 更通用（支持多种行动）。本质上 Self-Ask 是 ReAct 在"纯检索"场景的特化，简化了决策（只决定"下一个子问题是什么"），适合需要精准多跳检索的任务。

### 第二层：证据与定位

**Q：Self-Ask 在多跳问题上拆出的子问题链"跑偏了"（拆出的子问题检索不到答案），怎么定位？**

看子问题链的每一步。trace 日志记录每个子问题和其检索结果。跑偏的信号：1）子问题表述不适合检索——如子问题"他的导师是谁"里的"他"没消解（指代不明），检索召回歧义结果，根因是指代消解没做；2）子问题依赖的上一跳答案错误——如第一跳"获奖者"答错了（幻觉或检索不到），后续所有子问题都跟着错，根因在第一跳；3）子问题分解粒度不当——拆太细（一个事实拆成 5 个子问题，中间步骤检索不到）或太粗（没拆够，单次检索答不出）。定位方法：人工审核每个子问题的"表述是否清晰""上一跳答案是否正确""检索是否召回相关信息"，找到第一个出错的环节。

### 第三层：根因深挖

**Q：Self-Ask 的子问题分解完全靠 LLM，如果 LLM 分解能力弱（拆错或漏拆），有什么补救措施？**

三层补救。1）prompt 引导——给 LLM 明确的分解指引（如"按时间/因果/层级分解"）+ few-shot 示例（演示好子问题链长什么样），提升分解质量；2）规则辅助——对特定句式用规则预分解（如"X 的 Y 的 Z"自动拆成"X 的 Y"→"Y 的 Z"），减轻 LLM 负担；3）迭代修正——执行子问题时如果检索不到，让 LLM 反思"子问题表述是否合理"并改写后重检索（类似 Self-Ask + Reflexion 的结合）。根因治理：如果 LLM 持续拆错，说明 base model 推理能力不足，要么换更强模型，要么用多跳问答数据集（HotpotQA）做 SFT 提升分解能力。

**Q：既然子问题链是线性的（Q1→A1→Q2→A2），不支持并行，复杂任务里效率不高，为什么不改成"树形分解"并行检索？**

线性 vs 树形是 trade-off。线性（Self-Ask 原版）简单可靠——每步基于上一步答案构造下一步，逻辑清晰，适合"严格依赖"的多跳（A 的导师的导师，必须先查 A）；树形并行——把问题拆成多个独立子问题并行检索再合并，适合"独立子维度"（如"对比 X 和 Y"拆成"查 X""查 Y"并行）。Self-Ask 选线性的原因：多数多跳问题是严格顺序依赖的（后一跳依赖前一跳答案），树形并行用不上；且线性的 trace 可读、易调试。改进：对"部分可并行"的问题，用混合——先用 Self-Ask 做主链，遇到"独立子维度"时局部并行（如查到获奖者后，并行查他的"导师"和"论文"，两者独立）。

### 第四层：方案权衡

**Q：Self-Ask 和 ReAct 在多跳问答上，哪个更准？实测数据上差距多大？**

在标准多跳 benchmark（HotpotQA、Musique、2WikiMultihop）上，Self-Ask 和 ReAct 准确率接近（差距 1-3 个点），但特性不同。Self-Ask 的优势：子问题表述清晰、检索目标明确，在"每跳都是事实查询"的任务上检索召回率高，且 trace 可读性强（每个子问题-答案对清晰）。ReAct 的优势：支持非检索行动（如中间步骤要计算/推理），在"多跳+推理混合"任务上更灵活。选型：纯多跳事实问答用 Self-Ask（检索精准）；多跳+需要计算/工具调用用 ReAct（灵活）。如果业务是多跳问答，Self-Ask 通常更稳（检索召回率高 5-10%），因为它的子问题专门为检索优化。

**Q：既然 Self-Ask 这么适合多跳，为什么不所有 RAG 都用 Self-Ask，还要 Naive RAG？**

因为多数查询是单跳的。FAQ 类问题（如"退货政策是什么"）一次检索就能答，Self-Ask 的分解是多余开销（先拆成子问题再检索，比直接检索慢且多调 LLM）。Self-Ask 的价值在"需要多次检索才能回答"的多跳问题，但实际业务中多跳问题占比通常只有 10-30%，70-90% 是单跳。所以最优是"路由"：先用分类器判断问题是否多跳（基于问题结构/复杂度），单跳走 Naive RAG（快），多跳走 Self-Ask（准）。这样整体延迟和成本可控（多数查询走快的 Naive），多跳准确率有保障（走 Self-Ask）。

### 第五层：验证与沉淀

**Q：你怎么证明 Self-Ask 在多跳问题上确实比 Naive RAG 好？**

在多跳 benchmark 上对照。固定 HotpotQA/Musique 测试集，对比 Naive RAG（单次检索）和 Self-Ask（子问题链检索）。指标：1）EM/F1（精确匹配/ tokens F1）——Self-Ask 应显著高（多跳 Naive 答不出）；2）检索召回率——Self-Ask 多次检索应召回更多相关文档；3）延迟和成本——Self-Ask 更高（多步），要算 ROI。典型结果：Naive RAG 在 HotpotQA 上 F1 约 40-50%，Self-Ask 能到 60-70%（+15-25 点）。如果业务有多跳查询且准确率要求高，Self-Ask 的延迟/成本增加是值得的。

**Q：Self-Ask 的子问题分解和检索策略怎么沉淀成框架能力？**

封装成 MultiHopRetriever 组件：1）子问题分解器——内置分解 prompt 模板（含 few-shot）+ 指代消解模块（避免"他/它"歧义）；2）链式检索执行——自动执行子问题链，每步检索结果喂给下一步；3）检索失败处理——子问题检索不到时自动改写（同义词/HyDE）重试；4）部分并行优化——检测到独立子维度时自动并行检索；5）多跳评测集——内置 HotpotQA/Musique 评测，新 RAG 系统自动跑回归。这套写入团队 RAG 框架 SOP，业务接入多跳问答时组件复用，不重写分解逻辑。
