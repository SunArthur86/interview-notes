# 美团 Agent 二面：知识库检索会怎么问到底？

## 一、面试官的真实考察点

```
面试官问"知识库检索怎么做"时，真正想考察的：

初级回答（不够）：
  "用向量数据库，把文档向量化，查询时算相似度"
  → 评委：只知道概念，没有深度

中级回答（及格）：
  "文档分块→向量化→存向量库→查询时检索top-k→Rerank→生成"
  → 评委：知道流程，但缺细节

高级回答（加分）：
  全链路 + 每环节的优化 + 实战经验 + 权衡分析
  → 评委：有体系有深度有实践
```

## 三、面试官常追问的问题

### 追问 1：分块怎么切？

```
"你chunk_size设多少？为什么？"

回答策略：
  "我们对比了200/500/1000字符，最终选500。
   太小（200）：语义不完整，检索到但信息不全
   太大（1000）：一个chunk混多个主题，检索不精准
   500是recall和precision的平衡点。
   
   进阶：我们用父子分块——检索用小块(200)，返回用大块(1000)，
   兼顾精准和上下文完整。"
```

### 追问 2：检索召回率低怎么办？

```
"如果相关文档检索不到，怎么优化？"

回答策略：
  "分三步排查：
   1. 查查询侧：是不是口语化导致语义鸿沟？→ 查询改写/HyDE
   2. 查检索侧：单一向量检索有盲区？→ 加BM25混合检索
   3. 查数据侧：分块有没有切断关键信息？→ 父子分块/上下文扩展
   
   我们线上从recall 60%优化到85%，主要是加了混合检索和Rerank。"
```

### 追问 3：怎么防幻觉？

```
"LLM编造了文档里没有的内容怎么办？"

回答策略：
  "三层防护：
   1. Prompt约束：明确要求'只基于文档回答，不知道就说不知道'
   2. 后处理校验：用faithfulness指标检测，答案的每个claim
      检查是否有文档支持，不支持的标注或删除
   3. 引用标注：要求LLM在每个关键信息后标注来源，
      无法标注的说明可能是编造
   
   线上我们faithfulness从70%提升到92%。"
```

### 追问 4：大规模知识库怎么办？

```
"百万级文档，检索性能怎么保证？"

回答策略：
  "分层处理：
   1. 索引优化：HNSW索引，ef_search调优，O(logn)查询
   2. 分区检索：按业务域/时间分区，先路由再检索
   3. 缓存：热点查询缓存结果，语义缓存（相似问题）
   4. 分布式：Milvus集群，分片+副本
   
   百万文档，P99延迟<50ms。"
```

### 追问 5：怎么评估效果？

```
"你怎么知道你的RAG效果好？"

回答策略：
  "用RAGAS框架，四个指标：
   - faithfulness：防幻觉（最重要）
   - context_recall：检索全不全
   - context_precision：检索准不准  
   - answer_relevancy：答案切不切题
   
   建立100个测试case的评估集，每次优化后回归测试。
   线上持续收集Bad Case，形成优化闭环。"
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：知识库检索面试要展现"查询理解→多路召回→重排→上下文组装→生成→评估"全链路，为什么不只答"向量检索"（核心机制）？**

因为只答"向量检索"显得浅，面试官要的是"工程深度"。1）全链路体现系统性——"查询理解"（query 怎么处理）+"多路召回"（不只向量，还有 BM25/改写）+"重排"（精排）+"上下文组装"（怎么给 LLM）+"生成"（prompt 怎么写）+"评估"（怎么衡量），展现你理解 RAG 是系统而非单点；2）每环节有深挖点——面试官会层层追问（如"多路召回怎么融合""rerank 选什么模型"），只答向量检索没准备这些追问，露怯；3）区分度——初级答"向量检索"（基础），高级答"全链路+每环节的权衡和优化"（深度），面试官通过全链路区分水平。所以答全链路是"展现系统思维和工程深度"，应对层层追问，而非只答一个机制。

### 第二层：证据与定位

**Q：面试官问"你的 RAG 系统召回率多少，怎么提升的"，怎么用数据展现（而非空谈"调了调"）？**

用具体指标和优化路径。1）基线数据——初始召回率（如 Recall@5=65%）、问题（如某类 query 召回差）；2）优化动作——具体做了什么（如"加 BM25 混合检索，RRF 融合，Recall 升到 78%""加 rerank，MRR 从 0.6 到 0.8""query 改写，语义鸿沟 case 召回+10%"），每步带数据；3）最终效果——召回率（如 88%）、端到端准确率（如从 70% 到 85%）、用户满意度（如评分 4.2）；4）权衡——成本（如 rerank 增加 80ms 延迟，可接受）、复杂度（如多路召回增加维护成本）。这样答展现"数据驱动+全链路优化+权衡思考"，而非"调了调变好了"的空谈。面试官追问"为什么 rerank 提升 MRR 而非 Recall"（rerank 精排不扩召回），你能答（体现理解）。

### 第三层：根因深挖

**Q：面试官追问"你的多路召回（向量+BM25+改写）结果怎么融合，为什么用 RRF 而非加权"，怎么答才显深度？**

答 RRF 的"尺度无关"优势。1）加权的问题——向量分数（cosine 0-1）和 BM25 分数（无界）尺度不同，直接加权（如 0.5*向量+0.5*BM25）无意义（BM25 大数值压倒向量），要先归一化但归一化方式（min-max/z-score）影响结果且不鲁棒；2）RRF 的优势——只用排名（如文档在向量排第 1、BM25 排第 3，RRF=1/(1+1)+1/(3+1)=0.75），尺度无关（不受分数分布影响），简单（无需调权重）且鲁棒（业界 RAG 默认）；3）补充——如果想加权（如向量更重要），用加权 RRF（给向量排名的贡献乘 1.5），兼顾权重和鲁棒；4）进一步——有充足数据可用 LTR（学习排序，训练融合模型），最准但要数据。这样答展现"理解加权的尺度问题+RRF 的原理和优势+进阶选项"，深度够。

**Q：面试官问"rerank 的 Cross-Encoder 为什么比向量检索的 Bi-Encoder 准"，怎么答原理？**

答"深度交互"。1）Bi-Encoder——query 和文档独立编码成向量，相似度是向量点积/余弦，交互只发生在最后（浅层），query 和文档的词级语义对齐没做；2）Cross-Encoder——query 和文档拼接（[CLS] query [SEP] document）一起输入 Transformer，每层 attention 让 query 词和文档词交互（query 的"省钱"attend 文档的"降低成本"），深层对齐，细粒度语义匹配；3）为什么准——细粒度对齐能发现"表面不同但语义匹配"（Bi-Encoder 漏）或"表面相似但语义不匹配"（如"苹果手机"vs"苹果水果"，Cross-Encoder 通过上下文区分）；4）为什么慢——每个 query-文档对都要一次完整 forward（深度交互），不能预计算（不像 Bi-Encoder 文档向量预计算），所以只能精排少量候选（两阶段：向量粗筛 top-20→Cross-Encoder 精排 top-5）。这样答原理+权衡（准但慢→两阶段），显深度。

### 第四层：方案权衡

**Q：面试官问"你的 RAG 系统延迟多少，怎么优化的"，怎么答展现工程能力？**

答具体延迟和优化手段。1）延迟拆解——总延迟（如 P99=800ms）拆成各环节（查询理解 50ms + 多路召回 100ms + rerank 200ms + 上下文组装 30ms + LLM 生成 400ms），找瓶颈（如 LLM 生成占大头）；2）优化手段——针对瓶颈优化（如 LLM 生成用流式输出，首 token 200ms，用户感知快；rerank 用轻量模型/减 N，降 100ms；多路召回并行，降 50ms）；3）权衡——优化后的延迟（如 P99=500ms）+ 成本（如轻量 rerank 精度略降，可接受）；4）监控——线上 P99 延迟监控+告警（超 SLA 告警）。这样答展现"延迟归因（拆解）+ 针对性优化 + 权衡 + 监控"，是工程能力而非"调了调变快了"。

**Q：面试官问"怎么评估 RAG 系统好不好"，怎么答展现评估体系（而非只说"看准确率"）？**

答 RAGAS 四维+闭环。1）四维指标——上下文召回（检索全）、上下文精度（检索准）、忠实度（防幻觉）、答案相关性（切题），分层评估检索和生成；2）评估方法——合成评估集（LLM 基于文档生成 query，已知 golden）+ 人工标注（小规模精确）+ 线上采样（真实），三层结合；3）LLM 评判+人工校准——RAGAS 用 LLM 评判（大规模），抽样人工校准（保证可信）；4）闭环——bad case 收集→RAGAS 归因（哪维差）→优化→评估集回归，持续改进。这样答展现"四维评估+多源评估集+LLM 评判校准+改进闭环"，是体系而非单一指标。

### 第五层：验证与沉淀

**Q：面试官问"你怎么保证 RAG 上线后质量不退化"，怎么答展现质量保障？**

答 CI/CD+监控+闭环。1）CI/CD 门禁——每次迭代（改 prompt/换模型）自动跑核心评估集（RAGAS），退化阻断上线，保不退化；2）线上监控——关键指标（召回/延迟/准确率）实时监控+告警，异常（如召回突降）及时响应；3）bad case 闭环——线上 bad case 自动收集→归因→优化→验证，持续改进；4）版本管理——prompt/模型/评估集版本化，支持回滚（新版本有问题回退旧版本）；5）AB 测试——新版本灰度（部分流量），对比指标，确认好再全量。这样答展现"事前（CI 门禁）+ 事中（监控告警）+ 事后（闭环优化）+ 兜底（回滚）"的全流程质量保障。

**Q：知识库检索的全链路经验怎么沉淀成团队的能力？**

答平台化。1）RAG 框架——全链路组件标准化（查询理解/多路召回/rerank/组装/生成/评估），可配置组合，新系统快速搭建；2）评估平台——评估集管理+RAGAS 自动评估+CI 门禁，质量保障；3）监控平台——线上指标监控+bad case 收集，持续优化；4）最佳实践库——每环节的优化经验（如"混合检索用 RRF""rerank 两阶段"）文档化，新人按手册做；5）案例库——真实优化案例（数据驱动），经验复用。这套写入团队 RAG 平台 SOP，让"建知识库检索"从"每个项目摸索"变成"平台+最佳实践"，标准化产出高质量 RAG 系统。
