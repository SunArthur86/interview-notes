# 【RAG混合检索】为什么不能只用向量检索？

> 来源：小红书「前端 AI 项目必问：为啥不能只用向量检索？」

## 二、混合检索完整链路

```
用户Query
    │
    ├──→ 向量检索(pgvector)  ──→ Top 10 候选
    │    (语义匹配)                │
    │                              ├──→ RRF融合 ──→ Top 20 ──→ Reranker ──→ Top 5
    │                              │    (排名融合)    (重排)       (精排)
    └──→ BM25全文检索         ──→ Top 10 候选
         (精确匹配)                │
```

## 三、各环节技术选型

### 向量检索：pgvector

```sql
-- pgvector + HNSW索引
CREATE EXTENSION vector;

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536)  -- 1536维向量
);

-- HNSW索引（适合中小规模、低延迟）
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);

-- 检索
SELECT id, content, embedding <=> $1 AS distance
FROM documents
ORDER BY embedding <=> $1  -- 余弦距离
LIMIT 10;
```

### BM25全文检索：PostgreSQL原生

```sql
-- GIN索引支持全文检索
CREATE INDEX idx_docs_content ON documents 
USING gin(to_tsvector('chinese', content));

-- BM25检索（用ts_rank模拟）
SELECT id, content, 
       ts_rank(to_tsvector('chinese', content), query) AS score
FROM documents, to_tsquery('chinese', '退货 & 流程') query
WHERE to_tsvector('chinese', content) @@ query
ORDER BY score DESC
LIMIT 10;
```

### IVFFlat vs HNSW索引对比

| 特性 | IVFFlat | HNSW |
|------|---------|------|
| 原理 | 倒排文件+扁平量化 | 分层可导航小世界图 |
| 构建速度 | 快 | 慢 |
| 查询速度 | 中等 | 快 |
| 内存占用 | 小 | 大 |
| 适合规模 | 大规模数据 | 中小规模、低延迟 |

## 四、方案对比

| 方案 | 召回率 | 延迟 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| 纯向量检索 | 语义类高/精确类低 | 低 | 简单 | FAQ、语义搜索 |
| 纯BM25 | 精确类高/语义类低 | 低 | 简单 | 文档关键词搜索 |
| 向量+BM25+RRF | 全面高 | 中 | 中等 | 生产级RAG |
| 向量+BM25+RRF+Rerank | 最高 | 高 | 复杂 | 高质量RAG |

## 五、面试加分点

1. **量化分析**：能说出具体案例——"纯向量检索时，用户搜SKU编号的召回率只有30%，加入BM25后提升到95%"
2. **余弦相似度vs欧氏距离**：向量检索通常用余弦相似度（衡量方向而非距离），文本检索场景更合适，因为embedding的模长可能受文本长度影响
3. **chunk overlap设计**：文档切分时chunk之间要有overlap（如50字符），保证语义连续性——一个概念横跨两个chunk时不会断裂
4. **pgvector生产部署**：PostgreSQL同时做向量检索和全文检索，减少了系统组件数量，对前端开发者更友好
5. **RRF的k值调优**：k=60是经验值，k越小对高排名文档越敏感（更看重top-1），k越大越平缓（更看重整体排名）

## 结构化回答

**30 秒电梯演讲：** 向量检索擅长语义匹配但精确匹配差，BM25擅长关键词匹配但语义理解差——单用任何一种都有盲区，混合检索取两者之长——向量检索像'意会型'图书管理员。

**展开框架：**
1. **向量检索** — 语义匹配强，精确匹配差（搜编号/人名/专有名词漏召回）
2. **BM25** — 精确匹配强，语义理解差（同义词/口语化query漏召回）
3. **混合检索** — 混合检索=向量检索+BM25→RRF融合排名→取两者之长

**收尾：** 您想深入聊：向量检索和BM25的召回结果重叠率通常是多少？

## 苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | RAG为什么不能只用向量检索？根本矛盾是什么？ | 向量检索擅长语义相似但不擅长精确匹配（人名、编号、术语），只用向量会漏掉精确答案；混合检索是语义+精确的互补 |
| 证据追问 | 怎么证明只用向量检索会漏结果？ | 构造精确匹配query（产品型号、人名）对比纯向量vs混合检索的召回率，纯向量top-k里常找不到精确答案 |
| 边界追问 | 什么场景纯向量就够了，什么场景必须混合？ | 模糊语义问答纯向量够；涉及专有名词、代码、编号、精确数值的场景必须加BM25等稀疏检索 |
| 反例追问 | 向量模型很强，召回率还不够吗？ | 不够。向量是语义压缩有信息损失，精确token级匹配能力弱；且不同向量模型对专有名词编码不稳定，BM25的精确匹配不可替代 |
| 风险追问 | 混合检索的风险和成本是什么？ | 维护两套索引、融合策略调参复杂、延迟增加、稀疏检索对中文分词敏感、两套结果不一致时排序难 |
| 验证追问 | 怎么验证混合检索真的提升了效果？ | 构造评测集对比召回率@k、人工badcase分析、线上A/B测问答准确率、监控点击率 |
| 沉淀追问 | 混合检索方案怎么沉淀？ | 规范：默认混合检索、BM25+向量的权重调参、评测集回归、badcase监控 |

### 现场对话示例
**面试官**：RAG为什么不能只用向量检索？
**候选人**：向量擅长语义相似但精确匹配弱（人名、编号、术语会漏），必须混合BM25稀疏检索做语义+精确互补。
**面试官**：怎么证明纯向量会漏？
**候选人**：构造精确匹配query对比召回率，纯向量top-k常找不到型号/人名等精确答案，混合后召回率明显提升。
**面试官**：混合检索有什么成本？
**候选人**：维护两套索引、融合调参复杂、延迟增加、稀疏检索对分词敏感，需要评测集和badcase监控保障效果。
