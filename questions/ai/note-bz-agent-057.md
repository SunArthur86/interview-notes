# 什么场景必须用 GraphRAG？知识图谱如何构建和应用？

## 一、GraphRAG vs 向量 RAG

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 向量RAG             │ GraphRAG                │
├──────────────┼──────────────────┼──────────────────────┤
│ 数据结构      │ 文本块+向量索引     │ 实体+关系+图结构        │
│ 检索方式      │ 语义相似度          │ 图遍历                  │
│ 擅长          │ 局部事实查询        │ 多跳关系推理            │
│ 全局摘要      │ 弱（只看局部片段）  │ 强（社区摘要）          │
│ 构建成本      │ 低（切分+向量化）   │ 高（LLM抽取实体关系）   │
│ 更新成本      │ 低（重新向量化）    │ 高（重建图结构）        │
│ 延迟          │ 快(ms)             │ 中(图遍历)             │
└──────────────┴──────────────────┴──────────────────────┘
```

## 二、必须用 GraphRAG 的场景

### 场景 1：多跳推理（核心场景）

```
问题: "收购了DeepMind的公司，其创始人的母校在哪里？"

需要跨3跳推理:
  DeepMind → 被Google收购 → 创始人Larry Page → 母校密歇根大学

向量RAG: 一次检索找不到（信息分散在不同文档）
GraphRAG: 图遍历 DeepMind-[收购]→Google-[创始]→Page-[毕业]→密歇根
```

### 场景 2：关系密集型

```
场景: 药物相互作用分析
  药物A -[相互作用]- 药物B
  药物B -[禁忌]- 疾病C
  → 患者有疾病C，能用药物A吗？

向量RAG: 难以发现间接的相互作用链
GraphRAG: 图遍历自然发现 A→B→C 的禁忌链
```

### 场景 3：全局摘要

```
问题: "总结这个领域的所有主要观点"

向量RAG: 只能检索top-k局部片段，无法全局概览
GraphRAG: 社区检测聚类后，每个社区生成摘要，汇总成全局视图

GraphRAG的全局查询流程:
  1. 图结构 → Leiden算法社区检测
  2. 每个社区生成摘要
  3. 查询时汇总各社区摘要 → 全局答案
```

### 场景 4：因果/依赖分析

```
场景: 供应链风险分析
  "如果供应商A停产，影响哪些产品？"
  
  产品X -[依赖]- 组件Y -[来自]- 供应商A
  → 图遍历找到所有依赖A的产品
```

## 三、知识图谱构建流程

```python
class GraphRAGBuilder:
    def build(self, documents):
        # Step 1: 实体抽取
        entities = []
        for doc in documents:
            extracted = self.llm.extract_entities(doc)
            # "Google由Larry Page创立" 
            # → 实体: [Google, Larry Page]
            entities.extend(extracted)
        
        # Step 2: 关系抽取
        relationships = []
        for doc in documents:
            rels = self.llm.extract_relationships(doc)
            # → 关系: [(Google, 创立者, Larry Page)]
            relationships.extend(rels)
        
        # Step 3: 构建图
        graph = KnowledgeGraph()
        for entity in deduplicate(entities):
            graph.add_node(entity)
        for rel in relationships:
            graph.add_edge(rel.source, rel.target, rel.type)
        
        # Step 4: 社区检测（聚类）
        communities = graph.detect_communities(algorithm="Leiden")
        # 相关节点的聚类，如"AI领域""医疗领域"
        
        # Step 5: 社区摘要
        for community in communities:
            community.summary = self.llm.summarize(community.nodes)
        
        return graph
```

## 四、GraphRAG 查询流程

```python
class GraphRAGQuery:
    def query(self, question, graph):
        # 判断查询类型
        if self.is_global_query(question):
            # 全局查询：汇总各社区摘要
            return self.global_query(question, graph)
        else:
            # 局部查询：从相关实体图遍历
            return self.local_query(question, graph)
    
    def local_query(self, question, graph):
        # 1. 找到问题相关的起始实体
        entities = self.llm.extract_entities(question)
        start_nodes = graph.find_nodes(entities)
        
        # 2. 图遍历（扩展N跳）
        relevant = graph.traverse(start_nodes, max_hops=3)
        
        # 3. 收集相关信息
        context = self.collect_subgraph_info(relevant)
        
        # 4. LLM基于图信息生成答案
        return self.llm.answer(question, context)
    
    def global_query(self, question, graph):
        # 汇总所有社区摘要
        summaries = [c.summary for c in graph.communities]
        return self.llm.answer_global(question, summaries)
```

## 五、什么场景不适合 GraphRAG

```
不适合：
  ✗ 简单事实查询（向量RAG更快更便宜）
    "什么是Agent" → 向量一次检索就够
  
  ✗ 频繁更新的数据（图重建成本高）
    新闻/实时数据 → 向量库增量更新更快
  
  ✗ 数据量小（没必要建图）
    几十篇文档 → 向量RAG足够
  
  ✗ 无明确实体关系的文本
    纯叙述性文本 → 抽不出有意义的关系

判断标准：
  问题需要"跨实体关系推理" → GraphRAG
  问题只是"找相关信息" → 向量RAG
```

## 六、GraphRAG 的成本

```
构建成本（高）:
  - 每个文档都要LLM抽取实体和关系
  - 1000篇文档 → ~百万Token的LLM调用
  - 成本可能是向量RAG的10-50倍

查询成本（中）:
  - 图遍历比向量检索慢
  - 但通常比多轮Agentic RAG快

更新成本（高）:
  - 新增文档需要重新抽取+更新图结构
  - 不像向量库可以简单增量

优化:
  - 增量构建（只处理新文档）
  - 用小模型抽取（省成本）
  - 混合：关键数据建图，其余用向量
```

## 七、面试加分点

1. **多跳推理是核心价值**：用"A公司CEO的母校"例子说明向量 RAG 做不到的事
2. **承认成本高**：GraphRAG 构建贵更新难，不是银弹——只在需要时用
3. **混合架构**：向量+图谱混合（简单走向量，复杂走图谱）是生产实践

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：GraphRAG 用知识图谱做多跳推理，但向量 RAG 也能检索相关文档让 LLM 推理，为什么"关系密集型"场景必须用 GraphRAG？**

因为向量 RAG 在关系推理上有根本局限。1）多跳检索漏召回——关系型问题（如"A 的合作方的投资领域"）需要跨多个文档（A→合作方→投资），向量 RAG 检索"A 的合作"和"合作方的投资"分别可能召回，但中间实体（合作方）的连接关系没显式表达，LLM 靠拼接文档推理易出错（尤其跳数多时）；GraphRAG 在图上遍历（A→合作方→投资），显式走关系，准确；2）全局摘要——"所有领域的投资分布"这类全局问题，向量 RAG 检索 top-K 是局部的（无法覆盖全部分布），GraphRAG 遍历所有投资边聚合，全局准确；3）关系精确性——向量 RAG 检索到的文档可能含关系，但关系的"主语-谓语-宾语"没结构化（文本里），LLM 解析易错；GraphRAG 的关系是结构化边，精确。所以关系密集型（多跳/全局）GraphRAG 准，向量 RAG 在这些场景根本性弱。

### 第二层：证据与定位

**Q：GraphRAG 查询结果不对（关系推理错），怎么定位是图谱构建错（实体/关系抽取错）还是图遍历逻辑错？**

分层验证。1）图谱层——查图谱里相关实体/关系是否存在且正确（如"A 和 B 的合作关系"边是否存在），如果抽取时漏了/错了（如把"合作"抽成"竞争"），是图谱构建问题；2）遍历层——如果图谱对（实体/关系对），但查询遍历路径错（如该走"合作"边却走了"投资"边），是遍历逻辑/查询错；3）对齐层——用户 query 到图查询的转换（如"A 的合作方的投资"转成图遍历模式）可能错（转换的实体/关系映射错），是 query 到图的对齐问题。定位方法：打印查询的图遍历路径（走了哪些节点/边），对比"应该走什么"（人工判断），找不一致——路径对但结果错（图谱数据错），路径错（遍历/query 转换错）。常见根因：实体抽取漏（关键实体没入图）、关系抽取错（关系类型标错）、遍历模式错（query 转图查询的映射错）。

### 第三层：根因深挖

**Q：知识图谱构建要"实体抽取+关系抽取"，但抽取准确率有限（LLM/模型抽错或漏），怎么提升构建质量？**

模型+校验+迭代。1）抽取模型——用强 LLM（如 GPT-4）+ 精心 prompt（明确抽取规则/示例）抽取实体关系，比小模型/规则准；或微调专门抽取模型（如基于业务数据训 NER+RE 模型），领域更准；2）schema 约束——预定义实体/关系的 schema（如"公司/人/产品"实体，"投资/合作/竞争"关系），抽取按 schema 抽（限制输出空间，减少错抽），非 schema 的拒绝；3）校验——抽取后校验（如关系的主语宾语是否都是合法实体/关系类型是否符合 schema/重复实体是否合并），错的过滤或人工复核；4）迭代——从查询错误反推抽取缺陷（如"A 的合作"查不到→可能"A 的合作"关系漏抽），补充抽取/修正，持续提升。进阶：多源融合（从结构化数据如数据库直接导入，比文本抽取准）。

**Q：GraphRAG 和向量 RAG 各有优势（关系 vs 语义），实际系统为什么不二选一，而是混合？**

因为 query 类型多样。1）关系型 query（如"A 的合作方的投资"）——GraphRAG 准（图遍历），向量 RAG 弱；2）语义型 query（如"如何提升销量"）——向量 RAG 准（语义检索），GraphRAG 弱（图谱无开放语义）；3）混合 query（如"A 公司的销量提升策略"，既有实体 A 又有开放语义）——需要两者（GraphRAG 找 A 相关实体/结构化信息，向量 RAG 检索"销量提升策略"的文档），融合回答；4）路由——LLM 判断 query 类型（关系/语义/混合），路由到对应检索（GraphRAG/向量/两者），按需用；5）融合——两者结果融合（GraphRAG 的结构化事实 + 向量 RAG 的文档），LLM 综合生成答案。所以混合是"按 query 类型选最优检索"，覆盖全面，单一检索都有盲区。

### 第四层：方案权衡

**Q：GraphRAG 构建知识图谱成本高（抽取+校验+维护），什么时候值得投入，什么时候不值得？**

按 query 类型分布决策。1）值得投入——业务以"关系密集型 query"为主（如企业情报"投资关系"/社交"好友的朋友"/医疗"药物-疾病-基因关系"），GraphRAG 是刚需（向量 RAG 根本性弱），投入值得；2）不值得——业务以"语义型 query"为主（如文档问答/开放问答），向量 RAG 够，GraphRAG 投入回报低（关系 query 少）；3）混合场景——如果关系 query 占一定比例（如 30%），可建轻量图谱（只抽核心实体关系），向量为主+图谱补充关系查询，平衡成本和覆盖；4）演进——先向量 RAG 上线（快），观察 query 类型，如果关系型多再逐步建图谱（增量投入）。原则：query 类型驱动，关系密集才重投 GraphRAG，语义为主用向量，混合则两者协同。

**Q：GraphRAG 的图遍历深度（1跳/2跳/多跳）怎么定？深了准但慢（图爆炸），浅了快但可能漏关系？**

按 query 需求和性能权衡。1）query 需求——query 决定跳数（如"A 的合作"1跳，"A 的合作方的投资"2跳，"A 的合作方的投资的领域"3跳），按 query 实际需要的跳数遍历，不多跳（避免图爆炸）；2）图爆炸控制——多跳遍历组合爆炸（每跳邻居数乘积），深了节点数爆炸（千万级），延迟高，要剪枝（如只走特定关系类型/限 top-K 邻居/设深度上限）；3）混合——浅跳（1-2跳）实时遍历（快），深跳（3+）用预计算（如预先算好"X 的 3 跳邻居"缓存，查询时直接查），或用图嵌入（如 TransE 把图编码成向量，向量检索近似多跳）；4）动态——简单 query 浅遍历，复杂 query 深遍历（或预计算），按复杂度选。实务：默认 1-2 跳实时遍历，高频深跳 query 预计算缓存，超深用图嵌入近似。

### 第五层：验证与沉淀

**Q：你怎么衡量 GraphRAG 的效果（比向量 RAG 好多少、图谱质量如何）？**

多层指标。1）图谱质量——实体/关系抽取准确率（抽样人工评估，如"抽出的关系有多少正确"）、覆盖率（关键实体/关系是否都入图，漏了多少）、一致性（重复实体是否合并/关系是否冲突）；2）查询准确率——关系型 query 上 GraphRAG vs 向量 RAG 的答案准确率（GraphRAG 应显著高）；3）遍历效率——图遍历的延迟（P99，关系密集 query 要可控）；4）端到端——GraphRAG 融合向量 RAG 的整体准确率（应优于单一）。最优：图谱质量高（准确/覆盖/一致）+ 关系 query 准确率显著优于向量 + 延迟可控 + 端到端好 = GraphRAG 有效。还要监控图谱维护成本（抽取/更新的算力/人工）是否可接受。

**Q：GraphRAG 的构建和查询怎么沉淀成团队的知识图谱能力？**

建 GraphRAG 平台：1）图谱构建 pipeline——多源数据接入（文档/数据库/API）→ 实体关系抽取（LLM/微调模型）→ schema 校验 → 入图，支持增量更新（数据变更自动 reindex）；2）图数据库——选型（Neo4j/JanusGraph 等），支持高效图遍历；3）查询层——query 到图查询的转换（LLM 把 query 转成图遍历模式）+ 路由（关系 query 走图，语义 query 走向量）；4）混合检索——GraphRAG + 向量 RAG 融合，按 query 类型路由；5）评估——图谱质量监控（抽取准确率）+ 查询准确率评估 + bad case 反馈优化抽取。这套写入团队知识平台 SOP，让"建图谱+GraphRAG 查询"从"高门槛手工"变成"平台化能力"，业务按需接入。

## 结构化回答

**30 秒电梯演讲：** GraphRAG=用知识图谱做检索。实体为节点、关系为边，通过图遍历做多跳推理。适合"关系密集型"和"全局摘要"场景，向量RAG做不好的地方。

**展开框架：**
1. **核心** — 实体节点+关系边+图遍历
2. **构建** — LLM抽实体关系→建图→社区检测
3. **适合** — 多跳推理/关系分析/全局摘要
