# agent 模拟面试（结合项目场景题）怎么答？

## 一、场景题的答题框架

```
┌──────────────────────────────────────────────────┐
│            场景题答题五步法                          │
├──────────────────────────────────────────────────┤
│                                                    │
│  Step 1: 需求澄清（2分钟）                          │
│    问清楚：用户是谁/规模多大/关键约束              │
│    展示：你不急于解题，先理解问题                  │
│                                                    │
│  Step 2: 架构设计（5分钟）                          │
│    画出：整体架构图（分层）                        │
│    说明：每个组件的职责                            │
│                                                    │
│  Step 3: 关键决策（5分钟）                          │
│    解释：为什么选A不选B                            │
│    展示：技术选型能力和tradeoff分析               │
│                                                    │
│  Step 4: 风险应对（3分钟）                          │
│    识别：可能的失败模式                            │
│    方案：限流/熔断/降级/兜底                       │
│                                                    │
│  Step 5: 评估指标（2分钟）                          │
│    定义：怎么衡量效果                              │
│    展示：数据驱动的思维                            │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、模拟场景题示例

### 题目：设计一个企业内部知识库问答Agent

```
面试官：你们公司有10万份内部文档，要做一个问答Agent，
       日活5000，怎么设计？
```

### 参考回答

**Step 1: 需求澄清**

```
"我先确认几个问题：
  - 文档类型？（Word/PDF/Confluence/代码？）
  - 用户场景？（员工查SOP/技术文档/政策？）
  - 准确性要求？（必须准确还是允许参考？）
  - 是否需要权限控制？（不同部门看不同文档？）
  - 延迟要求？（秒级还是可以等？）"
→ 展示你不急于解题，先理解需求
```

**Step 2: 架构设计**

```
"整体架构分四层：

  ┌─────────────────────────────────────┐
  │  接入层：Web/IM/搜索框               │
  ├─────────────────────────────────────┤
  │  Agent层：                          │
  │  意图理解 → RAG检索 → 生成回答      │
  │  （含权限过滤/引用标注）             │
  ├─────────────────────────────────────┤
  │  数据层：                           │
  │  文档处理Pipeline + 向量库 + 元数据 │
  ├─────────────────────────────────────┤
  │  基础设施：LLM网关/监控/审计        │
  └─────────────────────────────────────┘"
```

**Step 3: 关键决策**

```
"几个关键设计决策：

  1. 检索方案：向量+BM25混合检索+Rerank
     为什么：向量擅长语义，BM25擅长关键词(产品名/代码)
            互补，Rerank保证精度
     tradeoff：比纯向量慢，但准确率高15%

  2. 分块策略：父子分块(检索300字/返回1500字)
     为什么：小块检索精准，大块上下文完整
     tradeoff：存储增加，但回答质量更好

  3. 权限控制：检索时元数据过滤
     为什么：不同部门只能看自己的文档
     tradeoff：索引时需打权限标签，增加维护成本"
```

**Step 4: 风险应对**

```
"主要风险和应对：

  1. 检索不到 → 混合检索+查询改写提升召回
  2. 幻觉 → Prompt约束+faithfulness检测
  3. 高并发 → 限流+缓存+模型路由
  4. 文档更新 → 增量索引(小时级更新)
  5. 敏感信息 → 输出脱敏+权限过滤"
```

**Step 5: 评估指标**

```
"评估体系：

  检索质量：recall@5 > 85%, precision > 80%
  生成质量：faithfulness > 90%
  业务指标：
    - 问答准确率（人工抽检）
    - 用户满意度（点赞率 > 85%）
    - 平均响应时间（< 3s）
    - 月活/留存
  
  建立Bad Case闭环持续优化。"
```

## 四、答题技巧

```
DO（加分）：
  ✓ 先问澄清问题（展示思考）
  ✓ 画架构图（更直观）
  ✓ 讲tradeoff（为什么选A不B）
  ✓ 给量化指标（展示数据思维）
  ✓ 承认不确定（诚实比硬编好）
  ✓ 从全局到细节（先架构再深入）

DON'T（减分）：
  ✗ 一上来堆技术名词
  ✗ 不问需求直接解题
  ✗ 只说用什么不说为什么
  ✗ 忽略非功能需求（性能/安全/成本）
  ✗ 答不上来就沉默（说思路也行）
  ✗ 过度设计（简单问题复杂化）
```

## 五、面试加分点

1. **五步框架**：需求→架构→决策→风险→评估，结构化答题
2. **先问后答**：澄清需求展示成熟——别急于表现
3. **tradeoff 是核心**：面试官最想听"为什么这么选"——这区分资深和初级

## 记忆要点

- 五步答题法：因为面试官考察全局观，所以按需求澄清→架构分层→关键决策→风险应对→评估指标展开
- 需求澄清必做：先反问文档类型、并发量、准确度要求，展示不急于写代码的资深工程师思维
- 关键决策必答Trade-off：选向量+BM25混合检索因为兼顾语义与关键词，加Rerank虽慢但能保精度

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：场景题你第一步是"需求澄清"反问面试官，但如果面试官说"你都假设一下，直接给方案"，你还要坚持问吗？这不显得啰嗦？**

不坚持追问，但要"陈述假设"而非"沉默假设"。面试官说"自己假设"时，正确的做法是"我假设文档以 Confluence/Word 为主、日活 5000、准确性要求高（内部知识库不能瞎答），基于这些假设我的方案是..."。这样既不啰嗦（不逼面试官逐个回答），又展示了你的假设（面试官可以纠正"假设错了，实际是 XX"）。直接闷头假设不做声明是危险的——如果假设错了（如假设不需要权限控制，实际要），整个方案跑偏。所以"陈述假设 + 邀请纠正"是折中，既快又稳。

### 第二层：证据与定位

**Q：你答到一半面试官打断说"你这个 RAG 召回率 85% 怎么来的，太乐观了吧"，你瞬间有点慌，怎么应对这种打断和质疑？**

先稳住，别防御性反驳。正确应对是"承认是估算 + 说清依据 + 请教实际数据"。具体："85% 是基于我之前项目（XX 场景）的经验值，不同文档质量下会有波动，如果贵司文档结构化程度高可能更高、如果是扫描件 OCR 可能更低。您那边有实测数据或文档质量的大致情况吗？我可以据此调整方案。"这样既没硬撑（承认是估算），又展示了依据（不是拍脑袋），还把对话变成协作（请教数据）。防御性反驳（"85% 是行业标准"）会被认为不虚心，慌乱沉默会被认为不扎实。被打断是正常的深挖信号，不是否定。

### 第三层：根因深挖

**Q：你的关键决策讲了"向量+BM25+Rerank"，面试官追问"那如果文档里有大量代码和 SQL（不是自然语言），你的向量检索还有效吗？"——这个场景你的方案有盲区，怎么接？**

诚实承认盲区 + 给应对方案。代码/SQL 的特点是"字面精确性强、语义性弱"，通用 embedding 模型（基于自然语言训练）对代码的语义理解差，向量召回率会掉。应对方案有三个层次：一是代码用专门的 embedding 模型（如 voyage-code、bge-code，专门在代码语料上训练）；二是代码检索更依赖 BM25（代码的函数名/变量名是强字面信号，BM25 精确匹配更有效）；三是混合文档要分流检索——自然语言段走通用向量，代码段走代码专用模型 + BM25，按文档类型路由。所以不是"一个向量模型打天下"，是按内容类型选检索策略。承认盲区 + 给方案，比假装"通用模型够用"更体现深度。

**Q：那为什么不直接对所有内容都用 BM25（代码和自然语言都能字面匹配），省去向量模型的复杂度？**

因为自然语言段用纯 BM25 召回率会掉。用户问"怎么改收货地址"，文档写的是"修改配送信息的步骤"——字面无重叠（没有"改/收货地址"这些词），BM25 召回不到，但向量能匹配语义。所以自然语言必须靠向量，代码必须靠 BM25/专用模型，两者不可互替。混合文档的正确架构是"内容感知的分流检索"——先判断 chunk 类型（自然语言/代码/表格），按类型走对应检索路径，最后融合。这是对"向量+BM25 混合"的进一步细化（不是简单两路并列，是按内容类型路由），体现了对文档异构性的理解。

### 第四层：方案权衡

**Q：风险应对你列了限流/缓存/降级，但面试官问"这套防护会让架构变复杂，5000 日活的小系统值得吗？是不是过度设计？"你怎么权衡？**

按规模分阶段，不是一上来就全上。5000 日活的内部知识库，QPS 峰值大概几十，限流（一个令牌桶几行代码）和缓存（语义缓存复用高频问题）成本低收益高，必上。熔断/多级降级链这种重型方案，5000 日活确实不必要（LLM 厂商自己有容量，不至于打爆），可以不上。判断标准：看"不做的代价"——不做限流，一个用户写脚本刷接口能烧光 API 预算（真实风险），必须做；不做熔断，LLM 偶发故障时用户重试放大（中等风险），可做轻量版（超时 + 简单重试）。所以诚实说"5000 日活我会上限流和缓存，熔断用轻量版，重型降级链留到规模涨 10 倍再上"，这比"全上"显得更懂按需设计，面试官反而认可。

**Q：为什么不直接按"最高标准"设计，一次到位，省得以后规模涨了再重构？**

因为"为不存在的规模过度设计"是工程大忌。5000 日活按 50 万日活的标准设计，会：浪费开发时间（花 3 个月做的熔断链可能 2 年内用不上）、增加维护成本（复杂系统 bug 多）、拖慢迭代（改个小功能要在复杂架构里穿行）。正确做法是"为当前规模设计 + 预留扩展点"——架构上留接口（如限流做成可插拔模块，未来要加熔断不用重构），但不实现未使用的复杂度。面试官问"规模涨 10 倍怎么办"是考你"有没有扩展性意识"，标准答法是"架构上留了 XX 扩展点，涨 10 倍时加 YY 即可，不用推倒重来"，而不是"现在就按 10 倍设计"。YAGNI（You Aren't Gonna Need It）原则。

### 第五层：验证与沉淀

**Q：你怎么验证场景题答得好不好，而不是自己觉得讲清楚了面试官没 get 到？**

看面试官的反馈信号。正向信号：面试官顺着你的方案深问细节（说明方案有料他想挖）、问你"还做过什么类似的"（认可能力想多了解）、讨论替代方案（把你当同行交流）。负向信号：面试官频繁打断纠错（方案有硬伤）、沉默不追问（方案平淡无亮点）、提前转下一题（这道题答完了没深挖价值）。面试后自我复盘：对照"五步框架"看哪步讲得不足（需求没澄清？决策没讲 trade-off？风险没覆盖？），下次改进。最好能找朋友复盘录音，自己讲的和面试官听的有偏差（你以为讲清了其实跳步了），第三方视角能发现盲点。

**Q：场景题的答题能力怎么长期提升，而不是每次面试临时练？**

平时用"场景拆解练习"积累。每周挑一个产品（自己用的或新闻里的 AI 产品），花 30 分钟按五步框架拆解它的架构（需求/架构/决策/风险/评估），写下来。比如拆解"Cursor 怎么做代码补全"——需求（低延迟补全）、架构（本地小模型 + 云端大模型分层）、决策（为什么本地优先）、风险（补全错误怎么处理）、评估（接受率）。这种拆解练多了，面试遇到任何场景都能快速套框架，因为底层是"系统化拆解产品"的能力。另外实战项目是最好的素材——自己做过并在生产踩过坑的，讲出来比任何模拟都真。所以长期提升 = 多拆解别人的产品 + 多做自己的真实项目。

## 结构化回答

**展开框架：**
1. **答题框架** — 需求→架构→决策→风险→评估
