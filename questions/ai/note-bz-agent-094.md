# agent 模拟面试（结合项目场景题）怎么答？

## 一、场景题的答题框架

```
┌──────────────────────────────────────────────────┐
│            场景题答题五步法                          │
├──────────────────────────────────────────────────┤
│                                                    │
│  Step 1: 需求澄清（2分钟）                          │
│    问清楚：用户是谁/规模多大/关键约束              │
│    展示：你不急于解题，先理解问题                  │
│                                                    │
│  Step 2: 架构设计（5分钟）                          │
│    画出：整体架构图（分层）                        │
│    说明：每个组件的职责                            │
│                                                    │
│  Step 3: 关键决策（5分钟）                          │
│    解释：为什么选A不选B                            │
│    展示：技术选型能力和tradeoff分析               │
│                                                    │
│  Step 4: 风险应对（3分钟）                          │
│    识别：可能的失败模式                            │
│    方案：限流/熔断/降级/兜底                       │
│                                                    │
│  Step 5: 评估指标（2分钟）                          │
│    定义：怎么衡量效果                              │
│    展示：数据驱动的思维                            │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、模拟场景题示例

### 题目：设计一个企业内部知识库问答Agent

```
面试官：你们公司有10万份内部文档，要做一个问答Agent，
       日活5000，怎么设计？
```

### 参考回答

**Step 1: 需求澄清**

```
"我先确认几个问题：
  - 文档类型？（Word/PDF/Confluence/代码？）
  - 用户场景？（员工查SOP/技术文档/政策？）
  - 准确性要求？（必须准确还是允许参考？）
  - 是否需要权限控制？（不同部门看不同文档？）
  - 延迟要求？（秒级还是可以等？）"
→ 展示你不急于解题，先理解需求
```

**Step 2: 架构设计**

```
"整体架构分四层：

  ┌─────────────────────────────────────┐
  │  接入层：Web/IM/搜索框               │
  ├─────────────────────────────────────┤
  │  Agent层：                          │
  │  意图理解 → RAG检索 → 生成回答      │
  │  （含权限过滤/引用标注）             │
  ├─────────────────────────────────────┤
  │  数据层：                           │
  │  文档处理Pipeline + 向量库 + 元数据 │
  ├─────────────────────────────────────┤
  │  基础设施：LLM网关/监控/审计        │
  └─────────────────────────────────────┘"
```

**Step 3: 关键决策**

```
"几个关键设计决策：

  1. 检索方案：向量+BM25混合检索+Rerank
     为什么：向量擅长语义，BM25擅长关键词(产品名/代码)
            互补，Rerank保证精度
     tradeoff：比纯向量慢，但准确率高15%

  2. 分块策略：父子分块(检索300字/返回1500字)
     为什么：小块检索精准，大块上下文完整
     tradeoff：存储增加，但回答质量更好

  3. 权限控制：检索时元数据过滤
     为什么：不同部门只能看自己的文档
     tradeoff：索引时需打权限标签，增加维护成本"
```

**Step 4: 风险应对**

```
"主要风险和应对：

  1. 检索不到 → 混合检索+查询改写提升召回
  2. 幻觉 → Prompt约束+faithfulness检测
  3. 高并发 → 限流+缓存+模型路由
  4. 文档更新 → 增量索引(小时级更新)
  5. 敏感信息 → 输出脱敏+权限过滤"
```

**Step 5: 评估指标**

```
"评估体系：

  检索质量：recall@5 > 85%, precision > 80%
  生成质量：faithfulness > 90%
  业务指标：
    - 问答准确率（人工抽检）
    - 用户满意度（点赞率 > 85%）
    - 平均响应时间（< 3s）
    - 月活/留存
  
  建立Bad Case闭环持续优化。"
```

## 四、答题技巧

```
DO（加分）：
  ✓ 先问澄清问题（展示思考）
  ✓ 画架构图（更直观）
  ✓ 讲tradeoff（为什么选A不B）
  ✓ 给量化指标（展示数据思维）
  ✓ 承认不确定（诚实比硬编好）
  ✓ 从全局到细节（先架构再深入）

DON'T（减分）：
  ✗ 一上来堆技术名词
  ✗ 不问需求直接解题
  ✗ 只说用什么不说为什么
  ✗ 忽略非功能需求（性能/安全/成本）
  ✗ 答不上来就沉默（说思路也行）
  ✗ 过度设计（简单问题复杂化）
```

## 五、面试加分点

1. **五步框架**：需求→架构→决策→风险→评估，结构化答题
2. **先问后答**：澄清需求展示成熟——别急于表现
3. **tradeoff 是核心**：面试官最想听"为什么这么选"——这区分资深和初级
