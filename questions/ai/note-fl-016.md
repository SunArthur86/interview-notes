# 【字节飞连面经】算法：合并两个有序链表（LeetCode 21）

## 一、核心思路：双指针 + dummy head

```python
# Definition for singly-linked list.
# class ListNode:
#     def __init__(self, val=0, next=None):
#         self.val = val
#         self.next = next

class Solution:
    def mergeTwoLists(self, l1: ListNode, l2: ListNode) -> ListNode:
        dummy = ListNode()    # 哨兵节点
        cur = dummy
        while l1 and l2:
            if l1.val <= l2.val:    # <= 保证稳定（相等时 l1 优先）
                cur.next = l1
                l1 = l1.next
            else:
                cur.next = l2
                l2 = l2.next
            cur = cur.next
        cur.next = l1 if l1 else l2   # 直接接剩余，别再 while
        return dummy.next
```

## 二、为什么用 dummy head

**不用 dummy**：头节点要特判（第一个节点要单独赋值，因为没有一个"前驱"可以 `.next`）。

**用 dummy**：所有节点都通过 `cur.next = x` 接入，统一处理，消除特判。最后返回 `dummy.next`（真正的头）。

```
不用 dummy:
  if not head: head = node; cur = head
  else: cur.next = node; cur = cur.next    # 特判头节点

用 dummy:
  cur = dummy
  cur.next = node; cur = cur.next           # 统一处理
  return dummy.next                          # 返回时跳过 dummy
```

## 三、复杂度

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | O(m+n) | 每个节点访问一次 |
| 空间 | **O(1)** | 只用几个指针，**没创建新节点**（在原节点上改 next） |

## 四、面试要点

1. **用 dummy head** 避免"头节点单独处理"的冗余
2. **比较用 `<=`** 保证稳定（相等时 l1 优先，保持原相对顺序）
3. **最后 `cur.next = l1 if l1 else l2`** 直接接上剩余的——别傻乎乎再 while 一遍
4. **面试默认要求迭代**（递归在长链表上会爆栈，递归深度 = 链表长度）

## 五、递归写法（≤5 行，但面试不推荐）

```python
def mergeTwoLists(self, l1, l2):
    if not l1 or not l2:
        return l1 or l2
    if l1.val <= l2.val:
        l1.next = self.mergeTwoLists(l1.next, l2)
        return l1
    else:
        l2.next = self.mergeTwoLists(l1, l2.next)
        return l2
```

**不推荐原因**：递归深度 = 链表长度，长链表（>1000）会 `RecursionError`。

## 六、追问：合并 K 个有序链表

**解法 1：最小堆**
```python
import heapq
def mergeKLists(lists):
    heap = [(head.val, i, head) for i, head in enumerate(lists) if head]
    heapq.heapify(heap)
    dummy = cur = ListNode()
    while heap:
        val, i, node = heapq.heappop(heap)
        cur.next = node
        cur = cur.next
        if node.next:
            heapq.heappush(heap, (node.next.val, i, node.next))
    return dummy.next
```
时间 O(N log K)，N=总节点数，K=链表数。

**解法 2：分治两两合并**——时间也是 O(N log K)。

## 七、扩展

- **链表 vs 数组合并**：链表合并空间 O(1)（改指针），数组合并空间 O(m+n)（要新建数组）
- **原地合并两个有序数组**（LeetCode 88）：从后往前填，避免覆盖——`nums1` 末尾有预留空间
- **合并后的稳定性**：用 `<=`（不是 `<`）保证相等元素保持原顺序，这在排序稳定性场景很重要

## 记忆要点

- 核心思路：双指针遍历比较，用 dummy head（哨兵节点）统一接入操作消除头节点特判。
- 末尾提速：剩余链表直接用 cur.next = l1 if l1 else l2 接上，无需再进循环。
- 复杂度：时间 O(m+n)，空间 O(1) 因为只改指针不建新节点。
- 考点细节：迭代优于递归防长链表爆栈，判断用 <= 保证排序稳定性。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：dummy head 你说是"消除头节点特判"。但特判头节点就两行代码（if not head），为什么非要用 dummy？是不是过度设计？**

不是过度设计，因为特判不仅是"两行代码"的问题，是"逻辑分叉"的问题。不用 dummy 时，每个节点接入都要判断"是不是头节点"，这个判断在循环内每次都执行（或用 if/else 分两段写），既增加心智负担（读代码要想着头节点特殊情况），又容易写错（边界 bug 高发）。dummy 的价值是"统一所有节点的接入操作为 `cur.next = x`"，代码线性、无分支、不易错。在链表题里，边界 bug（如忘记处理空链表、漏接最后一个节点）是最常见的错误来源，dummy 从结构上消除了这类 bug。两行特判的"简单"是表面的，维护时的心智成本和 bug 风险才是真实代价。这是"用空间换简单性和正确性"的工程权衡（dummy 只占一个节点的 O(1) 空间）。

### 第二层：证据与定位

**Q：你的代码 `cur.next = l1 if l1 else l2` 直接接剩余链表。但这里假设"剩余的链表已经是有序的"——这个假设成立吗？什么情况下会失效？**

成立。因为输入前提是"两个有序链表"，while 循环是"两链表都非空时取较小者"，当一链表空了退出循环时，另一链表剩余的部分一定是：原链表的尾部连续段，且这些节点的值都 >= 已合并部分的最后一个值（因为能剩下说明它们比另一链表的全部元素大或另一链表已空）。有序链表的尾部连续段自然有序，所以直接接上整体仍有序。失效的唯一情况是"输入不是有序链表"——如果 l1 或 l2 本身无序，这个假设不成立。所以这个优化的前提是"输入有序"，这是题目保证的。如果面试官追问"如何验证输入有序"，可以加个 O(n) 的预检，但 LeetCode 题目默认输入合法。

### 第三层：根因深挖

**Q：你说"迭代优于递归防爆栈"。但 Python 的默认递归深度限制是 1000，链表长度 < 1000 时递归没问题。为什么要一律推荐迭代？**

因为"链表长度 < 1000"是不可靠的假设。面试题的测试用例可能有长链表（如 10^4 或 10^5 节点），生产环境（如合并两个排序的日志链表）更可能超长。递归的栈空间是 O(n)（每层调用占栈帧），迭代是 O(1)。即使题目说"长度 < 1000"，工程习惯上应选"无隐含限制"的迭代，避免在边界翻车。更深的原因是递归的常数开销大（函数调用、栈帧创建），即使不爆栈，性能也差于迭代。唯一递归更优的场景是"代码简洁度"（递归 5 行 vs 迭代 15 行）——这在"面试快速写对"时有价值，但一旦被问"长链表怎么办"就要能改迭代。所以默认迭代是"安全和性能"优先，递归是"快速验证逻辑"时的备选。

**Q：那如果面试官要求"合并后保持稳定性"（相等元素保持原相对顺序），你用 `<=` 保证 l1 优先。但为什么是 l1 优先而不是 l2？这个选择有依据吗？**

"稳定性"的定义是"相等元素保持它们在原输入中的相对顺序"。如果 l1 的元素在输入中"排在前面"（l1 是第一个参数，语义上更"靠前"），那么相等时 l1 先接入就保持了"l1 的元素在前"这个顺序。这是约定俗成的——把 l1 当"左序列"、l2 当"右序列"，稳定排序要求左序列的相等元素先出。如果反过来用 `<`（相等时 l2 先），就破坏了稳定性（右序列的相等元素跑到了左前面）。这个选择本身的依据是"输入参数的顺序代表原始顺序"，l1 在前所以 l1 先。合并排序（merge sort）的稳定性就依赖这个细节，如果合并步骤不稳定，整个合并排序就不稳定（虽然 Python 的 sorted 是 TimSort，但底层合并也遵循这个原则）。

### 第四层：方案权衡

**Q：合并 K 个有序链表你给了最小堆解法，O(N log K)。但分治两两合并也是 O(N log K)。为什么推荐堆而不是分治？两者怎么选？**

两者复杂度相同但工程特性不同。最小堆的优势是"自然处理 K 不固定"的场景——流式数据（链表还在增长）也能处理（新链表加入就 push 进堆），且实现直观（一个堆管所有链表的当前最小）。分治的优势是"可以复用 mergeTwoLists 函数"（两两合并递归或迭代），代码复用性好，且某些硬件上分治的 cache 局部性更好（连续合并相邻链表）。选型：K 固定且小（如 K=3）用顺序合并（连调 mergeTwoList 两次，简单）；K 大（> 10）用堆（顺序合并退化成 O(NK)）；流式或 K 动态变化用堆。实践中堆更通用，分治更"教科书"。如果面试官追问"能否做到 O(1) 额外空间"，分治如果用迭代（bottom-up）可以做到 O(1)，堆不行（堆本身 O(K) 空间）。

**Q：链表合并空间 O(1)（改指针），数组合并空间 O(m+n)（新建数组）。为什么链表能原地、数组不能？**

因为链表的"指针"是可拆卸的——改 next 指针把节点从原链表"摘下来"接到新链表，不破坏其他节点的访问（通过 next 仍能遍历）。数组的元素是连续存储的，"原地合并"意味着要把一个数组的元素插到另一个数组的中间，插入点之后的元素全部要后移（O(n) 搬移），且如果两个数组没有预留空间根本插不进去。LeetCode 88（合并两个有序数组）能"原地"是因为 nums1 末尾预留了 m+n 的空间，从后往前填避免覆盖——这是特殊的"预留空间"前提，不是通用的原地合并。所以链表的原地合并是"指针特性"带来的优势，数组合并本质要 O(m+n) 空间（或特殊预留）。这是数据结构的本质差异，不是算法优劣。

### 第五层：验证与沉淀

**Q：你怎么证明你写的合并算法是对的？除了跑 LeetCode 测试用例，还有什么验证方法？**

三层验证。一是边界用例：空链表（l1 空、l2 空、都空）、单节点链表、一长一短链表、有重复值的链表、全相同值链表——这些是 bug 高发区，手测确认。二是性质测试（property-based testing）：随机生成两个有序链表，合并后验证三个性质——结果有序、结果长度 = m+n、稳定性（相等元素保持原顺序）。用 hypothesis 库自动生成大量随机用例跑，比手写用例覆盖广。三是对照基准：用语言内置的排序（如 Python `sorted(l1+l2)`）作为基准，对大量随机输入对比结果是否一致（注意内置排序可能不稳定，要验证稳定性时用 `sorted(key=..., stable=True)`）。证明逻辑是"性质 + 大规模随机测试 + 基准对照"，比几个固定用例的信心强得多。

**Q：怎么让团队在写链表/指针类代码时不犯常见的空指针、边界错误？**

沉淀编码规范和技巧清单而非靠个人经验。一是规范：所有链表题默认用 dummy head（强制），消除头节点特判；所有指针解引用前确认非空（防御性）。二是技巧清单：链表断链（改 next 前先存 next 指针）、环检测（快慢指针）、边界（while 条件用 `and` 还是 `or` 想清楚）。三是测试模板：链表题的测试用例必须覆盖空、单节点、两节点、长链表、有环（如适用），CI 模板化。四是 Code Review checklist：链表 PR review 时重点看"空指针、断链、死循环（环）、内存泄漏（手动管理语言）"。让规范和 checklist 把常见坑前置到编码和 review 阶段，而不是线上翻车才发现。
