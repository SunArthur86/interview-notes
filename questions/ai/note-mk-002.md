# 【月之暗面面经】桌面 AI 产品接入本地文件和目录时，前端如何设计权限提示？

## 一、为什么桌面文件权限如此重要

Web 应用的安全模型基于**同源策略 + 浏览器沙箱**，用户天然知道网页只能看到上传的文件。但桌面应用不同——它可以直接调用操作系统 API 访问文件系统。这意味着：

- AI 可能读取到用户不希望暴露的敏感文件
- AI 可能覆盖或删除已有文件
- AI 可能将本地文件内容上传到云端推理

**如果权限设计不到位，用户根本不敢用你的产品。**

## 三、权限提示的四个设计阶段

### 阶段 1：授权前——明确告知

```
┌─────────────────────────────────────────────────────┐
│  📁 Kimi 请求访问以下目录                            │
│                                                     │
│  路径：/Users/you/projects/my-app                    │
│  文件数：127 个文件                                  │
│  总大小：23.4 MB                                     │
│                                                     │
│  Kimi 将执行以下操作：                                │
│  ✓ 读取目录下所有文件内容（用于代码分析）              │
│  ✓ 在目录下创建分析报告（report.md）                  │
│  ✗ 不会修改任何已有文件                               │
│  ✗ 不会上传文件到云端（仅本地索引）                     │
│                                                     │
│  授权有效期：本次任务                                  │
│                                                     │
│  [ 仅授权读取 ]  [ 授权读取+写入 ]  [ 取消 ]          │
└─────────────────────────────────────────────────────┘
```

**设计要点**：
- 明确列出**会做什么**和**不会做什么**
- 区分读取和写入权限
- 设置授权有效期（本次任务 / 本次会话 / 永久）
- 显示文件数量和大小，让用户有量化感知

## 四、权限协议设计

```typescript
// 权限协议——每个Agent接入时必须声明
interface PermissionSpec {
  // 需要的权限
  required: {
    read?: string[];      // 需要读取的路径模式
    write?: string[];     // 需要写入的路径模式
    index?: string[];     // 需要索引的目录
    network?: boolean;    // 是否需要网络访问
    system?: string[];    // 需要的系统级能力
  };
  
  // 权限说明（展示给用户）
  description: {
    what: string;         // 会做什么
    why: string;          // 为什么需要
    output: string;       // 会产出什么
    retention: string;    // 数据保留多久
  };
  
  // 可选权限
  optional?: PermissionSpec['required'];
  
  // 撤销回调
  onRevoke: () => Promise<void>;
}
```

## 五、最佳实践总结

| 实践 | 说明 | 优先级 |
|------|------|--------|
| 最小授权 | 默认只请求当前任务必须的权限 | P0 |
| 预览变更 | 写入前展示 diff，让用户看到具体改动 | P0 |
| 分级确认 | 低风险自动通过，高风险二次确认 | P0 |
| 可撤销 | 设置页随时查看和撤销授权 | P0 |
| 审计日志 | 记录所有文件操作，支持回溯 | P1 |
| 沙箱模式 | 可选的只读模式，AI 不能写任何文件 | P1 |
| 权限过期 | 长时间未使用的授权自动失效 | P2 |

## 六、常见坑

- **一次性请求所有权限**：用户看到"访问所有文件"会直接拒绝，应该按需请求
- **写入无预览**：AI 直接覆盖文件不给 diff，用户无法控制风险
- **没有撤销路径**：授权后无法收回，用户只能卸载应用
- **忽略 .gitignore**：AI 索引目录时不跳过 node_modules/.env 等敏感文件

## 记忆要点

- 打破沙箱限制：桌面端可直接操作文件系统，权限管理是建立信任的核心
- 权限分层金字塔：L1引用、L2读取、L3写入确认、L4系统级操作
- 设计核心：授权前必须明确告知操作范围与有效期限，危险操作强制二次确认

### 第一层：目标与动机

**Q：桌面 AI 产品为什么要搞四层权限金字塔（L1 引用 / L2 读取 / L3 写入 / L4 系统级），而不是像 Web 一样用一个"允许访问文件"的二值开关？**

因为桌面应用拥有 OS 级文件访问能力，二值开关要么过度授权（用户给了"所有文件"权限，AI 就能读到 .env、密钥、聊天记录），要么过度限制（用户拒绝后 AI 什么都做不了）。四层金字塔的本质是把"文件操作"按可逆性和影响面拆分：L1 引用（看路径、显示图标）完全可逆零风险可自动通过；L2 读取（读内容、索引目录）涉及隐私但可逆，需告知用途；L3 写入（覆盖、删除）不可逆，必须 Diff 预览 + 二次确认；L4 系统级（执行命令、改注册表）影响整机安全，需单独审批。这样拆分后，80% 的低风险操作（引用、读取元数据）可以无感通过，只有真正危险的 20% 才打扰用户——既建立了信任又不牺牲体验。

### 第二层：证据与定位

**Q：你怎么定位"用户不敢授权"这个问题的根因是权限设计问题，而不是产品价值没讲清楚？**

用授权漏斗的数据定位。在授权弹窗埋三个埋点：弹窗曝光数、点击"仅授权读取"的转化率、点击"取消"的流失率。如果曝光到"仅授权读取"的转化率低于 40%，且取消率高于 30%，根因是权限设计——用户看到了权限请求但不信任。这时再看取消用户的热力图：如果他们停留在弹窗超过 5 秒才点取消（说明在犹豫），大概率是"授权范围不透明"（没说清会读哪些文件、保留多久）；如果 1 秒内点取消，是"信任未建立"（首次使用就要求全局权限）。如果转化率高于 60% 但后续任务完成率低，那根因才是产品价值——用户授权了但发现 AI 做的事没用。两种问题的解法完全不同：前者改权限 UI（加分层、加有效期、加 Diff 预览），后者改产品定位。

### 第三层：根因深挖

**Q：为什么写入操作必须强制 Diff 预览 + 二次确认，而读取操作只需告知即可？根因是什么？**

根因是"操作的可逆性"决定了"确认的成本合理性"。读取操作（L2）即使 AI 读了不该读的文件（如 .env 里的密钥），只要不上传到云端、不展示给用户，影响是可控的（数据还在原位），所以告知用途 + 事后审计日志就够。但写入操作（L3）一旦执行，原文件被覆盖、用户数据丢失，回滚成本极高（桌面端没有版本控制）。Diff 预览的价值是让用户在不可逆操作发生前看到"具体会改哪几行、加什么、删什么"，把"盲签"变成"知情同意"。二次确认的价值是防止 AI 的"权限蔓延"——用户以为只授权了读，结果 AI 顺便写了。所以根因不是"写入比读取更危险"，而是"写入的失败成本不可逆，必须用确认机制把不可逆操作降级为可逆决策"。

**Q：那如果 Diff 预览本身让用户看不懂（比如改了 500 行 JSON），为什么不直接信任 AI 自动写入？**

因为"用户看不懂 Diff"和"应该信任 AI 自动写入"是两个独立问题，不能用前者的困难论证后者。看不懂 Diff 的解法是优化 Diff 展示——把 500 行 JSON 的变更聚合成"新增依赖 3 个、修改配置项 5 个、删除废弃字段 2 个"的结构化摘要，再让用户决定是否展开看细节。而"信任 AI 自动写入"的代价是：一旦 AI 幻觉覆盖了关键文件（如删了 package.json 里的 react 依赖），用户毫无感知，排查时连改了什么都不知道。更合理的折中是"分级自动写入"：低风险变更（如新增字段、格式化）自动应用并记录到审计日志，高风险变更（如删除、覆盖已有值）必须用户确认。关键是"自动"的边界由影响面决定，而不是由"用户看不懂"来倒推放权。

### 第四层：方案权衡

**Q：授权有效期你设计成"本次任务 / 本次会话 / 永久"三档，为什么不学 Web 端"记住选择"的二值开关？**

三档的核心差异是"权限的衰减速率"和"任务粒度的匹配度"。本次任务：任务结束立即失效，适合一次性操作（如"分析这个目录生成报告"），风险最低但打扰最多；本次会话：App 重启失效，适合短期连续工作（如"连续改 5 个文件的样式"），平衡体验和安全；永久：除非手动撤销否则一直有效，适合长期信任的工作目录（如"我的项目文件夹"）。二值"记住选择"的问题在于：用户一旦勾选了"记住"，权限就永久有效，但用户的工作场景会变（今天分析项目、明天处理私人文件），永久授权会把私人文件也暴露给 AI。三档让用户按场景选衰减速率，本质上是用"时间维度"细化了最小权限原则——权限不仅要最小化范围，还要最小化时效。

**Q：那如果用户嫌三档太麻烦，普遍选"永久"来图省事，为什么不直接砍掉有效期，默认全部永久？**

因为"用户普遍选永久"恰恰说明权限设计出了问题，而不是有效期设计多余。用户选永久通常是因为"每次任务都要重新授权太烦"——解法是优化授权体验而不是放弃时效控制。正确做法是引入"信任累积"机制：用户对同一目录授权过 N 次且无异常操作（没有读取被拒绝的记录、没有写入冲突），自动把该目录的权限从"本次任务"升级到"本次会话"，再升级到"永久"，并用绿色徽章标识"已建立信任的目录"。同时把"永久"权限做定期复审——每 30 天弹一次轻提示"你已授权 Kimi 永久访问 X 目录，是否仍然需要？"，让用户有机会清理遗忘的授权。砍掉有效期会让权限只增不减，半年后用户的授权列表膨胀到几十个目录，反而更不安全。

### 第五层：验证与沉淀

**Q：你怎么验证"Diff 预览 + 二次确认"真的降低了事故率，而不是只是增加了用户的点击次数？**

用两个对偶指标验证：(1) 写入事故率 = 用户投诉"文件被误改"的工单数 / 总写入操作数，Diff 上线前后对比，如果事故率下降 50% 以上说明有效；(2) 确认通过率 = 用户点击"接受"的次数 / Diff 预览曝光次数，如果通过率高于 95% 说明 Diff 没起到拦截作用（用户无脑点接受），这时要看"Diff 展示时长"——如果用户 1 秒内就点接受，说明 Diff 太长或太技术化，用户根本没看。理想状态是：事故率显著下降 + 确认通过率在 70-90% 之间（说明用户在认真看 Diff 并拒绝了部分有问题的变更）。如果确认通过率是 99% 但事故率没降，说明 Diff 设计失效，要改成"只展示变更摘要 + 高亮风险字段（如删除、覆盖）"来强制用户关注关键变更。

**Q：怎么让团队在新 Agent 接入时，自觉声明权限协议（PermissionSpec），而不是直接调文件系统 API 绕过权限层？**

把权限协议做成"工程强制约束"而非"文档建议"。第一，文件系统访问层封装成一个统一的 `FileAccessGateway`，所有读写必须经过它，直接调 Node.js fs 模块的代码在 ESLint 规则里禁用（eslint-plugin-fs-ban）；第二，FileAccessGateway 要求调用方传入 PermissionSpec，没有声明权限的调用直接抛异常拒绝执行；第三，在 CI 流水线加权限扫描——扫描每个 Agent 的 PermissionSpec，如果 required 权限超出其功能需要（如一个"生成报告"的 Agent 申请了 system 级权限），CI 失败阻塞合并；第四，每周生成"权限使用报告"——统计每个 Agent 实际用到的权限 vs 声明的权限，声明了但从没用过的权限自动降级或移除，倒逼开发者精确声明。这样权限协议就从"自觉"变成了"不声明就跑不起来"的硬约束。

## 结构化回答

**30 秒电梯演讲：** 桌面AI产品需要访问本地文件和目录，前端必须设计清晰的权限边界——按粒度授权、事前说明用途、写入二次确认、事后可撤销——就像你雇了一个新助理，你不会直接把整个办公室钥匙给他。

**展开框架：**
1. **按文件、目** — 按文件、目录、应用动作区分授权粒度
2. **授权前说明** — 授权前说明会读取什么、产出什么、保留多久
3. **高影响写入** — 高影响写入动作要二次确认

**收尾：** 您想深入聊：如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
