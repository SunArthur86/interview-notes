# 【月之暗面面经】AI-Native 桌面产品怎样做结果回放，才能支持问题复盘和用户学习？

## 一、回放的两种价值

| 价值类型 | 目标用户 | 核心场景 |
|---------|---------|---------|
| 问题复盘 | 开发者/高级用户 | "AI生成的PPT第3页数据错了，是哪一步引入的错误？" |
| 用户学习 | 所有用户 | "这个任务做得很好，AI是怎么一步步完成的？我下次怎么更好地给指令？" |

## 四、五段数据模型

```typescript
interface ReplayTimeline {
  taskId: string;
  
  // ① 输入段
  inputs: {
    timestamp: number;
    rawInput: string;            // 用户原始输入
    inputRefs: InputRefSnapshot[]; // 引用的文件/URL
    contextSlots: ContextSlot[];  // 激活的上下文
  };
  
  // ② 计划段
  plan: {
    timestamp: number;
    steps: PlanStep[];           // AI分解的执行步骤
    reasoning: string;           // AI为什么这样分解
  };
  
  // ③ 执行段
  executions: ExecutionStep[];
  
  // ④ 产物段
  artifacts: ArtifactSnapshot[];
  
  // ⑤ 人工修改段
  modifications: Modification[];
}

interface ExecutionStep {
  index: number;
  stepName: string;
  startedAt: number;
  completedAt: number;
  
  // 消耗的输入
  consumedInputs: string[];      // 引用了哪些输入
  
  // AI推理过程
  promptSent?: string;           // 发给LLM的prompt
  responseReceived?: string;     // LLM的回复
  tokenUsage?: { input: number; output: number };
  
  // 产出
  outputArtifacts?: string[];    // 产出了哪些产物
  outputSummary?: string;        // 本步骤产出摘要
  
  // 异常
  warnings?: string[];
  errors?: string[];
}
```

  // 5. 高亮可能引入问题的部分
  highlightPotentialIssue(promptSection);
}

## 七、常见坑

- **回放只展示最终结果**：没有中间过程，用户无法理解AI是怎么得到结果的
- **无法从产物跳转到执行步骤**：发现问题后无法定位是哪一步引入的
- **不能沉淀模板**：每次都从头写指令，优秀任务无法复用
- **回放数据太大**：记录了所有中间token，导致存储和加载都很慢

## 记忆要点

- 核心双价值：面向开发者做问题复盘找Bug，面向普通用户做过程回放学习AI逻辑。
- 五段式回放模型：完整记录输入、计划、执行、产物、人工修改全生命周期的切片。
- 时间线交互设计：提供类似视频播放器的进度轴，可自由拖拽定位到特定执行步骤。
- 快照上下文：选中任一历史节点时，能完整展示当时的输入素材、AI推理与产物变化。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：回放系统为什么不只是给开发者调试用，还要给普通用户做"过程学习"？普通用户真的关心 AI 的执行过程吗？**

因为"用户对 AI 的信任"取决于"过程是否可理解"。AI 执行是黑盒——用户只看到输入和最终产物，中间发生了什么完全不知道。当产物质量好时用户不知道"为什么好"（无法复现），当产物质量差时用户不知道"为什么差"（无法改进）。回放系统的用户学习价值是打破黑盒：让用户看到"AI 在执行步骤 2 时引用了 PDF 第 12 页的数据，生成了定价对比表"——这样用户下次就知道"要改定价数据，应该更新 PDF 的第 12 页"。没有回放，用户和 AI 之间是"盲盒互动"——用户只能不断换输入碰运气，无法建立有效的协作策略。有回放后，用户能从优秀任务中学习"原来这样描述输入会得到更好的结果"，把隐性经验显性化。所以回放不是"调试工具的附属"，而是"建立人机信任的核心体验"——尤其在 AI 输出不确定的前提下，可回放是用户建立心理安全感的关键。

### 第二层：证据与定位

**Q：你怎么定位"用户看完回放还是不理解 AI 为什么这么做"是回放设计问题，还是 AI 执行过程本身不透明？**

用回放交互埋点定位。在回放页埋三个指标：(1) 单步停留时长——用户在某个执行步骤的平均查看时长，如果某步停留超过 30 秒且反复来回看，说明该步的展示不够清晰（用户在努力理解）；(2) Prompt 查看率——用户点击"查看完整 Prompt"的次数 / 回放查看次数，如果这个率高，说明默认展示（AI 执行摘要）不够，用户要看原始 Prompt 才能理解；(3) 定位跳转成功率——用户从产物问题点点击"定位来源"后，是否成功找到引入问题的步骤（通过用户后续行为判断：找到后通常会点击"更新数据源重跑"，没找到则关闭回放页放弃）。如果定位跳转成功率低于 50%，说明回放的"产物→步骤"关联逻辑有断链（可能某些步骤没记录 outputArtifacts）。如果三个指标都差，根因是回放设计；如果指标正常但用户仍抱怨"不理解"，可能要加 AI 推理过程的自解释（每步附带"我为什么这样做"的自然语言说明）。

### 第三层：根因深挖

**Q：为什么回放要按"输入→计划→执行→产物→人工修改"五段组织，而不是简单按时间线展示所有事件日志？**

因为"按时间线平铺事件日志"对用户毫无意义——一个任务可能产生几百条事件（每次 LLM 调用、每次文件读写、每次中间状态更新），平铺后用户看到的是一堆无结构的日志条目，完全无法理解"AI 整体在做什么"。五段组织的本质是"给无结构的事件赋予语义层级"：输入段回答"用户给了什么"；计划段回答"AI 打算怎么做"；执行段回答"AI 实际怎么做的"；产物段回答"AI 做出了什么"；人工修改段回答"用户改了什么"。用户回放时的心智是"我想知道某个环节发生了什么"——他可以直接跳到对应段，而非在几百条日志里翻找。五段的另一个价值是"跨任务对比"——两个类似任务的回放可以按段对齐（对比两者的计划段差异、执行段差异），帮用户理解"为什么同样的输入得到不同的结果"。平铺日志无法做这种语义级对比。所以五段不是"过度结构化"，而是"按用户理解模型组织事件"。

**Q：那如果某个任务的执行过程不符合五段模型（如 AI 中途推翻计划重新规划），为什么不放弃五段改成"事件流"来适配所有情况？**

因为"不符合五段的异常情况"恰恰是回放最有价值的部分，应该高亮而非隐藏。AI 中途推翻计划重新规划（re-planning）是一个重要信号——说明初始计划有误或输入有变化，用户最需要理解"为什么 AI 改变了计划"。五段模型可以通过"嵌套"来处理这种情况：在执行段中标记一个"re-planning 事件"，展开后显示新的计划子段和执行子段，形成"计划 A→执行→发现异常→计划 B→执行"的嵌套结构。这样既保留了五段的语义清晰性，又容纳了动态调整的现实。放弃五段改事件流的代价是丧失语义层级——所有事件（包括 re-planning）都变成同质的日志条目，用户无法快速定位"这是计划变更还是正常执行"。所以解法是"五段 + 异常事件高亮 + 嵌套子段"，而非"退化为无结构的事件流"。结构化的价值正是在异常情况下凸显——用户能立刻看到"这里发生了一次 re-planning"，比在一堆日志里找"哪个事件是计划变更"高效得多。

### 第四层：方案权衡

**Q：回放数据你记录"全量中间状态"（所有 Prompt、所有响应、所有 Token）还是"抽样快照"（只在关键节点记录），怎么选？**

选"分层记录"——关键节点全量记录、中间过程抽样记录。根因是"存储成本"和"回放完整性"的权衡。全量记录的问题：一个 5 分钟的任务可能调用 LLM 20 次，每次的 Prompt + 响应约 5000 token，总共 10 万 token 约 400KB 文本，加上输入文件快照和中间状态，单个任务的回放数据可能达 10MB。100 个历史任务就是 1GB，本地存储和加载都成问题。完全抽样的问题：只记录开始和结束，中间过程丢失，用户无法定位"哪一步引入了错误"。分层记录的方案：(1) 关键节点（计划生成、每次产物变更、异常事件）全量记录 Prompt + 响应 + 上下文快照；(2) 中间过程（进度更新、状态查询）只记录时间戳和摘要；(3) 输入文件只存引用（路径 + hash）而非完整内容副本（回放时按需重新加载）。这样单个任务的回放数据控制在 500KB-1MB，100 个任务约 50-100MB，可接受。

**Q：那如果用户反馈"回放时看不到完整的 Prompt 细节，定位不到问题"，为什么不全量记录所有中间 Prompt？**

因为"看不到完整 Prompt"的根因通常是"关键节点的记录不够完整"而非"中间过程没全量记录"。解法是优化关键节点的定义和记录粒度：(1) 扩展关键节点——除了计划和产物变更，把"每次 LLM 调用"都定义为关键节点（因为每次调用都可能引入错误），全量记录其 Prompt 和响应；(2) 延迟加载——回放数据只存元数据（调用了哪次 LLM、耗时多久、Token 消耗多少），完整的 Prompt 和响应对象存到独立的 blob 存储按需加载（用户点击"查看完整 Prompt"时才从 blob 加载），避免初始加载时拉取全部数据；(3) 过期清理——回放数据保留 30 天，超期后只保留摘要（计划 + 产物 + 关键异常），减少长期存储压力。全量记录所有中间 Prompt 的代价是存储爆炸，且 90% 的中间 Prompt 用户永远不会看（只有出问题时才回放），记录它们是纯浪费。所以根因是"记录策略要匹配用户的查看模式"——用户只在关键节点看细节，就在关键节点全量记录，其他走按需加载。

### 第五层：验证与沉淀

**Q：你怎么证明"回放系统"真的帮助用户复盘和学习了，而不是只是个没人用的功能？**

用三个指标验证：(1) 回放使用率 = 任务完成后用户打开回放页的次数 / 总任务数——如果这个率 >15%，说明用户真的有回看需求（不是没人用）；(2) 问题定位成功率 = 用户从回放中成功定位到问题步骤并采取行动（如更新数据源重跑）的比例——如果 >60%，说明回放的"产物→步骤"关联有效；(3) 模板沉淀率 = 从优秀任务创建模板的次数 / 优秀任务数（用户评分 4 星以上的任务）——如果 >20%，说明回放的"学习复用"价值被认可。还要看一个定性指标：用户工单内容——如果工单从"AI 结果不对，帮我查"变成"我在回放里看到第 3 步引用了旧数据，能更新吗"，说明回放成功把"黑盒投诉"变成了"可定位的精确反馈"。

**Q：怎么让团队在新增 Agent 能力时，自觉接入回放系统（记录五段数据），而不是跑完任务就丢掉中间过程？**

把回放数据记录做成"任务执行框架"的强制约束。第一，TaskRunner 框架在任务执行时自动记录五段数据——开发者在实现 Agent 逻辑时调用框架提供的 recordInput/recordPlan/recordExecution/recordArtifact/recordModification 钩子，框架自动序列化为回放数据，不调用这些钩子的任务在执行时框架会拒绝推进（强约束）；第二，回放数据的存储是框架托管的——开发者不需要关心存在哪、怎么序列化，框架统一处理，降低接入成本；第三，CI 流水线做回放数据完整性检查——测试任务执行后验证五段数据是否齐全（有输入段无计划段则 CI 失败），确保不会出现"部分接入"的半成品；第四，提供回放数据分析看板——团队可以看到哪些 Agent 的回放使用率高（说明用户关心其过程）、哪些 Agent 的问题定位率高（说明其回放数据有效），用数据驱动团队优化回放质量。这样回放接入就从"自觉规范"变成了"框架强制 + CI 校验"。

## 结构化回答

**30 秒电梯演讲：** AI-Native桌面产品的回放不仅为排障服务，也帮用户学习如何更好地使用产品。回放页按输入→计划→执行→产物→人工修改五段组织，用户能看到每步用了哪些文件和指令，问题点位能直接回到对应产物版本，优秀任务可一键沉淀为模板。

**展开框架：**
1. **回放页按输** — 回放页按输入、计划、执行、产物和人工修改组织
2. **用户能看到** — 用户能看到哪一步用了哪些文件和指令
3. **问题点位能** — 问题点位能直接回到对应产物版本

**收尾：** 您想深入聊：如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
