# 【月之暗面面经】桌面端为什么要把结果编辑、确认和发布拆成三步？

## 一、为什么不能一步到位

```
一步到位的后果：
  用户请求 → AI直接覆盖文件 → 发现结果有问题 → 已无法回退 💥

三步拆分的流程：
  用户请求 → AI生成草稿 → 用户审查编辑 → 确认发布 → 可随时回看版本
     ↓            ↓            ↓           ↓
   发起        可逆          可逆       不可逆（但有版本历史）
```

## 三、各步骤详细设计

### Step 1: 草稿态（Draft）

AI生成的内容**不直接写入目标文件**，而是进入草稿区：

```typescript
interface DraftArtifact {
  id: string;
  taskId: string;
  
  // 草稿内容
  content: string;          // AI生成的完整内容
  
  // 目标信息（草稿不会写入目标）
  targetType: 'file' | 'directory' | 'clipboard' | 'publish';
  targetPath?: string;      // 计划写入的路径
  
  // 对比信息
  originalContent?: string;  // 如果目标文件已存在，保存原始内容
  
  // 状态
  status: 'draft';
  createdAt: number;
}
```

```typescript
// 局部接受实现
interface PartialAccept {
  changes: Array<{
    id: string;
    type: 'add' | 'modify' | 'delete';
    accepted: boolean;        // 是否接受此变更
    originalRange?: [number, number];  // 原始行范围
    draftRange?: [number, number];     // 草稿行范围
  }>;
}

function applyPartialAccept(draft: DraftArtifact, accept: PartialAccept): string {
  let result = draft.originalContent || '';
  
  for (const change of accept.changes) {
    if (!change.accepted) continue;
    
    switch (change.type) {
      case 'add':
        result = insertLines(result, change.draftRange, draft.content);
        break;
      case 'modify':
        result = replaceLines(result, change.originalRange, change.draftRange, draft.content);
        break;
      case 'delete':
        result = removeLines(result, change.originalRange);
        break;
    }
  }
  
  return result;
}
```

### Step 3: 发布态（Publish）

确认后才执行实际写入操作：

```typescript
async function publish(draft: DraftArtifact, reviewedContent: string) {
  // 1. 创建版本快照（可回退）
  const version = await versionStore.createSnapshot({
    taskId: draft.taskId,
    artifactId: draft.id,
    content: reviewedContent,
    previousContent: draft.originalContent,
  });
  
  // 2. 执行写入
  switch (draft.targetType) {
    case 'file':
      await fs.writeFile(draft.targetPath, reviewedContent);
      break;
    case 'directory':
      await exportToDirectory(draft.targetPath, reviewedContent);
      break;
    case 'clipboard':
      await clipboard.writeText(reviewedContent);
      break;
    case 'publish':
      await publishToRemote(reviewedContent);
      break;
  }
  
  // 3. 更新状态
  draft.status = 'published';
  
  // 4. 记录审计日志
  auditLog.record({
    action: 'publish',
    artifactId: draft.id,
    target: draft.targetPath,
    version: version.id,
    timestamp: Date.now(),
  });
}
```

## 五、不同产物类型的三步差异

| 产物类型 | 草稿态 | 编辑态 | 发布态 |
|---------|--------|--------|--------|
| 文本文件 | Markdown预览 | Diff视图+局部接受 | 写入文件 |
| 站点 | 内嵌预览 | 代码编辑+实时预览 | 导出HTML压缩包 |
| PPT | 幻灯片缩略图 | 逐页编辑 | 导出PPTX |
| 表格 | 表格预览 | 单元格编辑 | 导出XLSX |

## 六、常见坑

- **AI直接覆盖文件**：没有草稿缓冲层，用户无法控制风险
- **编辑态不支持局部接受**：只能全盘接受或全盘拒绝，用户无法精细控制
- **发布后没有版本历史**：出了问题无法回退到之前的版本
- **没有审计日志**：无法追踪谁在什么时候改了什么

## 记忆要点

- 核心是状态隔离与可逆：因为直接覆盖文件不可逆，所以需拆分为草稿态、编辑态和发布态。
- 强调控制权：草稿不直接写入目标，而是进入隔离区供用户审查并对比。
- 版本机制兜底：即使到最后发布态不可逆，也必须依赖版本历史支持随时回退。
- 防患于未然：前置拆分能有效拦截AI幻觉或错误覆盖，保障数据安全。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么 AI 生成的结果不能直接写入目标文件，非要先放草稿区走三步确认？多一道步骤不是降低效率吗？**

因为 AI 输出具有不确定性（幻觉、格式错误、数据不准），直接写入正式文件是不可逆操作（原文件被覆盖），一旦写入错误内容，用户的数据就丢了。三步拆分的本质是"可逆性设计"——在不可逆操作前增加缓冲层：草稿态是可逆的（草稿没写入文件，丢弃零成本），编辑态是可逆的（用户可以局部接受、修改、重生成），只有发布态是不可逆的（实际写入文件），但发布时有版本快照兜底（可回退到前一版本）。效率的代价是"多一次确认点击"，但收益是"AI 幻觉不会直接造成数据损失"。对比"一步到位"的风险：用户让 AI 改 package.json 的依赖版本，AI 幻觉删了 react 依赖，直接覆盖后项目跑不起来，用户还不知道改了什么——这个损失远大于多点一次"确认发布"。所以三步不是冗余，是把 AI 的不可控性限制在草稿层的安全机制。

### 第二层：证据与定位

**Q：你怎么定位"用户发布后才发现内容不对"是草稿审查没做好，还是版本回退机制不够用？**

用发布后悔率 + 回退使用率双指标定位。发布后悔率 = 发布后 5 分钟内用户点击"回退到上一版"的次数 / 总发布次数。如果后悔率高（>15%），说明草稿审查环节失效——用户在编辑态没仔细看就点了发布，解法是强化编辑态的 Diff 展示（高亮风险变更、强制停留 N 秒才能点发布）；如果后悔率低但用户仍抱怨"内容不对"，说明是"审查了但没看出问题"——Diff 视图不够直观（如 500 行 JSON 的变更用户看不懂），解法是提供变更摘要（"本次修改了 3 个依赖版本、新增 2 个配置项"）。回退使用率 = 用户实际执行版本回退的次数 / 发布次数，如果这个率高（>10%），说明发布前的审查质量不够，应该前移拦截（在编辑态做更严格的结构校验）；如果回退使用率极低（<1%）且后悔率也低，说明三步设计有效——问题在发布前就被拦住了。

### 第三层：根因深挖

**Q：为什么编辑态必须支持"局部接受"（逐条勾选变更），而不是只给"全部接受/全部拒绝"两个选项？**

根因是"AI 生成的变更质量是不均匀的"——一个草稿里可能 80% 的变更是对的（标题更新、数据修正），20% 是错的（幻觉新增了一段不存在的内容、格式错误）。如果只给"全部接受/全部拒绝"，用户面临两难：接受则连带接受 20% 的错误，拒绝则丢失 80% 的有效修改——无论哪个选择都是损失。局部接受的价值是让用户对每条变更独立决策：正确的勾选接受，错误的取消，最终合并出一个"只包含用户认可变更"的版本。这本质上是把"接受/拒绝"的粒度从"整份文档"细化到"单条变更"，与 Git 的 hunk-level 暂存是同一个理念。实现上通过 Diff 算法把草稿和原文的差异拆分为独立的 change（add/modify/delete），每个 change 可独立勾选，最终用 applyPartialAccept 把勾选的变更应用到原文上。不支持局部接受的编辑态，实质是逼用户做"全有或全无"的二元决策，与 AI 输出的不确定性不匹配。

**Q：那如果变更太细碎（如 100 处微小的格式调整），局部接受的勾选本身就很累，为什么不退化成全盘接受？**

因为"勾选累"是 UI 设计问题，不是"局部接受"策略的问题。100 处微小变更的解法是"批量操作 + 智能分组"：(1) 按变更类型分组（格式调整 50 处 / 内容修改 30 处 / 新增内容 20 处），用户可以按组批量勾选（"接受所有格式调整"）；(2) 按风险等级分组（低风险：格式化、标点；中风险：数据更新、措辞修改；高风险：删除内容、新增段落），默认自动接受低风险、提示用户审核高风险；(3) 按变更位置分组（标题区/正文区/结论区），用户可以只审查关心的区域。这样 100 处变更的审查从"逐条勾选 100 次"降到"分组决策 3-5 次"，保留了局部接受的精细度又控制了操作成本。退化为全盘接受的代价是丧失对 AI 错误的控制力——100 处里有 5 处幻觉也会被全盘写入，后续排查成本远高于分组审查的成本。

### 第四层：方案权衡

**Q：三步状态机（Draft→Review→Publish）你用同步流转（用户手动点下一步）还是自动流转（草稿质量分高时自动进入发布），怎么选？**

选同步手动流转，不做自动发布。根因是"自动发布的收益（省一次点击）远小于其风险（错误内容被写入文件）"。自动发布的逻辑通常是"AI 自评打分 >80 分就跳过用户审查直接写入"，但 AI 自评的假阳性率高（AI 觉得自己写得很好但实际有幻觉），一旦误判就是不可逆的数据覆盖。而且自动发布破坏了三步设计的核心价值——"用户对 AI 输出的控制感"：如果用户知道"高分草稿会自动发布"，他们对高分草稿就会焦虑（"万一 AI 打错分了呢"），反而要在发布后手动检查，总成本更高。同步手动流转的成本只是"用户看一眼 Diff 点确认"，约 3-5 秒，但保证了"每一次写入都是用户知情的"。唯一可以考虑的自动化是"草稿质量分极低（<30 分）时自动进入重生成而非发布"——这是拦截低质量而非放过高质量，方向正确。

**Q：那如果高频用户抱怨"每次都要点确认太烦，我信任 AI 想跳过审查"，为什么不给他们开"自动发布"的开关？**

因为"高频用户想跳过审查"的根因是"审查体验差"而非"真的不需要审查"。正确做法是优化审查效率而非提供跳过开关：(1) 快捷键支持——高频用户可以用键盘快捷键秒过审查（Cmd+Enter 直接发布），比鼠标点确认快 3 倍；(2) 信任累积机制——对同一类任务（如"格式化 JSON 文件"）用户连续接受了 N 次且无回退，该类任务自动进入"低风险快速通道"，草稿只展示变更摘要不展开完整 Diff，用户一键确认；(3) 批量发布——用户可以勾选多个草稿一次性批量发布，而非逐个审查。开"自动发布"开关的致命问题是：一旦开启用户会忘记关，之后所有任务（包括高风险的覆盖操作）都自动发布，某天 AI 幻觉覆盖了重要文件，用户根本不知道跳过了审查——这是"便利性导致安全性崩塌"的典型陷阱。所以解法是"让审查更快"而非"跳过审查"。

### 第五层：验证与沉淀

**Q：你怎么证明"三步确认"比"一步到位"更能降低数据事故，有数据支撑吗？**

用三个指标验证：(1) 数据事故率 = 用户报告"文件被错误覆盖/内容不对"的工单数 / 总发布次数，三步确认上线后应从"每周 N 起"降到"几乎为零"，因为错误在草稿/编辑态就被拦住；(2) 草稿丢弃率 = 用户在草稿态点击"丢弃"的次数 / 总草稿数，这个率如果 >20%，说明三步设计成功拦截了大量低质量 AI 输出（这些在一步到位方案中会直接写入文件）；(3) 局部接受使用率 = 用户使用局部接受（而非全盘接受/拒绝）的发布次数 / 总发布次数，如果 >40%，说明局部接受是刚需（大量场景下用户只需接受部分变更）。A/B 测试时对照组用"一步到位"（AI 直接写文件 + 版本历史），实验组用"三步确认"，跑两周后对比数据事故率——如果实验组的事故率低 80% 以上，就证明三步设计有效。

**Q：怎么让团队在新接入产物类型（如思维导图、流程图）时，自觉实现三步状态机，而不是偷懒让 AI 直接写文件？**

把三步状态机做成"产物写入层 SDK"的强制约束。第一，文件系统写入层（FileWriter）统一封装，所有写入必须经过它，直接调 fs.writeFile 的代码在 ESLint 规则里禁用；第二，FileWriter 要求调用方传入 DraftArtifact 对象（必须经过 draft→review→publish 三态），没有 review 记录的写入直接抛异常；第三，每种产物类型在注册时必须实现 DraftRenderer（草稿态预览）、DiffRenderer（编辑态对比）、PublishHandler（发布态写入）三个组件，缺一个不允许注册；第四，CI 流水线扫描所有文件写入操作，如果有绕过 FileWriter 直接写文件的代码，CI 失败。这样三步状态机就从"产品规范"变成了"SDK 层的架构约束"，不遵循就无法写入文件系统。

## 结构化回答

**30 秒电梯演讲：** AI结果越接近真实产物，越不能一步到位直接落库。前端拆成三步——草稿态（AI结果先入草稿，避免误写正式文件）→编辑确认态（支持diff、局部接受和重生成）→发布态（确认后才触发导出/覆盖/发布），整条链路都能回看。

**展开框架：**
1. **AI** — AI结果先入草稿态，避免误写正式产物
2. **编辑态支持** — 编辑态支持diff、局部接受和重生成
3. **确认后再触** — 确认后再触发导出、发布或覆盖动作
