# Vibe Coding工具为什么最近会被拿来问，面试里应该怎么答？

## 什么是Vibe Coding

```
传统编程                          Vibe Coding
┌──────────────┐                 ┌──────────────┐
│ 程序员        │                 │ 程序员        │
│              │                 │              │
│ 思考逻辑      │                 │ 描述需求      │
│ 手写代码      │                 │ ↓ (自然语言)  │
│ 调试Bug       │                 │ AI工具生成    │
│ 重构优化      │                 │ ↓             │
│              │                 │ 审查+修改     │
│ 80%写代码    │                 │ ↓             │
│ 20%设计      │                 │ 20%描述需求   │
│              │                 │ 50%设计/架构  │
│              │                 │ 30%审查代码   │
└──────────────┘                 └──────────────┘
```

## 面试回答框架

### 回答策略：展示三个层次

```markdown
## 层次1：实操经验（展示"我真的在用"）

"我在日常开发中大量使用 Claude Code / Cursor 做Vibe Coding。
具体场景包括：
- 快速原型验证：用自然语言描述需求，5分钟生成MVP
- 重复代码生成：CRUD接口、单元测试模板
- 代码理解：让AI解释复杂开源项目的代码逻辑
- Bug修复：描述错误现象让AI定位和修复

我开发的Agent项目中，约70%的模板代码由AI生成，
但核心架构设计和关键逻辑仍由我亲自编写。"

## 层次2：理解范式变革（展示"我理解趋势"）

"Vibe Coding本质上是编程范式的转变：
- 从'语法驱动'到'语义驱动'
- 从'逐行编写'到'意图描述+结果审查'
- 程序员核心能力从'编码速度'变为'系统设计+需求表达+质量把关'

但我也清楚它的局限：
- 复杂架构设计AI做不了，需要人类架构决策
- 生成的代码需要严格Code Review，AI可能引入安全漏洞
- 性能关键路径仍需手写优化
- 对业务逻辑的理解AI不如领域专家"

## 层次3：思考深度（展示"我有独立见解"）

"我认为Vibe Coding对Agent开发岗位的影响特别大：
- Agent开发本身就是'用自然语言编程Agent'
- Harness工程师需要理解AI编程工具的内部机制
- 这正好是'用AI构建AI'的飞轮

我的实践体会：
- Vibe Coding最大的价值不是'写得快'，而是'试得快'
- 快速迭代→快速验证→快速失败→快速学习
- 但最终生产代码必须经过严格测试和审查"
```

   → 用GitHub Copilot辅助写ETL脚本
   → AI擅长：正则表达式、数据处理模板
   → 我负责：异常处理、边界case、性能调优
"""

# 如果被追问"Vibe Coding有什么风险？"
RISKS = """
1. 安全风险：AI可能生成有漏洞的代码（SQL注入、XSS等）
   → 对策：所有AI生成代码必须过安全扫描

2. 技术债：快速生成的代码可能不够优雅
   → 对策：定期重构，技术债务追踪

3. 过度依赖：长期不用手写可能导致基本功退化
   → 对策：核心算法/数据结构仍坚持手写

4. 理解偏差：AI可能误解需求生成错误代码
   → 对策：严格的Code Review + 测试驱动
"""

## 面试加分点

1. **不要只说"很好用"**：要展示具体的使用场景、量化效果（如"开发效率提升3x"）
2. **展示批判思维**：能说出Vibe Coding的局限和风险，体现你不是盲目跟风
3. **关联岗位**：将Vibe Coding与Agent开发岗位关联——"用AI构建AI"是核心能力
4. **实操Demo**：如果能现场展示用Claude Code快速实现一个小功能，加分极大
