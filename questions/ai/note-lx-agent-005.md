# 主Agent的意图识别应该怎么做，为什么单轮分类器经常不够用？

## 单轮分类器的三大盲区

```
盲区1: 省略句（上下文依赖）
  轮1: "我想买一件卫衣"          → intent: search_product
  轮2: "有红色的吗"              → intent: ???
  单轮分类器: 看到"有红色的吗" → 可能分类为"询问颜色"
  正确意图: search_product(color=红色)  ← 依赖轮1的上下文

盲区2: 指代消解
  轮1: "推荐一款iPhone"          → intent: recommend
  轮2: "它支持5G吗"              → intent: ???
  单轮分类器: "它"是谁？无法确定
  正确意图: query_spec(product=iPhone, feature=5G)

盲区3: 隐式意图
  用户: "这个月话费好贵啊"        → intent: ???
  单轮分类器: 分类为"闲聊/抱怨"
  正确意图: recommend(cheap_plan)  ← 用户隐含希望推荐便宜套餐
```

## 多轮意图识别方案

### 方案一：上下文窗口拼接

```python
def intent_with_context(dialog_history: list, current_input: str):
    """将最近N轮对话拼入Prompt，让LLM做意图识别"""
    recent = dialog_history[-5:]  # 最近5轮

    prompt = f"""根据对话历史，识别用户当前意图。

对话历史：
{format_dialog(recent)}

当前输入：{current_input}

请输出：
1. 意图类别（从以下选择：search/recommend/query/order/refund/chat/other）
2. 关键实体和槽位
3. 是否为隐式意图（用户没明说但有明确需求）
"""
    return llm_call(prompt)
```

### 方案二：对话状态追踪（DST）

```python
class DialogStateTracker:
    """维护对话状态，支持指代消解和意图链"""

    def __init__(self):
        self.state = {
            'current_intent': None,
            'mentioned_entities': [],   # 对话中提到的实体
            'slots': {},                 # 已填充的槽位
            'intent_chain': [],          # 意图演变链
        }

    def update(self, user_input: str, llm_response: dict):
        # 1. 意图识别（结合当前状态）
        intent = self._classify_intent(user_input, self.state)

        # 2. 指代消解
        resolved_input = self._resolve_references(user_input, self.state)

        # 3. 槽位更新
        self.state['slots'].update(llm_response.get('slots', {}))

        # 4. 实体追踪
        entities = extract_entities(resolved_input)
        self.state['mentioned_entities'].extend(entities)

        # 5. 意图链
        self.state['intent_chain'].append(intent)
        self.state['current_intent'] = intent

        return resolved_input, intent
```

### 方案三：意图链推理

```python
# 用户可能在同一个目标下连续追问
INTENT_CHAINS = {
    'shopping': ['search', 'query_detail', 'compare', 'decide', 'order'],
    'support': ['describe_problem', 'provide_info', 'try_solution', 'feedback'],
}

def predict_next_intent(intent_chain: list) -> str:
    """根据意图链预测下一步可能意图，提前准备"""
    for chain_name, sequence in INTENT_CHAINS.items():
        # 模糊匹配当前意图链
        matched = match_sequence(intent_chain, sequence)
        if matched and len(matched) < len(sequence):
            return sequence[len(matched)]  # 预测下一个意图
    return 'unknown'
```

## 多意图处理

```python
def multi_intent_detection(user_input: str) -> list:
    """一句话包含多个意图时拆分处理"""
    intents = llm_parse(f"""
    用户输入：{user_input}
    
    请识别所有独立意图，输出JSON列表：
    示例输入："帮我查下快递，顺便推荐个手机壳"
    示例输出：[{{"intent": "query_logistics"}}, {{"intent": "recommend", "keyword": "手机壳"}}]
    """)
    return intents
```

## 面试加分点

1. **分级策略**：先轻量分类器做初筛（7B），低置信度再走LLM精判（32B），平衡延迟和准确率
2. **意图数据飞轮**：收集人工标注的意图数据持续微调分类器，形成闭环
3. **冷启动方案**：新意图类型出现时，先用LLM few-shot处理，积累数据后再训分类器
4. **意图冲突**：用户同时表达矛盾意图（"想买又觉得贵"），需要识别情绪+给出折中建议

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：单轮分类器搞不定省略句和指代，但可以用 LLM few-shot 直接处理（把历史拼进 prompt）。为什么还要 DST（对话状态追踪）？LLM + 历史窗口不够吗？**

LLM + 历史窗口能处理部分上下文，但 DST 在工程上有不可替代的优势。一是长程依赖：LLM 的上下文窗口有限（即使 128k，几十轮对话后历史过长，注意力稀释，早期信息被"遗忘"）。DST 显式存储关键状态（当前意图、槽位、实体），不受窗口限制——第 50 轮的"那红色的呢"仍能通过 DST 知道"红色指代第 3 轮的卫衣"，而 LLM 可能因历史过长忘记第 3 轮。二是结构化与可控：DST 的状态是结构化的（slots={商品:卫衣, 颜色:待定}），下游 Agent 可直接用（查商品库），LLM 的自然语言输出需解析（易错）。三是效率：DST 只存关键信息（几行 JSON），LLM 重处理全部历史（冗余计算，延迟高、成本高）。四是可调试：DST 的状态可检查（某轮为什么这么识别），LLM 黑盒难调试。五是确定性：DST 的规则部分（槽位更新、实体追踪）是确定性的，LLM 的输出有随机性（同一对话不同次识别可能不同），关键业务（如下单）需确定性。LLM + 历史适合简单/短对话（闲聊、单轮），DST 适合长/复杂对话（多轮任务、电商客服）。两者常结合——DST 维护状态，LLM 在 DST 辅助下做意图识别（LLM 读 DST 状态 + 当前输入，而非全部历史），兼顾准确和效率。

### 第二层：证据与定位

**Q：你说 DST 维护状态提升多轮意图识别。但怎么量化证明 DST 比"LLM + 历史窗口"好？有哪些指标能对比两者的效果差异？**

多维对比。一是意图识别准确率：构建多轮测试集（含省略句、指代、隐式意图、多意图），对比 DST + LLM vs 纯 LLM + 历史窗口的准确率。典型差异：短对话（< 5 轮）两者接近（LLM 能记住历史）；长对话（> 20 轮）DST 优势显著（LLM 遗忘早期信息，准确率下降 10-20%，DST 稳定）。二是指代消解准确率：测试指代场景（"它""那个""红色的"），DST 显式追踪实体，准确率高；LLM 依赖注意力，长距离指代易错。三是槽位填充准确率：DST 的槽位更新是结构化的（准确率高），LLM 的槽位从自然语言解析（易漏/错）。四是延迟和成本：DST + LLM 只读状态 + 当前输入（短 prompt，快、省 token）；LLM + 全历史（长 prompt，慢、贵）。实测 20 轮对话，DST 方案延迟 200ms（短 prompt），LLM + 全历史延迟 1s+（长 prompt）。五是可调试性：DST 状态可导出检查（某轮错误可定位是状态更新问题还是识别问题），LLM 黑盒难调试。证明逻辑是"多轮测试集对比准确率 + 指代/槽位专项测试 + 延迟成本对比 + 可调试性"，DST 在长对话和结构化任务上全面占优，短对话两者接近。

### 第三层：根因深挖

**Q：意图链推理预测下一步意图（如购物链：搜索→查询→比较→下单）。但用户行为不一定遵循固定链（可能跳跃、回退、切换目标）。意图链会不会过于刚性，反而误预测？**

固定意图链确实过于刚性，真实用户行为多变。意图链的问题：一是用户不线性——用户可能在"搜索"后直接"下单"（跳过比较），或在"比较"后回退到"搜索"（换关键词），固定链预测下一步会错。二是目标切换——用户购物中途问"天气"（切换到天气意图），固定链仍预测"比较"或"下单"，误触发。三是多目标并行——用户同时购物和查物流，意图在两条链间切换，单一链无法建模。改进方法：一是用概率图模型（如 HMM、CRF）而非固定链——模型学习意图间的转移概率（P(下一步意图 | 历史意图序列)），允许跳跃和回退（概率分布而非确定路径），对未知路径给合理概率。二是结合用户状态——意图链 + DST 的用户状态（当前目标、已完成步骤），动态预测（如"已比较 3 次"预测"即将下单"，"刚开始搜索"预测"继续搜索"）。三是不强预测，只做准备——意图链用于"预加载资源"（如预测"可能下单"则预加载订单页），即使预测错也无害（只是多加载），而非"自动执行下一步"（误执行有害）。四是允许"unknown"——模型不确定时输出"不预测"，由用户行为驱动，避免误预测。所以意图链的价值是"概率性预测 + 软准备"，不是"刚性路径驱动"，设计时保留灵活性。

**Q：那如果用户意图不可预测（意图链不准），为什么要做意图预测？直接响应当前意图不就行？预测错了会不会浪费资源（预加载了不需要的）？**

意图预测的价值是"降低下一步的延迟"，即使偶有预测错也净收益。原因：一是预加载的成本低、收益高——如预测"可能下单"，预加载订单页（后端数据准备），若用户真下单则秒开（省 200ms 加载）；若用户没下单，预加载的资源被丢弃（成本低，只是后端查询，非用户可见）。预加载的"浪费"是后端资源（可承受），收益是用户体验（秒开 vs 等待）。二是预测准确率够用——基于意图链或概率模型，意图预测准确率通常 40-60%（不是随机猜，有信号），意味着 40-60% 的预加载有效，净收益为正。三是非预测的替代是"等用户明确后再加载"，延迟无法消除——用户说"下单"后才开始加载订单页，延迟 200ms+，体验差。四是预测可与流式 ASR 协同——ASR 中间结果送意图预测，用户还没说完就预判意图（如听到"帮...下...单"预测"下单"），提前准备，总延迟进一步降。所以意图预测是"低成本的延迟优化"，即使准确率不高也值得做，关键是"预测用于软准备（预加载）而非硬执行（自动触发）"，控制误预测的代价。误预测的浪费（后端资源）远小于延迟优化的收益（用户体验），这是工程上的非对称收益。

### 第四层：方案权衡

**Q：分级策略（轻量分类器初筛 7B，低置信度走 LLM 精判 32B）。为什么不直接全用 32B？分级增加了系统复杂度（两个模型、路由逻辑），有什么实际收益？**

分级的收益是"延迟和成本的平衡"，32B 全用在大流量下不可行。一是延迟：7B 分类器推理 50-100ms，32B 推理 200-500ms。高并发下全用 32B，GPU 资源紧张，排队延迟更高（可能秒级）。分级让 80% 高置信度请求走 7B（快），20% 低置信度走 32B（准），整体延迟和准确率都优。二是成本：32B 的 GPU 成本是 7B 的 4-5 倍（参数多、计算多）。全用 32B，推理成本高；分级让多数请求用便宜的 7B，成本降 60-80%。三是准确率：7B 对清晰意图（"下单""退款"）准确率已高（> 95%），只有模糊意图（省略句、隐式）需 32B 精判。分级不牺牲准确率（模糊的走 32B），只优化清晰的路由。四是复杂度的代价：分级需路由逻辑（置信度阈值）、两套模型维护、监控（7B 和 32B 各自指标）。但这个复杂度是"一次性的工程投入"，换取长期的延迟和成本收益，值得。实践配置：7B 做初筛，置信度 > 0.9 直接用，< 0.9 路由到 32B；阈值通过验证集调（平衡准确率和 32B 调用比例）。分级是"用工程复杂度换性能和成本"的经典权衡，大流量场景标配。

**Q：隐式意图（"这个月话费好贵"隐含"推荐便宜套餐"）的识别很难——用户没明说，怎么判断他真有"推荐"意图，而不只是闲聊/抱怨？误判隐式意图会不会让 Agent 过度推销，用户反感？**

隐式意图识别是高风险高收益，需谨慎。误判的后果：用户只是抱怨（"话费好贵"想吐槽），Agent 却推销套餐，用户觉得"被推销"，反感。识别方法：一是上下文信号——用户历史行为（曾查询过套餐、近期有换套餐意向）支持"推荐"意图；无历史信号则可能是闲聊。二是情绪与语气——"好贵啊"带抱怨情绪，"有没有便宜的"带询问意图，语气不同意图不同。用情感分析辅助判断。三是试探性回应而非直接推销——Agent 识别可能的隐式意图后，不直接推产品，而是试探（"您提到话费贵，是想了解更实惠的套餐吗？"），让用户确认，避免误判。这是"主动但克制"的策略——识别隐式意图后用"确认问句"而非"直接行动"。四是频率控制——即使识别了隐式意图，不每次都推销（防骚扰），用频率限制（如每 N 轮最多一次主动推荐）。五是用户反馈闭环——用户拒绝（"不用，就说说"）则记录，后续不推；用户接受则强化，数据飞轮提升识别准确率。隐式意图的正确做法是"谨慎识别 + 试探确认 + 频率控制 + 反馈闭环"，而非"识别就推销"。收益是主动服务（提升满意度），风险是骚扰，关键在克制和确认。

### 第五层：验证与沉淀

**Q：多轮意图识别上线后，怎么持续评估效果？意图识别的错误（误判意图）怎么发现和纠正？**

多层监控和反馈闭环。一是线上准确率监控：抽样线上对话，人工标注意图（对比 ASR 识别的意图），算准确率。准确率下降（如从 95% 降到 90%）触发告警，排查原因（新意图类型、分布漂移、模型退化）。二是用户行为信号：重说率（用户因意图识别错而重说，如 Agent 误解后用户纠正"我是说..."）、任务失败率（意图错导致任务没完成）、转人工率（用户因 Agent 不理解而转人工）。这些信号间接反映意图识别质量，异常时排查。三是错误案例库：收集识别错的案例（人工标注或用户反馈），分类错误类型（省略句未识别、指代消解错、隐式意图误判），针对性优化（补充训练数据、调整 DST 规则、改进 prompt）。四是 A/B 测试：优化（如微调模型、改 prompt、加 DST 规则）后线上 A/B，对比准确率和用户体验指标，验证效果。五是数据飞轮：错误案例标注后加入训练集，定期微调模型，持续提升。证明逻辑是"线上准确率监控 → 用户行为信号 → 错误案例库 → A/B 验证优化 → 数据飞轮"，形成"监控-发现-优化-验证"的闭环。意图识别不是一次上线就完，是持续监控和优化的动态系统。

**Q：怎么让团队在设计多轮意图识别时不各自为政（有人用 LLM、有人用 DST、有人用分类器），而是统一的系统架构？**

沉淀意图识别架构规范和选型决策树。一是架构规范：统一的意图识别架构（DST 维护状态 + LLM 识别 + 分级路由），各业务线遵循，避免碎片化。二是选型决策树：按场景选——简单单轮意图用分类器（快）；多轮复杂意图用 DST + LLM（准）；高并发用分级（7B 初筛 + 32B 精判，省成本）。三是 DST 规范：统一的 DST 数据结构（意图、槽位、实体、意图链），跨业务复用；DST 更新规则模板（槽位填充、实体追踪、指代消解）。四是意图分类体系：统一的意图标签集（search/recommend/query/order/refund/chat 等），各业务线映射到该体系，避免各自定义意图。五是评估规范：统一的测试集格式和评估指标（准确率、指代消解率、槽位填充率），跨业务可比。六是踩坑库：常见错误（单轮分类器处理多轮、DST 状态设计不全、分级阈值不当）及案例。让意图识别是"统一架构 + 决策树选型 + 规范数据结构 + 评估体系"的系统工程，而非各团队重复造轮子。
