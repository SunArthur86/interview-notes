# 【拼多多 AI大模型开发】大模型 vs 小模型 vs 规则引擎的选型场景

> 来源：拼多多AI大模型开发面经（小红书）— 原题：大模型 vs 小模型 vs 规则引擎 选型场景

## 二、第一性原理分析

**为什么不全用大模型？**

```
维度对比:
┌────────────┬────────────┬────────────┬────────────┐
│ 维度       │ 规则引擎    │ 小模型      │ 大模型      │
├────────────┼────────────┼────────────┼────────────┤
│ 单次成本    │ ~0元       │ ~0.001元    │ ~0.1元      │
│ 延迟       │ <1ms       │ 10-50ms     │ 500-3000ms  │
│ 准确率     │ 95%+(规则内)│ 85-92%      │ 90-95%      │
│ 幻觉率     │ 0%         │ <1%         │ 3-8%        │
│ 可控性     │ 完全可控    │ 较可控       │ 难以完全控制 │
│ 覆盖率     │ 仅规则覆盖  │ 分类+抽取    │ 通用生成     │
│ 硬件       │ CPU        │ CPU         │ GPU         │
│ 适合场景   │ FAQ/政策    │ 意图/分类    │ 复杂推理    │
└────────────┴────────────┴────────────┴────────────┘

成本差距: 规则→大模型 = 100-1000倍
→ 90%简单请求用规则 = 节省90%+成本
```

## 三、详细答案

### 3.1 三级处理架构

```
┌──────────────────────────────────────────────────────────────┐
│                    拼多多客服AI架构                             │
│                                                              │
│  用户消息 → 意图层(BERT轻量分类) → 分流                        │
│                    │                                         │
│         ┌─────────┼─────────┬──────────────┐                │
│         ↓         ↓         ↓              ↓                │
│     物流查询   退款流程   投诉纠纷      其他复杂                │
│     (规则匹配)  (规则匹配)  (大模型生成)   (大模型兜底)         │
│     90%请求    CPU秒解     8%请求       2%请求                │
│     CPU秒解    确定性输出   GPU推理      GPU推理               │
│                                                              │
│  风控层(所有大模型输出必过):                                   │
│    赔付规则强校验 → 资损拦截 → 敏感话术过滤 → 输出              │
│                                                              │
│  兜底层:                                                      │
│    静态话术模板 → 人工转接 → 超时降级                          │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 各层技术选型

**规则引擎层（90%流量）：**
```python
# 基于关键词+正则+FAQ知识库匹配
class RuleEngine:
    def match(self, query):
        # 1. 精确FAQ匹配（查退快递、查物流等）
        result = self.faq_db.search(query)
        if result.confidence > 0.95:
            return result.answer  # 直接返回，0延迟
        
        # 2. 政策文档检索（退款政策、售后流程等）
        policy = self.policy_db.search(query)
        if policy:
            return policy.template_fill(user_info)  # 模板填充
        
        return None  # 未命中，转下一层
```

**小模型层（意图分类+实体抽取）：**
```python
# BERT INT8量化 + CPU推理
class IntentClassifier:
    def __init__(self):
        self.model = load_bert_int8("bert-int8-quantized.bin")
        self.model.to(cpu_device)
    
    def classify(self, query):
        intent = self.model.predict(query)  # ~20ms CPU推理
        # 物流/退款/赔付/投诉/其他
        return intent
```

**大模型层（复杂生成）：**
```python
# LLM + RAG + 风控
class LLMHandler:
    def handle(self, query, context):
        # 1. RAG检索相关文档
        docs = self.rag.retrieve(query, top_k=3)
        
        # 2. 大模型生成
        response = self.llm.generate(
            prompt=build_prompt(query, context, docs),
            temperature=0.1  # 低温度减少幻觉
        )
        
        # 3. 风控拦截
        if self.risk_checker.has_promise(response):
            response = self.fallback_template  # 赔付承诺拦截
        
        return response
```

### 3.3 客服幻觉的致命后果

```
大模型幻觉 → 资损场景:

  用户: "我的包裹丢了，能赔多少？"
  大模型(幻觉): "根据平台政策，包裹丢失全额赔付500元。"
  
  实际政策: 按保价金额赔付，该用户未保价只赔3倍运费。
  后果: 多赔497元 × 日均1000单类似咨询 = 日损49.7万!
  
解决方案:
  1. 赔付规则强拦截: 大模型输出中的金额→校验规则引擎
  2. 禁止模型私自承诺: Prompt约束 + 后处理过滤
  3. 高风险场景转人工: 赔付/投诉类必转人工确认
```

## 四、选型决策矩阵

| 场景 | 推荐方案 | 原因 | 成本 |
|------|---------|------|------|
| 查物流 | **规则引擎** | 确定性、零成本 | ~0元 |
| 退款流程 | **规则引擎+模板** | 流程固定 | ~0元 |
| 商品咨询 | **小模型BERT分类** | 分类+FAQ匹配 | ~0.001元 |
| 售后纠纷 | **大模型+RAG+风控** | 需要理解和生成 | ~0.1元 |
| 投诉处理 | **大模型→人工兜底** | 高风险需人工 | ~0.1元 |

## 五、扩展知识

- **INT8量化**: 将FP32权重→INT8，模型体积缩小4倍，推理速度提升2-3倍，精度损失<1%
- **模型蒸馏**: 用大模型(Teacher)训练小模型(Student)，保留90%能力但体积减小10倍
- **batch合并**: 将多个请求合并为一个batch推理，提高GPU利用率

## 六、苏格拉底式面试提问

1. **"你说90%用规则引擎，但规则怎么维护？政策变了怎么办？"** — 引出规则热更新、版本管理、A/B测试
2. **"如果规则和小模型都判断不了，直接转大模型，但大模型也答错了怎么办？"** — 引出兜底链路、人工转接、BadCase回流
3. **"大模型成本降不下来，有没有办法让大模型也跑在CPU上？"** — 引出量化(INT4/INT8)、蒸馏、TinyLlama等轻量化方案
4. **"怎么评估三级架构的效果？用什么指标？"** — 引出分流准确率、各层解决率、整体满意度、成本/QPS
5. **"大促洪峰时，GPU资源不够用怎么办？"** — 引出降级策略：大模型层关闭，全部走规则+小模型兜底

## 七、面试加分点

1. **量化成本对比** — 规则0元 vs 小模型0.001元 vs 大模型0.1元，100-1000倍差距
2. **强调风控层** — 拼多多最看重的是资损控制，不是模型效果
3. **提到BadCase闭环** — 错误回答自动回流迭代，展示系统思维
4. **结合拼多多价值观** — 稳定优先、成本优先、风控优先、智能其次
5. **知道CPU轻量化部署** — INT8量化、蒸馏、batch合并，展示工程能力

## 结构化回答

**30 秒电梯演讲：** 不是所有问题都需要大模型。简单咨询用规则/小模型CPU推理，复杂纠纷用大模型兜底，极致降本是拼多多的核心选型逻辑——就像医院分诊。

**展开框架：**
1. **规则引擎** — 90%简单问题CPU秒解，零幻觉、零成本、确定性强
2. **小模型** — BERT/T5级别，CPU可推理，适合意图识别和分类
3. **大模型** — LLM兜底复杂问题，但成本高、延迟高、有幻觉风险

**收尾：** 您想深入聊：规则引擎如何维护？政策变更怎么办？
