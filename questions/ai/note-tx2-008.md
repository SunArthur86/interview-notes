# 【某讯面经】设计一个生产可用的 Agent 系统，从0到1怎么搭？

## 一、五层架构

```
┌─────────────────────────────────────────────────────────────┐
│  [可观测层] trace / metrics / log / 告警                     │
├─────────────────────────────────────────────────────────────┤
│  [接入层] API网关 / 鉴权 / 限流 / 协议转换                    │
├─────────────────────────────────────────────────────────────┤
│  [编排层] LangGraph 状态机 / Planning / 多Agent协作           │
├─────────────────────────────────────────────────────────────┤
│  [能力层] LLM(混元/豆包) / Tool(工具集) / Memory(记忆)        │
├─────────────────────────────────────────────────────────────┤
│  [数据层] Redis(短期状态) / Postgres(日志) / 向量库(知识)     │
└─────────────────────────────────────────────────────────────┘
```

## 二、各层职责详解

### 1. 接入层
```
- API 网关：统一入口，协议转换（HTTP/WebSocket/SSE）
- 鉴权：API Key / OAuth / JWT
- 限流：令牌桶（防滥用、控成本）
- 请求校验：参数合法性、prompt 注入检测
```

### 2. 编排层（核心）
```
LangGraph 状态机：
  State: messages / current_plan / tool_history / step_count
  Node: intent_recognition / planning / tool_execution / aggregation / reflection
  Edge: conditional（根据状态路由）

关键设计：
  - Planning 用 Plan&Execute（先规划后执行）
  - 工具执行是确定性 Workflow
  - 反思阶段决定是否再规划
  - 硬步数上限（10-15 步）防死循环
```

### 3. 能力层
```
LLM 池：
  - 大模型（混元 pro）：总结/复杂推理
  - 小模型（混元 lite）：意图识别/简单决策（省钱）
  - 兜底模型：主模型挂了用

Tool 注册中心：
  - 工具元数据（schema/权限/超时）
  - 服务端分发（不暴露给 LLM 直接调）

Memory：
  - 短期：当前会话（Redis）
  - 长期：用户画像/历史结论（Postgres）
  - 向量：知识库（Milvus）
```

### 4. 数据层
```
Redis：会话短期状态（TTL 30min）、分布式锁、限流计数
Postgres：agent_steps 日志（可回放/审计）、用户数据
向量库（Milvus）：知识库 embedding、历史对话向量化
对象存储（COS）：文档/图片/产物
```

### 5. 可观测层（生产必备）
```
Trace：每个请求的完整链路（OpenTelemetry）
  request_id → intent → plan → tool_calls → answer
  每步记录：input/output/latency/cost

Metrics：
  - 业务：任务完成率、用户满意度
  - 技术：P99 延迟、错误率、token 消耗、工具调用成功率

Log：结构化日志（JSON），可检索

告警：
  - 错误率 > 阈值
  - P99 延迟 > 阈值
  - token 消耗异常
  - 工具失败率 > 阈值
```

## 三、生产关键关注点

### 高可用（熔断降级）
```
LLM 调用熔断：
  错误率 > 50% → 熔断 30s → 降级到兜底模型
  兜底模型也挂 → 降级到规则引擎（确定性回复）

工具调用降级：
  工具超时 → 缓存结果 / 默认值 / 跳过
  关键工具挂 → 转人工

整体降级：
  Agent 链路挂 → 退化成简单 RAG（检索+生成）
```

### 成本控制
```
Token 预算：
  每请求预算（如 10K token）
  超预算 → 强制结束 + 返回当前结果

小模型决策 + 大模型总结：
  意图识别/工具选择 → 小模型（便宜）
  最终答案生成 → 大模型（贵但质量好）

缓存：
  相同 query → 缓存结果（embedding 相似度匹配）
  相同 tool_call → 缓存结果
```

### 安全
```
工具权限分级：
  read-only → 直接放行
  write → 日志审计
  destructive → 二次确认

PII 脱敏：
  用户输入先脱敏（身份证/手机号）再喂 LLM

Prompt 注入防御：
  输入检测（"忽略以上指令"等模式）
  输出过滤（敏感词/越权内容）

审计日志：
  所有工具调用落表（谁/何时/调了什么/结果）
```

### 可演进
```
模型版本管理：
  - 模型 ID 版本化（hunyuan-pro-v1.2）
  - 新模型灰度发布（10%流量先试）
  - 可回滚

A/B 测试：
  - 实验框架（分桶/指标/显著性检验）
  - 新 prompt / 新工具 / 新模型 都能 A/B

配置中心：
  - prompt 模板、工具列表、阈值 热更新
  - 不重启服务改配置
```

## 四、从 0 到 1 的落地路径

```
[阶段1] MVP（2周）
  - 单 Agent + 3-5 个核心工具
  - 简单 RAG（向量检索）
  - 基础 trace
  - 目标：跑通核心场景

[阶段2] 生产化（4周）
  - 加熔断/降级/重试
  - 加完整可观测（metrics/log/告警）
  - 加权限/审计
  - 目标：能 7x24 稳定跑

[阶段3] 优化（持续）
  - 成本优化（小模型/缓存）
  - 质量优化（Rerank/bad case 闭环）
  - 能力扩展（多 Agent/新工具）
  - 目标：提效降本
```

## 五、加分点

- 说出 **可观测层是生产 Agent 的命脉**：没 trace 的 Agent 出问题靠猜
- 说出 **小模型决策 + 大模型总结** 的成本优化策略
- 说出 **兜底链路**：Agent 挂了退化成 RAG，RAG 挂了退化成规则，永远有答案

## 六、雷区

- ❌ "demo 跑通了就能上线" → 缺熔断/可观测，上线必挂
- ❌ "全用大模型" → 成本爆炸
- ❌ "没有兜底" → 链路任何一环挂就整个不可用

## 七、扩展

- **Multi-Region 部署**：多机房容灾，Agent 服务无状态化（状态全在 Redis/DB）
- **容量规划**：按 QPS 估算 LLM 调用量、token 消耗、向量库规模
- **SLA 定义**：可用性（99.9%）、延迟（P99 < 5s）、质量（任务完成率 > 85%）

## 记忆要点

- 五层架构：接入网关、LangGraph编排、模型与工具能力、多模态存储数据、Trace可观测
- 编排核心：状态机控制流转，Plan&Execute分离规划与执行，硬步数上限防死循环
- 生产高可用：LLM与工具需配熔断降级，大模型挂了降级小模型或缓存，最终退化为规则
- 成本与安全：小模型做大模型意图识别省钱，写入工具需二次确认防注入


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：生产 Agent 系统为什么要分五层（接入/编排/能力/数据/可观测），而不是一个大单体？**

分层是为了"关注点分离"和"独立演进"。接入层（API 网关/鉴权/限流）关注"流量入口"，可以独立扩展应对高并发；编排层（LangGraph 状态机）关注"业务逻辑"，可以独立迭代不改接入；能力层（Tool/Memory/LLM）关注"原子能力"，可以独立升级（如换模型）；数据层关注"存储"，可以独立优化（如换向量库）；可观测层关注"监控"，贯穿所有层。单体让所有职责耦合，改一处影响全局。分层是"工程化的基础"，不是过度设计。

### 第二层：证据与定位

**Q：生产 Agent 出现"偶发超时"，怎么用五层架构定位瓶颈？**

逐层看 trace。1) 接入层——API 网关的请求队列是否积压（限流或鉴权慢）；2) 编排层——状态机的某步是否卡住（如等条件不满足）；3) 能力层——LLM 调用或工具调用是否慢（看每步的 wall time）；4) 数据层——向量库或 Redis 是否慢（看查询延迟）；5) 可观测层——trace 是否完整（如果 trace 断了，是某层没埋点）。用端到端 trace ID 串联五层，定位最慢的层。

### 第三层：根因深挖

**Q：Agent 的可观测层（trace/metrics/log）经常缺失关键信息，根因是埋点不够还是架构设计没考虑可观测？**

通常是架构设计没把可观测作为"一等公民"。很多团队先开发功能，后补埋点，导致关键路径（如 LLM 的 input/output、工具的参数和返回）没记录。根因是"把可观测当附属"而非"核心设计"。解法：1) 架构设计时定义"必须记录的事件"（每次 LLM 调用、每次工具调用、每次状态转移）；2) 用 OpenTelemetry 标准化埋点（自动注入 trace context）；3) CI 检查——关键路径没埋点的 PR 不让 merge。

**Q：那为什么不直接用现成的 APM（如 Datadog、SkyWalking），而要自己设计可观测层？**

现成 APM 适合"传统应用"（HTTP 请求、DB 查询），但 Agent 有特殊需求：1) 要记录 LLM 的 prompt 和 completion（token 级别），传统 APM 不支持；2) 要追踪多步 Agent 的状态转移（不是单次请求），传统 APM 的 trace 模型不匹配；3) 要统计 token 成本、tool_call_success_rate 等 Agent 专属指标。所以 Agent 可观测要"基于通用 APM + 自定义 Agent 指标"，不是纯自研。用 OpenTelemetry 采集 + 自定义指标扩展，比纯 Datadog 更贴合。

### 第四层：方案权衡

**Q：编排层用 LangGraph 还是自研状态机，怎么选？**

权衡"开发效率 vs 控制力"。LangGraph——提供状态机抽象（节点、边、条件转移），开发快，社区支持，但框架有"魔法"（调试难、性能瓶颈在框架内部）；自研状态机——完全可控，性能可优化，但要自己实现状态管理、异常处理、超时控制，开发慢。经验上：早期项目用 LangGraph（快速验证），规模化后评估是否自研（如果 LangGraph 的性能或灵活性成为瓶颈）。混合方案：用 LangGraph 的抽象 + 自研的核心调度。

**Q：为什么不直接用 LangChain 的 AgentExecutor（更简单），而要用 LangGraph？**

AgentExecutor 是"黑盒"——它的 ReAct 循环内部封装，难以插入自定义逻辑（如人工审批断点、状态持久化、错误恢复）。LangGraph 是"白盒"——状态转移显式定义，每个节点可自定义，支持复杂流程（如 HITL 断点、分支、循环）。生产 Agent 通常有复杂流程（审批、降级、重试），AgentExecutor 的简单循环不够用。所以 LangGraph 适合生产，AgentExecutor 适合原型。

### 第五层：验证与沉淀

**Q：怎么衡量五层架构的设计是否合理？**

三个指标：1) 层间独立性——每层能否独立测试、独立部署、独立扩展（如果不能，分层不到位）；2) 层间通信效率——层间调用的延迟占比（< 20% 合理，太高说明过度分层）；3) 故障隔离——一层故障是否影响其他层（应该不影响）。沉淀为架构 review checklist：每层的职责定义、接口契约、可独立部署性、可观测性。

## 结构化回答

**30 秒电梯演讲：** 设计生产可用 Agent 系统从0到1搭，按"接入层→编排层→能力层→数据层→可观测层"五层架构。接入层(API网关/鉴权/限流)，编排层(LangGraph状态机+Planning)。

**展开框架：**
1. **五层架构** — 接入(网关)→编排(状态机)→能力(Tool/Memory/LLM)→数据(Redis/PG/向量)→可观测(trace)
2. **高可用** — 熔断/降级/重试/兜底模型
3. **成本** — token预算/小模型决策+大模型总结/缓存

**收尾：** 您想深入聊：Agent 系统怎么做熔断降级？
