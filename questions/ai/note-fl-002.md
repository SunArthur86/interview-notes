# 【字节飞连面经】怎么设计 Tool Calling？Schema 怎么定义？防危险工具？

## 一、Schema 定义：双层校验

```python
# 模型侧：JSON Schema（OpenAI function calling 风格）
{
  "name": "get_user_info",
  "description": "查询用户基本信息",   # 模型主要看 description
  "parameters": {
    "type": "object",
    "properties": {
      "user_id": {"type": "string", "description": "用户唯一ID"},
      "fields": {"type": "array", "items": {"type": "string"}}
    },
    "required": ["user_id"]
  }
}

# 服务端侧：Pydantic 真正解析
from pydantic import BaseModel
class GetUserInfoParams(BaseModel):
    user_id: str
    fields: list[str] = []
```

字段名用 `snake_case`，**每个字段都写 description**（模型主要靠它理解意图）。输出 Schema 同样要定义，否则后置流程没法接。

## 二、工具失败：三分类处理

| 错误类型 | 处理策略 |
|---------|---------|
| 网络 / 5xx | 指数退避重试（1s, 2s, 4s） |
| 4xx 参数错 | **把错误信息塞回 prompt 让模型重写一次**，最多 2 次 |
| 业务侧拒绝（权限不足） | 立即终止 + 透传给用户 |

## 三、参数错误兜底

- **服务端默认值**：能给默认值的字段标记 `default`
- **枚举字段强制 fuzzy match**：模型输出"中国"，枚举只有"China"，做映射
- **必填字段缺失**：返回结构化错误让模型补一次，补不出来转人工

## 四、防止危险工具：分级 + 二次确认

```
工具危险分级：
  read-only   → 直接放行
  write       → 日志审计
  destructive → 绝不进 LLM 可见 tool list！
                由 LLM 提交"意图"，服务端拿到意图后
                弹二次确认（飞书卡片）→ 人确认 → 才执行
```

**核心原则**：deleterive 类工具（删数据库、转账）**绝不放进 LLM 可见的 tool list**。加 `allowlist` + 操作审计日志。

## 五、直接暴露 vs 服务端分发：永远服务端分发

理由：
1. **鉴权统一**：每个工具不重复写鉴权逻辑
2. **工具版本热更新**：不需要重新 prompt
3. **危险工具天然隔离**
4. **审计与限流好做**

## 六、判断是 Prompt 问题还是模型能力问题

- **控制变量**：同样 prompt 喂更大模型（豆包 1.6 → 豆包 pro / GPT-5），大模型能做对 → **多半是模型能力**；大模型也错 → **多半是 prompt**
- **看 log_prob**：模型对正确 token 给的概率不低但输出错 → prompt 引导不够；概率本身就低 → 能力问题
- **看 bad case 是否聚集**：80% 集中在某个工具 → 多半是 schema description 写得烂

## 七、加分点

提到 **MCP（Model Context Protocol）**：MCP 把"工具暴露"标准化了，未来工具发现可以走 MCP server，工具生态可以跨厂商复用。

## 八、雷区

- ❌ "我们让模型自己 retry" → 被秒挂，因为没控成本和死循环
- ❌ "危险工具直接放进 tool list 让模型谨慎用" → 安全红线

## 九、扩展

- Anthropic 的 MCP 把工具/资源/prompt 三类能力统一抽象，工具发现可以动态注册
- 高并发场景下 tool_call 要做令牌桶限流，防止 LLM 短时间内并发调用把下游 DB 打穿
- Structured Output（OpenAI 的 `response_format: json_schema`）比纯 function calling 更严格，能强制模型输出合法 JSON

## 记忆要点

- 双层校验防幻觉：模型侧用JSON Schema写清Description，服务端用Pydantic强制解析参数类型
- 工具失败分类处理：网络错指数退避重试，参数错报错让模型重写，业务拒绝立即透传终止
- 高危工具物理隔离：destructive类操作绝不可见，由LLM提交意图后触发人工二次确认（飞书卡片）
- 分发优于直暴：永远由服务端统一分发工具，因为天然隔离了危险操作且统一了鉴权与限流

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Schema 你为什么搞双层校验，模型侧 JSON Schema 已经够严了，服务端为什么还要再套一层 Pydantic？**

因为模型侧的 JSON Schema 只是"提示"，不是"约束"。LLM 仍可能产出 `user_id: 123`（数字）而不是 `"123"`（字符串），或者塞进 Schema 里没定义的字段。模型侧 Schema 的作用是引导模型按结构输出，真正拦住脏数据的必须是服务端的强类型校验。Pydantic 这层会在工具真正执行前抛 `ValidationError`，把不可信输入挡在业务逻辑之外——这是信任边界问题：模型输出是"非信任域"，服务端校验是"信任域入口"。

### 第二层：证据与定位

**Q：你怎么知道线上某次工具调用失败，到底是模型把参数拼错了，还是下游服务挂了？**

靠 `agent_steps` 表的全链路 trace。每次工具调用落四类字段：`tool_name`、`input_json`（模型产出）、`output_json`（或 error）、`latency_ms`。用 `trace_id` 串起来一查 SQL 就能分流：如果 `input_json` 里 `user_id` 是空串或类型错，是模型参数错（归到 4xx）；如果 `output_json` 是 5xx 或超时，是下游故障（走指数退避）；如果是 403/权限码，是业务拒绝。没这张表，定位全靠复现，复现不了就成悬案。

### 第三层：根因深挖

**Q：参数错你选择"把错误信息塞回 prompt 让模型重写，最多 2 次"。为什么不直接用 JSON Schema 做严格约束生成（Structured Output / Constrained Decoding）一劳永逸？**

严格约束生成（如 Outlines、OpenAI Structured Output 的 `response_format: json_schema`）能保证格式合法，但有代价：它会牺牲模型的部分推理能力，因为解码时每一步 token 都被 grammar 约束剪枝，模型不能自由思考；而且它只保证"格式合法"，不保证"语义正确"——`user_id` 必须是存在的用户 ID 这种业务约束它管不了。所以正确做法是分层：格式层用 Structured Output 兜底（如果模型支持且延迟可接受），语义层用塞回 prompt 重写 + Pydantic 校验 + 服务端业务校验。重试上限 2 次是成本控制——再多重试的边际收益极低，且烧 token。

**Q：那如果模型连续 2 次都拼错参数，为什么不在第 3 次直接微调模型而是放弃重试？**

微调是杀鸡用牛刀。单次 bad case 不代表分布问题，可能是这条 query 本身歧义或工具 description 写得烂。正确顺序是：先看是不是 description 没写清（改 description 比微调便宜 1000 倍），再统计这类错误在整体调用里的占比，只有当"参数错"在某个工具上稳定占比 > 10% 才考虑微调。第 3 次放弃是为了止损——再试下去既烧钱，用户体验也差，不如直接降级回"我不确定，请补充 X 信息"。

### 第四层：方案权衡

**Q：高危工具你用"物理隔离 + 飞书卡片人工二次确认"。为什么不让模型自己判断危险等级再决定要不要执行？**

因为模型的判断不可信。模型可能把"删除生产数据库"理解成普通操作，也可能被 prompt injection 诱导。高危操作的判定必须是确定性的，由人工预标注的危险工具清单（destructive 类）触发，不能交给概率模型。飞书卡片人工确认是把"最终决定权"从概率系统移交给确定性的人类，这是安全设计的铁律——不可逆操作的最后一关不能是 LLM。

**Q：那如果高危操作太多，每次都弹飞书卡片打扰人工，为什么不用规则引擎自动审批一部分？**

可以做分级，但分级规则必须比模型更确定。比如：读类操作（查询、统计）全自动；写类且影响范围 < 阈值（如改单个用户标签）规则引擎自动过；写类且不可逆（删库、清表、批量删除）必须人工。关键是自动审批的规则是 if-else 确定逻辑，不是模型概率判断。如果规则引擎也搞不准，宁可多打扰人工也别放行——高危操作误放的代价（数据丢了）远高于多弹一次卡片（烦一点）。

### 第五层：验证与沉淀

**Q：你怎么证明"服务端统一分发工具"比"把工具直接暴露给模型调用"更安全，有数据吗？**

拿三类指标对比：一是安全事故数——直暴模式下 prompt injection 触发的危险调用次数 vs 分发模式下被服务端拦截的次数。如果分发模式拦截率 > 99% 且直暴模式有真实事故，证明有效。二是鉴权一致性——直暴模式下每个工具自己实现鉴权，覆盖率难保证（漏一个就是洞）；分发模式鉴权在网关统一做，覆盖率 100%。三是限流效果——分发模式下能按工具粒度做令牌桶限流，直暴模式工具被绕过限流就失效，下游 DB 容易被打穿。

**Q：怎么让团队以后新增工具时都自觉走这套 Schema 规范，而不是偷偷加个裸接口？**

沉淀成工程约束而非口头约定：把 Schema 定义做成 CI 检查项——新增工具必须在 `tools/` 目录注册 JSON Schema + Pydantic Model + 危险等级标注，CI 校验三者字段一致才放行；工具分发走统一 SDK，裸接口在框架层直接拒绝注册；高危工具清单独立维护，新增 destructive 工具必须 PR review + 安全同学签字。让规范长在工具链里，而不是文档里。
