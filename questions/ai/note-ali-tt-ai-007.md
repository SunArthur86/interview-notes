## 二、工具一：链路追踪（理解调用关系）

### SkyWalking 链路追踪

```
一个请求的完整调用链（Trace）：

  用户下单请求
       │
       ▼
  ┌──────────────────────────────────────────────────────┐
  │ Trace: order-create-trace                            │
  │                                                      │
  │ [Gateway] 0ms─────────────────────────────────500ms  │
  │     │                                                │
  │     ├──[Order-Service] 5ms───────────200ms           │
  │     │     │                                          │
  │     │     ├──[User-Service] 10ms──────50ms (查用户)  │
  │     │     │                                          │
  │     │     ├──[Product-Service] 55ms────120ms (查库存) │
  │     │     │                                          │
  │     │     └──[Payment-Service] 130ms────190ms (支付)  │
  │     │                                                │
  │     └──[Inventory-Service] 210ms─────450ms (扣库存)   │
  │                                                      │
  │ 总耗时: 500ms  最慢: Inventory-Service(240ms)         │
  └──────────────────────────────────────────────────────┘

  从Trace中你能看到：
  ✅ 调用顺序：Gateway → Order → User/Product/Payment → Inventory
  ✅ 耗时分布：哪个服务最慢
  ✅ 同步/异步：哪些是并行调用
  ✅ 依赖关系：Order依赖User/Product/Payment/Inventory
```

### 如何使用链路追踪快速上手

```
Step 1: 找核心入口
  → 在SkyWalking中找一个高频接口（如 /api/order/create）
  → 查看它的完整Trace

Step 2: 画调用关系图
  → 根据Trace画出服务依赖图
  → 标注每个服务的职责和关键耗时

Step 3: 对比不同场景
  → 对比正常请求和慢请求的Trace差异
  → 理解什么情况下调用链会变长/变慢

工具选择：
  - SkyWalking（Apache开源，国内主流，阿里在用）
  - Zipkin（Twitter开源，轻量级）
  - Jaeger（CNCF项目，云原生生态）
  - 阿里内部：Eagle Eye（鹰眼）
```

## 三、工具二：注册中心（了解系统全貌）

### Nacos 注册中心

```
Nacos 控制台 → 服务管理 → 服务列表

┌────────────────────────────────────────────────┐
│ 服务名               │ 实例数  │ 健康实例  │ 命名空间│
├──────────────────────┼─────────┼──────────┼────────┤
│ order-service        │    3    │    3     │ prod   │
│ user-service         │    5    │    5     │ prod   │
│ product-service      │    4    │    3     │ prod   │ ← 1个不健康
│ payment-service      │    2    │    2     │ prod   │
│ inventory-service    │    3    │    3     │ prod   │
│ search-service       │    8    │    8     │ prod   │ ← 高并发服务，实例多
│ ai-agent-service     │    2    │    2     │ prod   │
└────────────────────────────────────────────────┘

  从服务列表你能了解到：
  ✅ 系统有哪些服务（全貌）
  ✅ 每个服务的规模（实例数反映流量大小）
  ✅ 服务健康状态（哪些有问题）
  ✅ 服务分组和命名空间（环境隔离）

  进一步：
  → 点击某个服务 → 查看订阅者（谁调用了这个服务）
  → 形成服务依赖关系图
```

### 通过Nacos梳理服务依赖

```
  在Nacos中查看每个服务的"订阅者列表"：

  order-service 的订阅者：
    ├── gateway-service（Gateway调Order）
    ├── payment-callback-service（支付回调调Order）
    └── report-service（报表服务查Order）

  product-service 的订阅者：
    ├── order-service（下单时查商品）
    ├── search-service（搜索时查商品）
    └── recommendation-service（推荐时查商品）

  → 逐个服务查看订阅者 → 反推出完整的服务依赖关系图
```

## 四、快速上手实战技巧

### 1. 跑通核心业务流程

```bash
# 找一个核心接口，从入口到出口完整走一遍
# 例如：电商下单流程

# Step 1: 找到入口（Gateway路由配置）
#   /api/order/create → order-service

# Step 2: 在SkyWalking中追踪这个请求
#   看到完整调用链：
#   Gateway → Order → User(查用户) + Product(查商品)
#                    → Payment(创建支付) → Inventory(扣库存)

# Step 3: 在代码中找到对应实现
#   OrderService.createOrder()
#     → userService.getUser()     // RPC调用
#     → productService.getProduct()
#     → paymentService.createPayment()
#     → inventoryService.deductStock()

# Step 4: 理解数据流
#   入参 → 处理逻辑 → 数据库写入 → 返回值
```

### 2. 代码走读优先级

```
优先级排序：

  P0：核心业务入口
  → 主流程API的Controller和Service
  → 理解"这个服务是做什么的"

  P1：数据模型
  → 数据库表结构、Entity/DO类
  → 理解"核心数据是什么"

  P2：外部依赖
  → RPC调用（Dubbo/Feign）、MQ消费
  → 理解"与其他服务的交互"

  P3：基础设施
  → 配置、中间件、工具类
  → 按需了解，不用一上来就看
```

### 3. 善用文档和同事

```
  必读文档：
  ├── 系统架构文档（架构图 + 技术选型）
  ├── API文档（Swagger/YApi）
  ├── 数据库设计文档
  ├── 开发规范文档（代码规范 + Git规范）
  └── oncall/排障文档（常见问题处理）

  主动沟通：
  → 找Mentor/老员工做15分钟"系统概览"讲解
  → 问"这个服务最核心的3个接口是什么？"
  → 问"最容易出问题的地方在哪？"
  → 带着具体问题去问，不要泛泛地问"系统怎么工作"
```

## 五、面试加分点

1. **提到APM工具的原理**：SkyWalking通过Java Agent字节码增强实现无侵入式链路追踪，理解其原理展示技术深度
2. **提到服务网格(Service Mesh)**：如果使用Istio，可以通过sidecar自动收集链路信息，不需要代码侵入
3. **提到文档建设**：主动提出"上手过程中整理一份新人文档"，展示主动性和团队意识
4. **提到AI辅助**：使用AI工具（如代码助手）快速理解陌生代码库，提升上手效率
5. **提到反模式**：不要一上来就改代码——先充分理解再动手，避免"好心办坏事"

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：两周内上手业务，你为什么要先跑通核心流程再读代码，而不是反过来？**

先跑通流程建立"系统观"，再读代码才有锚点。直接读代码是"盲人摸象"——不知道某个类在整个链路里的位置，读了也记不住、串不起来。先跑通"用户下单 → 支付 → 发货"这个核心流程（哪怕是本地环境模拟），知道了系统的输入输出和关键节点，再读代码时能带着"这个 RPC 对应流程里的哪一步"的问题去读，代码和业务对得上。动机是"自顶向下理解"——先宏观再微观，而不是自底向上迷失在细节里。

### 第二层：证据与定位

**Q：你说用 SkyWalking 追踪链路，具体看什么？怎么从 trace 里理解微服务间的调用关系？**

看三样东西：一是调用拓扑——trace 里每个 span 是一个服务调用，parent-child 关系就是调用链（A→B→C），把一个核心请求的 trace 展开，就是这条链路的微服务依赖图；二是耗时分布——每个 span 的 duration 占总耗时多少，定位性能瓶颈（如 B 服务占 80% 时间）；三是同步异步——span 之间是同步（RPC/HTTP）还是异步（MQ），理解服务的耦合方式。把几个核心业务（下单/支付/查询）的 trace 都看一遍，微服务调用关系就八九不离十了。

### 第三层：根因深挖

**Q：SkyWalking trace 只能看到有 Agent 的服务，如果有服务没接 SkyWalking（或用别的语言），链路就断了。根因怎么办？**

用注册中心（Nacos）补全。SkyWalking 断链的地方，去 Nacos 查该服务的下游依赖——Nacos 里每个服务注册了它调用的其他服务（通过配置或 SDK 自动注册），即使没有 trace 也能看到静态依赖关系。组合使用：SkyWalking 看动态调用链（有 trace 的部分），Nacos 看静态拓扑（所有注册服务的依赖），两者拼起来就是完整的微服务依赖图。如果有些服务既没 SkyWalking 也没 Nacos（如第三方系统），看代码里的 RPC 配置（如 Feign client、RestTemplate 的 URL）补。

**Q：那为什么不直接读代码里的 RPC 配置（如 Feign/Dubbo 注解）梳理依赖，省得依赖 APM 工具？**

读代码梳理是静态的、离线的，且工作量大——几十个微服务的代码逐个翻 RPC 配置，几天都看不完，且容易漏（如动态 URL、反射调用）。APM 工具是动态的、实时的——跑一个请求就能自动看到完整链路，且包含运行时信息（耗时、错误率、QPS）这些代码里看不到的。APM 是"让系统告诉你它的结构"，读代码是"你猜系统的结构"，前者准且快。当然，APM 覆盖不到的（未接入的服务）要用代码补，两者互补。

### 第四层：方案权衡

**Q：你提到用 AI 代码助手快速理解陌生代码库，具体怎么用？会不会被 AI 误导（理解错）？**

用法是"让 AI 解释代码段"——把不理解的类/函数贴给 AI，让它解释作用、调用关系、业务含义，比自己读快 3-5 倍。但有误导风险：AI 可能基于函数名臆测功能（如把一个废弃的方法当成核心逻辑），或基于通用模式误判（如以为某段是缓存逻辑实际是限流）。防误导的方法：AI 的解释必须回到代码验证——让 AI 指出它依据哪些代码行，你核对原文；关键逻辑（如核心算法、安全校验）必须自己读懂，不能全信 AI。AI 是加速器不是替代，最终理解责任在你。

**Q：为什么不直接问团队的老员工，省得自己查？老员工最懂业务。**

老员工时间宝贵，不能把所有问题都问他们（会消耗他们的时间、显得不独立）。正确策略是：自己能查的（代码、文档、APM、Nacos）先查，查不到的（业务背景、历史决策、为什么不这么设计）再问老员工。且问的时候要"带着理解问"——"我看到这块用了 X 方案，是不是因为 Y 约束？"，而不是"这块是干嘛的"。前者证明你做过功课，老员工愿意深入聊；后者是伸手党，老员工只会敷衍。两周上手的目标之一就是"问出高质量的问题"。

### 第五层：验证与沉淀

**Q：你怎么证明自己"上手了"，而不是只是看了两周代码？**

产出一个小的可验证成果：改一个小 bug 或做一个小需求，独立完成（设计→编码→测试→上线）。这证明你理解了代码结构（能定位到改动点）、开发流程（代码规范、CI/CD、上线步骤）、业务逻辑（知道改动的业务影响）。同时输出一份"新人文档"——把你这两周踩过的坑、理解的系统架构、核心流程梳理成文档，既能帮下一个新人，也是你理解的沉淀。能产出成果 + 能教会别人，才算真上手。

**Q：快速上手的方法论怎么沉淀成团队的新人 onboarding 流程？**

固化成"新人 14 天 onboarding checklist"：Day 1-3 看架构文档 + 跑通本地环境；Day 4-7 用 APM/Nacos 梳理核心链路 + 读关键代码；Day 8-10 做第一个小需求（配 mentor review）；Day 11-14 独立做第二个需求 + 写新人文档。配套"新人友好型 starter issue"（标好的简单任务）、"系统架构 wiki"（保持更新）、"新人 mentor 制度"。把每个新人补充的"踩坑记录"沉淀进 wiki，让 onboarding 越来越顺。
