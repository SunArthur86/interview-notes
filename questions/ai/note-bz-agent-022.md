# Agent Memory 怎么设计？

## 一、记忆的分类

```
┌──────────────────────────────────────────────────┐
│                  Agent记忆分类                      │
├──────────────────────────────────────────────────┤
│                                                    │
│  短期记忆 (Short-term / Working Memory)            │
│  = 当前对话上下文（LLM窗口内）                      │
│  特点：快、LLM直接可见、容量有限（如128K）           │
│  内容：当前对话历史、当前任务状态                    │
│                                                    │
│  长期记忆 (Long-term Memory)                       │
│  = 向量数据库（跨会话持久化）                        │

│  内容：用户偏好、历史事实、学到的知识                │
│                                                    │
│  工作记忆 (Working / Scratchpad)                   │
│  = 当前任务的中间状态                               │

│  = 历史执行轨迹（"经历"）                           │

│                                                    │

```

## 二、写入策略（什么时候记、记什么）

```python
class MemoryWriter:
    def should_write(self, content, context):
        """判断是否值得写入长期记忆"""
        
        # 1. 重要性评分（不是所有东西都记）
        importance = self.llm.score_importance(content, context)
        # 例: "用户喜欢简洁风格" → 0.9（重要）
        #     "用户说了句你好" → 0.1（不重要）
        if importance < 0.5:
            return False
        
        # 2. 去重（避免重复存储）
        similar = self.vector_db.search(content, threshold=0.85)
        if similar:
            # 合并而非新建
            self.merge(similar[0], content)
            return False
        
        # 3. 压缩（原始对话太长，提取要点）
        compressed = self.llm.summarize(content)
        # 例: 10轮对话 → "用户在讨论Python项目，偏好异步方案"
        
        # 4. 打标签（便于后续检索过滤）
        metadata = {
            "user_id": context.user_id,
            "type": self.classify(compressed),  # preference/fact/event
            "importance": importance,
            "timestamp": now(),
            "ttl": self.calc_ttl(importance)  # 重要性越高TTL越长
        }
        
        self.vector_db.add(compressed, metadata)
```

## 三、检索策略（怎么找到相关的）

```python
class MemoryRetriever:
    def recall(self, query, user_id, top_k=5):
        """多路召回 + 重排序"""
        
        # === 第1路：语义检索（向量） ===
        semantic_results = self.vector_db.search(
            embedding(query),
            filter={"user_id": user_id},  # 用户隔离
            n_results=top_k * 3
        )
        
        # === 第2路：关键词检索（BM25） ===
        keyword_results = self.bm25.search(
            query, filter={"user_id": user_id}
        )
        
        # === 第3路：时间近因（最近优先） ===
        recent = self.get_recent(user_id, limit=top_k)
        
        # === 融合 + 重排序 ===
        candidates = merge(semantic_results, keyword_results, recent)
        ranked = self.reranker.rerank(query, candidates)
        
        # === 过滤 ===
        valid = [
            m for m in ranked
            if not m.is_expired()           # 未过期
            and m.relevance > 0.3            # 相关性达标
            and m.user_id == user_id         # 用户隔离（双保险）
        ]
        
        return valid[:top_k]
```

## 五、遗忘机制（什么时候忘）

```python
class ForgettingManager:
    def cleanup(self, user_id):
        """多种遗忘策略组合"""
        
        # 1. TTL过期（时间驱动）
        expired = self.db.query(
            {"user_id": user_id, "ttl < now()})
        self.soft_delete(expired)  # 软删除，保留30天可恢复
        
        # 2. LRU淘汰（容量驱动）
        count = self.db.count({"user_id": user_id})
        if count > MAX_MEMORIES:
            oldest = self.db.query(
                {"user_id": user_id},
                sort="last_accessed", limit=count - MAX_MEMORIES
            )
            self.soft_delete(oldest)
        
        # 3. 重要性衰减（评分驱动）
        for memory in self.db.all(user_id):
            # score = importance × recency × frequency
            days = (now() - memory.last_accessed).days
            memory.score = (
                memory.importance *
                math.exp(-0.1 * days) *  # 时间衰减
                math.log(1 + memory.access_count)  # 频率加权
            )
            if memory.score < 0.1:
                self.soft_delete(memory)
        
        # 4. 主动遗忘（合规驱动）
        # GDPR right to be forgotten
        if user_requested_deletion(user_id):
            self.hard_delete(user_id)
```

## 七、记忆设计的核心权衡

```
┌──────────────┬──────────────┬──────────────────┐
│ 权衡维度      │ 一端           │ 另一端             │
├──────────────┼──────────────┼──────────────────┤
│ 写入门槛      │ 全记（不漏）    │ 严格筛选（不冗余）  │
│ 检索召回      │ 多召回（不漏）  │ 高精度（不噪）      │
│ 遗忘激进度    │ 少忘（保留多）  │ 多忘（省资源）      │
│ 隔离强度      │ 共享（省资源）  │ 隔离（保隐私）      │
└──────────────┴──────────────┴──────────────────┘

经验值：
  - 写入：重要性>0.5才记（宁可漏不可噪）
  - 检索：先多召回(3倍)再rerank到top-k
  - 遗忘：重要记忆永不删，普通记忆TTL 30天
  - 隔离：用户数据强制隔离，通用知识共享
```

## 八、面试加分点

1. **强调"存得下不等于找得到"**：记忆的核心难点不是存储而是检索——召回率和精度
2. **遗忘是 feature 不是 bug**：主动设计遗忘机制，避免记忆膨胀和噪声
3. **提"软删除+保留期"**：误删可恢复，体现工程严谨

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 记忆设计四要素（分类、写入、检索、隔离），为什么"写入"最难？决定"记什么"不比"检索"简单吗？**

"写入"是源头决策，错了下游全错。检索是在"已写入的记忆"里找相关的，写入决定了"记忆库的构成"——如果写入的都是噪声（寒暄、重复确认、失败尝试），检索再好也找不到有用信息。"记什么"难的点：1）重要性判断——LLM 要判断每条信息"值不值得记"，这是主观的（不同任务重要性不同）；2）时效性判断——有些信息时效短（如"今天股价"）记了很快过期，有些长效（如"用户偏好简洁风格"），要差异化处理；3）冲突处理——新信息和旧记忆矛盾时（如用户改了偏好），要更新还是保留历史？"记什么"决策错了，记忆库要么膨胀（全记）要么丢失关键信息（漏记），都会让检索质量崩塌。

### 第二层：证据与定位

**Q：Agent 该用记忆时没用（忘了用户偏好），怎么定位是"没写入"还是"写入了但检索不到"？**

查记忆库的两层日志。1）写入日志——看那条用户偏好（如"我喜欢简洁回答"）是否被写入记忆库（写入决策是否触发），如果日志里没有写入记录，是"写入漏了"（重要性判断错误，LLM 没觉得该记）；2）检索日志——如果写入了，看当前对话是否检索了这条记忆（query="用户偏好"是否召回这条），如果检索召回但 Agent 没用，是"Agent 没引用检索结果"（prompt 问题），如果检索没召回，是"检索质量差"（embedding/相关性算法问题）。三层定位：写入漏→优化写入 prompt（明确"用户偏好必记"）；检索没召回→优化 embedding/加 rerank；召回没用→优化 Agent prompt（"优先参考检索到的用户偏好"）。

### 第三层：根因深挖

**Q：Agent 记忆库越来越大，检索延迟从 50ms 涨到 500ms，怎么优化？是索引问题还是数据量问题？**

两者都有，分层优化。1）数据量——记忆库从 1 万条涨到 100 万条，向量检索的延迟自然上升（即使有 ANN 索引）。优化：分区检索（按用户/时间/类型分库，每次只检索单用户/近期的，缩小候选集）+ 预过滤（先用 metadata 如时间范围筛，再向量检索）；2）索引——HNSW/IVF 等索引参数没调好（如 HNSW 的 efSearch 太大），调参（efSearch 从 200 降到 50，延迟减半召回略降）；3）缓存——高频查询的记忆结果缓存（如"用户偏好"每次对话都查，缓存后命中直接返回），命中率高的场景能降 80% 延迟。优化顺序：先缓存（零成本高收益）→ 再分区（缩小数据量）→ 最后调索引参数。

**Q：记忆的"遗忘"机制（TTL/容量淘汰）会不会误删重要信息？怎么平衡"省空间"和"不丢关键信息"？**

用"重要性 × 时效性"二维评分做淘汰，而非单一 TTL。1）重要性——LLM 或规则给每条记忆打分（如"用户偏好=高""寒暄=低"），低分的优先淘汰；2）时效性——事实型记忆（如订单状态）短 TTL，偏好型长 TTL；3）访问频率——长期没被检索命中的记忆降权（类似 LRU），频繁命中的保留。淘汰策略：容量到上限时，按"重要性低 × 过期 × 低访问"优先删。安全网：高重要性记忆（如用户核心偏好）设"不可淘汰"标记，即使容量满也不删（宁可扩容不丢关键信息）。这套组合比单一 TTL 精细，能把记忆库控制在高质量小集合，检索又快又准。

### 第四层：方案权衡

**Q：Agent 记忆用"向量库"还是"关系数据库（存结构化记忆）"？还是两者结合？**

按记忆类型选存储。1）语义型记忆（用户偏好、事实、经验）——用向量库，因为检索靠语义相似度（如"用户喜欢什么风格"要语义匹配）；2）结构化记忆（用户信息、订单、配置）——用关系 DB，因为要精确查询（如"查用户 X 的订单"是 key-value 查询，向量检索不合适且不准）；3）时序记忆（对话历史）——用时序 DB 或日志（按时间范围查询）。实务三者结合：结构化的存关系 DB（精确查询），语义的存向量库（模糊检索），时序的存日志（历史回溯）。检索时按记忆类型路由到对应存储。单一存储（全向量或全关系）都有盲区——向量做不了精确查询，关系 DB 做不了语义检索。

**Q：记忆的"用户隔离"和"会话隔离"为什么要分？一个用户的所有记忆放一起不行吗？**

分场景。1）个人助手场景——一个用户的记忆跨会话共享（上周说的偏好这周也用），用户内不分会话，只按用户隔离（A 的记忆 B 看不到）；2）多任务场景——同一用户有多个独立任务（如"帮我规划旅行"和"帮我写简历"），任务间的记忆要隔离（旅行的 context 不污染简历任务），要按"用户+会话"双隔离；3）共享场景——团队 Agent 要共享部分记忆（如团队的项目文档），部分私有（个人偏好），要细粒度权限。隔离粒度按业务：个人助手用户级隔离够，多任务要会话级，团队要权限级。不隔离会导致"A 用户的敏感信息泄露给 B"或"任务 A 的 context 干扰任务 B"，都是严重问题。

### 第五层：验证与沉淀

**Q：你怎么衡量 Agent 记忆系统的健康度（写入/检索/淘汰都在正常工作）？**

四个指标。1）写入准确率——应写入的重要信息中实际写入的比例（人工抽检"这轮对话哪些该记"，对比实际写入），目标 >85%；2）写入噪声率——写入的记忆中"无用信息"占比（如寒暄被记），目标 <15%；3）检索命中率——应被检索到的记忆（人工标注"当前任务需要哪些历史记忆"）中实际召回的比例，目标 >80%；4）记忆利用率——检索召回的记忆中 Agent 实际引用的比例（召回但没用说明检索不准或 prompt 没引导），目标 >70%。四个指标打到 dashboard，任一低于阈值告警。还有"记忆库增长率"——如果增长过快说明写入不筛选（噪声多），增长停滞可能漏写。

**Q：Agent 记忆系统的四要素设计怎么沉淀成框架的 Memory 组件？**

封装成 MemoryManager 组件：1）写入决策器——内置重要性评分（LLM 或规则）+ 类型分类（事实/偏好/情景），自动决定记不记、存哪；2）多存储路由——按记忆类型自动路由到向量库/关系 DB/日志，开发者不用手选；3）检索器——内置向量检索 + rerank + metadata 过滤，返回 top-K 相关记忆；4）淘汰器——"重要性 × 时效 × 访问频率"三维评分自动淘汰，高重要性记忆不可淘汰；5）隔离层——按用户/会话/权限自动隔离。开发者只需配置"记什么类型/容量上限/隔离粒度"，框架自动跑写入-检索-淘汰。这套写入团队 Agent 框架 SOP，新 Agent 接入 Memory 即具备完整记忆能力，不重写四要素。
