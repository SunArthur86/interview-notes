# 【阿里淘天AI二面】如何避免Agent场景下模型的幻觉问题？

> 来源：阿里巴巴淘天淘工厂 AI应用开发 二面面经（小红书）

## 一、幻觉的本质与根源

### 🎯 什么是幻觉

```
幻觉 = LLM生成"看似合理但与事实不符"的内容

示例：
  用户：阿里巴巴是哪年成立的？
  模型（幻觉）：阿里巴巴成立于1998年3月。  ← 错误，实际是1999年

  模型不知道"不知道" → 总是生成最可能的回答
  这是概率生成模型的固有特性
```

### 幻觉的类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **事实性幻觉** | 生成与客观事实矛盾的内容 | "太阳是第9大行星" |
| **忠实性幻觉** | 输出与给定上下文/文档矛盾 | 文档说A，模型回答B |
| **推理幻觉** | 推理步骤逻辑错误 | 前提正确但推导过程有误 |
| **来源幻觉** | 编造不存在的引用/来源 | "根据Smith et al. (2023)..." |

### 为什么会产生幻觉

```
LLM本质：next-token prediction
  P(y | x) = 最大化给定上文x，下一个token y的概率

  当模型内部知识不充分时：
  ┌─────────────────────────────────┐
  │ 知识库中没有"阿里巴巴成立年份"     │
  │ 但训练数据中见过类似公司信息       │
  │ → 模型用概率推断"编造"一个答案    │
  │ → "1998年"（虽然错误，但概率最高） │
  └─────────────────────────────────┘

  核心原因：
  1. 训练数据噪声/过时
  2. 模型概率推断产生的"创造"
  3. 缺乏事实校验机制
  4. 没有显式的"不确定"输出机制
```

## 二、三层防线

### 第一层：提示词角度（预防）

```
策略1：约束模型基于知识库回答
  ┌─────────────────────────────────────────────┐
  │ System Prompt:                               │
  │                                              │
  │ 你是一个智能客服Agent。你必须：               │
  │ 1. 只基于提供的知识库文档回答问题             │
  │ 2. 如果知识库中没有相关信息，回答              │
  │    "抱歉，我暂时无法回答这个问题，            │
  │     正在为您转接人工客服"                     │
  │ 3. 不要编造、猜测或推断知识库中没有的信息      │
  │ 4. 每个回答必须标注信息来源                   │
  └─────────────────────────────────────────────┘

策略2：要求标注来源
  "请在回答末尾标注信息来源，格式：[来源：文档名-段落]"

策略3：设置"不知道"兜底
  "当你不确定答案时，必须回答'我不确定'而不是编造答案"

策略4：降低温度参数
  temperature = 0.1 ~ 0.3
  低温度 → 减少随机性 → 减少创造性幻觉
```

### 第二层：Workflow角度（过程控制）

```
策略1：结构化决策输出

  强制Agent输出JSON Schema：
  ┌─────────────────────────────────────────┐
  │ {                                       │
  │   "answer": "阿里巴巴成立于1999年",      │
  │   "source": "公司简介.pdf - 第3页",      │
  │   "confidence": 0.95,                   │
  │   "needs_human": false                  │
  │ }                                       │
  └─────────────────────────────────────────┘

  校验规则：
  - source为空 → 拒绝输出（无依据的回答）
  - confidence < 0.7 → 转人工
  - answer中包含source中没有的信息 → 拒绝输出

策略2：分步决策，每步可校验

  Step 1: 意图识别 → 输出intent + confidence
  Step 2: 知识检索 → 输出documents + relevance_score
  Step 3: 答案生成 → 输出answer + source + confidence
  Step 4: 答案校验 → 输出pass/fail + reason

  每步都有明确的输出Schema和校验规则
  → 任何一步校验失败，不会传递到下一步

策略3：RAG检索增强

  用户问题 → 向量检索Top-K文档 → Rerank精排
  → LLM基于检索结果生成答案（而非依赖内部知识）

  关键：答案必须引用检索到的文档内容
```

```python
from pydantic import BaseModel, Field

class AgentResponse(BaseModel):
    """结构化输出Schema"""
    answer: str = Field(description="回答内容，必须基于retrieved_docs")
    source: str = Field(description="信息来源文档名和段落")
    confidence: float = Field(ge=0, le=1, description="置信度")
    needs_human: bool = Field(description="是否需要转人工")

    def validate_against_docs(self, retrieved_docs: list[str]) -> bool:
        """校验answer中的关键信息是否在retrieved_docs中"""
        if not self.source:
            return False
        if self.confidence < 0.7:
            return False
        # 可选：NLI模型检查answer与docs的一致性
        return True
```

### 第三层：后处理角度（兜底校验）

```
策略1：双模型校验

  模型A（生成模型）→ 生成答案
       │
       ▼
  模型B（审核模型）→ 校验答案是否准确安全
       │
       ├── 通过 → 输出给用户
       └── 不通过 → 重新生成 OR 转人工

  为什么有效？
  - 不同模型的幻觉模式不同
  - 两个模型同时产生相同幻觉的概率很低
  - 审核模型可以基于检索文档做事实核查

策略2：自一致性投票（Self-Consistency）

  同一个问题，让模型生成N个答案（temperature=0.7）：
  答案1: "1999年"
  答案2: "1999年"
  答案3: "1998年"
  答案4: "1999年"
  答案5: "1999年"

  → 多数投票：1999年（4/5一致）

  幻觉是随机的，多次生成中幻觉内容通常不一致
  → 多数投票可以过滤掉随机幻觉

策略3：置信度过滤

  对Agent输出做置信度评估：
  - 基于logits的概率（如果可用）
  - 基于生成模型的self-evaluation
  - 基于检索结果的相关性分数

  confidence < threshold → 拒绝输出 / 转人工
```

## 四、面试加分点

1. **提到RAG幻觉的根因**：检索质量差（召回不相关文档）会导致模型基于错误信息生成答案，所以rerank很重要
2. **提到Grounding技术**：Google的Grounding API，将模型输出与搜索结果做实时事实验证
3. **提到Constitutional AI**：Anthropic的方法，让AI自己根据"宪法"（原则）来审查和修正自己的输出
4. **提到量化幻觉率**：使用RAGAS、TruLens等框架量化幻觉率（faithfulness score）
5. **提到温度与幻觉的关系**：temperature=0能显著减少随机性幻觉，但不能完全消除（模型本身的参数化知识可能就是错的）

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 场景防幻觉为什么要搞提示词 + workflow + 后处理三层，而不是把 RAG 做准就够了？**

因为 RAG 做准只解决"知识来源对"，解决不了"模型乱用知识"。即使检索到了正确文档，模型仍可能：偏离文档编造（temperature 发散）、混用训练记忆和文档（参数知识污染）、跨文档错误拼接（把 A 文档的数字套到 B 文档的实体上）。提示词约束（"只基于资料回答"）管模型的行为模式，workflow 结构化（分步可校验）管生成过程，后处理双模型审核管最终输出，三层分别防不同失败模式。单做 RAG 只堵住一个口，堵不住模型本身的概率性编造。

### 第二层：证据与定位

**Q：Agent 输出有幻觉，你怎么判断是 RAG 检索错了（召回错文档）还是模型编造（有正确文档但没用）？**

用 RAGAS 的 Context Recall + Faithfulness 交叉定位。Context Recall 低 → 正确文档没召回，是检索问题；Context Recall 高但 Faithfulness 低 → 文档召回了但模型没用，是生成层问题。具体看 Trace：检索回了哪几个 chunk、模型生成时引用了哪些 chunk、哪些生成的句子在 chunk 里找不到支撑。Faithfulness 的原子陈述拆解能精确指出"这句话没有文档支撑"，是编造的。

### 第三层：根因深挖

**Q：你发现模型有正确文档却仍然编造数字，根因是什么？temperature 已经设 0 了。**

根因是参数知识压过上下文知识。模型在预训练里见过类似但不准确的数字（如某产品旧参数），即使文档里有正确数字，模型生成时如果参数记忆的 token 概率更高，会输出记忆而非文档值。temperature=0 只是贪心解码（选最高概率 token），如果参数记忆的概率更高，照样稳定输出错的。治本是 prompt 里强约束（"数字必须严格引用资料原文，禁止用记忆补充"）+ 后处理校验（正则比对生成的数字是否在文档里出现）。

**Q：那为什么不直接对数字做 constrained decoding，只允许生成文档里出现的数字 token？**

理论上可以，但工程实现复杂且场景受限。数字的 constrained decoding 需要维护一个动态的"允许数字 token"集合（随文档变化），且数字可能以多种形式表达（100 / 一百 / 100.00），约束解码的 trie 难以覆盖。更关键的是幻觉不只在数字——日期、名称、因果关系都可能编造，对每种都做约束不现实。实际工程用"后处理校验"更通用——生成后用规则或 NLI 模型比对文档，不一致就触发重写或拒答，比 constrained decoding 灵活且易维护。

### 第四层：方案权衡

**Q：后处理你用"双模型审核"（第二个 Agent 审第一个的输出），为什么不直接用 NLI 模型做事实校验？**

精度和覆盖面。NLI 模型（如 DeBERTa）只能判断"这句话是否被文档支持"，对语义层面的错误（如因果颠倒、逻辑矛盾）捕捉弱，且对长句和复杂推理判断不准。双模型审核（让 LLM 审 LLM）能理解语义和逻辑，精度更高，且能给出"为什么错"的解释。代价是成本——LLM 审核比 NLI 贵 100 倍。工程上分层：NLI 做粗筛（快且便宜，拦明显的），NLI 不通过的再走 LLM 审核（贵但准），平衡成本和精度。

**Q：为什么不直接换 GPT-4o/Claude（幻觉更少）省掉三层防线？**

顶级模型幻觉率低但不到零。GPT-4o 在事实性任务上幻觉率约 3-5%，生产 SLA 要求 <1%，仍有差距。且换模型不能解决"业务私有知识"的幻觉——模型对训练后的新知识照样编造，仍需 RAG + 校验。更现实的是：用中等模型（如 GPT-4o-mini）+ 三层工程防线，成本远低于全量用 GPT-4o，且幻觉率可控。工程防线的价值是"模型无关"，换模型时防线复用，不把可靠性绑死在单一模型上。

### 第五层：验证与沉淀

**Q：你怎么量化三层防线的幻觉抑制效果，证明每一层都有贡献？**

构建对抗性评测集（含易幻觉 case：数字冲突、过时知识、诱导发散），埋点统计每层的拦截率。总幻觉 case 1000 条：RAG 召回正确拦下 30%（知识对就不编）、prompt 约束拦下 25%（引导模型用文档）、后处理审核拦下 20%（兜住前两层漏的），最终幻觉率从 baseline 20% 降到 2%。每层的拦截率证明其价值。用 RAGAS Faithfulness 每天抽样监控，Faithfulness <0.9 触发告警。

**Q：防幻觉方案怎么沉淀成 Agent 开发规范？**

固化成"防幻觉 checklist"：必须接 RAG（监控 Context Recall）、prompt 必须含"基于资料回答+不知道就说不知道"约束、关键场景（含数字/事实）必须过双模型审核、必须配拒答兜底（校验不通过就回"未找到相关信息"而非硬答）。封装统一的 `hallucination_guard()` 中间件，业务侧一行接入。把"易幻觉 case 库"做成回归测试集，每次模型/prompt 改动自动跑。

## 结构化回答

**30 秒电梯演讲：** 幻觉是LLM"一本正经地胡说八道"——模型生成看似合理但事实错误的内容。三层防线：(1)提示词角度——约束模型必须基于知识库回答，不知道就说不知道；(2)workflow角度——结构化决策，保证每步可校验。

**展开框架：**
1. **幻觉根源** — LLM是概率生成模型，没有"不知道"的概念，总是生成最可能的下一个token
2. **提示词防线** — 约束模型基于知识库回答、要求标注来源、设置"不知道"兜底
3. **workflow防线** — 结构化输出(JSON Schema)、分步决策、每步可校验

**收尾：** 您想深入聊：RAG检索不到相关信息时，如何防止模型"硬编"答案？
