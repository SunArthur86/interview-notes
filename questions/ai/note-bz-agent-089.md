# 如何将企业级 Agent 开发经验融入简历？

## 一、简历撰写的核心原则

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 错误(职责描述)      │ 正确(成就展示)          │
├──────────────┼──────────────────┼──────────────────────┤
│ 动词          │ "负责"            │ "设计/优化/提升"        │
│ 内容          │ 做了什么           │ 解决了什么+效果        │
│ 数据          │ 无                │ 有量化指标             │
│ 技术深度      │ 堆名词             │ 讲方案选型+tradeoff    │
│ 业务价值      │ 无                │ 讲对业务的影响         │
└──────────────┴──────────────────┴──────────────────────┘
```

## 二、STAR 结构化表达

```
STAR = Situation(情境) + Task(任务) + Action(行动) + Result(结果)

示例：
S: 客服系统日咨询量10万+，人工处理成本高
T: 开发智能客服Agent，自动处理常见问题
A: 设计意图识别+多技能路由+FAQ混合检索架构，
   实现RAG知识库+人工协作流程
R: 自动解决率70%，年节省人力成本200万，
   用户满意度4.3/5
```

## 三、企业级 Agent 经验的简历模板

```markdown
## XX公司 · AI Agent工程师 · 2024.X-X

### 智能客服Agent系统（日活5万+）
- **架构设计**：设计意图路由→多技能处理→人工协作三层架构，
  支撑日均10万+咨询量，P99延迟<2s
- **检索优化**：实现向量+BM25+Rerank混合检索，
  FAQ匹配准确率从60%提升至88%
- **成本优化**：设计模型路由策略（简单走小模型），
  单次对话成本降低65%
- **稳定性**：实现限流+熔断+降级三层防护，
  可用性达99.9%
- **技术栈**：LangGraph/Milvus/Redis/GPT-4o

### RAG知识库系统（百万级文档）
- **检索引擎**：设计父子分块+HyDE+多路召回，
  recall@5从65%提升至87%
- **评估体系**：搭建RAGAS评估闭环，
  faithfulness达92%，持续驱动优化
- **数据管道**：实现增量索引+版本管理，
  知识更新延迟<5分钟
```

## 四、技术深度的表达

```
不要只说"用了X"，要说"为什么用X、怎么用的、效果如何"

❌ "使用LangChain开发"
✅ "采用LangGraph图结构编排，支持人工审核节点
    和状态恢复，相比传统Chain提升了可控性"

❌ "用向量数据库"
✅ "对比Milvus/Pinecone/Qdrant后选Milvus，
    支撑亿级向量，P99检索延迟<50ms"

❌ "做了RAG"
✅ "设计三路混合检索(向量+BM25+元数据)，
    recall从65%提升至87%，配合Cross-Encoder Rerank
    精度提升15%"

关键：体现"技术选型能力"和"优化能力"
```

## 五、业务价值的表达

```
技术人容易只讲技术，但面试官也关心业务影响

技术成就 → 业务价值 的转化：
  "召回率提升22%" → "转人工率降22%，年省客服成本150万"
  "延迟降60%" → "用户等待时间减半，满意度提升0.5分"
  "成本降65%" → "单次对话成本从¥0.3降至¥0.1，支持规模化"
  "可用性99.9%" → "年故障时间<9小时，保障核心业务"

公式：技术指标 × 业务场景 = 业务价值
```

## 七、面试加分点

1. **成就 > 职责**：每条简历都是"做了什么+效果如何"，而非"负责什么"
2. **量化一切**：数字是最有力的证据——延迟、准确率、成本、规模
3. **双价值**：技术深度（怎么做的）+ 业务价值（影响了什么）——两个都要有

## 记忆要点

- 拒绝职责堆砌，因为面试官只想看成就，所以用“设计/优化”替代“负责”并辅以量化数据
- 熟练套用STAR法则，S明确场景与痛点，A展现选型与Trade-off，R必须提供绝对量化指标
- 技术深度公式：不仅说用了什么，更要说为什么选它、怎么用的、取得了什么量化效果
- 技术指标转化为业务价值：因为算法提升难直观感知，所以要转化为降本（省XX万）或增效指标
- 根据工作年限定策略：1-3年重深度与量化，3-5年重架构与选型，5年以上看全局与商业价值

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你强调"成就导向"要量化，但如果我的项目没有埋点、没有评测体系，上线后也没人测过准确率，我拿不出真实数据，简历怎么办？不能编吗？**

不能编，但可以"合理估算 + 事后补测"。编数据的红线是：面试官追问"这个 88% 怎么测的"，你答不上评测方法会被判定造假，直接出局。正确做法是事后补测——把当时的 FAQ 库和 200 条真实 query 拉出来，现在跑一遍召回率（即使项目已下线，数据一般还在），这叫"事后验证"，是真实的。如果数据真没了，写"约 80%（基于 XX 条样本估算）"并标注是估算，诚实比数字漂亮重要。最差的情况数据完全无法恢复，那就别写具体数字，改写选型决策和踩坑过程（这些不依赖数据也有深度）。

### 第二层：证据与定位

**Q：简历写"年节省人力成本 200 万"，面试官问"这 200 万怎么算的"，我怎么回答才能站得住？**

必须能拆解计算公式。200 万 = 节省的人力工时 × 单位工时成本。具体：自动解决率 70% × 日咨询量 10 万 = 日自动解决 7 万次；这些如果不自动需要人工处理，按人工客服单次处理 3 分钟、时薪 30 元算，单次成本 1.5 元；7 万次 × 1.5 元 × 365 天 = 383 万毛节省；扣掉 Agent 的 API 成本（7 万次 × 0.1 元 × 365 = 255 万），净节省约 128 万，取整写"约 130-200 万"。关键是每个参数（单次处理时长、时薪、自动解决率）都能说来源，面试官追问任何一个都能答。如果某个参数是假设（如时薪），标注"按 XX 行业均值估算"。

### 第三层：根因深挖

**Q：你说技术指标要转化成业务价值，但如果我是算法/研发岗，不直接接触业务侧的营收/成本数据，怎么知道我的优化值多少钱？**

自己建模型估算，不依赖业务方给数据。三个关键参数你能拿到或合理假设：优化的量级（如召回率提升 22%）、影响的业务量（如日咨询量 10 万）、单位价值（如单次转人工成本 1.5 元）。转化公式：召回率提升 22% → 转人工率降 22% → 日少转人工 2.2 万次 → 日省 3.3 万 → 年省约 1200 万（毛）。这个估算不精确但有量级参考，简历写"约 X 百万量级（基于 Y 假设估算）"。面试官看重的是你"能把技术影响量化到业务维度"的思维方式，不是数字精确到个位。如果你连这种估算都做不出，说明你不懂自己工作的业务价值，这才是减分项。

**Q：那为什么非得扯到钱，我是个工程师，把召回率从 60 提到 85 不就证明能力了吗？**

因为不同公司的"召回率 60→85"业务影响天差地别。在一个日咨询量 100 的小工具上提 25%，和在一个日咨询量 100 万的核心业务上提 25%，价值差 1 万倍。面试官无法单凭技术指标判断你的影响力，必须叠加业务规模。另外，高级岗位（P7+）的考察核心是"商业思维"——你能不能意识到自己的技术工作值多少钱、该往哪个方向优化 ROI 最高。只讲技术不讲业务，会被认为"格局不够"，卡在执行层升不上去。所以业务价值不是扯钱，是证明你有全局视角。

### 第四层：方案权衡

**Q：我是 3 年经验，按你说的该重"架构与选型"，但我项目里架构是 leader 定的、我只是执行，写架构会不会被认为吹牛？**

写"参与架构落地 + 负责其中某模块的选型决策"，不写"主导整体架构"。3 年经验的诚实定位是"在既定架构下独立完成模块设计"，不是"架构师"。你可以写"在 X 架构下，负责检索模块的选型（对比 Milvus/Pinecone 选 Milvus，考虑了 A/B/C）"，这个选型决策是你真实做的，有深度且不虚高。面试官问整体架构时，你讲清楚（说明你理解全局），但诚实说"架构是 leader 定的，我负责落地和模块选型"，这比吹"我设计的架构"更可信。虚高定位反而被一个"为什么这样设计"问穿，减分。

**Q：为什么不尽量往高了写（比如把"参与"写成"主导"），面试官又不知道真实情况？**

因为面试官会通过追问验证。写"主导架构设计"，面试官问"为什么选三层不选两层""当时考虑过什么替代方案""团队里谁反对过这个设计"，主导者能答出决策过程和争议，参与者答不出（因为没参与决策讨论）。一问就区分真假。另外背景调查（背调）会联系前同事确认职责，吹出来的会被打回。所以虚高写法的风险是"一个追问或一次背调就出局"，代价远大于诚实写法的"少一点光环"。诚信是底线，不是策略选择。

### 第五层：验证与沉淀

**Q：我怎么知道简历写得好不好，有没有效果？总不能投了没回应也不知道为啥被刷。**

用"A/B 投递 + 面试反馈"验证。准备两版简历（A 版偏职责描述、B 版偏成就量化），分别投同等规模的 20 家公司，看简历通过率（进面率）哪个高，数据说话。进面后，面试官的提问方向是简历质量的信号——如果面试官顺着你的成就深问（说明成就写得好勾起了兴趣），如果只问基础八股（说明简历没亮点只能问基础）。每次面试后记录"被问了什么"，反推简历哪条有效哪条无效，持续迭代。这是把简历当产品做 A/B 测试，不是写完就祈祷。

**Q：简历撰写能力怎么沉淀成长期能力？**

建一个"成就日志"习惯，而不是求职时才写简历。每周/每完成一个里程碑，花 10 分钟按 STAR 结构记一条（情境/动作/结果/数据），存到个人 wiki。求职时从日志里挑最亮的 3-5 条组装简历，不用临时回忆。这个习惯还有两个副产品：一是年终总结直接用（不用年底痛苦回忆做了啥），二是定期回顾日志能发现"这周好像没什么成就"——倒逼自己主动找有难度的事做而非舒适区重复。所以成就日志不只是简历素材库，是职业成长的反思工具。
