# 【字节飞连面经】项目中模型不听指令怎么办？怎么定位？反思会污染上下文吗？

## 一、典型 bad case 四类

| 类型 | 表现 | 例子 |
|------|------|------|
| 幻觉式工具调用 | 工具名拼错 | `get_user_info` → `getUserInfo` |
| 省略式作答 | 跳过工具直接给答案 | ReAct 流程里"我觉得答案是…" |
| 格式漂移 | 要求 JSON 但偶尔加 markdown | ```json ... ``` 包裹 |
| 中英混杂 | 要求纯中文但混英文术语 | "请用 React framework 实现" |

## 二、定位：必须有端到端 trace

```sql
-- 每步的 input/output/latency/cost 全落 agent_steps 表
SELECT step_id, tool_name, input_json, output_json, latency_ms, cost_usd
FROM agent_steps
WHERE trace_id = 'xxx'
ORDER BY ts;
```

**SQL 一查就知道在哪段崩**。没 trace 的项目 bad case 定位基本靠猜——这是大忌。

## 三、解决手段优先级（从便宜到贵）

```
[1] 流程约束（最便宜）
    │  在框架层写硬规则：必须先调工具才能答
    │  模型再不听话也走不通
    ▼
[2] 后处理（次便宜）
    │  JSON 解析失败 → 强制 regex 兜底
    │  格式漂移 → strip code fence
    ▼
[3] Prompt 优化
    │  加 1-3 条 few-shot 比改 system prompt 见效快
    ▼
[4] 模型参数
    │  temperature ↓（0.2）、top_p ↓
    │  稳定输出，但牺牲发散性
    ▼
[5] 微调（最贵）
       bad case 量级到几千条才值得做 SFT/DPO
```

**为什么这个顺序**：越靠前的手段成本越低、迭代越快、风险越小。微调要标数据、训模型、做评测，几周起步，且可能引入新 bad case。

## 四、评测数据：200-500 条 bad case 回归集

- 一份 **200–500 条**的人工标注 bad case 回归集
- 每次改完跑一遍，看两个指标：
  - **修复率**：原来错的现在对了多少
  - **回归率**：原来对的现在错了多少（这个最容易翻车）
- 业务侧再看 north star：工单解决率、人工兜底比例

## 五、反思会不会污染上下文？会

如果把"我刚才错了"原样塞回 context，下一轮模型容易：
- 过度自我怀疑（什么都先道歉）
- 重复道歉循环
- 被错误示例带偏

**处理方式**：
- 反思结果**摘要后再写回**（提炼成"应该先调工具"这种规则），不留原始错误文本
- 或反思放到独立 scratchpad，不进主 context

## 六、加分点

提到 **LLM-as-a-Judge** 做自动评测，并能说出 judge 模型的偏差：
- 偏好长答案（verbosity bias）
- 偏好自己生成的内容（self-preference）
- 位置偏好（A/B 测试时放前面的更易被选）

→ 要用 **swap position + 多 judge 投票** 校准。

## 七、雷区

- ❌ "改了 prompt 感觉好多了" → 没数据 = 没改
- ❌ "一上来就微调" → 成本失控，且可能引入新 bad case
- ❌ "反思直接塞回 context" → 污染

## 八、扩展

- 常见 ReAct bad case：模型把 `Observation:` 当成自己生成的内容继续编 → 用 `stop_sequence` 强制截断
- DPO 训练数据门槛：一般 5k-10k 偏好对才有显著效果，少于这个量不如调 prompt
- 评测体系三层：单元评测（单个工具调用）→ 集成评测（端到端流程）→ 业务评测（north star 指标）

## 记忆要点

- 不听指令四类雷：幻觉拼错工具名、省略工具直接作答、格式漂移加Markdown、中英文混杂输出
- 因为定位靠猜是大忌，所以每步的Input/Output必须落库，通过trace_id查SQL精准定位崩溃节点
- 解决手段看性价比：从流程约束、后处理到调Prompt和降Temperature，因为微调极贵，所以作为最后手段
- 反思必须防污染：原始报错文本绝不塞回上下文，必须摘要成规则经验或写入隔离的Scratchpad

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：模型不听指令，你为什么把"流程约束"排在第一位，而不是直接去调 Prompt？调 Prompt 不是最对症的吗？**

因为流程约束是确定性的，Prompt 优化是概率性的。"模型必须先调工具才能答"这条规则写在框架层，模型再怎么不听话也走不通——这是兜底。而调 Prompt 是降低模型出错的概率，永远会有漏网。工程上先上确定性兜底保证下限，再用概率优化提升上限。另一个原因是成本：框架层加一条 if 判断是 1 小时的活，改 Prompt + 跑 200 条回归集验证是几天，先做便宜的。

### 第二层：证据与定位

**Q：你说每次改完跑 200-500 条 bad case 回归集看"修复率 vs 回归率"。这 200 条怎么选的，怎么保证有代表性不是自选的 cherry-picking？**

样本来源必须分层抽样覆盖四类 bad case（幻觉工具调用、省略式作答、格式漂移、中英混杂），每类按线上真实发生比例抽取——不能全挑幻觉类的，否则格式漂移类的回归你根本发现不了。来源是线上 `agent_steps` 表里 `output_json` 解析失败或人工兜底的 trace，按 `trace_id` 去重后随机抽。代表性验证方法：把这 200 条跑一遍当前线上版本，看各项指标的分布是否和线上大盘一致，一致才算有代表性。

### 第三层：根因深挖

**Q：假设你调了 Prompt 加了 few-shot，200 条集上修复率 80% 看着很好，但上线后业务反馈没改善。你怎么定位？**

回归集和线上分布有 gap。常见三类：一是回归集是历史 bad case，但线上 bad case 的类型分布在变（比如新增工具后幻觉类变多，回归集里占比不够）；二是回归集是单轮评测，线上是多轮，多轮里上下文累积污染是单轮测不出来的；三是修复率 80% 但剩下的 20% 恰好是高频场景。定位方法是：拿线上最近一周的真实 trace 重新标注一批，跑回归集同样流程，看修复率是否掉得厉害。掉得多说明回归集不具代表性，要补充更新。

**Q：那如果回归集本身有代表性、修复率也真实，为什么还要怕微调？微调不是能根治吗？**

因为微调解决 A 类 bad case 的同时会引入 B 类 regression。DPO/SFT 在拟合几千条偏好对后，模型整体行为漂移——它可能在幻觉工具调用上好了 20%，但在格式漂移上回归 15%。而且微调模型一旦上线，后续迭代成本极高（每次改 Prompt 都要重训或重新评估）。更重要的是，微调需要 5k-10k 偏好对才有显著效果，标注成本几周起步。所以必须穷尽便宜手段（流程约束 + 后处理 + Prompt + 参数）确认都到瓶颈了，再动微调。

### 第四层：方案权衡

**Q：反思（Reflection）你要求"摘要后再写回，不留原始错误文本"。为什么不直接在 Prompt 里限制反思长度，比如"反思不超过 50 字"？**

Prompt 限长度是"请求"，模型不一定遵守——这正是"模型不听指令"的典型表现。而且摘要的价值不在短，在于"把错误抽象成规则"。原始报错是"我调了 getUserInfo 但传了 123，报错了"，摘要成规则是"调用用户查询工具前必须校验 user_id 是字符串"。前者塞回 context 是噪声，后者塞回 context 是可复用的经验。所以处理方式是双保险：用 Prompt 引导模型自己摘要成规则，再加一道后处理（如 LLM-as-Judge 或规则校验）确认摘要不是原文复制，确认不了就丢进独立 scratchpad 不进主 context。

**Q：那如果反思摘要后还是会污染（比如规则本身是错的），为什么不干脆关掉反思机制？**

关掉反思等于放弃模型自我纠错能力，成本更高。反思规则错误的比例通常 < 10%，而它能修复的 bad case 占比更高。正确做法是加一层校验：反思产出的规则用一个小的规则库比对，或用一个 judge 模型判断这条规则是否和已有规则矛盾/是否合理，合理的写回，不合理的丢弃。这是"带闸门的反思"——保留反思的收益，过滤它的噪声。

### 第五层：验证与沉淀

**Q：你怎么证明这次 Prompt 优化是真的有效，而不是这周业务量低、bad case 本来就少？**

做 A/B 测试而不是直接全量上线。对照组跑旧 Prompt、实验组跑新 Prompt，流量 50/50 分流，跑 1-2 周后比 north star 指标：人工兜底率、工单解决率、用户满意度。关键是要做流量归一化——把 bad case 数除以总请求数（bad_case_rate），消除业务量波动。如果实验组 bad_case_rate 显著低于对照组（比如 5% → 2.5%）且 p < 0.05，才能证明是 Prompt 效果不是流量因素。

**Q：怎么让团队以后改 Prompt 时都自觉跑回归集，而不是"感觉好多了"就上线？**

把"跑回归集"做成 CI 的强制 gate，而不是靠自觉。具体：Prompt 文件改动触发 CI 跑 200 条回归集，输出修复率和回归率报告；回归率 > 5% 或修复率 < 30% 直接 block 合并；报告里列出 regression 的具体 case 供 review。再配合上线 checklist：north star 指标的基线值和阈值，超过阈值自动告警回滚。让数据驱动长在工具链里，"感觉好多了"这种主观判断在 CI 这关就被拦住。

## 结构化回答

**30 秒电梯演讲：** 模型不听指令时，按"从便宜到贵"的优先级解：流程约束（最便宜，框架层硬规则）→ 后处理（regex/strip兜底）→ Prompt优化（加few-shot）→ 模型参数（temperature↓）→ 微调（最贵。

**展开框架：**
1. **解决优先级** — 流程约束 > 后处理 > Prompt > 模型参数 > 微调
2. **典型 bad case** — 幻觉式工具调用、省略式作答、格式漂移、中英混杂
3. **定位必须有端** — 定位必须有端到端 trace（每步 input/output/latency/cost 落表）
