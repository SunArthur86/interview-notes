# 用户一次提多个需求时，Agent的任务分解应该怎么做才不会互相污染？

## 问题场景

```
用户输入: "帮我查下明天北京的天气，顺便订一张去北京的机票，
          对了，给我把这周的会议纪要整理一下"

❌ 不做分解（同一上下文执行）：
  步骤1: 查天气 → 返回一大段天气信息
  步骤2: 订机票 → 被天气信息干扰，可能把天气参数误当航班参数
  步骤3: 整理纪要 → 上下文里全是天气和机票信息，纪要质量下降

✅ 做分解（隔离执行）：
  子任务1: 查天气 → 独立执行 → 返回结构化结果
  子任务2: 订机票 → 独立执行 → 返回结构化结果
  子任务3: 整理纪要 → 独立执行 → 返回结构化结果
  主Agent: 汇总三个结果 → 统一回复用户
```

## 分解架构

```python
class TaskDecomposer:
    """主Agent负责任务分解和结果整合"""

    async def process(self, user_input: str) -> str:
        # Step 1: 任务拆分
        subtasks = await self._decompose(user_input)

        # Step 2: 依赖分析
        execution_plan = self._plan_execution(subtasks)

        # Step 3: 执行（并行/串行）
        results = await self._execute_plan(execution_plan)

        # Step 4: 结果整合
        final_response = await self._synthesize(results, user_input)
        return final_response

    async def _decompose(self, user_input: str) -> list:
        """LLM拆分多需求为独立子任务"""
        prompt = f"""将以下用户输入拆分为独立的子任务。

规则：
1. 每个子任务必须目标明确、边界清晰
2. 标注子任务间的依赖关系
3. 每个子任务指定需要的工具

用户输入：{user_input}

输出JSON：
{{
  "subtasks": [
    {{
      "id": "task_1",
      "description": "任务描述",
      "tools": ["weather_api"],
      "depends_on": [],
      "output_schema": {{"city": "str", "temperature": "number", "condition": "str"}}
    }}
  ]
}}"""
        return parse_json(await llm_call(prompt))
```

## 隔离执行

```python
import asyncio

class IsolatedExecutor:
    """每个子任务在独立的上下文中执行"""

    async def execute_all(self, plan: dict) -> dict:
        results = {}

        # 按依赖层级分组执行
        for layer in plan['execution_layers']:
            # 同一层的任务可以并行
            tasks = []
            for subtask in layer:
                tasks.append(self._execute_isolated(subtask, results))

            layer_results = await asyncio.gather(*tasks)
            for subtask, result in zip(layer, layer_results):
                results[subtask['id']] = result

        return results

    async def _execute_isolated(self, subtask: dict, upstream_results: dict):
        """独立上下文执行——不共享对话历史"""
        # 只传入必要的上游结果（结构化数据，非原始对话）
        context = self._extract_minimal_context(subtask, upstream_results)

        # 创建独立的子Agent会话
        sub_agent = SubAgent(
            model='7b',  # 子任务用小模型
            tools=subtask['tools'],
            system_prompt=f"你是一个专注于'{subtask['description']}'的专业助手。",
        )

        result = await sub_agent.run(
            input=context,
            expected_output=subtask['output_schema'],
        )

        # 只返回结构化结果，不返回中间对话
        return {
            'task_id': subtask['id'],
            'status': 'success' if result else 'failed',
            'result': result,
        }
```

## 依赖处理

```python
def _plan_execution(self, subtasks: list) -> dict:
    """拓扑排序处理任务依赖"""
    # 构建依赖图
    layers = []
    remaining = list(subtasks)

    while remaining:
        # 找出无依赖（或依赖已满足）的任务
        ready = [t for t in remaining if all(
            dep in [l['id'] for layer in layers for l in layer]
            for dep in t.get('depends_on', [])
        )]
        if not ready:
            raise ValueError("检测到循环依赖！")

        layers.append(ready)
        remaining = [t for t in remaining if t not in ready]

    return {'execution_layers': layers}
```

## 防污染检查清单

```
✅ 上下文隔离检查：
  □ 每个子Agent使用独立的对话历史？
  □ 子任务间只通过结构化JSON传递数据？
  □ 上游结果只传"必需字段"，不传完整对话？
  □ 子Agent的System Prompt明确限定任务范围？

✅ 结果质量检查：
  □ 并行结果汇总后逻辑是否自洽？
  □ 时间/数量等跨任务数据是否一致？
  □ 子任务失败是否影响其他任务的独立性？
```

## 面试加分点

1. **任务粒度**：不是越细越好——过细的拆分会增加协调开销，粒度应以"能独立完成且有明确输出"为准
2. **Token效率**：隔离执行后每个子任务只需很小的上下文窗口（~1000 Token），可以全部用7B
3. **容错设计**：某个子任务失败不应阻塞其他任务，失败任务的输出标记为null由主Agent降级处理
4. **用户感知**：多任务并行时应有进度反馈（"正在查询天气...✅ 正在搜索机票..."），不能让用户干等

## 记忆要点

- 污染根源：多任务挤在同一上下文，历史信息干扰导致参数误传或质量降级
- 拆分原则：主 Agent 拆分为目标明确、边界清晰、带独立输出格式的子任务
- 隔离执行：各子任务启用独立上下文，仅通过上下游传递结构化参数
- 并发控制：按依赖关系分层，无依赖的同层任务可并行执行
- 结果整合：主 Agent 统一汇总各子任务的结构化输出并生成回复

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：多任务在同一上下文执行会"互相污染"。但"污染"具体指什么？LLM 不是有注意力机制，能区分不同任务的信息吗？为什么历史信息会"干扰"而非"帮助"后续任务？**

"污染"指无关信息稀释注意力和误导参数传递，即使有注意力机制仍发生。具体表现：一是注意力稀释——LLM 的注意力是有限的（softmax 权重总和为 1），上下文中信息越多，对关键信息的注意力权重被分散。任务 1（查天气）的工具返回（一大段天气 JSON）占据上下文，任务 2（订机票）时模型要关注"航班参数"，但注意力被天气信息分散，可能漏关键参数或误用天气字段（如把 temperature 当航班参数）。二是参数误传——多任务连续执行时，模型可能从前一任务的结果中提取错误参数。如任务 1 返回 {city:北京, date:明天}，任务 2（订机票）时模型可能误用 city=北京（正确）但 date=明天（机票日期，但用户可能要后天），因为日期语义在天气语境下被混淆。三是质量降级——任务 3（整理纪要）时，上下文充斥天气和机票信息，纪要任务的 prompt 被这些无关信息淹没，模型注意力偏离纪要内容，质量下降（纪要不完整或跑题）。四是 token 浪费——无关历史消耗 token（天气 JSON 可能 1000 token），挤占有效上下文窗口，纪要任务的核心输入（会议记录）被截断。LLM 的注意力帮助有限，因为注意力是"软"权重分配而非"硬"过滤，无关信息仍参与计算（只是权重低），累积影响仍显著。所以隔离执行（每任务独立上下文，只传结构化结果）是硬隔离，彻底消除污染，比依赖注意力更可靠。

### 第二层：证据与定位

**Q：你说隔离执行提升质量。但怎么量化证明"隔离 vs 不隔离"的质量差异？有哪些指标能对比？**

多维对比实验。一是任务成功率：构建多任务测试集（每含 2-5 个子任务），对比隔离 vs 不隔离的总体成功率（所有子任务都成功）。典型差异：不隔离 70-80%（污染导致部分任务失败），隔离 90-95%（独立执行质量高）。二是参数准确率：测子任务的参数传递准确性（如订机票任务是否传对日期、目的地），不隔离时参数误传率高（前任务的信息干扰），隔离时只传结构化结果准确率高。三是输出质量评分：对生成类任务（纪要、摘要）人工评分，不隔离时质量低（无关信息干扰），隔离时质量高。四是 token 效率：不隔离时上下文长（累积所有历史），token 消耗多、延迟高；隔离时每子任务上下文短（只核心输入），token 省、延迟低。五是错误归因：分析失败案例，不隔离的失败多归因于"污染"（参数误传、注意力分散），隔离的失败多归因于"单任务难度"（污染问题消除）。证明逻辑是"多任务测试集对比成功率 + 参数准确率 + 质量评分 + token 效率 + 错误归因"，隔离应在多维度占优。关键控制变量：同一测试集、同一底层模型、只改变"是否隔离"，确保差异来自隔离而非其他因素。

### 第三层：根因深挖

**Q：隔离执行要求子任务只传"结构化结果"而非"原始对话"。但结构化结果可能丢失信息（如用户的具体措辞、语气，可能对下游任务有用）。如何决定"传什么"？传多了污染，传少了丢信息，怎么平衡？**

平衡的关键是"按下游任务的输入需求传，而非全传或最少传"。一是基于 output_schema 传递——每个子任务定义 output_schema（结构化输出格式，如 {city, temperature, condition}），只传 schema 内的字段。schema 设计需考虑下游可能需要的信息（如订机票任务可能需要天气结论"是否需要外套"，而非原始温度数值）。二是下游任务声明依赖——子任务在 decompose 阶段声明 depends_on 和需要的字段（如订机票任务声明需要 task_1 的 city 和 condition），执行时只传声明字段，避免无关信息。三是保留关键上下文片段——对下游有用的非结构化信息（如用户的特殊偏好"我怕冷"），显式提取并传递（作为 structured 字段，如 {user_preference: "怕冷"}），而非传原始对话。四是"最小必要信息"原则——传足够下游决策的信息，不多不少。判断方法：问"下游任务做决策需要哪些信息？"列出来，只传这些。五是信息损失的权衡——如果丢失的信息（用户措辞）对下游决策无影响（订机票不需要措辞），丢弃无妨；如果可能影响（如用户语气暗示 urgency），显式提取为字段传递。实践：decompose 时设计 output_schema（覆盖下游需求）+ 下游声明依赖字段 + 执行时按字段传，形成"契约式"的信息传递，既不污染也不丢关键信息。这是工程化的信息流设计，非临时判断。

**Q：任务分解由 LLM 做（prompt 让它拆分）。但 LLM 可能拆错（拆得过细、过粗、漏依赖）。拆分质量直接影响后续执行，怎么保证分解的正确性？**

LLM 拆分的质量需校验和兜底。一是拆分规范（prompt 约束）：在 prompt 中明确拆分规则——"每个子任务必须目标明确、边界清晰、可独立完成""标注依赖关系""指定工具和输出 schema"，规则约束 LLM 的拆分行为，减少过细/过粗。二是粒度指导：prompt 中给粒度示例（如"查天气+订机票+写纪要"拆 3 个，而非把"订机票"再拆成"查航班+选座+支付"过细），示例引导合理粒度。三是拆分结果校验：LLM 输出拆分后，程序校验结构合法性（无循环依赖、schema 合法、工具存在），结构错误反馈 LLM 重新拆分。四是单意图直通：简单单意图（如只"查天气"）不拆分，直接执行，避免过度拆分。判断单意图（意图识别置信度高 + 无多需求标志）可跳过 decompose。五是拆分质量监控：线上收集拆分案例，人工审核（拆分是否合理、依赖是否正确、粒度是否合适），统计拆分错误率，针对性优化 prompt 或补充示例。六是关键场景人工兜底：对复杂或高风险任务的拆分（如涉及支付的复合任务），可选人工确认拆分（如客服场景，拆分后人工审核再执行）。证明逻辑是"prompt 约束 + 示例引导 + 程序校验 + 单意图直通 + 监控优化 + 关键场景兜底"，多层保证拆分质量。LLM 拆分不是黑盒，是可校验、可监控、可兜底的工程流程。

### 第四层：方案权衡

**Q：你说无依赖任务并行执行（如查天气+订机票+写纪要同时跑）。但并行会增加资源占用（3 个子 Agent 同时调 API、占 GPU）。并发太高会不会导致资源耗尽或限流？怎么控制并发度？**

并行确实增加资源压力，需控制并发度。资源风险：一是 GPU 显存——多个子 Agent 同时推理，显存占用叠加（7B 推理约 14GB，3 个并行 42GB，可能超单卡）。二是 API 限流——外部 API（天气、航班）有 QPS 限制，高并发触发限流（429 错误），任务失败。三是延迟——高并发导致 GPU 调度排队，单个任务延迟反而增加（资源争抢）。控制并发的策略：一是并发度上限——设全局并发限制（如 max_concurrency=5），超出排队等待，避免资源耗尽。上限根据资源定（GPU 显存、API QPS）。二是优先级调度——关键任务（支付、安全）高优先级优先执行，非关键（查询）低优先级排队，资源紧张时优先保证关键。三是批量大小控制——GPU 推理的 batch size 限制（如同时处理 8 个请求），超出排队，避免 OOM。四是 API 限流适配——对外部 API 的调用加 rate limiter（如每秒最多 10 次），避免触发限流；限流时自动退避重试（指数退避）。五是资源监控——监控 GPU 利用率、显存、API 错误率，资源紧张时动态降并发（如利用率 > 90% 降到 max=3），资源空闲时升并发（max=10）。六是串行降级——并发导致大量失败时，降级为串行（牺牲延迟保稳定）。实践配置：起步保守（max=3），监控资源利用和成功率，逐步调优。并发控制是"资源上限 + 优先级 + rate limit + 动态调整"的系统工程，避免并行反而导致不稳定。

**Q：结果整合时，多个子任务的结构化结果要汇总成连贯回复。但如果结果冲突（如天气任务说"明天北京下雨"，机票任务说"明天飞北京晴天"——时间或地点矛盾），主 Agent 怎么处理？**

冲突检测和仲裁是结果整合的关键。冲突类型：一是数据矛盾——如天气任务的 date=明天 与机票任务的 date=后天 不一致（用户可能改主意或 LLM 解析错）。二是逻辑矛盾——如订机票任务订了"北京"，但用户的行程是"上海"（某任务误解目的地）。三是时序矛盾——任务 A 的结果依赖任务 B，但 B 的结果与 A 假设不符。处理策略：一是冲突检测——整合前主 Agent 校验跨任务一致性（对比时间、地点、数量等关键字段），发现矛盾标记。二是优先级仲裁——按任务可信度或时序仲裁，如后执行的任务（反映用户最新意图）覆盖前者，或权威数据源（API 返回 > LLM 推断）优先。三是用户确认——关键矛盾（如目的地冲突）不自动仲裁，而是向用户确认（"您是要订北京的机票还是上海的？"），避免猜错。四是降级处理——矛盾无法自动解决时，主 Agent 诚实汇报（"查询结果有矛盾：天气显示北京下雨，但机票显示晴天，请您确认"），而非强行整合。五是根源追溯——矛盾常源于分解时的理解错误（某子任务误解用户意图），追溯根源并重新分解该任务。证明逻辑是"整合前一致性校验 → 检测矛盾 → 按优先级仲裁或用户确认 → 无法解决则诚实汇报"，不强行掩盖矛盾。冲突处理体现 Agent 的"智能"——不是机械整合，是理解结果间的关系并合理处理。

### 第五层：验证与沉淀

**Q：任务分解和隔离执行的系统上线后，怎么持续验证"分解合理 + 隔离有效"？不是一次性测试，而是长期监控？**

长期监控体系。一是分解质量监控：抽样线上对话，人工审核分解是否合理（拆分粒度、依赖标注、工具指定），统计分解错误率（如 < 5%），异常时优化 prompt。二是隔离效果监控：对比单上下文 vs 隔离的任务成功率和质量评分（A/B 或历史对比），隔离应持续优于单上下文。如果优势缩小（如隔离成功率从 95% 降到 90%），排查（是否某些任务的隔离实现退化了）。三是用户行为信号：多任务的完成率、用户重试率、转人工率。完成率下降或重试率上升可能反映分解或执行问题。四是错误案例库：收集失败案例（分解错、隔离泄漏、整合矛盾），分类错误模式，针对性优化（prompt、schema、冲突处理逻辑）。五是性能监控：多任务的延迟（分解 + 并行执行 + 整合的总时间）、资源占用（并发度、GPU）、成本。延迟或成本异常时优化（调整并发、简化分解）。六是分布漂移检测：用户需求模式变化（新的多任务组合）时，监控分解是否适应，不适应则更新分解 prompt 或 schema。证明逻辑是"分解质量监控 + 隔离效果 A/B + 用户行为 + 错误案例库 + 性能监控 + 漂移检测"，长期跟踪多维度指标，确保系统持续有效。多任务系统不是一次上线就稳定，是持续监控和优化的动态系统。

**Q：怎么让团队在设计多任务 Agent 时不各自实现（有人串行、有人并行、有人部分隔离），而是统一的分解-隔离-整合架构？**

沉淀统一架构规范和设计模式。一是架构规范：明确"分解器 + 隔离执行池 + 整合器"的三层架构，所有多任务场景遵循，避免各自实现。二是设计模式库：沉淀常见模式——独立任务并行（无依赖）、依赖任务串行（拓扑序）、冲突仲裁（优先级/用户确认）、失败降级（跳过/重试），新人按模式设计。三是数据结构规范：统一的子任务 schema（id/description/tools/depends_on/output_schema）、结果 schema（task_id/status/result），跨业务复用。四是工具规范：统一的子 Agent 接口（输入 context + output schema，输出结构化结果），隔离执行池统一调度。五是评估规范：统一的多任务测试集格式和评估指标（分解准确率、任务成功率、整合质量），跨业务可比。六是踩坑库：常见错误（单上下文污染、拆分过细、依赖漏标注、整合矛盾未处理）及案例。让多任务 Agent 是"统一架构 + 设计模式 + 规范 schema + 评估体系"的系统工程，而非各团队重复造轮子。

## 结构化回答

**30 秒电梯演讲：** 用户一次提多个需求时，Agent需要先拆分成独立子任务，每个子任务在隔离上下文中执行，通过结构化中间结果传递信息，避免互相污染。

**展开框架：**
1. **任务拆分** — 将多需求拆成独立子任务，明确边界
2. **上下文隔离** — 每个子任务独立会话，不共享中间状态
3. **结构化传递** — 通过JSON/结构化数据传递结果，不传原始对话
