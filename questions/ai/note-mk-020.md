# 【月之暗面面经】AI-Native 桌面产品从 Demo 到商用，前端最容易在哪些点翻车？

## 一、四大翻车点全景

```
┌──────────────────────────────────────────────────────────────────┐
│                Demo → 商用 四大翻车点                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   翻车率                                                         │
│    ▲                                                            │
│    │     ┌──────┐                                              │
│    │     │ 权限  │ ← 翻车点1: 本地权限边界不清                   │
│    │     │ 边界  │   用户不敢授权→产品无法使用                    │
│    │     └──────┘                                              │
│    │     ┌──────┐                                              │
│    │     │ 长任务 │ ← 翻车点2: 长任务回看不足                    │
│    │     │ 回看  │   任务中断/结果丢失/无法定位问题               │
│    │     └──────┘                                              │
│    │     ┌──────┐                                              │
│    │     │ 产物  │ ← 翻车点3: 产物落地链路不完善                  │
│    │     │ 落地  │   用户频繁重跑/格式错乱/无法编辑                │
│    │     └──────┘                                              │
│    │     ┌──────┐                                              │
│    │     │ 并发  │ ← 翻车点4: 多窗口/多任务状态错位               │
│    │     │ 状态  │   数据不一致/编辑冲突/状态丢失                 │
│    │     └──────┘                                              │
│    │                                                            │
│    └──────────────────────────────────→ 时间                     │
│         Demo阶段        商用阶段                                  │
│         (都不翻车)      (集中爆发)                                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、翻车点详解

### 翻车点1: 本地权限边界不清

**Demo表现**：开发者自己测试，所有权限都给了，一切正常。

**商用翻车**：
```
用户第一次使用 → 弹出"访问所有文件"权限请求 → 用户害怕 → 拒绝 → 产品无法使用

或者：
AI直接覆盖了用户的重要文件 → 用户数据丢失 → 差评/投诉

或者：
AI索引了 node_modules/.env 等敏感文件 → 上传到云端 → 隐私泄露
```

**根因**：Demo不涉及真实用户的安全顾虑，商用时权限是用户信任的第一道关卡。

**修复方向**：分层授权 + 写入二次确认 + 授权可撤销 + 敏感文件自动跳过。

### 翻车点2: 长任务回看不足

**Demo表现**：任务运行30秒就完成了，用户一直在等待。

**商用翻车**：
```
任务运行5分钟 → 用户切去做别的事 → 回来发现：
  • 任务失败了，但不知道在哪一步失败的
  • 任务完成了，但结果被另一个操作覆盖了
  • 任务部分完成了，但无法从断点继续
  • 任务"完成"了，但结果是空的或不完整
```

**根因**：Demo的短任务掩盖了长任务的可观测性需求。

**修复方向**：分段状态机 + traceId + 失败断点续跑 + 执行日志 + 回放系统。

### 翻车点3: 产物落地链路不完善

**Demo表现**：AI生成的文本内容直接显示在页面上，看起来很好。

**商用翻车**：
```
用户要导出PPT → 发现导出格式错乱
用户要编辑产物 → 发现只能看不能改
用户要修改一部分 → 发现只能整单重跑
用户要回退版本 → 发现没有版本历史
用户的文件被覆盖 → 发现无法恢复
```

**根因**：Demo只关注"AI能不能生成"，商用关注"用户能不能用这个产物继续工作"。

**修复方向**：草稿→确认→发布三步链路 + 局部重生成 + 版本管理 + 多格式导出 + Diff预览。

### 翻车点4: 多窗口/多任务状态错位

**Demo表现**：单窗口单任务，一切正常。

**商用翻车**：
```
用户开了3个窗口：
  窗口A：正在生成站点（长任务）
  窗口B：同时在编辑另一个产物
  窗口C：查看历史任务

  → 窗口A的任务完成通知，窗口B和C没收到
  → 窗口B的编辑保存覆盖了窗口A的产物
  → 窗口C看到的任务状态是过时的
  → 切换窗口时输入框内容和选中态丢失
```

**根因**：Demo的单窗口环境掩盖了多窗口状态同步的复杂性。

**修复方向**：事实同步/UI不同步 + 编辑锁 + 全局事件推送 + 窗口级状态隔离。

## 三、翻车的连锁反应

```
权限边界不清 → 用户不信任 → 减少使用 → DAU下降
     ↓
长任务回看不足 → 任务失败率高 → 用户流失
     ↓
产物落地不完善 → 用户频繁重跑 → 效率低下 → 失去竞争力
     ↓
并发状态错位 → 数据丢失/不一致 → 差评/投诉

最终结果：产品有很好的Demo效果，但商用后用户留存极低
```

## 五、常见坑

- **Demo先行，异常后补**：先做Demo拿融资，商用时发现一堆问题来不及修
- **忽视权限设计**：觉得"用户肯定会授权的"，结果用户第一秒就拒绝了
- **没有版本管理**：AI直接覆盖文件，出了事无法回退
- **不做多窗口测试**：只在单窗口测试，商用后多窗口问题集中爆发

## 记忆要点

- Demo到商用四大翻车点：权限边界、长任务回看、产物落地、并发状态
- 权限边界：因为全开全关易致隐私泄露，所以需分层授权与二次确认
- 长任务回看：因为短任务掩盖了中断风险，所以需状态机与断点续跑支持
- 产物落地：因为只读预览无法满足修改诉求，所以需提供版本历史与编辑链路
- 并发状态：因为多窗口容易数据错位，所以需引入冲突仲裁保障一致性
