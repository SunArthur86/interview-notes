# 【华为面经】AI DC 数据基础设施/数据湖在大模型训练中的角色

## 一、数据湖的基本概念

### 1.1 什么是数据湖

**数据湖（Data Lake）** 是一个集中式存储系统，以**原始格式**保存海量结构化和非结构化数据，支持**schema-on-read**（读取时才定义结构）。

```
传统数据仓库（Schema-on-Write）：
原始数据 → [ETL清洗转换] → 严格Schema表 → SQL查询
                ↑ 提前定义结构，不符合规则的数据被丢弃

数据湖（Schema-on-Read）：
原始数据 → [原样存储] → 需要时按需定义结构读取
                ↑ 先存再算，灵活应对未知的未来用途
```

### 1.2 数据湖在大模型时代的演进

传统数据湖主要处理**结构化业务数据**（CSV、Parquet、关系表）。大模型时代，数据湖的核心变化：

| 维度 | 传统数据湖 | 大模型数据湖 |
|------|-----------|-------------|
| **数据类型** | 结构化为主 | 多模态：文本、图片、视频、音频、代码 |
| **数据量** | TB级 | PB级（GPT-4训练用13万亿token） |
| **数据格式** | Parquet、ORC | JSONL、WebDataset、TFRecord、原始二进制 |
| **访问模式** | 批量SQL查询 | 高吞吐流式读取、随机采样 |
| **核心指标** | 查询延迟 | 数据质量、去重率、毒性过滤 |
| **治理重点** | 数据血缘、权限 | 数据来源、版权、PII脱敏、可复现性 |

大模型数据湖的存储架构：

```
物理层（对象存储 OBS/S3）
├── raw/                    # 原始数据区（Bronze层）
│   ├── web/2026-06/       # 按日期分区

├── 表结构定义
├── 数据血缘（哪个表从哪个表加工来）
├── 版本快照（可回溯任意历史版本）
```

#### 层3：处理层（Processing Pipeline）

```python
# 大模型数据预处理pipeline（运行在数据湖之上的Spark/Ray集群）
def pretrain_pipeline(raw_path, version="v3.2"):
    # 1. 语言识别 & 过滤
    lang_ok = filter_language(load(raw_path), langs=["zh", "en"])

    # 2. 质量过滤（perplexity + 启发式规则）
    quality_ok = filter_quality(lang_ok, min_score=0.7)

    # 3. 去重（MinHash LSH，模糊去重 + 精确去重）

    # 4. 安全过滤（毒性、PII脱敏）
    safe = filter_toxicity(mask_pii(deduped))

    # 5. Tokenize & 打包
    packaged = tokenize_and_pack(safe, tokenizer=tokenizer)

    # 写入curated区，带版本
    write_to_lake(packaged, f"s3://pangu-lake/curated/pretrain_{version}/")
    # 记录数据血缘：curated/pretrain_v3.2 ← processed/safe ← raw/web
```

#### 层4：交付层（Serving）

```python
# 训练集群从数据湖拉取数据
class DataLoader:
    def __init__(self, lake_path, version="v3.2"):
        # 从元数据层获取该版本所有文件列表
        self.files = lake.list_files(f"curated/pretrain_{version}/")

    def stream(self, batch_size=2048):
        """流式读取，避免一次性加载PB级数据"""
        for shard in shuffle(self.files):  # 跨文件shuffle
            for record in read_shard(shard):
                yield record
                if count % batch_size == 0:
                    yield flush_batch()
```

## 雷区

- **混淆数据湖和数据仓库**：数据湖存原始数据（schema-on-read），数据仓库存结构化数据（schema-on-write），不是一回事
- **忽视数据质量**：只关注存储容量，忽略"Garbage In Garbage Out"——再大的湖，装满脏水也训不出好模型
- **版权和合规盲区**：大模型数据涉及大量版权内容（书籍、网页、代码），数据湖必须有版权标记和合规审计能力

### 第一层：目标与动机

**Q：数据湖的核心卖点是"schema-on-read"（先存原始数据，读时再定结构）。但大模型训练数据最终都要 tokenize 成统一格式喂模型，schema-on-read 的灵活性在这个场景下有什么实际价值？为什么不一开始就按训练格式 ETL 好（schema-on-write）？**

价值在于"同一份原始数据服务于多个未来用途"，而训练格式会变。同一批网页语料，今天可能用 BPE tokenizer 训 LLaMA 架构，明天换 sentencepiece 训盘古，后天要做去重实验用不同阈值——如果一开始就 tokenize 成某格式存，换格式时要重新从原始数据 ETL（PB 级数据重处理成本极高）。schema-on-read 让原始数据原样存，每次按需读取和转换，灵活应对"格式/方法/参数变化"。另一个价值是"可复现"——原始数据存着，半年后发现训练效果异常，能精确恢复当时的原始数据重新处理验证，而只存 tokenized 的无法回溯（不知道当时的清洗参数）。训练格式是"下游"，原始数据是"上游"，上游保灵活、下游按需固化，这是数据工程的基本原则。

### 第二层：证据与定位

**Q：你说数据湖支撑"PB 级数据 + 高吞吐训练读取"。但如果训练集群要从对象存储（S3/OBS）流式读 PB 数据，网络 IO 容易成瓶颈。你怎么确认训练时数据加载不是瓶颈（GPU 不在等数据）？**

看 GPU 利用率和数据读取吞吐的对比。一是监控 GPU 利用率——如果利用率长期 < 80% 且数据加载队列频繁空（DataLoader 报 waiting），说明数据供给跟不上，GPU 在等。二是测数据读取吞吐——统计 DataLoader 每秒读取的 token 数或样本数，对比训练消耗速度（batch_size × steps/sec × tokens_per_sample），读取应 >= 消耗的 1.5 倍（留 prefetch 余量）。三是定位瓶颈层——对象存储的出口带宽、网络交换机吞吐、数据预处理（tokenize/过滤）的 CPU 算力，任一层卡住都会传导到 GPU 等待。常见优化：数据预 tokenize 存成二进制（WebDataset/TFRecord）减少读取后处理；本地 SSD 缓存热点 shard（避免每次从远端读）；多 worker 并行预取。证明逻辑是"GPU 利用率 + 读取/消耗比 + 各层吞吐定位"，数字说话。

### 第三层：根因深挖

**Q：数据质量你强调"Garbage In Garbage Out"。但大模型训练数据来自 Common Crawl 等开放数据，里面大量低质量/重复/有害内容。你怎么定义"质量"，又怎么在大规模（PB 级）下高效过滤？**

质量的定义是多维的：语言正确性（非乱码）、信息密度（非占位符/广告/导航栏）、重复度（近似重复的只留一份）、安全性（无毒/无偏见/无 PII）、领域相关性（对行业模型要过滤无关内容）。大规模过滤用"流水线 + 启发式 + 模型打分"组合：启发式规则（长度、符号比例、语言检测）快速过滤明显垃圾，成本低；perplexity 打分（用一个轻量语言模型给文本打分，分低的是乱码或低质量）过滤语义垃圾，中等成本；MinHash/LSH 去近似重复（精确去重大贵，近似去重够用）；毒性分类器过滤有害内容。关键是流水线顺序——便宜的先做（先过滤再打分，减少打分的数据量），且每层过滤率要监控（某层过滤率突增可能是规则误杀）。大规模下"精确"不现实，用"多层近似过滤"平衡效果和成本。

**Q：那如果过滤规则把高质量但少见的内容误删了（如方言、古文、专业术语），模型能力会下降。怎么平衡"过滤垃圾"和"保留多样性"？**

这是数据工程的核心矛盾。解法是"分桶 + 差异化过滤"。把数据按来源/领域分桶（网页、书籍、论文、代码、对话），每桶用不同过滤阈值——网页桶严格过滤（垃圾多），书籍/论文桶宽松（本身质量高），代码桶按语法正确性过滤而非 perplexity。多样性保护：对少见内容（方言、古文、小语种）单独成桶，用更宽松的规则或人工抽检确认保留。另一个机制是"过滤可回溯"——每条被过滤的数据记录过滤原因（哪个规则/哪个阈值），定期抽样审查过滤的是不是真垃圾，误杀率高的规则调整。A/B 验证：用不同过滤策略生成两个数据集，训同尺寸小模型看下游 benchmark，如果过严过滤导致某些能力（如小语种理解）下降，说明误杀了多样性，要放宽。核心是"过滤不是一刀切，是分桶差异化 + 可回溯 + A/B 验证"。

### 第四层：方案权衡

**Q：数据湖的 Bronze/Silver/Gold（raw/processed/curated）三层你提了。但每层都存一份数据，PB 级下存储成本翻 3 倍。为什么不只存最终层（Gold），省中间层的存储？**

因为中间层是"可复现和可调试"的关键。只存 Gold 的问题：一是无法调试——某次训练效果差，怀疑是去重或过滤的问题，但没有 Silver 层就无法回溯"去重前/去重后"的差异来定位。二是无法迭代——调整过滤阈值要重新从 raw 跑全流程，PB 级数据处理几小时到几天，频繁迭代极慢；有 Silver 层可以从中间层重跑下游（如只重跑 tokenize）。三是版本对比困难——想对比 v3.1 和 v3.2 的差异，中间层让差异可追溯（是去重阈值变了还是过滤规则变了）。存储成本控制靠"分层存储策略"——Gold（热，NVMe）小且贵但高频访问；Silver（温，OBS 标准）中频；Bronze（冷，OBS 归档）低成本长期存。归档存储成本是标准存的 1/10，3 份分层总成本可能只比单份 Gold 贵 50%，但换来的可复现和迭代效率价值远超成本。

**Q：Lakehouse（Iceberg/Delta/Hudi）你说是"数据湖 + 数据仓库的融合"。但大模型训练数据大多是 JSONL/WebDataset 等非结构化格式，Lakehouse 的 ACID 事务和 SQL 查询能力对它有什么用？**

Lakehouse 对大模型数据的价值不是"SQL 查询文本内容"，而是"元数据治理"。大模型数据集的元数据（文件列表、版本、统计信息、血缘、质量评分）是结构化的，适合用 Lakehouse 的表管理。具体场景：版本管理——Iceberg/Delta 的 time travel 让"v3.2 数据集包含哪些文件"可查询可回溯，比裸对象存储的版本管理强；数据血缘——"Gold 层的某文件由 Silver 层的哪些文件加工来"可 SQL 查询，方便调试；质量审计——每条数据的过滤状态、毒性评分作为列存，能 SQL 统计"v3.2 的去重率/毒性过滤率"；Schema 演进——新增字段（如质量评分）不重写数据。所以 Lakehouse 管"元数据层"，原始数据（文本/图片）仍以文件存在对象存储，Lakehouse 提供"结构化的元数据视图"。这是"非结构化数据 + 结构化治理"的分层，不是要求文本也进关系表。

### 第五层：验证与沉淀

**Q：你怎么证明数据湖这套基础设施（而非数据本身）对大模型训练效率有提升？基础设施的价值怎么量化？**

量化基础设施的价值看"数据交付效率"和"迭代效率"。数据交付效率：从"新数据进湖"到"训练可用"的时长——没有数据湖时（手动 ETL + 拷贝）可能要 1-2 周，有数据湖的 pipeline 自动化后 1-2 天，提升 5-10 倍。迭代效率：数据科学家调整过滤参数后重新生成数据集的时长——没有版本化和中间层时每次从头跑要几天，有 Silver 层只重跑下游几小时，提升 5 倍。可复现性：发现问题后恢复历史训练数据的时长——无数据湖要翻日志重 ETL 可能不可复现，有版本化精确恢复 1 小时内。成本：分层存储让 PB 级数据存储成本降 10 倍（归档），算总 TCO。证明逻辑是"对比有无数据湖时的交付/迭代/复现/成本四指标"，用"节省的时间和成本"说话，这是基础设施的 ROI。

**Q：怎么让团队的数据工程实践（分层、版本化、质量过滤）保持一致，而不是各项目自己搭 pipeline？**

把数据工程能力平台化。一是统一数据平台：团队共用一套数据湖（raw/processed/curated 分层）和 pipeline 框架（标准化的过滤/去重/tokenize 模块），各项目按规范接入而非自建。二是数据规范：文档化"L0/L1/L2 分层标准""质量过滤 pipeline 的标准步骤""版本命名规范"，新数据集必须按规范产出。三是数据 review：新数据集上线前过数据治理 review（分层是否合规、血缘是否记录、质量评分是否达标、PII 是否脱敏）。四是平台能力：封装"数据集管理 SDK"（create_version/get_lineage/quality_report），让版本化和血缘是 SDK 自动提供的而非手动维护。让数据工程是"平台能力"而非"项目技能"，规范一致性靠平台保障。

## 结构化回答

**30 秒电梯演讲：** 数据湖是大模型训练的"原料仓库+预处理工厂"——以原始格式集中存储海量多模态数据，提供统一的访问、清洗、版本管理能力，支撑从原始数据到训练就绪数据集的全链路。

**展开框架：**
1. **数据湖存储原** — 数据湖存储原始格式数据（schema-on-read），区别于数据仓库（schema-on-write）
2. **大模型数据湖** — 大模型数据湖需要支撑多模态（文本/图片/视频/音频）+ 多格式（parquet/jsonl/webdataset）
3. **核心能力** — 海量存储、统一元数据、数据版本、数据血缘、质量评估
