# RAG 系统的核心组件及工作逻辑？

## 一、五大核心组件

```
┌──────────────────────────────────────────────────┐
│                RAG Pipeline 五组件                  │
├──────────────────────────────────────────────────┤
│                                                    │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 1.索引器  │ →  │2.查询处理 │ →  │ 3.检索器  │   │
│  │ Indexer  │    │Query Proc│    │Retriever │   │
│  └──────────┘    └──────────┘    └────┬─────┘   │
│  文档→向量索引    查询优化          检索文档      │
│                                       │          │
│                                       ▼          │
│                 ┌──────────┐    ┌──────────┐   │
│                 │ 5.生成器  │ ←  │4.后处理器 │   │
│                 │Generator │    │Post-Proc │   │
│                 └──────────┘    └──────────┘   │
│                 生成答案        重排/过滤       │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、各组件详解

### 组件 1：索引器（Indexer）

```
职责：把原始文档转成可检索的向量索引

流程：
  文档加载 → 清洗 → 分块 → 向量化 → 存储

输入：PDF/Word/网页/DB
输出：向量数据库中的索引

关键决策：
  - 分块策略（固定大小/语义分块/递归分块）
  - chunk_size（200-1000字符）
  - overlap（10-20%重叠）
  - Embedding模型选择
```

### 组件 2：查询处理器（Query Processor）

```
职责：优化用户查询，提升检索效果

功能：
  - 查询改写：口语→检索友好的关键词
    "那个做AI的公司" → "OpenAI 人工智能公司"
  
  - 查询扩展：加同义词/相关词
    "Agent" → "Agent + 智能体 + autonomous agent"
  
  - HyDE：先让LLM生成假设答案，用答案检索
    （假设答案的语义更接近文档）
  
  - 子问题分解：复杂问题拆分
```

### 组件 3：检索器（Retriever）

```
职责：从索引中找到最相关的文档

方法：
  - 稠密检索（向量相似度）：语义匹配，主力
  - 稀疏检索（BM25）：关键词匹配，补充
  - 混合检索：向量+BM25融合，效果最好

参数：
  - top_k：返回多少个文档（通常5-20）
  - 相似度阈值：低于阈值的过滤掉
```

### 组件 4：后处理器（Post-Processor）

```
职责：对检索结果精排和过滤

功能：
  - Rerank：用Cross-Encoder重排序（比向量相似度更准）
  - 去重：相似文档合并
  - 过滤：按metadata过滤（时间/来源/权限）
  - 压缩：长文档提取关键句
  - 上下文扩展：找到相邻块补充上下文
```

### 组件 5：生成器（Generator）

```
职责：基于检索文档生成答案

输入：系统Prompt + 检索文档 + 用户问题
输出：答案（+引用来源）

关键：
  - Prompt模板设计
  - 防幻觉（"基于文档回答，不知道就说不知道"）
  - 引用标注（标注信息来自哪个文档）
```

## 三、工作逻辑（完整数据流）

```python
class RAGPipeline:
    def __init__(self):
        self.indexer = Indexer()
        self.query_processor = QueryProcessor()
        self.retriever = Retriever()
        self.post_processor = PostProcessor()
        self.generator = Generator()
    
    def answer(self, question):
        # 1. 查询处理
        processed_query = self.query_processor.process(question)
        
        # 2. 检索
        candidates = self.retriever.retrieve(processed_query, top_k=20)
        
        # 3. 后处理（重排+过滤）
        refined = self.post_processor.rerank_and_filter(
            question, candidates, top_k=5
        )
        
        # 4. 生成
        answer = self.generator.generate(question, refined)
        
        return {
            "answer": answer,
            "sources": [doc.metadata["source"] for doc in refined]
        }
```

## 五、各组件优化对效果的影响

```
效果提升贡献度（经验值）：

检索质量(组件2+3+4): 60%  ← 最关键
  - 查询处理: 15%
  - 检索方法: 25%
  - 后处理:   20%

  - LLM能力: 10%
```

## 六、面试加分点

1. **五组件清晰**：索引→查询处理→检索→后处理→生成，每环节职责明确
2. **强调检索是核心**：60%的效果由检索决定，生成只是锦上添花
3. **模块化可替换**：每个组件可独立优化/升级，体现工程架构思维

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RAG 核心组件是"索引器+检索器+生成器"，再加查询处理器和后处理器，为什么查询处理和后处理也要算核心？光有前三者不够吗？**

因为前三者只覆盖"理想流程"，处理处理器和后处理管"真实世界的噪声"。1）查询处理器——用户 query 往往不完美（口语化/有歧义/多轮指代/含错别字），直接检索效果差，查询处理器（改写/扩展/分解）把 query 优化成"适合检索的形式"，提升召回；2）后处理器——检索结果不完美（有噪声/冗余/重复/相关度低的），直接给 LLM 浪费 context 且干扰生成，后处理器（rerank/去重/过滤/压缩）把结果优化成"适合生成的形式"，提升质量。没有查询处理，召回差（garbage in）；没有后处理，生成差（noise to LLM）。所以五者构成完整 pipeline，缺查询处理和后处理会显著降效果。

### 第二层：证据与定位

**Q：RAG pipeline 各组件都跑了但答案还是不好，怎么定位是哪个组件的问题（索引/检索/查询处理/后处理/生成）？**

逐组件 trace。1）索引——检查文档是否正确索引（解析对不对/分块合理吗/embedding 生成正常吗），错则索引问题；2）查询处理——原始 query vs 处理后 query，对比召回结果，处理后的召回应更好，如果没变甚至变差，查询处理无效或有害；3）检索——看检索 top-K 是否包含 golden（人工标注的相关文档），不包含是召回问题；4）后处理——检索结果经过后处理（rerank/过滤）后，golden 是否还在（被误过滤则后处理有问题）；5）生成——检索结果含答案但 LLM 输出错，是生成问题（prompt/LLM）。定位方法：在每环节打 trace（输入/输出），找第一个"好输入坏输出"的组件。常见根因：索引（分块太碎/解析丢信息）、查询处理（改写过度引入噪声）、检索（embedding 不准）、后处理（rerank 模型差把 golden 排掉）、生成（prompt 没强调基于检索）。

### 第三层：根因深挖

**Q：索引器的"分块"策略对 RAG 效果影响大，固定块 vs 语义块 vs 结构化块，怎么选？**

按文档类型选。1）固定块（按字符/token 数切）——适合"无结构的纯文本"（如文章），简单快速，但可能切断语义（切在句子/段落中间）；2）语义块（按句子/段落/标题切）——适合"有段落结构的文档"（如 wiki/博客），保留语义完整性，质量好但实现稍复杂（要解析结构）；3）结构化块（按文档层级切，如"章-节-段"）——适合"强结构文档"（如手册/法规），块带层级元数据（所属章节），检索时可利用结构（如命中某章节优先返回该章节相关块）。选型：无结构文本用固定块（简单），段落文档用语义块（质量），强结构文档用结构化块（精准）。实务：默认语义块（兼顾质量和通用性），特殊文档用结构化块。

**Q：查询处理器要做"query 改写"，但改写错了（改写后的 query 比原 query 检索效果更差）怎么办？**

改写要"保守 + 验证"。1）保守改写——改写只是扩展/规范化（如加同义词/纠错/去停用词），不改变原意，避免过度改写引入噪声；2）双路检索——原 query 和改写 query 都检索，结果融合（不放弃原 query，改写只是补充召回），即使改写差原 query 兜底；3）改写验证——改写后对比召回（改写后 Recall 是否提升），提升才用改写，下降则回退原 query；4）改写质量监控——统计改写 query 的召回贡献（改写带来的新相关文档数），低贡献的改写规则优化或弃用。进阶：用 LLM 改写时让 LLM 输出"改写理由"（可解释），便于排查坏改写。原则：改写是增强不是替代，原 query 兜底，改写质量持续监控。

### 第四层：方案权衡

**Q：检索器可以用向量检索（快但粗）或 rerank 模型（准但慢），为什么不只用 rerank（一步到位准）？**

因为效率差异大。1）向量检索——从百万级文档库检索 top-K（如 K=100），毫秒级（ANN 索引如 HNSW），快但粗（embedding 相似度，有噪声）；2）rerank 模型——对 top-K 精排（cross-encoder，更准但慢，每个 query-文档对都要 forward），百毫秒级。只用 rerank——要对全库（百万文档）每个都 rerank，太慢（百万次 forward）不可行。所以实际是"向量检索粗筛（top-100）→ rerank 精排（top-5）"两阶段，兼顾速度和精度。选型：小库（<1000 文档）可直接 rerank（快），大库必须两阶段（向量先筛）。实务：默认两阶段，除非库小才单 rerank。

**Q：生成器要基于检索结果生成，但检索结果可能很长（多个文档拼一起），全塞给 LLM 费 token 且 attention 稀释，怎么压缩？**

三招压缩。1）截断——只取 top-K（如 top-5）的文档，每个文档截断到一定长度（如前 500 token），控制总量；2）抽取式压缩——从每个文档抽取"和 query 最相关的句子"（用相似度/LLM 判断），只保留相关句，大幅压缩；3）生成式压缩——用小 LLM 把每个文档"针对 query 总结"成简短摘要（如"该文档关于 X，要点是 Y"），只给 LLM 摘要。选型：简单用截断（快但可能丢信息），中等用抽取（保留相关句），极致压缩用生成（质量高但额外 LLM 调用）。原则：保证"压缩后的信息足以回答 query"，否则压缩过度导致答案不全。

### 第五层：验证与沉淀

**Q：你怎么衡量 RAG 各组件（查询处理/检索/后处理/生成）各自的贡献（哪个该优化）？**

组件级归因。1）消融实验——逐个关闭组件（如关查询处理用原 query、关 rerank 用向量 top-K、关后处理），对比答案准确率，关掉后准确率掉得多的组件贡献大；2）组件指标——查询处理（改写后 Recall 提升多少）、检索（Recall@K/MRR）、后处理（rerank 后 golden 排名提升、过滤掉的无噪声比例）、生成（给定检索结果，LLM 的答案准确率）；3）归因 bad case——每个错答案归因到组件（如"召回没找到"→检索/查询处理、"找到但没用"→生成），统计各组件的失败占比，占比大的优先优化。这样能定位"哪个组件是瓶颈"，而非盲目优化全部。

**Q：RAG 的五组件 pipeline 怎么沉淀成可复用的框架？**

建 RAG 框架：1）组件抽象——定义每个组件的标准接口（索引器：文档→索引；检索器：query→top-K；查询处理器：query→优化 query；后处理器：top-K→精排 top-K；生成器：query+top-K→答案），组件可插拔（换 embedding/rerank 模型不改其他）；2）pipeline 编排——配置化组装组件（如 YAML 定义 pipeline：查询处理=改写+扩展，检索=向量+BM25 混合，后处理=rerank+过滤），灵活组合；3）组件库——内置多种组件实现（不同 embedding/rerank/LLM），按场景选；4）评估集成——pipeline 自动跑评估集，组件级指标+端到端指标。这套写入团队 RAG 框架 SOP，让"搭 RAG"从"每次写代码"变成"配置组装"，快速搭建+灵活调优。
