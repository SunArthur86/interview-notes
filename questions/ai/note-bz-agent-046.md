# RAG 系统的核心组件及工作逻辑？

## 一、五大核心组件

```
┌──────────────────────────────────────────────────┐
│                RAG Pipeline 五组件                  │
├──────────────────────────────────────────────────┤
│                                                    │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 1.索引器  │ →  │2.查询处理 │ →  │ 3.检索器  │   │
│  │ Indexer  │    │Query Proc│    │Retriever │   │
│  └──────────┘    └──────────┘    └────┬─────┘   │
│  文档→向量索引    查询优化          检索文档      │
│                                       │          │
│                                       ▼          │
│                 ┌──────────┐    ┌──────────┐   │
│                 │ 5.生成器  │ ←  │4.后处理器 │   │
│                 │Generator │    │Post-Proc │   │
│                 └──────────┘    └──────────┘   │
│                 生成答案        重排/过滤       │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、各组件详解

### 组件 1：索引器（Indexer）

```
职责：把原始文档转成可检索的向量索引

流程：
  文档加载 → 清洗 → 分块 → 向量化 → 存储

输入：PDF/Word/网页/DB
输出：向量数据库中的索引

关键决策：
  - 分块策略（固定大小/语义分块/递归分块）
  - chunk_size（200-1000字符）
  - overlap（10-20%重叠）
  - Embedding模型选择
```

### 组件 2：查询处理器（Query Processor）

```
职责：优化用户查询，提升检索效果

功能：
  - 查询改写：口语→检索友好的关键词
    "那个做AI的公司" → "OpenAI 人工智能公司"
  
  - 查询扩展：加同义词/相关词
    "Agent" → "Agent + 智能体 + autonomous agent"
  
  - HyDE：先让LLM生成假设答案，用答案检索
    （假设答案的语义更接近文档）
  
  - 子问题分解：复杂问题拆分
```

### 组件 3：检索器（Retriever）

```
职责：从索引中找到最相关的文档

方法：
  - 稠密检索（向量相似度）：语义匹配，主力
  - 稀疏检索（BM25）：关键词匹配，补充
  - 混合检索：向量+BM25融合，效果最好

参数：
  - top_k：返回多少个文档（通常5-20）
  - 相似度阈值：低于阈值的过滤掉
```

### 组件 4：后处理器（Post-Processor）

```
职责：对检索结果精排和过滤

功能：
  - Rerank：用Cross-Encoder重排序（比向量相似度更准）
  - 去重：相似文档合并
  - 过滤：按metadata过滤（时间/来源/权限）
  - 压缩：长文档提取关键句
  - 上下文扩展：找到相邻块补充上下文
```

### 组件 5：生成器（Generator）

```
职责：基于检索文档生成答案

输入：系统Prompt + 检索文档 + 用户问题
输出：答案（+引用来源）

关键：
  - Prompt模板设计
  - 防幻觉（"基于文档回答，不知道就说不知道"）
  - 引用标注（标注信息来自哪个文档）
```

## 三、工作逻辑（完整数据流）

```python
class RAGPipeline:
    def __init__(self):
        self.indexer = Indexer()
        self.query_processor = QueryProcessor()
        self.retriever = Retriever()
        self.post_processor = PostProcessor()
        self.generator = Generator()
    
    def answer(self, question):
        # 1. 查询处理
        processed_query = self.query_processor.process(question)
        
        # 2. 检索
        candidates = self.retriever.retrieve(processed_query, top_k=20)
        
        # 3. 后处理（重排+过滤）
        refined = self.post_processor.rerank_and_filter(
            question, candidates, top_k=5
        )
        
        # 4. 生成
        answer = self.generator.generate(question, refined)
        
        return {
            "answer": answer,
            "sources": [doc.metadata["source"] for doc in refined]
        }
```

## 四、组件选型

```
┌────────────┬─────────────────────┬────────────────────┐
│ 组件        │ 选项                   │ 选型考虑            │
├────────────┼─────────────────────┼────────────────────┤
│ 分块        │ 固定/语义/递归         │ 文档类型            │
│ Embedding  │ OpenAI/BGE/本地模型   │ 语言/精度/成本      │
│ 向量库      │ Chroma/Milvus/Pinecone│ 规模/是否托管       │
│ 检索        │ 向量/BM25/混合        │ 精度要求            │
│ Rerank     │ Cross-Encoder/LLM    │ 延迟预算            │
│ 生成LLM    │ GPT-4/Claude/本地     │ 质量/成本/合规      │
└────────────┴─────────────────────┴────────────────────┘
```

## 五、各组件优化对效果的影响

```
效果提升贡献度（经验值）：

检索质量(组件2+3+4): 60%  ← 最关键
  - 查询处理: 15%
  - 检索方法: 25%
  - 后处理:   20%

生成质量(组件5): 25%
  - Prompt设计: 15%
  - LLM能力: 10%

索引质量(组件1): 15%
  - 分块策略: 10%
  - Embedding: 5%

结论：检索是RAG的核心，投入优化检索的ROI最高
```

## 六、面试加分点

1. **五组件清晰**：索引→查询处理→检索→后处理→生成，每环节职责明确
2. **强调检索是核心**：60%的效果由检索决定，生成只是锦上添花
3. **模块化可替换**：每个组件可独立优化/升级，体现工程架构思维
