# Agentic RAG 和普通 Naive RAG 的区别？什么时候需要 Agentic RAG？

## 架构对比

```
┌─────────────── Naive RAG ───────────────┐
│                                         │
│  Query ──► Embed ──► Search ──► LLM     │
│                            │             │
│                        Top-K Docs       │
│                                         │
│  特点：单次检索，不管结果好坏直接生成     │
└─────────────────────────────────────────┘

┌─────────────── Agentic RAG ─────────────┐
│                                         │
│  Query ──► Agent 决策中心               │
│             │                           │
│     ┌───────┼───────┐                   │
│     ▼       ▼       ▼                   │
│  Query    Multi   Rerank                │
│  Rewrite  Query   Eval                  │
│     │       │       │                   │
│     └───────┼───────┘                   │
│             ▼                           │
│         检索结果评估                     │
│      ┌────┴────┐                        │
│    够好      不够好                      │
│      │         │                        │
│   生成回答   改写query重试               │
│                                         │
│  特点：Agent主导，支持多轮检索和迭代     │
└─────────────────────────────────────────┘
```

## 核心区别

| 维度 | Naive RAG | Agentic RAG |
|------|-----------|-------------|
| **检索次数** | 1次 | 多次(按需) |
| **查询处理** | 原始query直接embedding | Query改写、分解、扩展 |
| **结果评估** | 无，直接拼接 | Agent评估相关性，不满意可重检 |
| **路由决策** | 固定向量库 | 可路由到不同知识源/API/工具 |
| **多跳推理** | 不支持 | 支持(Step 1结果指导Step 2检索) |
| **延迟** | 低(1次LLM+1次检索) | 高(多轮交互) |
| **准确率** | 简单问题好，复杂问题差 | 复杂问题显著提升 |
| **成本** | 低 | 高(多次LLM调用) |

## 什么时候需要 Agentic RAG

### ✅ 需要 Agentic RAG 的场景

1. **多跳推理**： "比XX公司市值高的AI公司有哪些？"
   - Step 1：检索XX公司市值 → 500亿
   - Step 2：检索AI公司市值列表 → 筛选>500亿的

2. **复杂条件查询**： "找2024年后发表的、引用次数>100的、关于RAG优化的论文"
   - Agent需要分解条件，可能先检索论文再过滤

3. **需要交叉验证**： 医疗/法律场景，需要多个来源确认

4. **动态知识源**： 先查内部知识库，不够再搜外部

### ❌ Naive RAG 就够的场景

1. **简单FAQ**： "退货政策是什么？"
2. **单文档问答**： "这篇合同的第3条说了什么？"
3. **延迟敏感**： 实时客服、搜索建议(<500ms)

## Agentic RAG 的实现模式

### 模式1：ReAct + RAG

```python
# Agent使用ReAct范式，将检索作为工具
tools = [
    Tool(name="vector_search", func=vector_db.search,
         description="搜索内部知识库"),
    Tool(name="web_search", func=web_search,
         description="搜索互联网"),
    Tool(name="sql_query", func=sql_db.execute,
         description="查询结构化数据"),
]

agent = ReActAgent(llm=llm, tools=tools)
# Agent会自己决定：何时搜索、搜什么、要不要再搜一次
```

### 模式2：CRAG (Corrective RAG)

```
Query → 检索 → 评估检索质量
                   │
          ┌────────┼────────┐
          ▼        ▼        ▼
        Correct  Ambiguous Wrong
          │        │        │
       直接生成  Web搜索   改写query
                          重新检索
```

### 模式3：Self-RAG (自反思RAG)

```
LLM在生成过程中自问：
- "需要检索吗？" → 需要 → 检索
- "检索结果相关吗？" → 不相关 → 重检
- "生成的内容有依据吗？" → 无依据 → 修正
```

## 生产级落地建议

- **模型路由**：简单query走Naive RAG(小模型)，复杂query走Agentic RAG(大模型)
- **缓存**：Semantic Cache缓存相似query的结果，避免重复检索
- **最大检索轮数**：限制3-5轮，防止Agent无限循环
- **超时控制**：总延迟控制在5s以内，超过则降级为Naive RAG结果

## 记忆要点

- 普通RAG单次检索直接生成，Agentic RAG以Agent为主导支持多轮迭代检索
- Agentic RAG核心能力：Query改写、结果评估、多跳推理、动态路由
- 因为Agentic RAG多次调用LLM导致延迟高成本高，所以简单FAQ和低延迟场景用Naive RAG即可
- 适用Agentic RAG：多跳推理、复杂条件查询、需交叉验证的医疗/法律场景

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agentic RAG 你说支持"多轮迭代检索"。Naive RAG（单次检索）为什么不够？什么场景必须多轮？**

Naive RAG 的单次检索无法处理"多跳推理"和"信息不足时的补充检索"。多跳推理场景：如"现任 CEO 的母校在哪个城市"，需要先检索"公司 CEO 是谁"（第一跳），再检索"这个人的母校"（第二跳），再检索"母校的城市"（第三跳）。Naive RAG 用原 query 一次检索，召回的 chunk 可能只有"公司 CEO"信息（没有"母校城市"），答案不完整。Agentic RAG 让 Agent 分步检索——第一轮检索 CEO 信息，基于结果生成新 query（"X 的母校"），第二轮检索母校，基于结果再检索城市，多轮迭代直到信息完整。信息不足场景：如第一轮检索结果不够回答，Agent 判断"信息不足，需补充"，改写 query 再检索。这些场景 Naive RAG 一次检索做不到，必须多轮。

### 第二层：证据与定位

**Q：Agentic RAG 的回答质量比 Naive RAG 差（某些简单问题反而答错）。怎么定位是 Agent 决策错（不该多轮的强行多轮）、还是检索质量差？**

看 Agent 的检索 trace。一是判断"是否过度检索"——对于简单问题（如"公司地址"），Agent 是否只检索一轮就回答（正确），还是强行多轮检索（过度，引入噪声）。如果简单问题被多轮检索（如查地址后又查公司历史、业务，把无关信息塞进 prompt），是 Agent 的"停止判断"逻辑差（该停没停）。二是看每轮检索的质量——多轮检索中每轮召回的 chunk 是否相关且增量（每轮有新信息），如果是"重复召回相似 chunk"或"召回无关 chunk"，是检索质量问题（embedding 差或 query 改写差）。三是看最终答案——Agent 综合多轮结果生成答案时，是否正确整合（而非被无关信息干扰）。治法：过度检索优化"停止判断"（Agent 评估"已有信息是否足够"，够了就停）；检索质量优化 embedding 或 query 改写。关键监控"平均检索轮数"——简单问题应 1 轮，复杂问题 2-3 轮，如果所有问题都多轮，是停止判断失效。

### 第三层：根因深挖

**Q：Agentic RAG 的"Query 改写"你说是核心能力。为什么需要改写？原 query 直接检索不行吗？**

原 query 常常"不适合检索"。场景一：多跳推理的原 query 是最终问题（如"CEO 母校的城市"），直接检索时 embedding 匹配的是"城市"相关的 chunk，而非"CEO 是谁"的中间信息，召回错方向。改写成中间 query（"公司 CEO 是谁"）才能召回正确信息。场景二：原 query 口语化或模糊（如"那个东西怎么用"），embedding 和文档（书面语）的语义匹配差，改写成清晰 query（"产品 X 的使用说明"）提升召回。场景三：原 query 有指代（如"他"，指代上文的人），检索时缺上下文，改写时补充指代（"张三的使用说明"）。Query 改写是 Agent 的"检索策略优化"——根据当前状态（已有信息 + 缺失信息）生成最适合检索的 query。Naive RAG 用原 query（不优化），对简单 query 够用，复杂 query 召回差。Agentic RAG 的改写是"主动优化检索"，提升召回质量。

**Q：那为什么不直接用"原 query + 多个变体"并行检索（而非 Agent 串行改写），并行更快？**

并行检索快但"变体是静态的"，Agent 改写是"动态的"。并行检索（如用 LLM 生成 5 个 query 变体，并行检索，结果融合）是"HyDE"或"multi-query"策略，快（一次生成 + 并行检索）。但变体是"一次性生成的"（基于原 query，不基于中间结果），无法适应"多跳推理"（第二跳的 query 依赖第一跳的结果）。如"CEO 母校城市"，并行生成的 5 个变体都是围绕"CEO 母居城市"（不知道 CEO 是谁），召回的都是泛泛的"公司城市"信息，无法找到具体 CEO 的母校。Agent 串行改写是"动态的"——第一轮检索 CEO（基于原 query 改写），第二轮检索母校（基于第一轮结果"CEO=张三"改写），每轮基于前轮结果优化 query。串行慢但精准（多跳推理必须串行）。折中：单跳问题用并行（快），多跳问题用 Agent 串行（准）。

### 第四层：方案权衡

**Q：Agentic RAG 延迟高（多轮 LLM 调用，3-5 秒）。你怎么权衡延迟和质量？什么场景值得高延迟？**

看"问题的复杂度"和"用户的延迟容忍度"。简单 FAQ（如"退货政策"）——Naive RAG 够用（单次检索，1 秒内），用户期望快答，Agentic RAG 的多轮（3-5 秒）反而损害体验。复杂研究（如"对比 A 和 B 产品的优劣"）——Naive RAG 可能召回不全（需要多次检索不同产品的信息），Agentic RAG 多轮检索更完整，用户愿意等（复杂问题用户预期慢）。医疗/法律（需交叉验证）——Agentic RAG 的多轮检索 + 结果评估能交叉验证信息（如"这个诊断是否一致于多个权威来源"），准确性重要，延迟可接受。决策方法：用"问题分类器"（或 Agent 自判断）——简单问题走 Naive RAG（快），复杂问题走 Agentic RAG（准）。或"渐进式"——先用 Naive RAG 快速回答，如果置信度低（信息不足）再升级到 Agentic RAG 深度检索（用延迟换质量，只在必要时）。

**Q：为什么不直接用最强模型 + 长上下文（把所有可能相关的文档塞进 prompt），让模型自己推理多跳，省得搞 Agentic RAG 的多轮？**

长上下文塞所有文档不现实且质量差。一是"文档量太大"——一个知识库可能有几万到几百万文档，全塞进 prompt 不可能（即使 200k 窗口也只够几百个 chunk）。二是"lost in the middle"——即使塞进去，模型对超长上下文的中间信息注意力弱（前面讨论过），质量降。三是"成本爆炸"——每次请求传几十万 token，API 费用极高。Agentic RAG 的多轮检索是"精准获取"——每轮只检索最相关的几个 chunk（小上下文，模型高效处理），多轮累积所需信息。相比"全塞进去"，Agentic RAG 是"按需检索"（只取需要的），成本和效率更优。且 Agentic RAG 的"结果评估"能过滤低质量 chunk（只保留高相关的），进一步精简上下文。长上下文模型适合"单次长输入"（如分析一篇报告），Agentic RAG 适合"从海量知识中检索推理"。

### 第五层：验证与沉淀

**Q：你怎么衡量 Agentic RAG 相比 Naive RAG 的提升，证明高延迟值得？**

定义指标：一是答案准确率（在多跳推理 golden set 上对比，Agentic RAG 应 >Naive RAG，如 85% vs 70%）；二是信息完整性（答案是否覆盖所有要点，Agentic RAG 多轮检索应更完整）；三是延迟（Agentic RAG 的 P99 应 <5 秒，Naive RAG <1 秒）；四是成本（Agentic RAG 的 token 消耗应 <Naive RAG 的 5 倍，否则不划算）。做对比实验：Naive RAG vs Agentic RAG，在"简单 FAQ"和"复杂多跳"两类测试集上分别对比。预期：简单 FAQ 上两者准确率接近（Agentic RAG 不该过度检索），复杂多跳上 Agentic RAG 显著优。关键验证"分场景路由"——简单走 Naive RAG（快）、复杂走 Agentic RAG（准），综合体验最优（简单问题不慢、复杂问题准）。A/B 测试线上效果——Naive RAG vs 分场景路由 vs 全 Agentic RAG，看用户满意度和成本。

**Q：Agentic RAG 方案怎么沉淀成 RAG 系统标配？**

固化成"自适应 RAG 框架"：问题分类器（简单/复杂）路由到 Naive RAG 或 Agentic RAG、Agentic RAG 的核心循环（检索 → 评估 → 改写 → 再检索，直到信息足够）、Query 改写的 prompt 模板、停止判断逻辑（信息足够时停止检索）。沉淀"各场景的路由策略"（FAQ 走 Naive、研究走 Agentic、医疗走 Agentic + 交叉验证）、"检索轮数控制"（简单 1 轮、复杂 3-5 轮、最多 10 轮）、"成本控制"（每次检索的 token 预算）。配套监控（答案准确率、平均检索轮数、延迟、成本），异常（准确率降/轮数暴涨）告警。把"分场景路由 + Agentic 核心循环"作为 RAG 系统的标配能力，新场景按规范配置路由和检索策略。积累"多跳推理的 query 改写案例库"，帮助优化改写 prompt。

## 结构化回答

**30 秒电梯演讲：** Naive RAG是一次性检索+生成；Agentic RAG让Agent主动决策检索策略，支持多轮检索、查询改写、结果评估和迭代优化。

**展开框架：**
1. **Naive RAG** — 单次检索→拼接→生成，简单但脆弱
2. **Agentic RAG** — Agent控制检索循环，支持查询改写/多路检索/结果评估/迭代
3. **何时需要** — 多跳推理、复杂条件、需要交叉验证的场景
