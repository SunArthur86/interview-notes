# 什么是 AI Agent？它和单纯的大模型（LLM）有什么区别和关系？

## 一、核心定义

**AI Agent（智能体）** = 以 LLM 为大脑，具备**感知、规划、记忆、工具使用**能力，能够**自主完成多步骤任务**的系统。

**单纯的大模型（LLM）** = 一个文本到文本的映射函数，输入 prompt 输出文本，**无状态、无记忆、无行动能力**。

## 二、五维度对比

```
┌────────────┬─────────────────────┬──────────────────────────────┐
│ 维度        │ LLM                  │ Agent                          │
├────────────┼─────────────────────┼──────────────────────────────┤
│ 主动性      │ 被动应答（你问它答）  │ 主动达成目标（给目标自己干）     │
│ 状态        │ 无状态（每次重新开始） │ 有状态（跨轮/跨会话记忆）        │
│ 行动        │ 只输出文本            │ 能调用工具产生真实效果           │
│ 控制流      │ 单次调用              │ 循环（规划→行动→观察→调整）      │
│ 任务粒度    │ 单步问答              │ 多步骤复杂任务                   │
└────────────┴─────────────────────┴──────────────────────────────┘
```

## 三、Agent 的三大核心能力

### 1. 规划（Planning）

```python
# LLM：给一步做一步
response = llm("帮我订一张去北京的机票")  # 只会输出文字建议

# Agent：自主分解任务
def agent_plan(goal):
    # LLM把大目标拆成可执行步骤
    steps = llm.plan("订去北京机票", context=memory.get())
    # 输出：1.查日程 2.比价 3.选航班 4.下单 5.通知
    for step in steps:
        result = execute(step)  # 调用真实工具
        if result.failed:
            steps = llm.replan(step, result.error)  # 失败重规划
```

### 2. 记忆（Memory）

```
短期记忆：当前对话上下文（Context Window）
长期记忆：向量数据库存储历史（跨会话）
工作记忆：当前任务的中间状态（任务进度、已收集信息）
```

### 3. 工具使用（Tool Use）

```python
# Agent能调用真实工具改变世界状态
tools = [
    {"name": "search_flight", "desc": "搜索航班"},
    {"name": "book_ticket", "desc": "下单订票"},
    {"name": "send_email", "desc": "发送通知"},
]
# LLM决定用哪个工具、传什么参数
action = llm.decide(context, tools)  # → book_ticket(flight=CA123)
```

## 四、关系：Agent 是 LLM 的"上层应用"

```
┌─────────────────────────────────────┐
│              Agent 系统               │
│  ┌─────────────────────────────┐    │
│  │      控制循环 (Loop)           │    │
│  │   while not goal_achieved:    │    │
│  │     plan → act → observe      │    │
│  │  ┌──────────────────────┐    │    │
│  │  │   LLM (决策大脑)      │ ← 记忆│    │
│  │  │   "下一步该做什么？"    │    │    │
│  │  └──────────┬───────────┘    │    │
│  │             ↓ 决策              │    │
│  │      [工具调用/工具调用]         │    │
│  │  search() book() email()        │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

**关键关系：**
- LLM 是 Agent 的**必要组件**（决策核心），但 LLM ≠ Agent
- Agent 是 LLM 的**增强包装**：加了循环 + 记忆 + 工具
- 同一个 LLM，配不同的工具和提示词，能构建完全不同的 Agent

## 五、一个直观例子

**任务：帮我分析竞品并写一份报告**

| 步骤 | LLM 单独 | Agent |
|------|---------|-------|
| 1 | 无法上网，只能凭训练数据编 | 调用搜索工具爬取最新竞品信息 |
| 2 | 一次性生成全文（可能过时/幻觉） | 分步骤：搜集→分析→起草→自检→定稿 |
| 3 | 出错无法纠正 | 写完自检发现错误，重新搜集修正 |
| 4 | 无状态 | 记住你的偏好风格，下次沿用 |

## 六、面试加分点

1. **强调"自主性"**：Agent 的核心是 autonomy——给定目标后自主完成，而非逐步指令
2. **区分 Agentic 程度**：Workflow（固定流程）→ Agent（动态决策）→ Autonomous Agent（完全自主），不是二元的
3. **提 Anthropic 的定义**：Augmented LLM + Tools + Loop，简洁权威

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你定义 Agent = LLM + 控制循环 + 外部能力，那如果一个应用只用 LLM 做单轮问答（如 ChatGPT 首版），它算 Agent 吗？边界在哪？**

不算。Agent 的核心判别是"自主决策 + 闭环行动"。单轮问答是"输入→输出"的开环，没有控制循环（不基于行动反馈决定下一步），没有外部能力调用（不调工具/不改环境）。边界在三点：1）是否有控制循环——Agent 会 Observe 行动结果再 Decide 下一步，单轮问答没有；2）是否调用外部工具——Agent 会 search/calc/API call，纯 LLM 只用权重里的知识；3）是否有目标驱动的多步——Agent 围绕目标拆解执行，单轮是一次性响应。ChatGPT 加了 Function Call + 多轮工具调用后才具备 Agent 特征。

### 第二层：证据与定位

**Q：你说 Agent 比纯 LLM 强，但 Agent 底层还是 LLM，怎么证明"加了循环和工具"真的带来增益，而不是增加复杂度还更差？**

在固定任务集上对比三个版本：1）纯 LLM 单轮（直接 prompt 问）；2）LLM + 单轮工具（给工具但只调一次）；3）完整 Agent（LLM+循环+多步工具）。指标：任务完成率、准确率、幻觉率。典型结果：复杂任务（多跳问答、代码调试）Agent 完成率比纯 LLM 高 30-50%，幻觉率低 40%（工具补充准确信息）；但简单任务（1+1）三者持平甚至 Agent 更慢（循环开销）。增益的来源是"工具补充知识 + 循环纠正错误"，这两个能力在纯 LLM 单轮里没有。如果 Agent 在所有任务上都不如纯 LLM，说明循环设计有问题（如过度调用工具）。

### 第三层：根因深挖

**Q：Agent 的"控制循环"具体怎么实现？是 while True 一直转吗？怎么决定停？**

控制循环本质是"状态机 + 终止条件"。while 循环每轮：Observe（看输入和上轮结果）→ Decide（LLM 决定下一步 Thought/Action）→ Act（执行工具）→ 把结果拼回 context。终止条件三类：1）模型输出 `<final_answer>` 标记——任务完成；2）达到 max_turns（如 20 轮）——防死循环；3）死循环检测——连续 N 步信息增益为 0（重复调用同工具同参数）。不能纯 while True，必须配终止条件，否则模型卡在"我再想想"的循环里烧 token。实务用 LangGraph 的图引擎实现——节点是动作，边是条件流转，自然支持终止节点和循环上限。

**Q：既然 Agent 这么强，为什么不所有 LLM 应用都做成 Agent？纯 LLM 单轮还有什么不可替代的场景？**

不可替代的场景：1）超低延迟场景（如实时翻译、输入法补全）——Agent 的循环+工具调用延迟 1-5 秒，单轮 LLM 流式输出首 token 200ms，体验完全不同；2）成本敏感场景——Agent 多轮调用 LLM + 工具 API，单次成本是单轮的 5-10 倍；3）确定性场景（如格式化输出、模板填充）——单轮 prompt 就能解决，Agent 的决策是过度设计；4）创造性生成（如写诗、续写故事）——不需要工具补充事实，纯 LLM 创造力更强。所以选型看任务：需要外部信息/多步推理用 Agent，纯生成/低延迟/低成本用单轮 LLM。

### 第四层：方案权衡

**Q：Agent 的"规划、记忆、工具"三大能力，哪个最难做？你在实际项目里花时间最多的是哪个？**

记忆最难，花时间最多。规划相对成熟（CoT/ReAct/Plan-Execute 框架 + LLM 推理能力够用），工具调用有标准接口（Function Call/MCP）。记忆的难点在于"什么时候记、记什么、怎么检索"——1）写入决策：不是所有对话都该存（存太多检索噪声大），要有"重要性评分"机制（如 LLM 给每条信息打分，高分才存）；2）检索相关性：向量检索的召回率在长程记忆里只有 60-70%，要配 rerank + 时间衰减 + 任务相关性加权；3）遗忘机制：记忆库无限增长会拖慢检索，要有 TTL 或容量上限 + 重要性淘汰。这三点每个都是独立的工程难题，规划一次调好 prompt 就行，记忆要持续迭代。

**Q：为什么不把 Agent 的记忆做成"全量保存 + 全文检索"，简单粗暴，不就不丢信息了？**

因为检索质量会崩。全量保存后，每次检索要在海量历史里找相关的，向量检索的精度随库增大而下降（候选多了，相似但无关的噪声也多），top-5 的命中率从 95% 降到 60%。而且全文检索（BM25）只匹配关键词，语义相关但用词不同的记忆召回不到。更糟的是延迟——检索几千条记忆要几百 ms，Agent 每轮都检索会严重拖慢。所以记忆要"精筛"：只存高价值信息（LLM 评分 + 用户反馈信号），配语义检索（embedding）+ rerank（cross-encoder）+ 时间衰减，把记忆库控制在高质量的小集合，检索又快又准。

### 第五层：验证与沉淀

**Q：你怎么衡量一个 Agent 比纯 LLM "更像智能体"，有没有量化指标？**

四个维度量化：1）自主性——用户给定目标后，Agent 不需要人工干预能完成的比例（autonomy_ratio = 无干预完成任务数/总任务数）；2）工具使用合理性——tool_call_success_rate（调用正确的工具+参数正确的比例），过度调用率（简单任务也调工具的比例）；3）多步推理能力——平均完成步数 / 最优步数（接近 1 说明规划高效，远大于 1 说明绕路）；4）错误恢复能力——故意注入工具失败，看 Agent 能否换方案恢复（recovery_rate）。一个"更像 Agent"的系统：autonomy_ratio>80%、tool_call_success_rate>85%、步数比<1.5、recovery_rate>70%。这套指标写入 Agent 评测集，每次迭代跑一遍对比。

**Q：Agent 的核心定义和能力边界怎么沉淀成团队共识，避免每个人对"Agent"理解不同？**

定一份 Agent 设计规范：1）定义——Agent = LLM + 控制循环 + 工具 + 记忆 + 目标，五个要素缺一不可，缺循环叫 Copilot（辅助），缺工具叫 Chatbot（聊天），缺目标叫 Demo（玩具）；2）能力矩阵——按自主性/工具数/记忆深度把 Agent 分级（L1 单工具 Copilot / L2 多工具 Agent / L3 多 Agent 协作），每级配技术栈和适用场景；3）反模式——列出"伪 Agent"特征（无终止循环、工具狂魔、记忆不检索），review 时识别。这套规范写入团队 Agent 工程手册，新人入职必读，新项目立项时按分级选型，让"Agent"从模糊概念变成可对齐的技术定义。
