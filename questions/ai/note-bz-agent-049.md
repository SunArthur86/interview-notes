# 如何提升 RAG 的检索召回率？（全链路优化）

## 一、全链路优化框架

```
┌──────────────────────────────────────────────────────┐
│              召回率全链路优化                            │
├──────────────────────────────────────────────────────┤
│                                                        │
│  索引层：分块优化 + Embedding选型 + 元数据              │
│    → 让文档更好地被"表示"                               │
│                                                        │
│  查询层：改写 + 扩展 + HyDE + 分解                      │
│    → 让查询更好地"匹配"文档                             │
│                                                        │
│  检索层：混合检索 + 多路召回 + 参数调优                  │
│    → 用多种方法"广撒网"                                 │
│                                                        │
│  后处理：Rerank + 上下文扩展                            │
│    → "精选"最相关的                                     │
│                                                        │
└──────────────────────────────────────────────────────┘
```

## 三、查询层优化

```python
class QueryOptimizer:
    def optimize(self, query):
        queries = [query]  # 原始
        
        # 1. 改写
        queries.append(self.rewrite(query))
        
        # 2. HyDE（假设答案）
        hyde = self.llm.generate(f"假设答案: {query}")
        queries.append(hyde)
        
        # 3. 多角度变体
        queries.extend(self.llm.variants(query, n=3))
        
        # 4. 子问题分解
        if self.is_complex(query):
            queries.extend(self.decompose(query))
        
        return queries  # 一个查询变成多个
```

## 四、检索层优化（核心）

### 混合检索

```python
class HybridRetriever:
    def retrieve(self, query, top_k=20):
        # 向量检索（语义）
        dense = self.vector_db.search(embed(query), k=top_k)
        
        # BM25（关键词，擅长专有名词/代码/数字）
        sparse = self.bm25.search(query, k=top_k)
        
        # 融合
        fused = self.rrf_merge(dense, sparse)
        return fused[:top_k]
```

### 多路召回

```python
def multi_route_retrieve(query, top_k=20):
    results = []
    
    # 路径1：原始查询 + 向量检索
    results += vector_db.search(query, k=10)
    
    # 路径2：改写查询 + 向量检索
    rewritten = rewrite(query)
    results += vector_db.search(rewritten, k=10)
    
    # 路径3：BM25关键词
    results += bm25.search(query, k=10)
    
    # 路径4：元数据过滤后检索（如限定文档类型）
    results += vector_db.search(query, filter={"type": "faq"}, k=5)
    
    # 去重合并
    return dedup(results)[:top_k]
```

## 六、召回率评估

```python
def evaluate_recall(test_cases):
    """
    test_cases: [{query, relevant_doc_ids}, ...]
    """
    total_recall = 0
    for case in test_cases:
        retrieved = retrieve(case["query"], k=20)
        retrieved_ids = {doc.id for doc in retrieved}
        relevant_ids = set(case["relevant_doc_ids"])
        
        # recall = 检索到的相关文档 / 所有相关文档
        recall = len(retrieved_ids & relevant_ids) / len(relevant_ids)
        total_recall += recall
    
    return total_recall / len(test_cases)

# 目标：recall@20 > 85%
```

## 八、面试加分点

1. **"广撒网再精选"**：召回阶段多召回（宁滥勿缺），排序阶段精排（宁缺勿滥）
2. **混合检索是关键**：向量+BM25 互补，能覆盖单一方法的盲区
3. **量化评估**：用 recall@k 指标衡量，而非凭感觉

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：提升 RAG 召回率的核心思想是"先广撒网再精选"，为什么不直接"精准撒网"（一次检索就准）？**

因为"一次精准"在真实场景几乎不可能。1）query 和文档的语义鸿沟——用户 query 多样（口语化/歧义/简短），文档表述固定，一次检索（无论向量还是 BM25）都难完全对齐，必然有漏召回；2）广撒网提升召回上限——多路召回（向量+BM25+不同改写 query），每路覆盖不同语义空间，并集召回远大于单路（如单路 60%，多路并集 85%）；3）精选保证精度——召回多了（广撒网）但 top-K 质量下降（噪声多），用 rerank 精选（把最相关的排到 top-K），兼顾召回和精度。所以"广撒网"是为了"召回全"（上限高），"精选"是为了"精度准"（top-K 质量好），两步配合才能"又全又准"。

### 第二层：证据与定位

**Q：RAG 召回率（Recall@K）只有 60%，怎么定位是"撒网不够广"（召回策略少）还是"网眼太大"（检索精度差，相关文档排不进 top-K）？**

分两步诊断。1）扩大 K 看趋势——Recall@5=60%，看 Recall@20/Recall@50，如果 Recall@20=85%（K 扩大召回显著升），说明相关文档被检索到了但排名靠后（网眼大，精度差），要优化排序（rerank）；如果 Recall@20=65%（K 扩大召回几乎不变），说明相关文档根本没被检索到（撒网不够广），要优化召回策略（多路/改写）；2）看漏召回 case——分析漏掉的文档为什么没检索到（query 和文档表述不一致→要 query 改写；文档没索引→索引问题；embedding 不匹配→换/微调 embedding）。定位方法：Recall@K 随 K 的变化趋势 + 漏召回 case 归因，区分"召回策略"还是"排序精度"问题。

### 第三层：根因深挖

**Q：多路召回（向量+BM25+多改写 query）能提升召回，但多路结果有大量重复/冗余，怎么去重和融合？**

去重 + 融合。1）去重——相同文档（同一文档 ID）只保留一次（多路都可能召回同一文档），按文档 ID 去重；2）分数融合——不同路的分数尺度不同，用 RRF（倒数排名融合，看排名不看分数）融合，每路召回的文档按排名贡献分数（rank 1 贡献 1/2、rank 2 贡献 1/3...），总和为融合分数，按融合分数排序；3） rerank 精排——融合后取 top-N（如 20）给 rerank 精排，选 top-K（如 5），rerank 基于语义精排（cross-encoder），质量比 RRF 更高。流程：多路召回→去重→RRF 融合→rerank 精排→top-K。这样既召回全（多路）又精度好（融合+rerank），且去重避免冗余。

**Q：查询改写（如多 query 改写）能让 LLM 生成多个 query 变体提升召回，但改写多了成本高（多次检索+LLM 调用），怎么平衡？**

按 query 复杂度动态改写。1）简单 query 不改写——明确的 query（如"X 产品的价格"）直接检索即可（改写无收益），用规则判断 query 是否明确（如含具体实体/短），明确的不改写；2）复杂 query 多改写——模糊/复杂的 query（如"如何提升业务"）改写成多个变体（如"业务增长方法""提升收入策略"等），多路检索扩召回；3）改写数量——一般 3-5 个变体（多了边际递减且成本高），配合 RRF 融合；4）异步改写——对延迟敏感场景，改写和检索并行（多个改写 query 同时检索），而非串行（改写完再检索），降延迟。原则：按需改写（简单不改、复杂多改），控制数量（3-5），并行降延迟。

### 第四层：方案权衡

**Q：混合检索（向量+BM25）的权重怎么调？向量权重高还是 BM25 权重高？**

按场景调，用 RRF 可免调。1）场景差异——语义型 query（如"如何省钱"，表述多样）向量权重高（语义匹配主导）；精确型 query（如"X-100 型号参数"，含专有名词）BM25 权重高（精确匹配主导）；2）调参方法——用评估集（query+golden），网格搜索向量/BM25 权重组合（如 0.3/0.7、0.5/0.5、0.7/0.3），选 Recall 最高的；3）RRF 免调——用 RRF 融合（只看排名），无需调权重，简单且效果好（业界默认）。实务：默认 RRF（免调），如果场景明确偏一侧（如强精确匹配场景）再调权重（BM25 高）。还要监控——线上 query 分布变了（如从语义型变精确型），权重可能要重调。

**Q：rerank 能提升精度但增加延迟和成本，召回率已经够（85%）的场景，还要加 rerank 吗？**

看精度是否够。1）召回够但精度差——Recall@20=85%（召回够）但 top-5 里 golden 比例低（如 MRR 低，golden 排第 10+），top-5 给 LLM 的话 golden 没进，答案仍差，要加 rerank（把 golden 排进 top-5）；2）召回够且精度够——Recall@5 已经高（golden 在 top-5），rerank 边际收益小，可不加（省成本）；3）判断方法——看 MRR/Recall@5（而非只 Recall@20），如果 top-K（实际给 LLM 的）质量够，不用 rerank。原则：rerank 是为了"把相关文档排进实际用的 top-K"，如果 top-K 已经好（不需要重排），省掉 rerank；如果 top-K 差（相关文档在更后面），加 rerank。

### 第五层：验证与沉淀

**Q：你怎么证明"广撒网+精选"的召回率优化有效（相比单路检索提升多少）？**

AB 对比。固定评估集（query+golden 文档），对比：1）基线（单路向量检索 top-K）；2）优化版（多路召回+RRF+rerank top-K）。指标：1）Recall@K——优化版应显著高于基线（如 60%→85%），证明广撒网提升召回；2）MRR/NDCG——优化版排序应更好（rerank 效果）；3）端到端准确率——基于优化版检索结果的答案准确率应提升；4）成本——优化版的 latency/调用成本是否可接受（多路+rerank 增加成本）。最优：Recall 显著升 + 排序好 + 端到端升 + 成本可接受 = 有效。还要监控线上真实 query 的召回（通过用户反馈/答案质量代理）。

**Q：全链路召回率优化怎么沉淀成团队的检索能力？**

建检索平台：1）多路召回组件——内置向量检索/BM25/多 query 改写等组件，可配置组合（如选哪些路、改写策略）；2）融合+rerank——RRF 融合 + 可插拔 rerank 模型（多种选择），标准化接口；3）诊断工具——自动化诊断（Recall@K 趋势/漏召回归因），定位召回瓶颈；4）评估闭环——评估集管理+自动评估（Recall/MRR/端到端），优化后验证；5）调参工具——网格搜索/自动调参（如向量/BM25 权重、K 值），辅助优化决策。这套写入团队 RAG 平台 SOP，让"提升召回"从"手动调参"变成"平台化能力"，新 RAG 系统快速复用。
