# Skill的渐进式披露是什么意思，为什么它能提升Agent的稳定性？

## 全量披露 vs 渐进式披露

```
全量披露（传统方式）：
┌──────────────────────────────────────┐
│ 上下文 = System Prompt + 全部工具定义   │
│                                        │
│ 工具1: search_products (完整Schema)    │  ← 800 Token
│ 工具2: place_order (完整Schema)        │  ← 600 Token
│ 工具3: process_refund (完整Schema)     │  ← 500 Token
│ ...                                    │
│ 工具100: export_report (完整Schema)    │  ← 400 Token
│                                        │
│ 总计: 50,000 Token（占窗口39%）        │
│                                        │
│ 问题：                                 │
│ ❌ 上下文被工具定义占满，对话空间不足   │
│ ❌ 100个工具选择困难，准确率下降        │
│ ❌ 模型被无关工具描述干扰               │
└──────────────────────────────────────┘

渐进式披露（分层加载）：
┌──────────────────────────────────────┐
│ Level 1: 常驻上下文（仅工具名+简介）    │
│   "search_products: 搜索商品"          │  ← 20 Token
│   "place_order: 下单购买"              │  ← 15 Token
│   "process_refund: 退款处理"           │  ← 15 Token
│   ...100个工具                          │
│   总计: 2,000 Token（占窗口1.6%）     │
└───────────────┬──────────────────────┘
                │ 模型选择search_products
                ▼
┌──────────────────────────────────────┐
│ Level 2: 按需展开（选中工具的完整Schema）│
│   search_products完整参数定义+约束     │  ← 800 Token
│   总计: 仅+800 Token                  │
└──────────────────────────────────────┘
```

## 实现架构

```python
class ProgressiveSkillDisclosure:
    """三级渐进式披露系统"""

    def __init__(self):
        # Level 1: 常驻摘要（所有工具的一句话描述）
        self.skill_summaries = {}
        # Level 2: 完整Schema（按需加载）
        self.skill_schemas = {}
        # Level 3: 使用示例（执行时注入）
        self.skill_examples = {}

    def build_context(self, available_skills: list, user_input: str) -> dict:
        """构建分层上下文"""

        # Step 1: 预筛选（从100个工具中选Top-5最相关的）
        top_skills = self._pre_filter(available_skills, user_input, k=5)

        # Step 2: 构建分层上下文
        context = {
            # Level 1: 所有工具的摘要（常驻）
            'skill_menu': [
                f"- {s['name']}: {s['brief']}"
                for s in available_skills
            ],  # ~2000 Token

            # Level 2: 预筛选工具的完整Schema
            'skill_details': [
                self.skill_schemas[s] for s in top_skills
            ],  # ~4000 Token（5个×800）

            # Level 3: 暂不加载，执行时再注入
        }
        return context

    def _pre_filter(self, skills: list, query: str, k: int = 5) -> list:
        """用embedding预筛选最相关的k个工具"""
        query_emb = embed(query)
        skill_embs = [embed(s['brief']) for s in skills]
        scores = cosine_similarity(query_emb, skill_embs)
        # 取Top-K
        top_indices = np.argsort(scores)[-k:]
        return [skills[i] for i in top_indices]
```

## 稳定性提升机制

```python
async def execute_with_progressive_disclosure(
    self, user_input: str, skills: list
):
    """渐进式披露执行流程"""

    # Round 1: 从工具菜单中选择（Level 1→2）
    selected = await self.llm_call(
        prompt=f"""用户需求：{user_input}

可用工具：
{self._format_menu(skills)}

请选择最合适的工具（1-3个），返回工具名："""
    )

    # Round 2: 用选中工具的完整Schema执行（Level 2→3）
    for skill_name in selected:
        schema = self.skill_schemas[skill_name]
        examples = self.skill_examples.get(skill_name, [])

        result = await self.llm_call(
            prompt=f"""使用工具 {skill_name} 完成任务。

工具定义：
{json.dumps(schema, ensure_ascii=False)}

使用示例：
{examples}

用户需求：{user_input}

请生成工具调用参数：""",
            tools=[schema]  # 只传入选中的工具
        )

    return result
```

## 效果对比

| 指标 | 全量披露 | 渐进式披露 | 提升 |
|------|---------|-----------|------|
| 常驻Token | 50,000 | 2,000 | -96% |
| 工具选择准确率 | 72% | 91% | +19% |
| 参数生成准确率 | 85% | 93% | +8% |
| 平均延迟 | 3.2s | 3.8s | +0.6s（多一轮） |
| 幻觉工具调用 | 8% | 1.5% | -81% |

## 面试加分点

1. **多轮权衡**：渐进式披露增加一轮LLM调用（+0.6s延迟），但准确率提升省去了后续重试成本，总延迟反而可能更低
2. **预筛选优化**：用轻量embedding模型做预筛选（~10ms），不需要大模型参与
3. **动态K值**：简单需求预筛选Top-3，复杂需求Top-8，动态调整而非固定
4. **Claude Code的实践**：Claude Code的Skill系统就是渐进式披露的典型案例——SKILL.md先给摘要，使用时才加载完整内容

## 记忆要点

- 全量加载导致Token爆炸和选择困难，渐进式按需加载提升准确率
- 三级架构：Level1常驻摘要列表，Level2按需展开Schema，Level3执行时注入示例
- 防上下文污染：常驻区只留工具名加一句话简介，预留充足对话空间

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：渐进式披露的核心是"分层按需加载"，减少常驻 Token。但现在的 LLM 上下文窗口很大（128K-200K），50K 的工具定义也装得下。窗口够用为什么还要渐进式披露？只是为了省 Token 成本吗？**

不只是省 Token，更重要的是"提升选择准确率"（Choice Paradox 的缓解）。即使窗口够大（128K），全量披露仍有问题：一是注意力稀释——LLM 的注意力是有限资源（softmax 权重总和为 1），上下文中工具越多，每个工具的注意力权重被分散。100 个工具时，模型对正确工具的注意力可能被其他 99 个工具的描述稀释，选择准确率下降（实测从 72% 降到更低）。窗口大不等于注意力充足。二是信息干扰——无关工具的描述（即使模型不选）仍参与注意力计算，可能误导。如工具列表有"process_refund"，用户问"查询订单"，refund 的描述可能与 query 语义部分重叠，干扰选择。三是指令遵循下降——长上下文（50K 工具定义）让模型对"指令"（system prompt 的任务要求）的注意力被工具定义淹没，遵循指令的能力下降（"lost in the middle"现象）。四是成本——虽窗口够，但每 token 都有推理成本（计算量与序列长度相关），50K 工具定义让每次推理慢且贵。渐进式披露（常驻 2K + 按需展开）解决这些问题：注意力集中（只看相关工具）、干扰少（无关工具不在上下文）、指令遵循好（上下文短）、成本低（序列短）。所以渐进式披露的核心价值是"质量提升"（选择准确率 +19%），省 Token 是附带收益。即使窗口无限大，渐进式披露仍值得用。

### 第二层：证据与定位

**Q：你说渐进式披露提升工具选择准确率 72%→91%。但这个数据怎么测的？怎么定义"选择准确率"？怎么确认提升是来自"渐进式"而非其他因素（如预筛选的 embedding 质量）？**

准确率的定义和测量需严谨。准确率定义：对测试集的每个 case（用户输入 + 正确工具标注），Agent 选择工具，正确（选了对的工具）记 1，错误记 0，准确率 = 正确数 / 总数。测试集构建：人工标注 case（用户输入对应哪个工具正确），覆盖各类意图和边界（如歧义输入、多工具场景），200-500 case 统计显著。测量方法：同测试集，对比"全量披露"（所有工具 schema 在上下文）和"渐进式披露"（分层加载）的选择准确率，控制其他变量（同模型、同 prompt、同测试集），确保差异来自披露方式。确认提升来源的 ablation：一是"全量披露 vs 渐进式"——差 19%，证明渐进式的总贡献。二是"渐进式去掉预筛选"（全量给 Level 1 摘要，模型从 100 个摘要选）vs 完整渐进式（预筛 Top-5）——差值是预筛选的贡献。三是"渐进式去掉分层"（直接给 Top-5 完整 schema，无 Level 1 摘要）vs 完整——差值是分层（Level 1 摘要 vs 直接 schema）的贡献。通过 ablation 分解贡献来源，确认是预筛选、分层、还是两者协同。还需控制 embedding 质量——用同 embedding 模型（如 text-embedding-3），确保预筛选质量一致。证明逻辑是"同测试集对比 + ablation 分解 + 控制变量"，确认提升来自渐进式披露的具体机制。关键是从"渐进式好"细化到"哪个机制贡献多少"，指导优化。

### 第三层：根因深挖

**Q：预筛选用 embedding 相似度（query 与工具 brief 的余弦相似度，取 Top-K）。但用户的 query 可能模糊或歧义，embedding 相似度可能选错工具（漏掉正确工具）。预筛选错了，后续就全错。怎么保证预筛选的召回率？**

预筛选的召回率（正确工具在 Top-K 内）是渐进式披露的关键。提升召回的方法：一是 K 值留余量——预筛 Top-5 而非 Top-1，即使相似度排序有小偏差，正确工具仍可能在 Top-5 内。K 太小（如 1）召回低（一个错全错），K 太大（如 20）失去预筛意义（仍信息过载）。K=5-8 是平衡点（召回率高，信息量可控）。二是动态 K——简单 query（明确意图）用小 K（如 3，明确工具），复杂 query（歧义、多意图）用大 K（如 10，覆盖可能工具）。用 query 的置信度（如 max similarity score）判断，高置信（> 0.9）用小 K，低置信（< 0.7）用大 K。三是多路召回——不只 embedding 相似度，结合关键词匹配（工具名或 brief 含 query 关键词）、历史使用频率（常用工具优先）。多路召回取并集（或加权排序），提升召回。四是 embedding 优化——用更好的 embedding 模型（如领域微调的 embedding），或工具 brief 优化（更清晰的描述，提升 embedding 质量）。五是兜底——预筛选后，模型在 Level 1 摘要（全工具）中仍能"看到"所有工具名，若预筛漏了正确工具，模型可在 Level 1 阶段从摘要中补选（"似乎还有个工具 X 相关，加入候选"）。这是"预筛选 + 模型补选"的双重保障。六是监控召回率——线上统计预筛选的召回率（通过模型最终选的工具是否在预筛 Top-K 内），召回率低（< 90%）时优化（调 K、改 embedding、优化 brief）。实践中 K=5-8 + 动态调整 + 多路召回 + 兜底，召回率可达 95%+，保证正确工具进入候选。

**Q：那如果预筛选召回率做不到 100%（总漏），漏了正确工具怎么办？模型在 Level 1 看到 100 个工具摘要，但 Level 2 只展开 5 个，如果正确的不在这 5 个，模型会被限制在错误候选里。怎么兜底？**

兜底机制是必须的，因为预筛选不可能 100% 召回。兜底策略：一是 Level 1 的"全可见"——模型在 Level 1 看到 100 个工具的摘要（名字+一句话），即使 Level 2 只展开 5 个，模型知道所有工具的存在。如果 query 与某未展开工具相关，模型可主动请求展开（"我想用 process_return，请展开它的 schema"）。实现：prompt 允许模型输出"request_more_tools: [tool_name]"，系统动态展开。二是 fallback 触发——如果模型在 Top-5 候选中选了工具但执行后失败（参数错、结果不符），触发 fallback：扩大预筛选范围（Top-20）或全量展开，重新选择。失败是预筛选错的信号，fallback 修正。三是置信度门限——模型选择时输出置信度，低置信（如 < 0.6）暗示候选可能不对，主动触发扩大候选（"我不确定，请给我更多相关工具"）。四是用户澄清——如果候选都不合适（模型都不满意），向用户澄清（"您是想退货还是退款？我需要确认"），而非在错误候选里硬选。五是召回监控与优化——线上统计 fallback 触发率（预筛漏的频率），高频触发说明预筛选需优化（K 太小或 embedding 差），迭代改进。实践：Level 1 全可见（模型能主动请求展开）+ fallback（失败后扩大候选）+ 置信度门限（低置信扩大）+ 用户澄清（都不合适时问）+ 监控优化（高频 fallback 改预筛选）。兜底机制确保预筛选的错误可恢复，不导致任务失败。关键是不把预筛选当"硬限制"，而是"软推荐"，模型和系统都有纠错能力。

### 第四层：方案权衡

**Q：渐进式披露增加一轮 LLM 调用（先选工具，再用工具），延迟 +0.6s。实时场景（如语音交互）对延迟敏感，这 0.6s 值得吗？能不能省掉这一轮（直接根据 query 选+用工具）？**

延迟的权衡取决于"准确率提升 vs 延迟成本"。0.6s 值得的场景：一是高准确率需求——工具选错导致任务失败（如订错机票），失败后重试或人工处理的成本远高于 0.6s。渐进式提升准确率 19%，减少失败，总延迟（含重试）反而更低。二是多工具场景——100+ 工具时，全量披露的准确率低（72%），渐进式的准确率提升显著，0.6s 换 19% 准确率划算。三是复杂任务——需多步工具调用，第一步选错后续全错，渐进式的首轮准确率保障整体成功。不值得的场景：一是少工具（< 10 个）——全量披露准确率已高（> 90%），渐进式提升小（1-2%），0.6s 延迟不划算，直接全量展开。二是简单任务——单工具、明确意图（如"查天气"），直接匹配，无需预筛选。三是极延迟敏感——如实时翻译、游戏交互，0.6s 影响体验，用全量或固定工具。省掉这一轮的方法：一是固定工具集——按场景预设工具（如客服场景固定 5 个工具），无需动态选择，省一轮。二是缓存——同 query 的工具选择缓存（用户问过"查天气"，下次直接用天气工具），省预筛选。三是并行——预筛选（embedding，10ms）与 Level 1 选择并行（而非串行两轮），减少延迟。实践：多工具+复杂任务用渐进式（0.6s 换准确率），少工具+简单任务用全量或固定（省延迟）。实时场景优化（并行、缓存）进一步降延迟。这是场景驱动的选择，不是"一律渐进式"或"一律全量"。

**Q：Claude Code 的 Skill 系统是渐进式披露的典型案例（SKILL.md 先给摘要，用时加载完整）。但 Claude Code 的 Skill 是"模型主动触发"（模型读到摘要决定加载），而你的方案是"系统预筛选+展开"（embedding 选 Top-K）。两种方式各有什么优劣？为什么不统一？**

两种方式是"模型驱动"vs"系统驱动"的权衡。Claude Code 的模型驱动：模型在上下文看到所有 Skill 的摘要（SKILL.md 的 YAML frontmatter，含 name + description），模型自主决定加载哪个 Skill 的完整内容。优势：一是灵活——模型理解语义，能处理复杂关联（query 与 Skill 的语义关联，而非 embedding 的表面相似），选择更准。二是简单——无需预筛选系统，模型直接选，架构简单。劣势：一是所有摘要占上下文（即使模型驱动，100 个摘要仍占 2K+ Token）；二是模型选择有延迟（模型推理选 Skill，比 embedding 慢）。系统驱动（embedding 预筛选）：系统用 embedding 预选 Top-K，模型只在 Top-K 中选。优势：一是快（embedding 10ms vs 模型推理 100ms+）；二是省上下文（只展开 Top-K 的完整 schema，非全部）。劣势：一是 embedding 可能漏（表面相似，漏语义关联的工具）；二是刚性（预筛选错了模型无法补救，除非有兜底）。不统一的原因：场景适配。Claude Code 的场景（开发辅助，Skill 数量中等如 20-50 个，质量优先）适合模型驱动（准确，数量适中延迟可接受）。大规模 Agent（工具 100+，延迟敏感）适合系统驱动（快，预筛选减少模型负担）。结合方案：系统预筛选（快速缩范围）+ 模型在缩范围后选（语义准确）+ 模型能请求展开未预筛的 Skill（兜底）。这是"系统+模型"协同，兼顾速度和准确。Claude Code 的实践是"数量适中用模型驱动"，大规模用"系统+模型"协同，按场景选。

### 第五层：验证与沉淀

**Q：渐进式披露上线后，怎么持续验证它比全量披露好？不只是离线测试，而是线上实际效果？**

线上多维验证。一是工具选择准确率监控——线上统计工具选择的正确率（通过任务成功率或用户反馈推断），对比渐进式 vs 全量（A/B），渐进式应更高。二是延迟监控——监控渐进式的额外延迟（预筛选 + 多一轮 LLM 调用）vs 全量，验证延迟可接受（如 < 1s）。如果延迟过高（如 > 2s），优化（并行、缓存）。三是成本监控——渐进式的 Token 消耗（常驻 2K + 按需展开）vs 全量（50K），渐进式应显著省 Token（> 90%）。四是 fallback 触发率——监控预筛选错误导致 fallback 的频率，高频（> 10%）说明预筛选需优化（K、embedding、brief）。五是用户满意度——A/B 对比渐进式 vs 全量的用户满意度（任务完成率、复用率），渐进式应更高（准确率高→完成好→满意）。六是长期趋势——随工具数量增长（如从 100 到 500），渐进式的优势应放大（全量更不可行，渐进式仍有效），验证可扩展性。证明逻辑是"准确率 + 延迟 + 成本 + fallback + 满意度 + 可扩展性"的线上 A/B 对比，渐进式应在准确率/成本/可扩展性上占优，延迟小幅增加可接受。如果某维度异常（如延迟过高或 fallback 频繁），针对性优化。

**Q：怎么让团队在做工具管理时不混乱（有人全量、有人渐进、有人固定工具集），而是统一的工具披露架构？**

沉淀工具披露架构规范和选型决策树。一是架构规范：统一的"分层披露 + 预筛选 + 兜底"架构，各 Agent 遵循，避免各自实现。二是选型决策树：按工具数量和场景选——少工具（< 10）用全量披露（简单够用）；中等多工具（10-50）用模型驱动（Claude Code 风格，准确）；大量工具（100+）用系统驱动预筛选 + 渐进式（快+准）。三是工具描述规范：统一的 brief 格式（工具名 + 一句话描述，清晰无歧义）、schema 格式（标准 JSON Schema）、示例格式（few-shot），跨工具一致，保证预筛选和模型理解的质量。四是预筛选规范：统一的 embedding 模型和 K 值策略（动态 K、多路召回），跨 Agent 一致。五是兜底规范：统一的兜底机制（模型请求展开、fallback 扩大候选、置信度门限、用户澄清），保证预筛选错误可恢复。六是监控规范：统一的指标（选择准确率、延迟、成本、fallback 率），跨 Agent 可比。让工具披露是"统一架构 + 决策树选型 + 规范描述 + 标准预筛选 + 兜底机制 + 监控"的系统工程，不靠各团队随意实现。
