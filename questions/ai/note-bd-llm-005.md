# 【字节面经】Few-shot 和 Chain-of-Thought 在什么场景下分别更有效？两者能否组合使用？

## 一、问题本质：两种不同的 Prompt 技术

Few-shot 和 Chain-of-Thought（CoT）是 Prompt Engineering 中最基础也是最重要的两种技术，但它们**激活的是 LLM 的不同能力维度**：

| 维度 | Few-shot | Chain-of-Thought |
|------|----------|-----------------|
| **核心机制** | 模式学习（Pattern Learning） | 推理引导（Reasoning Guidance） |
| **本质** | 教 LLM "做什么"（学格式和映射） | 教 LLM "怎么想"（展开推理过程） |
| **类比** | 给学生看标准答案学格式 | 要求学生写出解题过程 |
| **解决痛点** | 输出格式不可控、任务边界不清 | 复杂推理出错、中间步骤遗漏 |
| **最佳场景** | 分类、抽取、格式化输出 | 数学推理、逻辑推理、多步规划 |

**一句话总结**：Few-shot 解决"输出什么"，CoT 解决"怎么推理"。两者正交，可以组合。

---

## 二、Few-shot Prompting

### 2.1 核心思想

通过在 Prompt 中提供少量（通常 1~5 个）**输入-输出示例**，让 LLM 从示例中学习任务模式，从而在处理新输入时输出符合预期格式和逻辑的结果。

这本质上是一种**上下文学习（In-Context Learning, ICL）**——不需要更新模型权重，仅通过上下文中的示例来"激活"模型已有能力。

### 2.2 Prompt 示例

```
任务：情感分类

示例1：
输入：这家餐厅的菜品非常精致，服务也很到位！
输出：正面

示例2：
输入：等了半小时才上菜，味道还一般，不推荐。
输出：负面

示例3：
输入：价格还行，但环境比较吵，适合随便吃吃。
输出：中性

现在请分类：
输入：这部电影简直是浪费时间，剧情漏洞百出。
输出：
```

### 2.3 优势场景

- **格式控制**：需要严格输出 JSON / XML / 特定格式时
- **分类任务**：情感分析、意图识别、垃圾邮件检测
- **信息抽取**：NER、关系抽取、事件抽取
- **翻译/改写**：提供风格示例，控制输出风格
- **指令遵循**：零样本下模型不听话，加示例后立刻配合

### 2.4 局限性

- **不擅长复杂推理**：示例教的是"格式"，不教"推理过程"
- **Token 开销大**：每个示例都占用上下文窗口
- **示例选择敏感**：选错示例会引入偏差（Negative Bias）
- **模式过拟合**：模型可能机械模仿示例模式而非理解任务

## 三、Chain-of-Thought (CoT) Prompting

### 3.1 核心思想

CoT 的核心是**让 LLM 在给出最终答案前，先输出中间推理步骤**。通过显式展开推理链，将复杂问题分解为多个简单子问题，从而激活模型的逐步推理能力。

这利用了 LLM 的一个重要特性：**自回归生成机制下，中间 token 会成为后续生成的上下文**。展开推理步骤实际上是在为模型自己"铺路"。

### 3.2 Zero-shot CoT

最简单的 CoT 形式，只需加一句魔法咒语：

```
问题：一个商店打8折后再打9折，相当于打几折？

让我们一步步思考。

模型输出：
打8折后的价格是原价的 0.8
在此基础上再打9折：0.8 × 0.9 = 0.72
所以相当于打 7.2 折。

答案：7.2折
```

仅需添加 **"Let's think step by step"** 或 **"让我们一步步思考"**，无需任何示例。GSM8K 数学推理基准上可提升 10~20 个百分点。

### 3.3 Few-shot CoT（CoT 的完整形态）

在示例中不仅给出答案，还给出**完整的推理过程**：

```
任务：数学应用题

示例1：
问：小明有5个苹果，给了小红2个，又买了3个，现在有几个？
答：小明原有5个苹果。
    给了小红2个：5 - 2 = 3个。
    又买了3个：3 + 3 = 6个。
    所以小明现在有6个苹果。

示例2：
问：教室里有30把椅子，排成5排，每排椅子数相同，每排几把？
答：总共有30把椅子，排成5排。
    每排椅子数 = 30 ÷ 5 = 6把。
    所以每排有6把椅子。

现在请回答：
问：一列火车有12节车厢，每节坐8人，中途下了45人，还剩多少人？
答：
```

### 3.4 优势场景

- **数学推理**：算术应用题、代数推导
- **逻辑推理**：三段论、条件推理、谜题
- **常识推理**：因果推断、物理常识
- **多步规划**：任务分解、流程设计
- **符号操作**：字符串变换、正则推导

### 3.5 局限性

- **Token 开销大**：推理过程本身占用大量 token
- **简单任务过度复杂化**：分类任务用 CoT 反而可能降低性能
- **推理链可能跑偏**：一步错，步步错（Error Propagation）
- **并非所有模型受益**：小模型（<60B 参数）使用 CoT 效果反而变差

## 四、Few-shot vs CoT 全面对比

| 维度 | Few-shot | Chain-of-Thought |
|------|----------|-----------------|
| **核心目标** | 控制输出格式和任务模式 | 提升复杂推理的准确率 |
| **技术机制** | 上下文模式学习 | 中间推理步骤展开 |
| **Token 开销** | 中等（示例 × N） | 较高（推理链 + 示例） |
| **简单任务（分类）** | ⭐ 效果好 | 无提升甚至有害 |
| **复杂推理（数学）** | 效果有限 | ⭐ 显著提升（+10~30%） |
| **输出可控性** | ⭐ 高（格式明确） | 较低（推理过程不可控） |
| **模型能力要求** | 所有模型通用 | 需要足够大的模型（>60B） |
| **示例需求** | 必须（1~5个） | 可选（Zero-shot CoT 无需示例） |

### Prompt 示例对比

**同一个数学问题，三种写法对比**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❶ Zero-shot（不加任何技术）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
问：小明有50元，买了3本书每本12元，又买了一支5元的笔，还剩多少钱？
答：还剩9元。

→ 错误！模型直接猜答案，跳过了计算。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❷ Zero-shot CoT（只加引导句）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
问：小明有50元，买了3本书每本12元，又买了一支5元的笔，还剩多少钱？
让我们一步步思考。

答：
- 小明有50元
- 买书花了：3 × 12 = 36元
- 买笔花了：5元
- 总花费：36 + 5 = 41元
- 剩余：50 - 41 = 9元
所以还剩9元。

→ 正确！推理链展开后计算准确。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❸ Few-shot CoT（示例中含推理过程）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
示例：
问：小李有100元，买了2件衣服每件35元，退了1件，还剩多少钱？
答：小李有100元。
    买2件衣服花：2 × 35 = 70元。
    退了1件，退回35元。
    实际花费：70 - 35 = 35元。
    剩余：100 - 35 = 65元。

现在请回答：
问：小明有50元，买了3本书每本12元，又买了一支5元的笔，还剩多少钱？
答：小明有50元。
    买书花费：3 × 12 = 36元。
    买笔花费：5元。
    总花费：36 + 5 = 41元。
    剩余：50 - 41 = 9元。

→ 正确！且推理格式更规范，更可控。
```

## 五、组合使用：Few-shot CoT

### 5.1 两者正交且互补

Few-shot 和 CoT 是**正交技术**——前者管"做什么"（格式和任务定义），后者管"怎么想"（推理过程）。组合使用 = Few-shot CoT，效果 **1 + 1 > 2**：

```
                 推理能力
                    ↑
                    │      ┌──────────┐
                    │      │ Few-shot │
                    │      │   CoT   │ ← 最佳：既学格式又学推理
                    │      └──────────┘
        CoT ────────┼──────────────────→ 格式控制
                    │  ┌──────────┐
                    │  │Zero-shot │
                    │  │   CoT   │
                    │  └──────────┘
                    │
              Few-shot(无CoT)
              ┌──────────┐
              │ Few-shot  │
              │ (无推理)  │
              └──────────┘
```

### 5.2 实际效果数据

在 GSM8K（小学数学题）上的典型效果对比：

| 方法 | GPT-3.5 | GPT-4 | Llama-2-70B |
|------|---------|-------|-------------|
| Zero-shot | 15%~20% | 40%~50% | 10%~15% |
| Zero-shot CoT | 50%~60% | 85%~92% | 40%~50% |
| Few-shot | 25%~30% | 55%~60% | 15%~20% |
| **Few-shot CoT** | **60%~70%** | **90%~95%** | **50%~55%** |

> 注意：在简单分类任务上，Few-shot（不加 CoT）通常优于 Few-shot CoT，因为 CoT 对简单任务是噪声。

## 六、进阶：Self-Consistency

### 6.1 核心思想

CoT 的一个问题是**推理链可能出错**——同一道题用不同思路可能得到不同答案。Self-Consistency（自洽性）的解法是：**对同一问题采样多条推理链，取多数投票的结果**。

### 6.2 工作流程

```
              同一个问题
                 │
        ┌────────┼────────┐
        ▼        ▼        ▼
    推理路径1  推理路径2  推理路径N
    (temp=0.7)(temp=0.7) (temp=0.7)
        │        │        │
        ▼        ▼        ▼
     答案: A   答案: B   答案: A
        │        │        │
        └────────┼────────┘
                 ▼
           多数投票 → A
```

### 6.3 实现示例

```python
def self_consistency_answer(llm, question, n_samples=5):
    """Self-Consistency: 采样多条推理链，取多数票"""
    answers = []
    for i in range(n_samples):
        prompt = f"""
        问题：{question}
        让我们一步步思考。
        """
        # 使用较高 temperature 增加推理路径多样性
        response = llm.generate(prompt, temperature=0.7)
        # 提取最终答案
        answer = extract_final_answer(response)
        answers.append(answer)

    # 多数投票
    from collections import Counter
    most_common = Counter(answers).most_common(1)[0][0]
    return most_common
```

### 6.4 效果

Self-Consistency 在 GSM8K 上通常可以**再提升 5~15 个百分点**，代价是推理成本增加 N 倍（N = 采样次数，通常 5~40）。

## 八、面试回答要点总结

1. **先区分本质**：Few-shot = 模式学习（学格式），CoT = 推理引导（学思路），两者正交
2. **场景不同**：简单任务用 Few-shot（分类、抽取），复杂推理用 CoT（数学、逻辑）
3. **可以组合**：Few-shot CoT 是最常用且效果最好的组合——示例中既包含格式也包含推理过程
4. **进阶提 Self-Consistency**：多条推理链 + 多数投票，进一步提升准确率，但推理成本增加
5. **注意模型能力门槛**：CoT 对小模型（<60B）可能无效甚至有害
6. **给出具体数据**：GSM8K 上 Few-shot CoT 比 Zero-shot 提升 30~50 个百分点

## 记忆要点

- 核心对比：Few-shot教模型'输出什么'(学格式)，而CoT教模型'怎么想'(展推理)，两者维度正交
- Few-shot场景：解决输出格式不可控，擅长分类、抽取等简单模式映射任务，提供示例即可
- CoT场景：解决复杂推理易遗漏中间步骤，擅长数学与逻辑推理，通过生成中间token铺路
- 组合使用：Few-shot CoT是终极形态，在示例中不仅给答案更给出完整推导链路
- 避坑指南：简单任务用CoT可能过度复杂化降低性能，且小模型(<60B)用CoT反而变差

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Few-shot 和 CoT 你说是正交的两个维度，能不能讲清楚它们各自解决什么不同的失败模式？**

Few-shot 解决"输出格式/模式不可控"——模型不知道你要什么格式（JSON？分类标签？），给几个示例让它学会"输入→输出"的映射模式，解决的是"模式学习"。CoT 解决"复杂推理跳步出错"——数学/逻辑题直接给答案时模型可能跳过中间步骤导致错，CoT 引导模型"先写推理过程再给答案"，通过生成中间 token 为最终答案"铺路"，解决的是"推理质量"。两者正交：可以只 Few-shot（学格式）、只 CoT（引导推理）、或 Few-shot CoT（示例里带推理过程）。

### 第二层：证据与定位

**Q：你给模型加了 CoT 但准确率没提升甚至降了，怎么判断是 CoT 没起效还是任务不适合 CoT？**

看任务类型和模型大小。如果是简单任务（如分类、抽取、单步查询），CoT 是过度复杂化——模型本来能直接答，强制它写推理过程反而引入错误（推理链里某步错了带偏答案），这类任务 CoT 无效甚至有害。如果是复杂推理任务（数学、多跳逻辑）但模型太小（<60B），小模型推理能力弱，CoT 生成的推理链本身就错，自然不提升。验证方法：对比"有 CoT vs 无 CoT"在"简单任务 vs 复杂任务"上的表现，简单任务应无差异或 CoT 略差，复杂任务 CoT 应提升（大模型上）。

### 第三层：根因深挖

**Q：CoT 为什么能提升复杂推理？根因是"给模型更多思考时间"吗？这个直觉解释对吗？**

直觉解释不准确，LLM 没有"思考时间"（前向传播是即时的）。根因是"中间 token 扩展了计算路径"。Transformer 生成每个 token 时，前面的 token 作为 context 影响后续生成。直接给答案时，模型用一个 forward pass 从 query 映射到 answer，中间推理在"隐空间"完成，复杂推理时隐空间表达力不够。CoT 让模型把中间推理"显式生成成 token"，这些 token 重新作为输入参与后续计算，相当于"用更多层 forward 展开推理"，计算深度增加，推理更可靠。这是为什么 CoT 对多步推理有效——它把"一步难题"分解成"多步简单题"。

**Q：那为什么不直接在模型架构里加更多层（如 200 层 Transformer）替代 CoT，让模型一次推理到位？**

计算成本和效率。加层让所有请求（包括简单任务）都付更高的计算成本，而 CoT 只在复杂推理时"按需展开"。且研究表明，CoT 的"外部展开"和"加层"不完全等价——CoT 的中间 token 是"基于具体问题的动态推理"，而加层是"固定的更深的通用计算"，前者对特定推理任务更灵活。更现实的是训练成本——200 层模型训练成本是 100 层的数倍，且可能过拟合。CoT 用"推理时成本"换"推理能力"，比"训练时加层"性价比高。当然两者不互斥，大模型 + CoT 是当前最优组合。

### 第四层：方案权衡

**Q：Few-shot 你用几个示例？为什么不更多（如 10 个），示例越多模型学得越好吧？**

通常 3-5 个，不是越多越好。原因有三：一是 token 成本——每个示例占几百 token，10 个示例就是几千 token prompt，成本和延迟上升；二是注意力稀释——prompt 太长模型对每个示例的注意力下降，后面的示例可能被忽略（lost in the middle）；三是边际递减——3 个示例已经足够展示模式，第 4-10 个的提升很小。示例的质量比数量重要——3 个高质量、有代表性的示例（覆盖不同 case 类型）比 10 个冗余示例有效。如果 3 个不够，先检查示例是否有代表性，而非加数量。

**Q：为什么不直接用 Zero-shot CoT（"Let's think step by step"）省得写示例，简单又省 token？**

Zero-shot CoT 对强模型（GPT-4 级）有效，对弱模型效果差。强模型理解"step by step"的含义能生成好的推理链，弱模型可能生成流水账式的"伪推理"（看起来在 step by step 但逻辑无效）。Few-shot CoT 用示例"教"模型怎么推理，对弱模型更有效。且 Zero-shot 无法控制推理链的格式和深度，Few-shot 能通过示例引导（如"每步先写假设再验证"）。工程上：强模型用 Zero-shot CoT 省 token，弱模型用 Few-shot CoT 保效果，按模型能力选。

### 第五层：验证与沉淀

**Q：你怎么决定一个任务用 Few-shot、CoT、还是 Few-shot CoT？有没有系统方法？**

按任务的"格式复杂度"和"推理复杂度"两维度选。格式简单+推理简单（如情感分类）：Zero-shot 够，不用 Few-shot 也不用 CoT。格式复杂+推理简单（如结构化抽取）：Few-shot（教格式）。格式简单+推理复杂（如数学题）：CoT。格式复杂+推理复杂（如复杂分析报告）：Few-shot CoT。用评测集验证——在任务上对比四种策略的准确率，选最优。不要默认所有任务都上 CoT，简单任务上 CoT 有害。

**Q：Prompt 工程的 Few-shot/CoT 经验怎么沉淀？**

固化成"Prompt 策略选型指南"：按任务类型（分类/抽取/推理/生成）推荐策略，附示例模板。沉淀"高质量 Few-shot 示例的编写规范"（覆盖 case 类型、边界 case、格式一致）、"CoT 推理链的模板"（如"已知→推理→结论"三段式）。配套 Prompt A/B 实验框架，新 prompt 上线前自动对比效果。把"各模型对各策略的响应差异"（如 GPT-4 的 Zero-shot CoT 效果 vs 弱模型的 Few-shot CoT 效果）记录成模型-prompt 适配表。

## 结构化回答

**30 秒电梯演讲：** Few-shot通过示例教LLM格式和模式，CoT通过引导推理链提升复杂推理。两者可以组合且常常组合使用——Few-shot像给学生看标准答案学格式。

**展开框架：**
1. **Few-shot** — 格式学习/分类/抽取场景
2. **CoT** — 数学推理/逻辑推理/多步规划
3. **组合** — Few-shot-CoT(示例中含推理过程)
