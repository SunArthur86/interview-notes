# 【神州专车面经】事实表和维度表怎么设计？

## 一、事实表 vs 维度表

| 维度 | 事实表 | 维度表 |
|------|--------|--------|
| 内容 | 可度量业务事件 | 描述性属性 |
| 形状 | 长瘦（行多列少） | 短胖（行少列多） |
| 增长 | 高速（每天百万行） | 缓慢（司机/城市有限） |
| 主键 | 代理键（订单ID） | 代理键（司机ID） |
| 关系 | 含维度外键 | 被事实表引用 |
| 例子 | 订单、支付、点击 | 司机、乘客、城市、时间 |

## 二、事实表设计

### 三种事实表类型

| 类型 | 说明 | 出行场景 |
|------|------|---------|
| **事务型** | 每笔业务事件一行 | 订单表（每单一行） |
| **周期快照** | 定期全量快照 | 司机日快照（每天每个司机的状态） |
| **累计快照** | 业务流程的里程碑节点 | 订单生命周期（下单→接单→完成→支付 时间戳） |

### 事实表字段组成

```
事实表 = 外键 + 度量值 + 退化维度 + 时间戳
  外键：driver_id, passenger_id, city_id, date_id
  度量值：fare, distance, duration
  退化维度：order_no（订单号，无独立维度表）
  时间戳：create_time, complete_time
```

**度量值分三类**：
- **可加**：跨所有维度可加（如 fare）
- **半可加**：跨部分维度可加（如余额，不能跨时间加）
- **不可加**：比例/比率（如 转化率，不能直接加，要分别加分子分母再除）

## 三、维度表设计

### 常见维度

| 维度 | 出行场景字段 |
|------|------------|
| 时间维度 | date, year, month, week, is_holiday, is_weekend |
| 司机维度 | driver_id, name, level, join_date, city |
| 乘客维度 | passenger_id, name, vip_level, register_date |
| 城市维度 | city_id, city_name, province, tier |
| 车型维度 | car_type, brand, seats |

### 维度设计原则

1. **丰富属性**：维度表列要多（短胖），把所有描述性字段都放进来
2. **退化维度**：如果某维度只有 ID 没有属性（如订单号），直接放事实表，不建维度表
3. **一致维度**：同维度（如时间）被多个事实表共用，保证口径一致

## 四、缓慢变化维（SCD）

维度会变化（司机升等级、搬家），怎么处理历史？

| 类型 | 做法 | 适用 |
|------|------|------|
| **Type 1 覆盖** | 直接 UPDATE 新值 | 不关心历史（如纠正错误） |
| **Type 2 加行** | 新增一行，加 effective_date + is_current 标志 | **最常用**，保留完整历史 |
| **Type 3 加列** | 加 prev_value 列存旧值 | 只关心"前一次"变化 |

**Type 2 示例**：
```
driver_id | level | effective_from | effective_to | is_current
A         | B     | 2026-01-01     | 2026-06-30   | false
A         | A     | 2026-07-01     | 9999-12-31   | true
```

## 五、星型模型 vs 雪花模型

```
星型模型（推荐）：              雪花模型：
                                
    [司机]                        [司机]─[城市]─[省份]
      ↘                            ↗
    [订单]─[时间]              [订单]─[时间]
      ↗                            ↘
    [乘客]                        [乘客]─[会员等级]
```

| 维度 | 星型 | 雪花 |
|------|------|------|
| 维度规范化 | 否（维度扁平） | 是（维度再拆） |
| 查询 join 数 | 少（1层） | 多（多层） |
| 查询性能 | 快 | 慢（多 join） |
| 存储空间 | 略大（冗余） | 小（规范化） |
| 易用性 | 简单 | 复杂 |

**数仓默认选星型**：查询性能和易用性优先。

## 六、出行场景建模实例

```
事实表：
  dwd_order_di（订单明细）
    - order_id（退化维度）
    - driver_id（FK→dim_driver）
    - passenger_id（FK→dim_passenger）
    - city_id（FK→dim_city）
    - date_id（FK→dim_date）
    - fare, distance, duration（度量）
    - create_time, complete_time

维度表：
  dim_driver（driver_id, name, level, join_date, ...）
  dim_passenger（passenger_id, name, vip_level, ...）
  dim_city（city_id, city_name, province, tier）
  dim_date（date_id, year, month, is_holiday, ...）
```

## 七、加分点

- 说出 **退化维度**：只有 ID 没属性的维度（订单号）直接放事实表
- 说出 **一致维度**：跨事实表共用维度保证口径一致（Conformed Dimension）
- 说出 **无事实的事实表**：只记录"发生了什么事件"没有度量（如学生选课记录，只有学生ID+课程ID，记录"选了"这件事）

## 八、雷区

- ❌ 事实表塞维度属性（冗余，维度变化时事实表要全改）
- ❌ 维度表存度量值（违反建模原则）
- ❌ 所有维度都用 Type 2（不必要的字段变化也加行，表膨胀）

## 九、扩展

- **事实表压缩**：事实表行数多，按时间分区 + 列式存储（Parquet/ORC）大幅省空间
- **维度预热**：高频查询的维度（如时间）物化成宽表，避免每次 join
- **Data Vault 建模**：适合超大数据集成（Hub/Link/Satellite），比星型更灵活但更复杂

## 记忆要点

- 事实表记录业务事件且长瘦高速增长（含外键度量），维度表存描述属性且短胖缓慢增长
- 事实表分三种：事务型（单次事件）、周期快照（全量状态）、累计快照（里程碑时间戳）
- 缓慢变化维(SCD)：Type1直接覆盖，Type2拉链表加行（最常用），Type3增加旧值列


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：事实表和维度表为什么不能合并成一张表？分开存的动机是什么？**

动机是"减少冗余和提升查询灵活性"。如果合并，维度信息（如司机姓名、车型）会在每笔订单重复存储——一个司机一天 100 单，姓名存 100 次。维度变更时（司机改车型）要更新 100 行。分开存后，事实表只存"司机 ID"外键，维度表存一份司机信息，更新只改一行。这是数据库范式的核心思想——用 join 换取存储和一致性。

### 第二层：证据与定位

**Q：报表里司机姓名显示为空，怎么定位是事实表没有这个司机的记录，还是维度表缺这个司机？**

两步排查：1) 先查事实表——这个订单是否真实存在、driver_id 是否有值（如果事实表没记录，是数据丢失问题）；2) 再查维度表——这个 driver_id 在维度表里是否有记录（如果没有，是维度表未同步或司机未注册）。典型根因是"维度表是 T+1 同步，新司机当天没有维度记录"。验证：用 LEFT JOIN 事实表和维度表，看哪一端缺失。

### 第三层：根因深挖

**Q：事实表越来越大查询慢，根因是行数多还是字段多？**

通常两者都有但行数是主因。事实表按业务事件增长（订单表每天百万行），行数是字段数的万倍以上。查询慢的原因：1) 全表扫描——没有分区或查询没命中分区；2) 高基数 join——事实表 join 维度表时，如果维度表也大，shuffle 成本高。根因判断：看执行计划的 scan 行数，如果 scan 了全表而查询只要一天的数据，是分区问题；如果 scan 行数合理但 join 慢，是 join 性能问题。

**Q：那为什么不直接给事实表加索引加速，而要靠分区？**

数仓（Hive/Spark）和数据库（MySQL/PG）的索引机制不同。数仓是列式存储 + 批量扫描，B+ 树索引对大规模分析查询收益有限（索引随机访问反而慢）。分区是"物理分目录"，查询时按分区裁剪只读相关目录，效果显著。数据库 OLTP 场景用索引（点查快），数仓 OLAP 场景用分区 + 列裁剪（范围扫描快）。所以事实表优化主靠分区（如按 dt 分区），不靠传统索引。

### 第四层：方案权衡

**Q：星型模型和雪花模型怎么选？维度表要不要再拆细（规范化）？**

权衡"查询性能 vs 存储冗余和一致性"。星型模型——维度表不拆，冗余但 join 少、查询快，适合高频查询的数仓场景。雪花模型——维度表再拆（如城市表拆成城市-省份-国家），规范化好、冗余少，但 join 多、查询慢，适合存储敏感或一致性要求高的场景。经验上 OLAP 数仓优先星型（查询性能优先），OLTP 业务库用 3NF 雪花（一致性优先）。

**Q：为什么不把缓慢变化维度（SCD）直接覆盖更新，而要保留历史版本？**

直接覆盖会丢失历史。如果司机从"快车"改成"专车"，直接覆盖后查历史订单的车型都是"专车"，统计失真。SCD Type 2 保留历史版本——加一行新记录，标记有效期（start_date/end_date），查某时刻的车型用时间条件过滤。这是审计和分析的需要。代价是维度表行数增长（每次变更加一行），但对分析准确性必要。

### 第五层：验证与沉淀

**Q：怎么衡量事实表和维度表设计是否合理？**

三个指标：1) 查询性能——典型 BI 查询的响应时间（应该 < 10s）；2) 数据一致性——事实表外键在维度表的命中率（应该 > 99.9%，缺失是维度同步问题）；3) 冗余度——维度表的规范化程度（星型 vs 雪花的存储成本）。沉淀为维度建模规范：事实表只存外键和度量、维度表存描述性属性、SCD Type 2 处理缓慢变化、按业务事件选事实表粒度。
