# 向量检索召回原理：怎么通过一个向量就能找到对应的文本或文件？（小红书Java一面）

## 一、整体流程

```
用户提问："怎么部署Java应用？"
         │
         ▼
┌─────────────────┐
│  Embedding模型   │  (如 text-embedding-ada-002)
│  文本 → 向量      │  "怎么部署Java应用？" → [0.12, -0.34, 0.56, ..., 0.78]
└────────┬────────┘  维度通常 768 / 1024 / 1536
         │
         ▼
┌─────────────────┐
│  向量数据库       │  (Milvus/Pinecone/FAISS)
│  在已有向量中     │
│  搜索最相似的     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Top-K 召回       │  返回最相似的K个文档片段
│  + rerank 重排   │  (如bge-reranker精排)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  LLM 生成回答     │  用户问题 + 召回片段 → LLM → 答案
└─────────────────┘
```

## 二、Embedding：文本变向量

### 为什么能把语义编码成向量？

```
训练目标：语义相近的文本，向量距离相近

Word2Vec 训练示意：
  "Java" 附近常出现 "编程", "JVM", "Spring"
  "Python"附近常出现 "编程", "AI", "Django"

经过训练后向量空间：
  Java  = [0.8, 0.1, -0.3, ...]
  Python= [0.7, 0.2, -0.2, ...]  ← 和Java相近（都是编程语言）
  
  编程  = [0.9, 0.15, -0.25, ...] ← 也相近

  ⚡ 魔法：向量之间的距离 = 语义相似度
```

### Transformer Embedding（现代方案）

```
输入: "怎么部署Java应用？"
  ↓ Tokenize
  [CLS] 怎 么 部 署 Java 应 用 [SEP]
  ↓ Transformer Encoder (Self-Attention)
  各token的隐状态向量
  ↓ Pooling (取[CLS]或平均池化)
  sentence_embedding = [0.12, -0.34, ..., 0.78]  (768维)
```

## 三、相似度计算

```
余弦相似度 (Cosine Similarity) — 最常用：
  sim(A, B) = (A · B) / (|A| × |B|)
  
  值域: [-1, 1]，越接近1越相似
  
欧氏距离 (L2 Distance)：
  dist(A, B) = √(Σ(Ai - Bi)²)
  
  值越小越相似

点积 (Dot Product / 内积)：
  dot(A, B) = Σ(Ai × Bi)
  
  归一化后等价于余弦相似度
```

## 四、ANN 近似最近邻搜索（核心问题）

### 为什么不暴力遍历？

```
暴力搜索（KNN / Flat）：
  对1000万个文档向量逐一计算相似度 → 排序 → 取TopK
  时间复杂度: O(N × D)，N=1000万, D=768
  每次查询需 ~7.6 billion 次浮点运算 → 太慢

ANN（Approximate Nearest Neighbor）：
  牺牲少量精度（召回率~95%）换取100-1000倍加速
```

### 主流 ANN 算法

```
1. HNSW (Hierarchical Navigable Small World) ⭐最常用
   ┌──────────────────────┐
   │  Layer 2 (稀疏)       │ ← 快速跳到大致区域
   │  ○─────────○          │
   │       │               │
   │  Layer 1 (中等)       │ ← 逐步缩小范围
   │  ○──○──○──○           │
   │       │               │
   │  Layer 0 (全部节点)    │ ← 精确搜索
   │  ○○○○○○○○○○○○         │
   └──────────────────────┘
   类似跳表思想：从顶层快速定位→逐层下钻
   查询复杂度: O(log N)
   Milvus/Faiss/Qdrant 默认支持

2. IVF (Inverted File)
   先聚类(K-Means)将空间划分为nlist个区域
   查询时只搜索最近的nprobe个区域
   速度很快，可调精度

3. PQ (Product Quantization)
   将高维向量切成子向量分别量化压缩
   大幅减少内存占用（1000万×768维 压缩到 1/8）
```

## 六、关键指标

| 指标 | 含义 | 目标 |
|------|------|------|
| 召回率 (Recall) | 召回的正确文档/总正确文档 | >95% |
| QPS | 每秒查询数 | >1000 |
| 延迟 (Latency P99) | 99%请求的响应时间 | <50ms |
| 向量维度 | Embedding输出维度 | 768/1024/1536 |
