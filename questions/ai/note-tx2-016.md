# 【某讯面经】业务场景：某讯客服中心多智能体落地，设计思路

## 一、业务背景与目标

```
某讯客服中心场景：
  日均咨询量：百万级
  现状：80% 人工处理，成本高、响应慢
  目标：用多智能体自动化解决 60%+ 问题，人工兜底

核心指标：
  - 自动解决率（不需要转人工的比例）
  - 平均响应时间
  - 用户满意度（CSAT）
  - 单会话成本（token 消耗 + 计算）
```

## 三、设计思路

### 1. 意图分类（路由的关键）

```python
def classify_intent(query):
    # 第一层：规则匹配（快、便宜，处理明确的）
    rules = {
        r"退款|退钱": "refund",
        r"查询订单|查单": "order_query",
        r"转人工|找客服": "escalate",
    }
    for pattern, intent in rules.items():
        if re.search(pattern, query):
            return intent
    
    # 第二层：LLM 分类（语义，处理模糊的）
    prompt = f"""
    用户咨询：{query}
    请分类意图：faq / order_query / refund / escalate / other
    只输出类别名。
    """
    return llm.invoke(prompt)  # 用小模型，便宜快
```

**为什么混合**：规则快且免费（处理 70% 明确意图），LLM 处理 30% 模糊意图。

### 2. 各子 Agent 职责

| Agent | 触发条件 | 工具 | 解决率目标 |
|-------|---------|------|-----------|
| **FAQ Agent** | 一般咨询 | RAG（产品文档/政策） | 70% |
| **工单 Agent** | 订单查询 | DB查询 | 90% |
| **业务 Agent** | 退款/改签 | 业务API（需权限） | 60% |
| **升级 Agent** | 转人工/超能力 | 通知系统 | - |
| **质检 Agent** | 异步抽检 | LLM-as-Judge | - |

### 3. 子 Agent 无状态化

```
子 Agent 不存会话状态，状态全在共享存储：
  - 会话上下文：Redis（session_id 维度）
  - 工单数据：Postgres
  - 对话历史：向量库（召回用）

好处：
  - 子 Agent 可水平扩展（无状态）
  - 任一 Agent 挂了不影响其他
  - 新增 Agent 只需订阅消息总线
```

### 4. 关键约束：成本/延迟/质量

```
成本：
  - 多 Agent 调用 = 多次 LLM 调用 = token 翻倍
  - 简单问题（FAQ）用单 Agent，不用多 Agent
  - 小模型做意图分类，大模型做生成
  
延迟：
  - 串行调用累积延迟（Orchestrator → 子Agent → 工具）
  - 用 SSE 流式输出，让用户边等边看
  - 并行调用（多个子Agent 同时跑）
  
质量：
  - 自动解决不了的转人工（兜底）
  - 质检 Agent 抽检，发现 bad case 改进
  - 用户反馈（thumbs up/down）闭环
```

## 四、分阶段上线策略

```
[阶段1] MVP（1-2月）
  - 单 Agent + RAG（FAQ）
  - 解决简单咨询（20% 自动化）
  - 目标：验证可行性

[阶段2] 多 Agent（2-3月）
  - 加意图分类 + 工单/退款 Agent
  - 解决业务办理（40% 自动化）
  - 目标：提升解决率

[阶段3] 质检+优化（持续）
  - 加质检 Agent（抽检+改进）
  - 加用户反馈闭环
  - 加成本优化（小模型/缓存）
  - 目标：60% 自动化 + 降本
```

**为什么分阶段**：一次性上多 Agent 风险大（架构复杂+质量难控），分阶段逐步验证。

## 五、衡量多智能体相比单 Agent 的提升

| 指标 | 单 Agent | 多 Agent | 提升 |
|------|---------|---------|------|
| 自动解决率 | 30% | 60% | +30% |
| 平均响应时间 | 8s | 5s | -37%（专业Agent更快） |
| 单会话成本 | 1000 token | 1500 token | +50%（但人力成本大降） |
| 用户满意度 | 75% | 85% | +10% |

**关键洞察**：多 Agent token 成本上升，但人力成本下降更多（综合成本降）。

## 六、加分点

- 说出 **意图分类用规则+LLM 混合**：规则快免费处理明确意图，LLM 处理模糊意图
- 说出 **分阶段上线**：不一次性上多 Agent，MVP 验证再扩展
- 说出 **多 Agent 的真实成本**：token 上升但人力下降，综合成本降

## 七、雷区

- ❌ "所有咨询都走多 Agent" → 简单问题杀鸡用牛刀，成本高
- ❌ "不转人工硬撑" → 解决不了的问题硬答，用户体验差
- ❌ "一次上线全套" → 架构复杂+质量难控，容易翻车

## 八、扩展

- **混合人机协作**：AI 处理 80%，人工处理 20%（复杂/高价值），AI 辅助人工（推荐答案）
- **客服知识库自动更新**：质检 Agent 发现的知识缺口自动补充到 RAG
- **多模态客服**：图片（截图报错）+ 语音（电话）+ 文字，统一接入

## 记忆要点

- 核心架构是编排器+子Agent：规则快而LLM准，混合做意图路由分发
- 子Agent无状态化：会话上下文存Redis，数据存PG，保障水平扩展
- 成本和延迟控制：简单问题用单Agent，小模型分类大模型生成
- 辅以SSE流式输出和异步质检Agent，优化体验与质量


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：客服中心用多智能体落地，为什么不直接用一个"全能 Agent"处理所有问题？**

全能 Agent 的能力会相互干扰。一个 prompt 包含工单查询 + 退款处理 + 知识问答 + 升级投诉的能力，LLM 的注意力被稀释——每项能力的准确率都下降（如 tool_call_success_rate 从 90% 降到 70%）。专业分工（每个子 Agent 专注一个能力域）能让每个 Agent 的 prompt 聚焦，准确率更高。动机是"用专业分工换准确率"。客服场景的意图明确（查单/退款/投诉），按意图路由到专业 Agent 效果最优。

### 第二层：证据与定位

**Q：客服中心上线后，意图分类错误（用户要退款被分到查询），怎么定位是分类模型不行还是意图定义不清？**

看意图分类的 input 和 output。1) 分类模型——如果用户表述清晰（"我要退款"）但仍分错，是模型能力问题；2) 意图定义——如果意图边界模糊（如"我的订单有问题"既可能是查询也可能是退款），分类模型再强也难分。区分方法：统计错误 case 的特征——如果错误集中在"模糊表述"，是意图定义问题（要细化或加澄清步骤）；如果错误随机分布，是模型问题（要升级分类器）。

### 第三层：根因深挖

**Q：多 Agent 客服的成本高（每个意图一个 Agent），根因是 Agent 数太多还是每个 Agent 调用太贵？**

通常是 Agent 数太多导致总调用多。如 5 个意图 Agent + 1 个路由 Agent = 6 个 LLM 调用（串行），单次成本 6 × 单次 LLM 费用。根因是"每个请求都过多个 Agent"。解法：1) 合并相似意图（如查询和退款状态合并为"订单服务"）；2) 用小模型做路由（1.5B 分类意图，便宜快），大模型只处理复杂意图；3) 缓存——常见问题的答案缓存，命中缓存不走 Agent。经验上把 Agent 数从 5 降到 3 + 路由用小模型，成本能降 40-60%。

**Q：那为什么不直接用规则路由（关键词匹配意图），避免 LLM 路由成本？**

规则路由适合"意图明确"的场景（如"退款"关键词 → 退款 Agent），但用户表述多样（如"我不要了这个东西，钱退我"，没有"退款"关键词），规则漏召。LLM 路由能理解语义，覆盖更全。经验上：1) 高频明确的意图（如"查订单"）用规则（快且准）；2) 模糊或长尾意图用 LLM 路由（覆盖广）。两者配合——规则优先，规则没命中再走 LLM，平衡成本和覆盖。

### 第四层：方案权衡

**Q：多 Agent 客服的"转人工"兜底，什么时候触发？触发率多少合理？**

按"Agent 置信度 + 任务风险"触发。1) Agent 输出的置信度低（如分类概率 < 0.7）——可能处理不好，转人工；2) 涉及高风险操作（如大额退款）——必须人工确认；3) 用户主动要求人工——直接转。合理的触发率：5-15%。太低（< 5%）可能让 Agent 处理了不该处理的（用户不满）；太高（> 20%）说明 Agent 能力不足（没省人力）。触发率是 Agent 成熟度的反向指标，随迭代应下降。

**Q：为什么不直接让 Agent 处理所有问题（不转人工），最大化自动化率？**

因为某些问题 Agent 处理不好或不应处理。1) 复杂情绪问题（如愤怒投诉）——Agent 缺乏共情，可能激化矛盾，转人工安抚；2) Agent 不确定的问题——硬答可能给错误信息，转人工更安全；3) 高风险操作——Agent 自动退款可能出错，人工把关。纯追求自动化率会牺牲用户体验和安全性。合理的转人工是"兜底保障"，不是"自动化失败"。

### 第五层：验证与沉淀

**Q：怎么衡量多 Agent 客服系统的效果，证明它比单 Agent 或纯人工更优？**

对比三个 baseline：1) 纯人工客服——多 Agent 应该处理 80%+ 的请求（自动化率），只 20% 转人工，省人力；2) 单 Agent——多 Agent 的 task_success_rate 应该更高（专业分工）；3) 用户满意度——多 Agent 不能低于单 Agent（专业度提升体验）。核心指标：自动化率（> 80%）、task_success_rate（> 85%）、用户满意度（> 4.0/5）、转人工率（< 15%）。沉淀为客服 Agent 迭代规范：意图分类准确率、各 Agent 的成功率、转人工归因分析。
