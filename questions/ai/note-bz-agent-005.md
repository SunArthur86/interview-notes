# 企业级 Agent 如何搭建？必备组件有哪些？

## 一、企业级 vs Demo 的区别

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ Demo               │ 企业级                  │
├──────────────┼──────────────────┼──────────────────────┤
│ 并发          │ 单用户             │ 千万级并发              │
│ 可用性        │ 跑通即可           │ SLA 99.9%+             │
│ 容错          │ 崩了重启           │ 自动降级+故障转移        │
│ 安全          │ 无                 │ 权限/审计/脱敏/防注入    │
│ 可观测        │ print日志          │ 全链路Trace+监控告警     │
│ 成本          │ 不计               │ 每次调用都要算钱         │
│ 多租户        │ 无                 │ 用户/组织隔离            │
│ 迭代          │ 手动改             │ A/B测试+灰度+回滚        │
└──────────────┴──────────────────┴──────────────────────┘
```

## 二、六大必备组件

### 组件 1：编排引擎（Orchestrator）— 大脑

```python
class Orchestrator:
    """Agent的大脑，负责调度一切"""
    def run(self, goal, context):
        # 1. 意图理解
        intent = self.understand(goal)
        # 2. 任务规划
        plan = self.planner.decompose(intent)
        # 3. 循环执行
        for step in plan:
            result = self.execute_with_recovery(step)
            if result.need_replan:
                plan = self.planner.replan(plan, result)
        # 4. 结果聚合
        return self.aggregate(plan.results)
```

**关键能力：** 任务分解、路由决策、失败重规划、结果聚合、循环控制。

### 组件 2：记忆模块（Memory）— 状态

```
分层记忆架构：
├── Working Memory（工作记忆）
│   = 当前对话上下文（LLM可见）
├── Episodic Memory（情景记忆）
│   = 历史执行轨迹（用于复盘和重规划）
├── Semantic Memory（语义记忆）
│   = 知识库/RAG（向量检索）
└── User Profile（用户画像）
    = 用户偏好/历史（个性化）

工程要点：
- 用户隔离（namespace/user_id过滤）
- 检索召回（embedding + rerank）
- 遗忘机制（TTL + LRU + 重要性衰减）
```

### 组件 3：工具层（Tools）— 手脚

```python
# 企业级工具要有完整规范
class Tool:
    name: str               # 唯一标识
    description: str        # LLM看的描述
    parameters: Schema      # 参数JSON Schema
    permissions: List[str]  # 谁能调用
    rate_limit: int         # 调用频率限制
    timeout: int            # 超时时间
    retry_policy: dict      # 重试策略
    fallback: str           # 降级方案

# 工具治理（30+工具时的管理）
- 工具检索：用RAG按需召回工具描述（而非全塞给LLM）
- 版本管理：工具升级不影响历史调用
- 权限矩阵：不同角色可用不同工具
```

### 组件 4：知识库（RAG）— 参考书

```
企业知识接入：
├── 文档库（产品手册/SOP/FAQ）
├── 数据库（业务数据/API结果缓存）
├── 历史会话（客服记录/工单）
└── 实时数据（新闻/行情/库存）

RAG Pipeline：
文档加载 → 分块(Chunking) → 向量化(Embedding)
→ 存入向量DB → 查询时Embedding+Rerank → 注入Prompt
```

### 组件 5：安全层（Safety）— 免疫系统

```python
def security_check(request, action):
    # 1. 输入侧：防Prompt注入
    if detect_injection(request):
        return reject("检测到注入攻击")
    # 2. 权限校验
    if not has_permission(user, action):
        return reject("无权限")
    # 3. 内容合规
    if is_sensitive(action.params):
        return reject("涉及敏感信息")
    # 4. 高危操作
    if action.is_dangerous():  # 删除/支付/外发
        if not await human_approve(action):
            return reject("人工未授权")
    # 5. 输出侧：防泄露
    output = execute(action)
    return redact_sensitive(output)  # 脱敏
```

## 四、面试加分点

1. **强调"工程化"**：企业级 Agent 的难点不在智能，而在工程化——稳定、安全、可观测、可扩展
2. **提"LLM Gateway"**：模型路由层是多模型/降级/成本控制的关键组件
3. **可观测是上线前提**：没有 Trace 和监控的 Agent 不敢上生产，因为概率性故障无法定位

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：企业级 Agent 你说要有六大组件（编排、记忆、工具、知识库、安全、可观测），但 Demo 级只有编排+工具就能跑，为什么企业级必须加那四个？**

因为企业级要解决 Demo 级忽略的"可靠性、安全性、可维护性"。Demo 跑 happy path 就行，企业要处理 edge case：1）记忆——跨会话/跨用户的状态保持，没记忆每次对话从零开始，用户体验差；2）知识库——LLM 知识有截止日期，企业私有数据（产品文档、历史工单）要 RAG 注入，否则幻觉；3）安全——企业场景有合规要求（PII 脱敏、prompt injection 防护、工具权限），Demo 没安全会被攻击；4）可观测——线上故障要能定位（trace 日志、指标监控），Demo 黑盒跑出问题无法排查。这四个不是"锦上添花"，是"上线必需"。

### 第二层：证据与定位

**Q：企业级 Agent 上线后，用户反馈"答非所问"，你怎么从六大组件里定位问题？**

按用户 trace 逐环节查。1）编排——看 LLM 的 Thought 是否理解对意图（如果意图理解就错了，是编排/prompt 问题）；2）知识库——看检索的文档是否相关（如果检索召回的是无关文档，是 chunking/embedding/检索策略问题）；3）工具——看 tool_call 是否选对（选错工具或参数错，是工具描述/决策问题）；4）记忆——看是否引用了错误的历史记忆（记忆污染）；5）安全——看是否被安全中间件误拦截/改写（如敏感词过滤误杀）；6）可观测——通过 trace 日志还原整条决策链。80% 的"答非所问"出在前三个（编排理解错、知识库检索差、工具选错），后三个是少见但难排查的。

### 第三层：根因深挖

**Q：六大组件里，哪个组件的 bug 最难排查？你踩过最深的坑是什么？**

可观测层的"隐性错误"最难排查。显性错误（Agent 崩溃、工具调用失败）trace 日志直接能看到。隐性错误是"Agent 跑完了但答案微妙地错"——如记忆检索召回了一条过时的记忆（半年前的产品信息），LLM 基于它生成了错误答案，但整个流程看起来"正常"。这种错误的根因在记忆的"时效性管理"（没有 TTL 淘汰过时记忆）。踩过最深的坑：Agent 的知识库 chunking 切分不当，把一个完整的"产品定价规则"切成两半，检索只召回一半导致答案不完整。排查花了三天——因为答案"看起来对"（引用了产品名），只是漏了关键约束。教训：可观测不仅要记 trace，还要记"检索召回的完整 chunk 内容"供审计。

**Q：既然记忆的"过时信息污染"是难点，为什么不设计成"记忆全部带时间戳，检索时按时间衰减加权"，简单可靠？**

时间衰减加权是必要的但不充分。问题在于"什么算过时"是任务相关的——产品定价 1 个月就过时，但用户偏好（喜欢简洁风格）半年不过时，统一的时间衰减会错误地降低用户偏好的权重。正确做法是分类记忆 + 差异化 TTL：1）事实型记忆（产品信息、订单状态）——短 TTL（7-30 天）+ 强制刷新（查询时优先查实时 API，记忆只做缓存）；2）偏好型记忆（用户习惯、沟通风格）——长 TTL（半年-永久），不主动淘汰；3）情景型记忆（某次对话的具体内容）——中 TTL（90 天）。检索时不是单纯时间衰减，而是"类型 + 时间 + 相关性"三维加权。这套比单一时间衰减复杂，但能避免"事实过时"和"偏好误删"两个极端问题。

### 第四层：方案权衡

**Q：六大组件都要做，开发成本很高。能不能先用开源框架（如 LangChain/Dify）的现成组件，而非自研？**

分组件决策。1）编排+工具+记忆——用开源框架（LangChain/LangGraph），这些是通用能力，自研重复造轮子且不如开源成熟；2）知识库——RAG pipeline 用 LlamaIndex（专注 RAG），但企业私有数据的 chunking/embedding 策略要自研调优（开源默认策略效果一般）；3）安全——必须自研或用企业级安全产品，开源框架的安全能力弱（prompt injection 防护、企业合规审计），不能直接用；4）可观测——用 LangSmith/自建（trace 日志要对接企业监控体系）。所以策略是"通用组件用开源 + 业务/安全组件自研"，避免全自研（成本高）和全开源（安全/业务适配差）两个极端。

**Q：企业级 Agent 的"智能性 vs 可靠性"矛盾，你实际怎么平衡？是否为了可靠性牺牲智能性？**

不牺牲智能性，而是"分层保障可靠性"。1）LLM 决策层——保持智能性（用强模型+灵活 prompt），让 Agent 能处理复杂/长尾任务；2）安全兜底层——对高风险动作（如删数据、支付）加规则硬约束（不管 LLM 怎么决策，执行前必须过白名单校验），智能性不影响安全；3）关键输出审核——对面向用户的最终答案，用第二个 LLM 或规则做审核（如医疗 Agent 的诊断建议要过医学规则校验），智能生成+事后校验。这样智能性在"探索和生成"阶段充分发挥，可靠性在"执行和输出"阶段兜底，两者不矛盾。纯牺牲智能性（如全规则化）就退化成传统系统了，失去 Agent 价值。

### 第五层：验证与沉淀

**Q：你怎么证明企业级 Agent 的六大组件都"真在工作"，而不是"配置了但没生效"（如安全中间件被绕过）？**

每个组件做独立的健康检查 + 渗透测试。1）安全——做红队测试（构造 prompt injection 攻击，看是否被拦截；尝试调危险工具，看权限是否生效）；2）记忆——造一个有历史记忆的测试用户，看 Agent 是否引用记忆（不引用说明记忆检索没生效）；3）知识库——查一个知识库里有的文档内容，看是否检索到（检索不到说明 RAG 没生效）；4）可观测——触发一次错误，看 trace 日志和告警是否产生（没产生说明监控没接上）。每个组件配一个"探针任务"（专门验证该组件是否工作的测试用例），定期（如每天）自动跑一遍，任一组件失效告警。

**Q：企业级 Agent 的六大组件架构怎么沉淀成团队的平台能力，让每个新 Agent 不重新搭？**

建一个 Agent 平台：1）组件市场——六大组件各做成标准服务（编排服务、记忆服务、RAG 服务、安全服务、监控服务），新 Agent 通过配置接入而非重新开发；2）Agent 模板——按业务类型（客服/编程/分析）提供预置模板，已配好六大组件的默认参数，新人改业务逻辑即可；3）上线流水线——自动跑五项硬性检查（完成率/稳定性/成本/安全/可观测）+ 六组件探针，全绿才能上线；4）运维 dashboard——统一监控所有 Agent 的健康度（成功率、延迟、成本、安全事件）。这套平台让"搭企业级 Agent"从"60 人天"降到"5 人天"，团队聚焦业务逻辑而非重复搭基础设施。
