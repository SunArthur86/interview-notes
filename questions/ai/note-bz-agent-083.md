# 大模型推理性能优化方案？

## 一、推理性能瓶颈

```
LLM推理的瓶颈：

1. 显存带宽限制(Memory-Bound)
   生成每个token需读取全部权重
   7B模型FP16需14GB显存
   → 带宽是瓶颈，不是算力

2. KV Cache显存占用
   每个token的Key/Value都要存
   长上下文(100K)的KV Cache可能占几十GB

3. 批处理效率
   不同请求长度不同，简单批处理浪费GPU

4. 自回归串行
   必须逐token生成，无法并行
```

## 三、引擎优化：vLLM（最重要）

```python
# vLLM的两大核心技术：

# 1. PagedAttention（解决KV Cache显存碎片）
"""
传统：每个请求预分配最大KV Cache空间
      → 大量浪费（实际用不完）
      → 显存碎片，无法服务更多请求

PagedAttention：像OS虚拟内存一样分页管理
  - KV Cache分成固定大小的block(如16个token一块)
  - 按需分配，不预分配
  - 显存利用率从~60%提到~95%
  - 可服务更多并发请求
"""

# 2. 连续批处理（Continuous Batching）
"""
传统批处理：等一批请求都完成才开始下一批
  → 长请求拖慢短请求，GPU空闲

连续批处理：动态插入/移除请求
  - 某请求完成 → 立即移除，插入新请求
  - GPU永远不空闲
  - 吞吐量提升3-5倍
"""

# 使用vLLM
from vllm import LLM, SamplingParams
llm = LLM(model="qwen-7b", tensor_parallel_size=1)
outputs = llm.generate(["问题1", "问题2", "问题3"])  # 高效批量
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说推理是 memory-bound 不是 compute-bound，那为什么还要做连续批处理提高 GPU 利用率？如果瓶颈在带宽，批处理能帮上什么？**

因为批处理提高的是"带宽利用率"不是"算力利用率"。memory-bound 的意思是每次生成 token 都要把权重从显存搬到计算单元，单请求时搬运一次权重只算一个 token，带宽利用率极低（可能 < 5%）。批处理让多个请求共享同一次权重搬运——搬一次权重算 N 个请求的 token，带宽利用率线性提升。所以批处理对 memory-bound 场景恰恰是最有效的优化，它把"每次搬权重算 1 个"变成"每次搬权重算 N 个"，吞吐随 batch size 线性增长（直到带宽或显存打满）。这也是为什么 vLLM 的连续批处理能把吞吐提 3-5 倍。

### 第二层：证据与定位

**Q：你上了 vLLM 后，单请求延迟反而比 HuggingFace Transformers 还高了，为什么？**

这是正常的"延迟换吞吐"现象。vLLM 的连续批处理为了最大化吞吐，会把新请求塞进正在处理的 batch，单请求要等 batch 里的其他请求一起推进，首字延迟（TTFT）可能比 HF 的串行单请求高。定位方法：看吞吐和延迟两个指标——如果吞吐涨了 4 倍但 P99 延迟涨了 30%，这是正常的权衡（对吞吐优先的离线/批量场景值得）。如果吞吐没涨延迟还涨了，是配置问题（比如 max_num_seqs 设太大导致排队、或 gpu_memory_utilization 设太高触发频繁换页）。所以要分清是"正常权衡"还是"配置错误"，看吞吐是否同步提升。

### 第三层：根因深挖

**Q：PagedAttention 你说解决显存碎片，但分页本身有页表管理开销，为什么净收益是正的？**

因为碎片浪费的显存远大于页表管理开销。传统方式为每个请求预分配"最大可能长度"的连续 KV Cache（比如 2048 token 的空间），但大多数请求实际只生成 200 token，预分配的 1800 token 空间全浪费，显存利用率只有 60%。PagedAttention 按需分配（用多少分多少），利用率到 95%，多出来的 35% 显存能多服务一倍的并发请求。页表管理的开销是每次 attention 多一次 block table 查找（几纳秒级），相对省下的显存能多批处理的吞吐提升，净收益是正的。实测 vLLM 吞吐比 HF 高 3-5 倍，说明开销可忽略。

**Q：那为什么不用更小的 block（比如 1 token 一块），利用率不是更高？**

因为 block 太小会让 block table 巨大、查找开销线性增长。block size = 16 时，2048 token 的请求只有 128 个 block，block table 是 128 项的数组，查找快。block size = 1 时，同样请求有 2048 个 block，block table 是 2048 项，每次 attention 要查 2048 次指针，开销不可忽略。而且 GPU 的访存对连续地址友好，block size 太小导致 KV Cache 散落在显存各处，cache miss 率上升。所以 block size 是"利用率 vs 管理开销"的权衡，16 是实测出来的甜区，不是随便定的。

### 第四层：方案权衡

**Q：MoE 你说"参数容量大但推理快"，但 MoE 要把所有专家都加载到显存（Mixtral 8x7B 要 47B 参数的显存），显存占用比同等推理速度的 dense 模型大得多，怎么权衡？**

这是 MoE 的核心权衡：用显存换速度。Mixtral 8x7B 显存占 47B（约 90GB FP16），但每次只激活 13B，推理速度等同于 13B dense 模型。如果用 13B dense 模型，显存只要 26GB 但能力远不如 47B 的 MoE。所以 MoE 适合"要大模型能力但要求推理快"的场景，代价是显存。权衡方法：如果显存够（多卡部署），MoE 是最优（能力强且快）；如果显存紧（单卡），只能用小的 dense 模型或 MoE 的量化版（INT4 的 Mixtral 压到 24GB 能单卡跑）。所以不是 MoE 一定好，是看显存预算够不够。

**Q：为什么不直接用张量并行把大模型拆到多卡，而要搞 MoE 这种复杂架构？**

因为张量并行解决不了"单请求慢"的问题，只解决"放得下"。70B 模型 4 卡张量并行能放下，但每次生成 token 还是要算完 70B 参数（只是拆到 4 卡并行算），单请求延迟由 70B 的计算量决定，依然慢。MoE 的优势是每次只算 13B 参数，单请求延迟接近 13B 模型。而且张量并行有通信开销（每层都要 all-reduce），卡越多通信占比越高，4 卡以上边际收益递减。所以"要单请求快"选 MoE（减计算量），"要放下大模型"选张量并行（分摊显存），两者解决不同问题，大模型服务常两者叠加（MoE + 张量并行）。

### 第五层：验证与沉淀

**Q：你怎么证明 vLLM 的优化在你的业务负载下真的有效，而不是官方 benchmark 的理想场景？**

用业务真实流量做 benchmark。我的做法是录制线上 1 小时的真实请求（含真实的输入输出长度分布、到达模式），用 vLLM 和旧引擎分别回放，对比吞吐/QPS、P50/P99 延迟、显存利用率、单请求成本四个指标。官方 benchmark 通常是固定长度、固定到达率，和真实业务的长度分布（长尾）和突发流量不同，实测才有意义。我之前实测过：官方说 vLLM 快 5 倍，我业务场景（长短混合、有突发）只快 3 倍，因为突发时排队抵消了部分增益。有这个真实数据，容量规划（要多少卡）才准。

**Q：推理优化方案怎么沉淀？**

沉淀成"推理部署配置库"。按"模型 × GPU 型号 × 业务场景"三维建表，每格存最优配置（引擎选型、量化方案、batch size、block size、并行度）+ benchmark 报告。新服务按这三维度查表拿配置，不用重新调。配套一个自动 benchmark 工具——输入模型和 GPU，自动跑各种配置组合输出最优解，避免人工试。每季度更新（新引擎/新 GPU 出来重跑），保持配置库不过时。这样推理优化是平台能力，不是每个团队自己摸索。
