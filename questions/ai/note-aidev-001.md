# AI 项目如何量化业务收益？面试怎么说？

## 量化框架

```
┌─────────────────────────────────────────────────────┐
│            AI项目量化指标体系                        │
├──────────────┬──────────────────────────────────────┤
│  效率指标     │ 处理时间↓、吞吐量↑、首响时间↓        │
├──────────────┼──────────────────────────────────────┤
│  质量指标     │ 准确率↑、幻觉率↓、解决率↑            │
├──────────────┼──────────────────────────────────────┤
│  成本指标     │ 人力成本↓、API成本↓、GPU利用率↑      │
├──────────────┼──────────────────────────────────────┤
│  用户体验     │ 转人工率↓、满意度↑、留存率↑          │
├──────────────┼──────────────────────────────────────┤
│  安全指标     │ 数据泄露事件0、违规率↓                │
└──────────────┴──────────────────────────────────────┘
```

## 如何设计AB实验

```python
# AB实验设计
experiment = {
    "name": "AI客服 vs 规则客服",
    "对照组": "现有规则客服系统",
    "实验组": "AI Agent客服系统",
    "流量分配": "50/50",
    "样本量计算": {
        # 转人工率从35%降到20% → 需要~1200条/组
        # 效果: p=0.05, power=0.8
        "min_sample_per_group": 1200,
        "actual_sample": 50000,  # 远超最小样本量
        "confidence": 0.99
    },
    "核心指标": [
        "转人工率", "解决率", "首响时间", "满意度"
    ],
    "护栏指标": [  # 不能变差的指标
        "投诉率", "安全事件数"
    ],
    "实验周期": "2周",
    "结果": {
        "转人工率": {"before": 0.35, "after": 0.12, "p_value": 0.001},
        "解决率": {"before": 0.62, "after": 0.81, "p_value": 0.002},
    }
}
```

## 效果不达预期怎么办

```
面试加分回答:
"上线后发现解决率只提升了5%(预期20%)，我做了以下排查:

1. 日志分析: 发现80%的失败集中在'退换货'场景
2. 根因定位: 退换货知识库文档过时(2023年的政策)
3. 数据迭代: 更新知识库 + 增加退换货专项训练数据
4. Prompt优化: 增加Few-shot案例覆盖边界情况
5. 二次上线: 解决率从67%提升到82%

关键经验: AI项目不是上线就结束，线上迭代才是大头"
```

## 面试中的常见错误

```
❌ "我做了一个智能客服系统，用了RAG+Agent"
   → 面试官: 效果呢? 量化数据呢?

✅ "智能客服系统，转人工率从35%降到12%，节省60%人力，
    日均处理2万条对话，用户满意度4.5分"
   → 有数据、有对比、有业务价值

❌ "用了LangChain+GPT-4+Milvus"
   → 技术栈不是价值

✅ "选Milvus而非ES是因为需要亿级向量检索，P99<50ms，
    单节点QPS>5000，实测对比ES快3倍"
   → 有选型理由、有性能数据
```

## 记忆要点

- 核心框架：从效率、质量、成本、体验、安全五个维度量化AI收益
- 面试话术：一定要用'AB测试+百分比变化'展示（如：转人工率降30%，处理量升4倍）
- 实验设计：强调对照组设计，说出具体的流量分配、样本量和置信度
- 兜底策略：强调遇指标不及预期时，能通过日志快速定位场景并迭代数据
