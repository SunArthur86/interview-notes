# AI 项目如何量化业务收益？面试怎么说？

├──────────────┬──────────────────────────────────────┤
│  效率指标     │ 处理时间↓、吞吐量↑、首响时间↓        │
├──────────────┼──────────────────────────────────────┤

## 如何设计AB实验

```python
# AB实验设计
experiment = {
    "name": "AI客服 vs 规则客服",
    "对照组": "现有规则客服系统",
    "实验组": "AI Agent客服系统",
    "流量分配": "50/50",
    "样本量计算": {
        # 转人工率从35%降到20% → 需要~1200条/组
        # 效果: p=0.05, power=0.8
        "min_sample_per_group": 1200,
        "actual_sample": 50000,  # 远超最小样本量
        "confidence": 0.99
    },
    "核心指标": [
        "转人工率", "解决率", "首响时间", "满意度"
    ],
    "护栏指标": [  # 不能变差的指标
        "投诉率", "安全事件数"
    ],
    "实验周期": "2周",
    "结果": {
        "转人工率": {"before": 0.35, "after": 0.12, "p_value": 0.001},
        "解决率": {"before": 0.62, "after": 0.81, "p_value": 0.002},
    }
}
```

## 效果不达预期怎么办

```
面试加分回答:
"上线后发现解决率只提升了5%(预期20%)，我做了以下排查:

1. 日志分析: 发现80%的失败集中在'退换货'场景
2. 根因定位: 退换货知识库文档过时(2023年的政策)
3. 数据迭代: 更新知识库 + 增加退换货专项训练数据
4. Prompt优化: 增加Few-shot案例覆盖边界情况
5. 二次上线: 解决率从67%提升到82%

关键经验: AI项目不是上线就结束，线上迭代才是大头"
```

## 面试中的常见错误

```
❌ "我做了一个智能客服系统，用了RAG+Agent"
   → 面试官: 效果呢? 量化数据呢?

✅ "智能客服系统，转人工率从35%降到12%，节省60%人力，
    日均处理2万条对话，用户满意度4.5分"
   → 有数据、有对比、有业务价值

❌ "用了LangChain+GPT-4+Milvus"
   → 技术栈不是价值

✅ "选Milvus而非ES是因为需要亿级向量检索，P99<50ms，
    单节点QPS>5000，实测对比ES快3倍"
   → 有选型理由、有性能数据
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你的 AI 客服项目说"转人工率降 30%"，为什么这个指标能代表项目价值？依据是什么？**

因为转人工率直接绑定成本和体验。每次转人工意味着 AI 没解决、要付人工客服工资（每次 ~5-10 元）且用户要等待（体验差）。转人工率从 40% 降到 28%，按日均 10 万会话算，每天少转 1.2 万次，省 ~6-12 万/天人工成本，且用户等待减少。选这个指标的依据是它同时反映"AI 能力提升"（能解决更多问题）和"业务价值"（省钱 + 体验好），比"准确率"这种技术指标更能说服业务方。

### 第二层：证据与定位

**Q：转人工率降了，但用户满意度（CSAT）也降了，你怎么定位是哪里出了问题？**

这是典型的"压转人工压过头了"——AI 强行兜住本该转人工的复杂 case，解决了"量"但没解决"质"，用户被迫和解决不了问题的 AI 多轮纠缠，体验更差。定位要分场景看：按会话主题分组，看哪些场景的 CSAT 降得最多——大概率是原本就该转人工的复杂场景（如投诉、退款纠纷）被 AI 硬扛了。解法是调高这些场景的转人工阈值（或加意图识别，复杂意图直接转），宁可转人工率回升 5% 也要保 CSAT。

### 第三层：根因深挖

**Q：你发现 CSAT 下降集中在"退款投诉"场景，AI 的回答其实是对的（按流程），但用户不满。根因是什么？**

根因是"回答正确"不等于"解决满意"。退款投诉的用户情绪是愤怒/焦虑，AI 按流程回答"3-5 个工作日退款"在事实层面没错，但缺乏情绪安抚，用户感受是"被敷衍"。这是情感维度的问题，不是知识维度。治本是在 prompt 里加入情绪识别和共情模板（先共情"理解您的焦急"再给方案），或在 CSAT 低的会话尾部主动安抚。技术答案对但体验差，根因在对话策略而非知识库。

**Q：那为什么不直接让人工处理所有投诉类 case，AI 只做简单查询，省得 CSAT 翻车？**

因为成本和规模。投诉类 case 占比可能 30%+，全转人工意味着人工客服要扩容 3 倍，成本爆炸。AI 的价值是"过滤出真正需要人工的复杂投诉，自动化简单投诉"。正确做法是分层：简单投诉（如标准退款流程）AI 处理 + 情绪模板兜底；复杂投诉（如多次投诉、纠纷）识别后转人工。关键是识别准确率——把"简单但用户情绪激烈"的 case 也转人工（因为这类最容易 CSAT 翻车），而非纯按业务复杂度分流。

### 第四层：方案权衡

**Q：你用 A/B 测试证明效果，实验组对照组各 50% 流量跑 2 周。为什么是 2 周不是 1 周？流量分配怎么定？**

2 周是为了覆盖周期性波动（如工作日 vs 周末、发薪日 vs 平日），1 周可能恰好撞上特殊周期导致偏差。流量分配 50/50 是为了最快达到统计显著性——样本量公式 $n = Z^2 p(1-p)/e^2$，假设转人工率 baseline 40%、想检测 5% 的绝对差异（e=0.05）、置信度 95%（Z=1.96），每组需要约 7300 样本，按日均 10 万会话 50% 流量 1 天就有 5 万，2 周远超所需。如果是高风险改动（可能伤体验），先 5%/95% 小流量灰度，确认无 regression 再放 50%。

**Q：为什么不直接全量上线对比上线前的历史数据，省得搞 A/B 实验？**

全量上线无法排除时间混淆——上线后的转人工率降了，可能是 AI 变好，也可能是这段时间用户问题变简单了（如某产品下架了相关咨询减少）。A/B 实验对照组同时跑，两组面对相同的用户分布和时间因素，差异只来自 AI 改动，因果干净。历史对比是"前后"，A/B 是"同时"，后者排除了时间因素。没有 A/B 的"提升"不可信，面试官会质疑你把自然波动归功于自己。

### 第五层：验证与沉淀

**Q：你怎么在面试里把这套量化讲清楚，让面试官觉得你"真的懂"而不是背数字？**

三段式：先说北极星指标（转人工率 -30%，绑定成本），再说关键过程指标（首问解决率 +15%、平均轮次 -1.2 轮），最后说怎么证明（A/B 实验 50/50 流量 2 周，p<0.05 显著）。关键是要能回答"如果指标没达标你怎么 debug"——讲一个真实的 regression case（如 CSAT 降了，定位到某场景，调阈值修复）。能让面试官追问的细节越深、你答得越具体（如样本量计算、p 值、分场景归因），越证明是真做过而非编的。

**Q：这套量化方法论怎么沉淀成团队的项目评估规范？**

固化成"AI 项目上线 checklist"：必须有北极星指标（绑业务价值）、必须有 A/B 实验设计（流量、周期、样本量）、必须有过程指标（定位 regression 用）、必须有兜底预案（指标不达标如何回滚或迭代）。沉淀"各业务线的指标对照表"（客服看转人工率/CSAT、推荐看 CTR/GMV、风控看拦截率/误杀率），新项目按模板选指标。把"A/B 实验报告模板""regression 定位 SOP"做成团队文档。
