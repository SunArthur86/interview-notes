# Hermes Agent 七步精装，让智能体真正会干活？

## 一、七步精装总览

```
┌──────────────────────────────────────────────────┐
│              Agent 七步精装                         │
├──────────────────────────────────────────────────┤
│                                                    │
│  Step 1: 角色定义（Who）                           │
│    明确Agent的身份、能力边界、语气风格              │
│                                                    │
│  Step 2: 工具配置（What）                          │
│    选对工具、写好描述、设置权限                     │
│                                                    │
│  Step 3: 记忆设计（Remember）                      │
│    短期/长期/用户画像，写入和检索策略               │
│                                                    │
│  Step 4: 安全防护（Safe）                          │
│    权限/审计/防注入/高危确认                       │
│                                                    │
│  Step 5: 评估闭环（Evaluate）                      │
│    指标定义+测试集+持续监控                        │
│                                                    │
│  Step 6: 成本控制（Cost）                          │
│    模型路由+缓存+Token优化                         │
│                                                    │
│  Step 7: 持续迭代（Iterate）                       │
│    Bad Case收集+用户反馈+版本更新                  │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、各步详解

### Step 1：角色定义

```python
# 精装的角色定义（非简单一句"你是助手"）
AGENT_PERSONA = {
    "identity": "你是XX公司的智能客服",
    "expertise": ["订单查询", "退换货", "产品咨询"],
    "boundaries": {
        "can_do": ["查询", "建议", "发起流程"],
        "cannot_do": ["直接退款（需主管审批）", "承诺补偿"],
    },
    "tone": "专业、友善、简洁",
    "language_style": "口语化但不随意，用'您'",
    "fallback": "不确定或超范围时，诚实告知并转人工"
}
```

### Step 2：工具配置

```python
# 精装的工具（不只是注册，还有治理）
class ToolConfig:
    tools = [
        Tool(name="query_order", desc="...", permissions=["user"]),
        Tool(name="refund", desc="...", permissions=["admin"], 
             require_approval=True),  # 高危需审批
    ]
    
    # 工具路由（按上下文动态加载）
    def get_tools(self, context):
        if context.intent == "order":
            return [self.tools["query_order"]]
        # 而非全部塞给LLM
```

### Step 3：记忆设计

```python
# 精装的记忆系统
class MemoryConfig:
    short_term = {
        "window": 10,  # 最近10轮
        "overflow_strategy": "summarize"  # 超了摘要
    }
    long_term = {
        "storage": "vector_db",
        "write_threshold": 0.6,  # 重要性>0.6才记
        "ttl": 30 * 24 * 3600,   # 30天过期

    }
    profile = {
        "storage": "redis",
        "fields": ["name", "preferences", "history_summary"]
    }
```

class SecurityConfig:

## 三、精装前后对比

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 精装前(Demo)        │ 精装后(生产)            │
├──────────────┼──────────────────┼──────────────────────┤
│ 角色          │ "你是助手"          │ 明确身份/边界/语气       │
│ 工具          │ 全部塞给LLM        │ 动态路由+权限治理        │
│ 记忆          │ 无或简单窗口        │ 分层+检索+遗忘          │
│ 安全          │ 无                 │ 多层防护+审计           │
│ 评估          │ 手动看几个case      │ 自动化指标+测试集        │
│ 成本          │ 不计               │ 模型路由+缓存+预算       │

└──────────────┴──────────────────┴──────────────────────┘
```

## 四、面试加分点

1. **七步成体系**：角色→工具→记忆→安全→评估→成本→迭代，覆盖生产级全要素
2. **评估是核心**：没有评估闭环的 Agent 无法持续改进——强调"可度量"
3. **"精装"比喻好**：Demo 是毛坯，生产要精装，每步都不可省

## 记忆要点

- 七步精装：角色→工具→记忆→安全→评估→成本→迭代。
- 核心三要素：明确定位与边界(角色)，配置治理与路由(工具)，分级与摘要隔离(记忆)。
- 运行保障三支柱：防注入与高危确认(安全)，模型路由与缓存(成本)，指标监控与BadCase收集(评估与迭代)。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 七步精装（角色+工具+记忆+安全+评估+成本+迭代），为什么不直接拿个 demo Agent 上线，非要"精装"？**

因为 demo Agent 只在"理想场景"跑通，生产环境会暴露 demo 没考虑的问题。1）角色定义不清——demo 的 prompt 模糊，生产中遇到边界情况（如用户输入歧义）LLM 不知道怎么处理，输出失控；2）工具缺失——demo 只配了核心工具，生产中遇到要查额外信息/异常处理时没工具可用，Agent 卡住；3）记忆无设计——demo 不管理历史，长对话 context 爆炸或关键信息丢失；4）无安全防护——demo 不防 prompt 注入/越权/敏感输出，生产中被攻击或违规；5）无评估——demo 不知自己好不好，无法持续优化；6）无成本控制——demo 不限 token/调用次数，生产中烧钱；7）无迭代机制——demo 上线后不改进，能力停滞。精装就是把这七项补齐，让 Agent 从"能跑"到"生产可用"。

### 第二层：证据与定位

**Q：生产 Agent 上线后用户反馈"不好用"，但反馈很笼统（如"答非所问"），怎么定位是七步中哪一步的问题？**

用 trace 分层定位。1）角色/prompt——看 Agent 的输出是否偏离角色（如客服 Agent 开始闲聊），偏离是角色/prompt 问题；2）工具——看 Agent 是否调了对的工具、工具返回是否正确，调错/工具错是工具配置问题；3）记忆——看长对话是否丢失关键信息（如忘了用户前面说的），丢失是记忆设计问题；4）安全——看是否过度拦截（安全防护太严误伤正常请求）或漏拦截（违规输出），是安全配置问题；5）评估——看这个 case 在评估集里是否有覆盖，没覆盖是评估集不全。定位方法：拿用户反馈的具体 case，回放 trace，找第一个"出问题"的环节。常见根因：prompt 边界没写清（角色问题）、工具不足（工具问题）、context 截断丢了关键（记忆问题）。

### 第三层：根因深挖

**Q：七步里的"记忆设计"很关键（决定 Agent 记什么忘什么），但记忆设计错了（该记的没记/该忘的没忘）会导致什么问题，怎么设计好？**

记忆错位导致两类问题。1）该记没记——关键信息（如用户偏好/任务上下文）没存，长对话中 Agent 反复问/忘记前面说的，体验差；2）该忘没忘——无关信息（如闲聊/过时事实）全留，context 膨胀（费 token+attention 稀释），且旧信息干扰新决策。设计好的方法：1）分层记忆——短期记忆（当前对话的滑动窗口）+ 长期记忆（用户画像/历史任务，持久化）+ 工作记忆（当前任务的关键事实，显式提取）；2）写记忆决策——不是所有信息都记，LLM 判断"这条信息未来用得到吗"，用得到才写长期记忆；3）遗忘策略——短期记忆按时效/相关性淘汰（如超窗口的旧轮次淘汰），长期记忆按访问频率衰减（久不访问的淡化）。目标是"记得关键的，忘掉无用的"。

**Q：七步里有"安全防护"，但安全防护太严会误伤正常请求（如客服 Agent 拒绝回答正常问题），怎么平衡安全和可用性？**

分级防护 + 灰度。1）分级——安全规则按严重程度分（硬违规如涉黄涉政→坚决拦截；软风险如轻微敏感词→提示但不拦截；正常→放行），不同级别不同处理；2）精准规则——安全规则要精准（针对真正的违规模式），而非粗粒度关键词匹配（容易误伤，如"杀手"匹配到游戏讨论）；3）白名单/上下文——对已知正常场景（如客服的订单讨论）设白名单，绕过部分规则；4）灰度+监控——新规则先灰度（只拦截+告警不阻断），观察误伤率，低误伤再转硬拦截；5）申诉通道——误拦截的让用户/运维反馈，快速解封+优化规则。原则：宁可漏一点（软风险）也别大面积误伤（破坏可用性），硬违规才零容忍。

### 第四层：方案权衡

**Q：七步里"评估闭环"要建评估集+持续评估，但建评估集成本高（要标注），怎么平衡评估投入和产出？**

按 Agent 重要性分层投入。1）核心 Agent（高频/高价值，如主客服）——投入建完整评估集（几百到上千 case，覆盖各类场景+边界），持续评估，值得（一个问题影响大量用户）；2）次要 Agent（低频/辅助）——轻量评估（几十 case + 用户反馈驱动），不建大评估集（投入产出不值）；3）评估集增量——不一次性建完，而是从用户反馈/线上 case 持续补充（真实发现的问题比凭空设计的更有价值），逐步积累。原则：评估投入和 Agent 价值匹配，核心 Agent 重投（评估集完善），次要 Agent 轻投（够用即可），不一刀切。

**Q：七步精装后 Agent 变"重"了（七项都要维护），迭代变慢，怎么平衡"精装"和"敏捷迭代"？**

核心精装 + 非核心后补。1）上线前精装核心三步——角色定义（清晰 prompt）、工具配置（核心工具齐全）、安全防护（硬违规拦截），这三项不上线会出大问题（角色乱/卡住/违规）；2）上线后逐步补齐——记忆设计、评估闭环、成本控制、持续迭代，这些可以"上线后根据实际数据优化"（如先上线观察哪些信息该记忆，再设计记忆；先上线收集 case，再建评估集）；3）MVP 思维——第一版精装到"可用"（核心三步+基本安全），快速上线收集反馈，再迭代精装其余。原则：不追求一次精装到位，核心先稳（避免大问题），其余迭代补（基于真实数据）。

### 第五层：验证与沉淀

**Q：你怎么衡量七步精装的效果（比 demo 强多少）？**

对比 demo 和精装版。固定测试集+线上指标：1）任务完成率——精装应显著高于 demo（角色清晰+工具齐全，能力更强）；2）异常处理——精装能处理边界情况（demo 崩溃的 case 精装能优雅处理），看异常 case 通过率；3）安全性——精装的违规输出率应远低于 demo（有安全防护），看违规拦截率/误伤率；4）成本——精装的每次调用 token/工具调用次数（成本控制的效果），应优化；5）用户满意度——精装的评分/投诉率应优于 demo。综合：任务完成率高+异常处理好+安全+成本可控+满意度高 = 精装有效。还要看迭代速度（精装后持续改进的频率），持续迭代说明闭环建立。

**Q：七步精装的方法论怎么沉淀成团队的 Agent 生产 SOP？**

建 Agent 生产流水线：1）精装 checklist——每个 Agent 上线前过七步 checklist（角色/工具/记忆/安全/评估/成本/迭代），核心三步必过，其余按重要性；2）模板/脚手架——提供每步的工具（如 prompt 模板/工具注册框架/记忆模块/安全规则库/评估集模板/成本监控），降低精装成本；3）上线门禁——Agent 过 checklist+评估集（达标率 >阈值）才能上线；4）监控看板——线上 Agent 的七步指标自动采集（完成率/异常率/违规率/成本/满意度），异常告警；5）迭代机制——定期 review（基于监控+用户反馈），持续优化。这套写入团队 Agent 平台 SOP，让"精装 Agent"从"个人经验"变成"标准化生产"。
