# 【字节飞连面经】介绍 Agent 项目整体流程：为什么这么拆？状态怎么存？多 Agent 怎么协作？

## 一、整体流程：五段拆分

```
用户输入
  │
  ▼
[1] 意图识别（LLM）  ── 判断"用户到底想干什么"
  │
  ▼
[2] 计划生成（LLM）  ── 拆成 N 个可执行步骤
  │
  ▼
[3] 工具执行（Workflow，确定性）── 调 API、读 DB、跑脚本
  │
  ▼
[4] 结果聚合（规则模板）── 把多个工具结果拼成统一结构
  │
  ▼
[5] 反思（LLM）      ── 评估"是否完成 / 是否要补刀"
```

**拆分原则**：能 `if-else` 写出来的坚决不交给 LLM。"判断用户是否登录""检查工单类型在白名单"全是 Workflow；"猜用户真实意图""生成派单摘要"交给 LLM。

## 二、失败分支与重试

每个节点都包 `try / retry / fallback` 三段：

| 失败类型 | 处理 |
|---------|------|
| LLM 节点失败 | 换更小的兜底模型（豆包 1.6 失败 → 豆包 lite） |
| 工具节点失败 | 指数退避重试 3 次（1s, 2s, 4s）→ 进人工兜底队列 |
| 业务侧拒绝（权限不足） | 立即终止 + 透传给用户 |

**关键原则：fail loud，不要 fail silent**。失败必须落日志 + 上指标，否则线上出了问题靠猜。

## 三、状态怎么存：双层存储

```
短期会话：Redis Hash
  key = session:{user_id}:{conversation_id}
  字段：current_step, plan_json, tool_history, step_count
  TTL = 30 min

长期日志：Postgres
  表 agent_steps:
    (step_id, parent_step, tool_name, 
     input_json, output_json, status, ts, trace_id)
  → 支持完整回放、bad case 定位、审计
```

**为什么双层**：Redis 撑低延迟读写（每步都要读状态），Postgres 撑可追溯性（出问题能查）。两者双写，Redis 是热数据，Postgres 是冷账本。

## 四、多 Agent 分工 / 通信 / 终止

**Planner + Executor + Critic 三角色**：
- Planner 出计划
- Executor 执行工具
- Critic 评估"是否完成"

**通信方式（关键）**：共享状态对象（LangGraph 里的 `State`），**禁止 Agent 之间直接对话**。直接对话（AutoGen 那种）很容易扯皮——你说做完了，我说没做完，无限循环。

**终止条件三重上限**：
1. 硬上限：最多 N 步（典型 10–15）
2. 软上限：Critic 连续 2 次说"已完成"
3. 异常上限：连续 3 次工具失败

## 五、避免死循环 / 扯皮

- **硬步数上限**：到了强制返回当前最优结果
- **去重**：同工具 + 同参数在同一会话调用过 → 直接返回缓存
- **降权**：连续两次失败的工具，第三次从可选工具列表剔除

## 六、加分点

能画出 LangGraph `State` 的字段表：

```typescript
interface AgentState {
  messages: BaseMessage[];     // 对话历史
  current_plan: PlanStep[];    // 当前计划
  tool_history: ToolCall[];    // 工具调用历史
  step_count: number;          // 步数计数
  should_continue: boolean;    // 是否继续
}
```

提到 `interrupt()` / human-in-the-loop 兜复杂决策。

## 七、雷区

- ❌ "多 Agent 全交给 AutoGen 自己跑" → 立刻被追问"超时和死循环你怎么治"
- ❌ "状态存内存里就行" → 被追问"服务重启怎么办、水平扩展怎么办"
- ❌ 指标没数字（"效果很好"=没效果）

## 八、扩展

- LangGraph 的 `CheckpointSaver` 接口可以接 Redis/Postgres，实现状态持久化和断点续传
- 真正生产级的多 Agent 系统会用 message bus（如 NATS/Kafka）做异步通信，而非同步共享内存
- 反思阶段如果用 LLM，反思结果要**摘要后再写回**，不留原始错误文本，否则会污染上下文

## 记忆要点

- 五段式拆分原则：因为能if-else的确定性逻辑不该花Token，所以意图判断交LLM，执行全走Workflow
- 双层状态存储：Redis Hash存短期会话（低延迟），Postgres存全量日志（可追溯审计），双写保活
- 多Agent协作铁律：必须基于共享状态对象通信，禁止Agent直接对话扯皮引发无限死循环
- 防死循环三上限：硬限步数（10-15步）、软限Critic连续确认、异常限连续失败次数强制熔断

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说"能 if-else 的不交给 LLM"，但意图识别这步你交给了 LLM，意图识别本质也是分类（可以用规则或小模型），为什么这步算"创造性"不交给 Workflow？**

意图识别的边界 case 太多，规则写不全。比如用户说"上次那个帮我处理一下"——没有关键词，要结合上下文（上次的工单是什么）推断意图，这是语义理解不是字面匹配，规则做不到。规则能挡的是明确意图（含"退款"关键词 → 退款意图），挡不住的是指代、省略、口语化表达。所以我用"规则前置 + LLM 兜底"——80% 的高频意图规则秒判（省 Token），剩下 20% 的长尾才走 LLM。意图识别交给 LLM 不是因为它整体不可规则化，是因为那 20% 长尾规则覆盖不了，而这 20% 恰恰是用户体验的关键（挡不住就只能"无法理解"）。

### 第二层：证据与定位

**Q：你说状态用 Redis + Postgres 双写，但双写有一致性问题——Redis 写成功 Postgres 写失败，状态就不一致了。线上怎么定位和处理这种不一致？**

定位靠对账，处理靠"Redis 为准 + Postgres 异步补"。双写不保证强一致（要强一致得用分布式事务，成本太高），所以接受最终一致。具体：写顺序是先 Redis（成功才算这步状态生效）后 Postgres（异步落日志），如果 Postgres 写失败，记到失败队列由后台任务重试补写。定位不一致靠定时对账——每小时抽样对比 Redis 的 current_step 和 Postgres 最新 step 的 status，不一致告警。关键是明确"谁是 source of truth"：运行时读 Redis（低延迟），追溯读 Postgres（全量日志）。Redis 是热数据可以丢（TTL 30min），Postgres 是冷账本不能丢。所以即使短暂不一致，以 Redis 为准继续跑，Postgres 后补，不影响业务正确性。

### 第三层：根因深挖

**Q：你禁止 Agent 直接对话、必须走共享 State，但 AutoGen 这种"Agent 对话"范式很流行，你为什么不用？共享 State 比直接对话好在哪？**

直接对话的根因问题是"无状态扯皮"。Agent A 说"我做完了"，Agent B 说"你没做完"，两者各自有各自的 context，没有一个共同的"事实基础"，于是无限循环（真实坑）。共享 State 解决了"事实基础"——所有 Agent 读写同一个 State 对象（tool_history、step_count、current_plan），A 说"做完"必须把结果写进 State，B 看 State 就知道真做完没，不存在各说各话。另外共享 State 可序列化（存 Redis/Postgres），支持断点续传和回放；直接对话的消息流序列化困难（消息嵌套、引用关系复杂）。所以共享 State 好在：有事实基础（止扯皮）、可持久化（可恢复）、可观测（调试时看 State 一目了然）。这是工程可控性的根本差别。

**Q：那为什么不直接用一个超强的 LLM Agent 单干（不加 Planner/Executor/Critic 三角色），现在大模型这么强，一个 Agent 能规划能执行？**

因为单 Agent 的上下文会爆炸。一个 Agent 既要规划（想 N 步）又要执行（调工具收结果）又要反思（评估），所有中间产物（计划/工具结果/反思）都堆在一个 context 里，跑几步就几万 token，触发 Lost in Middle 且成本飙升。三角色的本质是"上下文隔离"——Planner 只看目标和历史摘要、Executor 只看当前步的工具结果、Critic 只看结果和目标，各自 context 精简。通过共享 State 传递必要信息（而非全量 context），每个角色的 context 控制在 2-3K token。所以拆角色不是为了"模仿人类团队"，是为了工程上的上下文管理，这是长任务 Agent 的硬约束。

### 第四层：方案权衡

**Q：反思阶段你交给 LLM，但反思的结论你说要"摘要后写回 State，不留原始错误文本"，为什么不直接把 Critic 的完整输出写进去，还要费劲摘要？**

因为完整反思会污染后续上下文。Critic 的输出可能很长（"这一步的工具 A 返回了 X 但其实应该是 Y，因为 Z 条件没满足..."），如果原样写回 State，下一步 Planner 读 State 时这些冗长反思会挤占 context 且干扰决策。摘要后只保留决策相关信息（"工具 A 不可靠，建议换工具 B"），10 个字代替 200 字。权衡是"信息完整 vs context 效率"——反思的完整内容已落 Postgres（可追溯），State 里只放摘要（供下一步决策），两者分工。所以摘要不是丢信息，是把信息分层存储：决策相关的进 State，审计相关的进日志。

**Q：为什么不直接限制反思输出长度（Prompt 里写"不超过 50 字"），还要额外做一道摘要？**

因为 Prompt 限长度不可靠。LLM 经常无视"不超过 50 字"的约束（尤其是反思类需要展开的内容），实测 30% 的输出超长。另外即使长度达标，内容可能不是"决策相关摘要"而是"流水账复述"（模型偷懒把过程复述一遍而非提炼结论）。独立的摘要步骤（用一个小 LLM 或规则模板提取"结论 + 建议"）能保证质量——它专门做提炼这件事，比让 Critic 边反思边控制格式可靠。成本上摘要用小模型（豆包 lite），单次几分钱，相对省下的 context Token 成本（每步省 200 token × 15 步 × 日均万次 = 可观）ROI 正。所以多一道摘要不是过度设计，是保证质量的必要环节。

### 第五层：验证与沉淀

**Q：你怎么证明这套五段架构 + 双层存储真的比"一个 Agent 单干"好，而不是为拆而拆？**

用对照实验。同一个任务集（100 个真实任务），方案 A 是五段架构 + 双层存储，方案 B 是单 Agent + 内存状态。对比四个维度：任务完成率（A 可能因拆分丢失全局信息略低？）、平均步数（A 应该更少因为不扯皮）、单任务成本（A 的 Token 消耗）、可调试性（出问题时能多快定位）。我预期 A 在成本和可调试性上显著优，完成率持平或略高（因为防死循环）。关键指标是"线上故障的平均定位时长"——双层存储 + agent_steps 回放能让定位从小时级到分钟级，这是架构拆分的最大业务价值。有这个对比数据，拆分就不是为拆而拆。

**Q：这套架构怎么复用到下个 Agent 项目？**

沉淀成"Agent 脚手架"，封装五段流程 + 双层存储 + 防死循环三上限 + LangGraph State 模板。新项目接入只需定义：业务工具集、意图分类体系、Plan 模板。脚手架内置通用能力（状态双写、断点续传、步数熔断、回放调试、Critic 摘要），不用重写。配套"Agent 上线 checklist"——必过项包括：agent_steps 表建好、硬步数上限配了、回放工具能跑、对账脚本上线。这样架构经验是平台化的，新项目 1-2 周出 MVP 而非从零搭 2 个月。

## 结构化回答

**30 秒电梯演讲：** Agent 项目的本质是把"创造性决策"（理解意图、规划）和"确定性执行"（API调用、字段校验）分离，前段交 LLM，后段交 Workflow；状态用 Redis（短期）+ Postgres（长期可追溯）双写。

**展开框架：**
1. **意图识别** — 意图识别 → 计划生成 → 工具执行 → 结果聚合 → 反思 五段拆分
2. **创造性环节交** — 创造性环节交 LLM，确定性环节交 Workflow
3. **Redis** — Redis Hash 存短期会话状态（TTL 30min），Postgres 存 agent_steps 长期日志

**收尾：** 您想深入聊：多 Agent 直接对话（如 AutoGen）vs 共享状态（LangGraph）各自什么坑？
