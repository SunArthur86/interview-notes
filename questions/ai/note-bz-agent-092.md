## 三、数字人落地

```python
class DigitalHuman:
    """数字人：LLM大脑+语音+形象"""
    
    def __init__(self):
        self.brain = LLM(persona="...")      # 大脑
        self.tts = TTSEngine()                # 语音合成
        self.avatar = AvatarRenderer()        # 形象渲染
        self.asr = ASREngine()                # 语音识别
    
    async def interact(self, audio_input):
        # 1. 语音识别
        text = await self.asr.transcribe(audio_input)
        
        # 2. LLM生成回复
        response = await self.brain.generate(text)
        
        # 3. 语音合成（带情感）
        audio = await self.tts.synthesize(
            response.text,
            emotion=response.emotion  # 开心/严肃/鼓励
        )
        
        # 4. 形象渲染（口型/表情/动作）
        video = await self.avatar.render(
            audio=audio,
            emotion=response.emotion,
            gesture=response.suggested_gesture
        )
        
        return video  # 返回说话的视频
    
    # 关键技术：
    # - ASR：语音→文本（识别用户说的）
    # - LLM：文本→文本（生成回复）
    # - TTS：文本→语音（合成说话）
    # - Avatar：语音→视频（渲染口型/表情）
    # - 延迟优化：全链路<1s（流式处理）
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：AI 名师你说"严禁直接给答案要用苏格拉底式引导"，但学生就是要答案（不做题想直接抄），不给答案用户就流失了，你怎么平衡"教学效果"和"用户留存"？**

用"梯度给"而不是"绝对不给"。纯不给答案用户确实流失（特别是应付作业的场景），纯给答案丧失教学价值。平衡做法：先引导一步（给提示不给答案），如果用户连续两次说"我只想知道答案"或表现出强抵触，就给答案但附带"解法拆解"（答案 + 为什么这么解 + 这类题的套路）。这样既满足要答案的即时需求，又保留了教学渗透。数据上监控"给答案后的留存 vs 引导后的留存"——如果引导的用户长期留存更高（学习有效果回头），说明引导策略长期价值大，短期流失可接受。所以不是教条地"绝不给"，是"先引导、必要时给但带解法"，兼顾体验和教学目标。

### 第二层：证据与定位

**Q：数字人交互延迟你目标全链路 < 1s，但 ASR + LLM + TTS + Avatar 渲染四步串行，光 LLM 就要 1-2s，怎么做到 1s？**

必须全链路流式 + 并行，不是串行等。ASR 用流式（边听边出文本，用户说话的同时开始识别），LLM 用流式输出（边生成边吐第一个 token，首字延迟 TTFT 控在 300ms），TTS 收到 LLM 的第一个分句就立即合成（不等整句生成完），Avatar 收到 TTS 的第一帧音频就立即驱动口型。这样四步是流水线重叠的，用户感知的延迟 = max(各步启动延迟) 而非 sum(各步耗时)。实测 ASR 流式 + LLM 流式 TTFT 300ms + TTS 首句 200ms，端到端首响应 < 800ms 可达。关键是每一步都不能等上一步完全结束才开始，要按"分句/分帧"粒度流水线。

### 第三层：根因深挖

**Q：数字人你让 LLM 输出 Emotion 标签驱动 TTS 和表情，但 LLM 经常输出"中性"（偷懒），导致数字人表情僵硬没感染力，根因是什么？**

根因是 Prompt 对 Emotion 的约束不够具体。LLM 倾向输出"中性"是因为中性最安全（不会错），如果没有明确要求"根据语境选具体情绪"，它就偷懒。解法是：在 Prompt 里枚举具体情绪（开心/鼓励/严肃/疑惑/共情）并给每个情绪的触发条件示例（"学生答对时用开心/鼓励，答错时用温和而非严肃"），再加"禁止输出中性"的硬约束。另外用 Few-shot 示范（输入 X → 情绪 Y），让模型学到情绪判别模式。如果还不行，用一个独立的情绪分类模型（小 BERT）基于 LLM 的文本输出判情绪，比让 LLM 自己标更准。所以情绪输出不准不是模型能力问题，是任务定义不够清晰，靠 Prompt 工程 + 专用模型解决。

**Q：那为什么不直接用更强的大模型（GPT-4 级）做情绪判断，而要专门搞个小分类模型？**

因为延迟和成本。数字人是实时交互，LLM 已经在生成回复（占主要延迟），再让它"仔细想情绪"会拖慢 TTFT。而且情绪判断是相对简单的分类任务（输入文本 + 语境 → 6 类情绪之一），用 fine-tuned 小 BERT（< 100M 参数）做是毫秒级、成本是 LLM 的 1/100，准确率反而更高（专精于这个任务）。GPT-4 虽强但是通才，在具体情绪分类上不一定比专门训练的小模型准。所以分工是：LLM 负责生成回复内容（需要创造力），小模型负责情绪分类（需要一致性和速度），各司其职。这是工程上的"用对的工具做对的事"，不是一味用大模型。

### 第四层：方案权衡

**Q：领域知识注入你列了 RAG / 微调 / Prompt 三种，但一个垂直 Agent 该选哪个？三个都用会不会效果最好但成本爆炸？**

按知识的"更新频率 + 量级 + 准确性要求"选，不是全用。RAG 适合"高频更新、量大、需溯源"的知识（如法规库、题库），灵活且可追溯。微调适合"稳定、需内化进模型风格/常识"的知识（如教学语气、医学术语），让模型"本来就是专家"而非"查资料"。Prompt 适合"少量、稳定、需精确控制"的知识（如合规红线、教学禁令），最轻量但占上下文。典型组合：Prompt 写红线（不可变）+ 微调内化专业语气 + RAG 提供动态知识库。不是三者简单叠加，是各管一类知识，避免重复。全用微调（把所有知识塞进模型）成本高且更新慢，全用 RAG（什么都查）风格不统一，所以要分工。

**Q：为什么不直接微调成领域专家模型，一步到位，还要 RAG 和 Prompt 这么麻烦？**

因为微调的知识是"快照"，无法实时更新且不可溯源。领域知识（如法规、医学指南、题库）是持续更新的，微调一次要几天 + 高成本，新法规出来到模型会答要等下一次微调周期（可能 1 个月）。RAG 的知识库更新是分钟级（加新文档即生效）。另外微调的知识无法溯源（模型说了一个医学结论，你不知道它从哪学的），RAG 可以返回来源（"根据《XX 指南》第 3 条"），这对医疗/法律等需溯源的领域是硬需求。所以微调适合"稳定的风格和常识"，RAG 适合"动态的、需溯源的事实"，不可互相替代。

### 第五层：验证与沉淀

**Q：AI 名师的教学效果怎么验证，证明它真的帮学生提升成绩，而不是学生觉得好玩但没用？**

做 A/B 对照实验。一组学生用 AI 名师辅导，对照组用传统方式（看视频课/刷题），固定周期（如 1 个月）后测同一套试卷，对比分数提升。关键控制变量：两组学生的初始水平相当（按前测分班）、学习时长相当、除了辅导方式其他条件一致。如果 AI 组提升显著（如平均提 15 分 vs 对照组 8 分，p<0.05），证明有效。短期指标看"知识点掌握率"（每学完一个知识点测一下），长期看"考试成绩"。没有对照实验，"学生喜欢"不等于"有效"（可能只是新鲜感），必须有数据证明教学价值，这是教育产品的核心壁垒。

**Q：垂直 Agent 的经验怎么复用到其他垂直领域？**

沉淀成"垂直 Agent 脚手架"，把通用的能力（人设管理、合规框架、工具集成、多模态链路、评估闭环）做成框架，新领域接入只换"领域知识包"（RAG 库 + 微调数据 + Prompt 模板 + 工具集）。比如从教育扩展到医疗，复用脚手架的合规框架（换成医疗合规规则）、多模态链路（ASR/TTS/Avatar 通用）、评估闭环（换评测集），只新做医学知识库和医疗工具。这样每个新垂直领域的落地成本从"从零搭"降到"换领域包"，是垂直化的规模化能力。脚手架 + 领域包的架构，是做"多垂直 Agent 矩阵"的公司的核心竞争力。
