# 大模型意图识别是怎么做的？

## 一、意图识别的定义

```
意图识别(Intent Recognition)
  = 理解用户输入"想做什么"，映射到预定义意图类别

例：
  "我要查订单"         → intent: query_order
  "退款"              → intent: refund
  "天气怎么样"         → intent: query_weather
  "帮我写首诗"         → intent: creative_writing
  "你好"              → intent: greeting
```

## 二、三种实现方法

### 方法 1：LLM 零样本/少样本分类（推荐）

```python
INTENT_PROMPT = """
判断用户意图，从以下选项中选择：

可用意图：
- query_order: 查询订单/物流
- refund: 退款/退货
- product_info: 产品咨询
- complaint: 投诉/不满
- greeting: 问候/闲聊
- other: 其他

用户输入: {user_input}

示例:
"我的快递到哪了" → query_order
"我要退那个手机" → refund
"你好" → greeting

只输出意图名称。
"""

intent = llm.classify(INTENT_PROMPT.format(user_input=user_msg))
# 优势：新增意图只需改prompt，无需训练
```

### 方法 2：微调专用分类器

```python
# 数据量大时，微调小模型更快更准
from transformers import AutoModelForSequenceClassification

# 标注数据
train_data = [
    ("查订单", "query_order"),
    ("退款", "refund"),
    # ... 几千条
]

# 微调BERT等小模型
model = AutoModelForSequenceClassification.from_pretrained("bert-base-chinese")
model.fit(train_data)
# 优势：快(ms级)、便宜、可离线
# 劣势：新意图需重训
```

### 方法 3：规则 + LLM 混合

```python
class HybridIntentRecognizer:
    """高频意图走规则(快)，长尾走LLM(准)"""
    
    RULES = {
        "query_order": ["订单", "快递", "物流", "到哪"],
        "refund": ["退款", "退货", "退钱"],
        "greeting": ["你好", "hi", "hello"],
    }
    
    def recognize(self, text):
        # 先走规则（快速）
        for intent, keywords in self.RULES.items():
            if any(kw in text for kw in keywords):
                return intent, 1.0  # 高置信
        
        # 规则未命中，走LLM（处理长尾）
        return self.llm_classify(text)  # 准但慢
```

## 三、多意图与槽位

```python
# 多意图识别（一句话多个意图）
def multi_intent(text):
    prompt = f"""
    用户输入可能包含多个意图。列出所有:
    
    输入: "{text}"
    输出: JSON数组，如 ["query_order", "refund"]
    """
    intents = llm.parse(prompt)
    # "查下我的订单，不对要退款" → ["query_order", "refund"]

# 槽位填充（意图确定后，提取参数）
def slot_filling(text, intent):
    if intent == "query_order":
        slots = llm.extract(f"""
        从以下文本提取: order_id(订单号)
        文本: {text}
        """)
        # "查我的订单A123" → {order_id: "A123"}
```

## 四、意图识别的挑战

```
┌──────────────┬──────────────────────────────────┐
│ 挑战          │ 对策                                │
├──────────────┼──────────────────────────────────┤
│ 意图模糊      │ 让LLM输出置信度，低的追问          │
│ "那个东西"   │ → "您是指查询还是购买？"            │
├──────────────┼──────────────────────────────────┤
│ 多意图        │ 多标签分类，按优先级处理            │
├──────────────┼──────────────────────────────────┤
│ 长尾意图      │ LLM兜底+定期把高频长尾加入规则     │
├──────────────┼──────────────────────────────────┤
│ 新意图出现    │ LLM零样本支持/识别unknown聚类      │
├──────────────┼──────────────────────────────────┤
│ 实时性        │ 规则优先(快)，LLM兜底              │
└──────────────┴──────────────────────────────────┘
```

## 五、面试加分点

1. **LLM 零样本是革命**：新增意图无需训练，只改 prompt——这是比传统 NLU 的巨大优势
2. **混合方案最实用**：规则(快)+LLM(准)结合，兼顾延迟和覆盖
3. **意图+槽位配合**：识别意图后还要提取参数，是完整的 NLU

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你推荐 LLM 零样本分类做意图识别，但 LLM 调一次 200-500ms，传统规则匹配只要 1ms。为什么不用纯规则，非要上 LLM？**

因为规则的覆盖率天花板低。规则只能命中"包含关键词"的输入（"我要退款"命中"退款"），但用户的真实表达千变万化——"这个东西我不想要了"、"上次买的那单帮我处理一下"、"商家发错货了怎么办"，这些没有明确关键词但意图明确是退款。纯规则在这些长尾输入上召回率只有 60-70%，剩下的用户会掉到"无法理解"的兜底回复。LLM 的价值是覆盖这 30-40% 的长尾。所以不是规则 vs LLM 二选一，是规则挡高频（80% 流量、1ms 延迟），LLM 兜长尾（20% 流量、500ms 延迟但召回高），整体延迟加权后只有 100ms 级别。

### 第二层：证据与定位

**Q：线上意图识别准确率从 92% 掉到 85%，怎么定位是 Prompt 退化、模型版本切换、还是用户输入分布变了？**

看三类证据。第一看错误样本的分布——如果错误集中在某类新表达（比如最近流行的新词/梗），是输入分布漂移，Prompt 里的示例没覆盖，要补示例。第二看模型版本灰度日志——准确率下降的时间点和模型版本切换吻合，是模型行为漂移。第三看同一批错误输入在旧 Prompt + 旧模型上的表现——如果旧组合准确，逐一变量切换定位是 Prompt 还是模型的问题。定位清楚后对症：输入漂移补示例，模型漂移加 Prompt 锚点，Prompt 退化回滚版本。

### 第三层：根因深挖

**Q：LLM 零样本你说新增意图只改 Prompt 不用训练，但实际加了新意图后，旧意图的准确率下降了，为什么？**

根因是"意图边界混淆"。LLM 零样本分类靠的是意图描述的语义区分度。新增一个意图（比如"投诉"），如果它的描述和已有意图（"退款"）语义重叠（投诉里常含退款诉求），LLM 在两者边界上就会摇摆，导致旧意图被误分到新意图。解法不是简单的加示例，而是明确意图间的边界规则——在 Prompt 里写"退款是要求退钱，投诉是表达不满不一定要求退钱；如果同时有，优先标 refund"。本质是把"意图定义"从模糊的自然语言描述变成有判别边界的规则，降低 LLM 的歧义。

**Q：那为什么不直接微调一个专用分类器，意图边界固化在权重里，比 Prompt 描述稳定？**

因为微调的迭代成本高。业务意图是高频变的（每周可能加新意图、调整边界），微调一次要标数据 + 训练 + 评测 + 上线，周期 1-2 周，跟不上业务节奏。LLM 零样本改个 Prompt 是几小时上线。所以微调适合"意图体系稳定、流量大、对延迟极敏感"的场景（比如头部高频意图 query_order/refund 用微调小模型做毫秒级分类），LLM 适合"意图体系频繁变、长尾覆盖"的场景。正确架构是分层：Top10 高频意图微调小模型（快且稳），其余走 LLM（灵活），不是全用一种。

### 第四层：方案权衡

**Q：意图模糊时你让 LLM 输出置信度，低的反问澄清。但反问太多用户体验差，你怎么决定什么时候该反问什么时候该猜？**

看反问的"代价 vs 猜错的代价"谁大。用置信度阈值 + 业务风险双重判断：置信度 > 0.8 直接执行（猜对概率高，不值得反问）；0.5-0.8 之间看动作的风险——低风险动作（查订单、查天气）直接猜着执行，猜错了用户能纠正且成本低；高风险动作（退款、支付）必须反问，因为猜错了执行了很难挽回。所以不是所有模糊都反问，是"模糊 + 高风险"才反问。我会在意图配置里给每个意图标 risk_level（high/medium/low），反问策略按 risk + 置信度矩阵决定，把反问率控制在 10% 以内。

**Q：为什么不直接每次都让 LLM 输出 Top-3 意图让用户选，准确率最高？**

因为大多数输入意图是明确的（"查订单"就是查订单），90% 的场景让用户从三个选项里选是多余的操作，体验比直接执行差。Top-3 选项只在"置信度灰度区（0.4-0.6）"用，且最多给 2-3 个选项（不是一堆），选项要用人话表达（"您是想查这个订单的物流，还是申请退款？"）而不是意图代号。所以 Top-3 是灰度区的手段，不是默认手段，默认是高置信度直接执行。滥用 Top-3 是把识别能力不足转嫁给用户，是不负责任的设计。

### 第五层：验证与沉淀

**Q：你怎么持续评估意图识别的质量，证明它在真实流量下不退化？**

维护一个"意图识别评测集" + 线上抽检闭环。评测集 1000+ 条人工标注的输入，覆盖所有意图 + 边界 case + 长尾表达，每次 Prompt/模型变更跑全量算 F1。线上每天抽 1% 流量做人工复核（或用更强的模型当裁判），算线上准确率。两个指标都监控趋势，下降 > 2% 触发告警。关键是评测集要持续更新——线上发现的新表达/新意图定期补进去，否则评测集和真实分布脱节，离线测得再好线上也退化。

**Q：意图体系怎么沉淀让团队可维护？**

沉淀成"意图知识库"平台。每个意图有完整定义：名称、描述、边界规则（和相邻意图怎么区分）、典型示例、反例、risk_level、对应的槽位 schema。意图变更走审批流（产品定边界、研发改 Prompt、评测集回归）。平台自动生成 LLM 的 Prompt（从知识库渲染），保证 Prompt 和意图定义一致。这样意图体系是产品/运营可维护的资产，不是写死在某个研发的代码里，新人接手看知识库就懂全貌。
