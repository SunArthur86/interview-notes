# 【月之暗面面经】你如何判断候选人真的做过 AI-Native 桌面产品，而不是普通 Web 前端？

## 一、核心判断逻辑

```
┌──────────────────────────────────────────────────────────────────┐
│              经验深度的四个层次                                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Level 1: 概念知道（读过文章/看过Demo）                            │
│  → 能说"AI桌面产品就是Electron+Vue+AI API"                       │
│  → 追问细节就卡住                                                 │
│                                                                  │
│  Level 2: 组件做过（做过部分功能）                                 │
│  → 能讲Vue组件树、Pinia状态管理                                   │
│  → 但讲不出文件系统、长任务、多窗口                                │
│                                                                  │
│  Level 3: 主链路做过（做过核心功能）                               │
│  → 能讲任务中心、产物面板、文件权限                                │
│  → 能说出具体的设计决策和坑                                       │
│                                                                  │
│  Level 4: 平台化做过（从0到1构建过完整系统）                       │
│  → 能讲协议设计、扩展架构、工程约束                                │
│  → 能主动补充权限/灰度/埋点/成本/失败恢复                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、六个验证维度

### 维度1: 任务中心

**普通前端**：会讲"用Pinia管理任务列表，v-for渲染任务卡片"

**桌面AI老手**：
```
"任务中心有五种状态：queued/running/paused/review/done
 每个任务关联输入引用、产物列表和权限记录
 任务状态变更通过事件总线广播到所有窗口
 长任务不绑定在组件生命周期上，组件卸载后任务继续执行
 我们遇到过一个问题：用户关闭窗口后任务还在跑，
 但回来时任务中心的进度条没有更新——因为事件订阅随组件销毁了
 后来把任务状态提升到全局单例 store 才解决"
```

### 维度2: 文件权限

**普通前端**：会讲"用<input type=file>上传文件"

**桌面AI老手**：
```
"文件权限分四层：引用/读取/写入/系统操作
 授权前展示文件数量和操作说明
 写入操作必须二次确认+diff预览
 撤销授权后要清理已索引的文件缓存
 最麻烦的是处理 .gitignore——用户不想让你索引 node_modules
 我们最终实现了一个忽略规则引擎，读取项目的 .gitignore"
```

### 维度3: 长任务

**普通前端**：会讲"用async/await，加个loading动画"

**桌面AI老手**：
```
"长任务最大挑战是用户会离开——切去做别的事
 所以前端必须做三件事：
 1. 系统通知——完成后即使应用在后台也能提醒用户
 2. 上下文恢复——用户点通知能回到完整任务现场
 3. 失败恢复——任务在某个步骤失败后可以从断点续跑
 
 我们踩过的坑：流式输出时网络断开，
 前端收到了部分内容但没有错误事件，
 导致产物不完整但状态显示'完成'
 后来加了心跳检测+内容完整性校验"
```

### 维度4: 多窗口

**普通前端**：可能知道"Electron可以开多个BrowserWindow"

**桌面AI老手**：
```
"多窗口最大的坑是状态同步
 我们最初把所有状态都通过IPC同步
 结果滚动位置/输入框内容都互相干扰
 后来分层：事实数据强同步、UI状态不同步
 编辑冲突用编辑锁解决——持有锁的窗口才能编辑
 锁5分钟无活动自动过期，防止窗口崩溃后锁不释放"
```

### 维度5: 本地与云端边界

**普通前端**：会讲"调API就行了"

**桌面AI老手**：
```
"本地索引段是CPU密集型——文件解析、OCR、摘要生成
 云端推理段是网络密集型——LLM调用
 产物导出段是IO密集型——写文件、格式转换
 三段的延迟特征和失败模式完全不同
 前端不能用一个loading覆盖一切
 我们用traceId贯穿三段链路
 出了问题能精确定位是哪一段卡住了"
```

### 维度6: 产物落地

**普通前端**：会讲"把AI返回的内容显示在页面上"

**桌面AI老手**：
```
"产物落地是三步流程：草稿→确认→发布
 AI结果先入草稿态，不直接写文件
 用户审查diff后确认，才触发实际写入
 每次写入都创建版本快照，支持回退
 最难的是大产物的diff——
 一个20页的PPT，用户只想看第3页改了什么
 所以我们做了页面级diff，而不是整文档diff"
```

## 三、验证问题清单

| 问题 | 检验维度 | 红旗信号（没做过） | 绿旗信号（做过） |
|------|---------|-------------------|-----------------|
| 描述一次任务失败后用户投诉最多的问题 | 异常处理 | 答不出具体场景 | 能说清失败原因和改进措施 |
| 文件权限是怎么设计的 | 安全设计 | 只说"用input上传" | 讲出分层授权+二次确认 |
| 多窗口同时编辑怎么办 | 状态管理 | 没想过这个问题 | 讲出编辑锁+冲突仲裁 |
| 长任务中断后怎么恢复 | 可靠性 | "重新跑就行了" | 讲出断点续跑+部分保留 |
| 怎么判断前端做得好不好 | 指标设计 | 说DAU/响应时间 | 说任务完成率/产物采纳率 |

## 四、常见坑

- **只考一次问答**：候选人可能准备了一两个场景，多问几个就能看出深度
- **被Vue组件树迷惑**：组件写得漂亮不代表做过桌面主链路
- **不追问失败场景**：正常流程谁都能讲，失败处理才是真功夫
- **不验证本地/云端边界**：这是桌面AI和Web前端最本质的区别

## 记忆要点

- 追问脱离浏览器的能力：验证是否懂本地文件系统操作、长任务与组件生命周期的解耦。
- 多窗口状态治理：问多窗口编辑冲突如何解决，考察强同步与不同步的分层状态管理经验。
- 云边协同全链路：是否掌握本地索引、云端推理、本地导出的差异化状态机与Trace设计。
- 权限与异常恢复：看其是否主动关注文件授权四层模型、心跳检测、断点续跑等工程难点。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么判断候选人是否做过 AI-Native 桌面产品，要追问"异常路径"而非"正常流程"？正常流程讲得好不也说明有经验吗？**

因为"正常流程"可以通过阅读文档和 Demo 在几小时内学会，而"异常路径"只有亲身踩坑才知道。一个候选人看了产品文档后能讲"任务中心有 queued/running/done 三态，用 Pinia 管理"——这是 Level 1 概念知道。但要回答"用户关闭窗口后任务还在跑，回来时进度条没更新，为什么？"——只有真正做过的人才知道"事件订阅随组件销毁了，任务状态要提升到全局单例 store"。异常路径之所以能区分经验深度，是因为 AI 桌面产品的复杂度集中在 Web 前端不会遇到的领域（本地文件系统、长任务生命周期、多窗口状态），这些领域的坑（如流式输出网络断开导致产物不完整、编辑锁窗口崩溃后不释放、多窗口 IPC 同步导致输入框互相覆盖）无法靠想象编造——没踩过就答不出具体的失败现象和修复方案。所以追问异常路径的本质是"验证肌肉记忆"——真正做过的人在回答时能脱口而出具体场景（"我们遇到过...""后来发现是...""解决方案是..."），没做过的人只能泛泛而谈（"应该可以...""大概需要..."）。

### 第二层：证据与定位

**Q：你怎么定位候选人回答的某个细节是"真实经验"还是"精心准备的面试话术"？现在候选人都会背八股文。**

用"细节追问深度"和"场景变形"定位。话术的特点是"只能讲一层深度"——候选人准备了"多窗口用编辑锁解决冲突"的话术，你追问"编辑锁的粒度是文件级还是字段级？"他卡住了（因为话术只准备了第一层）。真实经验的特点是"可以无限下钻"——做过的人不仅能讲文件级 vs 字段级，还能讲"我们最初用文件级锁，后来用户抱怨粒度太粗，改成了页面级锁"，甚至能讲"锁的实现是用 Map<artifactId, LockInfo> + 心跳续约 + 5 分钟超时自动释放"。另一个定位方法是"场景变形"——候选人讲了"流式输出网络断开的处理"，你接着问"那如果是云端返回了完整响应但内容是空的（HTTP 200 但 body 为空），你怎么检测？"真实经验的人会立刻说"这要加内容完整性校验，不能只看 HTTP 状态码"（因为他踩过类似的坑），话术党会卡住（因为准备的话术只覆盖了"网络断开"这一种场景）。真实经验的标志是"能应对变形场景"，话术的标志是"只在准备好的场景内流畅"。

### 第三层：根因深挖

**Q：为什么"只会讲 Vue 组件树"的候选人大概率没做过桌面主链路？组件写得好不也是核心能力吗？**

因为 Vue 组件树是"展示层"能力，而桌面 AI 产品的复杂度在"架构层"和"系统集成层"，两者是不同维度的能力。组件写得好说明候选人擅长"把 UI 切分为可复用组件 + 管理组件内状态"，这在 Web 前端是核心能力。但桌面 AI 产品的核心挑战不在 UI 切分，而在：任务状态如何脱离组件生命周期存活（组件卸载后任务继续跑）、多窗口如何同步事实数据而不互相干扰、本地文件系统如何做权限管理、长任务如何做断点续跑和失败恢复、产物如何走草稿→确认→发布三步链路——这些都不在 Vue 组件树的范畴内，属于"系统工程"和"架构设计"能力。一个候选人如果面试全程只讲"我用了 composition API 拆分逻辑、用 Pinia 管理状态、组件设计很优雅"，但讲不出"任务状态为什么要在组件外管理""多窗口同步策略""文件权限分层"，大概率他的工作集中在"渲染层"而非"主链路"——他可能做过产品的某个 UI 页面，但没做过任务中心、文件系统、产物落地这些核心链路。

**Q：那如果候选人确实做过桌面产品但只是做 UI 组件（没碰过主链路），为什么不认可他的组件经验而要判定"没做过桌面主链路"？**

因为"做 UI 组件"和"做桌面主链路"对团队的价值不同，面试要匹配的是岗位需求。如果岗位是"桌面 AI 产品的核心链路工程师"，需要的是能设计任务状态机、多窗口同步、文件权限系统的人——只做过 UI 组件的候选人无法胜任，判定"没做过主链路"是准确的评估（不是否定他的 UI 能力，而是不匹配岗位）。如果岗位是"UI 组件工程师"，那他的组件经验完全够用——但这个岗位通常不需要用"AI-Native 桌面产品"的特殊面试题来考察，普通前端面试（组件设计、状态管理、性能优化）就够了。所以"判定没做过桌面主链路"的价值是"帮助面试官区分候选人适合哪个岗位"——主链路岗位招做过主链路的人，UI 岗位招擅长 UI 的人，两者不混淆。这不是对候选人的否定，而是对岗位匹配度的精确评估。

### 第四层：方案权衡

**Q：验证候选人经验你用"追问异常路径"的方法，为什么不用"现场写代码"（如让他实现一个简单的任务状态机）来直接验证能力？**

因为"现场写代码"和"追问异常路径"验证的是不同维度的能力，面试应该组合使用而非二选一。现场写代码验证"编码能力"（语法熟练度、算法实现、代码组织），适合考察基础工程能力。但它有两个局限：(1) 时间短覆盖不全——30 分钟的 coding 面试只能写一个小功能，无法覆盖"多窗口同步""长任务恢复""权限系统"这些需要系统级设计的复杂场景；(2) 无法验证"踩坑经验"——候选人现场写一个任务状态机可能逻辑正确，但无法暴露他是否知道"组件卸载后状态丢失"这个坑（因为现场不会模拟组件卸载）。追问异常路径的互补价值是"验证系统级设计经验和踩坑记忆"——它能在 5 分钟内覆盖"多窗口怎么同步""长任务怎么恢复""文件权限怎么分层"等多个维度，快速判断候选人是否做过主链路。最优策略是"先用追问异常路径筛选出做过主链路的人，再用现场 coding 验证编码能力"——两者组合才能全面评估。

**Q：那如果团队觉得"追问异常路径太依赖面试官经验（面试官要自己懂这些坑才能判断）"，为什么不只用标准化题目（如算法题）降低面试门槛？**

因为"降低面试门槛"和"准确评估岗位匹配度"是矛盾的目标，选后者而非前者。标准化题目（算法题）的好处是"任何面试官都能判对错"，坏处是"考察的维度与岗位需求不匹配"——桌面 AI 产品主链路工程师不需要刷 LeetCode，需要的是系统设计和工程经验。如果只用标准化题目，可能招到一个算法很强但没做过多窗口/长任务/文件系统的人，入职后无法胜任。面试官不懂这些坑的解法是"提升面试官能力"而非"降低考察标准"：(1) 用结构化题库——把六个维度的验证问题和"红旗/绿旗信号"做成题库（本文已经提供了模板），面试官照着问并对照信号判断，不需要自己懂技术细节；(2) 双人面试——一个懂技术的面试官（判断答案准确性）+ 一个 HR（观察候选人表达和反应），降低单人面试的经验依赖；(3) 以"追问深度"而非"答案正确性"判断——面试官不需要知道编辑锁怎么实现，只需要不断追问"然后呢""为什么""遇到什么坑"，真实经验者越问越深，话术党越问越卡。

### 第五层：验证与沉淀

**Q：你怎么证明"追问异常路径"比"考正常流程"更能识别真实经验，有数据支撑吗？**

用"入职后表现回溯"验证。对过去入职的候选人做回溯分析：(1) 把面试评价（"做过主链路"/"没做过"）与入职后 3 个月的实际表现（能独立负责主链路功能 / 只能做 UI 辅助）对比，计算评估准确率——如果"追问异常路径"方法的准确率 >80%（即被判定"做过"的人 80% 确实能胜任），而"考正常流程"方法的准确率只有 50%（正常流程讲得好但实际做不了），就证明追问方法更有效；(2) 统计"误判率"——被判定"做过"但实际做不了的（假阳性），和被判定"没做过"但实际能做的（假阴性），追问方法的两种误判率都应低于正常流程方法；(3) 收集入职者的反馈——问他们"面试时的追问是否准确反映了你的经验"，真实经验者会说"这些问题只有做过才能答"（验证追问有效），话术党会说"我没准备这个场景所以没答好"（验证追问能区分）。

**Q：怎么让团队的面试官在面试 AI-Native 桌面产品岗位时，自觉用"追问异常路径"的方法，而不是退回到舒适的"考 Vue 组件"？**

把面试方法做成"结构化面试指南 + 面试官培训"的强制流程。第一，制定面试指南文档——明确岗位的六个验证维度（任务中心/文件权限/长任务/多窗口/云边边界/产物落地），每个维度提供 2-3 个异常路径追问问题和红旗/绿旗信号，面试官照着问；第二，面试官培训——对参与面试的工程师做 1 小时培训，讲解"为什么要追问异常路径""如何判断话术 vs 真实经验"，并用模拟面试演示；第三，面试反馈模板——面试官必须按模板填写"每个维度的追问问题和候选人的回答深度（Level 1-4）"，不允许只写"感觉还行/不行"的模糊评价，强制结构化评估；第四，面试质量回溯——每季度统计面试评估的准确率（通过入职后表现回溯），对准确率低的面试官做针对性辅导。这样面试方法就从"面试官自由发挥"变成了"结构化指南驱动"。

## 结构化回答

**30 秒电梯演讲：** 判断候选人是否真正做过AI-Native桌面产品而非普通Web前端——重点追问任务中心、文件权限、产物落地和多窗口协作的细节；要求讲一次长任务或导出失败的完整处理过程；看他能否讲清本地和云端的边界。

**展开框架：**
1. **重点问任务** — 重点问任务中心、文件权限、产物落地和多窗口协作
2. **追问** — 追问一次长任务或导出失败是怎么处理的
3. **看他能否讲** — 看他能否讲清本地和云端的边界

**收尾：** 您想深入聊：如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
