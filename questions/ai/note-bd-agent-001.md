# 如何搭建Agent工作流并提升生成准确性？

## 工作流架构设计

```
用户输入
  │
  ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Planner  │───→│   RAG    │───→│  Writer  │
│ 任务拆分  │    │ 知识检索  │    │ 内容生成  │
└──────────┘    └──────────┘    └────┬─────┘
                                     │
                                     ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  Save    │←───│  Repair  │←───│ Reviewer │
│ 结果存储  │    │ 修复生成  │    │ 质量检查  │
└──────────┘    └──────────┘    └──────────┘
```

## 核心节点说明

### 1. Planner（规划节点）
- 接收用户意图，将长文创作拆分为章节大纲
- 维护全局计划状态（已完成/待完成/冲突项）
- 输出结构化计划：章节列表 + 每章主题 + 预期字数

### 2. RAG（检索增强节点）
- **短期记忆**：最近N章的摘要，解决上下文窗口限制
- **长期记忆**：角色卡、世界观设定、已建立的关系网络
- 检索策略：Embedding相似度 + 关键词混合检索

### 3. Writer（生成节点）
- 输入 = 计划 + 检索上下文 + 前文摘要
- 使用结构化Prompt：角色设定 + 章节计划 + 风格要求
- 生成时注入长期记忆中的角色特征，保持人设一致

### 4. Reviewer（审稿节点）
- 检查维度：情节连贯性、角色一致性、逻辑冲突、风格匹配
- 输出结构化结果：`{pass: boolean, issues: [...], suggestions: [...]}`

### 5. Repair（修复节点）
- 根据Reviewer的issues定向修复
- 只修改有问题的段落，不重新生成全文
- 修复后重新进入Reviewer检查（最多3轮）

## 提升准确性的关键策略

| 策略 | 做法 | 效果 |
|------|------|------|
| **分层记忆** | 短期(最近章节) + 长期(角色设定/世界观) | 解决长篇设定遗忘 |
| **审稿修复循环** | Reviewer发现问题→Repair修复→Reviewer再验 | 将单轮85%提升至95%+ |
| **RAG上下文注入** | 检索历史章节+角色卡拼入Prompt | 保证人设和剧情连续性 |
| **结构化输出** | 每个节点输出JSON schema | 降低解析错误率 |
| **失败重试机制** | 节点失败后最多重试3次，超出则降级 | 提高整体可用性 |

## LangGraph状态编排实现

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class AgentState(TypedDict):
    user_input: str
    plan: dict
    context: str
    draft: str
    review_result: dict
    repair_count: int
    final_output: str

def should_repair(state: AgentState) -> str:
    """条件路由：审稿通过则保存，否则修复"""
    if state["review_result"]["pass"]:
        return "save"
    if state["repair_count"] >= 3:
        return "save"  # 超过3轮强制保存
    return "repair"

# 构建状态图
workflow = StateGraph(AgentState)
workflow.add_node("planner", plan_node)
workflow.add_node("rag", rag_node)
workflow.add_node("writer", write_node)
workflow.add_node("reviewer", review_node)
workflow.add_node("repair", repair_node)
workflow.add_node("save", save_node)

workflow.set_entry_point("planner")
workflow.add_edge("planner", "rag")
workflow.add_edge("rag", "writer")
workflow.add_edge("writer", "reviewer")
workflow.add_conditional_edges("reviewer", should_repair)
workflow.add_edge("repair", "reviewer")  # 修复后重新审稿
workflow.add_edge("save", END)

app = workflow.compile()
```

## 实际效果

- **单轮生成**：剧情连贯性~70%，角色一致性~65%
- **工作流+审稿修复**：剧情连贯性~92%，角色一致性~90%
- 核心提升来自Reviewer+Repair循环，而非单纯加大模型参数

## 面试加分点

1. **强调工程思维**：不是靠Prompt Engineering硬调，而是用工作流拆解降低单步错误率
2. **量化指标**：能说出具体准确率提升数字
3. **失败处理**：提及repair_count上限、降级策略，体现生产环境思维
4. **成本意识**：Repair只改有问题的段落而非全文重写，节省Token

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你的 Agent 工作流有"生成→审稿→修复"闭环，为什么不直接用一个更强的模型一次生成到位，省得多步开销？**

因为单次生成的错误率随任务复杂度指数上升，强模型也压不住。长文生成涉及逻辑一致性、事实准确、格式规范多个维度，一个 forward pass 同时保证所有维度极难。拆成"生成（专注内容）→审稿（专注挑错）→修复（专注改错）"让每个节点只关注一个目标，单节点准确率高，组合后整体超过单次生成。动机是"分工降复杂度"——把指数级错误率降为各节点错误率的加权和。

### 第二层：证据与定位

**Q：审稿节点反馈"初稿有逻辑断裂"，但修复节点改完仍有问题。你怎么定位是审稿没审全、还是修复没改对？**

看审稿和修复的 Trace 对照。把审稿指出的问题列表（issue list）和修复后的文本比对：如果审稿指出了 5 个问题，修复只改了 3 个（漏改），是修复节点问题；如果审稿只指出 3 个问题但实际有 5 个（漏审），是审稿节点问题。更系统的是做"修复后再审"——第二轮审稿如果发现新问题，看是新问题（第一轮审稿能力不足）还是老问题（修复没改好），按这个分类归因。

### 第三层：根因深挖

**Q：审稿节点的 Reviewer 用的是同一个 LLM（只是 prompt 不同），它能审出自己的错误吗？根因上自审有效吗？**

同模型自审有效但有上限。LLM 能识别"明显的逻辑错误、格式问题、事实冲突"（这些在 prompt 引导下可检出），但对"模型自身的认知盲区"检不出——如果模型在生成时就理解错了，自审时用同样的认知也审不出。治本是用不同模型族审稿（如 GPT 生成 + Claude 审），不同模型的错误模式不同，交叉审核能互补盲区。或用规则审稿（正则查格式、NLI 查事实一致性）补充 LLM 审稿的逻辑盲区。

**Q：那为什么不直接用更强的模型（如 GPT-4）做生成，弱模型（GPT-3.5）做审稿，反过来？**

因为生成的质量上限决定整体质量，审稿是"挑错"不是"创造"。如果弱模型生成的内容本身质量低（逻辑乱、信息缺），强模型审稿能挑出很多错，但修复还是要重新生成，等于退化成强模型生成。正确分工是强模型生成（保质量上限）+ 规则/小模型审稿（挑明显错误，省成本）+ 强模型定向修复（只改错处不全量重写）。审稿的核心是"覆盖率"（能挑出多少错）而非"创造力"，规则和小模型足够。

### 第四层：方案权衡

**Q：审稿-修复循环你设最多 3 次，为什么是 3 次？多循环几次质量会不会更高？**

3 次是边际收益和成本的拐点。实测：第 1 轮修复把准确率从 70% 提到 88%，第 2 轮提到 93%，第 3 轮提到 95%，第 4 轮只提到 95.5%——收益骤降。但每轮循环要 2 次 LLM 调用（审稿+修复），成本和延迟线性增加。3 轮后还修不好的通常是"结构性问题"（如素材不足、逻辑根本不通），再循环也救不回来，应该走兜底（转人工或返回"无法生成"）。超过 3 轮还循环也可能死循环（修复引入新错→审稿挑出→修复又引入），要硬限制。

**Q：为什么不直接在生成阶段加更多约束（如 few-shot 示例、CoT 推理），一次性提升质量，省得搞审稿修复？**

生成阶段的约束有边际效用递减。few-shot 和 CoT 能把基线从 70% 提到 80%，但要提到 95% 需要极长的 prompt（几十个示例），prompt 膨胀导致 token 成本和延迟上升，且模型对超长 prompt 的注意力下降。审稿修复是"后置质检"，不增加生成时的 prompt 负担，用独立的质检节点保证质量。两者互补：生成阶段适度约束（few-shot + CoT）拉高基线，审稿修复兜住尾部。全靠生成阶段约束是"预防"，加审稿修复是"检测+纠正"，工程上后者更可靠。

### 第五层：验证与沉淀

**Q：你怎么证明"生成→审稿→修复"闭环把准确率从 70% 提到 95%，而不是评测集偏简单？**

构建对抗性评测集（含难 case：多约束条件、需要长程一致性、含数字事实），在难集上测闭环效果。做消融实验：只生成（70%）→ 生成+审稿但不修复（审稿能识别但没改，仍是 70%，证明审稿本身不改质量）→ 生成+审稿+修复（95%），证明是修复环节起作用。每轮循环的贡献分别埋点统计（第 1 轮 +18%、第 2 轮 +5%、第 3 轮 +2%），证明每轮都有价值且收益递减符合预期。

**Q：这套审稿修复工作流怎么沉淀成通用 Agent 框架？**

封装成 `generate_review_repair(task, max_rounds=3)` 模板：生成节点可配模型/prompt，审稿节点可配规则+LLM 混合，修复节点只接受"定向修复指令"（改哪几处）。配套 Trace 看板，展示每轮的初稿、审稿 issue list、修复后文本，支持失败 case 回放。沉淀"各任务的审稿规则模板"（如代码审规则、文案审规则）、"修复 prompt 模板"（如何让模型只改错处不全量重写），新场景接入时按模板配。
