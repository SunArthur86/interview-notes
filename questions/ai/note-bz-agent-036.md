# 什么是 Skill？Agent 架构中 Skill 的本质是什么？

## 一、Skill 的定义

**Skill** = Agent 的**可复用能力模块**，封装了完成某类任务所需的 **Prompt + Tools + 执行流程**。

```
┌──────────────────────────────────────────────┐
│              Skill 的组成                       │
├──────────────────────────────────────────────┤
│                                                │
│  1. 触发条件 (Trigger)                         │
│     什么时候该用这个Skill                       │
│                                                │
│  2. 系统提示词 (Prompt)                        │
│     执行这个任务的行为指导                      │
│                                                │
│  3. 工具依赖 (Tools)                           │
│     这个Skill需要调用哪些工具                   │
│                                                │
│  4. 执行流程 (Flow)                            │
│     步骤编排、条件分支、错误处理                │
│                                                │
│  5. 输入输出Schema                             │
│     接受什么输入，产出什么输出                   │
│                                                │
└──────────────────────────────────────────────┘
```

## 二、Skill vs Tool 的区别

```
┌────────────┬──────────────────┬──────────────────────┐
│ 维度        │ Tool               │ Skill                  │
├────────────┼──────────────────┼──────────────────────┤
│ 粒度        │ 原子操作           │ 多步组合                │
│ 复杂度      │ 单一功能           │ 复杂能力                │
│ 组成        │ 一个函数           │ Prompt+Tools+Flow      │
│ 例子        │ search()/send()   │ "写技术博客"            │
│ 可复用      │ 代码级             │ 能力级（可分享）         │
└────────────┴──────────────────┴──────────────────────┘

例：
  Tool: web_search(q) → 单纯搜索
  Skill: "技术调研" = search + 分析 + 总结 + 生成报告
         （组合了多个Tool + 特定Prompt + 流程）
```

## 三、Skill 的本质：能力封装

```python
# 一个Skill的伪代码示例
class TechnicalResearchSkill:
    """技术调研Skill：从搜索到报告的完整流程"""
    
    name = "technical_research"
    description = "对某技术主题进行调研并生成报告"
    trigger = "用户要求'调研''分析''对比'某技术时"
    
    # 依赖的工具
    tools = ["web_search", "read_url", "write_file"]
    
    # 系统提示词（定义这个Skill的行为）
    system_prompt = """
    你是技术调研专家。流程：
    1. 搜索主题的最新信息
    2. 阅读关键文章提取要点
    3. 对比不同方案的优劣
    4. 生成结构化调研报告
    
    要求：客观、有数据支撑、标注来源
    """
    
    # 执行流程
    def execute(self, topic):
        # Step 1: 搜索
        results = self.tools["web_search"](topic)
        
        # Step 2: 深入阅读
        insights = []
        for url in results[:3]:
            content = self.tools["read_url"](url)
            insights.append(self.llm.extract_keypoints(content))
        
        # Step 3: 综合
        report = self.llm.generate_report(topic, insights)
        
        # Step 4: 保存
        self.tools["write_file"](f"{topic}_报告.md", report)
        
        return report
```

## 四、Skill 的价值

```
1. 标准化
   - 验证有效的流程固化，每次执行质量稳定
   - 不依赖"每次prompt写得好不好"

2. 可复用
   - 一次开发，多次使用
   - 跨项目/跨团队共享

3. 可分享（生态）
   - Skill可以打包成标准格式（如SKILL.md）
   - 社区共享，像"应用商店"
   - 例: Claude的Skills市场

4. 可组合
   - 复杂Skill可以调用简单Skill
   - 像积木一样构建复杂能力

5. 可迭代
   - Skill独立于Agent主逻辑
   - 优化某个Skill不影响整体
```

## 五、Skill 的设计原则

```
┌──────────────────────────────────────────────┐
│              Skill 设计原则                     │
├──────────────────────────────────────────────┤
│                                                │
│  1. 单一职责                                   │
│     一个Skill专注一类任务                       │
│     "写报告"不要既写技术报告又写财务报告         │
│                                                │
│  2. 清晰边界                                   │
│     明确什么场景用、什么场景不用                  │
│     避免与其他Skill职责重叠                      │
│                                                │
│  3. 明确Schema                                 │
│     输入输出格式定义清楚                        │
│     便于和其他Skill/Agent组合                   │
│                                                │
│  4. 容错设计                                   │
│     工具失败有降级方案                          │
│     不确定时返回"需要人工"                       │
│                                                │
│  5. 可测试                                     │
│     有标准测试用例                             │
│     能评估Skill质量                             │
│                                                │
└──────────────────────────────────────────────┘
```

## 七、面试加分点

1. **Tool vs Skill 的粒度区别**：Tool 是原子操作，Skill 是能力组合——这个区分是核心
2. **强调"复用"**：Skill 的价值是"一次开发多次使用"，像软件库一样
3. **提生态价值**：Skill 可分享可组合，是 Agent 生态的基础（类比 App 之于手机）

## 记忆要点

- 一句话定义：Skill是可复用的能力模块，封装了Prompt+Tools+执行流程
- 对比Tool与Skill：Tool是底层原子操作(如search)，Skill是多步组合的复杂业务能力(如调研)
- Skill五大核心组件：触发条件、提示词、依赖工具、执行流程、输入输出Schema
- 本质是高内聚低耦合：将常用复杂Agent处理逻辑固化为能力包，实现跨场景复用


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Skill 是"Prompt+工具+流程"的组合包，那它和直接写一个复杂 Prompt（包含工具说明和流程指引）有什么本质区别？为什么要封装成 Skill？**

Skill 的本质是"可复用、可管理的能力模块"，区别于一次性 Prompt 的三点：1）可复用——Skill 定义一次，多个 Agent/场景可调用（如"代码审查"Skill 在编程助手/CI/CD/教学场景都能用），复杂 Prompt 通常只服务于一个场景；2）元数据管理——Skill 带触发条件、输入输出 Schema、依赖声明、版本号，可以被框架发现、注册、调度，复杂 Prompt 是"裸文本"无元数据；3）可演进——Skill 独立迭代（改 Skill 不影响其他部分），复杂 Prompt 嵌在应用里改起来牵一发动全身。封装成 Skill 的价值是"从一次性脚本变成可管理的组件"，类似从"内联 SQL"到"存储过程"的演进。

### 第二层：证据与定位

**Q：一个 Skill 执行效果不好（用户说"这个能力不行"），怎么定位是 Skill 的 Prompt 写得差、工具不对，还是流程设计错？**

拆解 Skill 的三层。1）Prompt——单独把 Skill 的 prompt 拿出来跑（固定输入），看输出质量，差则是 prompt 问题；2）工具——看 Skill 内部调用的工具是否返回了正确结果（工具失败/不准会拖垮 Skill），查工具调用日志；3）流程——看 Skill 的多步骤流程是否合理（如"先查后改"还是"直接改"），流程错会导致即使每步对整体也错。定位方法：把 Skill 的执行 trace 分层（prompt 调用/工具调用/流程节点），找第一个"输入对输出错"的层。常见根因：prompt 不清晰（LLM 不知道该干什么）、工具能力不足（工具返回不准）、流程顺序错（该先验证后执行却反了）。

### 第三层：根因深挖

**Q：Skill 要带"触发条件"（什么时候用这个 Skill），但触发条件写不准（该触发没触发、不该触发误触发）怎么优化？**

触发条件要"显式 + 多维"。1）显式意图触发——Skill 注册时声明"触发意图"（如"用户请求代码审查时"），意图识别器（分类器/LLM）判断当前对话意图，匹配则触发；2）关键词/实体触发——声明触发关键词（如"review/审查/检查代码"），命中触发（补充意图识别的漏检）；3）置信度阈值——意图识别给置信度，高（>80%）直接触发，中（50-80%）询问用户"是否要代码审查"，低（<50%）不触发；4）误触发反馈——用户"取消 Skill"（触发错了）的信号反向优化触发条件（如"发现和 X Skill 混淆，加排除条件"）。触发条件的优化是持续的，上线后用真实触发数据迭代。

**Q：Skill 内部要调工具，但 Skill 和工具的边界经常模糊（这个能力是做成 Skill 还是做成工具），怎么划分？**

按"粒度和复杂度"划分。1）工具是"原子能力"——单一、通用、无状态（如"读文件""搜索""发邮件"），不包含业务逻辑；2）Skill 是"组合能力"——多个工具+流程+领域知识的组合，解决一个业务问题（如"代码审查"=读文件+静态分析+LLM 审查+生成报告），包含业务流程。判别：如果一个能力"只做一件事且通用"（如搜索），是工具；如果"做多件事且针对特定业务"（如代码审查），是 Skill。类比：工具是"螺丝刀"（原子），Skill 是"换轮胎"（组合螺丝刀+千斤顶+流程）。边界原则：工具尽量原子化（可被多个 Skill 复用），Skill 封装业务流程（组合工具）。

### 第四层：方案权衡

**Q：Skill 和"多 Agent"（把能力分给不同 Agent）都是"能力封装"，区别是什么？什么时候用 Skill 什么时候用多 Agent？**

粒度和自主性不同。1）Skill——是"单个 Agent 内的能力模块"，由主 Agent 决定何时调用，Skill 本身没有自主决策权（执行完返回结果给主 Agent），适合"流程明确、无需自主决策"的能力（如"格式化 JSON""生成报告模板"）；2）多 Agent——每个 Agent 有自主决策权（独立做规划/工具调用），适合"需要独立推理和决策"的能力（如"研究员 Agent 自主调研""审核员 Agent 自主判断"）。选型：流程固定的用 Skill（轻量，主 Agent 调用即可）；需要自主决策的用多 Agent（重量，独立运行）。实务：简单能力封装成 Skill（省协调成本），复杂能力才拆成多 Agent。

**Q：Skill 听起来好，但如果每个能力都做成 Skill，Skill 数量会爆炸（几十上百个），管理又是新问题，怎么控制？**

分层管理 + 复用。1）核心 Skill 精选——只把"高频+复杂+跨场景"的能力做成 Skill（如代码审查、数据分析、文档生成），低频/简单的用 prompt 即可（不值得封装）；2）Skill 组合——复杂 Skill 可由简单 Skill 组合（如"发布代码"=代码审查 Skill+测试 Skill+部署 Skill），避免每个能力都独立成 Skill；3）Skill 市场+治理——Skill 注册到市场，有评分/使用频率/维护者，低质/低频的淘汰，保持 Skill 库精简。目标是"几十个高质量 Skill"而非"几百个碎片 Skill"。原则：宁少而精，不要多而杂。

### 第五层：验证与沉淀

**Q：你怎么衡量一个 Skill 的质量（值得维护还是该下线）？**

四个指标。1）使用频率——Skill 被调用的次数/频率，低频（如月均 <10 次）考虑下线（不值得维护）；2）成功率——Skill 执行成功的比例（失败率高说明质量差或工具不稳），目标 >85%；3）用户满意度——Skill 输出的用户反馈（评分/是否采纳），低说明 Skill 能力不足；4）复用度——Skill 被多少场景/Agent 调用，低复用（只一个场景用）可能不值得封装成 Skill（直接内联更简单）。综合：高频+高成功+高满意+高复用的 Skill 是核心资产重点维护；低频/低质/低复用的下线。定期（季度）review Skill 库，淘汰长尾。

**Q：Skill 的设计和管理怎么沉淀成团队的 Skill 平台？**

建 Skill 平台：1）Skill 规范——定义 Skill 的标准结构（元数据+prompt+工具依赖+流程+输入输出 Schema+示例），不符合规范的拒绝注册；2）Skill 市场——注册中心，开发者上传/发现/复用 Skill，带评分/使用频率/版本；3）Skill 调度——框架根据对话意图自动触发匹配的 Skill（基于触发条件），开发者不用手写"if 话题是 X 则调 Skill Y"；4）Skill 监控——使用频率/成功率/满意度自动统计，低质的提示下线；5）Skill 组合——支持 Skill 组合（Skill 调 Skill），构建复杂能力。这套写入团队 Agent 平台 SOP，让 Skill 从"个人写的脚本"变成"团队共享的能力资产"。
