# 什么是渐进式披露（Progressive Disclosure）？

## 一、核心思想

```
传统做法（一次性全给）：
  给LLM: 所有文档全文 + 所有工具描述 + 完整记忆
  问题：上下文爆炸 + 关键信息被稀释 + 成本高

渐进式披露（按需给）：
  Level 1: 给概述（工具一句话描述）
       ↓ LLM判断需要哪个
  Level 2: 给摘要（选中工具的详细描述）
       ↓ LLM决定调用
  Level 3: 给详情（执行时的完整文档）
  
  每一层只展示必要信息，按需深入
```

## 二、三层披露模型

```
┌──────────────────────────────────────────────┐
│  Level 1: 概述层（Always visible）             │
│    工具名 + 一句话功能                         │
│    Token: ~50/工具                            │
│    作用：让LLM知道"有什么能力"                  │
│    示例: "web_search: 搜索互联网"              │
├──────────────────────────────────────────────┤
│  Level 2: 详情层（On demand）                  │
│    完整描述 + 参数schema + 使用示例             │
│    Token: ~200/工具                           │
│    作用：让LLM知道"怎么用"                     │
│    触发：LLM从L1选中后展开                     │
├──────────────────────────────────────────────┤
│  Level 3: 全文层（Just-in-time）               │
│    完整文档 + 所有示例 + 边界case               │
│    Token: ~1000+/工具                         │
│    作用：让LLM"精通使用"                       │
│    触发：复杂场景/LLM主动请求                   │
└──────────────────────────────────────────────┘
```

### 场景 3：Skill 披露

```

    "可用技能: 调研/写代码/做图表/翻译..."
```

## 四、实现机制

```python
class ProgressiveDisclosureSystem:
    def __init__(self):
        self.layers = {}  # 信息分层存储
    
    def serve(self, query, current_context):
        """根据当前需求，提供合适层级的信息"""
        
        # 判断当前需要哪个层级
        needed_level = self.assess_needs(query, current_context)
        
        if needed_level == "overview":
            # 信息充足，给概述即可
            return self.get_overview(query)
        
        elif needed_level == "detail":
            # 需要细节，展开
            return self.get_detail(query, focus=self.identify_focus(query))
        
        elif needed_level == "full":
            # 需要全部
            return self.get_full(query)
    
    def assess_needs(self, query, context):
        """评估当前需要哪个层级"""
        if context.has_seen_overview:
            if context.needs_detail:
                return "detail"
            return "overview"
        return "overview"
```

## 五、价值

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 全量展示            │ 渐进式披露             │
├──────────────┼──────────────────┼──────────────────────┤
│ Token成本    │ 高（全给）          │ 低（按需）             │
│ 准确率       │ 中（信息多易稀释）  │ 高（聚焦关键）         │
│ 延迟         │ 高（处理大量token） │ 低（处理少量）         │
│ 可扩展性     │ 差（工具多就爆）    │ 好（始终可控）         │
└──────────────┴──────────────────┴──────────────────────┘

特别适合：
  - 工具/Skill数量多（20+）
  - 文档库大（RAG）
  - 长期记忆丰富
```

## 六、面试加分点

1. **"剥洋葱"类比**：从概述到详情逐层深入，形象易懂
2. **核心是"匹配需求"**：不是藏信息，而是给当前最需要的粒度
3. **解决规模化问题**：工具/记忆越多，渐进式披露越重要——这是 Agent 规模化的关键

## 记忆要点

- 核心思想：按需给信息，避免一次性全塞导致上下文爆炸和关键信息被稀释
- 三层披露模型：L1概述常驻(知道有什么) → L2详情按需展开(知道怎么用) → L3全文即时加载(精通细节)
- 应用场景极广：工具描述、长文档RAG、Agent技能库均可采用此分层加载机制
- 对比传统方式：全量加载耗Token且易失焦，渐进式用最少Token实现精准唤醒


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：渐进式披露（Progressive Disclosure）是"不一开始全塞，按需逐步展示"，为什么不直接把所有信息给 LLM，让 LLM 自己挑有用的？**

因为 LLM 的 attention 在信息过多时会稀释，导致"信息在那但没用到"。1）全塞的问题——context 膨胀（塞几千 token 的详情）、attention 分散（关键信息被淹没）、成本上升（每轮 forward 全部信息）。2）渐进披露的价值——先给概述（如"这个项目有 10 个文件，主功能是 X"），LLM 判断"需要哪个文件的详情"时再加载详情，context 始终精简、attention 聚焦。本质是"把 LLM 当作注意力有限的决策者，给它当前决策所需的最小信息，而非全部信息"。类比人类查资料：先看目录（概述），决定看哪章，再看具体章节，而非一开始把整本书塞脑子。

### 第二层：证据与定位

**Q：渐进式披露要 LLM"判断需要什么详情再加载"，但 LLM 判断错了（加载了无关详情/漏了关键详情），怎么定位？**

看 LLM 的"加载决策"日志。1）加载决策——每次 LLM 决定"加载哪个详情"时记录（如"LLM 选择加载 file_A 详情"），对比"应该加载什么"（人工标注或 golden），不一致就是判断错；2）加载后效果——加载后 LLM 是否用了详情解决问题，如果加载了但还是答错，可能是详情不对（判断错）或详情对但 LLM 没用（attention 问题）。定位方法：trace 每次加载决策，统计"加载准确率"（加载对了/总加载），低则优化概述质量（概述不准导致 LLM 判断错）。常见根因：概述太粗（LLM 不知道有什么详情可选）或概述误导（概述和实际详情不符）。

### 第三层：根因深挖

**Q：渐进式披露的"概述层"很关键（LLM 靠它判断要什么详情），但概述怎么写得既精简又信息量足够？**

概述要"结构化 + 关键信号"。1）结构化——概述用固定结构（如"项目: X / 主功能: A,B,C / 文件: file1(功能D), file2(功能E)..."），而非自然语言段落，结构化让 LLM 快速扫描定位；2）关键信号——概述包含"决策所需的关键信息"（如文件的功能标签、数据的范围、工具的适用场景），让 LLM 据此判断"要不要加载详情"；3）精简——概述控制在几百 token（不超过总 context 的 20%），超出说明概述不够抽象。验证：给 LLM 概述，看它能否"不加载详情就大致判断该用哪个详情"，能则概述够，不能则概述缺关键信号。

**Q：渐进式披露要多轮交互（概述→加载详情→可能再加载更深详情），延迟比"一次全给"高，怎么平衡？**

按"信息量和延迟"权衡。1）小信息量——直接全给（如文档只有 2 页，一次塞完比多轮快）；2）大信息量——渐进披露（如代码库几百文件，全塞塞不下且慢，渐进反而更快——每次只 forward 小 context，比一次 forward 巨大 context 快）；3）预加载优化——对"高概率会加载的详情"预加载（如用户问代码问题，预测会用到的相关文件预读，减少交互轮次）。所以渐进披露在"信息量大"时反而更快（每轮 forward 小 context），不是绝对比全给慢。判断阈值：信息量 >几万 token 时用渐进，小于时全给。

### 第四层：方案权衡

**Q：渐进式披露 vs RAG（向量检索相关信息），两者都解决"信息多挑相关的"，区别是什么？怎么选？**

机制不同。1）渐进式披露——LLM 主动决策"加载什么"（基于概述判断），是"主动探索"，适合"信息有层次结构"（如代码库的文件树、文档的章节）的场景；2）RAG——向量检索"相关的是什么"（基于 query 语义匹配），是"被动检索"，适合"信息无明确结构、靠语义相关"（如知识库的文章）的场景。选型：信息有结构（代码/数据库 schema）用渐进披露（LLM 按结构探索）；信息无结构（文档库）用 RAG（语义检索）；混合——RAG 先召回 top-K 相关（语义），渐进披露让 LLM 决定"看哪个的详情"（结构化决策）。两者互补，不矛盾。

**Q：渐进式披露要求 LLM 多次决策（加载什么），每次决策都要调 LLM（成本），为什么不直接用规则预加载（代码写死加载逻辑）省成本？**

规则预加载快但僵化，LLM 决策慢但灵活。1）规则预加载——如"用户问代码问题就加载所有相关文件"，简单快，但"相关"的判断是硬编码的（可能漏/多），且无法适应动态情况（如用户其实只想问文档不想看代码）；2）LLM 决策——根据当前对话动态判断"需要什么"，准且灵活，但每轮多调 LLM。折中：1）规则兜底——常见模式用规则预加载（如"代码问题加载 README+主文件"），覆盖 80% 高频情况（快）；2）LLM 兜底——规则没覆盖的 20% 用 LLM 决策（灵活）。这样多数快、少数准，兼顾成本和灵活。

### 第五层：验证与沉淀

**Q：你怎么证明渐进式披露比"全塞"效果好（而非"省了 context 但漏了信息"）？**

AB 测试。固定复杂任务集（如代码问答、长文档问答），对比：1）全塞（信息全给，假设塞得下）；2）渐进披露（概述+按需加载）。指标：1）任务完成率——渐进应持平或更高（如果低，说明加载漏了信息）；2）context token 数——渐进应显著少（节省 forward 成本）；3）延迟——渐进在小信息量时可能略多（多轮交互），大信息量时更少（每轮小 context）。最优是"完成率持平 + token 省 + 延迟可接受"。还要监控"加载准确率"（LLM 加载对了/应该加载的），低则优化概述。

**Q：渐进式披露的设计怎么沉淀成框架能力？**

封装成 ProgressiveDisclosure 组件：1）概述生成器——自动从大量信息生成结构化概述（如代码库的文件树+功能标签、文档的章节摘要）；2）加载接口——LLM 通过统一接口（如 load_detail(file_id)）按需加载，框架负责检索具体内容；3）预加载策略——内置常见预加载规则（代码问题预加载 README 等）+ LLM 决策兜底；4）context 管理——自动管理加载的内容（已加载的缓存、不用的卸载，控制 context 总量）。开发者定义"信息的层次结构+概述生成方式"，框架自动支持渐进披露。这套写入团队 Agent 框架 SOP，新 Agent 处理大信息量时自动用渐进披露。
