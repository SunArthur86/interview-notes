# 大模型工具调用(Function Calling)能力是如何训练出来的？SFT和DPO分别解决什么问题？

## 训练全流程

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  阶段1: 预训练 (Pre-training)                     │
│  ────────────────────────────                    │
│  海量文本 → 语言理解能力                          │
│  ⚠️ 不具备工具调用能力                            │
│                                                  │
│  阶段2: SFT 监督微调 (Supervised Fine-Tuning)    │
│  ─────────────────────────────────────           │
│  工具调用数据集 → 学会格式 + 参数填充             │
│  ✅ 教会: 输出合法JSON、何时调用、补齐必填参数     │
│  ❌ 缺陷: 只能复刻样本，无法区分调用好坏          │
│                                                  │
│  阶段3: DPO/RLHF 偏好对齐 (Preference Alignment)  │
│  ────────────────────────────────────────        │
│  偏好对(好vs坏调用) → 优化决策质量                │
│  ✅ 教会: 自主判断要不要调、调用顺序、何时停止    │
│  ✅ 减少: 死循环、无效调用、幻觉调用              │
│                                                  │
└──────────────────────────────────────────────────┘
```

## SFT 阶段详解

### 数据集构造

```python
# 每条训练数据包含：工具描述 + 用户问题 + 标准调用JSON
training_example = {
    "tools": [
        {
            "name": "search_restaurant",
            "description": "搜索餐厅",
            "parameters": {
                "type": "object",
                "properties": {
                    "cuisine": {"type": "string"},
                    "location": {"type": "string"},
                    "price_max": {"type": "integer"}
                },
                "required": ["location"]
            }
        }
    ],
    "messages": [
        {"role": "user", "content": "附近有什么川菜馆"},
        {"role": "assistant", "content": None,
         "tool_calls": [{
             "name": "search_restaurant",
             "arguments": {"cuisine": "川菜", "location": "附近"}
         }]},
        {"role": "tool", "content": '{"results": [...]}'},
        {"role": "assistant", "content": "为您找到以下川菜馆..."}
    ]
}
```

### SFT 解决的问题

| 能力 | 说明 |
|------|------|
| **格式正确** | 输出合法JSON Schema，不是自然语言 |
| **参数完整** | 补齐必填参数，从用户输入中提取 |
| **时机判断** | 识别何时需要调用工具 vs 直接回答 |
| **多工具选择** | 从N个工具中选择正确的工具 |

### SFT 的局限

- 无法区分"好的调用"和"坏的调用"(都是合法JSON)
- 可能产生无意义的重复调用(格式对但逻辑错)
- 可能调用不存在的工具(幻觉调用)

## DPO/RLHF 阶段详解

### 偏好对构造

```python
# 同一个问题下，构造好vs坏的调用对
preference_pair = {
    "prompt": "帮我订明天晚上的川菜馆",

    "chosen": {  # ✅ 好的调用
        "tool": "book_restaurant",
        "arguments": {
            "cuisine": "川菜",
            "date": "2026-06-29",
            "meal": "dinner"
        },
        "reason": "参数完整，意图正确"
    },

    "rejected": {  # ❌ 坏的调用
        "tool": "search_restaurant",
        "arguments": {"cuisine": "川菜"},
        "reason": "用户要订座不是搜索，且缺少日期参数"
    }
}
```

### DPO vs RLHF 对比

| 维度 | DPO | RLHF |
|------|-----|------|
| **训练信号** | 偏好对(chosen vs rejected) | Reward Model打分 + PPO |
| **训练流程** | 直接优化(无需RM) | 先训RM，再PPO优化 |
| **计算成本** | 低(类似SFT) | 高(需要RM+Actor+Critic) |
| **收敛速度** | 快 | 慢(需多轮迭代) |
| **稳定性** | 高 | 低(PPO超参敏感) |
| **工业界偏好** | ⭐首选 | 大厂复杂场景用 |

### DPO 训练公式

```
L_DPO = -E[log σ(β · (log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x)))]

其中:
  y_w = chosen(好的调用)
  y_l = rejected(坏的调用)
  π = 当前策略, π_ref = 参考策略(SFT模型)
  β = 温度参数, σ = sigmoid
```

## 无微调方案：纯Prompt模拟

```python
# 开源模型不微调，纯靠Prompt也能模拟Function Calling
# 但稳定性远低于原生微调模型
FUNCTION_CALLING_PROMPT = """
你可以使用以下工具：
{tools_schema}

用户问题：{user_input}

如果需要调用工具，输出：
<tool_call>{"name": "...", "arguments": {...}}</tool_call>
"""

# ⚠️ 问题：
# 1. 格式不稳定(可能输出纯文本)
# 2. 参数提取不准
# 3. 无法可靠判断何时停止调用
# 4. 复杂多工具场景容易混乱
```

## 工业界最佳实践

```
工具调用能力 = SFT(70%能力) + DPO(30%提升)

SFT数据量: 5K-50K条高质量工具调用样本
DPO偏好对: 1K-10K对好/坏调用对比
评估指标:
  - 格式准确率: JSON Schema合规率
  - 参数准确率: 必填参数完整率
  - 工具选择准确率: 选对工具的比例
  - 冗余调用率: 不必要的重复调用比例
```

## 记忆要点

- SFT解决能力有无：教格式对齐(合法JSON)与参数填充，但只懂模仿
- DPO解决质量好坏：通过好vs坏偏好对，教自主判断、调用顺序与停止
- 因为SFT无法区分逻辑好坏(如死循环)，所以必须接DPO优化决策
- 训练三部曲：预训练打底 → SFT学格式工具 → DPO/RLHF对齐偏好


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：大模型的工具调用能力为什么不是预训练自带的，必须靠 SFT 训练？**

预训练学的是"自然语言的统计分布"（token 序列预测），工具调用要求的"结构化 JSON 输出 + 按 schema 填参数"不是自然语言的分布。预训练数据里虽然有代码和 JSON，但占比不足以让模型"稳定输出工具调用格式"。SFT 的作用是"用大量(指令, 工具 schema, 正确调用)数据微调，让模型把工具调用变成条件反射"。动机是把"概率性的格式遵从"固化成"稳定的结构化输出能力"，预训练做不到这种稳定性。

### 第二层：证据与定位

**Q：SFT 训练后模型在简单工具调用上准确率高，但复杂工具（参数嵌套）准确率低，怎么定位是数据不够还是任务太难？**

看训练数据的分布。1) 如果训练数据里"参数嵌套"的工具调用样本少（< 10%），是数据不够——模型没学会嵌套模式；2) 如果数据充足但仍不准，是任务本身难（嵌套参数的 schema 复杂，模型容量不够）。诊断：在 eval 集上按"工具复杂度"分桶看准确率，如果简单工具 95%、复杂工具 70%，且训练数据里复杂工具样本少，是数据问题。解法：补充复杂工具的 SFT 样本（5000+ 条）。

### 第三层：根因深挖

**Q：SFT 教会了"格式"和"工具选择"，但模型偶尔选对工具却填错参数（语义错，如 order_id 填了 user_id），根因是什么？**

根因是"模型没理解参数的语义"。SFT 学的是"(指令, schema, 正确 JSON)"的映射，但如果训练数据里没有"order_id 和 user_id 区分"的负样本（如故意填错让模型学区分），模型靠 schema 的 description 猜参数来源，可能猜错。解法：1) schema 的参数 description 写清楚"从哪里取"（如 order_id: "从用户的'我的订单'列表中选择的订单 ID，12 位数字"）；2) 训练数据加入"易混淆参数"的负样本（order_id 填成 user_id → 纠正）。

**Q：那为什么不直接用 DPO（不需要正确答案，只要偏好对）来教参数语义？**

DPO 可以但 SFT 先做。SFT 用"绝对正确"的样本教模型"标准答案长什么样"，DPO 用"偏好对"（A > B）教模型"在多个选项里选更好的"。参数语义的"正确性"是绝对的（order_id 就是 order_id，不是 user_id），适合 SFT（有标准答案）。DPO 更适合"多个选项都可行但要选最优"的场景（如"哪个回答更有用"）。所以教参数语义先 SFT（有标准答案），DPO 用于后续优化"在合法调用里选更优的"。

### 第四层：方案权衡

**Q：SFT 和 DPO 解决不同问题，为什么不只做 SFT（有标准答案就够了）？**

因为 SFT 有"天花板"。SFT 教模型"模仿标准答案"，但有些场景"标准答案"不唯一——如用户问"退款"，可以调 search_order 也可以调 search_refund，两者都"合法"但场景上可能有一个更优。SFT 只能教"标注的标准答案"（可能不是最优），DPO 用偏好对（"调 search_refund > search_order"）教模型"在合法选项里选更优的"。经验上 SFT 把准确率到 85%，DPO 再提升到 92-95%。DPO 解决的是"SFT 的优化天花板"。

**Q：为什么不直接用 RLHF（更通用的偏好优化），而用 DPO？**

RLHF 要训练奖励模型（Reward Model），额外成本和复杂度。DPO 直接用偏好对优化策略，绕过 RM，简单稳定。在"工具调用"这种"有明确偏好"的场景（正确调用 > 错误调用），DPO 够用且更省。RLHF 适合"奖励复杂、需要建模多目标"的场景（如 helpfulness + harmlessness 的组合）。工具调用场景偏好明确，DPO 性价比更高。DeepSeek 的实践也验证了 DPO 在工具调用优化上的有效性。

### 第五层：验证与沉淀

**Q：怎么衡量 SFT + DPO 训练的工具有效性？**

三层评估：1) 格式准确率——输出是否合法 JSON（> 98%）；2) 工具选择准确率——选对工具（> 92%）；3) 参数语义准确率——选对工具且参数语义正确（> 88%）。三层分阶段看：SFT 后格式和选择达标，DPO 后参数语义提升。沉淀为工具调用训练规范：SFT 数据量（50000+）、数据多样性（覆盖所有工具和参数类型）、DPO 偏好对的构建方法、eval 集设计。

## 结构化回答

**30 秒电梯演讲：** 大模型的工具调用能力不是预训练自带的，而是通过SFT教会格式+DPO/RLHF优化决策质量两阶段训练获得的——SFT是驾校学基本操作(打方向盘、踩油门)；。

**展开框架：**
1. **预训练阶段** — 预训练阶段不具备工具调用能力
2. **SFT** — 构造工具调用数据集，教会格式和参数填充
3. **DPO/RLHF** — 构造偏好对，优化调用决策(何时调/调什么/何时停)
