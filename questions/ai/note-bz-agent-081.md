# 如何有效缓解 / 检测 / 抑制大模型幻觉？

## 一、幻觉的本质与类型

```
幻觉(Hallucination) = LLM生成不真实/无依据的内容

类型：
┌──────────────────┬──────────────────────────────┐
│ 类型              │ 表现                            │
├──────────────────┼──────────────────────────────┤
│ 事实性幻觉        │ 编造不存在的事实                │
│                  │ "爱因斯坦2020年获得了诺贝尔奖"  │
├──────────────────┼──────────────────────────────┤
│ 忠实性幻觉        │ 无视给定的参考文档编造          │
│                  │ RAG给了正确文档但仍答错         │
├──────────────────┼──────────────────────────────┤
│ 推理幻觉          │ 推理过程错误导致结论错          │
│                  │ 数学计算步骤有误                │
├──────────────────┼──────────────────────────────┤
│ 来源幻觉          │ 编造引用来源                    │
│                  │ "据《Nature》2023年论文..."(不存在)│
└──────────────────┴──────────────────────────────┘
```

## 二、缓解：生成前（预防）

### RAG 提供事实

```python
# 最有效的幻觉缓解手段
def answer_with_rag(question):
    # 检索真实文档
    docs = retriever.search(question)
    
    # 强制基于文档回答
    prompt = f"""
    只基于以下文档回答。不要使用文档外的知识。
    如果文档中没有答案，说"根据提供的信息，我无法回答"。
    
    文档: {docs}
    问题: {question}
    """
    return llm.generate(prompt)
```

### Prompt 约束

```python
ANTI_HALLUCINATION_PROMPT = """
回答规则：
1. 只说你确定的内容

3. 不要编造数字/日期/人名/引用
4. 区分"事实"和"推测"，推测要标注
5. 被问到不知道的，诚实说不知道
```

## 四、缓解：生成后（验证）

### 自我验证（Self-Check）

```python
def generate_with_self_check(question):
    # 1. 生成初始答案
    answer = llm.generate(question)
    
    # 2. 让LLM自检
    check_prompt = f"""
    问题: {question}
    答案: {answer}
    
    请检查答案中的每个事实声明：
    - 哪些是你确定的？
    - 哪些可能不准确？
    - 有无编造的信息？
    """
    check_result = llm.generate(check_prompt)
    
    # 3. 根据自检修正
    if check_result.has_issues:
        return llm.revise(answer, check_result.issues)
    return answer
```

### 交叉验证（Cross-Check）

```python
def cross_validate(question, n=3):
    # 生成多个答案
    answers = [llm.generate(question, temperature=0.7) for _ in range(n)]
    
    # 比较一致性
    # 多个答案一致的部分 → 可信
    # 不一致的部分 → 可能是幻觉，标注不确定
    return reconcile(answers)
```

### 外部事实核查

```python
def fact_check(response):
```

## 五、检测幻觉的方法

```python
class HallucinationDetector:
    """检测生成内容的幻觉"""
    
    def detect(self, response, context=None):
        methods = []
        
        # 方法1: Faithfulness（有context时）
        if context:
            faith = self.faithfulness_score(response, context)
            # 答案中每个claim是否在context有支持
            methods.append(("faithfulness", faith))
        
        # 方法2: 置信度分析
        logprobs = self.get_logprobs(response)
        confidence = self.analyze_confidence(logprobs)
        # 低logprob的部分可能是不确定的(幻觉)
        methods.append(("confidence", confidence))
        
        # 方法3: 一致性检测
        consistency = self.check_consistency(response)
        # 多次生成，看关键事实是否一致
        methods.append(("consistency", consistency))
        
        # 方法4: 引用验证
        citations = self.verify_citations(response)
        methods.append(("citations", citations))
        
        return aggregate(methods)
```

## 六、各方法的局限

```
┌──────────────────┬──────────────────────────────┐
│ 方法              │ 局限                            │
├──────────────────┼──────────────────────────────┤
│ RAG              │ 文档本身可能有错/检索不到       │
│ Prompt约束        │ LLM可能不遵守                  │
│ 低温              │ 降低创造性，不能消除幻觉        │
│ 自检              │ LLM可能"自信地错"(检不出)      │
│ 交叉验证          │ 多次错则验证失效               │
│ 事实核查          │ 不是所有事实都能搜到           │
└──────────────────┴──────────────────────────────┘

结论：没有单一方法能完全消除幻觉
      → 多层防护 + 接受残余风险 + 人工兜底
```

## 七、不同场景的幻觉容忍度

```
┌──────────────┬──────────┬──────────────────────┐
│ 场景          │ 容忍度    │ 防护等级              │
├──────────────┼──────────┼──────────────────────┤
│ 医疗/法律/金融 │ 极低     │ RAG+人工审核+免责声明│

原则：风险越高，防护越重；风险低可放宽（避免过度工程）
```

## 八、面试加分点

1. **三阶段缓解**：生成前(RAG/Prompt)+生成中(低温/CoT)+生成后(自检/核查)
2. **无法完全消除**：幻觉是概率模型固有特性，只能减少——实事求是
3. **场景化容忍**：医疗零容忍vs闲聊高容忍——按风险分级防护

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说幻觉"无法完全消除只能减少"，那对一个医疗问答场景，"减少到多少"算可接受？你的验收标准是什么？**

医疗场景的可接受标准是"事实性幻觉率 < 0.1%"且"任何涉及诊断/用药的输出必须带来源引用"。这个标准不是拍脑袋，是对照行业——医疗 AI 产品（如 FDA 批准的）对事实错误的要求是接近零容忍。但完全 0% 不现实，所以用"硬约束兜底"：涉及诊断/用药的输出强制走 RAG + 人工复核，未带可信来源引用的直接拒绝输出（返回"建议咨询医生"）。所以验收标准是"可量化的幻觉率 + 关键场景的硬护栏"，不是追求绝对零。

### 第二层：证据与定位

**Q：你用 faithfulness 指标检测忠实性幻觉，但 faithfulness 高（答案都在文档里）不代表答案对——如果检索到的文档本身就是错的或过时的，faithfulness 检不出来，怎么办？**

faithfulness 只能保证"答案忠于检索文档"，不能保证"文档本身正确"，这是两个维度。文档正确性要在召回侧解决：一是知识库的更新机制（文档有版本和时效标记，过期的自动下线或标注），二是召回结果的质量评估（召回片段的相关性分数 < 阈值时不回答、转拒答）。定位方法：把幻觉样本分两类——faithfulness 低的是"模型无视文档编造"（生成侧问题），faithfulness 高但答错的是"文档本身错或召回错"（召回/知识库问题）。两类要分开治理，前者调 Prompt/模型，后者治知识库和召回。

### 第三层：根因深挖

**Q：你用交叉验证（多次采样取一致）防幻觉，但如果模型对某个错的事实"很自信"（多次都编一样的错答案），交叉验证就失效了，根因是什么？**

根因是"系统性幻觉"——模型在训练时学到了错误的知识（训练数据本身有错），所以多次采样都会稳定输出同一个错答案，交叉验证测的是"稳定性"不是"正确性"。这类幻觉靠交叉验证和自检都查不出来（因为模型内部认为这是对的）。解法必须引入"外部事实核查"——把答案里的关键事实声明抽出来，用外部知识源（搜索引擎、权威数据库）验证，外部查不到支持的标注为"未经验证"。所以对抗系统性幻觉的唯一办法是"模型外"的锚点（外部事实），模型内的方法（自检/交叉）都无效。

**Q：那为什么不直接所有输出都做外部事实核查，一步到位？**

因为成本和覆盖率的现实约束。外部事实核查要调搜索引擎或数据库，每条输出提取多个 claim 逐个查，延迟加 2-5 秒、成本翻倍，实时对话场景扛不住。而且不是所有事实都能查到——长尾知识、领域内部数据搜索引擎没有。所以我的策略是分级核查：高风险声明（数字/日期/人名/引用）强制查，低风险声明（观点/建议）不查。判断高风险用规则（含数字、含"根据研究表明"等措辞的 claim）+ 模型分类。这样把事实核查的量从全量降到 20%，在成本和覆盖率间取得平衡。

### 第四层：方案权衡

**Q：RAG 你说最有效，但 RAG 也可能让模型"过度依赖文档"——文档没说的它就不答，本来模型知道的知识也不敢用，导致召回率下降。你怎么权衡？**

这其实是期望的行为还是 bug，取决于场景。高风险场景（医疗/法律）"过度依赖文档、不知道就说不知道"是期望行为——宁可拒答也不能编。这类场景召回率下降可接受，用"拒答率"监控（拒答率 < 20% 算知识库覆盖够）。低风险场景（闲聊/创意）过度依赖文档反而伤害体验，这时 Prompt 要调成"文档有的优先用文档，文档没有的可以用你的知识但要标注'未经验证'"。所以不是 RAG 本身的问题，是 Prompt 约束的松紧要按场景调。我会按场景维护两套 Prompt 模板：严约束版（高风险）和松约束版（低风险）。

**Q：为什么不直接用更大的模型（GPT-4 级别），大模型幻觉率本来就低，省掉这一堆缓解手段？**

因为大模型只降低了"事实性幻觉"（训练知识更全），没降低"忠实性幻觉"（无视文档编造）和"实时性幻觉"（不知道训练后的事实）。在 RAG 场景，大模型和小模型的忠实性幻觉率差距没那么大（都可能无视文档），核心是 Prompt 约束和文档质量。而且大模型贵 10 倍、慢 5 倍，把缓解手段省下来的钱远不够付大模型的差价。正确架构是小模型 + RAG + 缓解手段，在 80% 场景达到接近大模型的事实准确率，只在最复杂的推理任务才上大模型。

### 第五层：验证与沉淀

**Q：你怎么持续监控幻觉率，证明你的缓解手段在线上长期有效，而不是上线那会儿好？**

建立"幻觉监控闭环"。线上全量采样（5-10%）跑 faithfulness 检测 + 每周人工抽检 200 条算 ground truth，输出幻觉率趋势图。关键是要分类型监控（事实性/忠实性/推理/来源），因为不同类型的治理手段不同，混在一起看趋势定位不了问题。当某类幻觉率上升时，对照近期的变更（Prompt 改了/模型升级了/知识库更新了）归因。这个监控要长期跑，因为幻觉率会随输入分布漂移和模型版本灰度而变化，不是上线验证一次就完。

**Q：幻觉治理经验怎么沉淀？**

沉淀成"幻觉防护配置矩阵"。按场景（医疗/客服/创意）× 防护手段（RAG/Prompt/自检/核查）建矩阵，每格写明该场景该手段的启用状态、参数、验收指标。新人接新场景查矩阵就知道该开哪些防护。配套一个"幻觉 case 库"——线上发现的幻觉样本入库，带类型/根因/治理动作，用于持续丰富评测集和训练分类器。这样幻觉治理是积累式的，case 库越丰富防护越准，而不是每次新项目从零开始。
