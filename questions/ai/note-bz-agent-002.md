# AI Agent 的核心能力有哪些？（规划、记忆、工具使用）

## 一、三大核心能力总览

```
                    目标输入
                       │
                       ▼
              ┌─────────────────┐
              │  1. 规划 Planning │ ← LLM大脑分解任务
              └────────┬────────┘
                       │ 步骤列表
                       ▼
              ┌─────────────────┐
              │  2. 记忆 Memory   │ ← 读取历史/上下文
              └────────┬────────┘
                       │ 相关信息
                       ▼
              ┌─────────────────┐
              │ 3. 工具 Tool Use  │ ← 执行真实操作
              └────────┬────────┘
                       │ 执行结果
                       ▼
                 观察并循环
                 （写到记忆里）
```

## 二、规划 Planning

**作用：** 把模糊目标拆解为可执行的有序步骤。

```python
# 1. 任务分解（Decomposition）
goal = "帮我做一份2026年大模型行业调研报告"
plan = llm.plan(goal)
# 输出：
# Step 1: 搜索"2026 大模型 行业"最新新闻
# Step 2: 整理头部公司动态（OpenAI/Anthropic/国内）
# Step 3: 提炼技术趋势（Agent/MoE/多模态）
# Step 4: 撰写报告大纲
# Step 5: 充实内容并自检

# 2. 失败重规划（Replanning）
result = execute(plan[0])
if result.failed:
    new_plan = llm.replan(goal, error=result.error, tried=plan[:0])
```

**规划范式：**
- **CoT（思维链）：** 线性逐步推理
- **ToT（思维树）：** 多分支探索，失败回溯
- **Plan-and-Execute：** 先全局规划再逐步执行
- **ReAct：** 推理与行动交替（Reason + Act）

## 三、记忆 Memory

**作用：** 跨步骤、跨轮次、跨会话保持状态。

```
┌──────────────────────────────────────────┐
│ 记忆层次                                    │
├──────────────────────────────────────────┤
│ 短期记忆 (Working/Context)                 │
│   = 当前对话窗口（LLM直接可见）              │
│   特点：快但容量有限（如128K token）         │
├──────────────────────────────────────────┤
│ 长期记忆 (Long-term)                       │
│   = 向量数据库（按语义检索）                 │
│   特点：容量大但需检索（有延迟+召回率问题）   │
├──────────────────────────────────────────┤
│ 任务记忆 (Episodic)                        │
│   = 当前任务的执行轨迹                       │
│   特点：记录"做了什么/结果如何"，用于重规划   │
└──────────────────────────────────────────┘
```

```python
# 记忆写入与检索
memory.write("用户偏好：喜欢简洁的报告风格", user_id="u1")
context = memory.retrieve("写报告", user_id="u1", top_k=5)
# 注入到下次prompt：[历史偏好] 用户喜欢简洁风格...
```

## 四、工具使用 Tool Use

**作用：** 让 Agent 拥有"感知"和"行动"能力，超越纯文本。

```python
tools = [
    # 感知类（读环境）
    {"name": "web_search", "desc": "搜索互联网"},
    {"name": "read_file", "desc": "读取本地文件"},
    {"name": "query_db", "desc": "查询数据库"},
    # 行动类（改环境）
    {"name": "send_email", "desc": "发送邮件"},
    {"name": "write_file", "desc": "写入文件"},
    {"name": "call_api", "desc": "调用外部API"},
]

# LLM决策：用哪个工具+传什么参数
decision = llm.tool_call(
    context="用户要订机票",
    tools=tools
)
# → {"name": "web_search", "args": {"q": "北京机票 6月"}}
observation = execute(decision)  # 真实执行
```

**工具使用的挑战：**
- 工具过多时选不准（30+工具命中率下降）
- 工具描述不清导致误用
- 调用失败如何重试/降级
- 参数格式不对（需要 schema 校验）

## 五、三大能力的协同：ReAct 循环

```python
# 经典ReAct循环，体现三大能力协同
def react_agent(goal, memory, tools):
    trajectory = []
    for step in range(MAX_STEPS):
        # 1. 规划+记忆：基于历史决定下一步
        thought = llm.reason(goal, trajectory, memory.recall(goal))
        # 2. 工具使用：决定调用什么
        action = llm.act(thought, tools)
        # 3. 观察：执行并记录
        observation = execute(action)
        # 4. 记忆更新
        trajectory.append((thought, action, observation))
        memory.write(trajectory[-1])
        # 5. 判断完成
        if llm.is_complete(goal, trajectory):
            return llm.finalize(goal, trajectory)
```

## 六、面试加分点

1. **强调"闭环"**：三大能力不是独立的，而是构成"感知-决策-行动"闭环，缺一环都会断链
2. **提"记忆最难"**：规划和工具相对成熟（靠强模型 + API 规范），记忆是最难工程化的（检索召回/用户隔离/遗忘策略）
3. **能力分级**：弱 Agent（固定流程+少量工具）→ 强 Agent（动态规划+长期记忆+多工具）→ 自主 Agent（自主设目标）

## 记忆要点

- 三大能力口诀：规划（拆解任务）、记忆（保持状态）、工具（执行操作）
- 规划范式：线性CoT、分支ToT、以及交替进行的ReAct（推理+行动）
- 记忆分层：短期为上下文窗口，长期为向量库检索，任务记忆记录执行轨迹
- 工具分类：分为感知类（读取环境如搜索）和行动类（改变环境如发邮件）


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你把 Agent 核心能力拆成"规划、记忆、工具"三个，为什么是这三个？加个"学习"或"沟通"能力不行吗？**

这三个是 Agent 闭环的最小完备集。规划解决"怎么干"（把目标拆成步骤），工具解决"能干什么"（突破 LLM 权重知识的边界），记忆解决"记住什么"（跨轮/跨会话的状态保持）。这三者覆盖了"感知-决策-行动"循环的全部环节。"学习"能力本质是通过记忆积累（把成功/失败经验存进长期记忆）实现的，不是独立能力；"沟通"是多 Agent 协作时的接口（属于工具/消息传递范畴）。如果把学习/沟通单列，会导致能力边界模糊（规划要不要学？工具要不要学？），反而难设计。三能力是正交分解：规划管决策，工具管执行，记忆管状态。

### 第二层：证据与定位

**Q：一个 Agent 在实际任务里表现差，你怎么判断是规划、记忆、还是工具的问题？**

用 trace 日志分层定位。把 Agent 每轮的 Thought/Action/Observation 打日志，按失败模式归类：1）规划问题——Thought 里把任务拆错了（如该先查 A 却先查 B）、漏掉关键步骤，表现是"方向就错了"；2）工具问题——选对方向但 tool_call 参数错（如 SQL 查错表）、选错工具（该 search 却 calc），表现是"方向对但执行错"；3）记忆问题——忘了之前的关键约束（如用户说"只查 2024 年"但后面忘了）、检索不到相关历史，表现是"重复已做的事或违反早期约束"。统计三类错误的占比，占比最高的就是短板。还可以做消融——分别禁用某能力看效果下降多少，下降最大的最关键。

### 第三层：根因深挖

**Q：假设定位是"规划能力弱"（任务拆解经常出错），这是 LLM 本身推理能力不行，还是规划框架（如 ReAct）的问题？**

用对照实验区分。固定同一个 LLM，分别用三种规划方式跑同一批任务：1）零规划（直接让 LLM 答）；2）CoT（"step by step"）；3）ReAct/Plan-Execute 框架。如果 0→1→3 的完成率递增（如 30%→50%→75%），说明框架有效、LLM 推理也够用，问题是之前没用对框架。如果 1→3 几乎没提升（如 50%→52%），说明 LLM 推理能力本身不足，换框架没用，要换更强的 base model。还可以看 Thought 的具体内容——如果 LLM 能写出正确的 Thought（推理对）但 Action 选错，是工具描述/决策层问题；如果 Thought 本身就错，是 LLM 推理能力问题，得上更强模型或 SFT 提升推理。

**Q：既然规划、记忆、工具是核心，为什么不三个能力都用最强的（最强 LLM 规划、最大向量库记忆、最多工具），堆到极致？**

因为三个能力互相耦合，堆到极致反而互相拖累。1）工具太多——30+工具的描述塞进 prompt 占 token，LLM 选不准（命中率从 90% 降到 60%），还会导致 context 过长稀释规划注意力；2）记忆太大——全量记忆检索慢、噪声多，反而降低相关性；3）规划太深——无限拆解子任务导致步骤过多，每步都要调 LLM+工具，延迟和成本爆炸。最优是平衡：工具精选（5-10 个高频的常驻，长尾的走 RAG 按需检索）、记忆分层（高价值精筛）、规划限深（max_turns 兜底）。Agent 设计是"够用就好"的工程，不是"越多越强"的堆料。

### 第四层：方案权衡

**Q：规划、记忆、工具三个能力，开发顺序怎么排？是并行做还是串行做？**

串行，顺序是"工具 → 规划 → 记忆"。理由：1）工具先行——没有工具 Agent 干不了实事（只会说），先把 1-2 个核心工具接通，跑通最小闭环（能调工具拿到结果）；2）规划跟上——有了工具，用 ReAct/CoT prompt 让 LLM 学会"何时调工具"，这时 Agent 能完成简单任务；3）记忆最后——简单任务不需要长期记忆（单轮就能完成），只有多轮/跨会话任务才需要记忆，且记忆的写入/检索逻辑复杂，放最后做（先验证 Agent 主流程通，再加记忆锦上添花）。这个顺序保证每一步都有可验证的产出，避免"三个能力一起做但都不通"。

**Q：为什么不把记忆设计成"自动全量记录所有交互"，简单可靠，而是要做复杂的"选择性记忆"？**

全量记录会让记忆变成垃圾场。1）信噪比崩塌——Agent 每轮的寒暄、重复确认、失败尝试都记下来，真正有价值的信息（用户偏好、关键决策、事实结论）被淹没，检索时 top-5 全是噪声；2）成本爆炸——全量记录意味着记忆库快速增长，几周后几十万条，检索延迟从 100ms 涨到 1s+，存储成本也飙升；3）遗忘的必要性——人类的记忆是有遗忘曲线的，不重要的自然淡忘，Agent 也需要。所以选择性记忆（LLM 给每条信息打重要性分，高分才存 + 低分自动 TTL 过期）不是"复杂"，是"必需"。全量记录是"懒得设计记忆策略"的偷懒，效果远差于精筛。

### 第五层：验证与沉淀

**Q：你怎么量化 Agent 三大能力的健康度，而不是"感觉规划差了点"这种模糊判断？**

每个能力配独立指标。规划：任务完成率、平均步数/最优步数比、规划错误率（人工标注 Thought 是否正确拆解）；记忆：记忆检索命中率（召回的相关记忆占应该召回的比例）、记忆引用率（Agent 生成时引用历史信息的比例）、记忆冲突率（新记忆和旧记忆矛盾的比例）；工具：tool_call_success_rate、工具选择准确率、参数填充正确率。三个能力的指标分别打到 dashboard，设健康阈值（如规划完成率>70%、记忆命中率>80%、工具成功率>85%），任一指标跌破阈值告警，定位到具体能力模块优化。

**Q：Agent 的三大能力设计经验怎么沉淀成团队可复用的组件库？**

封装成标准组件：1）Planner 组件——内置 CoT/ReAct/Plan-Execute 三种规划模式，开发者按任务复杂度选（简单 CoT、中等 ReAct、复杂 Plan-Execute），配 prompt 模板；2）Memory 组件——分层记忆（短期 context + 长期向量库 + 工作记忆文件），自动写入决策（LLM 打分）、自动检索（embedding+rerank）、自动 TTL，开发者配容量和阈值即可；3）Tool 组件——工具注册中心（MCP 标准）、工具检索（按需加载）、工具执行沙箱。三个组件通过标准接口（State/Action/Observation）协作，开发者像搭积木一样组合。这套组件库写入团队 Agent 框架，新项目复用，不再每个项目重新设计三大能力。
