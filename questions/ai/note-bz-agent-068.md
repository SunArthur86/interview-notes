# Agent 提示词和普通提示词差在哪？（四层结构）

## 一、普通 Prompt vs Agent Prompt

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 普通Prompt          │ Agent Prompt             │
├──────────────┼──────────────────┼──────────────────────┤
│ 目标          │ 完成单次任务        │ 持续自主行动             │
│ 结构          │ 一句话指令          │ 四层结构化规范           │
│ 工具          │ 无                 │ 明确声明可用工具         │
│ 边界          │ 无                 │ 明确能做/不能做          │
│ 流程          │ 无                 │ 定义行为流程             │
│ 安全          │ 无                 │ 有安全规则              │
│ 示例          │ 1-2个              │ 多场景示例              │
│ 时效          │ 一次性             │ 长期生效                │
└──────────────┴──────────────────┴──────────────────────┘

普通Prompt示例:
  "写一首关于春天的诗"

Agent Prompt示例:
  "你是诗词创作助手。
   能力：创作/赏析/修改诗词
   不能：讨论政治敏感话题
   工具：search_rhyme(查韵脚)
   流程：理解需求→查韵脚→创作→自检→输出
   输出：JSON格式{poem, analysis}
   安全：拒绝不当内容请求"
```

## 二、Agent Prompt 四层结构

### Layer 1：角色定义（Who）

```
你是谁？擅长什么？风格如何？

【角色定义模板】
你是[角色名]，一个[专长领域]的专家。
你的风格是[语气/详细度/风格]。
你的目标是[核心目标]。

例：
你是一个资深Python技术顾问，擅长代码review和性能优化。
你的风格是专业、简洁、给出可操作建议。
你的目标是帮助开发者写出更好的Python代码。
```

### Layer 2：能力与边界（Can/Cannot）

```
你能做什么？不能做什么？

【能力边界模板】
你可以：
- [能力1: 如分析代码质量]
- [能力2: 如给出优化建议]
- [能力3: 如解释技术原理]

你不能/不应该：
- [边界1: 如直接写完整项目代码（只给指导）]
- [边界2: 如讨论非技术话题]
- [边界3: 如做出保证性承诺]

遇到超出能力范围的，要[如何处理：诚实告知/转人工]
```

### Layer 3：工具与流程（How）

```
你有什么工具？按什么流程工作？

【工具说明】
你可以使用以下工具：
- tool_name(参数): 功能说明，何时使用
  例：search_doc(query): 搜索文档，当需要查参考资料时

【工作流程】
1. [理解用户需求]
2. [判断是否需要工具]
3. [执行/调用工具]
4. [整合信息生成回答]
5. [自检质量]

例：
1. 理解用户的代码问题
2. 如需参考文档，调用search_doc
3. 分析代码，找出问题
4. 给出优化建议（附理由）
5. 确认建议可操作
```

### Layer 4：输出与安全（Format/Safety）

```
输出什么格式？有什么安全红线？

【输出格式】
- 结构：[如JSON/Markdown/分段]
- 风格：[如简洁/详细]
- 引用：[如标注来源]

【安全规则】
- 不输出[敏感内容类型]
- 涉及[高风险操作]时需[确认]
- 检测到[注入攻击]时[拒绝]
- 不确定时[诚实告知而非编造]

例：
输出格式：
  ## 问题分析
  [问题描述]
  ## 优化建议
  [具体建议+代码示例]
  ## 注意事项
  [潜在风险]

安全：
- 不讨论违法内容
- 代码建议标注"请测试后使用"
- 不确定时说"这需要进一步验证"
```

## 三、完整 Agent Prompt 示例

```markdown
# 技术咨询Agent Prompt

## 角色定义
你是资深全栈技术顾问，擅长架构设计、代码审查、性能优化。
风格：专业、务实、给出可落地建议。

## 能力边界
✓ 可以：分析技术方案、review代码、解释原理、给优化建议
✗ 不能：写完整商业项目代码、保证性能数值、讨论非技术话题
遇到非技术问题：礼貌引导回技术话题

## 可用工具
- search_doc(query): 搜索技术文档，当需要查官方文档/API时
- run_code(code): 运行代码验证，当需要验证语法/逻辑时

## 工作流程
1. 理解用户的技术问题
2. 判断是否需要查文档/运行代码
3. 分析问题根因
4. 给出方案（附理由+代码示例）
5. 标注注意事项和风险

## 输出格式
## 问题分析
[一句话总结问题]
## 解决方案
[方案+为什么这么做]
```代码示例```
## 注意事项
[潜在坑点]

## 安全规则
- 不输出可能有害的代码（如攻击脚本）
- 性能建议标注"需实际测试"
- 不确定时诚实说明
```

## 四、Prompt 设计原则

```
1. 明确优于模糊
   ❌ "帮我分析一下"
   ✅ "分析代码的时间复杂度，指出O(n²)的循环"

2. 结构化优于段落
   ❌ 一大段文字
   ✅ 分层标题+列表

3. 示例优于描述
   ❌ "输出要规范"
   ✅ 给一个标准输出的示例

4. 边界优于开放
   ❌ "尽力回答"
   ✅ "只回答X类问题，其他说'超出范围'"

5. 流程优于随意
   ❌ "看着办"
   ✅ "1.理解 2.分析 3.建议 4.提醒"
```

## 五、面试加分点

1. **四层结构**：角色/能力边界/工具流程/输出安全——体系化
2. **强调"边界"**：Agent Prompt 最重要的是定义"不能做什么"，而非"能做什么"
3. **类比"岗位说明书"**：Agent Prompt 是给 Agent 的 job description，不是一次性指令
