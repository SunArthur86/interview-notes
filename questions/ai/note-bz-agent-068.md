# Agent 提示词和普通提示词差在哪？（四层结构）

## 一、普通 Prompt vs Agent Prompt

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 维度          │ 普通Prompt          │ Agent Prompt             │
├──────────────┼──────────────────┼──────────────────────┤
│ 目标          │ 完成单次任务        │ 持续自主行动             │
│ 结构          │ 一句话指令          │ 四层结构化规范           │
│ 工具          │ 无                 │ 明确声明可用工具         │
│ 边界          │ 无                 │ 明确能做/不能做          │
│ 流程          │ 无                 │ 定义行为流程             │
│ 安全          │ 无                 │ 有安全规则              │
│ 示例          │ 1-2个              │ 多场景示例              │
│ 时效          │ 一次性             │ 长期生效                │
└──────────────┴──────────────────┴──────────────────────┘

普通Prompt示例:
  "写一首关于春天的诗"

Agent Prompt示例:
  "你是诗词创作助手。
   能力：创作/赏析/修改诗词
   不能：讨论政治敏感话题
   工具：search_rhyme(查韵脚)
   流程：理解需求→查韵脚→创作→自检→输出
   输出：JSON格式{poem, analysis}
   安全：拒绝不当内容请求"
```

## 二、Agent Prompt 四层结构

### Layer 1：角色定义（Who）

```
你是谁？擅长什么？风格如何？

【角色定义模板】
你是[角色名]，一个[专长领域]的专家。
你的风格是[语气/详细度/风格]。
你的目标是[核心目标]。

例：
你是一个资深Python技术顾问，擅长代码review和性能优化。
你的风格是专业、简洁、给出可操作建议。
你的目标是帮助开发者写出更好的Python代码。
```

### Layer 2：能力与边界（Can/Cannot）

```
你能做什么？不能做什么？

【能力边界模板】
你可以：
- [能力1: 如分析代码质量]
- [能力2: 如给出优化建议]
- [能力3: 如解释技术原理]

你不能/不应该：
- [边界1: 如直接写完整项目代码（只给指导）]
- [边界2: 如讨论非技术话题]
- [边界3: 如做出保证性承诺]

遇到超出能力范围的，要[如何处理：诚实告知/转人工]
```

### Layer 3：工具与流程（How）

```
你有什么工具？按什么流程工作？

【工具说明】
你可以使用以下工具：
- tool_name(参数): 功能说明，何时使用
  例：search_doc(query): 搜索文档，当需要查参考资料时

【工作流程】
1. [理解用户需求]
2. [判断是否需要工具]
3. [执行/调用工具]
4. [整合信息生成回答]
5. [自检质量]

例：
1. 理解用户的代码问题
2. 如需参考文档，调用search_doc
3. 分析代码，找出问题
4. 给出优化建议（附理由）
5. 确认建议可操作
```

### Layer 4：输出与安全（Format/Safety）

```
输出什么格式？有什么安全红线？

【输出格式】
- 结构：[如JSON/Markdown/分段]
- 风格：[如简洁/详细]
- 引用：[如标注来源]

【安全规则】
- 不输出[敏感内容类型]
- 涉及[高风险操作]时需[确认]
- 检测到[注入攻击]时[拒绝]
- 不确定时[诚实告知而非编造]

例：
输出格式：
  ## 问题分析
  [问题描述]
  ## 优化建议
  [具体建议+代码示例]
  ## 注意事项
  [潜在风险]

安全：
- 不讨论违法内容
- 代码建议标注"请测试后使用"
- 不确定时说"这需要进一步验证"
```

## 三、完整 Agent Prompt 示例

```markdown
# 技术咨询Agent Prompt

## 角色定义
你是资深全栈技术顾问，擅长架构设计、代码审查、性能优化。
风格：专业、务实、给出可落地建议。

## 能力边界
✓ 可以：分析技术方案、review代码、解释原理、给优化建议
✗ 不能：写完整商业项目代码、保证性能数值、讨论非技术话题
遇到非技术问题：礼貌引导回技术话题

## 可用工具
- search_doc(query): 搜索技术文档，当需要查官方文档/API时
- run_code(code): 运行代码验证，当需要验证语法/逻辑时

## 工作流程
1. 理解用户的技术问题
2. 判断是否需要查文档/运行代码
3. 分析问题根因
4. 给出方案（附理由+代码示例）
5. 标注注意事项和风险

## 输出格式
## 问题分析
[一句话总结问题]
## 解决方案
[方案+为什么这么做]
```代码示例```
## 注意事项
[潜在坑点]

## 安全规则
- 不输出可能有害的代码（如攻击脚本）
- 性能建议标注"需实际测试"
- 不确定时诚实说明
```

## 四、Prompt 设计原则

```
1. 明确优于模糊
   ❌ "帮我分析一下"
   ✅ "分析代码的时间复杂度，指出O(n²)的循环"

2. 结构化优于段落
   ❌ 一大段文字
   ✅ 分层标题+列表

3. 示例优于描述
   ❌ "输出要规范"
   ✅ 给一个标准输出的示例

4. 边界优于开放
   ❌ "尽力回答"
   ✅ "只回答X类问题，其他说'超出范围'"

5. 流程优于随意
   ❌ "看着办"
   ✅ "1.理解 2.分析 3.建议 4.提醒"
```

## 五、面试加分点

1. **四层结构**：角色/能力边界/工具流程/输出安全——体系化
2. **强调"边界"**：Agent Prompt 最重要的是定义"不能做什么"，而非"能做什么"
3. **类比"岗位说明书"**：Agent Prompt 是给 Agent 的 job description，不是一次性指令

## 记忆要点

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 提示词有"四层结构"（角色定义+能力边界+工具说明+流程规范+输出约束+安全规则），为什么比普通 Prompt（只任务指令）复杂这么多？**

因为 Agent 要"自主决策"，普通 Prompt 是"一次性任务"。1）自主决策——Agent 要自己判断"该用什么工具/何时调用/调用几次/何时停止"，没有清晰的角色和能力边界会失控（如乱调工具/超范围回答），四层结构约束 Agent 行为；2）工具使用——Agent 要正确调工具（选对工具/传对参数/处理返回），工具说明（每个工具的功能/参数/边界）让 Agent 懂工具，普通 Prompt 不涉及工具；3）流程规范——Agent 的多步流程（如"理解→检索→判断→回答"）要规范（按流程执行而非乱序），流程规范指导；4）输出约束——Agent 输出要结构化（如 JSON/含引用），约束格式便于下游处理；5）安全——Agent 自主性带来风险（如调危险工具/输出违规），安全规则防护。普通 Prompt 是"给定任务让 LLM 一次回答"，无自主决策/工具/多步/安全需求，简单即可。

### 第二层：证据与定位

**Q：Agent 行为不对（如乱调工具/超范围回答/不按流程），怎么定位是 Prompt 哪层写错（角色/工具/流程/输出/安全）？**

对应症状定位。1）乱调工具（选错工具/不该调时调）——工具说明层问题（工具描述不清/边界没写，Agent 不懂何时用哪个），优化工具描述；2）超范围回答（答了不该答的，如客服 Agent 聊闲天）——角色定义/能力边界问题（角色不清/边界没限制），明确角色和"不干什么"；3）不按流程（跳步/顺序乱）——流程规范问题（流程没写清/顺序不明），显式写流程步骤；4）输出格式错（非结构化/缺字段）——输出约束问题（格式约束弱/示例缺），加格式约束+示例；5）违规输出（涉黄/越权）——安全规则问题（规则不全/不严），加安全规则。定位方法：看症状→对应层→优化该层。常见根因：工具描述模糊（Agent 误选）、角色边界不清（超范围）、流程不显式（乱序）。

### 第三层：根因深挖

**Q：Agent Prompt 的"工具说明"要让 Agent 懂工具（何时用/怎么用），但工具一多（几十个），Prompt 会很长（费 token+attention 稀释），怎么管理？**

分层+按需加载。1）工具分类——把工具按类别组织（如"查询类""操作类""通知类"），Prompt 只列类别概述（"有查询/操作/通知三类工具"），Agent 按需深入；2）按需加载——Agent 根据当前任务判断需要哪类工具，动态加载该类工具的详细说明（如"任务要查数据→加载查询类工具详情"），而非全加载；3）工具检索——用向量检索/分类器，根据 query 选 top-K 相关工具（如"用户问数据→检索出查询工具"），只把相关的给 Agent（而非全几十个）；4）工具描述精简——每个工具描述精简（功能+触发+参数+边界，控制在几十字），减少单工具 token；5）MCP 生态——用 MCP 等标准化协议（工具自描述，Agent 按需发现），动态加载。原则：不全加载（按需/检索），精简描述，类组织，控 Prompt 长度。

**Q：Agent Prompt 的"流程规范"要指导 Agent 按流程执行，但流程太死（每步写死）会僵化（应对不了意外），怎么平衡规范和灵活？**

规范框架+灵活判断。1）规范框架——定义流程的大框架（如"理解意图→判断是否需要工具→（需要）调用工具→基于结果回答"），Agent 按大框架执行（保证基本流程）；2）灵活判断——框架内的具体决策留给 Agent（如"判断是否需要工具""选哪个工具""调用几次"由 Agent 按情况判断），而非每步写死（应对意外）；3）边界条件——定义边界（如"最多调用 5 次""不确定时反问用户"），防止僵化执行导致死循环/错误；4）示例——给几个典型流程示例（如"简单 query 流程""复杂 query 流程"），让 Agent 学习按情况选流程；5）迭代——观察 Agent 执行（哪些情况流程不适），优化流程规范（增加灵活性）。原则：大框架规范（保基本流程）+细节灵活（应对意外）+边界防护（防失控），而非全写死或全放开。

### 第四层：方案权衡

**Q：Agent Prompt 要写这么多层（角色+工具+流程+输出+安全），Prompt 会很长（几千 token），费成本且 LLM 可能忽略部分（attention 稀释），怎么平衡完整性和简洁？**

精简+分层+强调。1）精简——每层只写必要的（如角色一句话而非一段，工具描述几十字而非几百字，流程列要点而非详述），去冗余保核心；2）分层+位置——重要信息（角色/核心约束）放 Prompt 开头/结尾（attention 强，避免 lost-in-the-middle），工具/示例等放中间；3）强调——关键约束用强调（如"【必须】基于工具结果回答""【禁止】调用未列出工具"），醒目让 LLM 重视；4）动态——按任务动态调整 Prompt（如简单任务精简 Prompt、复杂任务详细 Prompt），而非总是全量；5）结构化——用清晰结构（如 markdown 标题/列表），让 LLM 易解析定位（而非大段文本）。原则：精简（去冗余）+分层（位置/强调）+动态（按任务）+结构化（易解析），保证完整且 LLM 能用，而非长但被忽略。

**Q：Agent Prompt 和普通 Prompt 的"四层结构"差异，意味着 Agent Prompt 更难写，团队怎么保证 Prompt 质量（而非每人乱写）？**

规范化+评审+迭代。1）Prompt 模板——提供四层结构的模板（角色/能力/工具/流程/输出/安全各层的填写指南+示例），开发者按模板写，标准化；2）Prompt 评审——Prompt 像 code review 一样评审（角色清不清/工具描述准不准/流程合理吗/约束全吗），团队把关；3）最佳实践——文档化各场景的 Prompt 最佳实践（如"客服 Agent 的角色怎么写""编程 Agent 的工具怎么描述"），新人参考；4）测试——Prompt 上线前过测试集（典型 case 验证 Agent 行为对），退化阻断；5）迭代——线上 bad case 反馈→优化 Prompt→验证，持续改进；6）Prompt 库——沉淀验证有效的 Prompt（带场景/效果/维护者），复用。这套写入团队 Prompt 开发 SOP，让"写 Agent Prompt"从"每人乱写"变成"模板+评审+测试+迭代"，标准化产出高质量 Prompt。

### 第五层：验证与沉淀

**Q：你怎么衡量 Agent Prompt 的质量（四层都写好 vs 随便写，效果差异）？**

AB 对比。固定测试集，对比"四层结构 Prompt"vs"简单 Prompt"：1）行为正确率——四层应高（角色清不超范围/工具对不乱调/流程对不乱序），简单 Prompt 行为易失控；2）输出质量——四层的输出结构化/符合约束（格式对/引用全），简单 Prompt 输出乱；3）安全性——四层违规率低（安全规则约束），简单 Prompt 易违规；4）鲁棒性——四层对边界情况（如异常输入/工具失败）处理更好（流程/安全规则兜底），简单 Prompt 易崩。综合：行为对+输出好+安全+鲁棒 = 四层有效。还要看 Prompt 长度成本（四层长，token 贵，但效果好值不值），如果效果提升大值成本，提升小则简化 Prompt。

**Q：Agent Prompt 的四层结构怎么沉淀成团队的 Prompt 工程能力？**

建 Prompt 工程 SOP：1）结构规范——文档化四层结构（每层的写法/要点/示例），标准；2）模板库——按 Agent 类型（客服/编程/分析/创作）提供四层模板，脚手架；3）评审流程——Prompt 评审 checklist（角色/工具/流程/输出/安全各层是否合格），团队把关；4）测试集——按 Agent 类型的测试集（典型 case+边界），Prompt 验证；5）最佳实践——真实 Prompt 案例（如"某客服 Agent 的四层 Prompt 及效果"），经验复用；6）迭代机制——线上反馈→优化 Prompt→验证→上线，持续改进。这套写入团队 Prompt 工程 SOP，让"写 Agent Prompt"从"经验驱动"变成"规范+模板+评审+测试+迭代"的工程化能力，标准化高质量产出。
