# 多 Agent 协作有哪些模式？（串行、并行、仲裁三种）

## 一、三种协作模式总览

```
┌──────────────────────────────────────────────────┐
│                  三种协作模式                       │
├──────────────────────────────────────────────────┤
│                                                    │
│  1. 串行 (Sequential / Pipeline)                   │
│     A → B → C → 输出                               │
│     前一个的输出 = 后一个的输入                      │
│                                                    │
│  2. 并行 (Parallel / Fan-out)                      │
│        A                                           │
│       / \                                          │
│      B   C  → 聚合 → 输出                          │
│     同时执行，最后汇总                              │
│                                                    │
│  3. 仲裁 (Debate / Arbitration)                    │
│     A的方案 }                                      │
│     B的方案 } → 仲裁者 → 最终方案                  │
│     C的方案 }                                      │
│     多方提案，择优/融合                             │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、模式 1：串行（Pipeline 流水线）

```
工作流：
  Agent A (调研) → Agent B (写作) → Agent C (校对) → 输出

特点：
  + 简单清晰，易实现
  + 顺序明确，易调试
  - 无法并行，速度慢
  - 一环出错全链路阻塞

适用：任务有明确阶段顺序
```

```python
class Pipeline:
    def __init__(self, agents):
        self.agents = agents  # [agent_a, agent_b, agent_c]
    
    def run(self, task):
        data = task
        for agent in self.agents:
            data = agent.process(data)  # 前一个输出给下一个
            if data is None:
                return "流水线在{agent}中断"
        return data

# 示例：内容生产流水线
pipeline = Pipeline([
    ResearchAgent(),   # 调研 → 素材
    WriterAgent(),     # 写作 → 草稿
    EditorAgent(),     # 校对 → 终稿
])
final = pipeline.run("写一篇Agent技术博客")
```

## 三、模式 2：并行（Fan-out 分治）

```
工作流：
              ┌─ Agent A (查新闻)
  主管分发 ───┼─ Agent B (查论文)  ──→ 主管聚合 → 输出
              └─ Agent C (查社区)

特点：
  + 速度快（并发执行）
  + 信息覆盖广
  - 需要聚合逻辑
  - 子任务必须相互独立

适用：子任务相互独立，可同时进行
```

```python
import asyncio

class ParallelCoordinator:
    def __init__(self, workers):
        self.workers = workers
    
    async def run(self, task):
        # 分解子任务
        subtasks = self.decompose(task)
        
        # 并发执行
        results = await asyncio.gather(*[
            worker.run(sub) for worker, sub in 
            zip(self.workers, subtasks)
        ])
        
        # 聚合结果
        return self.aggregate(results)

# 示例：市场调研
coordinator = ParallelCoordinator([
    NewsResearcher(),    # 查新闻
    PaperResearcher(),   # 查论文
    ForumResearcher(),   # 查社区
])
report = await coordinator.run("调研2026 Agent趋势")
```

## 四、模式 3：仲裁（Debate 辩论）

```
工作流：
  Agent A 提案 ─┐
  Agent B 提案 ─┼─→ 仲裁者(Aggregator) → 最终方案
  Agent C 提案 ─┘

特点：
  + 多视角，减少单点偏差
  + 可靠性高（容错）
  - 成本高（多次生成）
  - 需要好的仲裁机制

适用：高风险决策、需要多视角验证
```

```python
class DebateSystem:
    def __init__(self, proposers, arbitrator):
        self.proposers = proposers  # 多个提案Agent
        self.arbitrator = arbitrator  # 仲裁Agent
    
    def run(self, question):
        # 1. 各Agent独立给方案
        proposals = [p.propose(question) for p in self.proposers]
        
        # 2. （可选）多轮辩论
        for round in range(DEBATE_ROUNDS):
            proposals = [
                p.revise(question, proposals, own=p) 
                for p in self.proposers
            ]
        
        # 3. 仲裁者综合
        return self.arbitrator.arbitrate(question, proposals)

# 示例：医疗诊断
system = DebateSystem(
    proposers=[Cardiologist(), Neurologist(), Generalist()],
    arbitrator=ChiefDoctor()
)
diagnosis = system.run("患者症状：胸痛+头晕")
```

### 仲裁策略

```python
def arbitrate(question, proposals):
    # 策略1：投票（多数决）
    if is_objective(question):
        return most_common(proposals)
    
    # 策略2：LLM综合（取各方优点）
    return llm.synthesize(question, proposals)
    
    # 策略3：置信度加权
    weighted = [(p, p.confidence) for p in proposals]
    return max(weighted, key=lambda x: x[1])[0]
```

## 五、三种模式对比

| 维度 | 串行 | 并行 | 仲裁 |
|------|------|------|------|
| **速度** | 慢（串行） | 快（并发） | 中（并发+汇总） |
| **成本** | 低 | 中 | 高 |
| **可靠性** | 中 | 中 | 高（容错） |
| **适用** | 有顺序依赖 | 独立子任务 | 需多视角/高风险 |
| **典型** | 内容生产流水线 | 多源调研 | 医疗诊断/投资决策 |

## 六、混合模式（生产实战）

```
真实场景：开发一个软件功能

阶段1：并行调研（并行）
  ├─ Agent: 调研需求
  ├─ Agent: 调研技术方案
  └─ Agent: 调研竞品
       ↓ 汇总

阶段2：串行实现（串行）
  Architect → Developer → Tester
       ↓

阶段3：仲裁审核（仲裁）
  ├─ 代码审查Agent A
  ├─ 代码审查Agent B
  └─ 安全审查Agent C
       ↓ 仲裁
  最终是否合并
```

```python
async def hybrid_workflow(feature_request):
    # 阶段1：并行调研
    research = await parallel_coordinator.run(feature_request)
    
    # 阶段2：串行实现
    design = architect.run(research)
    code = developer.run(design)
    test_result = tester.run(code)
    
    # 阶段3：仲裁审核
    if test_result.passed:
        review = debate_system.run({"code": code, "action": "merge"})
        return review.approved
    return False
```

## 七、面试加分点

1. **三种模式讲清差异**：串行重顺序、并行重速度、仲裁重可靠，各有适用
2. **强调"实际混合"**：生产中很少单用一种，常是"并行调研→串行执行→仲裁审核"
3. **提成本意识**：仲裁模式最贵，不能滥用；简单任务串行即可

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：串行、并行、仲裁三种多 Agent 协作模式，为什么按"子任务间的依赖关系"选，而不按"任务复杂度"选？**

因为依赖关系决定了"能不能并行"。子任务间无依赖（如"查 A 公司资料"和"查 B 公司资料"独立）可以并行，加速明显；子任务间有强依赖（如"先编码才能审核"）必须串行。复杂度本身不决定模式——一个复杂任务如果子任务全独立，仍可并行（多 Agent 同时干各部分再汇总）；一个简单任务如果有依赖（如"先查再算"），仍要串行。所以模式选型看"子任务的依赖图"：DAG 里无依赖边的是并行机会，有依赖边的是串行约束，有冲突/分歧的需仲裁。仲裁模式用于"多个 Agent 给出不同结论需要裁决"（如多个医生 Agent 诊断不一致，要仲裁 Agent 综合判断）。

### 第二层：证据与定位

**Q：并行模式下，两个 Agent 同时修改同一份文档导致冲突，你怎么发现和解决？**

并行写冲突是经典分布式问题。发现：1）锁机制——文档有版本锁，第二个 Agent 写时检测到版本号变了，触发冲突告警；2）diff 检测——并行结束后对比两份修改，自动检测冲突区域（同一行被改了不同内容）。解决：1）预防——设计时避免"多 Agent 写同一资源"，拆成不同 Agent 写不同部分（如 CoderA 写前端，CoderB 写后端，不冲突）；2）仲裁——冲突时交给仲裁 Agent 决定保留哪个（或合并）；3）串行化——对必须共写的资源，把并行改成串行（A 写完 B 再写），牺牲速度换一致性。最优是"设计时避免共享写"（无冲突），迫不得已才用锁/仲裁。

### 第三层：根因深挖

**Q：仲裁模式下，多个 Agent 给出分歧结论，仲裁 Agent 自己也是 LLM，怎么保证它裁决得对？**

仲裁 Agent 的可靠性靠"信息充分+多源交叉"。1）信息充分——仲裁 Agent 不只看各 Agent 的结论，还看它们的"推理过程"和"证据"（如 A 医生诊断 X 并附依据，B 诊断 Y 并附依据），仲裁 Agent 基于完整信息裁决比基于结论裁决准；2）置信度加权——各 Agent 附置信度（A 90% 确信是 X，B 60% 确信是 Y），仲裁优先采高置信度的，除非低置信度方有更强证据；3）规则兜底——关键决策（如医疗）仲裁后过规则校验（如"诊断 X 和 Y 都不满足某必须条件则转人工"），LLM 仲裁+规则兜底。如果仲裁 Agent 仍不可靠，升级到"人工仲裁"（最关键场景如医疗诊断，分歧时必须人工医生介入）。

**Q：三种模式里，串行最简单但最慢，并行最快但最复杂，实际项目多数用哪个？**

实际多数用"串行为主 + 局部并行"。原因：1）多数任务的子任务有依赖（如"分析→方案→执行"），天然串行；2）并行要求子任务独立，设计时要识别哪些能并行（需要架构能力），增加复杂度；3）并行的协调成本（同步、合并、冲突解决）可能抵消速度收益。所以实务是：默认串行（简单可靠），识别到"独立子任务"时局部并行（如串行的第 3 步可以拆成 3 个独立子任务并行），不全盘并行。这样兼顾简单和效率。纯并行适合"embarrassingly parallel"任务（如批量处理 100 个独立文档），这种场景并行收益最大。

### 第四层：方案权衡

**Q：仲裁模式比串行/并行都复杂（要额外仲裁 Agent），什么场景值得用？**

值得用的场景是"决策有分歧 + 错误代价高 + 单 Agent 不可靠"。1）医疗诊断——单 Agent 诊断可能有 20% 误诊率，多专科 Agent + 仲裁能把误诊率降到 10% 以下（交叉验证），误诊代价（错诊导致医疗事故）远超仲裁成本；2）高风险代码合并——多个 Agent 对同一 PR 的审核可能分歧（一个说 OK 一个说有 bug），仲裁综合判断避免漏审导致线上事故；3）投资决策——多分析 Agent 给出冲突建议（买/卖），仲裁综合风险偏好和证据做最终建议。不值得用的场景：低风险任务（如生成文案，分歧不大且代价低）、单 Agent 足够准的任务（仲裁是多余开销）。判别：错误代价 × 分歧概率 > 仲裁成本，才值得用仲裁。

**Q：为什么不所有多 Agent 系统都用仲裁（最可靠），而要分三种模式？**

因为仲裁的成本和延迟最高，对多数任务不划算。仲裁要等所有 Agent 给结论（最慢的那个决定延迟）、要额外调仲裁 Agent（成本）、要处理分歧（复杂度）。一个简单串行任务（如"查→算→答"）用仲裁是杀鸡用牛刀——没有分歧可仲裁（每步只有一个 Agent 做），纯属浪费。所以三种模式各有最优场景：串行适合"有依赖的流水线"，并行适合"独立子任务加速"，仲裁适合"高风险需要交叉验证"。选型看"任务结构（依赖）+ 风险等级（错误代价）+ 性能要求（延迟预算）"，不是一律用最可靠的（那是过度设计）。

### 第五层：验证与沉淀

**Q：你怎么证明多 Agent 协作模式选对了，而不是"该并行却串行导致慢"或"该串行却并行导致乱"？**

模式选型验证看三个指标。1）延迟——并行模式应显著快于串行（如 3 个独立子任务，串行 9s，并行 3s）；如果并行没快（如同步等待时间长），说明并行设计有问题；2）一致性——串行/并行模式下结果应一致（同一任务多次运行方差小）；如果并行模式方差大（有时对有时错），说明协调/合并逻辑有 bug；3）完成率——选的模式应不降低完成率（如为了并行而拆得太细导致子任务质量下降）。AB 测试：同任务分别跑串行/并行/仲裁，对比延迟、一致性、完成率，选"满足质量前提下最快"的模式。如果并行快但完成率掉 10%，说明拆分破坏了任务连贯性，要减少并行粒度或回退串行。

**Q：三种协作模式怎么沉淀成框架的标准能力，开发者按需选？**

封装成 CollaborationPatterns 组件：1）串行模式——Pipeline 执行器，Agent 按序执行，前者输出喂后者；2）并行模式——Fan-out/Fan-in 执行器，多个 Agent 并行，结果合并（提供合并策略：拼接/投票/择优）；3）仲裁模式——Debate 执行器，多 Agent 各出结论+证据，仲裁 Agent 综合裁决（支持置信度加权+规则兜底）。开发者通过配置（`mode: parallel`）选择，框架自动调度。配套：冲突检测器（并行写冲突告警）、死循环防护（仲裁反复分歧触发终止）、多 Agent trace（链路追踪定位问题）。这套写入团队 Agent 框架 SOP，开发者像选"并发模式"一样选协作模式，不重写调度逻辑。

## 结构化回答

**30 秒电梯演讲：** 多Agent协作三种模式——串行(流水线)、并行(分治)、仲裁(辩论)。串行求顺序，并行求速度，仲裁求可靠——像餐厅。

**展开框架：**
1. **串行** — 流水线，前输出后输入，适合有依赖
2. **并行** — 分治，子任务并发，适合独立任务
3. **仲裁** — 多Agent各出方案，投票/汇总，求可靠

**收尾：** 您想深入聊：三种模式能混用吗？——能，先并行调研再串行写作最后仲裁审核？
