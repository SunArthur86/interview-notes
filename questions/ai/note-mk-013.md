# 【月之暗面面经】如何设计 AI 桌面端的产物 diff，既能给新手看，也能给高手看？

## 一、三级Diff分层模型

```
┌──────────────────────────────────────────────────────────────────┐
│                    Diff 信息分层金字塔                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│           Level 3: 原始Diff（给开发者/极客）                      │
│          ┌───────────────────────┐                               │
│          │ 字符级/行级原始对比     │                               │
│         └───────────────────────┘                                │
│       Level 2: 结构化Diff（给有经验用户）                          │
```

## 六、实现：Diff渲染器

```typescript
class DiffRenderer {
  render(artifact: Artifact, diff: ArtifactDiff, level: DiffLevel) {
    switch (level) {
      case 'summary':
        return this.renderSummary(diff);
      case 'structured':
        return this.renderStructured(artifact, diff);
      case 'raw':
        return this.renderRaw(diff);
    }
  }
  
  // 摘要diff
  private renderSummary(diff: ArtifactDiff): SummaryView {
    return {
      totalChanges: diff.changes.length,
      categories: this.categorizeChanges(diff.changes),
      // { modified: 2, added: 1, deleted: 0 }
      changes: diff.changes.map(c => ({
        type: c.type,
        description: this.humanizeChange(c),
        impact: c.impact,
      })),
    };
  }
  
  // 根据用户专业度自动选择默认diff层级
  recommendLevel(userProfile: UserProfile): DiffLevel {
    if (userProfile.totalTasks < 10) return 'summary';
    if (userProfile.totalTasks < 50) return 'structured';
    return 'raw';
  }
}
```

## 七、常见坑

- **只有代码级diff**：普通用户看不懂 `@@ -1,5 +1,12 @@`，直接拒绝所有修改
- **不支持局部接受**：全盘接受或全盘拒绝，用户无法精细控制
- **大产物用文本diff**：站点/PPT这种结构化产物用文本diff，信息量太大
- **没有diff层级切换**：不能根据需要切换详细程度，要么太简略要么太详细

## 记忆要点

- 金字塔三级分层：Level 1摘要级（新手）、Level 2结构化（经验者）、Level 3原始对比（极客）。
- 新手看摘要：屏蔽代码细节，提炼一句话总结（如新增表格、修改标题）降低认知门槛。
- 高手看细节：提供字符级或行级纯文本对比，支持快速定位特定代码改动。
- 灵活UI设计：采用渐进式展示，默认只给摘要，提供下拉或开关让用户自由切换深度。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么 Diff 要做三级分层（摘要/结构化/原始），而不是统一用一个"代码级 Diff"或"摘要 Diff"给所有人？**

因为不同用户的理解力和需求完全不同，统一层次的 Diff 要么"对新手太复杂"要么"对高手太简略"。代码级 Diff（`@@ -1,5 +1,12 @@`）对开发者是标准语言，但对非技术用户（产品经理、运营）是天书——他们看到 `-# 季度报告` 和 `+# 2024 Q4 竞品分析报告` 完全不知道这是什么操作，只能盲目点"拒绝"。摘要 Diff（"改了标题"）对新手足够，但对需要精确审查的高级用户（如要确认"30% 是否改成了 45%"）信息不足——他们要看到具体改了哪些字符。三级分层的本质是"按用户专业度匹配信息抽象层次"：摘要层屏蔽代码细节提炼语义（"新增了竞品对比表格"），结构化层按区块展示语义化变更（表格可视化对比），原始层提供字符级精确对比（开发者审查代码改动）。用户从摘要层开始，需要更多细节时逐层展开，认知负担可控。

### 第二层：证据与定位

**Q：你怎么定位"用户看不懂 Diff 所以拒绝了所有修改"是 Diff 设计问题，而不是修改本身就有问题？**

用 Diff 层级切换率和局部接受率定位。在 Diff 视图埋：(1) 层级切换事件——用户从摘要层切到结构化层或原始层的次数，如果用户切到更详细层后改变了决策（从"全部拒绝"变成"部分接受"），说明摘要层信息不足导致误拒，根因是 Diff 设计（摘要不够清晰）；(2) 局部接受率——用户选择"逐条确认"后勾选了多少变更，如果平均勾选率超过 70%，说明用户认可大部分修改只是某些条目有疑问——拒绝整单是因为无法精细控制，根因是 Diff 不支持局部操作；(3) 修改质量评分——如果同一段修改被大量用户拒绝，但切到原始 Diff 后接受率提升，说明是摘要描述不准确（没说清楚改了什么）；如果切到原始 Diff 后仍被拒绝，说明修改本身有问题（AI 改错了）。这三种情况分别对应 Diff 层级不足、局部操作缺失、摘要质量差，解法各不同。

### 第三层：根因深挖

**Q：为什么大产物（站点/PPT）的 Diff 不能用文本行级对比，必须做"页面级 Diff"？根因是什么？**

根因是"文本行级 Diff 会把语义变更淹没在格式噪音里"。一个 PPT 的 JSON 表示有几千行，其中 90% 是格式定义（颜色、字体、位置坐标），10% 是语义内容（标题文字、数据值）。AI 修改了第 3 页的"增长率从 30% 改成 45%"，在 JSON 层面可能影响几十行（颜色值变了、图表数据数组变了、标签字符串变了），文本 Diff 会展示几十行的增删改，用户完全看不出"核心变更只是 30%→45%"。页面级 Diff 的本质是"按产物的语义单元（页面/幻灯片）组织对比"——只在"第 3 页"标记"有修改"，用户点击后看到的是渲染后的 PPT 第 3 页视觉对比（左侧原图、右侧新图），而非 JSON 行级对比。这样用户一眼就能看到"数据从 30% 变成了 45%"，语义清晰。文本行级 Diff 只适用于"纯文本产物"（如 Markdown 文档），结构化产物（站点/PPT/表格）必须按其语义单元做 Diff。

**Q：那如果用户是开发者，想看 PPT 修改的 JSON 原始变更（如确认某个动画参数改了），为什么不保留行级 Diff 给开发者用？**

因为"开发者想看 JSON 原始变更"不等于"对所有变更都用行级 Diff"，而是"在页面级 Diff 的基础上，对特定页面可以下钻到原始层"。三级分层已经预留了这个场景：开发者先用页面级 Diff（Level 2 结构化层）定位到"第 3 页有修改"，然后点击"查看原始 Diff"下钻到 Level 3，看第 3 页的 JSON 变更。这样开发者既能用语义层快速定位（不用翻几千行 JSON 找哪页改了），又能在定位后下钻看原始细节。直接对所有页面用行级 Diff 的问题：几千行 JSON 的变更里 90% 是格式噪音，开发者在噪音里找语义变更（增长率改了）效率极低。正确的做法是"页面级 Diff 做导航 + 原始 Diff 做细节"，而非"全部用原始 Diff"。这和 Git 的理念一致——Git 的 PR 视图默认按文件组织（文件级导航），点击文件后才展开行级 Diff，而非把所有文件的所有行平铺展示。

### 第四层：方案权衡

**Q：默认展示哪个 Diff 层级你用"按用户专业度自动推荐"（总任务数 <10 用摘要），为什么不用"统一默认摘要"让所有人手动切换？**

因为"统一默认摘要"会让高级用户每次都要手动切换，产生"为什么要多点一次"的挫败感。自动推荐的价值是"默认层级匹配用户习惯"——新手打开 Diff 看到摘要层（符合预期），高级用户打开看到结构化层或原始层（符合预期），都不需要手动切换。按总任务数（<10 新手 / <50 经验者 / >=50 高手）做分层是一个简单但有效的启发式：任务数少说明用户刚上手，大概率不熟悉代码级 Diff；任务数多说明用户是重度用户，大概率需要精确审查。自动推荐的实现成本极低（读用户画像字段即可），但省下了所有高级用户"每次切换"的累积成本。代价是推荐可能不准（如一个任务数 5 的开发者想看原始 Diff），但这通过"一键切换层级 + 记住用户上次选择"来缓解——用户切换一次后，后续默认用他上次的层级。

**Q：那如果用户每次都在摘要层和原始层之间来回切换（说明自动推荐不准），为什么不放弃自动推荐改成"每次都问用户想看哪层"？**

因为"每次都问"的交互成本远高于"推荐不准偶尔切换"。每次问用户意味着每次打开 Diff 都要弹一个选择框（"你想看摘要/结构化/原始？"），用户要点一次才能看到内容——这是典型的"把决策成本推给用户"。自动推荐偶尔不准时用户只需点一次"切换到原始层"，且切换后系统可以更新推荐（如检测到用户连续 3 次从摘要切到原始，把默认层级调成原始）。更优的做法是"学习用户偏好"——记录用户最近 5 次 Diff 查看的层级选择，用众数作为下次默认，这样推荐会逐渐收敛到用户的真实偏好。如果用户在摘要和原始之间频繁切换，说明他对不同任务有不同需求（简单任务看摘要、复杂任务看原始），这时可以按"任务类型"推荐而非全局推荐（如代码生成任务默认原始层、文档生成任务默认摘要层）。所以根因是"推荐策略要持续学习用户偏好"而非"放弃推荐改每次问"。

### 第五层：验证与沉淀

**Q：你怎么证明"三级 Diff 分层"比"单一 Diff 层级"更能帮用户做出正确的接受/拒绝决策？**

用决策质量指标验证：(1) 误拒率 = 用户拒绝了但实际上是正确修改的比例（通过后续用户手动改回正确值来判断）——三级分层（摘要让用户理解变更语义）应降低误拒率；(2) 误纳率 = 用户接受了但实际上是错误修改的比例（通过后续产物质量评分或回退操作判断）——结构化层让用户精确审查应降低误纳率；(3) 决策耗时 = 用户从打开 Diff 到做出接受/拒绝决策的平均耗时——如果三级分层的摘要层让新手快速理解，决策耗时应短于单一原始 Diff（新手看不懂要反复研究）。A/B 测试时对照组用单一原始 Diff，实验组用三级分层，跑两周后对比——如果实验组的"误拒率 + 误纳率"综合低于对照组 30% 以上，且决策耗时不超过对照组，就证明分层设计有效。

**Q：怎么让团队在新增产物类型时，自觉实现三级 Diff 渲染，而不是只做一个层级的 Diff 就交付？**

把三级 Diff 做成"产物类型注册表"的必填插槽。第一，ArtifactTypeSpec 接口里 differ 字段必须实现三个方法：renderSummary()、renderStructured()、renderRaw()，缺一个不允许注册——这从类型系统层强制每种产物都有三级 Diff；(2) 提供 Diff 基础设施——通用的文本 Diff 算法（行级对比）、JSON Diff 算法（结构化对比）、Markdown Diff 算法（按标题/段落对比）封装成工具函数，产物类型只需调用并定制渲染逻辑，不从零写 Diff 算法；(3) CI 做 Diff 完整性检查——注册新产物类型时自动运行一组测试变更，验证三级渲染器都能正确输出，缺层级 CI 失败；(4) 提供 Diff 渲染器模板——脚手架 CLI 生成新产物类型时自动产出三级渲染器的骨架代码，开发者填业务逻辑即可，降低实现成本。这样三级 Diff 就从"产品规范"变成了"注册接口的强制约束 + 工具链支持"。
