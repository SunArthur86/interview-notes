# 你对当下Agent能力怎么看待？

## Agent能力成熟度评估

```
能力维度          水平        评估
─────────────────────────────────────
任务拆解          ★★★★☆      显著进步
Tool调用          ★★★★☆      可靠性提升
知识检索(RAG)     ★★★★☆      成熟
长程推理          ★★☆☆☆      仍然不足
多步决策          ★★☆☆☆      稳定性差
自我纠错          ★★☆☆☆      能力有限
通用迁移          ★☆☆☆☆      基本做不到
```

## 已能稳定落地的场景

| 场景 | 代表产品 | 为什么能落地 |
|------|---------|------------|
| **客服Bot** | 各大平台 | 流程固定、容错高、有人工兜底 |
| **Research助手** | Perplexity/ChatGPT | 单轮检索+总结、不需要长链路 |
| **代码辅助** | Copilot/Cursor | 单文件级、有IDE反馈循环 |
| **数据分析** | NL2SQL+Chart | 输出结构化、可验证 |
| **文档处理** | 归档/分类/抽取 | 任务边界清晰、可批量校验 |

## 仍然困难的场景

```
困难场景的共同特征：

1. 长链路任务 → 错误率指数累积
   "帮我调研竞品并写一份50页报告"
   → 10步串行，每步90%→整体35%成功率

2. 需要精确推理 → 概率模型不擅长
   "分析这段代码的并发安全问题"
   → 需要确定性逻辑推理，LLM容易遗漏

3. 开放式探索 → 没有明确终止条件
   "帮我找到一个好的创业方向"
   → 评价标准模糊，无法自动判断完成

4. 跨系统协作 → 接口和状态同步复杂
   "同时操作CRM、ERP和邮件系统"
   → 每个系统API不同、状态不一致
```

## 核心瓶颈

```
┌─────────────────────────────────────────┐
│           Agent 三大瓶颈                │
├──────────────┬───────────┬──────────────┤
│   稳定性差    │  成本高    │  长链路易失败 │
├──────────────┼───────────┼──────────────┤
│ 同一输入不同  │ 每步调用   │ 10步串行      │
│ 输出不一致    │ LLM+Tool  │ 每步95%       │
│              │ 成本累积   │ 整体=60%      │
│ 无法保证SLA   │ 复杂任务   │ 中间任何一步   │
│              │ $1-5/次   │ 失败全盘失败  │
└──────────────┴───────────┴──────────────┘
```

## 面试回答要点

> "我觉得现在Agent能力进步很快，尤其任务拆解和Tool调用。

> **简单任务已经能稳定落地**，比如客服、Research、代码辅助。这些场景的共同特点是：流程短、容错高、有人工兜底。

> **但复杂任务问题还很多**，主要是稳定性差、成本高、长链路容易失败。一个10步的串行任务，即使每步95%成功率，整体也只有60%。

> 离真正通用Agent还有距离。现阶段更务实的做法是把Agent定位为特定场景的效率工具，而不是试图做万能助手。"

## 面试加分点

1. **辩证看待**：既肯定进步又指出局限，体现成熟的技术判断
2. **量化分析**：用错误率累积模型解释为什么长链路难
3. **场景化**：区分"能落地的"和"不能落地的"，并解释原因
4. **务实路线**：推荐"特定场景优先落地"而非"一步到位做通用Agent"

## 记忆要点

- 成熟度盘点：任务拆解与Tool调用成熟，长程推理与自我纠错依然薄弱
- 易落地场景：流程短、容错高、有人工兜底(如客服/代码辅助/数据简答)
- 长链路瓶颈：因为错误率指数级累积(10步x90%=35%)，所以多步决策极易全盘崩溃
- 务实落地观：现阶段做特定场景的效率工具，而非一步到位造通用Agent

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说现阶段做"特定场景效率工具"而非"通用 Agent"，判断依据是什么？凭什么说通用 Agent 还不行？**

依据是长链路错误率累积。单步 Agent 的成功率约 90%（工具调用、推理），但多步任务的错误率是乘法累积——10 步任务整体成功率 0.9^10=35%，20 步只有 12%。通用 Agent（如"完成任意开放任务"）往往需要 10-50 步，当前 LLM 的单步可靠性撑不住。特定场景（如客服/代码辅助）可以把任务限制在 3-5 步，配合人工兜底，整体可靠性可接受。动机是"在当前模型能力下，限制步数和场景来保证可靠性"，而非否认 Agent 的潜力。

### 第二层：证据与定位

**Q：你的 Agent 在 5 步以内任务成功率 92%，但 10 步任务掉到 60%。你怎么定位是模型推理弱、还是任务设计不合理？**

看每步的失败率和失败模式。如果每步失败率均匀（如每步都 90%），是正常的概率累积，说明模型推理能力是瓶颈；如果某步失败率畸高（如第 5 步只有 60%），是该步骤设计问题（如该步的工具不稳定、prompt 不清晰）。把 10 步任务拆成"2 个 5 步子任务"测，如果两个子任务各自成功率 90%，但串起来只有 60%，确认是累积效应，治本是减少步数（如用 Workflow 把 10 步压成 5 步）而非提升单步。

### 第三层：根因深挖

**Q：Agent 的"自我纠错"你说是薄弱项，根因是模型反思能力差，还是纠错机制设计不对？**

两者都有，但机制设计是主因。模型反思能力其实不弱（GPT-4 能识别自己的错误），问题是"纠错循环"会引入新错误——模型发现错误后修改，修改可能引入新错，再纠错再出错，振荡不收敛。根因是没有"纠错的验收标准"——模型不知道"改到什么程度算对"。治本是引入外部验收（如测试用例、规则校验、人工审核），纠错后必须过验收才算成功，否则超时兜底。纯靠模型自评的纠错是开环，加外部验收才闭环。

**Q：那为什么不直接用更强的模型（如 o1）做长程推理，它有内建的思维链纠错，省得搞外部机制？**

o1 的推理能力强（单次长 CoT），但仍有局限。一是成本——o1 的单次推理可能花 10k-50k token（思考链很长），10 步任务成本爆炸；二是 o1 的纠错是"单次推理内"的，跨工具调用的纠错（如调用失败后调整策略）仍需要外部 Agent 框架；三是 o1 也会错，在高可靠性场景仍需外部验收。o1 提升了单步推理质量（降低单步错误率），但长链路的累积问题仍在。正确做法是 o1 做关键步骤 + 外部框架做编排和验收，组合使用。

### 第四层：方案权衡

**Q：现阶段易落地的场景（客服/代码辅助/数据简答）共同特点是"流程短、容错高、有人工兜底"，为什么这三条是关键？**

因为它们恰好对冲了 Agent 的三大弱点。流程短（<5 步）对冲"错误率累积"——3 步 90% 成功率 73%，可接受；容错高（偶尔错可接受）对冲"LLM 随机性"——客服答错可以道歉重答，不像医疗/法律错一次是事故；有人工兜底对冲"Agent 能力边界"——搞不定的转人工，用户体验有保障。三条满足任意两条就有落地价值，三条全满足是"甜点场景"。反之，流程长、零容错、无兜底的场景（如全自动医疗诊断、自动驾驶决策）现阶段不该让 Agent 独立承担。

**Q：为什么不直接在容错低的场景（如医疗/法律）用 Agent + 强人工审核，非要等模型更强？**

人工审核的瓶颈和责任。医疗/法律场景即使 Agent 只做辅助，如果它给出看似合理的错误建议，可能误导专业人员（自动化偏见），且错误的责任归属复杂（是 Agent 的错还是医生的错）。更现实的是这些场景的"正确性标准"极高（99.9%+），Agent 当前达不到，强行用会增加审核负担（专业人员要逐条核查 Agent 输出，比自己从头做还累）。现阶段 Agent 在这些场景的价值是"信息检索和初筛"（如检索相关法条/文献），而非"直接给结论"，定位是工具而非决策者。

### 第五层：验证与沉淀

**Q：你怎么判断"某个新场景现在能不能上 Agent"，而不是拍脑袋？**

用三维度评估矩阵：流程长度（步数）、容错要求（错误代价）、兜底可行性（能否转人工）。打分：步数 <5 得 3 分、5-10 得 2 分、>10 得 1 分；容错（错可接受）3 分、中等 2 分、零容错 1 分；有人工兜底 3 分、部分兜底 2 分、无兜底 1 分。总分 >7 分适合 Agent，5-7 分需谨慎评估，<5 分暂不上。配合 PoC（概念验证）——在真实数据上跑 100 个 case 看实际成功率，成功率 >85% 且兜底可控才推进。

**Q：对 Agent 能力的务实认知怎么沉淀成团队的选型原则？**

固化成"Agent 适用性评估 checklist"：流程步数、容错度、兜底可行性、数据可得性、ROI（节省的人力成本 vs Agent 开发和运营成本）。沉淀"已验证可落地场景库"（客服/代码辅助/数据简答）和"暂不推荐场景库"（医疗诊断/法律判决/无人驾驶），附理由和 revisit 条件（如"模型能力提升到 X 再评估"）。避免团队对 Agent 能力"过度乐观"（什么都想 Agent 化）或"过度悲观"（什么都不敢用），用评估框架做客观决策。
