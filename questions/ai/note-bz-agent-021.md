# 医疗辅助诊断多智能体怎么落地？

## 一、医疗场景的特殊性

```
医疗诊断 vs 普通任务：
  ├─ 准确性要求极高：误诊可能致命
  ├─ 幻觉零容忍：不能编造药物/剂量
  ├─ 多维度：症状涉及多个专科
  ├─ 安全约束：药物相互作用/过敏/禁忌
  ├─ 合规严格：数据隐私/可追溯/责任界定
  └─ 需解释性：医生和患者都要能理解

→ 这些特性天然适合多Agent"会诊"模式
```

## 三、各 Agent 职责详解

### 1. Triage Agent（分诊）

```python
class TriageAgent:
    def triage(self, symptoms):
        """根据症状判断需要哪些专科会诊"""
        prompt = f"""
        患者症状: {symptoms}
        
        判断需要咨询哪些专科（可多选）：
        - 心内科：胸痛/心悸/高血压
        - 神经科：头痛/头晕/肢体麻木
        - 呼吸科：咳嗽/气喘
        - 消化科：腹痛/恶心
        - 全科：症状不明确时必选
        
        返回: [专科列表, 紧急程度]
        """
        return self.llm.triage(prompt)
```

### 2. 专科 Agent（会诊）

```python
class SpecialistAgent:
    def __init__(self, specialty):
        self.specialty = specialty  # "cardiology"
        self.knowledge_base = load_medical_kb(specialty)  # 专科知识库
    
    def diagnose(self, symptoms, patient_info):
        """基于专科知识库诊断（RAG防幻觉）"""
        # 关键：必须基于知识库，不能编造
        relevant = self.knowledge_base.retrieve(symptoms, top_k=5)
        
        prompt = f"""
        你是{self.specialty}医生。基于以下医学资料诊断：
        
        [检索到的医学资料]
        {relevant}
        
        [患者信息]
        症状: {symptoms}
        既往史: {patient_info.history}
        过敏: {patient_info.allergies}
        
        给出：
        1. 可能的诊断（附置信度）
        2. 建议的检查
        3. 初步治疗方案（必须基于资料，不可编造）
        4. 不确定的地方（诚实标注）
        """
        return self.llm.diagnose(prompt)
```

### 3. Review Agent（审核）

```python
class ReviewAgent:
    def review(self, diagnoses, patient_info):
        """综合多专科意见 + 安全检查"""
        # 1. 综合诊断
        combined = self.synthesize(diagnoses)
        
        # 2. 药物安全检查（关键！）
        for drug in combined.medications:
            # 药物相互作用
            interactions = self.check_interactions(
                drug, patient_info.current_meds)
            if interactions.dangerous:
                combined.add_warning(f"{drug}与当前用药有危险相互作用")
            
            # 过敏检查
            if drug in patient_info.allergies:
                combined.add_warning(f"患者对{drug}过敏！")
            
            # 禁忌症检查
            contraindications = self.check_contraindications(
                drug, patient_info.conditions)
        
        # 3. 置信度评估
        if combined.confidence < 0.7:
            combined.recommend_human = True  # 建议人工
        
        return combined
```

### 4. Safety Agent（安全兜底）

```python
class SafetyAgent:
    """高风险场景必须转人工"""
    RED_FLAGS = [
        "胸痛+呼吸困难",      # 可能心梗
        "剧烈头痛+呕吐",      # 可能脑出血
        "意识模糊",           # 危急
        "儿童高热>39.5",     # 高危
    ]
    
    def check(self, diagnosis):
        for flag in self.RED_FLAGS:
            if flag in diagnosis.symptoms:
                return {
                    "action": "URGENT_HUMAN",
                    "message": "检测到危急症状，请立即就医"
                }
        if diagnosis.uncertain:
            return {"action": "CONSULT_HUMAN", "message": "建议面诊医生"}
        return {"action": "PROCEED"}
```

## 四、防幻觉的关键设计

```
医疗Agent幻觉=灾难，必须多重防护：

第1层：RAG强约束
  - 所有诊断必须基于检索到的医学文献
  - Prompt明确："只能基于提供的资料，不可编造"

第2层：结构化输出
  - 药物/剂量必须从药品库精确匹配，不可生成
  - 用JSON Schema约束输出格式

第3层：交叉验证
  - 多专科Agent意见交叉比对
  - 矛盾处必须标注

第4层：安全审核
  - Review Agent专门查药物相互作用/禁忌
  - Safety Agent拦截高危场景

第5层：人工兜底
  - 低置信度/高危/不确定 → 必须转医生
  - Agent定位是"辅助"，不替代医生
```

## 七、面试加分点

1. **强调"辅助"定位**：医疗 Agent 是辅助决策，不是替代医生——这个边界很重要
2. **多层防幻觉**：RAG+结构化+交叉验证+安全审核+人工兜底，体现对医疗严肃性的理解
3. **会诊隐喻**：多 Agent = 多学科会诊，这个类比既贴切又能解释架构合理性

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：医疗辅助诊断用多智能体（分诊+专科+审核+沟通），为什么不直接用一个强 LLM 做诊断？多智能体不是更复杂吗？**

医疗的核心矛盾是"准确性 vs 安全性"，单 LLM 两头都难保证。1）准确性——单 LLM 的诊断准确率受限于训练数据，罕见病/并发症容易漏诊；多专科 Agent 各自专精（心内科 Agent 看心血管、神经科 Agent 看神经），覆盖更广、更准；2）安全性——单 LLM 输出没有交叉验证，错了也没人"反驳"；多 Agent 有审核 Agent（综合多专科意见）和规则校验（如"诊断与必查指标矛盾则告警"），安全性靠交叉验证+规则兜底。复杂性确实增加，但医疗是"错误代价极高"的场景（误诊可能致死），多 Agent 的交叉验证是必要的安全冗余，值得这个复杂度。

### 第二层：证据与定位

**Q：医疗多智能体诊断出错（误诊或漏诊），你怎么定位是哪个专科 Agent 或审核 Agent 的问题？**

按会诊流程 trace。1）分诊 Agent——看它分到哪些专科（如该看心内科却没分，是分诊漏诊）；2）专科 Agent——看每个专科的诊断是否合理（如心内科 Agent 诊断"心肌炎"但依据不足，是专科误诊）；3）审核 Agent——看它是否综合了所有专科意见（如心内科说心梗、消化科说胃炎，审核 Agent 没识别出"心梗放射到腹部"的典型表现，错误采信消化科，是审核综合能力弱）。每个 Agent 的诊断要附"依据"（症状+检查结果+医学指南引用），审核时能追溯。定位方法：把误诊 case 的完整 trace 给医学专家审核，标出"第一个出错的环节"，统计各环节错误占比，针对性优化。

### 第三层：根因深挖

**Q：医疗多智能体的"专科 Agent"怎么获得专科能力？是用通用 LLM + 专科 prompt，还是用专科数据微调？**

按专科知识深度决定。1）通用 LLM + 专科 prompt——适合"专科知识在通用模型训练数据里已有"的场景（如心内科常见病，LLM 训练时见过），prompt 加专科指南引用即可；2）专科数据微调——适合"专科知识是私有/前沿/罕见病"的场景（如某医院的专科病例数据、最新临床试验），这些通用模型没见过，要用专科数据 SFT 注入。实务组合：通用 LLM 做底座（保证基础医学推理能力）+ 专科 RAG（检索最新指南/文献，动态更新）+ 必要时专科 SFT（某专科的私有病例，提升罕见病准确率）。RAG 优于 SFT 的点：医学知识更新快（新指南/新药），RAG 实时更新不用重训。

**Q：医疗多智能体的"审核 Agent"如果是 LLM，它凭什么能比专科 Agent 更准地做最终判断？**

审核 Agent 不是"比专科更准"，是"综合多专科 + 规则校验"。1）综合多专科——专科 Agent 只看自己领域（如心内科只想到心源性胸痛），可能漏掉"胸痛也可能是消化道/心理原因"，审核 Agent 看所有专科意见，能发现"心内科和消化科都有部分证据"的复杂情况，提示进一步检查；2）规则校验——审核 Agent 配合医学规则引擎（如"胸痛+ST 投高=心梗，必须立即干预"），规则是确定性的，LLM 诊断后过规则，不符合则告警/转人工；3）置信度阈值——多专科一致时高置信直接出诊断，分歧时低置信转人工医生。所以审核 Agent 的价值是"全局视角 + 规则兜底 + 置信度路由"，不是"比专科更聪明的医生"。

### 第四层：方案权衡

**Q：医疗多智能体这么复杂，为什么不简化成"单 LLM + 医学知识库 RAG"，用 RAG 补充知识？**

RAG 补充知识但解决不了"交叉验证和安全性"。1）交叉验证缺失——单 LLM + RAG 仍是"一个模型一个判断"，没有多专科的多视角，罕见并发症（跨专科）容易漏；2）安全性冗余不足——单 LLM 的诊断没有"审核环节"，错了直接输出给医生/患者；3）规则校验——虽然可以加规则引擎，但没有"多专科讨论"的语义层交叉（如心内科和消化科意见的权衡），规则只能做硬约束。所以医疗场景"多智能体 + 规则 + RAG"的组合是最稳的：多智能体做多视角交叉验证，RAG 补充最新知识，规则做硬性安全校验。简单场景（如分诊导诊）可以单 LLM+RAG，复杂诊断要多智能体。

**Q：医疗多智能体的最终决策权归谁——审核 Agent 还是人类医生？Agent 能不能自主出诊断？**

取决于风险等级和法规。1）低风险场景（如分诊导诊、健康咨询）——Agent 可自主建议（如"建议看心内科"），因为这只是"导诊建议"不是"诊断"，错误代价低；2）中风险场景（如辅助诊断）——Agent 给"诊断建议+置信度"，但最终诊断必须人类医生确认（Agent 输出是"参考意见"不是"最终诊断"），法规要求医疗诊断责任在人；3）高风险场景（如治疗方案推荐、用药）——Agent 只能给"参考方案"，必须医生审核签字才能执行。所以 Agent 的自主权按风险分级：低风险自主、中风险建议、高风险只参考。这是医疗合规的要求，不是技术做不到（技术上 Agent 能给诊断），是法规不允许 Agent 替代医生做最终决策。

### 第五层：验证与沉淀

**Q：医疗多智能体的诊断准确率怎么验证？不能拿真实病人做实验，怎么获得 ground truth？**

用历史病例的回顾性验证 + 前瞻性影子测试。1）回顾性验证——用医院已确诊的历史病例（有明确最终诊断的），把病例的"症状+检查"输入多智能体，看它给的诊断是否和"最终确诊"一致，准确率 = 一致/总数。这要求医院提供脱敏的历史病例库（有伦理审批）；2）前瞻性影子测试——Agent 和真实医生并行处理新病例（Agent 不影响实际诊疗，只做"影子判断"），对比 Agent 诊断和医生最终诊断的一致率，验证"真实场景"效果；3）专科 benchmark——用公开医学考试题（如 USMLE）或专科基准（如 MedQA）测各专科 Agent 的专科能力。三个层面交叉验证，准确率达阈值（如回顾性 85%+）才考虑辅助上线。

**Q：医疗多智能体的架构和专科能力怎么沉淀成可复用的医疗 Agent 平台？**

封装成 MedicalAgentPlatform：1）专科 Agent 模板——心内科/神经科/消化科等专科 Agent 的 prompt 模板 + 专科 RAG 知识库（指南/文献），新专科接入改模板即可；2）分诊 Agent——基于症状的分诊路由（症状→可能涉及专科），复用医学知识图谱；3）审核 Agent——多专科意见综合 + 规则引擎校验（内置常见"诊断-检查矛盾"规则）；4）合规层——分级自主权（低/中/高风险）、审计日志（每次诊断可追溯）、人类医生确认接口；5）评测框架——回顾性/影子/专科 benchmark 三层评测自动化。这套写入医疗 Agent 平台 SOP，新医院/新专科接入时复用平台能力，聚焦于本地化（该院的数据/指南）而非重写架构。

## 结构化回答
