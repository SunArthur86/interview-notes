# 【月之暗面面经】本地索引、云端推理和产物导出是三段链路，前端怎样串起来？

## 一、三段链路全景图

```
┌──────────────────────────────────────────────────────────────────┐
│                     桌面AI任务全链路                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ① 本地索引段        ② 云端推理段        ③ 产物导出段             │
│  (Local Index)       (Cloud Inference)    (Artifact Export)     │
│                                                                  │
│  ┌──────────┐       ┌──────────┐         ┌──────────┐          │
```

## 二、分段状态机

```typescript
// 每段链路有独立的状态机
type IndexPhase = 
  | { status: 'idle' }
  | { status: 'scanning'; filesProcessed: number; totalFiles: number }
  | { status: 'extracting'; currentFile: string }
  | { status: 'summarizing'; processed: number; total: number }
  | { status: 'done'; fileCount: number; totalTokens: number }
  | { status: 'failed'; error: string; partialResults: boolean };

type InferencePhase =
  | { status: 'idle' }
  | { status: 'preparing'; promptTokens: number }
  | { status: 'streaming'; receivedChars: number; receivedChunks: number }
  | { status: 'done'; outputTokens: number; duration: number }
  | { status: 'failed'; error: string; retryable: boolean }
  | { status: 'timeout'; duration: number };

type ExportPhase =
  | { status: 'idle' }
  | { status: 'converting'; artifactKind: string }
  | { status: 'writing'; bytesWritten: number; totalBytes: number }
  | { status: 'done'; exportPath: string; fileSize: number }
  | { status: 'failed'; error: string };

// 任务总状态 = 三段的组合
interface TaskPhases {
  traceId: string;
  index: IndexPhase;
  inference: InferencePhase;
  export: ExportPhase;
}
```

## 四、分段错误处理

```
┌──────────────────────────────────────────────────────────────────┐
│  任务失败                                                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ① 本地索引    ✅ 完成                                           │
│  ② 云端推理    ❌ 失败                                           │
│     ┌────────────────────────────────────────────────┐          │
│     │ 错误：请求超时 (30s)                             │          │
│     │ 原因：云端推理服务响应超时                        │          │
│     │ 建议：检查网络连接或减少输入素材                   │          │
│     │                                                 │          │
│     │ [ 重试此段 ]  [ 减少上下文后重试 ]  [ 查看日志 ] │          │
│     └────────────────────────────────────────────────┘          │
│  ③ 产物导出    ⏳ 未开始（因上一步失败）                           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

```typescript
// 分段重试：只重试失败的段
async function retryPhase(taskId: string, phase: 'index' | 'inference' | 'export') {
  const task = await taskStore.getTask(taskId);
  
  switch (phase) {
    case 'index':
      // 重新索引（云端推理和导出需要重新执行）
      await reindexInputs(task);
      break;
      
    case 'inference':
      // 只重试推理（复用已有的索引结果）
      const indexResult = task.phases.index;
      if (indexResult.status === 'done') {
        await retryInference(task, indexResult);
      }
      break;
      
    case 'export':
      // 只重试导出（复用推理结果）
      const inferenceResult = task.phases.inference;
      if (inferenceResult.status === 'done') {
        await retryExport(task, inferenceResult);
      }
      break;
  }
}
```

## 六、常见坑

- **一个loading覆盖一切**：用户不知道当前在索引、推理还是导出，也不知道卡在哪里
- **失败提示不区分**：只显示"任务失败"，不说明是哪一段失败、什么原因
- **整单重试**：索引已经完成了，失败在推理，却要重新索引
- **没有traceId**：出问题后无法关联本地日志和云端日志进行排查

## 记忆要点

- 三段特征差异：本地索引CPU密集、云端推理网络密集、产物导出IO密集，失败模式各不相同。
- 全链路追踪：必须使用唯一的traceId贯穿本地解析、云端大模型与本地文件写入。
- 分段状态机：总状态等于各段状态的组合，前端需针对不同阶段展示独立的加载与失败态。
- 拒绝一降到底：不能用单一Loading掩盖所有过程，需分段提示（如提取中、生成中、保存中）。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么要把本地索引、云端推理、产物导出拆成三段独立链路展示，用一个"任务进行中 45%"的进度条覆盖一切不行吗？**

因为三段链路的延迟特征、失败模式和用户可采取的补救措施完全不同，混合展示会让用户在出错时"不知道卡在哪、能做什么"。本地索引段（CPU 密集，秒到分钟级）失败通常是文件不存在或格式不支持，补救是换文件；云端推理段（网络密集，秒到分钟级）失败通常是网络超时或 API 限流，补救是重试或减少输入；产物导出段（IO 密集，毫秒到秒级）失败通常是磁盘空间不足或权限问题，补救是换路径或清理磁盘。如果用一个混合进度条，用户看到"卡在 45%"完全不知道是索引卡了（换文件就好）还是推理卡了（重试就好），只能取消重跑整单。分段展示让用户看到"索引完成✓ / 推理失败❌ / 导出未开始"，立刻知道问题在推理段，且可以只重试推理段（复用已完成索引的结果），省下重新索引的时间。所以三段拆分的本质是"把混合的失败模式拆解为可独立诊断和补救的单元"。

### 第二层：证据与定位

**Q：你怎么定位"任务卡住了 30 秒没动静"是哪一段的问题——本地索引卡了、云端推理慢了，还是网络断了？**

用分段状态机 + traceId 链路日志定位。分段状态机记录每段的 status 和最后一次更新时间戳，如果某段的 status 超过 N 秒没更新（如 inference 段的 streaming 状态超过 15 秒没收到新 chunk），判定为该段"疑似卡住"。然后用 traceId 查链路日志：(1) 如果 index 段的日志停在"正在解析第 3 个文件"，说明本地索引卡在某个文件的解析（可能是文件过大或格式异常）；(2) 如果 inference 段的日志显示"请求已发出但无响应"，且本地网络 ping 云端超时，说明是网络问题；(3) 如果 inference 段的日志显示"云端返回 429 Rate Limit"，说明是 API 限流。三种情况对应不同的用户提示：索引卡住→"第 3 个文件解析困难，是否跳过？"；网络断了→"网络连接中断，请检查后重试"；限流→"请求过于频繁，30 秒后自动重试"。没有分段状态机和 traceId，只能告诉用户"任务异常"，无法定位。

### 第三层：根因深挖

**Q：为什么 traceId 必须贯穿三段链路，不能每段用自己的 requestId？**

因为"分段 requestId"无法关联跨段的因果关系。一个完整的任务是三段的因果链：索引段的输出（提取的上下文）是推理段的输入，推理段的输出（生成的产物）是导出段的输入。如果推理段返回了异常结果（如产物内容为空），要排查是"索引段提取的上下文有问题"还是"推理段本身生成有 bug"，必须用同一个 traceId 串联三段的日志。如果每段用不同 requestId，排查时要先从推理段日志找到对应的索引段 requestId（可能记录在某个字段里），再去索引段日志查——这依赖人工关联，容易断链。traceId 贯穿三段后，一次查询 `GET /api/traces/{traceId}` 就能拿到按时间排序的全链路日志（索引日志→推理日志→导出日志），因果链清晰可见。这在分布式系统中是标准实践（如 OpenTelemetry 的 trace context 跨服务传播），桌面 AI 的三段链路本质上是"本地服务→云端服务→本地服务"的分布式调用，必须用统一的 traceId。

**Q：那如果三段链路里有子步骤（如索引段内部有"文件解析→OCR→摘要"三个子步骤），为什么不每个子步骤一个 traceId 精确定位？**

因为"每个子步骤一个 traceId"会破坏"一个任务一个 traceId"的简洁性，且子步骤的定位用 spanId 解决更合适。正确做法是用"traceId + spanId"的层级结构：一个任务一个 traceId（贯穿三段），每段内部的子步骤用 spanId（如 index 段有 span-1 文件解析、span-2 OCR、span-3 摘要）。查询时先按 traceId 拿到全链路，再按 spanId 展开某段的内部细节。这比"每子步骤独立 traceId"更优：(1) 查询简洁——一次 traceId 查询拿到完整链路，不需要记多个 ID；(2) 因果清晰——同一个 traceId 下的 span 有父子关系（OCR span 的 parent 是文件解析 span），能还原执行树；(3) 聚合方便——统计"索引段的平均耗时"时，按 traceId 分组取 index span 的耗时即可。这是 OpenTelemetry 的标准 tracing 模型（一个 trace 包含多个 span，span 之间有 parent-child 关系），直接复用即可，不需要发明新的多 traceId 方案。

### 第四层：方案权衡

**Q：分段重试你设计成"只重试失败的段"（如推理失败只重试推理，复用索引结果），为什么不简化成"整单重跑"？**

选分段重试，原因是长任务的重跑成本极高。一个完整任务：索引 5 秒 + 推理 30 秒 + 导出 2 秒 = 37 秒。如果推理段在第 28 秒失败（网络超时），整单重跑要再花 37 秒（含重新索引的 5 秒），分段重试只花 30 秒（复用索引结果只重跑推理）。对于更长的任务（索引 2 分钟 + 推理 5 分钟），差异更显著：整单重跑 7 分钟 vs 分段重试 5 分钟。更重要的是"索引结果的复用性"——索引段做了文件解析、OCR、摘要提取等 CPU 密集工作，结果是可以缓存的（只要输入文件没变，提取结果不变），没有理由因为后续段失败就丢弃。分段重试的前提是每段有 checkpoint（序列化中间结果），实现成本是每段要设计 checkpoint 的序列化/反序列化，但这对于长任务是值得的工程投入。

**Q：那如果分段重试时发现前一段的 checkpoint 已经过期（如索引的文件被用户修改了），为什么不放弃分段重试直接整单重跑？**

因为"checkpoint 过期"是可检测的，检测到过期才需要重跑该段，而非默认放弃所有分段。检测方法：每个 checkpoint 记录生成时的输入指纹（如文件的 hash、URL 的内容版本），重试前对比当前输入的指纹与 checkpoint 记录的指纹：一致则 checkpoint 有效，可直接复用并重试失败段；不一致则该段 checkpoint 过期，需要重跑该段（但不影响其他段）。具体场景：推理段失败要重试，检查索引段 checkpoint 的文件 hash——如果文件没变，索引 checkpoint 有效，只重跑推理；如果文件变了，索引 checkpoint 过期，先重跑索引段再做推理。导出段失败要重试时，检查推理段 checkpoint（生成的产物内容）是否存在——产物内容不会"过期"（它已经生成好了），所以导出段的重试几乎总能复用推理结果。所以根因是"过期检测要精确到段级别"，而非笼统地"可能过期就全部重跑"——后者把 1 段的过期代价放大到 3 段的重跑代价。

### 第五层：验证与沉淀

**Q：你怎么证明"分段状态机展示"比"单一进度条"更能帮助用户理解和应对任务失败？**

用两个指标验证：(1) 任务取消率 = 用户主动取消任务的次数 / 总任务数——如果分段展示让用户知道"只是推理段超时，重试就好"，取消率应显著下降（从"以为整个任务废了"降为"重试这一段"）；(2) 分段重试使用率 = 用户点击"只重试失败段"的次数 / 失败任务数——如果这个率 >50%，说明分段重试是真需求（用户确实想复用已完成段的结果）。还要看一个定性指标：用户工单内容——如果工单从"任务失败了不知道为什么"变成"推理段超时了能加长超时时间吗"，说明分段展示成功帮用户定位了问题。A/B 测试时，对照组用单一进度条 + "任务失败"提示，实验组用三段状态机 + 分段重试，跑两周后对比取消率和重试成功率——如果实验组的"失败任务最终完成率"（重试后成功）高出 30% 以上，就证明分段设计有效。

**Q：怎么让团队在新增链路段（如新增"云端后处理段"或"本地缓存段"）时，自觉遵循分段状态机 + traceId + checkpoint 的模式，而不是把新段塞进已有段的逻辑里？**

把三段架构做成"链路编排框架 SDK"的强制约束。第一，定义 Phase 接口（status 状态机 + checkpoint 序列化 + retry 逻辑 + traceId 日志），所有链路段必须实现该接口并通过 PhaseRunner 编排，不允许开发者自己写裸的 async 函数串联段；第二，PhaseRunner 自动管理 traceId 传播（每段自动注入 traceId 到日志和请求头）、自动做 checkpoint（每段结束时序列化结果）、自动支持分段重试（失败时只重跑指定段），开发者只需实现段的业务逻辑；第三，UI 层的进度展示由 PhaseRunner 驱动——它自动把各段状态推送到 TaskStore，UI 订阅后渲染分段进度条，开发者不写 UI 更新逻辑；第四，新增段时必须注册到 PhaseRegistry，未注册的段无法接入编排——这强制开发者为新段定义状态机和 checkpoint。这样分段模式就从"自觉规范"变成了"框架约束"，不遵循就无法接入链路编排。
