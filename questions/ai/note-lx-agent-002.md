# ASR选型怎么做，为什么不能只看字错率？流式ASR延迟由哪些环节决定？

## 为什么不能只看字错率

```
ASR选型的五维评估框架：

         准确率(CER)
          /    \
         /      \
    鲁棒性 ----- 延迟(RTF)
         \      /
          \    /
     部署成本 - 领域适配

字错率(CER)只覆盖了"准确率"一个维度！

实际部署中的关键指标：
┌────────────┬──────────────────────────┬──────────┐
│ 指标        │ 含义                      │ 目标值   │
├────────────┼──────────────────────────┼──────────┤
│ CER         │ 字错率（准确率倒数）       │ <5%      │
│ RTF         │ 实时率（处理时间/音频时长）│ <0.3     │
│ 首字延迟    │ 从说话到第一个字出现的延迟  │ <300ms   │
│ 完句延迟    │ 说完后完整结果返回的延迟    │ <500ms   │
│ 方言鲁棒性  │ 非标准普通话的准确率       │ >85%     │
│ 噪声鲁棒性  │ 背景噪声下的准确率         │ >80%     │
│ 并发能力    │ 单GPU同时处理的流数        │ >100路   │
└────────────┴──────────────────────────┴──────────┘
```

## 流式 vs 非流式ASR

```
非流式ASR（Offline）：
用户说话: "帮我查一下明天的天气" (说完)
          ████████████████████████ 3s音频
                                        ↓ 等待
                                   ┌─────────┐
                                   │ 全量识别  │ 1s
                                   └────┬────┘
                                        ↓
                                   返回结果  总延迟=4s

流式ASR（Streaming）：
用户说话: "帮我查一下明天的天气"
          ████                          ← 200ms出第一段
               ████████                 ← 持续输出
                      ████████          ← 持续输出
                              ████      ← 说完后200ms出最后一段
          ↑                              ↑
      首字延迟~200ms              完句延迟~200ms
```

### 根本区别

```python
# 非流式：必须等待完整音频才能开始识别
def offline_asr(audio_stream):
    full_audio = audio_stream.read_all()  # 阻塞等待
    text = model.transcribe(full_audio)   # 一次性处理
    return text

# 流式：边接收音频边输出识别结果
async def streaming_asr(audio_stream):
    chunk_buffer = []
    async for chunk in audio_stream:
        chunk_buffer.append(chunk)
        # 增量识别：每次新chunk到达就更新结果
        partial_text = model.transcribe_streaming(chunk_buffer)
        yield partial_text  # 实时输出中间结果
    yield model.get_final()  # 输出最终结果
```

## 流式ASR延迟拆解

```
用户开口说话
    │
    ├─→ VAD端点检测: 检测到人声开始   300-500ms ← 瓶颈1
    │
    ├─→ 音频分帧: 每100ms一帧送入模型  ~10ms
    │
    ├─→ 声学模型推理: 编码音频帧       50-100ms ← 瓶颈2
    │
    ├─→ 语言模型解码: 解码为文本       30-50ms
    │
    ├─→ 网络传输: 结果回传             20-50ms  ← 瓶颈3
    │
    ▼
首字延迟总计: 400-700ms（理想200-300ms）
```

## ASR选型对比

| 方案 | CER | 流式 | 延迟 | 成本 | 适用场景 |
|------|-----|------|------|------|---------|
| Whisper-large | 3% | ❌ | 高 | 高 | 离线转写、字幕 |
| Paraformer | 4% | ✅ | 中 | 中 | 通用语音交互 |
| SenseVoice | 5% | ✅ | 低 | 低 | 实时对话、多语言 |
| WeNet | 6% | ✅ | 极低 | 极低 | 本地部署、低延迟 |

## 面试加分点

1. **场景驱动选型**：客服转写用Whisper（准但慢），实时对话用SenseVoice（快但略逊），边缘设备用WeNet（小快省）
2. **端到端优化**：ASR延迟不只是模型问题，VAD+网络+解码都是瓶颈，需要全链路优化
3. **热词增强**：业务专有名词（产品名、人名）通过热词表增强，CER可降低2-3%
4. **与LLM协同**：ASR的中间结果可以先送给LLM做意图预判，不等说完就开始准备回答

## 记忆要点

- 核心结论：CER 准确率只占一维，需综合考量 RTF、延迟、鲁棒性与并发
- 流式非流式对比：流式边收音边出结果，首字延迟低交互体验好
- 延迟瓶颈1：VAD 端点检测耗时较长，约占 300-500ms
- 延迟瓶颈2：声学模型推理耗时，约占 50-100ms
- 延迟瓶颈3：网络传输与解码耗时，合理控制可优化整体首字延迟
