## 三、推理层优化

### KV Cache

```python
# 问题：每生成一个token，要重新计算所有之前的attention
# KV Cache：缓存已计算的Key/Value，避免重复计算

# 无Cache: 生成N个token要O(N²)计算
# 有Cache: 生成N个token要O(N)计算（只算新的）

# 进阶：Prefix Caching
# 相同的system prompt，KV Cache可复用
# 多个请求共享同一system prompt → 只算一次
```

### 投机解码（Speculative Decoding）

```python
# 原理：小模型快速"猜"多个token，大模型并行"验证"
# 小模型猜得对 → 省时间（用了小模型的速度）
# 小模型猜错 → 大模型纠正（保证质量）

# 效果：吞吐提升2-3倍，质量无损
class SpeculativeDecoding:
    def generate(self, prompt):
        # 1. 小模型快速生成5个候选token
        draft = small_model.generate(prompt, n=5)
        
        # 2. 大模型并行验证5个
        verified = large_model.verify(prompt, draft)
        
        # 3. 接受正确的，重新生成错误的
        accepted = verified.accepted
        if not verified.all_accepted:
            accepted += large_model.regenerate(verified.rejected_pos)
        
        return accepted
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说模型路由能降 70% 成本，但路由本身要先用一个模型判断复杂度，这一步不是也要花钱？怎么保证净收益是正的？**

路由判断用极小成本换大收益。复杂度判断不用大模型——用一个轻量分类器（fine-tuned BERT 或者直接用规则 + embedding 相似度），单次成本是大模型的 1/100，耗时 < 50ms。假设 80% 请求被路由到 7B（成本是 GPT-4 的 1/20），路由判断成本即使算上，总成本还是降 70%+。净收益为负只有一个场景：路由误判率高，把简单题错判成难题走大模型（白花钱），或把难题判简单走小模型（质量差返工）。所以路由的 ROI 关键在分类器的准确率，我会持续标注误路由样本迭代分类器。

### 第二层：证据与定位

**Q：你上了 vLLM 之后吞吐没明显提升，反而偶发 OOM，怎么定位是 vLLM 配置问题还是模型不兼容？**

看两个证据。第一看 vLLM 启动日志的 KV Cache 显存占比——如果 PagedAttention 的 block_size 配太大或 gpu_memory_utilization 配太高（>0.95），KV Cache 抢显存导致 OOM，是配置问题，调小到 0.9。第二看模型格式——vLLM 对某些量化格式（如老的 GPTQ 格式）支持不好，会走慢路径或显存翻倍。用 `--dtype auto` 让它自动选，或者把模型转成 vLLM 原生支持的 safetensors。定位方法：先用官方 benchmark 脚本跑同模型，如果官方 benchmark 快、你的慢，就是你的部署配置问题；如果都慢，是模型/版本不兼容。

### 第三层：根因深挖

**Q：INT8 量化你说几乎无损，"几乎"到底是多少？怎么量化这个损失，什么场景不能用 INT8？**

量化的损失用业务评测集的准确率下降衡量，不是 PPL。我的做法：FP16 和 INT8 各跑同一套 500 条业务评测集，对比准确率。典型数据：通用 QA 指标下降 < 1%（可接受），但数学推理/代码生成下降 3-5%（GSM8K 这种对数值精度敏感的任务）。根因是量化把权重从 FP16 压到 INT8，激活值的微小误差在多步推理里被放大。所以 INT8 不能用的场景：数学计算、代码生成、需要严格遵循格式的任务。这些场景要么用 FP16，要么用对激活更友好的量化方案（如 AWQ，比朴素 INT8 在这些任务上损失小）。

**Q：那为什么不直接全用 FP16 不量化，反正现在 GPU 显存也不贵？**

因为量化省的不只是显存，还有吞吐。INT8 的矩阵乘法用 INT8 Tensor Core，吞吐是 FP16 的 2 倍（A100 上 INT8 算力是 FP16 的 2 倍）。所以在"显存够但追求吞吐"的场景（高并发 API 服务），量化也能让单卡 QPS 翻倍，直接摊薄每请求成本。显存便宜不代表算力便宜——GPU 的算力才是瓶颈。所以量化是显存和算力双省，不是只为省显存。

### 第四层：方案权衡

**Q：投机解码你说吞吐翻倍质量无损，但小模型猜错时大模型要重算，最坏情况比直接用大模型还慢，你怎么保证不劣化？**

保证不劣化的关键是控制"接受率"（acceptance rate）。投机解码的加速取决于小模型猜对的比例——猜对 5 个就省 5 次串行解码，猜错全废。我用接受率作为开关指标：如果实测接受率 > 60%，开投机解码净赚（加速 1.5 倍+）；接受率 < 40%，关掉（不如直接大模型）。提升接受率的方法是让小模型和大模型"对齐"——用大模型的输出蒸馏小模型，让两者分布接近，接受率能到 70-80%。所以不是无脑开投机解码，是看接受率决定开不开，且持续蒸馏提升接受率。

**Q：为什么不直接全用大模型 + KV Cache，反正 Cache 能省 70% 计算，还要搞投机解码那么复杂？**

因为 KV Cache 和投机解码优化的是不同环节，不互斥要叠加。KV Cache 优化的是"输入部分"——相同的 system prompt 复用已算的 KV，省的是 prefill 阶段。投机解码优化的是"输出部分"——生成每个 token 的串行解码，用小模型并行猜多个。一个请求 = prefill（输入）+ decode（输出），KV Cache 加速 prefill，投机解码加速 decode，叠加才能把端到端延迟压到最低。只做 KV Cache 的话 decode 阶段还是一个个 token 串行生成，长输出依然慢。

### 第五层：验证与沉淀

**Q：你上了一大堆优化（路由+量化+vLLM+缓存），怎么证明每个都有效，而不是叠加在一起分不清谁的功劳？**

用消融实验（ablation）。在固定评测集和压测流量下，逐个开关各优化项测指标：基线（无优化）→ +缓存 → +路由 → +量化 → +vLLM → +投机解码，每加一项记录吞吐/延迟/成本/准确率四个指标。这样每项的边际贡献清晰可量化。我之前的项目数据：缓存贡献成本降 30%，路由再降 35%，vLLM 提吞吐 3 倍，量化提吞吐 1.5 倍。有这个归因表，下个优化该投哪里（看哪个指标还没达标）就有数据依据，不是凭感觉堆优化。

**Q：这套优化怎么沉淀让团队不重复踩坑？**

沉淀成"推理优化 Playbook"，按成本/延迟/质量三个目标各给一组推荐组合。比如"成本优先档"= 路由 + INT8 + 缓存，"延迟优先档"= vLLM + 投机解码 + KV Cache，"质量优先档"= FP16 大模型不量化。每个档位附带压测报告（吞吐/延迟曲线）和踩坑记录（如某量化格式在 vLLM 上的兼容问题）。新服务按目标选档位直接套，不用从零摸索。Playbook 每季度更新——新模型/新引擎出来重新跑 benchmark，保持推荐组合不过时。
