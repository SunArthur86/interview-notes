# 本地生活场景下怎么做意图识别与槽位填充(Slot Filling)？

## 场景分析

用户输入："我想找个附近人均50左右好吃的川菜馆，最好能订座"

```
┌─────────────────────────────────────────────┐
│           用户自然语言输入                    │
│ "找个附近人均50的好吃的川菜馆，最好能订座"    │
└──────────────────┬──────────────────────────┘
                   ▼
┌──────────────────┴──────────────────────────┐
│              NLU 处理层                      │
├─────────────────────────────────────────────┤
│  意图识别(Intent)    │  槽位填充(Slot Filling)│
│  → intent: SEARCH   │  → location: 附近      │
│    RESTAURANT       │  → cuisine: 川菜       │
│                     │  → price: ~50/人       │
│                     │  → action: 订座        │
└──────────────────┬──────────────────────────┘
                   ▼
┌──────────────────┴──────────────────────────┐
│           后续 Agent 执行                    │
│  调用搜索API → 返回川菜馆列表 → 调用订座API  │
└─────────────────────────────────────────────┘
```

## 三种技术方案对比

### 方案一：规则+NER(传统方案)

```python
# 意图识别：关键词匹配
INTENT_RULES = {
    "SEARCH_RESTAURANT": ["找", "搜索", "推荐", "附近", "好吃"],
    "BOOK_TABLE": ["订座", "预约", "预订", "排队"],
    "WRITE_REVIEW": ["点评", "评价", "打分"],
}

# 槽位提取：NER模型(如BERT-CRF)
# 输入序列 → B-PERSON B-PRICE I-PRICE B-CUISINE
SLOT_DEFINITIONS = {
    "location": "位置",
    "cuisine": "菜系(川菜/粤菜/日料等)",
    "price_range": "人均价位",
    "party_size": "用餐人数",
    "time": "用餐时间",
}
```

**优点**：速度快、可控、成本低
**缺点**：维护成本高、泛化差、新意图需手动添加规则

### 方案二：小模型微调(BERT/RoBERTa)

```python
# 意图分类：BERT分类头
from transformers import BertForSequenceClassification

intent_model = BertForSequenceClassification.from_pretrained(
    "bert-base-chinese", num_labels=8  # 8种意图
)

# 槽位填充：BERT-CRF 或 BERT-Softmax
slot_model = BertForTokenClassification.from_pretrained(
    "bert-base-chinese", num_labels=20  # B-LOC I-LOC B-CUISINE O ...
)
```

**优点**：准确率高、推理速度快(BERT-tiny ~20ms)
**缺点**：需要标注数据、新意图需重新训练

### 方案三：LLM(Zero-shot / Few-shot) ← 推荐方案

```python
SYSTEM_PROMPT = """你是本地生活意图识别助手。分析用户输入，返回JSON。

意图类型：
- SEARCH_RESTAURANT: 搜索餐厅
- BOOK_TABLE: 预订座位
- GET_REVIEW: 查看评价
- NAVIGATE: 导航到店

槽位定义：
- location: 位置描述
- cuisine: 菜系偏好
- price_per_person: 人均价位(整数)
- party_size: 用餐人数(整数)
- reservation_time: 预订时间

输出格式：
{"intent": "...", "slots": {"key": "value"}, "confidence": 0.0-1.0}

示例：
输入：附近人均80的日料店
输出：{"intent": "SEARCH_RESTAURANT", "slots": {"location": "附近", "cuisine": "日料", "price_per_person": 80}, "confidence": 0.95}
"""

response = llm.chat(system=SYSTEM_PROMPT, user=user_input)
result = json.loads(response)  # 直接拿到结构化结果
```

**优点**：零样本启动、灵活扩展新意图、理解复杂表达
**缺点**：延迟高(200-500ms)、成本高、输出格式不稳定

## 生产级最佳实践：分层架构

```
┌──────────────────────────────────────┐
│  Layer 1: 缓存层 (Semantic Cache)    │ ← 相似query直接返回
├──────────────────────────────────────┤
│  Layer 2: 规则快通道                  │ ← 高频简单意图秒回
│  (正则+关键词)                        │
├──────────────────────────────────────┤
│  Layer 3: 小模型分类                  │ ← BERT-tiny 处理80%流量
│  (BERT分类+NER)                      │
├──────────────────────────────────────┤
│  Layer 4: LLM兜底                    │ ← 复杂query/低置信度走LLM
│  (GPT-4/GPT-3.5)                    │
└──────────────────────────────────────┘
```

## 保证LLM输出稳定JSON的技巧

1. **System Prompt约束**：明确要求JSON格式，提供few-shot示例
2. **Function Calling**：使用OpenAI function calling原生支持结构化输出
3. **JSON Mode**：启用response_format={"type": "json_object"}
4. **Schema验证**：输出后用Pydantic/jsonschema校验，失败则重试
5. **降级策略**：JSON解析失败→正则提取→规则兜底→人工客服

## 记忆要点

- 三大技术方案对比：规则+NER速度快泛化差，小模型微调需重训，LLM零样本启动灵活但延迟高
- 意图识别定动作(如订座)，槽位填充抓关键参数(如地点/价格/菜系)
- 因为LLM输出JSON不稳定，所以生产级最佳实践需增加结构化解析与兜底处理
- 小模型方案常采用BERT分类头定意图，配合BERT-CRF序列标注做槽位提取

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：意图识别 + 槽位填充，你对比了"规则+NER / 小模型微调 / LLM"三方案。为什么生产最终选了 LLM，而非传统小模型（BERT-CRF）？**

LLM 的"零样本泛化"和"维护成本"优势显著。传统小模型（BERT 分类头 + BERT-CRF）的问题：一是需标注数据——每个意图和槽位要标几百到几千条样本，新意图上线要重新标注 + 重训（周期长，1-2 周）；二是泛化差——小模型对训练分布外的表达泛化弱（如训练数据是"订餐厅"，用户说"帮我找个地方吃饭"，小模型可能识别不出"订餐厅"意图）；三是维护重——模型迭代要训练管线、版本管理、A/B 测试，工程成本高。LLM 的优势：一是零样本——无需标注，prompt 描述意图即可识别（新意图只需改 prompt，几小时上线）；二是泛化强——LLM 对各种表达方式鲁棒（"订餐厅"和"找地方吃饭"都能识别）；三是维护轻——改 prompt 即迭代，无需训练管线。LLM 的代价是延迟高（几百 ms vs 小模型几十 ms）和成本高（API 费用），但对大多数 Agent 场景可接受（延迟 <1s，成本可控）。选型看"意图数量和变化频率"——意图多且常新增用 LLM，意图少且稳定用小模型。

### 第二层：证据与定位

**Q：LLM 做意图识别，某个意图的识别准确率突然降到 50%（原本 90%）。怎么定位是 LLM 版本变了、prompt 不清晰、还是该意图本身难识别？**

三步定位。一是看失败样本——识别错误的 query 长什么样，是"和其他意图混淆"（如"订餐厅"和"订外卖"混淆，意图边界模糊）还是"完全识别错"（如"订餐厅"识别成"查天气"）。混淆多因 prompt 的意图描述不清晰（"订餐厅"和"订外卖"的 prompt 描述太像，LLM 分不清）；完全错可能是 LLM 没理解意图（prompt 太笼统）。二是查 LLM 版本——服务端是否切换了模型（如从 GPT-4-0613 换成新版本，新版本的指令遵循能力变了，同样的 prompt 效果不同）。三是对比历史——该意图之前是否一直 90%（突然降可能是版本 regression），还是一直偏低（该意图本身难，需优化 prompt 或加 few-shot）。治法：混淆改 prompt（明确区分相似意图，加对比示例）；版本问题锁定模型版本（不要随意升级）；难识别加 few-shot（给几个该意图的示例 query）。

### 第三层：根因深挖

**Q：LLM 输出 JSON 不稳定（如意图识别结果有时不是合法 JSON），根因是什么？为什么 LLM 会"跑偏"？**

根因是"LLM 是概率生成，无强制保证"。LLM 生成 JSON 靠 prompt 引导（"请输出 JSON 格式"），但模型基于概率采样，大多数时候输出合法 JSON，少数时候会跑偏（如多一个逗号、漏括号、加了自然语言解释）。跑偏的原因：一是 prompt 不够强约束（如只说"输出 JSON"没给 schema，模型自由发挥）；二是 query 复杂导致模型"分心"（如长 query 让模型关注内容而非格式）；三是 temperature 高（采样随机性大，可能选了非法 token）。治本：一是用 Function Calling（API 强制 schema，可靠性 99%+）；二是用约束解码（Outlines/GBNF，token 级掩码，100% 合法）；三是后处理修复（json-repair 库 + 重试）。生产级必须用 Function Calling 或约束解码，纯 prompt 引导的 JSON 不可靠（90-95% 成功率，线上 5% 失败不可接受）。

**Q：那为什么不直接用小模型（BERT 分类头）做意图识别（输出固定类别的 softmax，100% 是合法类别），省得担心 JSON 不稳定？**

小模型输出稳定但"泛化和维护"差。BERT 分类头的输出是 softmax 概率分布（固定 N 个意图类别，必输出合法类别），格式 100% 稳定。但问题：一是新意图要重训（加一个意图类别要重新标注 + 训练，1-2 周上线）；二是泛化弱（前面说的，对训练分布外的表达差）；三是需要标注数据（每个意图几百条）。LLM 的 JSON 不稳定是"工程问题"（可用 Function Calling 解决），小模型的泛化差是"模型能力问题"（需重训解决）。工程问题好解决（上 FC），能力问题难解决（标注 + 训练成本高）。所以即使 LLM 有 JSON 不稳定问题，通过 Function Calling 解决后，LLM 的"零样本 + 泛化 + 维护轻"优势仍远超小模型。选 LLM + FC 是当前主流（除非意图极度稳定且追求低延迟，才用小模型）。

### 第四层：方案权衡

**Q：槽位填充你用 LLM（输出 JSON 的槽位键值对）。为什么不直接用 NER 模型（如 BERT-CRF）专门做实体抽取，精度更高？**

NER 精度高但"槽位定义变化时需重训"且"复杂槽位难抽取"。NER 模型（BERT-CRF）对"命名实体"（如地点、人名、时间）抽取精度高，但槽位填充的槽位不一定是"命名实体"——如"菜系"（川菜、粤菜）、"价格区间"（便宜、中等、贵）、"人数"（2 人、4 人），这些槽位是"任务特定"的，通用 NER 模型不覆盖（要标注 + 微调）。且槽位定义变化（如新增"包间要求"槽位）要重训 NER，周期长。LLM 的优势：槽位定义在 prompt 里（JSON Schema 声明槽位名和类型），改槽位只需改 prompt，无需重训；且 LLM 能理解复杂槽位（如"价位中等偏上" → price_level: "medium-high"，NER 难抽取这种模糊表达）。LLM 的代价是延迟和成本，但对槽位填充（通常和意图识别一次调用搞定）可接受。选型：通用实体（地点/时间）可用 NER 精度优，任务特定槽位用 LLM 灵活优。多数 Agent 场景用 LLM（槽位定义常变，灵活更重要）。

**Q：为什么不直接用"规则 + 关键词匹配"做意图识别和槽位填充（最快最省），省得调 LLM？**

规则快但泛化极差且维护爆炸。规则方案（如"query 包含'订餐' → 意图=订餐"，正则抽"XX 人" → 槽位人数）对"标准表达"快且准（如"帮我订 4 人的餐厅"），但对"非标准表达"失效（如"找个能坐 4 个人的地方吃饭"，没有"订餐"关键词，规则识别不出）。且规则要覆盖所有表达变体（"订餐"、"预订餐厅"、"找地方吃饭"、"book a table"等），规则数量爆炸（每个意图几十条规则，维护噩梦）。新意图上线要写一堆规则（几天），LLM 改 prompt 几小时。规则的唯一优势是"延迟极低"（正则匹配 ms 级）和"成本零"，适合"意图极简单 + 表达标准化"的场景（如命令式交互"/order"）。自然语言交互（用户随意说）必须用 LLM（泛化）或小模型（折中）。生产 Agent 用自然语言交互，规则不够用。

### 第五层：验证与沉淀

**Q：你怎么衡量意图识别 + 槽位填充的效果，证明 LLM 方案可行？**

定义指标：一是意图识别准确率（intent_accuracy，在 golden set 上测，应 >90%）；二是槽位填充 F1（slot_F1，抽取的槽位是否正确 + 完整，应 >85%）；三是端到端任务成功率（E2E task_success_rate，意图 + 槽位都对了才能正确路由和执行）；四是延迟（P99 <500ms）和成本（每次调用 token 数）。做对比实验：规则 vs BERT-CRF vs LLM（无 FC）vs LLM（有 FC），在相同 golden set 上对比准确率/延迟/成本。关键验证"LLM 的泛化"——在"非标准表达"的测试集上（规则和 BERT 表现差的），LLM 应明显优。验证"FC 的稳定性"——LLM + FC 的 JSON 成功率应 99%+（vs 无 FC 的 90-95%）。golden set 要覆盖各意图的多种表达变体（标准 + 非标准），确保评估泛化能力。A/B 测试线上效果——LLM 方案的用户满意度（意图识别准 → 正确执行 → 用户满意）。

**Q：意图识别 + 槽位填充方案怎么沉淀成 Agent 框架标配？**

固化成"意图理解模块"：统一接口 `understand(query) → {intent, slots}`，内部用 LLM + Function Calling（Schema 声明意图和槽位）。沉淀"意图和槽位的 Schema 设计规范"（意图名清晰、槽位类型明确、必填/选填标注）、"prompt 模板"（含意图描述 + few-shot 示例）、"兜底策略"（LLM 失败时降级到关键词匹配或询问用户澄清）。配套监控（intent_accuracy、slot_F1、JSON 成功率、延迟），准确率降告警。把"LLM + FC"作为意图理解的默认方案，新 Agent 按规范定义意图和槽位 Schema 即可获得理解能力。积累"常见意图的混淆矩阵"（哪些意图易混淆 + 解法），帮助优化 prompt。

## 结构化回答

**展开框架：**
1. **意图识别** — 文本分类任务，输出intent label
2. **槽位填充** — 序列标注任务(NER)，提取实体参数
3. **方案演进** — 规则→小模型(BERT)→LLM(Zero-shot/Few-shot)
