# 如何保证大模型生成内容的合规性？

## 一、合规的四层防护

```
┌──────────────────────────────────────────────────┐
│              合规四层防护                            │
├──────────────────────────────────────────────────┤
│                                                    │
│  Layer 1: 输入过滤（预防）                         │
│    拦截违规请求在进入LLM前                         │
│                                                    │
│  Layer 2: 模型对齐（内化）                         │
│    RLHF/SFT让模型本身不想生成违规内容              │
│                                                    │
│  Layer 3: 输出审核（检测）                         │
│    LLM生成后，审核是否合规                         │
│                                                    │
│  Layer 4: 流程审计（追溯）                         │
│    全链路日志，可追溯可问责                        │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、Layer 1：输入过滤

```python
class InputComplianceFilter:
    """请求进入LLM前的合规过滤"""
    
    def check(self, user_input):
        checks = [
            self.check_illegal_content,    # 违法内容
            self.check_sensitive_words,    # 敏感词
            self.check_personal_info,      # 个人信息(脱敏)
            self.check_injection,          # 注入攻击
            self.check_policy_violation,   # 政策违规
        ]
        
        for check in checks:
            result = check(user_input)
            if result.blocked:
                return Reject(result.reason)
        
        return Allow(result.cleaned_input)
```

## 四、Layer 3：输出审核

```python
class OutputComplianceReview:
    """LLM生成后的合规审核"""
    
    async def review(self, response, context):
        results = {}
        
        # 1. 内容安全分类
        safety = await self.content_filter.moderate(response)
        # 返回: {violence: 0.01, sexual: 0.00, ...}
        if any(v > THRESHOLD for v in safety.values()):
            results["blocked"] = True
            results["reason"] = "内容安全审核未通过"
        
        # 2. 事实核查（防虚假信息）
        if context:
            factuality = await self.fact_check(response, context)
            if factuality.has_error:
                results["warning"] = "可能包含不准确信息"
        
        # 3. 隐私检测
        pii = self.pii_detector.detect(response)
        if pii:
            results["action"] = "redact"  # 脱敏后输出
        
        # 4. 业务合规
        business_violations = self.check_business_rules(response)
        
        return self.decide(results, response)
    
    def decide(self, results, response):
        if results.get("blocked"):
            return self.safe_reject()
        if results.get("action") == "redact":
            return self.redact(response)
        if results.get("warning"):
            return self.add_disclaimer(response)
        return response
```

## 五、内容审核实现

```python
class ContentModerator:
    """内容审核：规则+模型+人工"""
    
    # 第一道：规则过滤（快，覆盖明确违规）
    RULES = {
        "exact_blacklist": ["明确违规词1", "违规词2"],
        "regex_patterns": [r"手机号模式", r"身份证模式"],
    }
    
    # 第二道：分类模型（准，覆盖模糊case）
    def model_check(self, text):
        # 用专门的分类模型判断
        # 如OpenAI Moderation API / 自训练分类器
        scores = self.moderation_model.predict(text)
        return scores  # {violence: 0.9, sexual: 0.1, ...}
    
    # 第三道：人工审核（金标准）
    async def human_review(self, text):
        # 模型不确定的 → 人工
        if self.uncertain(text):
            return await human_moderator.review(text)
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：合规你说漏检率要 < 0.1%，但 LLM 是概率模型，怎么承诺这么低的漏检率？这不是违反模型本质吗？**

0.1% 不是靠模型单独承诺的，是靠"模型 + 规则 + 人工"三道防线叠加达成的。单靠 LLM 的对齐训练，漏检率大概在 2-5%（攻击者能越狱）。加上输出侧的内容安全分类模型（专门训练的违规检测器，比通用 LLM 在违规识别上准），漏检率降到 0.5%。再叠加规则黑名单（对明确违规词 100% 拦截）和人工抽审（高风险场景全量人审），把残余漏检压到 0.1%。所以 0.1% 是系统工程的结果不是单模型能力，合规必须靠纵深防御，这也是为什么我强调四层而不是只靠模型对齐。

### 第二层：证据与定位

**Q：线上被监管通报了一次"输出了不当内容"，你怎么快速定位是哪一层防线失效？**

用审计日志做全链路回溯。拿到那条违规输出的 incident_id，查审计日志看：输入侧过滤是否触发（如果输入有明显违规词却没拦，是规则层失效）、模型对齐是否生效（如果输入合法但模型主动生成违规，是对齐/模型版本问题）、输出审核是否拦截（如果内容安全分类器的分数 < 阈值却实际违规，是分类器召回漏洞或阈值太松）。三个环节的日志一目了然定位到哪层漏了。典型情况是输出审核的分类器对该类违规（比如新型话术）没见过、分数给低了，那就是要补这类样本重训分类器。

### 第三层：根因深挖

**Q：误杀率你目标 < 1%，但实际跑下来有 5%，正常用户被拦了疯狂投诉。根因是什么？只调松阈值行不行？**

根因通常是规则太粗或分类器在边界 case 判断不稳。调松阈值能降误杀但会升漏检，是按下葫芦浮起瓢，不行。真正的根因解法是细化判定：把"误杀"的样本拿出来分析，会发现误杀集中在几类边界 case（比如医学科普讨论被误判为违规医疗建议）。对这几类单独训练细分类器（医学科普 vs 违规医疗的判别），而不是用一个粗分类器管所有。第二是加"申诉回溯"——被拦的请求允许用户申诉，申诉进人工审核，人工判正常的反哺成负样本训练分类器。所以降误杀靠精细化分类 + 闭环反馈，不是调阈值。

**Q：那为什么不直接对所有内容都人工审核，准确率最高，还要费劲搞模型分类？**

因为成本和延迟。全量人工审核一天几十万条，一条 2-5 分钟，要几千人，成本不可承受；而且人工审核加 1-3 分钟延迟，实时对话场景用户等不了。模型分类的价值是"把人工量从全量降到 5-10%"——模型能高置信判定的（95% 的正常内容、明显的违规）自动处理，只把模型不确定的（置信度在 0.3-0.7 之间的灰度区）送人工。这样人工只审真正需要判断的，成本和延迟可控。所以模型 + 人工是"规模 + 质量"的分工，不是要谁替代谁。

### 第四层：方案权衡

**Q：你提到不同地区合规策略不同（GDPR/个保法），但维护多套规则成本高，你怎么权衡全球统一 vs 本地化？**

用"全球基线 + 本地增量"的分层架构。全球基线是最严的那套（基本把各地重叠的要求都覆盖），保证默认情况下各地都合规。本地增量是各地特有的差异化要求（比如 GDPR 的"被遗忘权"、中国的数据出境审查），做成可插拔的插件。运行时按用户所在地加载"基线 + 对应插件"。这样维护成本是 1 套基线 + N 个小插件，而不是 N 套独立规则。判断一个要求是进基线还是插件：如果多地都有（即使措辞不同）进基线，只有一地有的进插件。

**Q：为什么不直接用最严的规则（比如把所有地区的最严要求叠加）全球统一执行？**

因为会过度合规伤害体验。比如某地区要求"任何 AI 生成内容必须标注"，如果全球都强制标注，在没这要求的地区用户体验受损（每条回复带免责声明显得啰嗦）。再比如 GDPR 的数据留存限制最严，全球都按它来会让其他地区丢失本可利用的数据（影响个性化）。过度合规的代价是产品竞争力下降。所以正确做法是"各地按当地法规执行"，用基线+插件实现，而不是一刀切用最严的。

### 第五层：验证与沉淀

**Q：你怎么证明这套合规体系真的扛得住监管检查，而不是自说自话？**

对标行业标准 + 外部审计两步。对标是拿系统的能力对标行业基准（如 OpenAI Moderation API 的违规召回率、Trust & Safety 的行业 SLA），如果我们的指标（漏检率/误杀率）达到或超过基准，说明在业内属合格。外部审计是请第三方合规机构（如 ISO 27701 认证机构、律师事务所）做定期审计，出具合规报告。监管检查时出示这些第三方报告比自己说"我们做了四层防护"可信得多。另外保留完整的审计日志（每次拦截/放行的证据链），监管要查时能追溯到具体 case，这是合规可问责的硬证据。

**Q：这套合规体系怎么沉淀成团队/公司标准？**

沉淀成"合规中台"，把输入过滤/输出审核/审计日志/规则管理做成平台能力，各业务线接入而非自建。规则在统一后台管理（法务/安全团队能直接配规则和阈值，不用研发改代码）。配套"合规变更流程"——法规更新时法务评估影响、安全团队调规则、测试验证后灰度上线，全流程可追溯。这样合规能力是公司级的，法规变化时一处调整全局生效，避免各业务线各自为政导致某条线漏了新法规。
