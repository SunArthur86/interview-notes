# 不同场景下提示词怎么写？

## 一、四大场景的 Prompt 策略

```
┌──────────────┬──────────────────────────────────┐
│ 场景          │ Prompt 核心策略                     │
├──────────────┼──────────────────────────────────┤
│ 客服/助手     │ 流程SOP + 边界 + 语气 + 兜底        │
│ 编程/技术     │ 精确性 + 格式约束 + 测试验证        │
│ 创意/写作     │ 风格示例 + 自由度 + 灵感引导        │
│ 分析/推理     │ 逻辑链 + 数据引用 + 结论明确        │
└──────────────┴──────────────────────────────────┘
```

## 二、场景 1：客服/助手

```python
CUSTOMER_SERVICE_PROMPT = """
你是XX公司的智能客服。

【工作流程】
1. 理解用户问题，判断类型（查询/操作/投诉/咨询）
2. 查询类：调用query工具获取信息
3. 操作类：确认后调用对应工具
4. 回复用户，语气友善

【能力边界】
- 可以：查询订单/物流/产品信息，发起退款申请
- 不能：直接审批退款（需转主管）、承诺补偿
- 超出范围：礼貌告知并转人工

【语气】
- 用"您"，礼貌专业
- 简洁明了，不啰嗦
- 遇到投诉：先共情再解决

【兜底】
- 不确定时：不要编造，说"我帮您查询一下"
- 用户不满：主动道歉并提供解决方案
- 情绪激动：安抚+转人工
"""
```

## 三、场景 2：编程/技术

```python
CODING_PROMPT = """
你是资深开发者，帮助解决编程问题。

【要求】
1. 代码必须可直接运行（不要伪代码）
2. 标注语言版本和依赖
3. 关键逻辑加注释
4. 附带简单测试用例

【输出格式】
```[语言]
[完整可运行代码]
```

【测试】
```[语言]
// 测试用例
assert function(input) == expected
```

【说明】
- 时间/空间复杂度
- 关键设计决策的理由

【约束】
- 不写不安全的代码（如SQL拼接）
- 不确定API时标注"请核实版本"
- 给多个方案时说明各自优劣
"""
```

## 七、面试加分点

1. **场景差异化**：不同场景用不同策略，而非一套 Prompt 通吃
2. **客服重边界**：客服 Prompt 最重要的是"不能做什么"和"兜底"
3. **编程重验证**：编程 Prompt 要要求可运行代码+测试，体现严谨

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：不同场景的 Prompt 策略不同（客服重流程/编程重精确/创作重风格/分析重逻辑），为什么不统一用"最强通用 Prompt"（一个 Prompt 打天下）？**

因为场景的核心挑战不同，通用 Prompt 无法针对性优化。1）客服——核心是"流程合规+边界控制"（按 SOP 回答/不超范围/转人工），通用 Prompt 没强调流程/边界会失控（客服乱答/超范围），要流程规范+能力边界；2）编程——核心是"精确+可验证"（代码正确/可测试），通用 Prompt 易写错代码（不强调测试/边界条件），要精确约束（如"考虑边界/写测试"）；3）创作——核心是"风格+示例"（符合品牌风格/有创意），通用 Prompt 输出平庸（无风格/无创意），要风格指引+few-shot 示例；4）分析——核心是"逻辑+数据"（推理严谨/基于数据），通用 Prompt 易臆断（不基于数据/逻辑松），要逻辑规范+数据引用。所以场景化的核心是"针对该场景的核心挑战定制 Prompt"，通用 Prompt 不针对，效果差。

### 第二层：证据与定位

**Q：某场景的 Agent 效果差（如客服答非所问/编程代码错），怎么定位是 Prompt 策略不适配场景还是 Prompt 内容问题？**

对比+归因。1）场景适配——判断当前 Prompt 策略是否匹配场景核心挑战（如客服 Prompt 是否强调流程/边界，编程 Prompt 是否强调精确/测试），如果不匹配（如客服 Prompt 没流程规范），是策略不适配，要调整策略；2）内容质量——如果策略适配（如客服 Prompt 有流程规范）但效果差，看具体内容（如流程规范写得对不对/清不清），内容差优化内容；3）对比——用"场景化 Prompt"vs"通用 Prompt"对比效果，如果场景化显著好（证明场景化必要）但仍差（内容问题）或场景化和通用差不多（策略可能不适配，要重新设计）；4）症状归因——答非所问（理解错→Prompt 角色不清）、代码错（不精确→没强调测试/边界）、风格不对（没风格指引）。定位方法：判断策略适配（对照场景核心挑战）→对比通用→症状归因→优化。

### 第三层：根因深挖

**Q：编程场景 Prompt 要"精确+可验证"，具体怎么写才能让 LLM 生成正确且可测试的代码？**

精确约束+验证要求。1）明确需求——Prompt 要把需求写清楚（输入/输出/边界/约束，如"函数接受 list[int] 返回 list[int]，空列表返回空，含负数按原序"），减少 LLM 猜测；2）边界强调——显式要求考虑边界（如"考虑空输入/极值/异常类型，给出处理"），防 LLM 忽略边界；3）测试要求——要求 LLM 同步给测试（如"写单元测试覆盖正常/边界/异常 case"），可验证；4）示例——给 few-shot（典型输入输出对），让 LLM 学习预期行为；5）代码规范——要求遵循规范（如"有注释/类型注解/符合 PEP8"），可维护；6）自我检查——要求 LLM 生成后自检（如"检查代码是否有 bug/边界是否覆盖"），自我纠错。反例："写个快排"（太模糊，LLM 可能忽略边界/无测试）。原则：需求明确+边界强调+测试要求+示例+规范+自检，生成精确可验证代码。

**Q：创作场景 Prompt 要"风格+示例"，但风格是抽象的（如"年轻化/有创意"），怎么把抽象风格变成 Prompt 能用的具体指引？**

具体化风格+示例示范。1）风格具体化——把抽象风格拆成具体维度（如"年轻化"→用词活泼/语气轻松/引用流行梗；"有创意"→比喻新颖/角度独特/出人意料），每维度给具体指引（如"用'绝绝子''yyds'等流行词"）；2）示例示范——给几个符合风格的示例（输入→风格化输出），让 LLM 模仿（few-shot，最有效的风格传递）；3）反向示例——给反例（如"不要像这样正式/枯燥"），让 LLM 避免不对的风格；4）风格词库——提供风格词库（如年轻化的常用词/句式），LLM 选用；5）评估校准——生成后评估风格（人工/LLM 评判"是否符合年轻化"），不一致则优化 Prompt（加强风格指引）。原则：抽象风格→具体维度+示例（正反）+词库，让 LLM 可执行而非模糊理解。

### 第四层：方案权衡

**Q：分析场景 Prompt 要"逻辑+数据"，但 LLM 可能臆断（不基于数据编结论），怎么用 Prompt 约束 LLM 基于数据推理？**

显式约束+推理结构。1）数据优先——Prompt 明确"结论必须基于提供的数据，不得臆断/编造数据"（强调基于数据），如"分析必须引用具体数据点"；2）推理结构——要求结构化推理（如"列出数据观察→归纳模式→得出结论，每步标注依据"），让推理过程显式可查（而非直接抛结论）；3）不确定处理——要求"数据不足时说明'数据不足以支持该结论'，不得编造"，防臆断；4）引用——要求引用具体数据（如"数据：销量 Q1=100，Q2=150，增长 50%"），可追溯；5）反例约束——"不要做无数据支撑的推断，如'可能是因为 X'（除非有数据）"，明确禁止。反例："分析销量"（LLM 可能编原因）。原则：数据优先+结构推理+不确定说明+引用+反例约束，让 LLM 基于数据严谨推理。

**Q：客服场景 Prompt 要"流程+边界"，但流程太严会僵化（如所有问题都按 SOP 答得机械），怎么平衡流程和灵活（有温度的客服）？**

流程框架+灵活表达+温度。1）流程框架——核心流程（如"理解问题→查知识库→回答/转人工"）规范执行（保证准确/合规），但框架内表达灵活；2）灵活表达——回答的措辞/语气灵活（如不强制话术，允许自然对话），避免机械；3）温度指引——Prompt 加温度指引（如"礼貌/共情/主动帮助"，如"理解您的困扰，我帮您查一下"），让客服有温度而非冷冰冰；4）个性化——根据用户情绪/问题类型调整（如用户急躁→安抚+快速解决；用户咨询→详细解答），而非千篇一律；5）边界内灵活——流程/边界（如不超范围/合规）是硬约束（必须遵守），表达/温度是软指引（灵活调整），分清硬软。原则：硬约束（流程/边界）不妥协，软指引（表达/温度）灵活，保证准确+有温度。

### 第五层：验证与沉淀

**Q：你怎么衡量场景化 Prompt（vs 通用 Prompt）的效果提升（值不值得场景化）？**

AB 对比。固定场景测试集，对比"场景化 Prompt"vs"通用 Prompt"：1）核心指标——该场景的核心指标（如客服的流程合规率/编程的代码正确率/创作的风格符合度/分析的推理严谨度），场景化应显著高；2）端到端——用户满意度/任务完成率，场景化应提升；3）成本——场景化 Prompt 可能更长（token 贵）+开发成本（写场景化 Prompt 要时间），是否值；4）维护——多场景多 Prompt 的维护成本（每场景一套 Prompt 要维护），是否可接受。综合：核心指标显著升+端到端升+成本可接受+维护可控 = 场景化值得。如果提升小（如+2%）但成本高（多套 Prompt 维护），不值得（用通用+轻量调整）。

**Q：场景化 Prompt 的经验怎么沉淀成团队的 Prompt 库？**

建场景 Prompt 库：1）场景分类——按场景分类（客服/编程/创作/分析/教育等），每场景积累 Prompt；2）Prompt 模板——每场景的 Prompt 模板（核心挑战/结构/示例），新需求按模板写；3）最佳实践——每场景的最佳实践文档（如"客服 Prompt 怎么写流程/边界""编程 Prompt 怎么强调测试"），新人参考；4）示例库——每场景的验证有效 Prompt（带效果数据/适用情况），复用；5）迭代——线上反馈→优化场景 Prompt→验证→更新库，持续改进；6）跨场景复用——某些 Prompt 技巧跨场景（如"边界强调"客服/编程都用），提取通用技巧。这套写入团队 Prompt 库 SOP，让"写场景化 Prompt"从"每人重新研究"变成"按模板+最佳实践+示例库"，高效产出高质量场景 Prompt。
