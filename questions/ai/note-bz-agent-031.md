# Agent 怎么选工具？如何确保意图决策正确、选中合适工具？

## 一、工具选择的流程

```
用户输入 → 意图理解 → 工具匹配 → 决策校验 → 执行
            ↑            ↑          ↑
         LLM理解      语义匹配    合理性检查
         用户要什么   匹配工具描述  防止误选
```

## 二、工具描述的关键

```python
# 好的 vs 坏的工具描述
BAD = {
    "name": "search",
    "description": "搜索功能"  # 太模糊，LLM不知道何时用
}

GOOD = {
    "name": "web_search",
    "description": """搜索互联网获取实时信息。
    使用场景：用户问最新新闻/天气/股价/时事时。
    不使用：用户问常识/历史/定义时（这些模型知道）。
    参数：q(搜索词，用关键词而非整句)""",
    "parameters": {"q": {"type": "string", "description": "搜索关键词"}}
}
# 关键：说清"何时用"、"何时不用"、"参数怎么填"
```

## 三、确保选中正确工具的手段

### 1. 清晰的工具描述（基础）

```python
# 每个工具描述包含：能力 + 触发条件 + 反例
TOOL_PROMPT = """
可用工具：

1. query_order
   功能：查询订单物流状态
   用：用户问"我的订单""到哪了""物流"
   参数：order_id（订单号）

2. refund
   功能：发起退款
   用：用户明确要求退款/退货
   不用：用户只是抱怨，需先确认是否要退款
   参数：order_id, reason
"""
```

### 2. Few-shot 示例

```python
# 给LLM看"什么意图该选什么工具"的例子
EXAMPLES = [
    {"input": "我的快递到哪了", "tool": "query_order", "args": {"order_id": "?"}},
    {"input": "我要退款", "tool": "refund", "args": {"order_id": "?", "reason": "?"}},
    {"input": "你好", "tool": "none", "args": {}},  # 闲聊不调工具
]
```

### 3. 决策校验（防误选）

```python
def validate_tool_choice(choice, user_intent):
    """校验工具选择是否合理"""
    checks = [
        # 工具能力是否匹配意图
        intent_matches_tool(user_intent, choice.tool),
        # 参数是否齐全
        all_required_params_present(choice),
        # 参数值是否合法
        validate_param_values(choice.args),
    ]
    if not all(checks):
        return {"valid": False, "reason": "..."}
    return {"valid": True}
```

### 4. 多工具时：RAG 检索工具

```python
# 工具多时，先检索相关工具子集
def select_tools(user_query, all_tools):
    # 把工具描述向量化
    tool_embeddings = {t: embed(t.description) for t in all_tools}
    query_emb = embed(user_query)
    
    # 检索top-k最相关的工具
    relevant = top_k_by_similarity(query_emb, tool_embeddings, k=5)
    
    # 只把相关的5个工具给LLM选（而非全部30个）
    return relevant
```

## 四、意图理解增强

```python
class IntentAnalyzer:
    def analyze(self, user_message):
        """显式的意图分析，而非让LLM隐式判断"""
        intent = self.llm.analyze(f"""
        分析用户意图：
        消息: {user_message}
        
        返回：
        - primary_intent: 主要意图（查信息/操作/闲聊/投诉）
        - required_action: 需要什么动作（查询/修改/创建/删除）
        - entities: 关键实体（订单号/产品名/时间）
        - confidence: 置信度
        """)
        
        if intent.confidence < 0.6:
            # 低置信度，追问而非乱猜
            return {"action": "clarify", "question": "您是想查询还是...？"}
        return intent
```

## 五、面试加分点

1. **描述质量是基础**：工具选不准，90%是描述没写好——这是最容易被忽略的
2. **RAG 选工具**：工具多时先检索子集再让 LLM 选，而非全塞
3. **校验+追问**：低置信度时主动追问比乱选工具更可靠

## 记忆要点

- 核心结论：工具选不准，90%是因为工具描述（能力+触发+反例）写得模糊不清
- 应对多工具：使用RAG检索机制，先向量召回最相关的Top-K工具子集再给LLM
- 决策校验防误选：执行前必须检查意图匹配度、必填参数是否齐全及合法
- 低置信度处理：显式分析意图，置信度不足时宁可向用户追问，绝不乱猜


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 选工具靠"意图理解+工具匹配+决策校验"，为什么不直接让 LLM 输出"用哪个工具"，LLM 自己理解意图不就行了？**

LLM 直接输出确实能选，但准确率不够（尤其工具多时）。三步拆解的价值在于"提升准确率 + 可干预"：1）意图理解显式化——先明确用户意图（如"用户想查订单状态"），作为后续匹配的 query，比 LLM 隐式理解更可控、可调试；2）工具匹配精准化——意图明确后，用结构化匹配（语义相似度+规则约束）从工具库召回 top-K 候选，比 LLM 从几十个工具描述里直接选更准（LLM 在多选项时准确率下降）；3）决策校验兜底——选完工具后校验"参数是否完整/权限是否允许/是否和意图一致"，不合法的拒绝或重选，防 LLM 误选。三步是"显式化+精准化+兜底"，比 LLM 一步直选准确率高 15-25%。

### 第二层：证据与定位

**Q：Agent 选错了工具（该查天气却调了计算器），怎么定位是意图理解错、工具匹配错、还是校验没拦住？**

逐环节查。1）意图理解——看 LLM 提取的意图是什么（如"用户想算数"还是"查天气"），如果意图就提取错了，是意图理解问题（LLM/prompt）；2）工具匹配——意图正确（查天气）但召回的工具是计算器，是匹配问题（工具描述不清或 embedding 把"天气"匹配到了"计算器"）；3）校验——选了计算器但校验没拦住（校验规则没覆盖"查天气不该用计算器"的情况）。定位方法：trace 每个环节的输出，找第一个错的。常见根因：工具描述写得太相似（如"计算器"和"天气计算"描述混淆）或意图理解 prompt 不清晰。

### 第三层：根因深挖

**Q：意图理解靠 LLM，但用户表达模糊（如"帮我看看那个"），LLM 怎么理解这种模糊意图？**

模糊意图要"澄清"而非"猜"。1）意图置信度——LLM 理解意图时输出置信度（如"查订单 70%、查天气 20%、查日历 10%"），低置信度（最高 <50%）触发澄清；2）澄清提问——"您是想查订单还是查天气？"给用户选项，不盲目猜；3）上下文消歧——结合对话历史消歧（如上文聊了订单，"那个"大概率指订单），用 context 提升理解；4）多轮确认——执行前确认（"我帮您查最近的订单，对吗？"），用户确认才执行。核心原则：宁可多问一句也不盲目猜错（猜错了用户体验更差）。只有高置信度（>80%）才直接执行，否则澄清/确认。

**Q：工具匹配用 embedding 语义相似度（如 query 和工具描述的相似度），但有些工具"功能相似但参数不同"（如"按日期查"和"按订单号查"），相似度区分不开怎么办？**

embedding 相似度的局限是"语义相似不代表功能可替换"。补充手段：1）结构化约束——工具描述加"适用条件"字段（如"本工具仅用于按订单号查询，按日期请用 date_search"），匹配时不仅看相似度还看条件约束；2）参数 schema 校验——用户意图里提取的参数（如"用户提供了订单号 12345"），匹配时校验工具的参数 schema（按订单号查的工具要求 order_id 参数，用户提供了一致），参数不匹配的工具降权；3）rerank——embedding 召回 top-10 后，用 cross-encoder 或 LLM 做 rerank（更精确地判断"功能+参数"双重匹配），选 top-1。单纯 embedding 在"功能相似"场景不够，要加结构化约束+rerank。

### 第四层：方案权衡

**Q：工具选择的三步（意图理解+工具匹配+决策校验），每步都要调 LLM，成本高，能不能合并成一步（LLM 直接选）？**

合并成一步（LLM 直接选）成本低但准确率差。三步拆解的价值在"工具多时准确率显著高"——LLM 从 30 个工具里直接选，准确率约 60-70%（选项多注意力分散）；三步拆解（意图理解→匹配召回 top-5→LLM 从 5 个里选）准确率 85-90%。成本对比：三步多调 2 次 LLM（意图理解+rerank），但每次输入短（不是全部工具描述），单次成本低；一步直选虽只 1 次 LLM 但输入长（全部工具描述），单次成本高。算总账：三步的总 token 成本可能和一步持平甚至更低（单次输入短），且准确率高。所以"看似多步多成本"实际"单步成本降+准确率高"，三步在工具多时更优。工具少（<5 个）时一步直选够用。

**Q：决策校验（选完工具后校验合法性）会不会"过度拦截"——合法的工具调用被校验拦住导致 Agent 卡住？**

校验设计不当会过度拦截。避免方法：1）校验只拦"硬性不合法"——如权限不足（无权调危险工具）、参数格式错（order_id 不是数字）、意图完全不匹配（查天气调删库工具），这些必须拦；2）软约束不拦只告警——如"这个工具可能不是最优选择"（有更好的工具），给 LLM 提示但不强制阻止，让 LLM 自主决定；3）被拦后的恢复——校验拦住后不卡住，而是把"拦截原因"反馈给 LLM（如"权限不足，请换一个工具或转人工"），LLM 据此决策下一步；4）校验日志——记录每次拦截的原因，定期 review 是否过度拦截（拦截率高但实际合法的比例），调校验规则。核心：校验是"安全网"不是"审查局"，只拦真不合法的。

### 第五层：验证与沉淀

**Q：你怎么衡量工具选择的准确率，而不是"感觉选得还行"？**

明确指标。1）工具选择准确率——在标注集（每个 query 标注"应该选哪个工具"）上，Agent 选对的占比，目标 >90%；2）参数填充正确率——选对工具后，参数值正确的占比（如 order_id 填对），目标 >85%；3）误调用率——不该调用工具时调用了（如简单问题强行调工具），应 <10%；4）漏调用率——该调用工具时没调用（如该查却直接编造答案），应 <5%。四个指标从"选对/填对/不误调/不漏调"全面衡量。指标打到 dashboard，每次 prompt/工具描述改动后跑回归，确保不退步。

**Q：工具选择的三步流程怎么沉淀成框架的 ToolSelector 组件？**

封装成组件：1）意图理解器——LLM 提取意图+置信度，低置信触发澄清；2）工具匹配器——embedding 召回 + 结构化约束过滤 + rerank 精选，返回 top-K 候选；3）决策校验器——权限/参数/意图一致性校验，拦截硬性不合法，软约束告警；4）工具描述管理——工具注册时强制要求"功能+场景+参数示例"完整描述，描述质量不达标的拒绝注册；5）准确率监控——自动统计工具选择准确率/误调率/漏调率，异常告警。开发者注册工具+配置权限，框架自动跑选择三步。这套写入团队 Agent 框架 SOP，新 Agent 工具选择开箱即用且准确率有保障。
