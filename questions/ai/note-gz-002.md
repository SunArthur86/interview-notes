## 方向1: 智能客服 RAG

```python
class CarSourceRAG:
    """接入实时车源知识库的客服Agent"""

    def __init__(self):
        self.knowledge_base = VectorDB(collection="car_sources")
        # 实时同步: MySQL车辆数据变更 → CDC → Kafka → 向量化 → 更新

    def answer(self, user_question, car_id=None):
        # 1. 检索车源信息
        car_info = self.knowledge_base.search(
            query=user_question,
            filter={"car_id": car_id} if car_id else None,
            top_k=3
        )

        # 2. 生成回答
        response = llm.chat(
            system="你是瓜子二手车客服，基于车况数据如实回答",
            context=car_info,
            user=user_question
        )
        return response

# 关键指标:
# - P99延迟 < 80ms (BERT-tiny做意图识别 + 缓存)
# - 车源数据延迟 < 5min (CDC近实时同步)
# - 回答准确率 > 90% (定期人工抽检)
```

```python
class PricingAssistant:
    """LLM理解非结构化维修/车况描述，辅助定价"""

    def adjust_price(self, car_description, base_price):
        # 1. LLM提取影响价格的因素
        factors = llm.extract(factors_prompt, car_description)
        """
        输入: "2023年上牌,左前门补漆,天窗正常,右后视镜有划痕"
        输出: [
            {"factor": "左前门补漆", "impact": -1500, "confidence": 0.9},
            {"factor": "右后视镜划痕", "impact": -500, "confidence": 0.8},
        ]
        """

        # 2. 结合传统AI定价模型
        adjusted = base_price
        for f in factors:
            adjusted += f["impact"] * f["confidence"]

        return adjusted

# 价值: 传统定价模型只看结构化数据(年份/里程/品牌)
#       LLM补充了非结构化描述(维修/外观/功能)
#       → 定价偏差率从±8%降到±3%
```

## 数据闭环

```
Step 1: 采集 — 客服对话/用户浏览/交易数据
Step 2: 清洗 — 去PII/去噪/质量校验
Step 3: 标注 — 高质量对话标注为训练数据
Step 4: 训练 — SFT微调/DPO对齐
Step 5: 评估 — AB实验验证模型效果
Step 6: 部署 — 替换线上模型
Step 7: 监控 — 收集新的bad case → 回到Step 1

关键: 闭环不是一次性的，是飞轮效应
     数据越多 → 模型越好 → 体验越好 → 更多数据
```

## 数据分层规范

```
ODS (原始层):   原始对话日志/车辆数据/用户行为
                → 不做处理, 原样存储

DWD (明细层):   清洗/脱敏/标准化
                → 对话文本去噪, 车辆字段标准化
                ⚠️ 必须做维度建模, 保证复用性

DWS (汇总层):   按主题汇总
                → 用户画像汇总, 车辆定价汇总

ADS (应用层):   直接服务业务
                → 推荐特征, 定价因子, 客服FAQ
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：辅助定价你让 LLM 理解"左前门补漆"转成 -2000 元。但定价是高精度场景（差几百元影响成交），LLM 输出有不确定性（同一描述每次可能不同）。为什么敢把定价交给概率性的 LLM，而不是坚持用结构化字段的传统定价模型？**

因为 LLM 不是"替代"传统定价模型，而是"补充"传统模型用不了的非结构化信号。传统定价模型只看结构化字段（年份、里程、品牌），但维修描述（"左前门补漆"）是非结构化文本，传统模型要么忽略它（信息浪费），要么靠人工标注成结构化字段（成本高、滞后）。LLM 的价值是"把非结构化描述实时转成结构化定价因子"，再喂给传统模型或叠加到基础价。不确定性问题通过"置信度加权"控制——LLM 输出每个因子带 confidence（如 0.9），低置信度的因子影响权重小（`impact * confidence`），且定价最终要过传统模型的校验（LLM 的因子作为输入特征，不是直接出最终价）。所以 LLM 是"信号提取层"不是"决策层"，决策仍由确定性模型或人工拍板。

### 第二层：证据与定位

**Q：客服 RAG 你说回答准确率 > 90%（定期人工抽检）。但抽检样本怎么选？如果只抽"容易的"query，90% 是虚高的。你怎么保证这个数字有代表性？**

抽检必须分层随机覆盖不同难度和类型。按 query 类型分层：车况查询（"有事故吗"）、价格查询（"这车能便宜吗"）、流程咨询（"过户怎么办"）、闲聊/超范围（"你叫什么名字"）；每类按线上真实占比随机抽样（不能只抽车况类）。难度分层：简单（单实体）、中等（多实体）、困难（多跳推理或歧义）。抽检员不能看模型答案后再判（有锚定偏差），要独立标注"正确答案"再对比模型输出。还要做"对抗样本"——故意构造容易错的 query（如同名不同款的车、模糊的代词指代）测模型的边界。代表性验证：抽检集的 query 类型分布和线上大盘对比，偏差 < 5% 才算代表。如果只抽容易的，90% 是自欺欺人。

### 第三层：根因深挖

**Q：画像增强你从对话提取"预算10万、偏好日系"。但用户对话里可能说反话（"日系车太脆了不考虑"）、改主意（"本来想要SUV现在觉得轿车好"）、或随口一说（"预算嘛随便"）。LLM 怎么区分真实偏好和噪声？**

单轮对话很难区分，必须靠多轮对话的"交叉验证"和"时间衰减"。机制是：LLM 提取每轮对话的偏好信号并打"强度标签"（明确陈述=高、随口提及=低、否定=反向），多轮信号累积——如果三轮里有两轮明确说"想要SUV"，一轮随口说"轿车也行"，累积后 SUV 偏好占主导。改主意场景用"时间衰减+最新优先"——最新的明确陈述权重最高，旧的偏好衰减。否定场景（"不考虑日系"）LLM 要识别为"排除项"而非"偏好项"。所有提取的偏好带"置信度"和"证据原文"（traceback），下游推荐系统按置信度加权使用，低置信度的偏好不影响排序或只做轻度调整。关键是 LLM 不做"最终判断"，它提供"带置信度的信号"，推荐系统融合多信号决策。

**Q：那如果 LLM 把"不考虑日系"误提取成"偏好日系"（否定理解错），画像被污染会影响所有后续推荐。为什么不在提取后加一层人工校验？**

全量人工校验成本太高（每天百万级对话），不现实。正确做法是"抽样校验 + 异常检测"。抽样：每天抽 0.1%-1% 的提取结果人工复核，统计 LLM 的错误率（如否定理解错占 3%），错误率超阈值（如 5%）告警，说明 Prompt 或模型退化要优化。异常检测：自动校验提取结果的合理性——如"预算 100 万"明显异常（超出正常分布）、"偏好日系又偏好德系"矛盾（需消歧）、"同一用户一周内偏好翻转 5 次"不稳定。异常的提取结果不直接进画像，标为"待确认"或降置信度。否定理解这类高频错误，通过 Prompt 优化（明确"区分肯定和否定陈述，否定要标 exclude"）+ few-shot 示例降低，而不是靠人工逐条校验。

### 第四层：方案权衡

**Q：数据闭环你说是"飞轮效应"——数据越多模型越好。但飞轮的前提是"数据质量高"，低质量数据只会越转越差。你怎么保证回流进训练的数据是高质量的？**

数据回流要设"质量闸门"而非全量回流。一是用户反馈信号：用户点踩的对话、转人工的对话、用户重复提问（说明没答好）的标为负样本，不回流或作为 DPO 的"坏回答"；点赞的、用户追问深入的标为候选正样本。二是人工筛选：候选正样本经人工标注员确认（答案准确、风格合规）才进训练集，不合格的剔除。三是多样性控制：不能只回流某一类高频问题（如车况查询），要覆盖全类型，否则模型偏向高频类型、低频问题变差。四是去重和去污染：相似对话去重（避免重复样本主导），且要把"评测集"从训练集剔除（防数据泄漏，评测虚高）。飞轮的核心不是"量"而是"高质量信号的持续流入"，低质量数据回流是"反向飞轮"——越多越差。

**Q：数据分层你提了 ODS/DWD/DWS/ADS 四层。但 LLM 应用很多是实时性要求（如客服 P99 < 80ms），走四层 ETL 延迟太高。为什么不全走实时链路，还要保留批处理的分层？**

因为实时和批处理解决不同问题，两者要共存（Lambda 架构）。实时链路（CDC → Kafka → 向量库）解决"客服要毫秒级拿到最新车源"——这是低延迟、单条记录的更新。批处理分层（ODS→DWD→DWS→ADS）解决"全局分析和模型训练"——如统计某车型的历史成交均价（要扫全量数据）、构建训练集（要高质量清洗和标注），这些是高吞吐、批量处理。实时链路不适合做复杂分析（ Join 多表、窗口聚合慢），批处理不适合做低延迟更新。所以客服 RAG 走实时（向量库按 car_id 查），定价模型走批处理（DWS 层算好定价因子），两者数据最终一致（实时是增量的，批处理是全量的，T+1 对齐）。全实时的问题是"复杂分析慢且贵"，全批处理的问题是"数据延迟"，Lambda（实时+批处理双链路）是工程权衡。

### 第五层：验证与沉淀

**Q：你怎么证明这一整套大模型应用（客服+画像+定价）对瓜子二手车的业务有实际价值，而不是"为了用 AI 而用 AI"？**

算业务指标的 A/B 对比，不是看技术指标。客服：A/B 测试 RAG 客服 vs 传统规则客服，比"人工转接率"（应下降）、"工单解决时长"（应缩短）、"用户满意度 CSAT"（应上升）。画像增强：A/B 测试 LLM 画像 vs 传统行为画像驱动的推荐，比"推荐点击率 CTR"、"试驾预约转化率"、"最终成交转化率"——文中提到推荐准确率提升 15-25%，要验证这个数字是 A/B 出来的还是感觉。辅助定价：A/B 测试 LLM 辅助定价 vs 纯传统定价，比"定价偏差率"（实际成交价 vs 定价的差距，文中说从 ±8% 降到 ±3%）、"定价到成交的周期"（定价准的卖得快）。关键是 A/B 的样本量要够、要分流量桶（消除季节性波动）、要看长期指标（短期提升可能是新鲜感）。如果业务指标没显著提升，技术再炫也是"为 AI 而 AI"。

**Q：怎么让团队在构建大模型应用时，自觉遵循"数据闭环+分层规范"，而不是各项目各搞各的数据管道？**

把数据规范做成平台能力而非项目级选择。一是统一数据平台：团队共用一套 ODS/DWD/DWS/ADS 数仓，各项目按规范接入，不自己搭管道。二是统一 LLM 应用框架：封装"对话采集→清洗→标注回流→训练评测"的标准 SDK，新项目接入即获得数据闭环能力，而非自己实现。三是数据治理 review：新项目的数据流必须过数据治理团队的 review（分层是否合规、PII 是否脱敏、质量闸门是否设了），不合规不上线。四是监控大盘：按项目看"数据回流质量"（正样本数、人工确认率、评测分数趋势），异常项目告警。让数据闭环是"平台默认提供的能力"，项目方只需用而非建，这样规范的一致性靠平台保障而非靠自觉。
