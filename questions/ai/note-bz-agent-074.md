# 生成参数 Temperature、Top-P、Top-K 在业务中如何调优？

## 一、三个参数的本质

```
LLM生成下一个token时：
  1. 计算所有词的概率分布 P(word1), P(word2), ...
  2. 三个参数控制如何从这个分布中"采样"

┌──────────────┬──────────────────────────────────┐
│ 参数          │ 作用                                │
├──────────────┼──────────────────────────────────┤
│ Temperature  │ 调整概率分布的"陡峭度"              │
│              │ 低→分布尖锐(总选最高概率词=确定)    │
│              │ 高→分布平坦(更可能选低概率词=随机)  │
├──────────────┼──────────────────────────────────┤
│ Top-P        │ 核采样：只从累积概率达P的词中选      │
│              │ P=0.9→只选概率累积前90%的词        │
│              │ 过滤掉长尾低概率词                  │
├──────────────┼──────────────────────────────────┤
│ Top-K        │ 只从概率最高的K个词中选              │
│              │ K=40→只在前40个候选词中采样         │
│              │ 硬性截断                            │
└──────────────┴──────────────────────────────────┘
```

## 二、Temperature 详解

```python
import numpy as np

def softmax_with_temperature(logits, temperature):
    """带温度的softmax"""
    # Temperature调整logits的尺度
    scaled = logits / temperature
    exp = np.exp(scaled - np.max(scaled))
    return exp / exp.sum()

# 效果对比
logits = [5.0, 3.0, 1.0, 0.5]  # 词A/B/C/D的原始logits

# Temperature = 0.1（极低 → 几乎总选A）
# probs ≈ [0.999, 0.001, 0, 0]  → 确定性输出

# Temperature = 1.0（正常）
# probs ≈ [0.87, 0.12, 0.01, 0.01]

# Temperature = 2.0（高 → 更随机）
# probs ≈ [0.50, 0.28, 0.12, 0.10]  → 可能选其他词

# 直觉：
#   低温 = 保守，总选"最安全"的词 → 适合事实任务
#   高温 = 冒险，可能选"意外"的词 → 适合创意任务
```

## 三、业务调优指南

```
┌──────────────────┬──────────┬──────────┬──────────────────┐
│ 任务类型          │ Temp     │ Top-P    │ 说明              │
├──────────────────┼──────────┼──────────┼──────────────────┤
│ 代码生成          │ 0-0.2    │ 0.9      │ 要准确，不能随机   │
│ 数学计算          │ 0        │ 0.9      │ 确定性输出        │
│ 数据提取/分类      │ 0-0.3    │ 0.9      │ 格式稳定          │
│ 事实问答          │ 0.3-0.5  │ 0.9      │ 准确为主          │
│ 翻译             │ 0.3-0.5  │ 0.9      │ 忠于原文          │
│ 对话/客服         │ 0.5-0.7  │ 0.9      │ 自然但有控        │
│ 摘要             │ 0.5-0.7  │ 0.9      │ 适度多样          │
│ 创意写作          │ 0.7-1.0  │ 0.95     │ 鼓励创意          │
│ 头脑风暴          │ 1.0-1.2  │ 0.95     │ 越发散越好        │
└──────────────────┴──────────┴──────────┴──────────────────┘

原则：
  事实/精确任务 → 低温（0-0.3），要稳定
  平衡任务 → 中温（0.5-0.7），自然
  创意任务 → 高温（0.7-1.0），要多样
```

## 四、Top-P vs Top-K

```python
# Top-K: 固定数量截断
# 不管概率分布如何，只保留top-K个词
# 缺点：有时top-5已经覆盖95%，有时top-40还有大量长尾

# Top-P（nucleus sampling）: 动态数量截断
# 保留累积概率达到P的最小词集合
# 优势：自适应——分布集中时少选，分布均匀时多选

# 例：分布 = [0.5, 0.3, 0.1, 0.05, 0.05]
# Top-P=0.8: 选{0.5, 0.3}（累积0.8）→ 只2个词
# Top-K=3: 选{0.5, 0.3, 0.1} → 固定3个

# 实践：Top-P比Top-K更智能，现代多用Top-P
# 两者通常只用一个，不叠加
```

## 五、实际业务调优示例

config = {
    "temperature": 0.6,   # 中低：自然但不发散

## 六、调优方法论

```python
# A/B测试找最优参数
def find_optimal_params(test_cases, param_grid):
    """
    param_grid = {
        "temperature": [0.3, 0.5, 0.7],
        "top_p": [0.85, 0.9, 0.95]
    }
    """
    results = {}
    for temp in param_grid["temperature"]:
        for top_p in param_grid["top_p"]:
            score = evaluate(test_cases, temp=temp, top_p=top_p)
            results[(temp, top_p)] = score
    
    return max(results, key=results.get)

# 评估维度：
# - 准确性（事实任务）
# - 多样性（创意任务）
# - 一致性（多次运行相似度）
# - 用户满意度
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说代码生成用 Temperature=0.1、创意写作用 0.9，这个 0.1 和 0.9 是怎么定出来的？是经验值还是有依据？**

有依据不是拍脑袋。Temperature 的本质是把 logits 除以 T 再 softmax，T 越小分布越尖锐。我定 0.1 的依据是：在代码生成评测集（HumanEval）上跑 T=0/0.1/0.2/0.3，T=0 会有"贪心退化"（模型卡在重复 token，因为完全确定性会把概率接近的两个词反复选），T=0.1 既接近确定又避开贪心陷阱，pass@1 最高。创意 0.9 的依据是：用 distinct-n（输出 token 的去重率）做指标，T<0.7 时多样性饱和（distinct-2 < 0.4），T>1.0 开始出现语义不通的乱跳，0.7-1.0 是"多样且通顺"的甜区。所以数值是评测出来的，不是默认值。

### 第二层：证据与定位

**Q：线上代码生成功能偶尔输出"半截代码"或重复片段，你怎么判断是 Temperature 问题还是 max_tokens 或 Prompt 问题？**

看三个特征。如果是"重复片段"（同一行循环出现），是 Temperature 太低触发贪心退化，调高到 0.2-0.3 或加 frequency_penalty。如果是"半截就停"（输出在中间戛然而止），看 finish_reason——如果是 `length` 就是 max_tokens 不够，调大；如果是 `stop` 就是 Prompt 里的停止词触发太早。如果是"语法错乱"（括号不匹配），是 Temperature 太高采样到非法 token，调低。所以不是一上来调 Temperature，是先看 finish_reason 和输出特征归类再对症。

### 第三层：根因深挖

**Q：Top-P 你说比 Top-K 智能自适应，但既然 Top-P 这么好，为什么很多 API 默认还提供 Top-K？Top-K 在什么场景反而比 Top-P 好？**

Top-K 在"候选词分布本身稳定"的场景比 Top-P 好。Top-P 的自适应是优势也是风险——当分布异常平坦（所有词概率接近，比如模型对下一步很不确定时），Top-P=0.9 会把几十上百个词都纳入候选，反而引入噪声。Top-K=40 在这时硬性限制只在前 40 个，兜底更稳。所以 Top-K 适合"分布可控、要硬护栏"的场景（比如分类任务，输出空间就是那几个类别），Top-P 适合"开放生成、要自适应"的场景（比如写文章）。这也是为什么很多 API 两者都开放——不是冗余，是给不同场景选。

**Q：那为什么不直接 Temperature + Top-P + Top-K 三个全开，多重约束更稳？**

因为三者叠加会过度压缩采样空间，让输出退化。Temperature 已经调了分布形状，Top-P 又截断长尾，Top-K 再硬截断，三重叠加后候选词可能只剩 2-3 个，等价于近乎贪心解码，输出会重复且生硬。实测 Temperature=0.3 + Top-P=0.9 + Top-K=40 的输出多样性比单独 Temperature=0.3 还差。正确用法是 Temperature + 一个截断参数（Top-P 或 Top-K 二选一），留出合理的采样空间。

### 第四层：方案权衡

**Q：客服场景你定 Temperature=0.6，但客服最怕"每次回答不一样"被投诉，为什么不用 0 保证完全一致？**

因为完全一致反而损害体验。Temperature=0 时同一问题模型永远输出同一句话，但用户追问"换种说法"或上下文微变时，模型会僵化地重复原话，显得"傻"。0.6 的意义是：核心事实（话术要点）稳定，但表达自然有变化，更像真人客服。防"不一致投诉"不靠 Temperature=0，靠"事实约束层"——Prompt 里固化必须说的要点（退款政策、时间），允许措辞浮动。所以权衡是：要点层用 Prompt 锁死（确定性），表达层用 0.6 留自然度（多样性）。

**Q：为什么不干脆都用 Temperature=0，反正事实任务都需要确定性？**

因为 Temperature=0 有两个硬伤。一是贪心退化——概率接近的两个 token 会陷入"选 A 还是 B"的循环，输出重复片段（"我我我我觉得"）。二是无法做多样性采样——RAG 的多路召回、多数投票（self-consistency）需要多次采样取多数，Temperature=0 每次都一样，投票失去意义。所以事实任务用 0.1-0.2 而不是 0，既接近确定又避开贪心陷阱，还能在需要 self-consistency 时调高采样。

### 第五层：验证与沉淀

**Q：你 A/B 测试 Temperature，怎么保证结论可信？因为 LLM 本身有随机性，0.6 比 0.5 高 3% 可能只是噪声。**

每个参数组合至少跑 3 次取均值，且做统计显著性检验。具体做法：固定 200 条评测集，每组参数跑 3 次（不同随机种子），算准确率均值和标准差，用配对 t 检验看差异是否 p<0.05。如果 0.6 的均值 87%±1.2%，0.5 是 84%±1.5%，差 3% 但 p=0.08，就不显著、不能下结论说 0.6 更好。只有 p<0.05 才采纳。这样能避免把随机波动误判成参数效果。

**Q：这套参数怎么沉淀让团队复用，而不是每个场景重新调？**

沉淀成"场景参数预设表"。按任务类型（抽取/分类/对话/创作）各定一组推荐参数 + 评测指标 + 调参边界（如 Temperature 范围 0.1-0.5），新人按场景查表直接用。每个预设都带"为什么是这个值"的评测报告（比如"客服=0.6，因为 distinct-2=0.45 且人工评分最高"）。新场景上线先查表用预设，评测不达标才在该范围内微调。这样 80% 场景不用重新调参，省掉重复劳动。
