# 【月之暗面面经】桌面端长任务执行时，前端怎样做通知、回看和继续执行？

## 一、长任务的用户旅程

```
用户发起任务 → 切去做别的事 → 任务进行中 → 任务完成/失败
     ↓                                    ↓
  任务中心记录                        系统通知
                                         ↓
                                   用户点击通知
                                         ↓
                                   回到任务现场
                                         ↓
                              ┌──── 查看产物
                              ├──── 确认/编辑
                              └──── 补充材料重跑
```

## 三、通知系统设计

### 通知分级

| 级别 | 触发条件 | 通知方式 | 示例 |
|------|---------|---------|------|
| 关键 | 任务完成/失败/需确认 | 系统通知 + Dock角标 + 声音 | "站点生成完成，点击查看" |
| 重要 | 产物版本更新/部分完成 | 应用内通知栏 | "表格已生成（1/3）" |
| 普通 | 进度里程碑 | 任务中心进度更新 | "索引完成50%" |
| 静默 | 常规进度更新 | 不通知，仅任务中心可见 | "正在处理第45页..." |

```typescript
// 智能通知策略
class NotificationManager {
  send(taskEvent: TaskEvent) {
    const strategy = this.getStrategy(taskEvent);
    
    switch (strategy.level) {
      case 'critical':
        // 系统级通知（即使应用不在前台也能看到）
        new Notification('Kimi Desktop', {
          body: strategy.message,
          icon: '/icon.png',
          tag: taskEvent.taskId,  // 防止重复通知
        });
        // Dock 角标
        app.dock.setBadge('1');
        break;
        
      case 'important':
        // 应用内通知栏
        this.addToInAppNotifications(strategy.message);
        break;
        
      case 'normal':
        // 只更新任务中心进度
        taskStore.updateProgress(taskEvent.taskId, taskEvent.progress);
        break;
        
      case 'silent':
        // 不做任何主动通知
        break;
    }
  }
  
  private getStrategy(event: TaskEvent): NotificationStrategy {
    if (event.type === 'completed') return { level: 'critical', message: `✓ ${event.taskName}完成` };
    if (event.type === 'failed') return { level: 'critical', message: `✗ ${event.taskName}失败` };
    if (event.type === 'needs-confirmation') return { level: 'critical', message: `⚠️ ${event.taskName}需要确认` };
    if (event.type === 'partial-complete') return { level: 'important', message: `产物 ${event.index}/${event.total} 已生成` };
    if (event.type === 'progress' && event.progress % 25 === 0) return { level: 'normal' };
    return { level: 'silent' };
  }
}
```

async function handleNotificationClick(taskId: string) {
  // 1. 恢复任务对象

    params: { id: taskId },

  });
  
  // 6. 高亮新产物
  nextTick(() => {
    highlightNewArtifacts(artifacts);
  });
}

## 六、常见坑

- **通知塞太多细节**：用户被频繁打扰，最终关闭所有通知
- **点通知回不到现场**：只跳到任务列表，用户还要手动找到对应任务
- **失败后只能整单重跑**：用户只想换个文件继续，却要从头开始
- **没有执行日志**：任务失败后无法定位是哪一步出了问题
