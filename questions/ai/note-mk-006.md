# 【月之暗面面经】桌面端长任务执行时，前端怎样做通知、回看和继续执行？

## 一、长任务的用户旅程

```
用户发起任务 → 切去做别的事 → 任务进行中 → 任务完成/失败
     ↓                                    ↓
  任务中心记录                        系统通知
                                         ↓
                                   用户点击通知
                                         ↓
                                   回到任务现场
                                         ↓
                              ┌──── 查看产物
                              ├──── 确认/编辑
                              └──── 补充材料重跑
```

## 三、通知系统设计

### 通知分级

| 级别 | 触发条件 | 通知方式 | 示例 |
|------|---------|---------|------|
| 关键 | 任务完成/失败/需确认 | 系统通知 + Dock角标 + 声音 | "站点生成完成，点击查看" |
| 重要 | 产物版本更新/部分完成 | 应用内通知栏 | "表格已生成（1/3）" |
| 普通 | 进度里程碑 | 任务中心进度更新 | "索引完成50%" |
| 静默 | 常规进度更新 | 不通知，仅任务中心可见 | "正在处理第45页..." |

```typescript
// 智能通知策略
class NotificationManager {
  send(taskEvent: TaskEvent) {
    const strategy = this.getStrategy(taskEvent);
    
    switch (strategy.level) {
      case 'critical':
        // 系统级通知（即使应用不在前台也能看到）
        new Notification('Kimi Desktop', {
          body: strategy.message,
          icon: '/icon.png',
          tag: taskEvent.taskId,  // 防止重复通知
        });
        // Dock 角标
        app.dock.setBadge('1');
        break;
        
      case 'important':
        // 应用内通知栏
        this.addToInAppNotifications(strategy.message);
        break;
        
      case 'normal':
        // 只更新任务中心进度
        taskStore.updateProgress(taskEvent.taskId, taskEvent.progress);
        break;
        
      case 'silent':
        // 不做任何主动通知
        break;
    }
  }
  
  private getStrategy(event: TaskEvent): NotificationStrategy {
    if (event.type === 'completed') return { level: 'critical', message: `✓ ${event.taskName}完成` };
    if (event.type === 'failed') return { level: 'critical', message: `✗ ${event.taskName}失败` };
    if (event.type === 'needs-confirmation') return { level: 'critical', message: `⚠️ ${event.taskName}需要确认` };
    if (event.type === 'partial-complete') return { level: 'important', message: `产物 ${event.index}/${event.total} 已生成` };
    if (event.type === 'progress' && event.progress % 25 === 0) return { level: 'normal' };
    return { level: 'silent' };
  }
}
```

## 六、常见坑

- **通知塞太多细节**：用户被频繁打扰，最终关闭所有通知
- **点通知回不到现场**：只跳到任务列表，用户还要手动找到对应任务
- **失败后只能整单重跑**：用户只想换个文件继续，却要从头开始
- **没有执行日志**：任务失败后无法定位是哪一步出了问题

## 记忆要点

- 用户体验闭环：发起任务防阻塞、系统级通知防漏看、一键恢复防断点
- 任务中心看板：全局聚合展示进行、待确认、完成、失败四类状态列表
- 设计要求：支持跨页追踪进度，保留任务历史日志，支持一键回看与重跑

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么长任务不能用前端的 loading 动画让用户等着，非要做"发起→离开→通知→回看"的异步闭环？**

因为长任务的耗时（5-30 分钟）远超用户的注意力保持周期（秒级）。如果用 loading 阻塞等待，用户会面临两个选择：盯着 loading 等 10 分钟（不可能），或者切去做别的事——但一旦切走，loading 所在的组件可能被卸载、窗口可能被关闭、App 可能被重启，任务状态全丢。异步闭环的本质是"把任务的生命周期从 UI 交互中解耦"：任务在后台跑（状态存在 TaskStore，不依赖 UI 组件），用户可以自由切换窗口/页面/应用，任务完成时用系统通知召回（即使 App 在后台也能收到），用户点通知回到任务现场时恢复完整上下文（任务对象、产物列表、对话历史、素材引用）。这样用户可以在等待任务 A 的同时发起任务 B，把桌面端从"单任务阻塞"升级为"多任务并发"，这是桌面 AI 产品相对 Web 聊天页的核心体验优势。

### 第二层：证据与定位

**Q：你怎么定位"用户点了通知却回不到任务现场"的根因，是通知链接错还是上下文恢复逻辑错？**

用通知点击漏斗定位。在通知点击链路上埋四个埋点：(1) 通知点击事件（用户点了系统通知）；(2) App 激活事件（App 被拉到前台）；(3) 路由跳转事件（router.push 到 task-detail）；(4) 上下文加载完成事件（task/artifact/contextSlots 全部就绪）。如果 (1) 到 (2) 的转化率低，是系统通知的 deeplink 配置错（App 没被正确唤起）；如果 (2) 到 (3) 的转化率低，是路由跳转逻辑错（taskId 没正确传递或路由守卫拦截）；如果 (3) 到 (4) 的转化率低且耗时长（超过 2 秒），是上下文加载逻辑问题（TaskStore 持久化数据没恢复、异步加载未做 loading 态）。最常见的是第四种：上下文加载了但不完整——产物列表回来了但版本历史丢了、对话消息只加载了最近 10 条，用户感觉"回到了现场但东西少了"，这通常是 TaskStore 持久化字段遗漏导致。

### 第三层：根因深挖

**Q：为什么通知要分级（关键/重要/普通/静默），而不是每个任务事件都通知用户？**

根因是"通知疲劳"——如果每个进度更新（"处理到第 3 页""处理到第 4 页"）都发系统通知，用户会在 5 分钟内收到 20 条通知，第一反应是关闭所有通知权限，之后连任务完成的"关键通知"也收不到。分级的本质是"按用户需要介入的程度排序通知优先级"：关键级（完成/失败/需确认）是用户必须知道的、且需要行动的，用系统通知 + Dock 角标强制提醒；重要级（部分产物完成）是用户可能想知道但不需立即行动的，用应用内通知栏不打扰；普通级（进度里程碑如 25%/50%/75%）是用户主动查看时关心的，只更新任务中心；静默级（常规进度）完全不通知。这样用户每天收到的系统通知从"几十条降到 3-5 条"，每条都是有价值的"需要你做决定"的信号，通知的有效点击率从 5% 提升到 60% 以上。

**Q：那如果用户抱怨"任务失败了我没及时知道"，为什么不把所有失败都升级为关键级系统通知？**

因为"所有失败都关键"等于"没有关键"。失败本身也分级别：整任务崩溃（后端报错、文件丢失）是关键失败，必须立即通知；单步骤失败但可自动重试（如某页 OCR 超时）不是关键失败，自动重试成功后用户根本不需要知道；产物质量低但任务跑完了（自评打分 50 分）是"半失败"，应该用重要级通知并附带"建议重试"。如果把所有失败都升级为关键级，用户会被"第 3 页 OCR 失败，已自动重试""第 5 页解析慢，已重试成功"这类噪音通知淹没，对真正的关键失败（"整个任务崩溃了，你的输入文件格式不对"）脱敏。解法是"失败也要分级"：fatal（整任务终止）→ 关键通知；recoverable（自动重试中）→ 静默，重试成功后不提，重试 N 次仍失败才升级为关键；partial（部分产物质量低）→ 重要级通知 + 降级展示。根因是"通知的价值 = 信息重要性 × 用户行动可能性"，用户无法行动的失败不该用关键级打扰。

### 第四层：方案权衡

**Q：失败恢复你设计成"从失败步骤继续执行"，为什么不简化成"整单重跑"或"完全手动重试"？**

三种方案的权衡：整单重跑最简单（重新发起任务），但浪费已完成的计算——一个 5 步任务在第 4 步失败，前 3 步的结果（如文件解析、索引构建）全部丢弃，重跑要多花 3 步的时间和 Token 成本；完全手动重试（用户重新选文件、重新写指令）最灵活但用户体验最差，用户要重新配置整个任务上下文。从失败步骤继续执行是折中：利用执行日志记录每一步的输入输出（checkpoint），失败时恢复到失败前一步的 checkpoint，只重跑失败步骤。代价是实现复杂——需要每步做 checkpoint（序列化中间状态）、失败时做状态恢复（反序列化并跳到指定步骤）。选这个方案的原因是长任务的重跑成本极高（一个 20 分钟的任务失败在第 18 分钟，整单重跑要再等 20 分钟，断点续跑只要 2 分钟），省下的时间价值远大于实现的工程成本。而且断点续跑让用户敢于尝试（"失败了也能接着跑"），提升了产品试错体验。

**Q：那如果断点续跑的 checkpoint 数据本身不一致（如第 3 步的输出有 bug 导致第 4 步必然失败），为什么不放弃断点续跑直接整单重跑？**

因为"checkpoint 数据有 bug"是可检测的，不该用"放弃断点续跑"来回避。检测方法：断点续跑前先做 checkpoint 完整性校验——检查第 N 步的输出是否满足第 N+1 步的输入 schema（字段齐全、类型正确、值在合理范围）。如果校验失败，提示用户"检测到第 3 步的数据可能有问题，建议从第 1 步重跑"——这是有选择的整单重跑，而非默认整单重跑。另一个保障是"checkpoint 版本化"——每个步骤的 checkpoint 记录生成它的代码版本号，如果任务执行过程中 App 升级了（代码变了），旧 checkpoint 自动失效，强制整单重跑（避免新旧代码的不兼容）。所以根因不是"断点续跑不可靠"，而是"需要配套校验和版本管理"——做了这两点，断点续跑的可靠性就能逼近整单重跑，同时保留 10 倍的速度优势。

### 第五层：验证与沉淀

**Q：你怎么证明"通知分级"比"全量通知"更有效，用户真的更满意吗？**

用三个指标验证：(1) 通知权限保留率 = 保持通知开启的用户数 / 总用户数，全量通知下用户会因为疲劳关闭通知，这个率通常不到 50%；分级通知应能保持 80% 以上。(2) 关键通知点击率 = 用户点击"任务完成"通知的次数 / 该通知曝光次数，全量通知下用户对所有通知脱敏，点击率低于 10%；分级通知应达到 50% 以上（因为每条都是高价值信号）。(3) 任务回看率 = 用户通过通知回到任务详情页的次数 / 完成的任务数，这个率直接反映"异步闭环"是否成立——如果用户完成任务后不回来查看产物，说明通知没有有效召回。A/B 测试两组用户（全量通知 vs 分级通知），跑两周后对比这三个指标，如果分级通知组在"任务回看率"上高出 2 倍以上，就证明分级设计有效。

**Q：怎么让团队在新增长任务类型时，自觉接入任务中心 + 通知分级 + 断点续跑，而不是自己写一套简单的 loading？**

把这三件事做成"长任务基础设施 SDK"而非可选规范。第一，任务执行框架（TaskRunner）统一封装——所有长任务必须实现 TaskHandler 接口（定义 steps/checkpoint/onFailure），通过 TaskRunner.execute() 发起，不允许开发者直接用 setTimeout/setInterval 写异步逻辑；第二，TaskRunner 自动注册到任务中心、自动按策略发通知、自动做 checkpoint，开发者只需实现业务逻辑（每步做什么），基础设施由框架提供；第三，通知策略通过声明式配置（taskManifest 里写 notification: { onComplete: critical, onProgress: silent }），不允许在代码里手动调 new Notification()——所有通知必须走 NotificationManager 统一分发；第四，断点续跑通过 TaskHandler 的 checkpoint() 钩子自动实现——开发者在每步结束时调 checkpoint(state)，框架负责持久化和恢复，开发者不写恢复逻辑。这样"接入基础设施"就从"自觉遵循规范"变成了"用 SDK 就自动获得"，不接入就只能用同步阻塞（产品体验极差）。
