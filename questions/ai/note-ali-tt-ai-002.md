# 【阿里淘天AI二面】Multi-Agent是如何共享上下文的？子Agent会使用父Agent的所有Context吗？

> 来源：阿里巴巴淘天淘工厂 AI应用开发 二面面经（小红书）

## 一、Multi-Agent 两种基础范式

```
                    Multi-Agent 架构
                         │
            ┌────────────┴────────────┐
            ▼                         ▼
    ┌───────────────┐        ┌───────────────┐
    │  层级式        │        │  协作式        │
    │ (Hierarchical)│        │(Collaborative)│
    │ SubAgents     │        │ Agent Team    │
    └───────┬───────┘        └───────┬───────┘
            │                        │
  子Agent之间：无通信        子Agent之间：有通信
  适合：子任务相互独立        适合：子任务彼此依赖
```

### 1. 层级式（Hierarchical / SubAgents）

```
        ┌─────────────┐
        │  父Agent     │  任务拆解 + 调度
        │ (Orchestrator│
        └──────┬──────┘
               │ 分配子任务
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌───────┐ ┌───────┐ ┌───────┐
│子Agent│ │子Agent│ │子Agent│  各自独立执行
│   A   │ │   B   │ │   C   │  互相不通信
└───┬───┘ └───┬───┘ └───┬───┘
    │         │         │
    └─────────┼─────────┘
              ▼
        ┌───────────┐
        │  父Agent   │  结果汇总
        └───────────┘

特点：子Agent A 不知道子Agent B 的存在
      每个 子Agent 只与父Agent通信
```

### 2. 协作式（Collaborative / Agent Team）

```
    ┌─────────────┐
    │   Planner   │  全局规划
    │   Agent     │
    └──────┬──────┘
           │ 分发计划
    ┌──────┼──────────┐
    ▼      ▼          ▼
┌───────┐    ║     ┌───────┐
│Executor│←──╫──→│Executor│  执行者之间
│   1   │    ║     │   2   │  可以直接通信
└───┬───┘    ║     └───┬───┘
    │        ║         │
    └────────╫─────────┘
             ║
         结果写回
         共享状态

特点：Executor 1 和 Executor 2 之间可以直接交换信息
      适合子任务之间有数据依赖的场景
```

### 两种范式的关键区别

| 维度 | 层级式 (SubAgents) | 协作式 (Agent Team) |
|------|-------------------|-------------------|
| **子Agent间通信** | 无 | 有 |
| **调度方式** | 父Agent集中调度 | 分布式协作 |
| **适合场景** | 子任务相互独立 | 子任务彼此依赖 |
| **复杂度** | 低，易于实现 | 高，需要协调机制 |
| **典型框架** | AutoGen, CrewAI | LangGraph, AutoGen |

## 二、上下文共享的两种方式

### 方式一：直接通信

```
Agent A ────消息──────────→ Agent B
        ←───回复──────────

实现方式：
  1. RPC/HTTP：Agent暴露REST/gRPC接口
  2. WebSocket：实时双向通信
  3. 暴露为Tool：Agent将自身能力包装成MCP工具，其他Agent通过工具调用
  4. Agent-to-Agent Protocol：标准化的A2A通信协议

代码示例（Tool方式）：
  // Agent B 暴露的能力
  @tool
  def search_product(query: str) -> str:
      """搜索商品信息"""
      return product_db.search(query)

  // Agent A 可以调用Agent B的工具
  agent_a.invoke({
      "tools": [search_product],  // Agent B的能力
      "task": "帮用户找到合适的手机"
  })
```

### 方式二：间接通信

```
Agent A ──写入──→ ┌──────────────┐ ──监听──→ Agent B
                   │  共享状态     │
                   │ (文件/数据库) │
                   └──────────────┘

实现方式：
  1. 文件系统：Agent读写共享文件（如Claude Code的inbox方案）
  2. 数据库：Agent写入数据库表，其他Agent查询
  3. 消息队列：通过Kafka/Redis Pub-Sub中转
  4. 共享内存：共享变量/状态存储（如Redis Hash）
```

## 三、Claude Code 的 Multi-Agent 实现

Claude Code 使用**文件系统作为消息队列**，这是一个非常优雅的间接通信方案：

```
~/.claude/teams/
├── config.json              # 成员列表（所有Agent的信息）
└── inboxes/                 # 每个Agent一个inbox文件
    ├── agent-alpha.json     # Agent Alpha的收件箱
    ├── agent-beta.json      # Agent Beta的收件箱
    └── agent-gamma.json     # Agent Gamma的收件箱

消息格式（inbox JSON中的每条记录）：
{
  "from": "agent-alpha",
  "text": "我已经完成了数据分析，结果在 /tmp/result.json",
  "timestamp": "2024-01-15T10:30:00Z",
  "read": false
}

工作流程：
  1. Agent要给谁发消息 → 往对方的inbox JSON追加一条记录
  2. Agent执行任务时 → 不检查inbox（专注当前任务）
  3. Agent当前turn（推理+工具调用+返回）完成后 → 检查inbox
  4. 发现新消息 → 处理消息 → 发送回复
```

## 四、子Agent会使用父Agent的全部Context吗？

**答案：不会。** 子Agent只接收与其任务相关的精简上下文。

### 为什么不全量传递？

```
❌ 原因1：上下文窗口有限
   父Agent上下文：50K tokens（包含完整对话历史）
   子Agent任务：只需要2K tokens的上下文
   → 全量传递 = 浪费48K tokens + 可能超限

❌ 原因2：信息过载降低推理质量
   父Agent上下文可能包含大量与子任务无关的信息
   → 无关信息干扰子Agent的推理（注意力分散）

❌ 原因3：安全边界
   父Agent可能拥有敏感信息（API keys, 用户隐私数据）
   → 子Agent不应获得超出其职责的信息

✅ 正确做法：最小必要上下文原则
   子Agent只获得：
   - 任务描述（要做什么）
   - 必要的上下文（做这件事需要知道什么）
   - 结果约束（输出格式/限制）
```

### 上下文传递模型

```
父Agent上下文（完整）：
┌────────────────────────────────────────────┐
│ 用户完整对话历史                              │
│ 之前所有工具调用的结果                         │
│ 中间推理过程                                  │
│ 系统提示和约束                                │
│ ────────────────────────────────             │
│  提取任务相关部分 ↓                           │
│  ┌──────────────────────────┐               │
│  │ 任务描述："查询用户123的订单"│               │
│  │ 约束："返回JSON格式"       │               │
│  │ 必要数据：用户ID=123       │               │
│  └──────────────────────────┘               │
│        │                                    │
│        ▼ 只传递这部分                         │
└────────┼────────────────────────────────────┘
         │
         ▼
子Agent上下文（精简）：
┌──────────────────────────┐
│ 任务描述："查询用户123的订单"│
│ 约束："返回JSON格式"       │
│ 必要数据：用户ID=123       │
│ Agent系统提示              │
│ 可用工具列表               │
└──────────┬───────────────┘
           │
           ▼ 执行完成后
┌──────────────────────────┐
│ 结果：{"orders": [...]}   │ ← 只返回结果给父Agent
└──────────────────────────┘
```

## 五、面试加分点

1. **提到A2A Protocol标准**：Google提出的Agent-to-Agent通信标准，类似MCP但面向Agent间通信
2. **提到上下文压缩技术**：父Agent在传递上下文给子Agent前，可以做summarization/context window compression
3. **提到错误传播问题**：层级式中，一个子Agent的错误会影响父Agent的决策，需要重试机制
4. **提到成本控制**：每个子Agent都是一次LLM调用，Multi-Agent会显著增加token消耗
5. **提到Claude Code的真实实现细节**：文件系统inbox方案是一个低成本、高可靠的工程实现，适合展示对工业界Agent系统的深入理解

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Multi-Agent 为什么要做上下文隔离，让子 Agent 看到父 Agent 的全部 Context 不是更省事、信息更全吗？**

省事但会爆炸和污染。父 Agent 的 Context 包含所有历史对话、中间推理、工具返回，可能已经 10k+ token，全量传给子 Agent 既浪费 token 又稀释信号——子 Agent 只需要和自己任务相关的片段。更危险的是污染：父 Agent 的某些上下文（如其他子 Agent 的中间结果）可能误导当前子 Agent 的决策。隔离的目的是"最小权限原则"，子 Agent 只拿到完成自己任务所需的最小上下文，精度高且成本低。

### 第二层：证据与定位

**Q：子 Agent 的输出质量差，你怎么判断是上下文传少了（信息不足）还是传多了（被污染）？**

看两类证据：一是子 Agent 的推理过程（Thought 链）是否在"找信息"——如果反复说"我没有 X 信息"，是传少了；如果推理被无关信息带偏（如父 Agent 上下文里有其他任务的细节，子 Agent 错误参考了），是传多了。二是消融实验：分别测传"全量上下文"和"精简上下文（只传任务描述+相关片段）"的子 Agent 成功率，如果精简版更高，证明传多了有污染。

### 第三层：根因深挖

**Q：父 Agent 给子 Agent 传上下文，你用"全量 copy"还是"摘要+指针"？摘要会丢信息吗？**

用摘要 + 指针。全量 copy 的问题是 token 爆炸——3 个子 Agent 各拿一份父 Agent 的 10k token 上下文，总开销 30k+。摘要（让 LLM 把父上下文压成 500 token）能控成本，但确实可能丢信息——摘要模型可能漏掉子 Agent 需要的关键细节。治本是"结构化摘要"：不是让 LLM 自由摘要，而是按子 Agent 的任务类型提取固定字段（如子任务是"查订单"就传订单相关上下文），且保留原始上下文的指针（如文件路径或 trace_id），子 Agent 需要时能回查全量。

**Q：那为什么不直接用共享黑板（blackboard）模式，所有 Agent 读写同一个 Context 存储，省得传来传去？**

黑板模式（所有 Agent 共享一个可读写的 context store）的问题是并发污染和一致性。多个子 Agent 并发写黑板，可能互相覆盖或引入冲突（A 写了 X，B 读到 X 被误导）。且黑板是"全可见"的，违反最小权限——每个 Agent 都能看到其他 Agent 的中间状态，容易被无关信息干扰。工程上更稳的是 Orchestrator 模式：中心调度器维护全局状态，按需把相关片段注入各子 Agent，子 Agent 之间不直接共享，通过 Orchestrator 中转。Claude Code 的文件系统 inbox 就是这个思路——用文件做通信中介，天然持久且解耦。

### 第四层：方案权衡

**Q：上下文传递你选 JSON payload 还是文件 inbox（Claude Code 做法）？为什么？**

看任务的生命周期和并发度。JSON payload 适合短生命周期、同步调用的子 Agent（调一次拿结果），简单直接但 payload 大时有大小限制（如 MCP 消息大小）。文件 inbox 适合长生命周期、异步、需要持久化的场景——父 Agent 把上下文写文件，子 Agent 读文件，结果也写文件，天然支持异步、断点续传、崩溃恢复。Claude Code 用文件 inbox 是因为代码任务往往长且需要可中断/可恢复。选型本质是"任务是否需要持久化和异步"。

**Q：为什么不直接用 gRPC/消息队列做 Agent 间通信，非要搞上下文传递？**

Agent 间通信（gRPC/MQ）传的是"消息"（结构化数据），不是"上下文"（LLM 的 prompt 语义）。Agent 调用另一个 Agent 的本质是"让另一个 LLM 用它的推理能力处理某个子任务"，必须把任务背景（上下文）传给它，而不是传 RPC 参数。gRPC 适合传确定性的函数调用（输入参数 → 输出结果），Agent 间需要的是"模糊的任务委托 + LLM 推理"，性质不同。当然，底层传输可以用 gRPC/MQ，但 payload 里装的是上下文 + 任务描述，不是 RPC 参数。

### 第五层：验证与沉淀

**Q：你怎么证明 Multi-Agent 的上下文隔离方案比"全量共享"更好？**

定义三个指标：总 token 成本（隔离应显著低于全量）、子 Agent 任务成功率（隔离应不低于全量，理想情况更高因减少污染）、端到端任务完成时间（隔离的摘要/传递有额外开销，但并发度可能更高）。在固定任务集上对比两组方案，如果隔离组成本降 50%+ 且成功率持平或更高，证明隔离有效。重点看成功率——如果隔离后成功率反降，说明摘要丢了关键信息，要调摘要策略。

**Q：Multi-Agent 的上下文管理怎么沉淀成框架能力？**

封装成 Orchestrator 框架：支持上下文模板（按子任务类型定义该传哪些字段）、自动摘要（全量上下文 → 精简上下文，可配摘要模型）、上下文持久化（文件/DB 存储，支持回查和恢复）。沉淀"各子任务类型的上下文模板""摘要模型的选型与质量评估""并发隔离的锁机制"，新 Multi-Agent 系统接入时按模板配置，不重复设计。
