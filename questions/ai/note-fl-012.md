# 【字节飞连面经】缓存穿透 / 击穿 / 雪崩：含义和解法，布隆过滤器

## 一、三大问题对比

| 概念 | 含义 | 解法 |
|------|------|------|
| **穿透** | 查一个**不存在**的 key，每次都打到 DB | 缓存空值（短 TTL）+ **布隆过滤器**前置判断 |
| **击穿** | 一个**热点 key 过期**瞬间大量并发打到 DB | **互斥锁**（只让一个线程回源）/ 永不过期 + 异步更新 |
| **雪崩** | **大量 key 同时过期** + 突发流量打到 DB | **过期时间加随机扰动** / 多级缓存 / 限流降级 |

## 二、穿透：缓存空值 + 布隆过滤器

```
查询 user:999（不存在）
  │
  ├─[无布隆过滤器] Redis miss → DB miss → 返回 null
  │                                  ↑ 每次都打 DB（被恶意攻击就挂）
  │
  └─[有布隆过滤器] 布隆过滤器先判断 → 说不存在 → 直接返回（不查 DB）
                                         说存在 → 可能误判 → 查 Redis/DB
```

**缓存空值**：DB miss 后把 `user:999 → null` 缓存（短 TTL，如 60s），下次直接命中。

**布隆过滤器**：
- 用多个 hash 函数把 key 映射到 bit 数组
- **说不存在一定不存在**（100% 准确）
- **说存在可能误判**（false positive）
- 用来挡掉绝大部分"查不存在 key"的穿透

## 三、击穿：互斥锁防并发回源

```
热点 key 过期瞬间，1000 个请求同时来
  │
  ├─[无锁] 1000 个请求都 Redis miss → 1000 个都打 DB → DB 挂
  │
  └─[互斥锁] 
       第 1 个请求拿到锁 → 回源 DB → 写回 Redis → 释放锁
       其余 999 个请求等锁 → 锁释放后读 Redis（命中）
```

**互斥锁实现**：`SETNX lock:key 1 EX 10`（拿锁，10s 过期防死锁）。

**另一解**：永不过期 + 异步更新（后台定时刷新，热点 key 永远在缓存）。

## 四、雪崩：过期时间加随机扰动

```
[问题] 大量 key 同时过期（如批量预热的缓存同一时刻 TTL 到）
       → 突发流量全部打 DB → DB 雪崩

[解法] 过期时间 = base + random(0, jitter)
       TTL = 3600 + rand(0, 600)   # 1 小时 ± 10 分钟随机
       → 打散过期时间点
```

**其他解法**：
- 多级缓存（本地缓存 Caffeine + Redis + DB）
- 限流降级（Hystrix / Sentinel 保护 DB）

## 五、布隆过滤器详解

| 特性 | 说明 |
|------|------|
| 空间 | 极省（1 亿 key 约 100MB） |
| 查询 | O(k)，k=hash 函数个数 |
| **说"不存在"** | **100% 准确** |
| **说"存在"** | **可能误判**（false positive） |
| 删除 | 标准布隆过滤器**不能删**（bit 不能复位） |

**误判率控制**：增加 bit 数组大小 + 增加 hash 函数个数。典型配置 1 亿 key 误判率 1% 约 114MB。

**Counting Bloom Filter**：用计数器代替 bit，支持删除（但空间翻几倍）。

## 六、加分点

- 说出 **布隆过滤器不能删**，要删得用 Counting Bloom Filter 或 Cuckoo Filter
- 说出 **互斥锁要设过期时间**（防持锁线程崩溃死锁）
- 说出 **雪崩 vs 击穿的区别**：击穿是单 key，雪崩是多 key

## 七、扩展

- **缓存预热**：服务启动时主动加载热点 key，避免冷启动雪崩
- **多级缓存一致性**：本地缓存（Caffeine）+ 分布式缓存（Redis），更新时本地缓存怎么失效（广播失效 / TTL 短）
- **Redis 集群的雪崩**：不仅是 key 过期，主节点宕机切换期间也会"雪崩"——需要熔断降级保护

## 记忆要点

- 穿透(查不存在)解法：缓存空值短TTL，配合布隆过滤器前置拦截挡掉恶意请求
- 击穿(热Key过期)解法：互斥锁(SETNX)只让单线程回源查DB，或永不过期加异步更新
- 雪崩(大量同时过期)解法：过期时间加随机扰动打散，配合多级缓存与限流降级
- 布隆过滤器特性：说不存在100%准，说存在有误判，且标准版不支持删除元素

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：穿透你用"缓存空值 + 布隆过滤器"两层。缓存空值已经能挡掉重复查同一个不存在 key 的请求了，为什么还要加布隆过滤器？**

因为缓存空值有盲区。空值缓存的前提是"这个不存在的 key 被查过一次"——第一次查时仍会打到 DB。如果攻击者用大量随机不存在的 key 轮询（每次都不同的不存在 ID），每个 key 都是"第一次查"，缓存空值挡不住（每次都 miss 到 DB）。布隆过滤器解决了"第一次查就挡"的问题——它预先装载了所有存在的 key，新请求先问布隆过滤器，不存在直接返回，根本不碰 Redis/DB。所以两者分工：布隆过滤器挡"首次查不存在"（攻击防御），缓存空值挡"重复查同一不存在"（节省布隆过滤器的查询开销）。高安全场景两者都要，低风险场景缓存空值可能够。

### 第二层：证据与定位

**Q：线上 DB 突然慢查询飙升，你怎么确认是缓存击穿/穿透/雪崩中的哪一个，而不是单纯流量涨了？**

看 Redis 和 DB 的指标对比。先看 Redis 命中率——如果命中率从 95% 突降到 60%，确认是缓存失效导致 DB 穿透。再细分原因：看 Redis 的 evicted_keys 和 expired_keys 增速。如果是某个热点 key 的过期时间点（能从 key 的 TTL 历史或创建时间推算）和 DB 慢查询飙升时间吻合，是击穿（单 key 失效）。如果是大量 key 在同一时间段集中过期（比如整点批量预热的缓存都设了相同 TTL），是雪崩（多 key 齐失效）。如果命中率降但 expired_keys 没明显波动，且请求里有大量"查不存在 key"的模式（如 user_id 连续自增疑似遍历），是穿透。还要排除纯流量增长——如果 Redis 和 DB 的 QPS 同比涨、命中率没降，是流量问题不是缓存失效。

### 第三层：根因深挖

**Q：击穿你用互斥锁（SETNX）只让一个线程回源。但互斥锁会让其他 999 个请求等待，如果回源的那个请求很慢（DB 慢查询 5 秒），这 999 个请求全卡住 5 秒，用户体验更差。为什么不直接让所有请求都回源，至少不阻塞？**

因为"999 个请求等 5 秒"比"999 个请求打挂 DB 然后全部超时"好。如果不加锁，999 个请求同时打到 DB，DB 本来就慢（5 秒），叠加 999 倍并发直接打满连接池，最终所有请求超时（可能 30 秒后失败），DB 可能直接 OOM 或锁表。互斥锁的代价是"等待"，但等待是有上限的（锁的超时时间 + 一次回源时间），且保住了 DB 不被压垮。优化体验的做法是：锁等待设超时（如 2 秒），等不到锁的请求降级返回（旧数据或"稍后重试"），而不是无限等。或者用"双重检查"——等锁的请求拿到锁后先查一次缓存（可能已被第一个回源请求填上了），命中就直接返回。本质是用"可控的等待 + 降级"换"DB 不挂"，这是系统稳定性的权衡。

**Q：那如果不想让用户等，为什么不用"永不过期 + 异步更新"方案？热点 key 永远在缓存里，不就没有击穿了吗？**

"永不过期"听上去完美但有代价。一是数据陈旧——热点 key 永远在缓存，但底层数据变了（如库存被买了），用户看到的是旧库存，业务可能不允许。二是异步更新的复杂性——要后台定时刷新每个热点 key，刷新逻辑出错（如 DB 查询失败）时缓存一直是旧的，且要管理刷新任务的调度和失败重试。三是内存占用——永不过期的 key 永远占内存，热点 key 多了内存压力大。所以"永不过期"适合"数据陈旧可容忍"的场景（如商品详情页的浏览数），不适合"数据实时性要求高"的场景（如库存、价格）。实时性要求高的场景仍用互斥锁 + 短 TTL（如 30s），接受击穿的瞬时等待但保证数据较新。没有银弹，看业务对"陈旧 vs 等待"的容忍度。

### 第四层：方案权衡

**Q：雪崩你用"过期时间加随机扰动"。但扰动只能打散过期时间，如果根本原因是流量突增（不是 key 同时过期），这个解法没用。你怎么区分和应对？**

雪崩的根因有两类：一是 key 同时过期（时间维度），二是流量突增压垮缓存层导致缓存 miss 暴涨（流量维度）。过期时间随机扰动只解第一类。第二类的解法是限流降级——在 DB 前加 Sentinel/Hystrix，流量超阈值时直接拒绝或返回降级响应，保住 DB 不挂。区分方法是看"缓存层是否健康"：如果 Redis 本身 QPS/延迟正常但 DB 被打高，是 key 过期导致的穿透（第一类）；如果 Redis 也被打高（QPS 飙升、延迟升高），是流量突增（第二类）。实际雪崩常是两者叠加（流量高峰撞上批量过期），所以防御要组合：过期时间打散（解时间）+ 限流降级（解流量）+ 多级缓存（本地缓存兜底 Redis 失效）。单一手段都不够。

**Q：布隆过滤器你说"说不存在 100% 准，说存在可能误判"。那如果误判率 1%，1 亿次查询里有 100 万次误判打到 DB，这不算穿透吗？为什么还说布隆过滤器解决了穿透？**

因为布隆过滤器解决的是"绝大多数不存在的 key 被挡掉"，不是"100% 挡掉"。没有布隆过滤器时，1 亿次查询里如果有 2000 万次查不存在的 key，全部打到 DB；加了布隆过滤器（误判率 1%），2000 万次里 1980 万次被准确挡掉，只有 20 万次（1% 误判）漏到 DB。DB 压力从 2000 万次降到 20 万次，是 100 倍缓解。这是"降低风险"不是"消除风险"。如果业务要求 0 穿透（如金融防欺诈），布隆过滤器不够，要用精确集合（如 Redis Set 或 DB 索引），但代价是内存或查询开销大。布隆过滤器是"概率防御"，适合 99% 场景的性价比之选，误判率可通过增大 bit 数组调到 0.1% 或更低。

### 第五层：验证与沉淀

**Q：你怎么证明布隆过滤器的误判率确实是配置的 1%，而不是实际更高？**

实测验证。准备一份测试数据集：N 个"存在的 key"（用于初始化布隆过滤器）和 M 个"不存在的 key"（用于测误判）。初始化布隆过滤器后，查询 M 个不存在 key，统计布隆过滤器"误报存在"的数量，误判率 = 误报数 / M。如果实测误判率显著高于配置（如配 1% 实测 5%），可能原因：实际 key 数量超过预期容量（布隆过滤器是按预设容量算 bit 数的，超容量误判率飙升）、hash 函数配置错、或 key 分布不均。纠正方法是重新按实际容量估算 bit 数组大小并重建。线上监控：持续统计"布隆过滤器说存在但 DB 查不存在"的比例（需要 DB 层配合记录），这是实际误判率的线上观测值，超阈值告警重建。

**Q：怎么让团队在缓存设计上不踩这三大坑，而不是每次出事才补？**

把三大问题的防御做进缓存框架和 review checklist。一是框架层封装：团队统一缓存 SDK，内置"所有写缓存自动加随机 TTL 扰动"（防雪崩）、"热点 key 自动加互斥锁回源"（防击穿）、"可选启用布隆过滤器前置"（防穿透），让正确用法是默认用法。二是 review checklist：新增缓存 key 必须回答"这个 key 失效会击穿吗？批量过期会雪崩吗？查不存在会穿透吗？"，reviewer 逐项确认。三是压测验证：上线前用压测工具模拟"热点 key 失效"和"批量 key 过期"场景，确认 DB 不被打挂。四是监控告警：缓存命中率、DB QPS、慢查询三指标联动监控，命中率突降自动告警。让防御靠框架和流程，不靠"记得加随机 TTL"这种口头提醒。
