# 【字节飞连面经】Redis 过期删除策略 + 内存淘汰策略：Key 过期会立刻删吗？

## 一、Key 过期会立刻删吗？不会

Redis 用 **惰性删除 + 定期删除** 组合：

```
惰性删除：
  下次访问这个 key 时才检查 → 过期就删
  优点：CPU 友好（不主动扫描）
  缺点：过期但永不访问的 key 会占内存（靠定期删除兜底）

定期删除：
  每 100ms 抽样若干个有 TTL 的 key 检查
  过期的删掉
  如果过期 key 比例超过 25%，继续抽样（自适应）
```

**为什么不全量扫描**：百万级 key 全扫一次会阻塞 Redis（单线程）。抽样 + 自适应是性能和准确性的折中。

## 二、内存淘汰策略 8 种（maxmemory-policy）

| 策略 | 范围 | 算法 |
|------|------|------|
| `noeviction` | - | 默认，满了直接报错（写操作失败） |
| `allkeys-lru` | 所有 key | 最久未使用 |
| `allkeys-lfu` | 所有 key | 最少使用频率 |
| `allkeys-random` | 所有 key | 随机 |
| `volatile-lru` | 设了 TTL 的 key | 最久未使用 |
| `volatile-lfu` | 设了 TTL 的 key | 最少使用频率 |
| `volatile-random` | 设了 TTL 的 key | 随机 |
| `volatile-ttl` | 设了 TTL 的 key | TTL 最短（最快过期） |

**volatile-*** 只淘汰设了 TTL 的 key（保护持久数据）；**allkeys-*** 淘汰所有。

## 三、Agent 会话状态选什么

**选 `allkeys-lru` 或 `volatile-lru`**：

- 会话状态本身就设了 TTL（30 min）
- 会话本身有时效性——最久没碰的会话最可能已经结束
- LRU 自然把"僵尸会话"踢掉，保留活跃会话

**为什么不用 LFU**：会话状态访问频率不是"价值"信号（一个刚启动的会话访问少不代表不重要）。

## 四、LRU 是严格 LRU 吗？不是，是近似

Redis 的 LRU 是**近似 LRU**：
- 不维护全局双向链表（内存开销大）
- 淘汰时**采样 N 个 key**（`maxmemory-samples`，默认 5），从 N 个里选最久未用的
- N 越大越接近严格 LRU，但 CPU 开销越大

**Redis 4.0 引入 LFU**：按访问频率淘汰，更适合"热点数据"场景。LFU 有衰减机制（长期不访问频率会降），避免历史热点永远占内存。

## 五、加分点

- 说出 **noeviction 是默认**，但生产环境几乎没人用（满了就写失败）
- 说出 **volatile-ttl** 适合"缓存"场景（最快过期的先踢，延长其他缓存寿命）

## 六、扩展

- **持久化与过期的交互**：RDB 快照不存已过期的 key；AOF 写入时会检查过期
- **主从复制的过期**：从节点不主动删过期 key（等主节点 DEL 命令同步），导致从节点可能读到"过期但未删"的数据——读到后惰性删除
- **Redis 7.0** 的多线程 I/O 对过期策略无影响（过期逻辑仍在主线程）

## 记忆要点

- 过期不立刻删：组合使用惰性删除(访问才查)与定期删除(100ms抽样)，避免阻塞主线程
- 8大淘汰策略：noeviction默认报错，allkeys-*管所有键，volatile-*只踢设了TTL的键
- 淘汰算法看场景：LRU淘汰最久未用(保活跃)，LFU淘汰频次低(保热点)
- 淘汰非严格LRU，而是采样近似(maxmemory-samples默认取5个选最老)

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：惰性删除 + 定期删除的组合，看起来定期删除已经能清过期 key 了，为什么还要惰性删除兜底？反过来呢？**

因为定期删除是抽样的（每 100ms 抽一部分），不是全量扫描，所以一定有"过期但没被抽样到"的 key 漏网——这些 key 靠惰性删除在下次访问时清掉。反过来，惰性删除的问题是"过期但永不访问的 key"永远不会触发清理（比如用户发完就走再不回来的会话状态），这些 key 靠定期删除兜底抽样清。两者是互补关系：定期删除是"主动但有遗漏"，惰性删除是"被动但精准"。缺任何一个都会导致过期 key 堆积占内存。组合的本质是"高频低成本的抽样 + 按需精准的访问检查"，覆盖彼此盲区。

### 第二层：证据与定位

**Q：线上 Redis 内存使用率到了 90%，但你看 `INFO keyspace` 发现没有明显的大 key。你怎么确认是淘汰策略没生效，还是过期 key 没清干净？**

分流查。先看 `INFO stats` 的 evicted_keys（被淘汰的累计 key 数）和 expired_keys（被过期删除的累计 key 数）。如果 evicted_keys 增长缓慢但内存高，说明淘汰策略配置可能是 noeviction（不淘汰只报错），查 `CONFIG GET maxmemory-policy` 确认。如果 expired_keys 增长缓慢但有过期 key 堆积，说明定期删除跟不上（抽样比例低或 TTL key 太多），调 `hz` 参数（后台任务频率，默认 10，调高到 100 能加速定期删除）。再用 `redis-cli --bigkeys` 或 `MEMORY USAGE` 抽查是否有隐藏大 key。还可以 `DBSIZE` 对比 `INFO keyspace` 的 keys with expires 数，判断是"过期 key 没清"还是"有效 key 太多"。定位逻辑是先看淘汰/过期的计数器是否在动，再查配置和参数。

### 第三层：根因深挖

**Q：Agent 会话状态你选 LRU 而不是 LFU，理由是"会话访问频率不是价值信号"。但高频访问的活跃会话用 LFU 不是更能保护吗？为什么 LRU 更合适？**

因为会话状态的"价值"是"最近是否活跃"，不是"累计访问频率"。LRU 保护的是"最近被访问的"——一个用户 5 分钟前还在对话的会话，LRU 不会淘汰它，这符合预期。LFU 保护的是"累计访问多的"——一个已经结束的会话如果历史上交互了 100 轮（高频），LFU 不会淘汰它，但它实际已经没价值了，这是误保。会话场景的访问模式是"近期密集 + 随时间衰减"（用户活跃期高频，结束后骤降），LRU 的"最近访问"语义匹配这个模式，LFU 的"累计频率"会被历史高频误导。唯一适合 LFU 的场景是"长期热点"（如全局配置缓存），会话状态不是。

**Q：那如果用 volatile-lru 而不是 allkeys-lru，万一内存满了但所有 TTL key 加起来还不够，Redis 会怎么办？**

volatile-lru 只在设了 TTL 的 key 里淘汰，如果这些 key 都被淘汰完了内存还不够（说明大量没设 TTL 的持久 key 占了内存），Redis 不会动那些持久 key，而是开始报错（写操作返回 OOM，相当于退化为 noeviction 的行为）。这就是 volatile-* 策略的风险——它"保护持久数据"的代价是：持久数据如果失控增长，缓存 key（有 TTL 的）会被全淘汰且写仍可能失败。解法是监控：设 TTL key 的占比告警，如果占比降到 < 20% 说明持久 key 太多挤占了缓存空间，要排查持久 key 是否该设 TTL。生产环境如果没把握区分，用 allkeys-lru 更安全（至少不会 OOM 报错，最坏是淘汰了不该淘汰的持久 key，但这比写失败好排查）。

### 第四层：方案权衡

**Q：Redis 的 LRU 是近似（采样 N 个选最老），不是严格 LRU。为什么不实现严格 LRU？采样近似的代价是什么？**

严格 LRU 要维护一个全局双向链表，每次访问都把 key 移到链表头（O(1) 但要改数据结构），淘汰时取链表尾。代价是每个 key 多存两个指针（prev/next，64 位系统 16 字节），百万 key 就是 16MB 额外内存；且每次访问都要更新链表位置，在 Redis 单线程模型下是 CPU 开销。近似 LRU 用采样（默认采 5 个）避免了全局链表，内存省、CPU 省，代价是淘汰的 key 不一定是"全局最久未用的"，只是"采样这 5 个里最久的"——可能误淘汰一个其实没那么老的 key。实测中 `maxmemory-samples=10` 时近似 LRU 的表现已经接近严格 LRU（命中率差距 < 1%），所以这个权衡在大部分场景是值得的。只在对命中率极敏感的场景才需要调大 samples（代价是 CPU）。

**Q：noeviction 是默认策略，但你生产几乎没人用。为什么不把默认改成 allkeys-lru？让用户少踩坑？**

因为 noeviction 是"最安全"的默认——它不会偷偷删数据。Redis 作为缓存用时 allkeys-lru 合理，但 Redis 也常被用作"数据存储"（如会话状态、分布式锁），这时偷偷淘汰一个锁 key 会导致并发问题，noeviction 的"满了报错"反而让问题暴露而非隐藏。把默认设成 allkeys-lru 会让"把 Redis 当存储用"的用户在不知情下丢数据，这是更危险的默认。所以 Redis 保守地选 noeviction，强制用户显式声明"我知道缓存场景，允许淘汰"（设成 allkeys-lru）。这是"危险操作要显式 opt-in"的设计哲学，类似数据库默认不开启 DROP 权限。生产部署的第一件事就是把策略改成匹配场景的，这是运维常识。

### 第五层：验证与沉淀

**Q：你怎么证明当前淘汰策略（如 allkeys-lru）的命中率是 acceptable 的，而不是在频繁淘汰热数据导致缓存雪崩？**

核心指标是缓存命中率。`INFO stats` 里的 keyspace_hits / (keyspace_hits + keyspace_misses) = 命中率。生产 Redis 命中率应 > 95%，如果 < 80% 说明在频繁淘汰有用数据（可能是内存太小或策略不匹配）。再细看：evicted_keys 的增速，如果每秒淘汰数百 key 且命中率低，确认是淘汰问题。验证策略是否合适的 A/B 方法：切到 LFU 或调大 maxmemory-samples（5→10），观察命中率是否提升——提升说明原策略/参数不优，无变化说明是内存本身就不够（加内存或拆分）。还要看业务影响——缓存 miss 多是否导致 DB 压力上升（DB QPS 监控），如果 DB 被打高，淘汰策略的代价传导到了下游，必须优化。

**Q：怎么让团队在 Redis 内存治理上不踩坑（比如有人不设 TTL 导致 key 堆积）？**

把治理做进规范和工具。一是编码规范：所有写入 Redis 的 key 必须设 TTL（除非显式标注"持久 key"且有 review），CI 用 `redis-cli --bigkeys` 定期扫描，无 TTL 的 key 列表告警。二是容量规划：每个业务的 Redis 内存配额，超配额自动淘汰或告警，防止单业务挤占全局。三是监控大盘：内存使用率、命中率、淘汰速率、过期 key 清理速率四指标按实例展示，异常自动告警（如命中率突降 10%、淘汰速率突增）。四是定期审计：每月跑一次 key 分析（key 数量、TTL 分布、大 key），治理"僵尸 key"（长期无访问的）和"大 key"（单 key > 10MB）。让内存治理是持续运营而非出事才查。
