# 为什么做了 SFT 还要做 DPO？

## SFT 的天花板

```
SFT训练目标: P(标准答案 | 输入) → 最大

问题: 标准答案只有一个，但同一个问题有多种回答方式
  - 有些回答更安全
  - 有些回答更详细
  - 有些回答更简洁
  - 有些回答更有帮助

SFT无法表达"回答A比回答B更好"这种偏好信息
→ 需要DPO/RLHF来学习偏好
```

## SFT vs DPO vs RLHF 对比

```
┌────────────────────────────────────────────────┐
│                                                │
│  SFT (监督微调)                                 │
│  数据: (输入, 标准答案)                          │
│  目标: 模仿标准答案                              │
│  类比: 抄写课文                                 │
│  能力: ✅ 学会格式和基础能力                      │
│        ❌ 不知道什么回答更好                      │
│                                                │
│  RLHF (人类反馈强化学习)                         │
│  数据: (输入, 多个回答) → 人工排序 → 训练RM      │
│  流程: SFT → 训练Reward Model → PPO优化         │
│  类比: 老师打分 → 学生根据打分改进                │
│  能力: ✅ 学会人类偏好                           │
│        ❌ 训练复杂、不稳定(PPO超参敏感)            │
│                                                │
│  DPO (直接偏好优化)                              │
│  数据: (输入, 好答案, 坏答案) 偏好对              │
│  流程: SFT → 直接用偏好对优化(无需RM)            │
│  类比: 红蓝对比 → 直接告诉哪个好                 │
│  能力: ✅ 学会偏好                               │
│        ✅ 训练简单(类似SFT)、稳定                │
│                                                │
└────────────────────────────────────────────────┘
```

## DPO 原理

### 偏好对数据

```python
preference_data = {
    "prompt": "如何学习机器学习？",
    "chosen": "建议从基础数学(线代/概率)开始，然后看Andrew Ng的课程...", # 有条理
    "rejected": "随便看看书就行了，不用太认真。" # 敷衍
}
```

### DPO 目标函数

```
L_DPO = -E[log σ(β · (log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x)))]

解读:
  π: 当前模型 (训练中)
  π_ref: SFT模型 (固定参考)
  y_w: 好答案 (chosen)
  y_l: 坏答案 (rejected)
  β: 控制偏离参考模型的程度

直觉:
  让 π(好答案) / π_ref(好答案) > π(坏答案) / π_ref(坏答案)
  即: 相比SFT模型，当前模型更倾向于输出好答案
```

### 为什么 DPO 不需要 Reward Model

```
RLHF流程: 数据 → Reward Model → PPO优化 → Policy
  问题: RM可能过拟合, PPO训练不稳定

DPO的数学推导:
  RL的目标 = 最大化 reward - KL(π, π_ref)
  → 最优解: π*(y|x) = π_ref(y|x) * exp(r(x,y)) / Z(x)
  → 反解: r(x,y) = β * log(π*(y|x) / π_ref(y|x)) + const
  → 代入偏好概率: P(y_w > y_l) = σ(r(y_w) - r(y_l))
  → 最终: 不需要显式r，直接用log-ratio!
```

## 什么时候 DPO 会让模型变差

```
⚠️ DPO的已知问题:

1. 过度优化(Over-optimization)
   → DPO训练太多步 → 模型变得过于保守
   → 所有回答趋于模板化 → 多样性下降

2. 分布外偏好(Distribution Mismatch)
   → 偏好数据来自GPT-4的输出
   → 但训练的是7B小模型
   → 小模型模仿不了GPT-4的风格 → 训练信号无效

3. 在线vs离线
   → 标准DPO是离线的(用固定数据集)
   → 模型不探索新的好答案
   → 改进: Online DPO / Iterative DPO

4. 简单被拒绝的答案
   → 如果rejected答案太差(如乱码)
   → 模型学到的是"不要乱码"而非"要更好"
   → 偏好对质量 > 数量
```

## 工业界实践

```
典型训练流程:
  1. 预训练 (Pre-training): 海量文本, ~1T tokens
  2. SFT (Supervised Fine-Tuning): 5万-50万条高质量对话
  3. DPO (Direct Preference Optimization): 1万-10万偏好对
  4. (可选) RLHF/PPO: 更精细的在线优化

DPO数据量经验:
  - 1万对: 可见效果
  - 5万对: 明显提升
  - 10万对: 收益递减
  - 质量远比数量重要
```

## 记忆要点

- SFT天花板：只学模仿标准答案，无法表达A比B更好的偏好信息
- DPO目标：利用(好答案,坏答案)偏好对，类似SFT般稳定地学偏好
- 为何不用RLHF：DPO通过数学转化去除了Reward Model，训练更简单稳定
- 避坑指南：防止过度优化导致多样性下降，坏样本不能太差且要注意分布偏移


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：做了 SFT 为什么还要做 DPO？SFT 难道没教会模型"回答好"吗？**

SFT 教的是"模仿标注答案"，但"好"和"更好"它分不清。SFT 用"(prompt, 标准答案)"训练，模型学到"生成标准答案"，但标准答案是唯一的（或随机的），SFT 无法告诉模型"在多个合法答案里哪个更优"。如用户问"解释递归"，答案 A 简洁准确、答案 B 啰嗦但也对，SFT 只能教"生成某个答案"，DPO 用偏好对（A > B）教"选更好的 A"。动机是"SFT 有优化天花板——它不知道什么是更优"。

### 第二层：证据与定位

**Q：DPO 训练后模型在某些任务上变好了，但另一些任务变差了（对齐税），怎么定位？**

看 DPO 的训练数据。1) 如果 DPO 数据集中在某类任务（如对话），其他任务（如推理）的偏好对少，DPO 会让模型"偏移向对话优化"，牺牲推理；2) 如果 DPO 数据里有"偏好对选错"（A 实际不如 B 但标成 A>B），会教坏模型。区分方法：按任务类型统计 DPO 前后的准确率变化，定位哪些任务退化。检查退化任务的偏好数据质量。解法：补充退化任务的偏好对、用多任务平衡的 DPO 数据。

### 第三层：根因深挖

**Q：DPO 比 SFT 提升，但提升幅度有限（如准确率 +3%），根因是偏好数据不够还是 SFT 已经很强？**

两者都有。1) SFT 强——如果 SFT 已经把模型训练到接近"标注答案的最优"，DPO 的提升空间小（边际递减）；2) 偏好数据不够——如果 DPO 数据少（< 10000 对）或质量差，学不到"更优"的模式。诊断：看 DPO 的 reward margin（偏好对的 reward 差），如果 margin 小（模型对 A 和 B 打分接近），是数据质量问题（A 和 B 差异不够明显）。解法：1) 用更高质量的偏好对（人工标注或用强模型生成）；2) 接受 SFT 已强，DPO 是锦上添花。

**Q：那为什么不直接用 RLHF（更通用的偏好优化），而用 DPO？**

RLHF 要训练奖励模型（RM），额外成本和复杂度。DPO 直接用偏好对优化策略，绕过 RM，简单稳定。RLHF 的优势是"能建模复杂的 reward"（如多目标加权），适合需要精细 reward shaping 的场景。DPO 适合"偏好明确"的场景（A > B 直接给）。在多数对齐任务里，DPO 的效果接近 RLHF 且更简单，所以成为主流。RLHF 在需要"在线探索"（如提升推理能力）时仍有优势（DPO 是离线的）。

### 第四层：方案权衡

**Q：SFT + DPO vs SFT + RLHF，怎么选？**

权衡"复杂度 vs 灵活性"。1) DPO——简单（不用训 RM）、稳定（直接偏好优化）、适合偏好明确的场景（如有大量人工偏好对）；2) RLHF——复杂（要训 RM）、灵活（能建模复杂 reward）、适合需要在线探索的场景（如提升数学推理）。经验上：通用对齐（有用、无害）用 DPO 够；能力提升（推理、规划）用 RLHF/GRPO 更有效（能探索新策略）。DeepSeek R1 的推理能力是 GRPO 训出来的，DPO 做不到这种探索。

**Q：为什么不直接用 DPO 从头训练（不做 SFT），避免两个阶段？**

DPO 需要一个"有基础能力"的起点。DPO 是在 SFT 模型基础上做"偏好微调"（用 SFT 模型作为 reference），如果从预训练模型直接 DPO，模型还没有"按指令回答"的基础能力，DPO 的偏好信号无法有效学习（模型连合法答案都生成不了，谈不上选优）。所以顺序是：预训练（学语言）→ SFT（学指令跟随）→ DPO（学选更优）。每阶段以前一阶段为基础，不能跳过。

### 第五层：验证与沉淀

**Q：怎么衡量 DPO 训练的效果（相比 SFT 的增量）？**

三个维度：1) 偏好准确率——DPO 后的模型在 hold-out 偏好对上，是否把 A 排在 B 前面（应该 > 70%）；2) 任务准确率——在目标任务（如 MT-Bench、Arena）上的提升（相比 SFT 应该 +3-10%）；3) 通用能力保持——在 MMLU 等通用 benchmark 上不退化（避免对齐税）。如果偏好准确率高但任务没提升，可能是偏好数据和任务不匹配。沉淀为 DPO 训练规范：偏好数据构建（人工标注/强模型生成）、reference 模型选择、学习率、beta 参数。

**展开框架：**
1. **SFT局限** — 只有一个标准答案，无法学习偏好
