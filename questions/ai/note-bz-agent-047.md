# RAG 可以从哪些细节深挖？

# 检索用小块（精准），返回用大块（上下文全）
class ParentChildChunking:
    def chunk(self, doc):
        parent_chunks = split(doc, size=2000)    # 大块
        for parent in parent_chunks:
            child_chunks = split(parent, size=200)  # 小块
            for child in child_chunks:
                child.metadata["parent"] = parent  # 关联父块
        # 检索小块，返回时取父块（上下文更全）

## 三、Layer 2：检索层深挖

### 查询改写

```python
class QueryRewriter:
    def rewrite(self, query):
        # 1. 口语→规范
        # "咋装" → "如何安装"
        
        # 2. 扩展同义词
        # "Agent" → "Agent 智能体 autonomous"
        
        # 3. HyDE（假设答案检索）
        hypothetical_answer = llm.generate(f"假设答案: {query}")
        # 用假设答案（而非原问题）去检索
        # 因为答案的语义更接近文档
        
        # 4. 多查询（Multi-Query）
        queries = llm.generate_variants(query, n=3)
        # 生成3个不同角度的查询，分别检索后合并
```

### 混合检索

```python
class HybridRetriever:
    """向量检索 + 关键词检索 融合"""
    
    def retrieve(self, query):
        # 向量检索（语义）
        dense = self.vector_db.search(embed(query), k=10)
        
        # BM25检索（关键词）
        sparse = self.bm25.search(query, k=10)
        
        # 融合（RRF: Reciprocal Rank Fusion）
        fused = rrf_merge(dense, sparse)
        return fused[:10]
    # 向量擅长语义匹配，BM25擅长精确关键词，互补
```

### Rerank（重排序）

```python
# 检索召回多（top-20），Rerank精选（top-5）
class Reranker:
    def rerank(self, query, docs):
        # Cross-Encoder比向量相似度更准（但慢）
        scores = self.cross_encoder.score(
            [(query, doc.content) for doc in docs]
        )
        ranked = sorted(zip(docs, scores), key=lambda x: -x[1])
        return [d for d, s in ranked[:5]]
```

### 闭环迭代

```
评估闭环：
  线上case → 评估打分 → 发现bad case → 分析原因 → 优化 → 回归测试
  
常见bad case原因：
  - 检索没找到：优化分块/查询改写/混合检索
  - 找到了但没用：优化Rerank/Prompt
  - 用了但答错：优化生成Prompt/换更强LLM
```

## 六、优化优先级

```
ROI从高到低：
  1. Rerank（加个重排序，效果立竿见影）
  2. 查询改写（低成本高收益）
  3. 混合检索（向量+BM25）
  4. 分块优化（父子分块）
  5. 数据清洗（去噪/去重）
  6. Prompt优化
  7. 换更强Embedding模型
  8. 元数据过滤

建议：先做1-3（投入小收益大），再按评估结果针对性优化
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RAG 可以从"数据/检索/生成/评估"四层深挖优化，为什么不一次性把所有优化都堆上（分块+改写+rerank+微调全用），反而要分层？**

因为优化的"边际收益和成本"不同，盲目全堆会浪费且可能相互干扰。1）边际收益递减——简单优化（如调 K/加 rerank）收益大成本低，应先做；复杂优化（如微调 embedding/训练 rerank）收益增量小成本高，等简单优化做满再做；2）相互干扰——多个优化叠加可能冲突（如激进的 query 改写 + 微调 embedding，两者都改语义对齐，叠加可能过拟合），分层逐步加（加一个评估一次，确认有效再加下一个）能定位每个优化的贡献；3）问题驱动——不同 RAG 系统瓶颈不同（有的召回差、有的生成差），按瓶颈优化（哪层差优化哪层）而非全堆。所以分层深挖是"按需优化、逐步验证"，而非"全堆"。

### 第二层：证据与定位

**Q：RAG 系统整体准确率 70%，想提升但不知从哪层优化，怎么定位瓶颈在数据/检索/生成哪层？**

用分层指标定位。1）检索层——Recall@K（golden 文档在 top-K 比例），如果低（如 60%），瓶颈在检索（没找到对的文档）；2）生成层——给定检索结果（含 golden），LLM 答案准确率（context fidelity），如果检索 Recall 高但生成准确率低，瓶颈在生成（找到了但没用好）；3）数据层——如果 Recall 低，进一步看是文档质量问题（如文档本身不全/分块碎了 golden）还是检索问题（embedding 不准），对比"理想分块"和"当前分块"的 Recall 差异。定位方法：逐层测指标，找第一个不达标的层。常见模式：Recall 低→检索/数据层；Recall 高但答案差→生成层。

### 第三层：根因深挖

**Q：数据层的"分块"深挖，分块优化能显著提升召回，但怎么判断当前的分块是不是瓶颈？**

对比测试。1）Recall 对比——用当前分块跑 Recall@K，再用"理想分块"（人工标注的语义完整块）跑 Recall，如果当前远低于理想（如 60% vs 85%），分块是瓶颈（块切得不好导致召回差）；2）块质量分析——抽查检索失败的 case，看 golden 文档的块是否被切碎（关键信息分散在多块，单块相似度低）或块过大（块内信息杂，相似度被稀释），是则分块不合理；3）分块策略 AB——试不同分块（固定大小 vs 语义 vs 结构化），对比 Recall，差异大说明分块敏感（优化空间大）。优化方向：语义切分（保留完整语义）、重叠切分（避免切断关键信息）、结构化切分（利用文档层级）。

**Q：生成层的 prompt 深挖，"基于检索结果回答"的 prompt 写法不同效果差很多，怎么写好生成 prompt？**

prompt 要"约束 + 引导"。1）约束——明确要求"答案必须基于检索结果，不得使用检索结果外的知识"（防幻觉），加"如果检索结果不足以回答，说'信息不足'"（防编造）；2）引用——要求"答案中标注引用来源"（如 [文档1]），强制 LLM 基于具体文档，且便于用户核实；3）结构引导——给输出结构（如"先直接回答，再展开说明，最后列引用"），避免 LLM 啰嗦或跑题；4）检索结果格式——检索结果用清晰格式（如 [文档1] 标题: ... 内容: ...），让 LLM 易区分和引用；5）few-shot——给几个"基于检索结果正确回答"的示例，示范预期行为。反例：只说"根据以下文档回答"太弱（LLM 可能忽略或幻觉）。验证：AB 不同 prompt，测答案准确率/幻觉率/引用率。

### 第四层：方案权衡

**Q：检索层的 rerank 深挖，可以用现成 rerank 模型（如 bge-reranker）或微调 rerank，怎么选？**

按"领域特异性和准确率要求"选。1）现成模型——通用 rerank（如 bge-reranker-large），适合"通用场景/快速上线"，零训练成本，但领域特异性差（如医疗/法律术语可能排不准）；2）微调 rerank——用业务数据（query-文档-相关性标签）微调 rerank，适合"领域特异/准确率要求高"，准确率高（针对业务优化）但需标注数据和训练成本。决策：通用场景用现成（快），领域特异用微调（准）。折中：现成 rerank + 业务 few-shot（给 rerank 一些业务示例），介于两者之间。还要看数据量——标注数据少（<1000）不值得微调（过拟合），用现成+示例；数据多（>10000）微调收益大。

**Q：评估层的深挖，RAG 评估用 RAGAS（自动化指标），但 RAGAS 是 LLM 评判（本身可能不准），怎么保证评估可信？**

LLM 评判 + 人工校准。1）RAGAS 指标——faithfulness（答案是否忠于检索结果）、answer relevancy（答案是否切题）、context relevancy（检索结果是否相关），自动化跑大规模评估；2）人工校准——从 RAGAS 评估结果中抽样（如 10%），人工复核"RAGAS 判定对不对"，计算 RAGAS 和人工的一致率，一致率高（如 >85%）则 RAGAS 可信，低则 RAGAS 不准（要换评判模型/调 prompt）；3）多评判模型——用多个 LLM 评判（如 GPT-4 + Claude），取一致结果（降低单模型偏差）；4）关键 case 人工——高影响 case（如线上 bad case）必人工评判，不只依赖 RAGAS。原则：RAGAS 做大规模初筛，人工做关键校准，两者结合。

### 第五层：验证与沉淀

**Q：你怎么证明某一层的优化（如加 rerank）真的有效（而非"加了但没提升"）？**

AB 对比。固定评估集，对比"优化前"和"优化后"：1）该层指标——如加 rerank 后，看检索质量指标（MRR/NDCG）是否提升；2）端到端指标——看答案准确率/满意度是否提升（如果该层提升但端到端没提升，说明该层不是瓶颈或优化被其他层抵消）；3）成本——优化引入的成本（如 rerank 增加 latency/调用成本）是否可接受。最优：该层指标提升 + 端到端提升 + 成本可接受 = 有效。如果该层提升但端到端没提升，说明瓶颈在其他层（该层已不是瓶颈），应转优化其他层。这就是分层优化的价值——每层独立评估贡献，不盲目优化。

**Q：RAG 四层深挖的优化经验怎么沉淀成团队的知识？**

建优化手册 + 组件库：1）优化手册——记录每层优化的方法（如分块的语义切分法、query 改写的扩展法、rerank 的两阶段法、prompt 的约束法），含适用场景/成本/收益（基于实战数据）；2）优化 checklist——按层列出"常见问题→排查方法→优化手段"（如召回差：查 Recall→分块/embedding/query 改写），新人按 checklist 定位；3）组件库——把验证有效的优化做成可插拔组件（如多种 rerank/分块器/query 改写器），新 RAG 系统直接用；4）案例库——记录真实优化案例（如"召回 60%→85% 的优化路径：先加混合检索+10%，再调分块+15%"），用实战经验指导。这套写入团队 RAG 平台 SOP，让"优化 RAG"从"个人摸索"变成"按手册+用组件"。
