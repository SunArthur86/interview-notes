# 【神州专车面经】介绍做过最复杂的数据项目，从需求到落地怎么做？

## 一、STAR-B 叙述结构

```
[S] Situation 背景（30秒）
    什么业务、为什么要做、上下游是谁
    ↓
[T] Task 任务（融入背景）
    你负责哪部分、目标是什么
    ↓
[A] Action 方案（90秒，核心）
    方案 A vs 方案 B 的对比
    为什么选 A（技术选型理由）
    ↓
[R] Result 结果（30秒）
    上线了什么 + 量化指标
    （QPS、准确率、人力节省%、延迟降低%）
    ↓
[B] Building 反思（30秒）
    现在回头看哪里可以更好
```

## 二、难点怎么讲（关键）

**必须列 2 个真正卡过你的点**，不要列"工期紧""需求变"这种废话。

好的难点例子：
- "数据倾斜导致任务跑了 6 小时，定位到是某个 key 数据量是其他的 100 倍"
- "实时和离线指标对不上，差 3%，排查一周发现是去重逻辑在两个链路实现不一致"

坏的难点例子：
- "工期紧"（废话）
- "需求频繁变更"（甩锅）
- "跨部门沟通难"（软技能不算技术难点）

## 三、方案：A vs B 对比

```
方案 A：用 Flink 实时计算
  优：延迟低（秒级）
  劣：运维复杂、成本高

方案 B：用 Spark 批处理 + 1 小时调度
  优：稳定、便宜
  劣：延迟高（1小时）

最终选 A，因为业务要求分钟级延迟，且团队有 Flink 运维能力
```

**为什么必须对比**：证明你不是"随便选了个会用的"，而是权衡过的。

## 四、结果：必须量化

| 差的表述 | 好的表述 |
|---------|---------|
| "提升很多" | "查询延迟从 30s 降到 3s，提升 90%" |
| "效果不错" | "准确率从 85% 提升到 92%，人工审核量减少 60%" |
| "节省了人力" | "每周节省 20 人时，相当于 0.5 个 HC" |

**没数字 = 没做**。如果你记不清具体数字，说个合理区间（"大概从 X 提升到 Y"）也比"提升很多"强。

## 五、反思：证明成长性

- "现在回头看，当时用 XX 方案会更好，因为..."
- "如果重做，我会先做 XX 验证，避免后期返工"
- "这个项目让我意识到 XX 的重要性，后来在新项目里我..."

反思证明你不是"做完就忘"，而是"会沉淀经验"。

## 六、雷区

- ❌ 把"我跟着 mentor 做了 xxx"挂嘴边 → 立刻被追问"你独立完成了什么"
- ❌ 把架构图背出来但讲不出为什么这么拆 → 下一题被打穿
- ❌ 指标没数字（"提升很多"=没提升）
- ❌ 难点列"工期紧"（废话，显得没真东西可讲）

## 七、扩展

- **怎么讲独立贡献**（如果跟 mentor 做的）：聚焦"我负责的模块"，主动说"mentor 负责 XX，我独立负责 YY，YY 的设计/实现/排错都是我做的"
- **量化指标说不出来**：面试前必须回溯项目，把关键指标查清楚或合理估算
- **面试官抓细节"你写的哪个函数"**：诚实说具体函数名 + 它解决什么问题 + 为什么这么实现；如果记不清，说"具体函数名记不清了，但逻辑是 XX"
