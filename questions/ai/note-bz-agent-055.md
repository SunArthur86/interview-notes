# 文档的加载和分割（Chunking）怎么做？

## 一、为什么需要分块

```
原始文档可能很长（一本手册100页=50000字）

不分块的问题：
  1. 整篇文档作为一个向量 → 一个向量表示太多内容（语义稀释）
  2. 检索到整篇 → 超出LLM上下文窗口
  3. 不相关的内容也带进去 → 噪音

分块的好处：
  ✓ 每块语义聚焦（一个主题）
  ✓ 检索精准（找到最相关的段落）
  ✓ 控制上下文大小（top-k块不超token）
```

## 二、分块策略

### 策略 1：固定大小分块（最简单）

```python
from langchain.text_splitter import CharacterTextSplitter

splitter = CharacterTextSplitter(
    chunk_size=500,      # 每块500字符
    chunk_overlap=50,    # 相邻块重叠50字符
    separator="\n\n"     # 尽量在段落边界切
)
chunks = splitter.split_text(long_document)
```

### 策略 2：递归分块（推荐）

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "，", " ", ""]  # 按优先级递归切
)
# 先尝试按段落切，太大则按句子，再大则按字符
# 尽量在自然边界切，保留语义
```

### 策略 3：语义分块

```python
# 用Embedding检测语义边界
class SemanticChunker:
    def chunk(self, text):
        sentences = split_sentences(text)
        
        # 计算相邻句子的语义相似度
        for i in range(len(sentences) - 1):
            sim = cosine(embed(sentences[i]), embed(sentences[i+1]))
            if sim < threshold:  # 语义跳变点
                # 这里是分块边界
                start_new_chunk(i + 1)
# 在语义自然断点处分块，最尊重语义
```

### 策略 5：父子分块（进阶）

```python
class ParentChildChunker:
    """检索小块，返回大块"""
    def chunk(self, doc):
        # 先切大块（父）：2000字符
        parents = split(doc, size=2000)
        
        for parent in parents:
            # 再切小块（子）：200字符
            children = split(parent, size=200)
            for child in children:
                # 子块建索引（用于检索）
                child.metadata["parent_id"] = parent.id
                self.index.add(child)
            # 父块存储（用于返回）
            self.parent_store[parent.id] = parent
    
    def retrieve(self, query):
        # 检索子块（精准）
        child_hits = self.index.search(query, k=5)
        # 返回父块（上下文全）
        parent_ids = {c.metadata["parent_id"] for c in child_hits}
        return [self.parent_store[pid] for pid in parent_ids]
```

## 五、分块质量的影响

```
分块好坏对RAG效果影响巨大：

好的分块：
  块1: "Agent的规划能力指任务分解"     ← 一个主题
  块2: "Agent的记忆分为短期和长期"      ← 一个主题
  → 查"规划"精准命中块1

坏的分块（固定500字符硬切）：
  块1: "Agent的规划能力指...记忆分为短期"  ← 两个主题混在一起
  → 查"规划"命中了，但块里一半是"记忆"，噪音大
```

## 六、面试加分点

1. **强调"粒度匹配"**：没有最优块大小，要看查询需求——事实用小块，分析用大块
2. **父子分块**：检索小块精准+返回大块上下文全，这是进阶技巧
3. **按文档类型分**：代码/对话/表格各有适合的分块方式，不能一刀切

## 记忆要点

- 不分块痛点：整篇向量化导致语义稀释，检索带入大量噪音且超LLM上下文。
- 策略对比：固定分块简单但易切碎语义，递归分块找自然边界最常用。
- 语义分块：用Embedding检测语义跳变点切分，最准但计算成本高。
- 父子分块：子块（200字）建索引精准匹配，父块（2000字）返回保上下文。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：文档分块（Chunking）要权衡"太大（检索不准/超 token）vs 太小（丢失上下文）"，为什么不直接把整篇文档作为一个向量（不分块）检索？**

因为整篇文档检索效果差。1）检索不准——整篇文档 embedding 是"全文语义的平均"，query 只和文档某部分相关时，全文 embedding 相似度被稀释（如 query"手机的电池"vs 整篇手机评测文档，全文 embedding 涵盖屏幕/性能/电池等，相似度被稀释，可能排不进 top-K）；2）超 token——整篇文档（如几千到几万字）直接给 LLM 超其 context（早期 LLM 4K-8K context），即使现代 LLM（128K）也费 token 且 attention 稀释；3）分块解决——分块让每块聚焦单一主题（embedding 准，检索精准），块大小可控（适配 LLM context，不超 token）。所以分块是"把大文档拆成语义聚焦的小块"，每块独立 embedding/检索，检索准且 LLM 能用。

### 第二层：证据与定位

**Q：分块后检索召回差，怎么定位是分块问题（块切得不好）而非 embedding 或检索问题？**

对比测试。1）理想分块对比——用当前分块跑 Recall@K，再用"人工标注的理想分块"（语义完整的块）跑 Recall，如果当前远低于理想（如 60% vs 85%），分块是瓶颈；2）块质量分析——抽查漏召回 case，看 golden 文档的块是否被切碎（关键信息分散在多块，单块相似度低，检索不到）或块过大（块内信息杂，相似度被稀释），是则分块不合理；3）分块策略 AB——试不同分块（固定/语义/结构化），对比 Recall，差异大说明分块敏感（优化空间大）；4）排除 embedding/检索——如果换分块策略 Recall 显著变（如语义块比固定块高 20%），是分块问题；如果所有分块策略 Recall 都差（embedding 不匹配），是 embedding 问题。定位方法：控制变量（固定 embedding/检索，只变分块），看 Recall 变化。

### 第三层：根因深挖

**Q：固定分块（按字符数切）简单但可能切断语义，语义分块（按段落/句子切）好但实现复杂，实际怎么选？**

按文档类型和效果要求选。1）固定分块——按字符/token 数（如每 500 token 一块），简单快速，适合"无结构的纯文本"（如文章/日志），但可能切断句子/段落（语义受损）；2）语义分块——按句子/段落/标题边界切，保留语义完整，适合"有段落结构的文档"（如 wiki/博客/报告），质量好但需解析结构；3）结构化分块——按文档层级（章-节-段）切，块带层级元数据，适合"强结构文档"（如手册/法规/产品文档）；4）选型——无结构文本用固定（简单），段落文档用语义（质量），强结构用结构化（精准）。实务：默认语义分块（通用质量好），特殊文档用结构化，固定分块只用于快速原型。还要加分块重叠（overlap，如相邻块重叠 50 token），避免切断关键信息。

**Q：分块重叠（overlap）能避免切断关键信息，但重叠多了冗余（费存储/检索重复），重叠多少合适？**

按"避免切断"和"冗余"权衡。1）避免切断——关键信息（如一个完整论点）可能跨块边界（块尾+块首），重叠让这部分在两块都出现（至少一块完整），重叠要够大覆盖"典型跨边界信息"（如一句话/一个短语）；2）冗余——重叠多了同一信息在多块（存储重复/检索时多块召回同一信息，浪费 top-K 名额）；3）经验值——一般 10-20% 的块大小（如块 500 token，重叠 50-100 token），覆盖一句话又不太多冗余；4）动态——如果文档关键信息密集（如技术文档要点密集），重叠大一点（防切断）；如果稀疏（如叙事文），重叠小（少冗余）。AB 测试：不同重叠跑 Recall，找最优（如 overlap=50 比 0 和 200 都好）。原则：重叠够覆盖典型跨边界信息（一句话级，50-100 token），不过度（控冗余）。

### 第四层：方案权衡

**Q：分块大小（chunk_size）怎么定？小（200 token）vs 大（1000 token）各有利弊，怎么选？**

按 embedding 模型和 LLM context 权衡。1）embedding 模型——不同 embedding 模型对输入长度敏感（如有的 max 512 token，超出截断；有的支持 8192），chunk_size 不能超模型 max（否则截断丢信息）；模型擅长的长度也影响（如句向量模型在短文本好，长文本可能语义平均稀释），按模型擅长选；2）LLM context——检索 top-K 给 LLM，总 token = K * chunk_size，要 ≤ LLM context（如 4K context，K=5 则 chunk_size ≤ 800），超 context 要截断；3）语义完整——chunk_size 要够包含"完整语义单元"（如一个知识点/一个论点），太小（如 50 token）语义碎片，太大（如 2000 token）语义杂；4）经验——一般 200-500 token（兼顾 embedding/LLM/语义），AB 测试选最优。原则：不超模型 max + 不超 LLM context/K + 语义完整，经验值 200-500，AB 验证。

**Q：结构化分块（按文档层级）质量好但要解析文档结构，解析失败（如 PDF 无明确标题）怎么办？**

降级 + 启发式。1）降级到语义分块——如果文档无明确结构（如纯文本 PDF），结构化解析失败，降级到语义分块（按句子/段落切），保证基本可用；2）启发式解析——用启发式提取结构（如字体大小判断标题、缩进判断层级、关键词如"第X章"识别章节），PDF/Word 可提取样式信息辅助；3）LLM 辅助——用 LLM 解析文档结构（如"识别这篇文档的章节标题和层级"），准但费成本，适合重要文档；4）混合——先启发式（快），失败再用 LLM（准），平衡成本和质量。实务：支持结构化的用结构化（如 Word/HTML 有明确标签），不支持的降级语义+启发式（PDF 提取字体），关键文档用 LLM 辅助解析。原则：尽力提取结构（质量），失败降级（可用），不因解析失败而无法分块。

### 第五层：验证与沉淀

**Q：你怎么衡量分块策略的效果（哪种分块召回好/答案准）？**

AB 对比。固定评估集（query-文档对），对比不同分块策略：1）Recall@K——理想分块（语义/结构化）应高于固定分块（如 85% vs 60%），证明语义分块好；2）块质量——抽查检索结果，看块是否语义完整（无切断）、聚焦（无杂糅）；3）端到端——基于不同分块检索结果的答案准确率（分块好答案好）；4）成本——分块/索引的计算成本（结构化分块需解析，成本高）。最优：Recall 高 + 块质量好 + 答案准 + 成本可接受 = 分块策略优。还要看不同文档类型的最佳分块（如技术文档用结构化好，叙事文用语义好），按主要文档类型选。

**Q：分块策略怎么沉淀成文档处理的标准能力？**

建分块平台：1）多分块策略——内置固定/语义/结构化/递归等分块器，可配置组合（如按文档类型自动选）；2）文档解析——支持多格式（PDF/Word/HTML/Markdown），提取结构（标题/层级），启发式+LLM 辅助；3）参数调优——chunk_size/overlap/策略的可配置，AB 测试工具辅助选参；4）质量评估——分块后自动评估（块语义完整性/Recall），辅助优化；5）增量更新——文档变更时增量分块（只处理变更部分），高效。这套写入团队文档处理 SOP，让"分块"从"每个项目自己实现"变成"平台标准化能力"，开发者配置即可用。
