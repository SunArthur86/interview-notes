# 上下文窗口是什么？突破长上下文有哪些技术？

## 一、上下文窗口定义

```
上下文窗口(Context Window)
  = LLM单次推理能处理的token数量上限

  GPT-3:        2K tokens
  GPT-4:        8K / 32K / 128K
  Claude 3:     200K
  Gemini 1.5:   1M / 2M
  国产(通义/Kimi): 128K - 1M+

  意义：
  - 窗口大 → 能处理更长的文档/对话
  - 但更大 ≠ 更好（成本高/中间遗忘/精度下降）
```

## 二、为什么有上限

```
限制因素：

1. 注意力计算的O(n²)复杂度
   每个token要关注所有其他token
   n=100K → 10^10次运算 → 显存爆炸

2. 位置编码的表达能力
   训练时的最大长度限制了推理长度
   超过训练长度，位置编码外推效果差

3. 显存限制
   KV Cache随序列长度线性增长
   100K token的KV Cache可能占几十GB

4. "中间遗忘"(Lost in the Middle)
   即使支持长上下文，模型对中间部分的处理质量下降
```

## 三、突破技术：模型层面

### 1. 位置编码扩展（RoPE）

```python
# RoPE (Rotary Position Embedding)
# 通过旋转矩阵编码相对位置
# 优势：可外推到训练时没见过的长度

# 扩展方法：
# - NTK-aware: 调整RoPE基频，平滑外推
# - YaRN: 分段插值，兼顾近远距离
# - Dynamic NTK: 推理时动态调整

# 效果：训练4K的模型，通过RoPE扩展可处理32K+
```

### 2. 滑动窗口注意力

```python
# 标准注意力：每个token看所有token O(n²)
# 滑动窗口：每个token只看附近W个token O(n×W)

class SlidingWindowAttention:
    def __init__(self, window_size=4096):
        self.window = window_size
    
    def attention(self, query, key, value, positions):
        # 每个位置只关注 [pos-W, pos+W] 范围
        for i in range(len(query)):
            start = max(0, i - self.window)
            end = min(len(key), i + self.window)
            # 只在这个窗口内做attention
            attend(query[i], key[start:end], value[start:end])

# Mistral等模型用此技术支持32K+
```

### 3. Flash Attention

```python
# 不改变注意力结果，但优化GPU内存访问
# 把注意力计算分块，减少HBM读写
# 效果：同样的上下文，显存占用降3-5倍，速度提升2-4倍

# 不是"算法突破"而是"工程优化"
# 让长上下文在有限显存下可行
```

### 4. 稀疏注意力 / 线性注意力

```python
# 稀疏：只关注"重要"的token（如Longformer）
# 线性：用核函数近似，降到O(n)（如Performer/Linformer）
# 代价：精度有损，目前不如标准注意力准
```

## 四、突破技术：应用层面

### 1. RAG（外置记忆）

```python
# 不把所有文档塞进上下文，而是按需检索
# 这是处理"超大知识库"的标准方案

def answer_with_rag(question, huge_docs):
    # 从百万文档中检索相关的5-10个片段
    relevant = retriever.search(question, top_k=5)
    # 只把相关的塞进上下文
    context = format(relevant)  # ~2000 tokens
    return llm.generate(question, context)
# 突破窗口限制：外部存储无限，上下文只放精华
```

### 2. 摘要压缩

```python
# 长文档→摘要→塞进上下文
def handle_long_doc(long_doc):
    if count_tokens(long_doc) > window:
        # 分块摘要
        chunks = split(long_doc, size=window)
        summaries = [llm.summarize(c) for c in chunks]
        # 摘要的摘要（层次化）
        final_summary = llm.summarize("\n".join(summaries))
        return final_summary
    return long_doc
```

### 3. 滑动窗口处理

```python
# 分批处理超长输入
def process_long_text(text, window=8000, overlap=500):
    chunks = []
    for i in range(0, len(text), window - overlap):
        chunk = text[i:i+window]
        result = llm.process(chunk)
        chunks.append(result)
    return merge(chunks)
```

### 4. Map-Reduce 模式

```python
# Map: 分块独立处理
# Reduce: 汇总结果
def map_reduce_long_doc(docs, question):
    # Map: 每块独立找答案
    partial_answers = []
    for chunk in split(docs, window):
        ans = llm.ask(question, chunk)
        partial_answers.append(ans)
    
    # Reduce: 汇总
    final = llm.synthesize(question, partial_answers)
    return final
```

## 五、长上下文的问题

```
┌──────────────┬──────────────────────────────────┐
│ 问题          │ 说明                                │
├──────────────┼──────────────────────────────────┤
│ 中间遗忘      │ 模型对上下文中间部分处理质量下降      │
│ (Lost in     │ 首尾好，中间差                      │
│  Middle)     │                                    │
├──────────────┼──────────────────────────────────┤
│ 成本高        │ Token数∝成本，100K输入很贵          │
├──────────────┼──────────────────────────────────┤
│ 延迟高        │ 更长上下文=更慢的首token延迟         │
├──────────────┼──────────────────────────────────┤
│ 精度下降      │ 信息过多会"稀释"注意力              │
│              │ 关键信息被淹没                      │
├──────────────┼──────────────────────────────────┤
│ 位置外推      │ 超过训练长度效果下降                │
└──────────────┴──────────────────────────────────┘

应对：
  - 重要信息放首尾（避免中间）
  - 能用RAG就别硬塞（成本+精度）
  - 摘要压缩减少token
```

## 六、面试加分点

3. **RAG 仍是主流**：即使有 1M 窗口，RAG 在成本和精度上仍更优——务实判断

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：现在 Claude/Gemini 已经支持 1M-2M 窗口了，你为什么还说 RAG 是主流而不是直接全塞进去？**

因为"能塞"和"该塞"是两回事，三个现实约束让 RAG 不可替代。一是成本——1M token 输入按 GPT-4 定价一次几十美金，RAG 只塞 2000 token 是几分钱，差 100 倍。二是延迟——首字延迟（TTFT）随输入 token 线性增长，1M 输入 TTFT 可能到 30s+，用户等不了。三是精度——Lost in the Middle 现象在超长上下文里更严重，关键信息被噪声稀释，召回准确率反而比 RAG 精选片段低。所以长窗口解决的是"能放下"，RAG 解决的是"放对的且便宜"，不是替代关系。

### 第二层：证据与定位

**Q：你说有 Lost in the Middle，怎么验证你用的模型确实有这个问题，而不是凭论文结论假设？**

做一个"针插测试"（needle in haystack）。在一个长文档（比如 100K token 的无关文本）的不同位置（头部 5%、中部 50%、尾部 95%）插入一个事实陈述（"密码是 xyz123"），然后问模型"密码是什么"。跑 100 次统计每个位置的召回率。如果头部尾部 > 95%、中部 < 70%，就确认这个模型有 Lost in Middle。不同模型程度不同——GPT-4o 比 GPT-4 好很多，所以不能假设，必须针对自己用的模型实测。

### 第三层：根因深挖

**Q：RoPE 外推你说能让 4K 训练的模型扩展到 32K，但为什么不能无限外推到 1M？外推失效的根因是什么？**

根因是"相对位置表达力的衰减"。RoPE 的旋转角度按位置线性增长，短距离时角度差明显（模型能区分位置 1 和位置 2），长距离时角度差被周期折叠（位置 10000 和 10001 的旋转角度差异在浮点精度内被淹没）。NTK-aware 和 YaRN 通过调基频缓解，但本质是训练数据里没见过这么远的相对位置，模型对超长距离的注意力权重是"瞎猜"。所以外推到训练长度的 4-8 倍还行（4K→32K），到几十倍（4K→1M）就崩。要支持 1M 必须在长序列上继续预训练，纯外推不行。

**Q：那为什么不直接在训练时就用 1M 长度训练，一步到位？**

因为训练成本是 O(n²) 甚至更高。1M 长度的注意力计算量是 4K 的 6 万倍（(1000/4)²），单次前向就要几秒，全量预训练的算力成本是天文数字。而且训练数据里真正高质量的超长文本（完整一本书、完整代码库）很少，大部分数据是 4K 以内的网页/文章。所以工业界的做法是：先用 4K-8K 训练底座（性价比高），再在精选长文本上做长上下文继续预训练（只微调位置编码相关参数），把成本控制在可接受范围。

### 第四层：方案权衡

**Q：长文档处理你列了滑动窗口和 Map-Reduce，两个都能处理超长输入，你怎么选？**

看任务是否需要跨段信息。滑动窗口（带 overlap）适合"局部任务"——比如逐段翻译、逐段抽取实体，每段独立处理再拼接，不需要跨段推理。Map-Reduce 适合"全局任务"——比如"这篇文档的核心论点是什么"，需要看完所有段才能综合，所以 Map 阶段每段抽局部信息，Reduce 阶段汇总。判断标准：任务的答案是否依赖多个段落的关联，是就 Map-Reduce，否就滑动窗口。滑动窗口快且便宜，Map-Reduce 准但贵（多一轮 LLM 调用）。

**Q：为什么不直接用支持长窗口的模型一把读进去，还要拆 Map-Reduce 多费一道？**

因为精度和成本。一把读进去（1M 窗口）的 Lost in Middle 会让中段信息召回率掉到 70%，而 Map-Reduce 每段都在短窗口内处理（召回 > 95%），Reduce 阶段只汇总高召回的局部结果，整体准确率更高。成本上，Map-Reduce 虽然多调几次，但每次都是短输入（4K），总 token 数 100K，远少于一次塞 1M（省 10 倍）。所以长窗口模型解决"能放下"，Map-Reduce 解决"算得准又省"，后者是工程最优。

### 第五层：验证与沉淀

**Q：你用 RAG + 摘要组合处理长文档，怎么证明这个组合比直接塞长窗口效果好，而不是你的主观判断？**

用标注好的长文档 QA 评测集做 A/B。准备 100 个长文档（50K-200K token）+ 每个文档 5 个问题（覆盖细节/全局/跨段推理三类）。方案 A 直接全塞 Claude 200K 窗口，方案 B 用 RAG 检索 + 摘要压缩后塞。对比准确率和成本。我之前实测：方案 B 细节题准确率高 12%（因为 RAG 精选片段无噪声），成本降到方案 A 的 1/15，延迟降到 1/8。有这个对比数据，选 RAG 就不是主观而是数据驱动。

**Q：长上下文的实践经验怎么沉淀给团队？**

沉淀成"长输入处理决策树"：第一步判断输入长度，<8K 直接塞；8K-32K 评估 Lost in Middle 风险，关键信息前置；>32K 走 RAG。第二步判断任务类型，局部任务走滑动窗口，全局任务走 Map-Reduce。决策树配上每种方案的代码模板和评测指标（准确率/成本/延迟），新人按输入特征查树就能选对方案，不用每次重新设计。
