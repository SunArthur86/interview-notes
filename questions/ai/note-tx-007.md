# 【腾讯面经】你这个 Agent 项目怎么设计的？包含什么模块？Agent 整体运行流程是什么？

> 这是 Agent 方向的**系统性架构题**，面试官想看你是否真正做过 Agent 项目，而不只是调过 API。回答要像系统设计题一样展开：先给整体架构图，再逐模块讲设计决策，最后落到运行流程和工程细节。核心框架：五大模块 → 运行流程 → 关键设计决策 → 评估方法。

## 二、模块详解

### 模块一：编排引擎（Orchestrator）—— Agent 的大脑

编排引擎是整个系统的核心，负责五件事：

```python
class Orchestrator:
    def run(self, user_input: str) -> str:
        # 1. 意图理解：判断用户要做什么
        intent = self.understand_intent(user_input)

        # 2. 任务分解：复杂任务拆成子任务
        subtasks = self.decompose(intent)

        # 3. 路由决策：每个子任务用什么策略
        plan = self.plan(subtasks)

        # 4. 循环执行：ReAct / Plan-and-Execute
        for step in plan:
            result = self.execute_step(step)
            if self.is_complete(result):
                break

        # 5. 结果聚合：合并子结果，生成最终回复
        return self.synthesize(results)
```

**两种主流编排模式**：

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| **ReAct** | 思考-行动-观察循环，边想边做 | 探索性任务、步骤不确定 |
| **Plan-and-Execute** | 先规划全量步骤再执行 | 步骤明确、可并行优化 |

生产中常用**混合模式**：先 Plan 出大致步骤，执行中允许 ReAct 动态调整。

### 模块二：记忆模块（Memory）—— Agent 的状态

```
记忆分层:
├── 工作记忆（Working Memory）
│   ├── 当前对话上下文（在 prompt 中）
│   └── 任务状态快照（外部存储）
│
├── 短期记忆（Short-term · Redis）
│   ├── 最近 N 轮对话摘要
│   └── 当前会话的工具调用结果
│
└── 长期记忆（Long-term · 向量DB）
    ├── 用户偏好（按 user_id 隔离）
    ├── 历史执行轨迹
    └── 抽取的事实知识
```

**关键设计**：上下文窗口不够时用 Auto-Compact 压缩 + Task State Snapshot 保证任务不丢；跨会话记忆走向量DB，按相关性 + 时间衰减检索。

### 模块三：工具模块（Tools）—— Agent 的手脚

```
工具体系:
├── MCP 工具（Model Context Protocol）
│   └── 标准化外部接口：文件系统、数据库、浏览器、代码执行
│
├── Skill（能力封装）
│   ├── 复用已验证的 Prompt + 工具组合
│   └── 如 "搜索并总结" 作为一个 Skill 封装
│
└── 安全沙箱
    ├── 代码执行工具 → Docker 容器隔离
    ├── 文件操作 → 限定目录 + 权限白名单
    └── 网络请求 → 域名白名单 + 超时控制
```

```python
# 工具注册示例
@tool(name="web_search", description="搜索互联网获取最新信息")
def web_search(query: str, max_results: int = 5) -> list:
    # 带超时 + 重试 + 结果缓存
    return search_api.search(query, timeout=3, retries=2)

@tool(name="execute_code", description="在沙箱中执行Python代码",
      risk_level="high")  # 高风险 → 需人工确认
def execute_code(code: str) -> str:
    return sandbox.run(code, timeout=10)
```

### 模块四：知识模块（RAG）—— Agent 的参考书

```
RAG 管线:
用户问题
  ↓ Query 改写（扩展/分解）
  ↓ 向量检索（top-K=10~20，高召回）
  ↓ Rerank 精排（top-N=3~5，高精度）
  ↓ 上下文组装（按相关性排序 + token 预算控制）
  ↓ 注入 Prompt
```

**分块策略**：固定大小 vs 语义分块 vs 按文档结构（标题/段落）。生产推荐语义分块 + overlap，保证检索完整性。

### 模块五：安全与可观测层（Security）—— Agent 的免疫系统

```
安全四件套:
├── 权限控制
│   ├── 工具白名单（只允许调用预授权工具）
│   ├── 最小权限原则（只给完成任务所需的最小权限）
│   └── 高风险操作人工确认（删数据/发邮件/付款）
│
├── 全链路 Trace
│   ├── 每步决策记录（LLM输入/输出/工具调用/耗时）
│   ├── Token 消费追踪（成本归因到每个请求）
│   └── 异常告警（连续失败/超时/异常输出）
│
├── 限流熔断
│   ├── 单用户 QPS 限制
│   ├── 全局并发控制
│   └── 连续失败熔断 → 降级
│
└── Prompt 注入防护
    ├── 输入清洗（去除注入模式）
    ├── 系统提示隔离（用户输入不覆盖 system prompt）
    └── 输出过滤（检测敏感信息泄露）
```

### 循环执行的状态机

```python
class AgentLoop:
    MAX_STEPS = 20          # 防死循环
    STEP_TIMEOUT = 30       # 单步超时（秒）
    MAX_RETRIES = 3         # 单步重试

    def run(self, task):
        state = AgentState(task=task, step=0)
        while not state.is_complete:
            # 安全检查
            if state.step >= self.MAX_STEPS:
                return self.fallback(state, reason="超过最大步数")
            if state.total_cost > task.budget:
                return self.fallback(state, reason="超过成本预算")

            # 执行一步
            try:
                action = self.llm_decide(state)  # Thought + Action
                if action.is_high_risk:
                    if not self.human_confirm(action):
                        continue  # 用户拒绝，重新决策
                result = self.execute_with_timeout(action)
                state.observe(result)
            except TimeoutError:
                state.observe("工具执行超时")
            except Exception as e:
                state.observe(f"执行失败: {e}")

            state.step += 1

        return state.final_answer
```

## 四、关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 最大循环次数 | 限制 20 步 | 防死循环，控制成本 |
| 单步超时 | 30 秒 | 防工具卡死，保障可用性 |
| 高风险操作 | 人工确认 | 防误操作（删数据/付款） |
| 全链路 Trace | 必须有 | 可调试、可审计、可优化 |
| 成本预算 | 每任务设上限 | 防异常消耗 |
| 失败处理 | 重试 → 降级 → 兜底 | 三级容错保障稳定性 |

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 拆成编排引擎、记忆、工具、知识、安全监控五个模块，为什么是这个划分而不是更粗或更细？**

这个划分对应"大脑 + 记事本 + 手脚 + 参考书 + 免疫系统"的角色分工，每个模块职责单一且可独立演进。编排引擎（大脑）负责决策，记忆（记事本）负责状态，工具（手脚）负责执行，知识（参考书）负责信息，安全监控（免疫系统）负责防护。更粗的划分（如把记忆和知识合并）会让职责混淆（短期状态 vs 长期知识的访问模式不同）；更细的划分（如把规划单独拆出）会增加模块间通信成本。五模块是职责清晰和系统简单的平衡点。

### 第二层：证据与定位

**Q：Agent 运行时某个模块出问题（如工具调用超时），怎么快速定位是哪个模块的锅？**

用分布式 trace。每个模块的调用作为一个 span，标注模块名和耗时。1) 如果工具调用 span 显示超时，是工具模块（外部 API 慢）；2) 如果编排引擎 span 显示决策延迟，是 LLM 推理慢；3) 如果记忆召回 span 慢，是向量库或 Redis 慢。看 trace 的 span 时序，定位最长的 span 就是瓶颈。关键是有"端到端 trace ID"串联所有模块的调用，否则各模块日志对不上。

### 第三层：根因深挖

**Q：Agent 整体响应慢，但单个模块都不慢，根因是什么？**

通常是模块间的"等待和序列化"开销。1) 串行编排——能并行的模块被串行调用（如先查记忆再查知识，其实可以并行），累积延迟；2) 序列化开销——模块间传递数据要序列化/反序列化（如 JSON 编解码），数据量大时耗时；3) 网络往返——模块在不同进程/机器，每次调用有网络延迟（RPC 开销）。根因是"模块化引入的通信成本"。解法：能并行的并行、用更高效的序列化（如 protobuf）、同进程部署高频调用的模块。

**Q：那为什么不直接把所有模块合并到一个进程，避免模块间通信开销？**

权衡"性能 vs 可维护性"。合并到一个进程性能最好（无通信开销），但：1) 单点故障——一个模块崩全崩；2) 无法独立扩展——记忆模块要扩容时整个进程要扩；3) 技术栈耦合——所有模块必须用同一语言。微服务式拆分（每模块独立进程）牺牲性能换可维护性和可扩展性。经验上：高频核心模块（编排引擎、记忆）同进程部署（低延迟），低频独立功能（知识检索、安全审计）独立服务（可扩展）。混合部署比"全合并"或"全拆分"更合理。

### 第四层：方案权衡

**Q：安全监控作为独立模块 vs 嵌入到每个模块里（如编排引擎自带护栏），怎么选？**

权衡"一致性 vs 性能"。独立安全监控——所有模块的输出都经过它，检查一致（统一规则），但增加一次调用延迟；嵌入式护栏——每个模块自带检查，快但规则可能不一致。经验上分层：1) 嵌入式做"轻量实时检查"（如输入长度限制、输出格式校验）——快；2) 独立监控做"重量深度检查"（如内容安全审核、越狱检测）——准。两者配合，不是二选一。

**Q：为什么不直接用现成的 Agent 框架（LangChain/LangGraph），而要自研模块化架构？**

现成框架解决"快速搭建"，但不解决"生产治理"。LangGraph 的状态机好用，但生产需要：1) 细粒度的权限控制（每个工具谁能调）；2) 成本控制（token 预算、调用频次限制）；3) 审计日志（谁、何时、调了什么、结果）；4) 多租户隔离。这些框架不提供或不够成熟。所以"用框架的组件 + 自研的治理层"是常见做法。完全自研是为了对每一层有完全控制权，适合对安全性和可控性要求高的场景（金融、医疗）。

### 第五层：验证与沉淀

**Q：怎么衡量 Agent 架构设计是否合理（模块化是否到位）？**

三个指标：1) 模块独立性——每个模块能否独立测试、独立部署、独立扩展（如果不能，模块化不到位）；2) 通信效率——模块间通信的延迟占总延迟的比例（< 20% 合理，太高说明过度拆分）；3) 故障隔离——一个模块故障是否影响其他模块（应该不影响，否则耦合太紧）。沉淀为架构 review checklist：模块边界清晰度、接口契约完整性、故障隔离性、可独立演进性。

## 结构化回答

**30 秒电梯演讲：** Agent项目=编排引擎（大脑）+记忆（状态）+工具（手脚）+知识（参考书）+安全监控（免疫系统）——Agent像一个公司。

**展开框架：**
