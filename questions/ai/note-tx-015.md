# 【腾讯面经】代码 AI 生成比例多少？做 Agent 踩过哪些坑？

## 一、问题本质：面试官在考察什么？

这道题看似是"闲聊体感"，实际考察三个维度：

```
┌─────────────────────────────────────────────────────┐
│  考察维度                                           │
├─────────────┬───────────────────────────────────────┤
│ 1. 诚实度   │ 是否夸大 AI 能力？是否回避自身短板？ │
│ 2. 工程判断 │ 人机协作的边界划在哪里？为什么？     │
│ 3. 实战深度 │ 踩过的坑是否真实？调试方法是否系统？ │
└─────────────┴───────────────────────────────────────┘
```

> **答题策略**：给出真实数据 → 解释比例背后的判断逻辑 → 用具体案例讲踩坑经验 → 总结方法论。**诚实 > 包装**。

## 二、AI 生成比例：真实数据与分层结构

### 2.1 我的实际数据（2024-2025 年实际项目统计）

| 代码类型 | AI 生成占比 | 人工审查占比 | 说明 |
|---------|-----------|------------|------|
| **样板代码**（CRUD、DTO、配置） | 80-90% | 10-20% | AI 几乎全自动，人工只审命名规范 |
| **业务逻辑** | 50-60% | 40-50% | AI 出框架，人工调边界和异常 |
| **核心算法/架构** | 20-30% | 70-80% | 人工主导设计，AI 辅助实现细节 |
| **调试/修复** | 40-50% | 50-60% | AI 定位问题快，人工验证修复 |
| **测试代码** | 70-80% | 20-30% | AI 生成单测框架，人工补边界 case |
| **加权平均** | **60-70%** | **30-40%** | — |

### 2.2 为什么是 60-70% 而不是 90%？

```
AI 代码生成的核心矛盾：
┌──────────────────────────────────────────────┐
│  AI 擅长             │  AI 不擅长            │
├──────────────────────┼──────────────────────┤
│  局部代码生成        │  全局架构设计         │
│  模式匹配/模板代码   │  跨模块的语义理解     │
│  已知 API 调用       │  业务规则的隐含约束   │
│  单一函数实现        │  系统级性能权衡       │
│  测试用例生成        │  安全性边界判断       │
└──────────────────────┴──────────────────────┘

结论：AI 生成效率高 → 但缺乏全局视角 → 人工必须做架构设计和安全审查
```

### 2.3 分层协作模型

```
┌─────────────────────────────────────────────────┐
│                  人机协作金字塔                  │
│                                                 │
│   ┌─────────┐                                   │
│   │ 人工决策 │  架构选型、技术方案、安全审查     │
│   │  10%    │  （不可替代，最关键）             │
│   └────┬────┘                                   │
│   ┌────┴────┐                                   │
│   │人机协作  │  业务逻辑、接口设计、调试修复     │
│   │  30%    │  （AI 提议 → 人工审查 → 迭代）    │
│   └────┬────┘                                   │
│   ┌────┴────┐                                   │
│   │ AI 生成  │  样板代码、测试、文档、重构       │
│   │  60%    │  （AI 主导，人工抽检）            │
│   └─────────┘                                   │
└─────────────────────────────────────────────────┘
```

> **面试加分点**：不要说"90% 都是 AI 生成的"——这要么说明项目太简单，要么说明你没有认真审查。**坦诚的 60-70% + 清晰的分层逻辑** 更有说服力。

### 3.1 故障全景图

     ▼

### 3.2 故障①：工具调用失败

**典型表现**：
- LLM 生成的工具参数格式错误（如 JSON 字段名错误、类型不匹配）
- 外部 API 超时、限流、返回异常
- 工具描述不清导致 LLM 调用错误的工具

**真实案例**：
```python
# 问题：LLM 生成调用，但参数格式错误
{
  "tool": "search_code",
  "params": { "query": "login function" }  # ❌ 实际参数名是 "keyword"
}

# 调试方法：
# 1. 结构化校验 —— 用 Pydantic 强制约束参数
from pydantic import BaseModel

class SearchCodeParams(BaseModel):
    keyword: str          # 强制类型
    max_results: int = 10 # 默认值

def search_code(params: SearchCodeParams):  # 类型安全
    ...
```

**解决方案体系**：
```
工具调用稳定性三板斧：
├── 1. 输入校验层：Pydantic / JSON Schema 强约束
├── 2. 重试容错层：指数退避重试 + 超时控制
│      retry(max_attempts=3, backoff=exponential)
└── 3. 降级兜底层：工具失败时返回结构化错误而非崩溃
       error → LLM 感知失败 → 选择替代方案
```

### 3.3 故障②：上下文断裂

**典型表现**：
- 长对话超过 context window，前面的关键信息被截断
- 多步骤 Agent 执行中，中间状态丢失
- RAG 检索的上下文与当前任务不相关

**真实案例**：
```python
# 问题：10 步 Agent 任务，第 8 步突然"忘记"第 2 步的约束
# 原因：context window 只能装 6 步的历史

# 解决方案：分层上下文管理
class ContextManager:
    def __init__(self, max_tokens=8000):
        self.max_tokens = max_tokens
        self.working_memory = []  # 短期：当前任务关键信息
        self.episodic_memory = [] # 中期：本轮对话摘要
        self.semantic_memory = [] # 长期：RAG 知识库

    def get_context(self):
        """优先级压缩策略"""
        context = []
        # 1. 系统提示 + 任务目标 —— 永远保留
        context.append(self.system_prompt)
        # 2. 当前步骤 + 前一步结果 —— 高优先级
        context.extend(self.working_memory[-4:])
        # 3. 历史摘要 —— 低优先级，token 不够就压缩
        context.extend(self.summarize(self.episodic_memory))
        return self.truncate_to_tokens(context, self.max_tokens)
```

**调试方法**：
```
上下文断裂排查清单：
☑  打印每一步的 context 内容，确认关键信息是否在
☑  检查 token 计数，确认是否超限截断
☑  对比"有上下文"和"无上下文"的输出差异
☑  用 LangSmith / Phoenix 等工具可视化 trace
☑  检查 RAG 检索的 recall（是否检索到相关文档）
```

### 3.4 故障③：幻觉输出

**典型表现**：
- 编造不存在的 API、函数名、参数
- 引用错误的文档版本
- 逻辑推理跳步（A→C，跳过 B）

**真实案例**：
```python
# 问题：Agent 生成的代码引用了不存在的 API
result = some_lib.awesome_function()  # ❌ 这个函数不存在

# 调试方法：
# 1. 输出约束 —— 用结构化输出限制 LLM 的自由度
from instructor import llm_validator

@llm_validator("回答必须基于检索到的文档，不能编造")
class CodeAnswer(BaseModel):
    code: str
    source: str  # 必须引用来源文档
    confidence: float  # 置信度

# 2. 事实校验 —— 对 AI 输出做后验检查
def verify_code_safety(code: str):
    checks = [
        ("import 存在性", check_imports_exist),
        ("API 签名匹配", check_api_signatures),
        ("类型一致性", check_type_consistency),
    ]
    results = {name: fn(code) for name, fn in checks}
    failed = [k for k, v in results.items() if not v]
    return failed  # 返回失败项，触发重试或人工介入
```

**解决策略**：
```
幻觉治理四层防御：
├── 1. 预防：清晰的 system prompt + Few-shot 示例约束输出格式
├── 2. 检测：对输出做事实校验（代码能否编译/运行/通过测试）
├── 3. 纠正：检测到幻觉时自动重试（带"修正提示"）
└── 4. 兜底：无法纠正时降级为"我不确定，建议人工确认"
```

### 3.5 故障④：循环调用

**典型表现**：
- Agent 反复调用同一个工具，陷入死循环
- 工具 A 失败 → 调用工具 B → B 也失败 → 回到 A
- 自我对话/自我修正的无限螺旋

**真实案例**：
```python
# 问题：Agent 找不到答案 → 调用搜索 → 搜索无结果 → 再次搜索 → 死循环

# 解决方案：状态追踪 + 循环检测
from collections import defaultdict

class LoopDetector:
    def __init__(self, max_same_action=3, max_total_steps=20):
        self.action_history = []
        self.max_same_action = max_same_action
        self.max_total_steps = max_total_steps

    def check(self, action: str) -> bool:
        """返回 True 表示检测到循环，应终止"""
        self.action_history.append(action)

        # 检查 1：同一动作连续超过阈值
        recent = self.action_history[-self.max_same_action:]
        if len(recent) == self.max_same_action and len(set(recent)) == 1:
            return True  # 连续重复

        # 检查 2：总步数超限
        if len(self.action_history) >= self.max_total_steps:
            return True

        # 检查 3：短周期循环（A→B→A→B 模式）
        if len(self.action_history) >= 4:
            seq = self.action_history[-4:]
            if seq[0] == seq[2] and seq[1] == seq[3]:
                return True

        return False

# 在 Agent 主循环中使用
detector = LoopDetector()
while not task_done:
    action = llm_decide(state)
    if detector.check(action):
        print("⚠️ 检测到循环，强制终止并返回当前最佳结果")
        break
    result = execute(action)
    state.update(result)
```

## 四、踩坑经验总结（面试高价值）

### 4.1 工具设计踩坑

| 踩坑 | 现象 | 教训 |
|------|------|------|
| 工具描述太模糊 | LLM 不知道何时用哪个工具 | **工具描述要包含"何时使用"和"何时不使用"** |
| 工具粒度太细 | 20+ 个小工具，LLM 选错率高 | **合并为 5-7 个粗粒度工具**，内部路由 |
| 返回格式不统一 | LLM 解析各种格式崩溃 | **统一返回 `{success, data, error}` 结构** |
| 没有超时控制 | 外部 API 挂了 Agent 卡死 | **每个工具必须有 timeout + 降级** |

### 4.2 Prompt 工程踩坑

```python
# ❌ 踩坑：prompt 没有限制输出格式
prompt = "分析这段代码的性能问题"  # LLM 可能输出长篇大论

# ✅ 修正：结构化 prompt + 输出约束
prompt = """
分析代码性能问题，按以下格式输出：
- 问题1：[问题描述] | 影响：[高/中/低] | 建议：[修复方案]
- 问题2：...
最多列出 3 个最严重的问题。不要编造不存在的问题。
"""
```

### 4.3 上下文管理踩坑

```
最大教训：不要把所有历史都塞进 context window

错误做法：messages = [全部历史消息]
正确做法：messages = [系统提示] + [任务摘要] + [最近3轮对话] + [RAG检索结果]

token 预算分配建议（以 8K context 为例）：
├── 系统提示 + 工具定义    1500 tokens
├── 任务摘要 + 关键约束   1000 tokens
├── 最近对话历史          2000 tokens
├── RAG 检索上下文        3000 tokens
└── 输出预留              500 tokens
```

### 4.4 评估体系踩坑

```
最大教训：没有评估体系就无法迭代优化

┌──────────────────────────────────────────┐
│         Agent 评估三层体系               │
├──────────────┬───────────────────────────┤
│ 1. 单元评估  │ 每个工具的准确率/延迟      │
│ 2. 集成评估  │ 端到端任务完成率/步数      │
│ 3. 回归评估  │ 每次改 prompt 跑 100 case │
└──────────────┴───────────────────────────┘

关键指标：
- 任务完成率（Task Success Rate）：目标 >85%
- 平均步数（Avg Steps）：越少越好
- 工具调用准确率（Tool Accuracy）：目标 >90%
- 幻觉率（Hallucination Rate）：目标 <5%
```

## 五、面试回答框架

### 30 秒精简版

> 我项目中 AI 生成代码占比约 **60-70%**——样板代码、测试和文档几乎全用 AI，但核心架构和业务逻辑以人工为主。做 Agent 最大的坑有四个：**工具调用失败**（解决方案：结构化校验 + 重试 + 降级）、**上下文断裂**（解决方案：分层上下文管理 + 摘要压缩）、**幻觉输出**（解决方案：输出约束 + 事实校验）、**循环调用**（解决方案：循环检测器 + 步数限制）。核心经验是：**Agent 工程的关键不是"能跑起来"，而是"出错时如何优雅降级"**。

### 加分句（选择 1-2 句说出来）

1. "AI 生成比例不是越高越好——**60-70% + 30-40% 深度审查**比 90% 不审查更可靠"
2. "Agent 最难的不是写 prompt，而是**设计评估体系和降级策略**"
3. "每改一次 prompt，我都会跑 100 个 case 的回归测试——**没有评估就没有迭代**"
4. "工具调用的本质是**把 LLM 的不确定性约束到确定性管道里**"

```

│              AI 协作 & Agent 工程核心原则                │
├─────────────────────────────────────────────────────────┤
│                                                         │

│     → 60-70% AI 生成 + 分层逻辑 > "90% 都是 AI"          │

│                                                         │
│  5. 人机边界 > 全自动幻想                                │

└─────────────────────────────────────────────────────────┘
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：代码 AI 生成比例这个指标，为什么面试官要问？它在考察什么？**

考察"对 AI 辅助开发的真实理解和工程判断力"。如果回答"100% AI 生成"——要么是吹牛、要么是不负责任（核心逻辑和安全代码不能全靠 AI）；如果回答"完全不用 AI"——要么是保守、要么是没用过现代工具。合理的回答是"按代码类型分层"：样板代码 80%+ AI 生成、业务逻辑 30-50% AI 辅助、安全/核心逻辑 < 10%（人写为主）。面试官想看你是否知道"AI 适合什么、不适合什么"的边界。

### 第二层：证据与定位

**Q：AI 生成的代码上线后有 bug，怎么定位是"AI 写错了"还是"人 review 漏了"？**

看代码的 author 标记和 review 记录。1) 如果代码是 AI 生成、人没改直接 merge，bug 的第一责任人是 reviewer（没尽到把关）；2) 如果 AI 生成后人改过引入 bug，是修改的问题不是 AI 的问题；3) 如果 AI 生成的边界 case 错（如 null 处理），是 AI 的已知弱点，review 时应该重点查。关键原则：AI 生成的代码 bug，责任永远在 merge 它的人——AI 是工具不是责任人。

### 第三层：根因深挖

**Q：Agent 工程中你踩过最大的坑是什么？根因是技术判断错还是业务理解错？**

这道题考的是反思深度。好的回答要具体（不是"沟通不畅"这种空话）。常见坑：1) 过度信任 LLM——把不可靠的输出直接用，没做校验，导致线上错误（根因是技术判断错，低估了 LLM 的不确定性）；2) 工具 schema 设计模糊——LLM 反复调错工具，准确率低（根因是工程设计不严谨）；3) 没做成本控制——多步 Agent 调用累积 token 成本爆炸（根因是没预估资源消耗）。讲坑时要讲"根因 + 教训 + 后续怎么避免"，体现成长。

**Q：那为什么不直接承认"我没踩过坑"（如果真没有）？**

如果真没踩过坑，要么是项目太简单、要么是没意识到坑（后者更可能）。Agent 工程的复杂度高，没人能"零坑"。面试官更欣赏"踩过坑、反思清楚、有改进"的候选人，而不是"零坑"的（可能缺乏深度或隐瞒）。如果确实经验少，诚实说"目前踩过的坑是 X（小坑），随着项目复杂度增加预期会遇到 Y 类问题"，比强行编大坑好。

### 第四层：方案权衡

**Q：AI 生成代码 vs 人写代码，怎么权衡两者的比例？**

按"代码类型和风险等级"分层。1) 样板代码（CRUD、配置、单测骨架）——AI 生成 80%+，人 review 即可，效率优先；2) 业务逻辑（中等复杂度的业务规则）——AI 辅助 30-50%，人主导设计，AI 补细节；3) 安全/核心逻辑（鉴权、权限、资金）——人写为主，AI 只做格式辅助，安全优先。比例不是固定值，按代码的"出错代价"调整——出错代价高的代码人写比例高。

**Q：既然 AI 生成效率高，为什么不强制要求团队所有代码用 AI 生成（提升效率）？**

因为"效率"不是唯一目标。强制 AI 生成会导致：1) 代码质量方差大——不同人用 AI 生成的质量不一，维护成本上升；2) 技术债——AI 倾向于"堆代码"而非"抽象复用"，长期代码重复；3) 能力退化——新人过度依赖 AI，不理解底层，遇到 AI 不会的就卡住。所以"鼓励 AI 辅助"但"不强制 AI 生成"，让人根据代码类型自主选择。规范是"AI 生成占比上限"（如 < 50%），不是"AI 生成占比下限"。

### 第五层：验证与沉淀

**Q：怎么衡量团队用 AI Coding 的实际收益，而不是"感觉快了"？**

三个量化指标：1) 开发周期——同类 feature 从开工到上线的小时数，对比引入 AI 前后；2) AI 生成占比——用 git blame + AI 工具标记统计（目标 30-50%，过高说明可能滥用）；3) 缺陷率——AI 生成代码的 bug 密度 vs 人写代码的 bug 密度（AI 生成不应显著高于人写）。如果开发周期缩短 30%、缺陷率持平，AI Coding 有效。沉淀为团队 AI Coding 规范：可用场景、review 标准、占比上限。

## 结构化回答




**30 秒电梯演讲：** 就像问一个厨师"你的菜多少是预制菜、多少是现做的"——考察你对工具依赖度和工程掌控力的认知。

**展开框架：**
1. **AI** — AI 生成 vs 人工审查的边界
2. **Agent** — Agent 常见故障模式
3. **工具调** — 工具调用稳定性

**收尾：** Agent 在什么场景下最容易出错？你是如何调试的？
