# 【月之暗面面经】如果桌面端支持多窗口协作，前端怎么避免不同窗口状态打架？

## 一、问题本质：多窗口状态冲突

```
场景：用户同时打开两个窗口
  窗口A：正在编辑产物PPT的第3页
  窗口B：同时在编辑同一个PPT的第5页
  → 保存时谁的版本覆盖谁？💥

场景：窗口A正在执行长任务
  窗口B也触发了同一个任务
  → 重复执行，浪费资源 💥

场景：窗口A修改了文件授权
  窗口B不知道，继续使用旧授权
  → 数据不一致 💥
```

## 三、实现架构

```typescript
// 跨窗口通信机制（Electron/Tauri）
class WindowSyncManager {
  private mainWindow: BrowserWindow;
  private popoutWindows: Map<string, BrowserWindow> = new Map();
  
  // 强同步：立即广播
  broadcastFact(event: FactEvent) {
    // 广播到所有窗口
    for (const [, window] of this.popoutWindows) {
      window.webContents.send('fact-update', event);
    }
    this.mainWindow.webContents.send('fact-update', event);
  }
  
  // 弱同步：节流广播
  private throttleMap = new Map<string, NodeJS.Timeout>();
  broadcastCoop(event: CoopEvent) {
    const key = `${event.type}:${event.targetId}`;
    if (this.throttleMap.has(key)) {
      clearTimeout(this.throttleMap.get(key)!);
    }
    this.throttleMap.set(key, setTimeout(() => {
      this.broadcastToAll('coop-update', event);
    }, 100));  // 100ms节流
  }
}

// 窗口侧监听
class WindowStateReceiver {
  constructor() {
    ipcRenderer.on('fact-update', (_, event: FactEvent) => {
      this.handleFactUpdate(event);
    });
    ipcRenderer.on('coop-update', (_, event: CoopEvent) => {
      this.handleCoopUpdate(event);
    });
  }
  
  private handleFactUpdate(event: FactEvent) {
    switch (event.type) {
      case 'task-status-changed':
        taskStore.updateTaskStatus(event.taskId, event.status);
        break;
      case 'artifact-version-updated':
        artifactStore.updateVersion(event.artifactId, event.version);
        break;
      case 'permission-changed':
        permissionStore.update(event.permissionId, event.changes);
        break;
    }
  }
}
```

```typescript
// 编辑权管理
class EditLockManager {
  private locks: Map<string, EditLock> = new Map();
  // key: artifactId, value: { windowId, acquiredAt, lastActivityAt }
  
  acquire(artifactId: string, windowId: string): AcquireResult {
    const existing = this.locks.get(artifactId);
    
    if (!existing || existing.windowId === windowId) {
      // 无锁或自己已持有
      this.locks.set(artifactId, {
        windowId,
        acquiredAt: Date.now(),
        lastActivityAt: Date.now(),
      });
      return { success: true };
    }
    
    // 检查锁是否过期（超过5分钟无活动自动释放）
    if (Date.now() - existing.lastActivityAt > 5 * 60 * 1000) {
      this.locks.set(artifactId, {
        windowId,
        acquiredAt: Date.now(),
        lastActivityAt: Date.now(),
      });
      return { success: true };
    }
    
    // 锁被其他窗口持有
    return {
      success: false,
      conflict: {
        holderWindowId: existing.windowId,
        holdingFor: Date.now() - existing.acquiredAt,
      },
    };
  }
  
  // 心跳续约（编辑中定期调用）
  heartbeat(artifactId: string, windowId: string) {
    const lock = this.locks.get(artifactId);
    if (lock && lock.windowId === windowId) {
      lock.lastActivityAt = Date.now();
    }
  }
  
  release(artifactId: string, windowId: string) {
    const lock = this.locks.get(artifactId);
    if (lock && lock.windowId === windowId) {
      this.locks.delete(artifactId);
    }
  }
}
```

## 六、常见坑

- **全量同步所有状态**：滚动位置、输入框内容都同步，窗口间互相干扰
- **没有编辑权管理**：两个窗口同时编辑同一个产物，最后保存的覆盖前者
- **锁不过期**：窗口崩溃后锁不释放，其他窗口永远无法编辑
- **长任务结果只通知发起窗口**：其他窗口的任务中心看不到最新状态

## 记忆要点

- 状态分层模型：按重要程度分为强同步（事实数据）、弱同步（协作数据）、不同步（UI状态）。
- 强同步防数据冲突：任务、产物、权限等核心状态必须立即广播至所有窗口保证强一致。
- 不同步防互相干扰：滚动位置、输入框草稿等窗口本地状态绝对不跨窗口同步。
- 并发编辑冲突解决：引入编辑锁机制，持有锁的窗口才能编辑，并设置超时自动释放防死锁。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么多窗口不能全量同步所有状态，非要分"强同步/弱同步/不同步"三层？全同步不是最简单吗？**

因为全量同步会导致"窗口间互相干扰"和"性能灾难"。全同步意味着窗口 A 的滚动位置、输入框草稿、浮层开关都广播到窗口 B——后果是：用户在窗口 A 输入"帮我做 PPT"，窗口 B 的输入框也突然出现"帮我做 PPT"（用户以为见鬼了）；用户在窗口 A 滚动到第 5 页，窗口 B 的视图也跳到第 5 页（用户正在 B 看第 1 页）。这种干扰让用户完全无法在多个窗口做不同的事。性能层面：输入框每打一个字就触发一次跨窗口 IPC 广播，高频操作（打字、滚动、鼠标移动）会导致 IPC 通道拥堵，主进程忙于转发消息而卡顿。三层分类的本质是"按数据价值决定同步策略"：强同步（任务/产物/权限）是所有窗口都必须看到一致的事实，立即广播；弱同步（编辑锁/光标位置）是协作辅助信息，延迟 100ms 广播即可（容忍短暂不一致）；不同步（滚动/输入框/浮层）是窗口本地的 UI 状态，完全独立。这样 90% 的高频 UI 操作（打字、滚动）不产生跨窗口通信，只有 10% 的低频事实变更（任务完成、产物更新）才广播，性能和体验都可控。

### 第二层：证据与定位

**Q：你怎么定位"两个窗口看到的任务状态不一致"是同步机制问题，还是某个窗口的状态没更新？**

用"状态版本号 + 同步日志"定位。每个事实数据（Task/Artifact/Permission）携带版本号（version 字段），每次变更递增。跨窗口同步时携带版本号，接收窗口对比本地版本：如果同步消息的版本号 > 本地版本号，更新本地状态；如果 < 本地版本号，说明消息乱序（旧消息晚到），丢弃；如果 = 本地版本号，说明重复消息，忽略。当发现两个窗口状态不一致时，查它们的本地版本号——版本号低的窗口说明漏掉了某次同步消息。再查同步日志（记录每次广播的 event 和目标窗口的 ack），定位是哪个窗口没收到消息（IPC 发送失败）还是收到了但没处理（handleFactUpdate 逻辑有 bug）。最常见的根因是：窗口 B 在广播消息时还未完成初始化（ipcRenderer.on 的监听器还没注册），导致消息被丢弃。解法是窗口初始化完成后主动拉取一次全量状态（而非只依赖增量广播），保证不会漏掉启动前的消息。

### 第三层：根因深挖

**Q：为什么并发编辑冲突要用"编辑锁（悲观）"，而不是"无锁 + 自动合并（乐观）"或"最后写入获胜（LWW）"？**

因为桌面端产物是结构化文档（PPT/表格/站点），三种策略的适用性不同。LWW（最后写入获胜）最简单但最危险——窗口 A 编辑了第 3 页标题后保存，窗口 B 编辑了第 5 页内容后保存，B 的保存覆盖 A 的修改，第 3 页标题丢失，用户完全无感知。自动合并（乐观并发）理论上最优但实践极难——需要结构化 Diff 算法（把两个窗口的编辑合并），对于 PPT 这种复杂结构（每页有文本/图片/动画/布局），合并算法的冲突率极高，且合并结果可能产生语义错误（如两个窗口都改了同一页但改了不同元素，合并后页面布局错乱）。编辑锁（悲观）最保守但最可靠——同一时刻只有一个窗口能编辑，其他窗口只读，从源头消除冲突。选编辑锁的根因是"桌面端文件没有 Git 这样的版本控制兜底，合并出错的代价是数据损坏且回滚困难"。悲观锁的代价（其他窗口要等）远小于数据损坏的代价。乐观合并只适用于纯文本协作（如 Google Docs 的 CRDT），结构化文档场景下悲观锁是更安全的选择。

**Q：那如果用户抱怨"我急着改 PPT 但被锁了改不了"，为什么不去掉编辑锁改成"各自编辑各自保存，冲突时让用户手动选"？**

因为"冲突时手动选"的体验比"等一下编辑锁"更差。手动选冲突意味着用户要在两个版本之间逐项对比（"窗口 A 改了标题，窗口 B 改了内容，你要哪个"），这个认知负担极重——尤其当冲突涉及几十处修改时，用户根本看不过来，最终要么随机选一个（丢数据）要么放弃编辑（白干）。而编辑锁的等待通常很短（其他窗口保存后就释放，约 1-3 分钟），且可以优化体验：(1) 锁的粒度细化——不是锁整个产物，而是锁到"页级别"或"字段级别"（窗口 A 锁第 3 页，窗口 B 可以编辑第 5 页）；(2) 锁的状态可见——显示"窗口 #2 正在编辑，已编辑 3 分钟"，让等待方有预期；(3) 超时自动释放——5 分钟无活动自动解锁（防崩溃死锁），等待方最多等 5 分钟；(4) 协同编辑——长期看可以引入 OT/CRDT 算法实现真正的实时协同（像 Figma），但这是大工程，短期内悲观锁 + 粒度细化是性价比最高的方案。

### 第四层：方案权衡

**Q：弱同步层（编辑锁/光标位置）你用 100ms 节流广播，为什么不实时广播或完全不同步？**

100ms 节流是"协作感知度"和"性能开销"的平衡点。实时广播的问题：光标位置每移动一次就广播（鼠标移动事件约 60 次/秒），跨窗口 IPC 通道会被高频消息占满，主进程忙于转发导致所有窗口卡顿。完全不同步的问题：协作者看不到对方的编辑意图（"窗口 A 的人正在改哪一页"），协作体验断裂——用户以为对方在改第 3 页，实际对方已经切到第 5 页，两边对话错位。100ms 节流的意义：人类对光标位置的感知粒度约 100ms（再快肉眼也看不出区别），所以每 100ms 广播一次光标位置，对协作体验无损（对方看到的轨迹仍然流畅），但消息频率从 60 次/秒降到 10 次/秒，性能压力降 6 倍。编辑锁的获取/释放是低频操作（秒级），不需要节流，直接实时广播。光标位置、在线状态这类"高频但容忍延迟"的协作数据走 100ms 节流，是分布式协作系统的标准实践。

**Q：那如果协作者说"100ms 的延迟让我感觉对方的光标有延迟，影响协作流畅度"，为什么不降到 10ms？**

因为"光标延迟感"的根因通常不是广播频率，而是网络/IPC 传输延迟，单纯提高频率解决不了。从 100ms 降到 10ms 会让消息频率从 10 次/秒升到 100 次/秒，IPC 通道负载增加 10 倍，反而可能导致消息积压（排队延迟超过 100ms），体感更卡。真正改善光标流畅度的方法是：(1) 插值预测——接收方窗口收到光标位置后，用插值算法平滑过渡（而非直接跳到新位置），即使广播间隔是 100ms，视觉上也是连续移动的；(2) 操作意图预测——基于对方的编辑动作（如正在输入）预测光标的下一步位置，提前渲染；(3) 优先级队列——光标位置消息走高优先级 IPC 通道（不与任务状态等事实数据排队），保证低延迟。所以解法是"优化渲染和传输策略"而非"盲目提高广播频率"。10ms 的频率只有在局域网低延迟环境（<5ms 传输延迟）下才有意义，桌面端多窗口的 IPC 传输延迟通常在 1-5ms，100ms 广播间隔配合插值已足够流畅。

### 第五层：验证与沉淀

**Q：你怎么证明"三层同步策略"比"全量同步"更优，用户真的感知到差异吗？**

用三个指标验证：(1) 多窗口并发事故率 = 用户报告"多窗口数据不一致/编辑丢失"的工单数 / 多窗口使用会话数——三层同步（强同步事实 + 编辑锁防冲突）应将事故率从"全量同步的频繁覆盖"降到接近零；(2) IPC 消息频率 = 单位时间内跨窗口消息数——三层同步因 90% 的 UI 操作不同步，消息频率应比全量同步低一个数量级（从每秒数百条降到每秒数十条），直接反映性能改善；(3) 多窗口任务并发率 = 用户同时操作多个窗口完成不同任务的会话比例——如果三层同步的体验好（不互相干扰），这个率应显著提升（用户敢于多窗口并发），全量同步时因干扰严重用户会回避多窗口。A/B 测试两组（全量同步 vs 三层同步），跑一周后对比——如果三层同步组的多窗口并发率高出 50% 以上且事故率接近零，就证明设计有效。

**Q：怎么让团队在新增跨窗口共享的数据时，自觉判断"该走强同步还是不同步"，而不是习惯性全走强同步（保险）或全走不同步（省事）？**

把同步策略做成"数据分类 + 自动路由"的框架约束。第一，定义 SyncCategory 装饰器/标签，开发者在定义状态时必须声明类别（@sync('strong') / @sync('weak') / @sync('none')），不声明的字段默认 none（不同步），强制开发者做有意识的选择；第二，WindowSyncManager 根据类别自动路由——strong 走实时广播、weak 走节流广播、none 完全不经过 IPC，开发者不需要手动调 broadcast 方法；第三，Code Review 检查清单加一条："新增的共享字段是否选对了同步类别？事实数据（任务/产物/权限）必须 strong，UI 状态（选中/滚动/输入）必须 none"；第四，提供同步诊断面板（开发模式）——可视化展示当前所有窗口的状态版本号和同步消息频率，如果某个 strong 字段的消息频率异常高（说明被高频修改），提示考虑细化粒度或改用 weak。这样同步策略就从"每开发者自己判断"变成了"框架强制分类 + 工具辅助诊断"。

## 结构化回答




**30 秒电梯演讲：** 就像团队协作写文档——核心内容（任务和产物）存在共享服务器上（全局同步），但每个人自己的编辑器界面（字号/主题/滚动位置）各自独立，不互相干扰。

**展开框架：**
1. **全局任务数据** — 全局任务数据和局部展示状态分开
2. **UI** — 跨窗口共享只同步高价值事实，不同步所有UI细节
3. **冲突时先提示哪个窗口持** — 冲突时先提示哪个窗口持有编辑权

**收尾：** 如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
