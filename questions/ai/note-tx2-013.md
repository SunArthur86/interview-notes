# 【某讯面经】并发锁、熔断、限流、降级实现思路

## 一、四件套定位

```
请求流入
  │
  ▼
[限流] 挡住超额流量（令牌桶/漏桶）
  │
  ▼
[并发锁] 保护共享资源（分布式锁）
  │
  ▼
[业务调用]
  ├─ 调下游 → [熔断] 下游故障快速失败
  └─ 主链路挂 → [降级] 兜底方案
```

## 二、并发锁：防资源竞争

### 分布式锁（多实例场景）
```python
# Redis 实现（SETNX + TTL）
def acquire_lock(key, ttl=10):
    token = str(uuid.uuid4())
    # SET key token NX EX ttl（原子操作）
    if redis.set(key, token, nx=True, ex=ttl):
        return token  # 拿到锁
    return None       # 锁被占

def release_lock(key, token):
    # Lua 脚本保证"检查+删除"原子
    script = """
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
    else
        return 0
    end
    """
    redis.eval(script, 1, key, token)
```

### 用途
- **防止重复执行**：定时任务多实例只跑一个
- **保护共享状态**：并发改会话状态
- **限并发**：信号量（限制同时N个请求）

### 陷阱
- **死锁**：持锁进程崩溃不释放 → TTL 兜底（设过期时间）
- **误删**：A 的锁被 B 删 → token 校验（释放时验证持有者）
- **续期**：业务超 TTL → 看门狗续期（Redisson 实现）

## 三、熔断：防下游故障蔓延

### 熔断器三状态
```
[关闭] 正常调用
  │ 错误率 > 阈值（如50%）
  ▼
[打开] 快速失败（不调下游，直接返回错误/降级）
  │ 等待恢复期（如30s）
  ▼
[半开] 放少量请求试探
  ├─ 成功 → 回到[关闭]
  └─ 失败 → 回到[打开]
```

### 实现（Python 示例）
```python
from circuitbreaker import circuit

@circuit(failure_threshold=5, recovery_timeout=30)
def call_llm(prompt):
    return llm.invoke(prompt)
    # 连续失败5次 → 熔断30s
    # 期间调用直接抛 CircuitBreakerError
```

### 触发条件
- **错误率**：错误数/总请求数 > 阈值（如 50%）
- **慢调用率**：超时请求比例 > 阈值
- **连续失败数**：连续 N 次失败

### 为什么要熔断
```
不熔断：
  下游LLM挂 → 我的请求都超时（30s）→ 线程池占满 → 我也挂 → 雪崩

熔断：
  下游LLM挂 → 错误率上升 → 熔断打开 → 快速失败（10ms）→ 降级处理 → 我还活着
```

## 四、限流：防流量过载

### 令牌桶（允许突发）
```
桶里有令牌，每秒加 R 个（速率）
请求来了，拿一个令牌：
  有令牌 → 通过
  没令牌 → 拒绝/排队

特点：
  - 允许突发（桶满时一次性用完）
  - 平均速率 = R
```

```python
# Redis + Lua 实现令牌桶
def token_bucket(key, capacity, rate):
    script = """
    local current = tonumber(redis.call('get', KEYS[1]) or capacity)
    if current > 0 then
        redis.call('decr', KEYS[1])
        redis.call('expire', KEYS[1], 1)
        return 1  -- 允许
    else
        return 0  -- 拒绝
    end
    """
    return redis.eval(script, 1, key)
```

### 漏桶（匀速）
```
请求进桶，以固定速率 R 漏出（处理）
桶满了 → 拒绝

特点：
  - 严格匀速（不允许突发）
  - 适合保护下游（如LLM只能10 QPS）
```

### 对比
| 算法 | 突发 | 匀速 | 适合 |
|------|------|------|------|
| 令牌桶 | ✅ | 平均 | 入口限流（允许偶尔突发） |
| 漏桶 | ❌ | 严格 | 保护下游（严格匀速） |

## 五、降级：主链路挂了用兜底

### LLM 服务的降级链
```
[主] 大模型（混元 pro）
  │ 挂/超时
  ▼
[降级1] 小模型（混元 lite）
  │ 也挂
  ▼
[降级2] 缓存结果（相似query命中）
  │ 没命中
  ▼
[降级3] 规则引擎（确定性回复）
  │ 规则也没有
  ▼
[兜底] "服务繁忙，请稍后重试 / 转人工"
```

### 降级策略
- **返回默认值**：非核心功能直接返回默认
- **返回缓存**：相似请求的旧结果
- **简化逻辑**：Agent 挂 → 退化成简单 RAG
- **异步处理**：返回"处理中"，结果后通知

### 降级原则
- **核心功能优先保**：客服系统，回答能力 > 个性化
- **降级要透明**：告知用户"当前为简化模式"
- **降级要可恢复**：主链路恢复后自动切回

## 六、Agent 系统的综合应用

```
请求入口
  ↓
[限流] 每用户 10 QPS，全局 1000 QPS（令牌桶）
  ↓
[并发锁] 会话状态更新加锁（防并发改冲突）
  ↓
Agent 编排
  ├─ 调 LLM
  │   ↓
  │  [熔断] 错误率 > 50% → 熔断 → [降级] 用小模型
  │   ↓
  │  [超时] 30s 超时 → [降级] 缓存/规则
  │
  ├─ 调工具
  │   ↓
  │  [熔断] 工具挂 → [降级] 跳过/默认值
  │
  └─ 调向量库
      ↓
     [熔断] 向量库挂 → [降级] 用 BM25 或缓存
```

## 七、加分点

- 说出 **熔断防雪崩**：不熔断会导致"下游挂→我超时→我也挂→全链路雪崩"
- 说出 **降级链设计**：多层兜底，保证总有答案返回
- 说出 **令牌桶 vs 漏桶**：令牌桶允许突发（入口），漏桶严格匀速（保护下游）

## 八、雷区

- ❌ 只限流不熔断 → 下游挂了自己也跟着挂
- ❌ 降级没有兜底 → 最后返回 500，用户体验差
- ❌ 分布式锁不设 TTL → 持锁崩溃导致死锁

## 九、扩展

- **Sentinel / Hystrix / Resilience4j**：成熟的熔断限流降级框架
- **自适应限流**：根据系统负载动态调整限流阈值（如 CPU/RT 自适应）
- ** chaos engineering **：故意注入故障测试降级是否生效

## 记忆要点

- 限流挡超额流量：令牌桶允许突发，漏桶严格匀速，入口防过载选令牌桶
- 并发锁防竞争：分布式锁需设TTL防死锁，释放时必须用Lua校验token防误删
- 熔断防下游雪崩：错误率超阈值快速失败不等待，经历关闭-打开-半开三态恢复
- 降级保主链路：大模型故障则降级小模型、缓存或规则引擎，核心业务必须兜底


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：高可用四件套（锁、熔断、限流、降级）为什么必须组合，单用一个不行吗？**

四者防护不同故障，单用有漏洞。1) 只用限流——挡住入口洪水，但下游故障时仍会拖垮自己（要熔断）；2) 只用熔断——下游故障快速失败，但入口洪水仍压垮（要限流）；3) 只用降级——兜底返回，但资源已被耗尽（要限流+熔断先保命）；4) 只用锁——防数据竞争，但不防流量过载。四者协同：限流挡入口、熔断防下游蔓延、降级保核心可用、锁防数据竞争。这是"纵深防御"。

### 第二层：证据与定位

**Q：系统开了四件套仍偶发雪崩，怎么定位是哪一环没生效？**

看各环的触发日志。1) 限流——是否触发（如果 QPS 没超阈值就没触发，可能是阈值设太高）；2) 熔断——下游故障时是否打开（如果错误率达标但没熔断，是配置错）；3) 降级——熔断后是否走降级（如果没降级逻辑，熔断了直接报错）；4) 锁——并发冲突时是否生效（如果数据竞争了，是锁粒度不够）。用监控看每个防护的触发次数和时机，定位"该触发没触发"的环节。

### 第三层：根因深挖

**Q：限流配了令牌桶但仍被流量打挂，根因是限流阈值不对还是限流位置不对？**

通常是限流位置不对。如果限流只配在"应用层"（请求进来后），DDoS 流量已经打满了网络带宽或负载均衡器，应用层限流来不及生效。根因是"流量在到达应用前就压垮了基础设施"。解法：多层限流——1) 网络层（防火墙/WAF）防 DDoS；2) 负载均衡层（Nginx/网关）限连接数；3) 应用层（令牌桶）限 QPS。每层挡一类流量。单靠应用层限流防不住基础设施层的攻击。

**Q：那为什么不直接扩容（加机器），扛住流量，而要限流？**

成本和有效性。1) 成本——按峰值流量扩容，平时 90% 机器闲置，浪费；2) 有效性——某些流量（恶意攻击、下游故障引起的重试洪水）扩容也扛不住（会继续累积）。限流是"主动控制流量在系统能力范围内"，扩容是"被动提升系统能力"。两者配合：扩容提升常态能力，限流挡异常峰值。纯靠扩容是"用钱解决问题"，限流是"用工程解决问题"。

### 第四层：方案权衡

**Q：熔断用"错误率触发"还是"超时触发"，怎么选？**

两者互补。错误率触发——下游返回错误多（如 5xx > 50%），说明下游故障，熔断；超时触发——下游响应慢（如 P99 > 5s），可能是下游过载，熔断避免自己被拖死。经验上两者都配：1) 错误率 > 50% 持续 10 秒 → 熔断 30 秒；2) 超时率 > 30% 持续 10 秒 → 熔断。错误率熔断防"错误传播"，超时熔断防"资源占用"。Hystrix/resilience4j 都支持两者组合。

**Q：为什么不直接用"重试"解决瞬时故障，而要熔断？**

重试会放大流量。如果下游因过载而偶发失败，客户端重试会让下游承受 2 倍流量，加剧过载，形成"重试风暴"。熔断是"发现下游不行了就别再试了"，快速失败避免加重下游负担。经验策略：重试有限次（1-2 次）+ 熔断兜底（多次失败后停止重试）。纯重试是"赌下游会恢复"，熔断是"承认下游不行了保护自己"。

### 第五层：验证与沉淀

**Q：怎么验证高可用四件套配置合理（不多不少）？**

故障注入测试（Chaos Engineering）。主动注入：1) 流量洪峰——限流是否触发、拒绝是否优雅；2) 下游故障——熔断是否打开、降级是否生效；3) 并发冲突——锁是否正确。衡量：1) 系统在故障下是否仍服务（可用性 > 99%）；2) 恢复时间（MTTR < 5 分钟）；3) 是否有"误触发"（正常流量被限流/熔断）。沉淀为高可用配置规范：每件套的触发阈值、降级策略、监控告警。

## 结构化回答

**30 秒电梯演讲：** 高可用四件套——并发锁(防资源竞争，分布式锁SETNX+TTL)、熔断(下游故障时快速失败不拖垮自己，错误率/超时阈值触发)、限流(防流量过载，令牌桶/漏桶)、降级(主链路挂了用兜底，如LLM挂→规则引擎)。

**展开框架：**
1. **并发锁** — 分布式锁(SETNX+TTL)防资源竞争
2. **熔断** — 下游故障快速失败，错误率/超时阈值触发
3. **限流** — 令牌桶(QPS)/漏桶(匀速)防流量过载
