# 【某讯面经】并发锁、熔断、限流、降级实现思路

## 一、四件套定位

```
请求流入
  │
  ▼
[限流] 挡住超额流量（令牌桶/漏桶）
  │
  ▼
[并发锁] 保护共享资源（分布式锁）
  │
  ▼
[业务调用]
  ├─ 调下游 → [熔断] 下游故障快速失败
  └─ 主链路挂 → [降级] 兜底方案
```

## 二、并发锁：防资源竞争

### 分布式锁（多实例场景）
```python
# Redis 实现（SETNX + TTL）
def acquire_lock(key, ttl=10):
    token = str(uuid.uuid4())
    # SET key token NX EX ttl（原子操作）
    if redis.set(key, token, nx=True, ex=ttl):
        return token  # 拿到锁
    return None       # 锁被占

def release_lock(key, token):
    # Lua 脚本保证"检查+删除"原子
    script = """
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
    else
        return 0
    end
    """
    redis.eval(script, 1, key, token)
```

### 用途
- **防止重复执行**：定时任务多实例只跑一个
- **保护共享状态**：并发改会话状态
- **限并发**：信号量（限制同时N个请求）

### 陷阱
- **死锁**：持锁进程崩溃不释放 → TTL 兜底（设过期时间）
- **误删**：A 的锁被 B 删 → token 校验（释放时验证持有者）
- **续期**：业务超 TTL → 看门狗续期（Redisson 实现）

## 三、熔断：防下游故障蔓延

### 熔断器三状态
```
[关闭] 正常调用
  │ 错误率 > 阈值（如50%）
  ▼
[打开] 快速失败（不调下游，直接返回错误/降级）
  │ 等待恢复期（如30s）
  ▼
[半开] 放少量请求试探
  ├─ 成功 → 回到[关闭]
  └─ 失败 → 回到[打开]
```

### 实现（Python 示例）
```python
from circuitbreaker import circuit

@circuit(failure_threshold=5, recovery_timeout=30)
def call_llm(prompt):
    return llm.invoke(prompt)
    # 连续失败5次 → 熔断30s
    # 期间调用直接抛 CircuitBreakerError
```

### 触发条件
- **错误率**：错误数/总请求数 > 阈值（如 50%）
- **慢调用率**：超时请求比例 > 阈值
- **连续失败数**：连续 N 次失败

### 为什么要熔断
```
不熔断：
  下游LLM挂 → 我的请求都超时（30s）→ 线程池占满 → 我也挂 → 雪崩

熔断：
  下游LLM挂 → 错误率上升 → 熔断打开 → 快速失败（10ms）→ 降级处理 → 我还活着
```

## 四、限流：防流量过载

### 令牌桶（允许突发）
```
桶里有令牌，每秒加 R 个（速率）
请求来了，拿一个令牌：
  有令牌 → 通过
  没令牌 → 拒绝/排队

特点：
  - 允许突发（桶满时一次性用完）
  - 平均速率 = R
```

```python
# Redis + Lua 实现令牌桶
def token_bucket(key, capacity, rate):
    script = """
    local current = tonumber(redis.call('get', KEYS[1]) or capacity)
    if current > 0 then
        redis.call('decr', KEYS[1])
        redis.call('expire', KEYS[1], 1)
        return 1  -- 允许
    else
        return 0  -- 拒绝
    end
    """
    return redis.eval(script, 1, key)
```

### 漏桶（匀速）
```
请求进桶，以固定速率 R 漏出（处理）
桶满了 → 拒绝

特点：
  - 严格匀速（不允许突发）
  - 适合保护下游（如LLM只能10 QPS）
```

### 对比
| 算法 | 突发 | 匀速 | 适合 |
|------|------|------|------|
| 令牌桶 | ✅ | 平均 | 入口限流（允许偶尔突发） |
| 漏桶 | ❌ | 严格 | 保护下游（严格匀速） |

## 五、降级：主链路挂了用兜底

### LLM 服务的降级链
```
[主] 大模型（混元 pro）
  │ 挂/超时
  ▼
[降级1] 小模型（混元 lite）
  │ 也挂
  ▼
[降级2] 缓存结果（相似query命中）
  │ 没命中
  ▼
[降级3] 规则引擎（确定性回复）
  │ 规则也没有
  ▼
[兜底] "服务繁忙，请稍后重试 / 转人工"
```

### 降级策略
- **返回默认值**：非核心功能直接返回默认
- **返回缓存**：相似请求的旧结果
- **简化逻辑**：Agent 挂 → 退化成简单 RAG
- **异步处理**：返回"处理中"，结果后通知

### 降级原则
- **核心功能优先保**：客服系统，回答能力 > 个性化
- **降级要透明**：告知用户"当前为简化模式"
- **降级要可恢复**：主链路恢复后自动切回

## 六、Agent 系统的综合应用

```
请求入口
  ↓
[限流] 每用户 10 QPS，全局 1000 QPS（令牌桶）
  ↓
[并发锁] 会话状态更新加锁（防并发改冲突）
  ↓
Agent 编排
  ├─ 调 LLM
  │   ↓
  │  [熔断] 错误率 > 50% → 熔断 → [降级] 用小模型
  │   ↓
  │  [超时] 30s 超时 → [降级] 缓存/规则
  │
  ├─ 调工具
  │   ↓
  │  [熔断] 工具挂 → [降级] 跳过/默认值
  │
  └─ 调向量库
      ↓
     [熔断] 向量库挂 → [降级] 用 BM25 或缓存
```

## 七、加分点

- 说出 **熔断防雪崩**：不熔断会导致"下游挂→我超时→我也挂→全链路雪崩"
- 说出 **降级链设计**：多层兜底，保证总有答案返回
- 说出 **令牌桶 vs 漏桶**：令牌桶允许突发（入口），漏桶严格匀速（保护下游）

## 八、雷区

- ❌ 只限流不熔断 → 下游挂了自己也跟着挂
- ❌ 降级没有兜底 → 最后返回 500，用户体验差
- ❌ 分布式锁不设 TTL → 持锁崩溃导致死锁

## 九、扩展

- **Sentinel / Hystrix / Resilience4j**：成熟的熔断限流降级框架
- **自适应限流**：根据系统负载动态调整限流阈值（如 CPU/RT 自适应）
- ** chaos engineering **：故意注入故障测试降级是否生效

## 记忆要点

- 限流挡超额流量：令牌桶允许突发，漏桶严格匀速，入口防过载选令牌桶
- 并发锁防竞争：分布式锁需设TTL防死锁，释放时必须用Lua校验token防误删
- 熔断防下游雪崩：错误率超阈值快速失败不等待，经历关闭-打开-半开三态恢复
