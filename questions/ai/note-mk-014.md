# 【月之暗面面经】如果产品要加更多 Agent，前端怎样保持扩展能力而不是越做越重？

## 一、反模式：一Agent一工作台

```
❌ 错误做法：
  代码生成Agent → 自己的输入框 + 自己的产物面板 + 自己的设置页
  PPT生成Agent  → 自己的输入框 + 自己的产物面板 + 自己的设置页
  数据分析Agent → 自己的输入框 + 自己的产物面板 + 自己的设置页
  网站生成Agent → 自己的输入框 + 自己的产物面板 + 自己的设置页

  结果：4个Agent = 4套UI = 维护噩梦
  增加第5个Agent = 再写一套UI
```

## 三、三套统一协议

### 1. 输入协议（Input Protocol）

```typescript
// 所有Agent使用统一的输入格式
interface AgentInput {
  // Agent标识
  agentType: string;
  
  // 用户输入
  rawInput: string;              // 原始输入（命令或自然语言）
  
  // 上下文
  inputRefs: InputRef[];         // 引用的文件/URL
  contextSummary: string;        // 上下文摘要
  
  // 参数
  params: Record<string, any>;   // 结构化参数
  
  // 偏好
  preferences: {
    outputFormat?: string;       // 输出格式偏好
    language?: string;
    style?: string;
  };
}

// 每个Agent声明自己接受的输入类型
interface AgentCapability {
  agentType: string;
  displayName: string;
  
  // 接受的输入类型
  acceptedInputs: {
    fileTypes: string[];         // ['.md', '.pdf', '.png']
    urlSchemes: string[];        // ['https://', 'file://']
    textInput: boolean;          // 是否接受纯文本输入
  };
  
  // 输出类型
  outputTypes: ArtifactKind[];   // ['site', 'ppt', 'sheet']
  
  // 命令别名
  commands: string[];            // ['/gen-site', '/make-website']
}
```

### 2. 结果协议（Result Protocol）

```typescript
// 所有Agent返回统一的结果格式
interface AgentResult {
  // 标识
  taskId: string;
  agentType: string;
  
  // 状态
  status: 'success' | 'partial' | 'failed';
  
  // 产物
  artifacts: Artifact[];         // 统一产物格式
  
  // 执行过程
  executionLog: ExecutionStep[]; // 统一执行日志
  
  // 元数据
  metadata: {
    duration: number;
    tokenUsage: { input: number; output: number };
    cost?: number;
  };
  
  // 下一步建议
  nextActions?: SuggestedAction[];
}
```

### 3. 授权协议（Permission Protocol）

```typescript
// 所有Agent使用统一的权限声明
interface AgentPermissionSpec {
  agentType: string;
  
  // 必需权限
  required: PermissionRequirement[];
  
  // 可选权限
  optional?: PermissionRequirement[];
  
  // 数据处理声明
  dataHandling: {
    localProcessing: boolean;    // 是否本地处理
    cloudUpload: boolean;        // 是否上传云端
    dataRetention: string;       // 数据保留策略
  };
}
```

## 四、Agent适配层

```typescript
// 新Agent只需实现适配器接口
interface AgentAdapter {
  // 能力声明
  capability: AgentCapability;
  
  // 权限声明
  permissions: AgentPermissionSpec;
  
  // 执行
  execute(input: AgentInput): Promise<AgentResult>;
  
  // 流式输出（可选）
  streamExecute?(input: AgentInput): AsyncGenerator<StreamChunk>;
  
  // 产物预览（可选，如果需要特殊渲染）
  renderPreview?(artifact: Artifact): Vue.VNode;
}

// 注册新Agent只需几行代码
agentRegistry.register({
  capability: {
    agentType: 'code-review',
    displayName: '代码审查',
    acceptedInputs: { fileTypes: ['.ts', '.js', '.py'], textInput: true },
    outputTypes: ['document'],
    commands: ['/review', '/code-review'],
  },
  permissions: {
    required: [{ type: 'file-read', target: 'selected' }],
  },
  execute: async (input) => { /* ... */ },
});
```

## 六、常见坑

- **每个Agent一套工作台**：UI代码膨胀，新Agent接入成本高
- **协议不统一**：每个Agent定义自己的输入输出格式，平台层无法复用
- **差异散落在平台层**：Agent的特殊逻辑侵入共用组件，if-else越来越多
- **没有能力描述**：用户不知道新Agent能做什么、接受什么输入

## 记忆要点

- 坚决反模式：拒绝一Agent一套UI，避免重复造轮子导致维护成本呈指数级增长。
- 平台加插件架构：抽象统一输入、任务中心、产物面板作为平台底座，各Agent作为适配层接入。
- 统一三大协议：通过标准化的输入、结果、授权协议，实现底层与具体Agent的解耦。
- 配置化扩展：新增Agent只需编写配置文件描述其输入约束和产物渲染器，前端零侵入。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么不能给每个 Agent 做一套独立的工作台（输入框+产物面板+任务中心），非要抽象成"平台+插件"架构？每个 Agent 做自己的 UI 不是更灵活吗？**

因为"每个 Agent 一套 UI"会让维护成本随 Agent 数量呈指数级增长，最终压垮团队。如果有 N 个 Agent 和 M 个共用模块（输入框/任务中心/产物面板/权限/通知/回放），独立 UI 模式下每个 Agent 都要实现 M 个模块，总复杂度是 N×M。4 个 Agent 时是 4×6=24 个模块，还能维护；20 个 Agent 时是 20×6=120 个模块，团队完全无法维护——每个共用模块的 bug 要修 20 处，每个 UI 升级要改 20 套代码。"灵活"的代价是"重复造轮子"。"平台+插件"架构把 M 个共用模块抽象为平台层（只实现一次），每个 Agent 只提供差异化的适配层（AgentAdapter），总复杂度降为 N+M——20 个 Agent 时是 20+6=26 个模块，是独立 UI 的 1/5。而且新 Agent 接入时间从"周级"（重写一套 UI）降到"小时级"（写适配器 + 注册）。灵活性的丧失通过"适配层可定制"来弥补——Agent 可以提供自定义渲染器（renderPreview）和特殊参数，但不触碰平台层的共用逻辑。

### 第二层：证据与定位

**Q：你怎么定位"新 Agent 接入成本过高"是架构设计问题，而不是 Agent 本身功能太复杂？**

用"接入成本拆解"定位。把新 Agent 的接入工时按模块分解：(1) 实现 execute 业务逻辑（Agent 的核心功能，这是不可压缩的）；(2) 写输入框/任务中心/产物面板/权限页面/通知/回放页等 UI 模块（这些应该是平台层复用的）；(3) 调试协议对接（输入协议/结果协议/授权协议的适配）。如果 (2) 的工时占比超过 50%，说明架构有问题——Agent 在重复实现本应由平台层提供的 UI 模块；如果 (2) 的工时占比低于 20% 且 (1) 占主导，说明架构合理，成本主要在业务逻辑（不可压缩）；如果 (3) 的工时占比高，说明协议设计不够清晰或 Agent 的输入输出与协议不匹配，需要优化协议或加适配层。还可以横向对比——如果不同 Agent 的接入工时差异大（Agent A 接入 1 天，Agent B 接入 2 周），且差异主要在 (2) UI 模块上，说明平台层的能力覆盖不全，某些 Agent 被迫自建 UI。

### 第三层：根因深挖

**Q：为什么用三套协议（输入/结果/授权）做平台与 Agent 的解耦边界，而不是用更多协议（如再加"执行协议""通知协议"）做更细粒度的解耦？**

因为"协议数量"和"接入成本"是正相关的——协议越多，Agent 要实现的接口越多，接入越复杂。三套协议是覆盖"Agent 全生命周期关键契约"的最小完备集：输入协议定义"Agent 接受什么"（acceptedInputs/params），结果协议定义"Agent 返回什么"（artifacts/executionLog/status），授权协议定义"Agent 需要什么权限"（required/dataHandling）。这三者覆盖了"用户给 Agent 输入→Agent 执行→Agent 产出结果→Agent 用了什么权限"的完整链路。执行过程和通知不属于 Agent 的契约——执行过程由 TaskRunner 框架统一编排（Agent 只需在 execute 里调 recordExecution 钩子，不需要实现"执行协议"），通知由平台层根据任务状态自动分发（Agent 不需要关心通知逻辑）。如果把执行和通知也做成协议，Agent 要多实现两个接口，但这两个接口的逻辑对所有 Agent 都一样（都是标准化的），让 Agent 各自实现是重复劳动。所以三套协议的根因是"只把 Agent 间真正有差异的部分做成契约，相同的部分由平台层统一实现"——这是接口最小化原则。

**Q：那如果某个 Agent 有特殊的执行模式（如需要用户中途确认某步骤），为什么不在"执行协议"里支持这种差异，而要逼它套用统一的 execute 接口？**

因为"需要用户中途确认"不是"执行协议的差异"，而是"执行过程中的状态变更"，可以通过 TaskRunner 的交互钩子解决，不需要 Agent 自己实现执行协议。具体方案：Agent 的 execute 函数遇到需要确认的步骤时，调用 TaskRunner.requestUserConfirm(question) 钩子，TaskRunner 暂停任务并把状态置为 review，平台层的任务中心自动展示"需确认"提示，用户确认后 TaskRunner 恢复 execute 继续执行。这样 Agent 仍然只实现统一的 execute 接口，"中途确认"是通过框架提供的钩子实现的，不需要额外的执行协议。如果做成"执行协议"让 Agent 自己实现（如定义 onConfirm/onPause/onResume 接口），每个 Agent 都要写一套暂停-恢复逻辑，重复且容易出 bug。通过 TaskRunner 统一托管的交互钩子（requestUserConfirm/requestUserInput/requestFileSelect），所有 Agent 共享同一套交互能力，Agent 只需在需要时调钩子。所以根因是"差异化的交互需求应该用平台层的钩子机制满足，而非让 Agent 各自实现协议"。

### 第四层：方案权衡

**Q：Agent 适配层你用"配置化注册"（agentRegistry.register）还是"约定大于配置"（如自动扫描 agents/ 目录），怎么选？**

选显式配置化注册，不选自动扫描。根因是"可控性"和"可发现性"。配置化注册要求开发者显式调用 register 并传入 capability/permissions/execute，好处是：(1) 注册位置明确——想知道系统有哪些 Agent，看注册文件（如 agents/index.ts）即可，不用全局搜索；(2) 注册参数明确——开发者必须声明 capability（接受什么输入）和 permissions（需要什么权限），平台层据此做校验和路由，不会出现"未声明能力但被调用"的幽灵 Agent；(3) 可条件注册——某些 Agent 可以根据环境条件决定是否注册（如企业版才注册的 Agent），自动扫描做不到。自动扫描（约定大于配置）的问题是：(1) 隐式注册——加了文件就自动注册，开发者可能不知道某个 Agent 被意外激活（如调试文件没删）；(2) 能力声明分散——每个 Agent 文件各自声明能力，没有统一的注册表可查；(3) 注册顺序不可控——扫描顺序依赖文件系统，可能导致依赖顺序问题。配置化的代价是"多写一行 register 代码"，但收益是"注册可追溯、能力可审计、条件可控制"，远大于代价。

**Q：那如果团队觉得"每个 Agent 都要写 register 太重复"，为什么不做成自动扫描 + 装饰器（如 @RegisterAgent）减少样板代码？**

因为"装饰器自动注册"和"配置化注册"不冲突——可以用装饰器减少样板代码，但仍保留显式注册的可控性。具体方案：定义 @RegisterAgent(capability) 装饰器，它内部仍然调用 agentRegistry.register，只是把注册代码从"单独的 register 调用"变成了"类装饰器"，开发者不用单独写注册文件。但装饰器要求开发者显式标注（在类上加 @RegisterAgent），这仍然是"显式声明"——不是文件被扫描就自动注册，而是开发者主动加了装饰器才注册。这样既减少了样板代码（不用单独写 register），又保留了显式性（装饰器是可见的注册标记）。自动扫描（文件被发现就注册）和装饰器（开发者主动标注才注册）的区别在于"注册的触发条件"——前者是隐式的（文件存在就触发），后者是显式的（开发者意图触发）。选装饰器方案是"优化配置化的写法"，不是"改成自动扫描"。所以根因是"减少样板代码要用装饰器/宏等语法糖，而非放弃显式注册的语义"。

### 第五层：验证与沉淀

**Q：你怎么证明"平台+插件"架构比"一 Agent 一套 UI"更优，新 Agent 接入成本真的降低了吗？**

用工时指标和代码指标验证：(1) 新 Agent 平均接入工时——平台化之前（一 Agent 一套 UI）通常需要 1-2 周（写 UI + 调对接），平台化之后应降到 1-3 天（写适配器 + 注册），工时降低 70% 以上；(2) 新 Agent 代码量——平台化前每个 Agent 约 3000-5000 行（含 UI），平台化后应降到 500-1000 行（仅业务逻辑 + 适配器），代码量降低 80%；(3) 共用模块的 bug 修复成本——平台化前修一个"产物面板"的 bug 要改 N 个 Agent 的产物面板代码，平台化后只改 1 处（平台层），修复成本从 O(N) 降到 O(1)。收集团队过去半年的 Agent 接入数据，对比平台化前后的这三个指标，如果都显著改善就证明架构有效。还可以看一个定性指标：团队是否敢于快速尝试新 Agent——平台化后因为接入成本低，团队应该更愿意实验新 Agent（每月新接入数增加），而不再因为"又要写一套 UI"而抗拒。

**Q：怎么让团队在接入新 Agent 时，自觉走"平台+插件"架构（实现 Adapter + 注册），而不是习惯性地自己写一套独立 UI？**

把"走平台架构"做成"唯一可用的接入路径"。第一，平台层模块（输入框/任务中心/产物面板/权限/通知/回放）不暴露给 Agent 直接使用的 API——Agent 只能通过 AgentAdapter 接口接入，想自己写 UI 就无法调到平台层的能力（如无法注册任务、无法发通知），独立 UI 跑不起来；第二，AgentAdapter 接口是唯一注册入口——只有通过 agentRegistry.register 注册的 Agent 才能被输入路由器发现和调用，未注册的 Agent 完全不可达；第三，脚手架 CLI 一键生成 Adapter 骨架——`npm run new-agent code-review` 自动产出实现 AgentAdapter 的类骨架、capability 声明模板、permissions 声明模板、注册代码、测试文件，开发者照着填业务逻辑，写独立 UI 反而比填模板更费劲；第四，Code Review 加硬规则——拒绝任何在 Agent 目录下出现 UI 组件（.vue 文件）的代码，Agent 目录只允许有 adapter.ts/capability.ts/permissions.ts，UI 渲染必须走平台层或自定义渲染器（renderPreview）。这样平台架构就从"推荐规范"变成了"唯一路径"。

## 结构化回答

**30 秒电梯演讲：** 要让前端在接入更多Agent时保持扩展能力，核心是统一三套协议（输入协议、结果协议、授权协议），把产物面板/任务中心/回放页做成共用模块，Agent差异放在能力描述和适配层，不让每个Agent自己重新做一套工作台。

**展开框架：**
1. **统一输入协** — 统一输入协议、结果协议和授权协议
2. **把产物面板** — 把产物面板、任务中心和回放页做成共用模块
3. **Agent** — Agent差异放在能力描述和适配层

**收尾：** 您想深入聊：如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
