# 【某讯面经】BM25 关键词检索 + 向量检索混合检索优势，只单向量会有什么问题

## 一、两种检索器的互补性

| 维度 | BM25 强项 | 向量检索强项 |
|------|----------|------------|
| 精确词面匹配 | ✅ 专有名词、代码、数字、SKU | ❌ |
| 同义/近义 | ❌ | ✅ "VPN连不上"↔"虚拟专网异常" |
| 跨语言 | ❌ | ✅ 中英文互检 |
| 概念查询 | ❌ | ✅ "怎么提速"↔"性能优化" |
| 拼写错误容忍 | ❌ | ✅（一定程度） |
| 长尾词 | ✅ | ❌（训练数据少） |

**核心互补**：BM25 抓"字面一样"，向量抓"意思相近"。

## 二、只单向量会有什么问题

### 问题1：专有名词漏召回
```
Query: "VPN 连不上怎么办"
向量可能召回: "网络连接异常的排查方法"（语义近）
但漏掉: "VPN 配置指南"（字面含 VPN，但语义距离可能远）

BM25 能召回: "VPN 配置指南"（字面匹配 VPN）
```

### 问题2：数字/代码场景弱
```
Query: "错误码 5002 怎么解决"
向量: 可能召回各种"错误"相关文档，但 5002 这个具体数字权重低
BM25: 精确匹配 "5002"，直接命中
```

### 问题3：长尾词/罕见术语
```
Query: "K8s NodeAffinity 配置"
向量: "K8s NodeAffinity" 是技术长尾词，embedding 训练数据少，表示不准
BM25: 精确匹配 "NodeAffinity"，稳
```

### 问题4：跨语言不如配合
```
Query: "如何 use Function Calling"
向量: 中英混杂，表示可能漂移
BM25: "Function Calling" 精确匹配英文文档
```

## 三、混合检索怎么融合

### 方案1：RRF（Reciprocal Rank Fusion）—— 推荐简单有效

```
RRF_score(d) = Σ 1 / (k + rank_i(d))

  d: 文档
  rank_i(d): 文档 d 在第 i 个检索器中的排名（第1名rank=1）
  k: 平滑常数，通常取 60
```

```python
def rrf(bm25_results, vector_results, k=60):
    scores = {}
    for rank, doc in enumerate(bm25_results, 1):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
    for rank, doc in enumerate(vector_results, 1):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
    return sorted(scores.items(), key=lambda x: -x[1])
```

**RRF 优点**：
- 不需要校准分数（BM25 和向量分数量纲不同，直接加权难）
- 只用排名，鲁棒
- 简单，工业界常用

**k 为什么取 60**：经验值，平衡"头部权重"和"长尾权重"。k 大→排名差异被平滑，k 小→头部权重高。

### 方案2：加权融合
```
final_score = α · normalize(bm25_score) + (1 - α) · normalize(vector_score)

α: 按场景调
  - 词面查询多（代码/型号）→ α 大（如 0.7）
  - 语义查询多（自然语言）→ α 小（如 0.3）
```

**缺点**：需要校准两个分数到同一量纲（normalize），调 α 麻烦。

## 四、完整混合检索流程

```
Query 进入
  │
  ├─ BM25 检索 → top-50（按词面）
  │
  ├─ 向量检索 → top-50（按语义）
  │       └─ query embedding（用 BGE / OpenAI embedding）
  │
  ▼
RRF 融合 → top-100 排名
  │
  ▼
Rerank（cross-encoder）→ top-5（精排）
  │
  ▼
喂给 LLM 生成答案
```

## 五、加分点

- 说出 **RRF 不需要分数校准**，比加权融合简单
- 说出 **BM25 的变体**：BM25F（字段加权，标题权重高于正文）、BM25+（处理长文档惩罚）
- 说出 **向量检索的过滤**：先按元数据过滤（如"只查2024年文档"）再向量检索，提升精度和速度

## 六、雷区

- ❌ "向量检索一定能替代 BM25" → 专有名词/数字/代码场景必漏
- ❌ 加权融合不 normalize → 分数量纲不同，加权无意义
- ❌ 只混合不 Rerank → 召回多但精度不够

## 七、扩展

- **Sparse Vector（稀疏向量）**：如 SPLADE，结合了 BM25 的精确性和向量的语义性，用学习的稀疏向量替代 BM25
- **Multi-vector**：ColBERT 风格，每个 token 一个向量，做 late interaction，精度更高
- **混合检索的工程实现**：Milvus/Qdrant/Weaviate 都原生支持混合检索（一个 API 同时查 BM25 + 向量）

## 记忆要点

- BM25重字面精确匹配（专有名词/数字），向量重语义近似匹配（同义/跨语言），两者互补
- 单向量短板：因为靠语义匹配，所以遇到精确代码、罕见长尾词、特定数字时极易漏召回
- 融合方案首选RRF：因为只用排名计算无需校准分数量纲，k常取60，比加权融合更简单鲁棒
- 工业级完整链路：双路召回Top-50 -> RRF融合 -> Cross-encoder重排Top-5 -> 喂给LLM


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：BM25 + 向量混合检索的优势是"互补"，但为什么不直接用更强的向量模型，让它覆盖 BM25 的能力？**

因为向量模型在"精确匹配"上天生弱。向量模型的语义相似度是"软匹配"——"VPN 连不上"和"虚拟专网异常"语义相似，向量召回；但"iPhone 15 Pro Max 256GB"这种精确产品名/数字/代码，向量相似度可能不高（向量更捕捉语义而非精确字面）。BM25 是 TF-IDF 的升级版，强在"精确词面匹配"。即使向量模型再强，专有名词、SKU 编号、代码标识符的精确匹配仍需要 BM25。所以两者互补不是"谁替代谁"。

### 第二层：证据与定位

**Q：混合检索后效果反而下降，怎么定位是融合方法（RRF）的问题还是两路召回的质量问题？**

分别看两路召回。1) BM25 单独的 recall@10 和向量单独的 recall@10——如果两者都低（< 70%），是召回质量问题（embedding 模型差或分词错）；2) 如果两路 recall 都高但融合后下降，是融合方法问题（RRF 的 k 参数或权重不对）。具体看融合后 top-10 里的 gold chunk，对比它在 BM25 排名和向量排名——如果 BM25 排第 2、向量排第 50，RRF 融合后可能排第 10（被向量的低排名拉下来），是融合权重问题。

### 第三层：根因深挖

**Q：只单向量检索的问题，根因是 embedding 模型能力不够还是向量方法本身有局限？**

向量方法本身有局限。向量是"把文本压缩成固定维度的语义表征"，压缩过程会丢失"精确字面信息"。如"订单号 ABC123"和"订单号 ABC124"的向量可能很接近（语义都是"订单号"），但它们是不同订单。这不是 embedding 模型不够强（即使最强的模型也难区分），是"向量压缩"的信息损失。根因是"语义表征和精确匹配是不同的任务"，单一表征无法同时做好两者。BM25 不压缩（直接用词面），所以精确匹配强。

**Q：那为什么不直接用更大的 embedding 模型（如 8B 参数）提高精度？**

收益递减且成本高。大 embedding 模型在"语义相似"上略强，但在"精确匹配"上的提升有限（因为根本问题是向量压缩损失字面信息，不是模型容量）。且大 embedding 模型的推理成本高（8B 模型编码一篇文档要几百毫秒），大规模检索时成本爆炸。BM25 是"计算便宜 + 精确匹配强"，加一个大 embedding 模型不如"中等 embedding + BM25 混合"。所以混合检索是"性价比最优"，不是"能力最强"。

### 第四层：方案权衡

**Q：BM25 和向量检索的结果融合用 RRF 还是加权分数，怎么选？**

RRF（Reciprocal Rank Fusion）——按排名融合（rank 1 得分 1/1，rank 2 得分 1/2...），不用校准分数。加权分数——BM25 分数 × α + 向量分数 × β，要校准两路分数的尺度。RRF 优势：简单、不用调参、对分数尺度不敏感；劣势：只看排名不看分数（rank 1 和 rank 2 的差距可能很大也可能很小，RRF 一视同仁）。加权分数优势：利用分数信息；劣势：两路分数尺度不同（BM25 是 TF-IDF 分数、向量是 cosine），校准难。经验上 RRF 更鲁棒、更常用，加权分数在"两路分数都校准好"时略优。

**Q：为什么不直接用 learning-to-rank（LTR，训练一个融合模型），而要用 RRF 这种启发式？**

LTR 效果上限更高但有成本。1) 训练数据——要标注"query + 文档 + 相关性"三元组，量大；2) 模型维护——LTR 模型要随数据变化重新训练；3) 复杂度——多一个模型要部署和推理。RRF 是零参数启发式，无需训练、部署简单、效果在多数场景接近 LTR（差距 < 5%）。所以"RRF 先做 baseline，如果效果不够再上 LTR"是合理路径，不要一上来就 LTR（过度工程）。

### 第五层：验证与沉淀

**Q：怎么衡量混合检索相比单一检索的增量价值？**

对比实验：1) 只 BM25 的 recall@10；2) 只向量的 recall@10；3) 混合（RRF）的 recall@10。混合应该 > max(BM25, 向量)，增量 > 5% 才值得维护两套系统。同时看 MRR（平均倒数排名）——混合应该让 gold chunk 排名更靠前。沉淀为混合检索配置规范：BM25 的分词器选择、embedding 模型选择、RRF 的 k 参数、两路的召回数（如各 top-20 融合成 top-10）。
