# 为什么选向量数据库而不是传统数据库？向量数据库如何选型对比？

## 一、为什么需要向量数据库

```
传统数据库（MySQL/PG）：
  查询: SELECT * FROM docs WHERE title = "Agent"
  本质：精确匹配 / 范围查询
  索引：B+树（擅长等值/范围）
  
  问题：无法做"语义相似"查询
  "查和'智能体'语义相近的文档" → 传统DB无能为力

向量数据库：
  查询: 找与 [0.1, 0.3, ...] 最相似的top-10向量
  本质：近似最近邻搜索（ANN）
  索引：HNSW / IVF / PQ（擅长高维相似度）
  
  能力：语义检索（"智能体"≈"Agent"≈"autonomous"）
```

## 二、传统 DB 硬做向量检索的问题

```python
# 传统DB硬做（不用向量库）
def naive_vector_search(query_vec, all_docs):
    """暴力计算所有文档的相似度"""
    scores = []
    for doc in all_docs:
        sim = cosine_similarity(query_vec, doc.vector)
        scores.append((doc, sim))
    return sorted(scores, key=lambda x: -x[1])[:10]

# 问题：
# 1. O(n)复杂度，100万文档要算100万次
# 2. 每次查询全表扫描，无法利用索引
# 3. 延迟：100万文档要几秒，不可接受

# 向量库用ANN索引：
# HNSW：O(logn)，100万文档<10ms
```

## 三、向量库的核心技术

```
┌──────────────────────────────────────────────┐
│              向量索引算法                       │
├──────────────────────────────────────────────┤
│                                                │
│  HNSW (Hierarchical Navigable Small World)    │
│    分层图结构，O(logn)查询                      │
│    精度高/速度快，内存占用大                    │
│    最主流的选择                                 │
│                                                │
│  IVF (Inverted File)                          │
│    先聚类分桶，只搜索相关桶                     │
│    速度快/内存省，精度略低                      │
│                                                │
│  PQ (Product Quantization)                    │
│    向量压缩，省内存                            │
│    精度有损，适合超大规模                       │
│                                                │
│  组合：IVF + PQ + HNSW                        │
│    大规模场景的常见组合                         │
│                                                │
└──────────────────────────────────────────────┘
```

## 五、选型决策树

```
你的场景是什么？
    │
    ├─ 原型验证/小项目(<100万向量)
    │   → Chroma（嵌入式，零运维）
    │
    ├─ 已有PostgreSQL，不想加组件
    │   → pgvector（SQL集成，简单）
    │
    ├─ 已有ES，需要全文+向量混合
    │   → Elasticsearch（一套搞定）
    │
    ├─ 中大规模，性能敏感
    │   → Qdrant（Rust高性能）
    │
    ├─ 企业级，超大规模(亿级)
    │   → Milvus（分布式，可扩展）
    │
    └─ 不想运维，快速上线
        → Pinecone（全托管SaaS）
```
