# 为什么选向量数据库而不是传统数据库？向量数据库如何选型对比？

## 一、为什么需要向量数据库

```
传统数据库（MySQL/PG）：
  查询: SELECT * FROM docs WHERE title = "Agent"
  本质：精确匹配 / 范围查询
  索引：B+树（擅长等值/范围）
  
  问题：无法做"语义相似"查询
  "查和'智能体'语义相近的文档" → 传统DB无能为力

向量数据库：
  查询: 找与 [0.1, 0.3, ...] 最相似的top-10向量
  本质：近似最近邻搜索（ANN）
  索引：HNSW / IVF / PQ（擅长高维相似度）
  
  能力：语义检索（"智能体"≈"Agent"≈"autonomous"）
```

## 二、传统 DB 硬做向量检索的问题

```python
# 传统DB硬做（不用向量库）
def naive_vector_search(query_vec, all_docs):
    """暴力计算所有文档的相似度"""
    scores = []
    for doc in all_docs:
        sim = cosine_similarity(query_vec, doc.vector)
        scores.append((doc, sim))
    return sorted(scores, key=lambda x: -x[1])[:10]

# 问题：
# 1. O(n)复杂度，100万文档要算100万次
# 2. 每次查询全表扫描，无法利用索引
# 3. 延迟：100万文档要几秒，不可接受

# 向量库用ANN索引：
# HNSW：O(logn)，100万文档<10ms
```

## 三、向量库的核心技术

```
┌──────────────────────────────────────────────┐
│              向量索引算法                       │
├──────────────────────────────────────────────┤
│                                                │
│  HNSW (Hierarchical Navigable Small World)    │
│    分层图结构，O(logn)查询                      │
│    精度高/速度快，内存占用大                    │
│    最主流的选择                                 │
│                                                │
│  IVF (Inverted File)                          │
│    先聚类分桶，只搜索相关桶                     │
│    速度快/内存省，精度略低                      │
│                                                │
│  PQ (Product Quantization)                    │
│    向量压缩，省内存                            │
│    精度有损，适合超大规模                       │
│                                                │
│  组合：IVF + PQ + HNSW                        │
│    大规模场景的常见组合                         │
│                                                │
└──────────────────────────────────────────────┘
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：向量数据库专为相似度搜索优化，但传统数据库（如 PostgreSQL）也支持向量扩展（pgvector），为什么还需要专用向量库？**

因为规模和性能差异。1）规模——专用向量库（Milvus/Qdrant）支持十亿级向量，针对 ANN（近似近邻）索引（HNSW/IVF）深度优化，亿级向量毫秒检索；pgvector 在百万级尚可，千万级以上性能急剧下降（索引构建慢、查询延迟高）；2）性能——专用向量库的 ANN 算法（HNSW 召回率 95%+ 且毫秒级）工程优化极致，pgvector 的 HNSW 实现较基础（召回/延迟差）；3）功能——专用向量库支持过滤+向量混合检索（标量过滤+向量相似度）、分布式扩展、多副本，pgvector 是单机扩展（分布式要自己搞）。选型：小规模（<千万）+ 已有 PostgreSQL 用 pgvector（省事）；大规模（亿级）+ 高性能要求用专用向量库。实务：原型用 pgvector（快），生产大规模用 Milvus/Qdrant。

### 第二层：证据与定位

**Q：向量库检索延迟从 10ms 飙到 500ms，怎么定位是数据量增长、索引退化，还是查询问题？**

分层排查。1）数据量——看向量总数，如果从百万增到千万，延迟升可能因数据量超索引容量（如 HNSW 的 ef 参数没调），检查数据量增长曲线和延迟曲线是否吻合；2）索引——HNSW 索引可能因频繁删除/更新导致图结构退化（碎片化），重建索引看延迟是否恢复，恢复则是索引退化；3）查询——看 query 向量的维度/过滤条件，高维（如 1536 维）比低维慢，复杂过滤（如多条件 AND）拖慢检索；4）资源——看 CPU/内存是否瓶颈（向量检索是算力密集），监控资源使用率。定位方法：对比延迟升前后的变化（数据量/索引状态/查询模式/资源），用 EXPLAIN（如 Milvus 的查询计划）看检索耗时分布。常见根因：数据量超预期（要调 HNSW 参数或分片）、索引退化（要重建）、资源不足（要扩容）。

### 第三层：根因深挖

**Q：向量库的 HNSW 索引召回率 95%，但漏召回的 5% 正好是高频 query 的 golden 文档，怎么提升这部分的召回？**

针对性优化。1）分析漏召回——这 5% 的漏召回 case 有什么特征（如 query 特别短/向量分布特殊/文档向量异常），找规律；2）调 HNSW 参数——增大 ef_construction（建图时邻居数，提升图质量）和 ef_search（查询时搜索邻居数，提升召回），如 ef_search 从 64 提到 128，召回升但延迟增，权衡；3）多索引互补——除了 HNSW 加 IVF 索引（另一种 ANN），两者漏召回的 case 不同，多索引检索+融合能互补；4）混合检索——加 BM25 关键词检索，向量漏召回的（如精确匹配 query）关键词可能召回，融合提升；5）rerank——召回 top-50（扩大候选，提升召回上限）再 rerank 选 top-5，把漏在 top-20 外的 golden 通过扩大召回+精排捞回来。原则：先诊断漏召回特征，针对性调参/加索引/混合检索，而非盲目提召回。

**Q：向量库选型要考虑"过滤+向量"混合检索（如"category=A 且相似度高"），但过滤太严会减少候选导致召回差，怎么平衡？**

过滤策略优化。1）先过滤后检索——先用标量过滤（category=A）筛出子集，再在子集内向量检索，过滤太严（子集太小）时候选少，相似度高的可能不在子集，召回差；2）先检索后过滤——先全库向量检索 top-N，再按标量过滤，但如果 top-N 里符合过滤的少（如 category=A 的不在 top-N），召回差；3）平衡策略——宽松过滤（如不只 category=A，放宽到 category in [A,B]）扩大候选，或调检索参数（扩大 top-N 再过滤），或用向量库的"过滤+检索一体化"（如 Milvus 在 ANN 遍历时边过滤，兼顾）；4）数据分布——如果 category=A 的数据占比小（如 1%），严格过滤必然候选少，考虑分库（每个 category 单独索引）或调业务（放宽过滤条件）。原则：过滤和召回要权衡，太严牺牲召回，太松引入噪声，按数据分布和业务需求调。

### 第四层：方案权衡

**Q：HNSW（图索引）和 IVF（聚类索引）是两种主流 ANN，召回率和延迟特性不同，怎么选？**

按场景选。1）HNSW——多层图结构，查询时从顶层粗搜到底层精搜，召回率高（95%+）、延迟低（毫秒）、但内存占用大（图结构）、构建慢；适合"内存充足+追求高召回低延迟"的场景（大多数在线检索）；2）IVF——聚类（如 K-means）把向量分桶，查询时只搜最近的几个桶，召回率中（调 nprobe 提升但延迟增）、内存省、构建快、支持大规模；适合"数据量超大规模（十亿级）+ 内存敏感"的场景。选型：默认 HNSW（召回/延迟好），内存瓶颈或超大规模用 IVF（省内存）。折中：IVF-HNSW（IVF 分桶+每桶内 HNSW），兼顾规模和召回，Milvus 等支持。

**Q：向量库要支持高频写入（如实时更新文档向量），但写入会触发索引更新（HNSW 重建慢），怎么平衡读写？**

读写分离 + 增量索引。1）读写分离——写副本（接收写入，索引实时更新）+ 读副本（只查询，定时同步），读不受写影响，但读副本有延迟（数据不是最新）；2）增量索引——HNSW 支持增量插入（加新向量到图，不重建全索引），但删除/更新会导致图碎片化（要定期 compaction 重建），控制写入频率（如批量写入而非单条）；3）分段索引——把索引分成多个 segment（如按时间），新数据写新 segment（小索引快），定期 merge 成大 segment，平衡写入和查询；4）业务层——非实时数据用批量更新（如每小时全量重建），实时数据用增量+读旧（查询时旧索引兜底，新索引异步构建）。原则：写快（增量/分段）+ 读稳（读写分离），按数据时效性要求选策略。

### 第五层：验证与沉淀

**Q：你怎么衡量向量库选型是否正确（满足业务需求）？**

四个维度。1）召回率——业务评估集上 Recall@K 是否达标（如 >90%），不达标说明 ANN 索引或参数不对；2）延迟——P99 检索延迟是否满足 SLA（如 <50ms），超 SLA 影响用户体验；3）吞吐——QPS 是否支持业务峰值（如 1000 QPS），不够要扩容；4）成本——资源成本（机器/内存）是否可接受，向量库是内存密集（贵），要评估性价比。综合：召回达标+延迟达标+吞吐够+成本可接受 = 选型正确。还要看扩展性（数据量翻倍时性能是否可接受，要扩容的话成本翻倍能否承受）。线上监控：召回（采样评估）、延迟（P99 告警）、吞吐（QPS 监控）、资源（CPU/内存水位）。

**Q：向量库的选型和运维怎么沉淀成团队的检索基础设施？**

建向量库平台：1）选型规范——按场景推荐（小规模 pgvector、大规模 Milvus/Qdrant），含选型决策树（数据量/延迟/成本）；2）运维工具——部署/监控/扩缩容自动化（如 K8s operator），索引健康检查（退化告警+自动重建），备份恢复；3）性能基准——定期跑性能基准（不同数据量/参数的召回+延迟），指导选型和调参；4）混合检索支持——内置向量+BM25 混合检索（统一接口），开发者不用自己拼；5）评估集成——接入业务评估集，召回退化告警。这套写入团队基础设施 SOP，让"用向量库"从"每个团队自己选型运维"变成"平台标准化能力"，开发者专注业务。
