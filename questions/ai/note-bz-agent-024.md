# 多轮对话的上下文如何建模？

## 一、核心矛盾

```
对话轮次增长 vs 上下文窗口有限

轮次1: [user1, ai1]                        ~100 tokens
轮次5: [u1,a1,...,u5,a5]                   ~500 tokens
轮次20: [u1,a1,...,u20,a20]               ~2000 tokens
轮次100: ...                              ~10000 tokens
                    │
                    ▼
         上下文窗口上限（如128K）

问题：轮次多了，要么超限报错，要么丢历史失忆
```

## 二、四种建模方法

### 方法 1：滑动窗口（最简单）

```
策略：只保留最近N轮，旧的丢弃

┌──────────────────────────────────┐
│  完整对话: [u1,a1,u2,a2,...,u20,a20]│
│                                    │
│  滑动窗口(保留最近5轮):             │
│  [u16,a16,u17,a17,...,u20,a20]     │
│  前面的丢弃                         │
└──────────────────────────────────┘

优点：实现极简，token可控
缺点：丢失早期重要信息（如用户姓名/需求）
```

```python
def sliding_window(messages, keep_recent=10):
    """保留最近N轮"""
    return messages[-keep_recent * 2:]  # 每轮2条(user+ai)
```

### 方法 2：摘要压缩（常用）

```
策略：旧对话压缩成摘要，最近对话保留原文

┌──────────────────────────────────┐
│  [摘要: "用户在咨询Python项目，    │ ← 旧对话压缩
│   偏好异步方案，已讨论了架构设计"]  │
│  [u18,a18,u19,a19,u20,a20]        │ ← 最近3轮原文
└──────────────────────────────────┘

优点：保留关键信息，token省
缺点：摘要有损，可能丢细节
```

```python
class SummaryMemory:
    def __init__(self, llm):
        self.llm = llm
        self.summary = ""
        self.recent = []  # 最近几轮原文
        self.SUMMARY_THRESHOLD = 10  # 每10轮压缩一次
    
    def add(self, turn):
        self.recent.append(turn)
        if len(self.recent) >= self.SUMMARY_THRESHOLD:
            # 压缩旧轮到摘要
            old = self.recent[:-3]  # 保留最近3轮
            new_summary = self.llm.summarize(
                f"已有摘要: {self.summary}\n"
                f"新增对话: {old}\n"
                f"请更新摘要，保留关键信息"
            )
            self.summary = new_summary
            self.recent = self.recent[-3:]
    
    def get_context(self):
        return [{"role": "system", "content": f"历史摘要: {self.summary}"}] + self.recent
```

### 方法 3：记忆检索（智能）

```
策略：历史存外部记忆，每轮按需检索相关

┌──────────────────────────────────┐
│  当前问题: "我之前说的异步方案..."  │
│                                    │
│  Step1: 从记忆库检索相关历史        │
│    → 找到: "轮次5讨论了asyncio"    │
│    → 找到: "轮次12确定了方案"      │
│                                    │
│  Step2: 组装上下文                 │
│    [相关记忆] + [最近2轮] + [当前] │
└──────────────────────────────────┘

优点：精准召回相关历史，不浪费token
缺点：依赖检索质量，有延迟
```

```python
class RetrievalMemory:
    def __init__(self, vector_db):
        self.memory = vector_db
        self.recent = []
    
    def get_context(self, current_query):
        # 1. 检索相关历史
        relevant = self.memory.search(
            current_query, 
            filter={"session": self.session_id},
            top_k=3
        )
        # 2. 组装：相关记忆 + 最近对话 + 当前问题
        context = []
        for mem in relevant:
            context.append({"role": "system", 
                           "content": f"[历史] {mem.content}"})
        context.extend(self.recent[-4:])  # 最近2轮
        return context
```

### 方法 4：分层管理（生产级）

```
组合策略，按信息重要性分层：

┌──────────────────────────────────────┐
│ Layer 1: 永久信息（始终保留）          │
│   - 用户画像/偏好/核心需求             │
│   - System Prompt                    │
├──────────────────────────────────────┤
│ Layer 2: 摘要（压缩保留）              │
│   - 早期对话的要点总结                 │
├──────────────────────────────────────┤
│ Layer 3: 最近原文（滑动窗口）          │
│   - 最近N轮的完整对话                  │
├──────────────────────────────────────┤
│ Layer 4: 检索记忆（按需）              │
│   - 存向量库，当前问题相关时才召回      │
└──────────────────────────────────────┘

最终上下文 = Layer1 + Layer2 + Layer3 + (Layer4 if relevant)
```

## 三、上下文组装示例

```python
def build_context(user_id, session_id, current_message):
    context = []
    
    # Layer 1: 永久信息
    profile = db.get_user_profile(user_id)
    context.append({
        "role": "system",
        "content": f"用户画像: {profile.preferences}"
    })
    
    # Layer 2: 会话摘要
    summary = redis.get(f"summary:{session_id}")
    if summary:
        context.append({
            "role": "system",
            "content": f"对话摘要: {summary}"
        })
    
    # Layer 4: 检索相关历史
    relevant = memory.retrieve(
        current_message, 
        session_id=session_id,
        top_k=3
    )
    for mem in relevant:
        context.append({
            "role": "system",
            "content": f"[相关历史] {mem}"
        })
    
    # Layer 3: 最近对话
    recent = redis.get_recent(session_id, limit=6)  # 最近3轮
    context.extend(recent)
    
    # 当前消息
    context.append({"role": "user", "content": current_message})
    
    return context
```

## 四、Token 预算管理

```python
def manage_token_budget(context, max_tokens=8000):
    """确保上下文不超过token预算"""
    total = count_tokens(context)
    
    if total <= max_tokens:
        return context  # 没超，全保留
    
    # 超了，按优先级裁剪
    # 优先级：永久信息 > 当前问题 > 最近对话 > 检索记忆 > 摘要
    
    # Step1: 减少检索记忆数量
    context = trim_retrieval(context, max_tokens * 0.9)
    # Step2: 缩短最近对话窗口
    context = trim_recent(context, max_tokens * 0.8)
    # Step3: 进一步压缩摘要
    context = compress_summary(context, max_tokens * 0.7)
    
    return context
```

## 五、方法对比

| 方法 | Token成本 | 信息保留 | 实现复杂度 | 适用 |
|------|----------|---------|-----------|------|
| 滑动窗口 | 低 | 差（丢早期） | 极低 | 简单聊天 |
| 摘要压缩 | 中 | 中（有损） | 中 | 长对话 |
| 记忆检索 | 中 | 好（精准） | 高 | 知识型对话 |
| 分层管理 | 中 | 优 | 高 | 生产级 |

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：多轮对话上下文建模的核心是"决定带什么历史进下一轮"，为什么不直接把所有历史塞进 context？**

三个约束。1）窗口上限——LLM 的 context window 有限（128K/200K），超长对话（如客服聊 100 轮）根本塞不下；2）attention 稀释——即使塞得下，context 越长每个 token 的 attention 越分散（lost in the middle），早期关键信息反而被"淹没"，模型表现下降；3）成本和延迟——context 越长，每次 forward 的计算量 O(n²) 增长，prefill 延迟和 API 成本飙升。所以"全塞"在长对话里不可行也不划算，必须选择性带历史——带"对当前回复最有价值"的部分，舍弃冗余。

### 第二层：证据与定位

**Q：Agent 在多轮对话里"忘了"早期约束（如用户说"只查 2024 年"但后面忘了），怎么定位是上下文截断丢了还是 LLM 没注意？**

看 context 的实际内容。1）截断检查——把当前轮发给 LLM 的完整 context 打出来，看"只查 2024 年"这句是否还在 context 里。如果不在了，是截断/压缩丢了（上下文管理问题）；如果还在但 LLM 没遵守，是 attention 稀释或 prompt 没强调（LLM 问题）。2）如果是截断丢了——优化截断策略：关键约束（如时间范围、用户明确要求）标"不可截断"标记，截断时保留；3）如果在但没用——优化 prompt（把关键约束放 context 末尾/near 当前 query，attention 权重更高）或用"约束提醒"（每轮在 system prompt 重复关键约束）。区分这两类是定位的前提，否则盲目优化方向错。

### 第三层：根因深挖

**Q：滑动窗口（只带最近 N 轮）是最简单的上下文建模，但它的问题是什么？什么时候不够用？**

滑动窗口丢了"早期的关键约束"。问题：1）约束遗忘——用户在第 1 轮说"只查 2024 年数据"，到第 10 轮（超出窗口）这句话被丢弃，Agent 又开始查全量数据，违反约束；2）任务连贯性断裂——长任务的早期决策（如"选了方案 A"）被忘，后面又重新讨论方案选择。不够用的场景：1）有"全局约束"的对话（如时间范围、用户身份、权限）；2）长任务（几十轮的咨询/分析）；3）需要"早期事实"的对话（如用户第 1 轮提供的姓名/订单号后面要用）。改进：滑动窗口 + 关键信息提取（把约束/事实提取出来单独保留，不随窗口滑出）+ 摘要压缩（早期对话摘要成要点）。

**Q：摘要压缩早期历史会丢失细节，怎么保证"摘要不丢关键信息"？**

结构化摘要 + 关键信息保护。1）结构化摘要——不是让 LLM 自由摘要，用固定 schema（如"用户目标/已确认信息/关键约束/待办事项/已完成步骤"），强制保留关键维度；2）关键信息保护——摘要前先抽取"绝对不能丢的实体"（如时间范围、订单号、用户明确偏好、已做的决策），摘要后再校验这些实体是否都在，丢了的补回；3）渐进式摘要——不是一次性摘要全部历史，而是分层（最近 3 轮保留全文，4-10 轮轻度摘要，10 轮以前深度摘要），保留近期细节；4）外部记忆兜底——原始历史存外部存储，摘要里放索引，需要细节时按需检索回填。这套组合能让摘要的关键信息保留率从自由摘要的 70% 提到 95%+。

### 第四层：方案权衡

**Q：上下文建模的"截断、摘要、检索"三种策略，怎么组合使用？**

分层组合。1）近期（最近 N 轮）——保留全文（不截断不摘要），因为近期信息最相关且 LLM attention 强；2）中期（N 到 2N 轮）——轻度摘要（每轮压成一句话要点），保留要点丢细节；3）远期（2N 轮以前）——深度摘要（按主题归纳）+ 关键约束单独提取保留；4）超远期/外部信息——存外部记忆，按需检索（当当前对话提到相关主题时检索回填）。这个"近期全文+中期轻摘+远期深摘+外部检索"的组合，在 context 限制内最大化信息保留。N 的取值按 context 窗口和对话特点调（如客服场景 N=5，长咨询 N=10）。

**Q：为什么不直接用支持超长 context 的模型（如 Gemini 2M context），把所有历史塞进去，省去上下文建模的复杂度？**

长 context 不能替代建模。1）成本——2M context 的 prefill 计算量是 128K 的 15 倍，每轮对话都要重新 forward 全部历史，成本和延迟爆炸；2）attention 稀释——2M context 的中间信息召回率比首尾低 20-30%（lost in the middle 更严重），关键信息可能被淹没；3）不必要——多数对话的"有用历史"集中在近期和关键约束，全量塞入是浪费（80% 的历史对当前回复无用）。所以即使有 2M context，仍要做"选择性带历史"——近期全文+关键约束保留+长尾摘要，控制在几十 K context，兼顾信息保留和成本/效果。长 context 是"兜底"（极端情况下不溢出），不是"建模替代"。

### 第五层：验证与沉淀

**Q：你怎么衡量上下文建模策略（截断/摘要/检索）的效果，而不是"压缩了但 Agent 变笨了"？**

AB 测试。固定多轮对话集，对比"全量历史（不压缩，假设塞得下）"和"压缩后（截断/摘要）"在：1）回复准确率——压缩版应持平或略降（降太多说明丢信息）；2）约束遵守率——压缩版是否遵守早期约束（如时间范围），遵守率应 >90%；3）成本和延迟——压缩版应显著降低。最优策略是"准确率持平 + 约束遵守率高 + 成本降"。再细分：近期对话的准确率应持平（全文保留），远期的约束遵守率是关键指标（摘要是否保住关键约束）。如果压缩版约束遵守率 <70%，说明摘要太激进或截断丢了约束，要调保护策略。

**Q：多轮对话的上下文建模策略怎么沉淀成框架的 ContextManager 组件？**

封装成组件：1）分层策略配置——开发者配"近期全文轮数/中期摘要轮数/远期深度摘要/外部记忆"，框架自动执行；2）结构化摘要——内置固定 schema 的摘要 prompt（用户目标/约束/事实/待办/已决），强制保留关键维度；3）关键信息保护——自动抽取"时间/数值/用户明确要求"等关键实体，摘要后校验保留；4）约束持久化——用户明确的全局约束（如时间范围）自动标"不可截断"，单独保留；5）外部记忆接口——超远期历史自动写外部存储，按需检索回填。开发者配层级和阈值，框架自动跑上下文管理，新 Agent 接入即具备专业级上下文建模，不重写截断/摘要逻辑。
