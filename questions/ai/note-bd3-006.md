# 训练一个14B参数的FP16模型，理论上需要多少显存？

> 来源：字节跳动大模型技术面试二面

## 理论显存计算

### 1. 模型权重

```
14B 参数 × 2 bytes (FP16) = 28 GB
```

### 2. 优化器状态 (AdamW)

AdamW 为每个参数维护4个副本：

```
┌──────────────────────────────────────────────────┐
│        每个FP16参数的显存开销 (AdamW)             │
├──────────────────┬────────┬──────────────────────┤
│     组成部分      │ 精度   │    bytes/参数        │
├──────────────────┼────────┼──────────────────────┤
│ 模型权重 W        │ FP16   │  2                   │
│ 梯度 ∇W           │ FP16   │  2                   │
│ AdamW m (一阶矩)  │ FP32   │  4                   │
│ AdamW v (二阶矩)  │ FP32   │  4                   │
│ FP32主权重 (混合) │ FP32   │  4                   │
├──────────────────┼────────┼──────────────────────┤
│ 合计              │        │  16 bytes/参数       │
└──────────────────┴────────┴──────────────────────┘

14B × 16 bytes = 224 GB  ← 仅权重+优化器!
```

> **注**：纯FP16训练（无FP32主权重）为12 bytes/参数 = 168GB。  
> 混合精度训练（推荐）为16 bytes/参数 = 224GB。

### 3. 激活值

激活值显存取决于前向传播中需要保存的中间结果：

```
激活值 ≈ batch_size × seq_len × hidden_dim × num_layers × bytes

以14B模型典型配置 (hidden=5120, layers=40):
  batch=1, seq=4096:
  激活值 ≈ 1 × 4096 × 5120 × 40 × 2 (FP16)
         ≈ 1.68 GB/层 × 40 ≈ 不对，实际更复杂
```

更精确的估算（Transformer每层激活值）：

```
每层激活 ≈ batch × seq × hidden × (34 × hidden + 5 × seq) × 2 bytes

使用梯度检查点后:
  激活值 ≈ batch × seq × hidden × √layers × 2 bytes
  (只保存√N层的激活，其余重算)
```

### 4. 总计

```
┌──────────────────┬─────────────┬────────────────────┐
│      组成部分     │   显存(GB)  │      占比          │
├──────────────────┼─────────────┼────────────────────┤
│ 模型权重 (FP16)   │    28       │    15%             │
│ 梯度 (FP16)       │    28       │    15%             │
│ 优化器状态 (FP32) │   168       │    60% ← 最大!     │
│ 激活值 (FP16)     │   ~60       │     ~10%          │
├──────────────────┼─────────────┼────────────────────┤
│ 总计              │  ~284 GB   │    100%            │
└──────────────────┴─────────────┴────────────────────┘

→ 约需 4× A100-80GB (320GB) 或 6× A100-40GB
```

## INT8 量化的节省

```
┌──────────────────┬───────────┬───────────┬──────────┐
│      组成部分     │  FP16 (GB)│ INT8 (GB) │  节省    │
├──────────────────┼───────────┼───────────┼──────────┤
│ 模型权重          │    28     │    14     │  14 GB   │
│ 梯度             │    28     │    14     │  14 GB   │
│ 优化器状态        │   168     │   ~168    │  ~0      │
│ 激活值           │    60     │    30     │  30 GB   │
├──────────────────┼───────────┼───────────┼──────────┤
│ 总计             │   284     │   ~226    │  ~58 GB  │
└──────────────────┴───────────┴──────────┴──────────┘

注意: 优化器状态仍需FP32精度，INT8主要节省权重和激活值
总节省约 20-25%
```

## 显存优化手段

| 方法 | 节省部分 | 节省比例 | 代价 |
|------|---------|---------|------|
| **ZeRO-1** | 优化器状态分片 | ~4× | 通信增加 |
| **ZeRO-2** | +梯度分片 | ~8× | 通信增加 |
| **ZeRO-3** | +权重分片 | ~N× | 通信大增 |
| **梯度检查点** | 激活值 | ~√N | 计算增加33% |
| **混合精度** | 权重+梯度 | ~50% | 精度风险 |
| **LoRA/QLoRA** | 只训练少量参数 | ~99% | 效果略降 |

## 记忆要点

- 参数显存乘以16：AdamW混合精度训练中，每参数需16Bytes(权重2+梯度2+优化器状态12)。
- 优化器状态是大头：FP32的主权重、动量m和v占据12Bytes，占整体显存超一半。
- 总显存需四张卡：14B模型纯训练显存高达约224GB，需4张A100-80G才能装下。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：训练显存你说"每参数 16 Bytes"（权重 2 + 梯度 2 + 优化器状态 12）。为什么优化器状态占 12 Bytes？AdamW 的状态具体是什么？**

AdamW 的状态是"一阶动量 m"和"二阶动量 v"，加上"FP32 主权重"。混合精度训练中，前向/反向用 fp16（权重和梯度各 2 Bytes），但 AdamW 的更新需要 fp32 精度（fp16 的累加会数值溢出/精度损失）。具体：一是 FP32 主权重（master weight，4 Bytes）——fp16 权重的 fp32 副本，优化器在 fp32 上更新，再转回 fp16 用于前向；二是一阶动量 m（4 Bytes，FP32）——梯度的指数移动平均，用于动量更新；三是二阶动量 v（4 Bytes，FP32）——梯度平方的指数移动平均，用于自适应学习率。合计 $4 + 4 + 4 = 12$ Bytes 优化器状态。加上 fp16 权重（2）和 fp16 梯度（2），每参数 $2+2+12 = 16$ Bytes。14B 模型的训练显存：$14 \times 10^9 \times 16 = 224\text{GB}$（纯参数，不含激活值和 KV），需 4 张 A100-80G。

### 第二层：证据与定位

**Q：训练 7B 模型时 GPU 显存 OOM。你怎么定位是 batch size 太大（激活值大）、序列长度太长（激活值随长度增长）、还是优化器状态没分片（单卡装不下）？**

逐项排查。一是 batch size——激活值（每层的中间输出）与 batch size 成正比，减小 batch size（如从 32 到 8）看是否还 OOM，如果减 batch 解决，是激活值问题（加 gradient checkpointing 用计算换显存）。二是序列长度——激活值与 seq_len 成正比（attention matrix 是 $O(\text{seq\_len}^2)$），减小 seq_len（如从 4096 到 2048）看是否解决，如果解决是序列长度问题。三是优化器状态——用 ZeRO-1（分片优化器状态）看是否解决，如果 ZeRO-1 后不 OOM，是优化器状态没分片（单卡装不下 12 Bytes/参数的状态）。排查顺序：先看显存峰值在哪个阶段（前向/反向/优化器更新），前向峰值大是激活值（batch/seq_len），优化器更新峰值大是状态。用 PyTorch 的 `torch.cuda.max_memory_allocated()` 打印各阶段显存。

### 第三层：根因深挖

**Q：混合精度训练为什么要"FP32 主权重"（fp16 权重 + fp32 副本）？直接用 fp16 做优化不行吗？**

fp16 的数值精度不够做优化更新。AdamW 的更新公式 $w_{t+1} = w_t - \eta \cdot m_t / (\sqrt{v_t} + \epsilon)$，当更新量 $\eta \cdot m_t / \sqrt{v_t}$ 很小（如 1e-5）而权重 $w_t$ 较大（如 0.1）时，fp16 的精度（约 1e-3 的相对误差）会让更新被"舍入为 0"（$0.1 + 0.00001 = 0.1$ in fp16），权重不更新（训练停滞）。FP32 主权重用 fp32 精度做更新（fp32 的相对误差约 1e-7，能表示小更新），更新后再舍入到 fp16 用于前向。这就是"混合精度"——前向/反向用 fp16（快、省显存），优化器更新用 fp32（精度保证）。bf16（Brain Float 16）是另一种方案——bf16 的动态范围大（指数位多），不易数值溢出，但精度仍不如 fp32，所以即使 bf16 训练也常保留 fp32 主权重。fp32 主权重是"训练稳定性的保险"。

**Q：那为什么不直接用 fp32 训练（全精度），省得维护 fp16 + fp32 两套权重？**

fp32 训练显存翻倍且速度慢。fp32 每参数 4 Bytes（权重），加上梯度（4）和优化器状态（12），每参数 20 Bytes（相比混合精度的 16，多 4 Bytes 的权重和梯度）。7B 模型的 fp32 训练显存约 140GB（vs 混合精度 112GB），且前向/反向的矩阵乘法在 fp32 下慢 2-4 倍（GPU 的 fp16 算力是 fp32 的 2-4 倍，如 A100 的 fp16 312 TFLOPS vs fp32 19.5 TFLOPS）。混合精度的优势是"前向/反向快（fp16）+ 优化器精度保证（fp32 主权重）"，兼顾速度和稳定性。纯 fp32 只在"对精度极度敏感"的场景（如科学计算、数值仿真）才用，LLM 训练用混合精度（fp16 或 bf16 + fp32 主权重）是标准实践。

### 第四层：方案权衡

**Q：优化器状态占 12 Bytes（大头）。为什么用 AdamW（状态多），不用 SGD（无状态，省显存）？**

AdamW 的收敛质量远优于 SGD。SGD 无优化器状态（只更新权重，每参数 4 Bytes，省显存），但学习率难调（需要手动调 schedule）、收敛慢、容易陷入局部最优。AdamW 的一阶/二阶动量提供"自适应学习率"（每个参数根据历史梯度自适应调整），收敛快且稳定（几乎不用调学习率 schedule），是 LLM 训练的标准优化器。状态占 12 Bytes 的代价（显存）远小于"训不动"的代价（SGD 可能训不出好模型）。如果显存极度紧张，可以用"低显存优化器"（如 8-bit Adam，把 m/v 量化到 8-bit，状态从 12 降到 6 Bytes），或用 ZeRO 分片（把状态分散到多卡），而非退回 SGD。AdamW 是"质量优先"，显存问题用分布式训练（ZeRO/并行）解决，而非牺牲优化器质量。

**Q：为什么不用 Adafactor（状态更少，只需一阶动量，省显存）替代 AdamW？**

Adafactor 省显存但收敛质量略差。Adafactor 用"矩阵分解"近似二阶动量（不存完整的 v，存行和列的统计量），状态从 12 降到约 6 Bytes，省一半。但实验证明 Adafactor 的收敛质量略低于 AdamW（在 LLM 训练上，loss 曲线更抖、最终精度差 1-2%），且 Adafactor 的超参更敏感（factorization 的超参要调）。AdamW 的 12 Bytes 状态通过 ZeRO-1 分片（分到多卡）或 8-bit Adam（量化）解决，不必换优化器。当前主流（Llama/GPT）都用 AdamW，Adafactor 是"显存极度紧张且能接受质量损失"的备选（如早期 T5 训练用 Adafactor）。选型原则：优先 AdamW + 显存优化（ZeRO/8-bit），质量不达标再考虑 Adafactor。

### 第五层：验证与沉淀

**Q：你怎么估算训练所需的显存，证明"4 张 A100 够训 14B"？**

用显存公式估算。训练显存 = 模型权重（fp16，2 Bytes/参数）+ 梯度（fp16，2）+ 优化器状态（AdamW，12）+ 激活值（取决于 batch/seq_len/layers）。14B 模型的权重+梯度+状态 = $14 \times 10^9 \times 16 = 224\text{GB}$。激活值（以 batch=8、seq_len=2048、layers=40 为例）约 50GB。总计约 274GB。单卡 A100-80G 装不下，用 ZeRO-3（全分片）分到 4 张卡，每卡 $274/4 \approx 68\text{GB}$（加通信 buffer 约 75GB），单卡 80GB 够。验证方法：实际跑一个 batch（小 batch），用 `nvidia-smi` 或 PyTorch 的 memory tracker 打印显存峰值，对比估算。如果实测显存超估算，检查是否有显存泄漏（如激活值没释放）或估算遗漏（如 attention 的中间结果）。关键留 20% 余量（显存碎片和峰值），4 张 A100-80G 训 14B 是紧但可行。

**Q：显存估算和优化怎么沉淀成训练规范？**

固化成"训练显存计算器"：输入模型参数量、batch size、seq_len、优化器类型、并行策略（ZeRO 级别/张量并行/流水线并行），输出所需 GPU 数和每卡显存。沉淀"各模型规模的训练配置推荐"（7B 用 2 卡 ZeRO-2、14B 用 4 卡 ZeRO-3、70B 用 8 卡 ZeRO-3 + 张量并行）、"显存优化手段对照表"（gradient checkpointing 省 50% 激活值但慢 20%、ZeRO-1 省 12 Bytes 状态/卡、8-bit Adam 省一半状态）。配套训练前的"显存预检脚本"（模拟一个 batch 的显存占用，预估是否 OOM），避免训练中途 OOM 浪费时间。把"显存估算"作为训练启动的必经步骤，确保资源配置合理。

## 结构化回答

**30 秒电梯演讲：** 训练显存 = 模型权重 + 梯度 + 优化器状态 + 激活值，通常是模型权重的4-6倍——就像搬家。

**展开框架：**
1. **FP16模型权重** — 14B × 2 bytes = 28GB
2. **AdamW优化器** — 权重 × 4倍 (权重2B + 梯度2B + m 2B + v 2B = 8B/参数)
3. **激活值** — 与batch_size × seq_len × hidden_dim成正比
