# 7B和32B大模型选型逻辑，什么场景用小模型蒸馏部署？

## 选型决策矩阵

```
                        任务复杂度
                    低 ←─────────→ 高

              ┌──────────┬──────────┬──────────┐
    低延迟    │  ✅ 7B   │  ⚠️ 14B  │  ❌ 32B  │
    高并发    │  分类    │  简单    │  延迟    │
     ↑        │  提取    │  推理    │  不可接  │
     │        │  路由    │          │  受      │
     │        ├──────────┼──────────┼──────────┤
    低延迟    │  ✅ 7B   │  ✅ 14B  │  ⚠️ 32B  │
    低并发    │          │          │  可接受  │
     │        ├──────────┼──────────┼──────────┤
     ↓        │  ✅ 7B   │  ✅ 14B  │  ✅ 32B  │
    高延迟    │  浪费    │          │  最佳    │
    可接受    │          │          │  质量    │
              └──────────┴──────────┴──────────┘
```

## 具体场景选型

| 场景 | 推荐模型 | 原因 | 延迟 | 成本/千次 |
|------|---------|------|------|----------|
| 意图分类 | 7B | 类别有限，7B准确率已>95% | 0.3s | ¥0.5 |
| 实体提取 | 7B | 结构化任务，规则明确 | 0.5s | ¥0.5 |
| 简单对话 | 7B | FAQ类回答 | 0.5s | ¥0.5 |
| 摘要生成 | 14B | 需理解全文但不需要复杂推理 | 1.5s | ¥2 |
| 单步推理 | 14B | 需要一定的逻辑链 | 1.5s | ¥2 |
| 多步规划 | 32B | 需要长链推理和任务拆解 | 3s | ¥8 |
| 复杂创作 | 32B | 需要创意和风格控制 | 4s | ¥8 |
| Function Call | 14B+ | 参数生成需要精确理解 | 1s | ¥2 |

## 蒸馏部署流程

```python
"""
小模型蒸馏：让7B学习32B的能力
"""

# 第一步：用32B模型生成高质量训练数据
def generate_distillation_data(tasks: list):
    """32B教师模型生成高质量输出"""
    training_data = []
    for task in tasks:
        # 用32B生成详细、高质量的回答
        response = call_32b(task['prompt'], temperature=0.3)
        training_data.append({
            'input': task['prompt'],
            'output': response,  # 32B的输出作为标签
            'task_type': task['type'],
        })
    return training_data

# 第二步：用7B学生模型学习
"""
# 使用LoRA微调，保留7B的基础能力
# 训练配置
model_name: Qwen/Qwen2.5-7B-Instruct
lora_r: 64
lora_alpha: 128
learning_rate: 1e-4
num_epochs: 3
train_data: distillation_data.jsonl  # 32B生成的数据

# 训练命令
llamafactory-cli train \
    --model_name_or_path Qwen/Qwen2.5-7B-Instruct \
    --dataset distillation_data \
    --lora_rank 64 \
    --output_dir ./7b-distilled
"""

# 第三步：评估蒸馏效果
def evaluate_distillation(test_set: list):
    """对比原始7B vs 蒸馏7B vs 32B"""
    results = {'original_7b': [], 'distilled_7b': [], 'teacher_32b': []}
    for item in test_set:
        results['original_7b'].append(call_7b(item['input']))
        results['distilled_7b'].append(call_distilled_7b(item['input']))
        results['teacher_32b'].append(call_32b(item['input']))

    # 计算与32B输出的相似度
    for model in results:
        sim = avg_similarity(results[model], results['teacher_32b'])
        print(f"{model}: 与32B相似度 = {sim:.2%}")
```

## 动态模型路由

```python
class DynamicModelRouter:
    """根据任务难度动态选择模型"""
    def __init__(self):
        self.complexity_classifier = load_model('complexity-bert')  # 轻量分类器

    def route(self, user_input: str, context: dict) -> str:
        # 1. 快速分类任务难度
        complexity = self.complexity_classifier.predict(user_input)

        # 2. 根据难度选择模型
        if complexity == 'simple':
            return 'qwen-7b-distilled'
        elif complexity == 'medium':
            return 'qwen-14b'
        else:
            return 'qwen-32b'

        # 3. 兜底：Token预算或并发限制时降级
        if self.current_load > self.threshold:
            return 'qwen-7b-distilled'  # 高负载时统一用小模型
```

## 面试加分点

1. **蒸馏不是万能**：蒸馏后7B在训练分布内的任务接近32B，但分布外（OOD）任务仍差距明显
2. **成本量化**：32B单次推理成本约7B的8-10倍，日均100万次调用可节省¥7500/天
3. **渐进式策略**：先全用32B保证质量 → 统计任务难度分布 → 对简单任务蒸馏7B → 逐步替换
4. **A/B测试**：模型替换必须做A/B测试，监控准确率/满意度/投诉率变化

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：7B 和 32B 的选型，为什么要按场景分而不是统一用 32B（能力最强）？**

成本和延迟。32B 的推理成本是 7B 的 4-5 倍（参数量决定算力），延迟也是 3-4 倍。统一用 32B 意味着：1) 简单任务（分类、提取）也用 32B，浪费算力；2) 高并发场景（如客服每秒几百请求）32B 的吞吐撑不住；3) 边缘部署（手机、私有化）32B 显存放不下。所以按任务复杂度分流——简单任务用 7B（便宜快），复杂任务用 32B（准确）。这是"按需用算力"的工程思维。

### 第二层：证据与定位

**Q：怎么判断一个任务"简单"用 7B 够，还是"复杂"必须 32B？**

用准确率门槛判断。对每个任务类型，分别用 7B 和 32B 跑 eval 集：1) 如果 7B 准确率 > 90% 且和 32B 差距 < 3%，用 7B 够；2) 如果 7B 准确率 < 85% 或和 32B 差距 > 10%，必须 32B。典型分布：分类/提取/简单问答 → 7B 够；多步推理/规划/代码生成 → 32B 明显更优。用历史任务数据统计每类的 7B vs 32B 准确率差，建分流规则。

### 第三层：根因深挖

**Q：7B 在复杂推理上不如 32B，根因是参数量不够还是训练数据不够？**

主要是参数量（容量）。7B 和 32B 用相似规模的训练数据（万亿 token），但 7B 的参数容量不足以"压缩"复杂推理模式——模型容量小，学到的推理能力有上限。根因判断：如果给 7B 更多数据，准确率提升有限（已经饱和），是容量瓶颈；如果给更多数据能持续提升，是数据瓶颈。经验上，7B 在常见任务上已接近数据饱和，进一步提升要靠扩参数或蒸馏（用 32B 的输出发音 7B）。

**Q：那为什么不直接用蒸馏（用 32B 输出训练 7B），让 7B 也有 32B 的能力？**

蒸馏能提升 7B 的能力，但有上限。蒸馏的本质是"让小模型模仿大模型的输出分布"，能学到 7B 容量范围内的 32B 行为。但 7B 的容量天花板决定了它无法完全复现 32B 的复杂推理——有些推理模式需要更大的参数空间才能表达。经验上蒸馏能让 7B 在目标任务上接近 32B 的 85-90%，但不是 100%。所以蒸馏是"性价比优化"，不是"完全替代"。复杂任务仍需 32B。

### 第四层：方案权衡

**Q：小模型蒸馏部署的代价是什么？什么场景值得做？**

代价：1) 训练成本——要准备大量（query, 32B_output）对作为蒸馏数据，32B 推理生成这些数据要算力；2) 维护成本——蒸馏后的 7B 是定制版，模型升级要重新蒸馏；3. 泛化风险——蒸馏到特定任务后，7B 在其他任务上可能退化（过拟合到蒸馏数据）。值得做的场景：1) 高频调用（如每天百万次）——单次成本节省累积巨大；2) 特定领域（如客服、代码）——蒸馏到领域内，7B 能接近 32B；3) 边缘部署——必须用小模型。不值得的场景：低频长尾任务（蒸馏成本 > 收益）。

**Q：既然 7B 便宜快，为什么不所有任务都用 7B + 蒸馏，彻底放弃 32B？**

因为有些任务 7B 即使蒸馏也达不到要求。如：1) 复杂规划（Agent 多步任务拆解）——需要 32B 的推理深度；2) 长尾知识问答——32B 参数多，记忆的知识更广；3) 创意写作——32B 的表达能力更强。这些场景下 7B 的准确率天花板低于业务可接受线（如 < 80%），蒸馏也无法弥补。所以 7B 和 32B 共存——7B 处理 80% 的高频简单任务，32B 处理 20% 的复杂任务，按需路由。

### 第五层：验证与沉淀

**Q：怎么持续验证"7B 和 32B 的分流"是否合理？**

三个监控：1) 分流准确率——被路由到 7B 的任务，其准确率是否达标（如果 7B 路由的任务准确率 < 85%，说明分流太激进，该用 32B 的用了 7B）；2) 成本节省——实际 7B vs 32B 调用比例和对应的成本节省（验证 ROI）；3) 路由错误率——被路由到 7B 但实际需要 32B 的任务比例（7B 处理失败后回退到 32B 的次数）。沉淀为模型选型决策表：任务类型 × 准确率要求 × 延迟要求 → 推荐模型，定期更新。

## 结构化回答

**30 秒电梯演讲：** 7B适合高并发低延迟场景（分类/提取/简单对话），32B适合复杂推理（规划/多步推理/创作）。小模型蒸馏部署是用大模型的输出去训练小模型，保留核心能力。

**展开框架：**
1. **7B优势** — 速度快、成本低、可本地部署
2. **32B优势** — 推理能力强、指令遵循好、适合复杂任务
3. **蒸馏流程** — 32B生成高质量数据→7B学习→接近32B效果
