# 大模型思维链（CoT）原理是什么？

## 一、CoT 核心思想

**CoT（Chain-of-Thought）** = 让 LLM 在给出最终答案前，**先输出推理过程（思维链）**。

```
普通Prompting（直接要答案）：
  Q: 小明有5个苹果，给了小红2个，又买了3个，现在有几个？
  A: 6个  ← 可能算错（尤其复杂题）

CoT Prompting（先推理再答）：
  Q: 小明有5个苹果，给了小红2个，又买了3个，现在有几个？
  A: 小明原来有5个，给出去2个剩5-2=3个，
     又买了3个所以3+3=6个。
     答案：6个  ← 推理过程让答案更可靠
```

## 二、为什么 CoT 有效（原理）

### 自回归视角

```
LLM是自回归模型：P(y | x) = Π P(y_i | x, y_<i)

直接生成答案：
  P("6" | "问题") 
  → 从问题一步跳到答案，跨度大，搜索空间大，易错

CoT生成答案：
  P("先算5-2=3" | "问题") × P("再算3+3=6" | "问题,先算5-2=3")
  → 中间步骤分解了难度，每步跨度小，更准
```

### 直观理解

```
没有CoT：问题 ──跳过推理──→ 答案（黑盒，易错）

有CoT：  问题 ──→ 步骤1 ──→ 步骤2 ──→ ... ──→ 答案
                  ↑每步都为下一步提供上下文

中间推理步骤 = 给最终答案的生成"搭了脚手架"
```

## 三、两种 CoT 方式

### 1. Zero-shot CoT（零样本）

```python
# 最简单：在问题后加一句话
prompt = f"""
{question}

让我们一步一步思考。
"""
# 这句话触发模型展示推理过程
```

**优点：** 简单，无需示例。
**缺点：** 推理质量依赖模型自身能力。

### 2. Few-shot CoT（少样本）

```python
prompt = f"""
Q: 罗杰有5个网球，又买了2筒，每筒3个，共几个？
A: 罗杰原有5个。买了2筒每筒3个=6个。总共5+6=11个。

Q: 自助餐每位30元，8人去吃有2人免单，共花多少？
A: 8人中有2人免单，付费的6人。6×30=180元。

Q: {实际问题}
A:
"""
# 用带推理的示例引导模型
```

**优点：** 推理质量更高、更规范。
**缺点：** 需要构造示例，消耗更多 Token。

## 四、CoT 的效果（论文数据）

```
GSM8K（小学数学）准确率提升：

┌──────────────┬─────────┬──────────┐
│ 方法          │ GPT-3    │ PaLM-540B │
├──────────────┼─────────┼──────────┤
│ Standard     │ 17.7%    │ 17.7%     │
│ Few-shot CoT │ 56.9%    │ 56.9%     │
│ Zero-shot CoT│ 10.4%    │ 56.9%     │
└──────────────┴─────────┴──────────┘

关键发现：
  1. CoT对复杂推理（数学/逻辑）提升显著
  2. 简单任务提升不明显（甚至可能下降）
  3. 小模型(<60B)几乎没有CoT效果（涌现能力）
```

## 五、CoT 的演进

```
CoT家族演进：

CoT（线性推理）
  ↓
ToT（Tree of Thoughts，树状探索）
  → 多分支推理，失败可回溯
  ↓
GoT（Graph of Thoughts，图状推理）
  → 推理节点可合并/循环
  ↓
Self-Consistency（自洽性）
  → 生成多条CoT，投票选最常见答案
  ↓
ReAct（推理+行动）
  → CoT + 工具调用结合
```

### Self-Consistency（自洽性）示例

```python
# 生成多条推理链，投票
chains = [generate_cot(question) for _ in range(5)]
# ["答案是6", "答案是6", "答案是7", "答案是6", "答案是6"]
answer = most_frequent(extract_answers(chains))  # 6
# 多数投票，比单条更可靠
```

## 六、CoT 在 Agent 中的角色

```
CoT是Agent认知的基础：

ReAct的"Thought" → 本质就是CoT
  Thought: 我需要先查天气（推理）
  Action: search_weather()（行动）

Plan-Execute的"Plan" → 用CoT分解任务
  Thought: 写报告需要先查资料、列大纲...
  Plan: [step1, step2, ...]

Reflexion的"Reflect" → 用CoT反思
  Thought: 上次失败是因为边界条件没处理
  Lesson: 注意边界
```

## 七、CoT 的局限

```
1. Token消耗大：推理过程占用输出Token
   → 优化：用"内部推理"（如o1的hidden CoT）

2. 仍会出错：推理过程本身可能错
   → 优化：Self-Consistency投票 + 工具验证

3. 可能"强词夺理"：先给答案再编推理
   → 优化：先生成推理再生成答案（顺序很重要）

4. 小模型无效：CoT是涌现能力
   → 优化：换大模型，或用蒸馏让小模型学CoT
```

## 八、面试加分点

1. **讲清原理**：CoT 利用自回归特性，中间步骤为最终答案提供上下文约束——不是"魔法"，是利用了模型生成机制
2. **区分 Zero/Few-shot**：Zero 加指令，Few 给示例，后者质量更高
3. **强调"涌现"**：CoT 是大模型的涌现能力（>60B），小模型无效——这体现了规模的重要性

## 记忆要点

- 核心原理：自回归生成，中间推理步骤为最终答案提供了上下文约束
- 两种方式：Zero-shot加上“一步一步思考”，Few-shot提供带推理的示例
- 适用条件：大模型涌现能力，对复杂推理提升显著，简单任务无明显效果
