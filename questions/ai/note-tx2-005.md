# 【某讯面经】文档分块四种方案优缺点：固定窗口、递归分割、语义分割、结构感知分块

## 一、四种方案对比

| 方案 | 原理 | 优 | 劣 |
|------|------|----|----|
| **固定窗口** | 按固定 token 数切 | 简单、快 | 切断语义、边界粗糙 |
| **递归分割** | 按分隔符层级递归切 | 平衡、常用 | 仍可能不完美 |
| **语义分割** | 按句子/段落语义边界 | 质量高 | 慢、需 NLP |
| **结构感知** | 按文档结构（标题/段落） | 最准 | 需结构化输入 |

## 二、固定窗口分块（Fixed-Size Chunking）

```
文档: "今天天气真好。我想去公园。但是要下雨了。明天再去吧。"
chunk_size = 10 tokens, overlap = 2

Chunk 1: "今天天气真好。我想去公"
Chunk 2: "想去公园。但是要下雨了"    ← overlap "想去"
Chunk 3: "下雨了。明天再去吧。"
```

**优点**：
- 实现最简单（按 token 数硬切）
- 速度快
- chunk 大小一致（向量库索引友好）

**缺点**：
- **可能切断语义**（"我想去公"和"园"被切开）
- 不考虑内容边界
- 检索时 chunk 可能答不全

**适用**：纯文本、无结构、对质量要求不高。

## 三、递归分割分块（Recursive Splitting）

```
分隔符层级（优先级从高到低）：
  1. "\n\n"（段落）
  2. "\n"（换行）
  3. "。" / "."（句子）
  4. "，" / ","（子句）
  5. " "（空格）
  6. 字符

算法：
  1. 用最高级分隔符切
  2. 如果某块 > chunk_size，用下一级分隔符继续切
  3. 直到所有块 ≤ chunk_size
```

**优点**：
- **尽量保持语义完整**（优先按段落切，切不动才降级）
- 平衡质量和速度
- **LangChain 默认方案**（`RecursiveCharacterTextSplitter`）

**缺点**：
- 仍可能不完美（段落本身超长时降级到句子）
- 不理解语义（纯靠分隔符）

**适用**：通用场景的默认选择。

## 四、语义分割（Semantic Chunking）

```
用 embedding 计算相邻句子的语义相似度：

句子1 → 句子2: 相似度 0.9（同主题，合并）
句子2 → 句子3: 相似度 0.9（合并）
句子3 → 句子4: 相似度 0.3（主题变了，切断！）
  → Chunk A = [句1,2,3]
  → 继续...
句子4 → 句子5: 相似度 0.8（合并）
```

**优点**：
- **按语义边界切**（主题转换处切断）
- chunk 内语义高度一致
- 检索质量最高

**缺点**：
- **慢**（每对句子算 embedding）
- 需要 embedding 模型
- 阈值难调（相似度多少算"主题变了"）

**适用**：对检索质量要求高、文档量不大的场景。

## 五、结构感知分块（Structure-Aware Chunking）

```
Markdown 文档：
  # 第一章 概述       ← H1 标题 → chunk 边界
  正文...
  ## 1.1 背景         ← H2 标题 → chunk 边界
  正文...
  ## 1.2 现状         ← H2 标题 → chunk 边界
  正文...

按标题层级切：
  Chunk 1: "# 第一章 概述" + 该节正文
  Chunk 2: "## 1.1 背景" + 该节正文
  Chunk 3: "## 1.2 现状" + 该节正文
```

**优点**：
- **最准**（按文档作者定义的结构切）
- 保留标题上下文（chunk 自带"在讲什么"）
- 支持 Markdown/HTML/PDF（用解析器提取结构）

**缺点**：
- **需要结构化输入**（纯文本无结构）
- 依赖解析器质量（PDF 解析可能出错）
- chunk 大小不一致（有的章节长有的短）

**适用**：结构化文档（技术文档、产品手册、法律合同）。

## 六、生产实践推荐

### 参数设置
```
chunk_size: 200-500 tokens
  - 太小（<100）：语义不全
  - 太大（>1000）：检索不精确
  
overlap: 10-20%（如 chunk_size=400, overlap=50-80）
  - 防止边界信息丢失
  - 关键信息在边界时不被切成两半
```

### 按文档类型选方案

| 文档类型 | 推荐方案 | 理由 |
|---------|---------|------|
| 纯文本（日志/对话） | 固定窗口 / 递归分割 | 无结构 |
| Markdown/HTML | **结构感知**（按标题） | 有明确结构 |
| PDF | 结构感知（解析后）+ 递归兜底 | 解析可能不全 |
| 代码 | 结构感知（按函数/类） | 代码有 AST |
| 长文章 | 递归分割 + 语义分割 | 平衡 |
| 问答对/QA | 按 QA 切（一个 QA 一个 chunk） | 天然单元 |

### 混合策略
```
1. 先按结构感知切（大块）
2. 大块超 chunk_size → 递归分割（小块）
3. 小块之间加 overlap
```

## 七、加分点

- 说出 **chunk 大小和检索场景的关系**：
  - 事实查询（"X是什么"）→ 小 chunk（精准）
  - 总结/综述（"概述X"）→ 大 chunk（完整）
- 说出 **Parent-Child 分块**：切小 chunk 做检索（精准），命中后返回父 chunk（完整上下文）
- 说出 **Late Chunking**（2024 新方法）：先对整个文档 embedding 再切，保留全局上下文

## 八、雷区

- ❌ 所有文档都用固定窗口 → 结构化文档质量差
- ❌ chunk 太大（>1000 token）→ 检索混入无关内容
- ❌ 没有 overlap → 边界信息丢失

## 九、扩展

- **chunk 元数据**：每个 chunk 带元数据（来源文档、标题、页码），检索时能溯源
- **动态 chunk 大小**：按内容密度动态调整（密集段落切小，稀疏切大）
- **chunk 质量评估**：用"检索召回率"+"答案完整度"反向评估分块策略好坏

## 记忆要点

- 四大方案对比：固定窗口最糙、递归分割最通用、语义分割最高质、结构感知最准
- 默认首选：递归分割，因为按段落/句子优先级降级切，完美平衡速度与语义
- 优缺对比：结构感知能保全文层级最准，但极度依赖输入文档的结构化解析质量
- 参数铁律：chunk建议200-500 tokens，overlap保持10-20%防语义硬截断
- 混合策略：生产中常先按文档结构感知切大块，超长再降级用递归兜底切小块


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：文档分块（chunking）为什么必要？直接把整篇文档做向量不行吗？**

整篇文档一个向量会丢失"局部相关性"。一篇 10000 token 的文档讲 5 个主题，整篇向量是 5 个主题的混合表征，相似度被稀释（用户问一个主题，整篇向量相似度不高不低）。分块让每个 chunk 是"聚焦的语义单元"，向量表征更准，检索精确性更高。即使检索到，10000 token 塞进 prompt 也挤占上下文预算。分块是"局部性优化"——每个 chunk 聚焦，向量更准，检索后只塞相关部分。整篇向量化只适合"文档级分类"。

### 第二层：证据与定位

**Q：RAG 效果差，怎么定位是分块策略的问题还是 embedding 模型的问题？**

控制变量。1) 固定分块策略，换 embedding 模型（如从 text-embedding-ada-002 换到 bge-large）——如果 recall@5 提升明显，是 embedding 问题；2) 固定 embedding，换分块策略（如从固定窗口换到语义分割）——如果 recall@5 提升，是分块问题。具体看错误 case：如果 gold chunk 在文档里但没被召回，且 gold chunk 跨越了分块边界（被切断了），是分块问题；如果 gold chunk 完整但 embedding 相似度低，是 embedding 问题。

### 第三层：根因深挖

**Q：固定窗口分块会把一句话切成两半，根因是窗口大小不匹配还是固定窗口方法本身有缺陷？**

方法本身有局限。固定窗口按 token 数机械切，不理解语义边界（句子、段落）。即使窗口设得再大，只要文档长度不是窗口的整数倍，总会有切断。根因是"窗口是 token 维度的，语义是内容维度的，两者不对齐"。解法：1) 递归分割——优先按段落/句子分隔符切，切不开再按 token；2) 语义分割——用模型找语义边界切；3) overlap——相邻 chunk 重叠 10-20%，被切断的信息在相邻 chunk 里有备份。

**Q：那为什么不直接用语义分割（最准），而要推荐递归分割（折中）？**

性能和成本的权衡。语义分割要用模型判断每个位置的"语义边界"，处理一篇文档要多次模型推理，慢且贵。递归分割用预定义分隔符（段落符\n\n、句号。、空格）按优先级递归切，纯字符串操作，快且免费。质量上递归分割在结构化文档（技术文档、论文）上接近语义分割（因为段落分隔符本身就是语义边界），在非结构化文档（聊天记录）上略差。经验上生产推荐递归分割（性价比高），对质量要求极高的场景才用语义分割。

### 第四层：方案权衡

**Q：chunk size 设 200、500、还是 1000 token？怎么权衡？**

权衡"语义完整性 vs 检索精确性"。chunk 太小（< 200）——语义被切碎，单 chunk 信息不全，LLM 拿到无法回答；chunk 太大（> 1000）——一个 chunk 多主题，向量表征稀释，检索精确性下降。经验最优 256-512 token，配合 10-20% overlap。最终按数据特性 A/B：技术文档（结构化）用小 chunk（256），叙事文本（连续）用大 chunk（512）。用 eval 集测不同 size 的 recall@5 和 answer_correctness，选最优。

**Q：为什么不直接把 chunk 设成"按段落"（语义最自然），而要按 token 数？**

因为段落长度不可控。有些段落只有 1 句话（< 50 token，向量表征不稳定），有些段落几百句（> 2000 token，超出 embedding 模型的最优输入长度，如 BERT 系上限 512 token）。按 token 数（带语义分隔符）能保证 chunk 长度均匀，向量表征稳定。纯按段落会让 chunk 长度方差极大，检索质量不稳定。所以"按 token 数 + 优先在段落边界切"是两者的平衡。

### 第五层：验证与沉淀

**Q：怎么衡量分块策略的合理性？**

三个指标：1) recall@K——gold chunk 是否在 top-K（应该 > 85%）；2) chunk 完整性——gold chunk 是否被完整保留（没有被切断，用人工抽检）；3) answer_correctness——基于检索 chunk 的答案准确率（应该 > 80%）。对比不同分块策略的三个指标，选最优。沉淀为分块配置规范：chunk size、overlap、分隔符优先级，按文档类型（技术文档/论文/聊天）分别配置。
