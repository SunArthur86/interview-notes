# 工具调用失败如何治理？

## 一、工具调用的失败类型

```
┌──────────────┬─────────────────────┬────────────────────┐
│ 失败类型      │ 原因                  │ 可恢复性            │
├──────────────┼─────────────────────┼────────────────────┤
│ 超时          │ 网络慢/服务负载高      │ 可重试              │
│ 网络错误      │ 连接断/DNS失败        │ 可重试              │
│ 参数错误      │ LLM生成的参数不合法    │ 改参数重试          │
│ 权限不足      │ 用户无权限            │ 不可恢复（告知用户）│
│ 服务不可用    │ 下游服务挂了          │ 降级/换方案         │
│ 结果异常      │ 返回了脏数据          │ 校验后换方案        │
│ 限流          │ 调用太频繁            │ 退避后重试          │
└──────────────┴─────────────────────┴────────────────────┘
```

## 二、四级容错策略

```
工具调用失败
     │
     ▼
┌─────────────────┐
│ Level 1: 重试    │ ← 瞬时错误（超时/网络）
│ 指数退避重试3次  │
└────────┬────────┘
         │ 仍失败
         ▼
┌─────────────────┐
│ Level 2: 降级    │ ← 服务不可用
│ 用缓存/简化版    │
└────────┬────────┘
         │ 无法降级
         ▼
┌─────────────────┐
│ Level 3: 换方案  │ ← 让LLM重新决策
│ 错误回传LLM      │
│ LLM选其他工具    │
└────────┬────────┘
         │ 无替代方案
         ▼
┌─────────────────┐
│ Level 4: 告知    │ ← 不可恢复
│ 诚实告知用户     │
│ 提供替代建议     │
└─────────────────┘
```

## 三、实现代码

```python
class ToolFailureHandler:
    def execute(self, tool_call):
        # Level 1: 带重试的执行
        result = self.execute_with_retry(tool_call, max_retries=3)
        if result.success:
            return result
        
        # Level 2: 尝试降级
        degraded = self.try_degrade(tool_call)
        if degraded:
            return degraded
        
        # Level 3: 错误回传LLM，换方案
        alternative = self.ask_llm_for_alternative(tool_call, result.error)
        if alternative:
            return self.execute(alternative)
        
        # Level 4: 告知用户
        return self.user_friendly_error(tool_call, result.error)
    
    def execute_with_retry(self, tool_call, max_retries=3):
        """指数退避重试"""
        for attempt in range(max_retries):
            try:
                result = self.tool.execute(tool_call, timeout=30)
                if self.validate(result):
                    return Success(result)
            except RetryableError as e:  # 超时/网络错
                wait = 2 ** attempt  # 1s, 2s, 4s
                sleep(wait)
            except ValidationError as e:
                # 参数错，让LLM修参数
                fixed = self.llm.fix_params(tool_call, str(e))
                tool_call.arguments = fixed.arguments
            except NonRetryableError as e:
                return Failure(e)  # 权限/致命错，不重试
        return Failure("重试耗尽")
    
    def try_degrade(self, tool_call):
        """降级策略"""
        # 缓存降级
        if cached := self.cache.get(tool_call.cache_key()):
            return Success(cached, source="cache")
        
        # 简化版降级
        if simpler := self.get_simpler_version(tool_call):
            return self.execute(simpler)
        
        return None
    
    def ask_llm_for_alternative(self, failed_call, error):
        """错误回传LLM，让它换方案"""
        message = {
            "role": "tool",
            "content": f"工具{failed_call.name}调用失败: {error}"
                       f"请换一种方式完成用户需求。"
        }
        # LLM看到错误，可能：
        # 1. 换个工具
        # 2. 用已有信息直接回答
        # 3. 告诉用户无法完成
        response = self.llm.replan(self.context + [message])
        return response.alternative_action
    
    def user_friendly_error(self, tool_call, error):
        """对用户友好的错误提示"""
        return {
            "response": f"抱歉，{tool_call.user_facing_name}暂时不可用。"
                       f"您可以稍后再试，或{self.suggest_alternative()}",
            "fallback": True
        }
```

## 四、预防性措施

```python
# 1. 超时控制
@timeout(30)
def execute_tool(tool_call):
    return tool.execute(tool_call)

# 2. 熔断（错误率过高停止调用）
class CircuitBreaker:
    def __init__(self):
        self.failures = 0
        self.threshold = 5
    
    def can_call(self):
        if self.failures >= self.threshold:
            return False  # 熔断，不再调
        return True
    
    def record_failure(self):
        self.failures += 1

# 3. 限流
@rate_limit(calls=100, per=60)
def execute_tool(tool_call):
    ...

# 4. 结果校验
def validate_result(result, schema):
    """校验工具返回是否符合预期"""
    if not schema.validate(result):
        raise DataError("工具返回数据异常")
```

## 五、错误回传 LLM 的威力

```
传统做法：工具失败→直接报错给用户
  用户体验差，且没有利用LLM的推理能力

智能做法：工具失败→错误回传LLM→LLM自主决策
  ┌──────────────────────────────────────────┐
  │ LLM看到: "search_flight 超时"              │
  │ LLM推理: "航班查询超时，可能是网络问题"     │
  │         "我可以先用get_time确认日期"       │
  │         "然后告诉用户稍后重试"              │
  │ LLM输出: 优雅的降级回复                    │
  └──────────────────────────────────────────┘
  
  → 用户感知不到后端失败，体验流畅
```

## 六、面试加分点

1. **分级容错**：重试→降级→换方案→告知，体系化而非单点
2. **错误回传 LLM**：这是 Agent 区别于传统程序的关键——失败后 LLM 自主决策而非硬编码
3. **预防+治理**：既要有事后治理（重试/降级），也要有事前预防（超时/熔断/限流）

## 记忆要点

- 容错四级防线：L1重试(瞬时错) → L2降级(服务挂) → L3换方案(回传LLM重选) → L4告知(不可恢复)
- 重试必加指数退避：针对超时或网络错，按2^n秒间隔重试以防雪崩
- 参数错误让LLM自愈：拦截异常报错并抛给大模型，引导其修正后重新生成
- 对比重试与放弃：瞬时错重试到底，而权限或致命错立即放弃并友好告知用户


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：工具调用失败要"重试+降级+换方案+告知用户"，为什么不直接报错让用户自己处理？Agent 自己解决不更好吗？**

Agent 自己解决正是这四招的目的——"失败不崩溃，自主决策下一步"。直接报错让用户处理是"低质量体验"（用户用 Agent 就是期望它解决问题，结果还要自己处理）。但"自主解决"不是无脑重试（有些错误重试也没用，如权限不足），要分类处理：可恢复错误（网络抖动）重试、有备选的（主 API 挂）降级、方法错的（参数错）换方案、实在不行的告知用户。这四招覆盖"从轻到重"的失败处理，让 Agent 在失败时仍能给用户"尽力而为"的体验，而非"崩溃"。

### 第二层：证据与定位

**Q：工具调用失败，怎么判断是该"重试"（临时故障）还是"换方案"（方法根本错）？**

看错误类型。1）临时性/外部故障——如超时、网络错误、API 5xx、限流，这些重试可能成功（过一会就好了），优先重试（2-3 次，指数退避）；2）永久性/方法错误——如参数格式错（order_id 不是数字）、业务逻辑错（订单不存在）、权限不足，重试也没用，要换方案（如换工具/换参数/转人工）；3）未知错误——错误信息不明确时，先重试 1 次（防临时故障），仍失败则按"方法错误"处理（换方案）。判断依据：错误码/错误信息分类，工具实现层要返回结构化错误（如 {error_type: "timeout"} vs {error_type: "invalid_param"}），上层据此决策。

### 第三层：根因深挖

**Q：重试策略（指数退避）能处理临时故障，但如果所有重试都失败（如 API 持续故障），Agent 怎么办？**

降级 + 告知。1）降级——主工具（如实时搜索 API）持续故障，降级到备用（如本地缓存/另一个搜索 API/知识库），虽然质量差点但功能可用；2）告知用户——如果降级也做不到（无备选），诚实告知用户"查询服务暂时不可用，已记录您的问题，稍后回复/转人工"，不编造答案（幻觉）；3）异步重试——对非紧急任务，记录失败稍后重试（如 API 恢复后自动重试并推送结果给用户）。核心：宁可"诚实告知+降级"也不"编造答案"，编造（幻觉）比承认失败对用户体验伤害更大（用户信了错误信息）。

**Q：换方案（换个工具/方法解决同一问题）听起来好，但 Agent 怎么知道"还有什么方案"？**

靠工具的"功能等价组"设计 + LLM 推理。1）功能等价组——注册工具时标注"等价工具"（如 search_web 和 search_kb 都能"搜索信息"，互为等价），主工具失败时框架自动推荐等价工具给 LLM；2）LLM 推理——LLM 看到工具失败后，基于"用户意图+可用工具列表"推理替代方案（如"查询 API 失败，我可以用计算工具手动算"），这需要 prompt 引导（"工具失败时思考替代方案"）；3）Reflexion 经验——历史"某工具失败→换某工具成功"的经验存在记忆里，下次类似失败检索出来作为参考。三者结合：框架推荐等价工具 + LLM 推理 + 经验积累，让"换方案"有依据而非瞎试。

### 第四层：方案权衡

**Q：失败处理的四招（重试/降级/换方案/告知）按什么顺序执行？是固定的还是动态判断？**

动态判断，按错误类型路由。固定顺序（重试→降级→换方案→告知）的问题：有些错误重试无用（如权限不足，重试 3 次浪费时间）。正确做法：1）错误分类——工具返回错误类型（临时故障/方法错误/权限不足/无解）；2）按类型路由——临时故障→重试（成功则继续，失败则降级）；方法错误→换方案（换工具/参数）；权限不足→告知用户（要授权）；实在无解→告知+转人工。这个路由是"一步到位"（不固定重试 N 次再降级），减少无谓等待。实务用决策表（错误类型→处理策略），工具实现层返回结构化错误类型，上层查表决策。

**Q：Agent 频繁失败（如 10 次调用 3 次失败）会让用户体验很差，怎么从源头减少失败率？**

监控 + 优化。1）失败率监控——按工具/按错误类型统计失败率，失败率高的工具重点优化（如某 API 失败率 20%，换更可靠的或加缓存）；2）工具质量提升——失败多的工具优化实现（如超时从 5s 提到 30s、加重试、换更稳定的依赖）；3）工具下线/替换——持续高失败率的工具下线（找替代品），避免持续拖垮 Agent；4）参数预校验——LLM 输出参数后、调用前校验（格式/范围），不合法的直接让 LLM 重生成（不浪费一次工具调用）。源头治理让失败率从 30% 降到 <10%，再配合失败处理四招兜底，用户体验可控。

### 第五层：验证与沉淀

**Q：你怎么衡量失败处理机制的有效性（失败时 Agent 真的恢复了而非崩溃）？**

两个指标。1）失败恢复率——工具调用失败后，Agent 通过重试/降级/换方案"成功完成任务"的比例（而非崩溃或转人工），目标 >70%（多数失败能自恢复）；2）用户感知失败率——用户实际感知到"Agent 没解决问题"的比例（包括崩溃+转人工+告知不可用），应 <10%（多数失败用户无感）。两个指标从"Agent 视角"和"用户视角"衡量失败处理效果。还要分错误类型统计恢复率（临时故障恢复率应 >90%，方法错误换方案恢复率应 >50%），针对性优化薄弱环节。

**Q：失败处理四招怎么沉淀成框架的 ResilienceLayer？**

封装成组件：1）错误分类器——工具实现层返回结构化错误类型，框架自动分类；2）重试策略——按错误类型配重试（临时故障：3 次指数退避；其他：不重试）；3）降级链——工具注册时配等价/备选工具，主工具失败自动降级；4）换方案引擎——LLM 推理替代方案 + Reflexion 经验检索；5）告知模板——优雅的用户告知模板（不生硬报错）。开发者注册工具+配等价关系，框架自动处理失败。这套写入团队 Agent 框架 SOP，新 Agent 失败自动恢复，不重写容错逻辑。

## 结构化回答

**30 秒电梯演讲：** 工具调用失败治理=重试+降级+换方案+告知用户。核心是"失败不崩溃"，把错误信息反馈给LLM让它自主决策下一步——像开车遇到封路。

**展开框架：**
1. **失败类型** — 超时/参数错/权限/服务不可用
2. **策略** — 重试→降级→换工具→告知用户
3. **关键** — 错误信息回传LLM，自主决策

**收尾：** 您想深入聊：重试几次合适？——3次，指数退避？
