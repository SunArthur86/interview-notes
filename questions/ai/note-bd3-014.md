# RAG和微调应该如何取舍？什么情况下优先用RAG？什么情况下考虑微调？如何结合？

> 来源：字节跳动大模型技术面试二面

## 决策框架

```
┌─────────────────────────────────────────────────────────┐
│                 RAG vs 微调决策树                        │
│                                                         │
│                  需求是什么？                             │
│                      │                                  │
│            ┌─────────┼─────────┐                        │
│            ▼                   ▼                        │
│       知识注入              能力获取                     │
│       (需要准确信息)        (需要特定行为)               │
│            │                   │                        │
│       知识更新频率?       什么能力?                      │
│        /         \        /        \                    │
│      高           低    格式/风格    推理能力             │
│       │           │       │           │                │
│     RAG ✅    微调 ✅   微调 ✅     微调 ✅              │
│                                                         │
│  组合场景: 微调学格式/风格 + RAG提供知识 = 最优          │
└─────────────────────────────────────────────────────────┘
```

## 对比矩阵

| 维度 | RAG | 微调 |
|------|-----|------|
| **知识更新** | ✅ 实时更新 | ❌ 需要重新训练 |
| **知识溯源** | ✅ 可引用来源 | ❌ 无法溯源 |
| **数据量需求** | ✅ 无限制 | ⚠️ 需要标注数据 |
| **计算成本** | 低（推理时检索） | 高（训练成本） |
| **输出风格** | ❌ 受限于prompt | ✅ 可深度定制 |
| **推理模式** | ❌ 基础模型行为 | ✅ 可改变推理方式 |
| **幻觉控制** | ✅ 有事实约束 | ❌ 可能强化幻觉 |
| **延迟** | ⚠️ 有检索开销 | ✅ 无额外延迟 |
| **隐私** | ✅ 可私有化部署DB | ✅ 可本地训练 |
| **领域适应** | ⚠️ 依赖prompt | ✅ 深度适配 |

## 何时优先RAG

```
✅ 适合RAG的场景:

1. 知识频繁更新
   - 企业FAQ/知识库（每天更新）
   - 新闻/政策文档（实时变化）
   - 产品文档（版本迭代频繁）

2. 需要事实准确性
   - 法律/医疗/金融知识查询
   - 客服系统（回答必须基于真实产品信息）
   - 技术文档问答

3. 需要可溯源
   - 研究报告（需要引用来源）
   - 审计/合规场景（回答必须有依据）

4. 知识量极大
   - 企业全量文档库（百万级文档）
   - 多语言知识库

5. 数据隐私敏感
   - 内部数据不出库（检索时实时查）
```

## 何时考虑微调

```
✅ 适合微调的场景:

1. 特定输出格式
   - JSON/XML结构化输出
   - 特定模板的报告生成
   - 代码风格适配

2. 特定语言风格
   - 品牌语气/口吻
   - 客服话术规范
   - 文学创作风格

3. 特定推理能力
   - 数学推理（CoT增强）
   - 代码生成（语言特定模式）
   - 多步规划（Agent能力）

4. 减少Token消耗
   - 微调后不需要冗长的prompt
   - 系统提示词内化到模型权重

5. 低延迟要求
   - 无需检索的实时场景
   - 边缘设备部署
```

## 工程结合方案

### 方案1: 先微调后RAG（最常见）

```
┌──────────────────────────────────────────────┐
│  Step 1: 微调（格式+风格）                    │
│                                              │
│  基础模型 → SFT → 领域模型                    │
│  (学习输出格式、回答风格、行业术语)             │
│                                              │
│  Step 2: RAG（知识注入）                      │
│                                              │
│  用户Query → 检索领域知识库                    │
│           → 组装Prompt → 领域模型 → 回答      │
│                                              │
│  效果: 模型"说话得体" + "内容准确"            │
└──────────────────────────────────────────────┘
```

### 方案2: RAFT (RAG Fine-Tuning)

```
RAFT训练策略:
  
  训练数据格式:
  {
    "question": "xxx是什么?",
    "context": [正确chunk, 干扰chunk, 干扰chunk],
    "answer": "正确答案（基于正确chunk推理）"
  }

  目标: 教模型在RAG场景下:
  - 正确从多个检索结果中筛选信息
  - 忽略不相关的干扰chunk
  - 学会引用来源

  效果: 比单独微调或单独RAG提升15-25%
```

### 方案3: 动态路由

```python
class HybridSystem:
    """根据查询类型动态选择RAG或微调模型"""
    
    def route(self, query):
        # 分类: 知识查询 vs 能力任务
        intent = self.classify(query)
        
        if intent == "knowledge_query":
            # RAG: 查知识库
            return self.rag_pipeline(query)
        elif intent == "formatting_task":
            # 微调模型: 特定格式输出
            return self.ft_model.generate(query)
        elif intent == "complex_reasoning":
            # 两者结合
            context = self.rag_retrieve(query)
            return self.ft_model.generate(query, context=context)
```

## 实际案例分析

```
案例: 企业智能客服系统

需求:
  1. 回答基于公司产品文档（知识查询）
  2. 按客服话术规范回答（风格控制）
  3. 能处理多轮对话（上下文理解）
  4. 无法回答时转人工（边界感知）

方案: 微调(LoRA) + RAG 组合

  微调数据: 5万条高质量客服对话（格式+话术+边界）
  RAG知识库: 产品文档 + FAQ + 历史工单

  流程:
    用户问题 
      → 检索知识库 (RAG获取相关文档)
      → 微调模型生成回答 (LoRA控制格式和风格)
      → 置信度判断 (低置信度→转人工)

  指标:
    准确率: 85% (纯RAG: 72%, 纯微调: 65%)
    风格一致性: 92% (纯RAG: 68%)
    转人工率: 15% (纯RAG: 30%)
```

**面试加分点**：提到微软的GraphRAG将知识图谱与RAG结合用于复杂推理；提到RAFT论文(2024)通过在训练数据中混入干扰文档来提升RAG鲁棒性；提到Tuning-playground模式（先RAG验证PMF，再微调优化体验）是产品迭代最佳实践；提到微调后的模型仍可以叠加RAG，两者是正交的增强手段而非互斥方案。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RAG 解决"知识注入"，微调解决"能力获取"。为什么不直接用微调注入知识（把知识训练进权重），省得搞 RAG 的检索链路？**

微调注入知识有"更新慢 + 不可溯源 + 知识容量受限"的问题。一是更新慢——知识（如产品信息、政策）频繁更新，每次更新都要重新微调（成本高、周期长，一周训一次），而 RAG 只需更新文档库（秒级生效）；二是不可溯源——微调把知识融进权重，模型无法说出"知识来自哪个文档"（溯源难，幻觉风险高），RAG 的答案可以标注引用来源（用户信任）；三是知识容量受限——模型权重容量有限（70B 模型的知识容量约几亿 token，超了会"遗忘"），大量知识微调会导致"灾难性遗忘"（老知识被新知识覆盖），RAG 的知识存在外部库（容量无限）；四是更新成本——微调要标注数据 + 训练 + 部署（重），RAG 只需写文档入库（轻）。所以"频繁更新的知识"用 RAG，"稳定的能力"（格式/风格/推理）用微调，两者互补。

### 第二层：证据与定位

**Q：你的业务场景：用户问"最新的退货政策"，模型回答了旧政策。怎么定位是 RAG 没检索到新文档、还是微调的模型"固执"地用训练时的旧知识？**

看 RAG 的检索结果和模型生成。一是检索结果——RAG 是否召回了新政策文档（如果召回的是旧文档，是检索问题——新文档没入库或 embedding 没更新；如果召回了新文档但模型仍答旧政策，是模型问题）；二是模型生成——如果召回了新文档（在 prompt 里），但模型仍答旧政策，是模型"依赖参数化知识"（训练时学的旧政策）而非"看 prompt 的新政策"，这是微调导致的"知识固执"（模型对训练时的知识更自信，忽略 prompt 的新信息）；三是 prompt 设计——prompt 是否明确要求"基于检索结果回答，忽略你的先验知识"，如果没有，模型可能混用先验和检索。治法：检索问题更新文档库；模型固执问题强化 prompt（"严格基于以下资料回答，不要用你的先验知识"）或改用更强的模型（对 prompt 更服从）；极端情况用 RAG-only 模式（封闭域生成，禁止自由发挥）。

### 第三层：根因深挖

**Q：微调你说"改变推理模式"。微调怎么能改变推理模式？不是只学数据吗？**

微调通过"数据模式"重塑模型的推理习惯。模型的"推理模式"是训练数据分布塑造的——如果微调数据全是"逐步推理"格式（如 CoT，"第一步... 第二步... 结论"），模型学会"逐步推理"的习惯（即使推理时没明确要求 CoT，模型也会倾向分步）；如果微调数据是"直接给答案"，模型学会"直觉回答"（快但不深入）。微调的本质是"调整模型的输出分布"，让模型在特定任务上更擅长（如代码微调让模型更会写代码，数学微调让模型更会做数学）。RAG 无法改变推理模式（它只提供知识，不改模型的生成习惯）——给一个"直接回答"习惯的模型加 RAG，模型仍直接回答（只是用检索到的知识），不会变成"逐步推理"。所以"改变推理模式"必须微调，RAG 只能"提供事实"。

**Q：那为什么不直接用 Prompt 工程改变推理模式（如 prompt 里写"请逐步推理"），省得微调？**

Prompt 工程有效但不稳定且成本高。在 prompt 里加"请逐步推理"（CoT prompt）确实能让模型逐步推理，但问题：一是"不一致"——模型有时遵循 prompt（逐步推理），有时忽略（直接回答），取决于 prompt 的措辞和模型的理解，不稳定；二是"token 成本"——每次请求都要在 prompt 里写 CoT 指令（消耗 token），且 CoT 输出长（逐步推理的输出比直接答案长 3-5 倍），推理成本高；三是"能力上限"——prompt 只能"激发"模型已有的能力（如果模型本身不擅长推理，prompt 让它推理也推不好），微调能"内化"推理能力（通过大量 CoT 数据训练，模型的推理能力提升到新水平）。微调后的模型无需 CoT prompt 也会逐步推理（习惯内化），且推理质量更高（训练提升了能力）。所以 Prompt 是"轻量激发"，微调是"重量内化"，对核心能力（如推理）用微调，对临时需求用 Prompt。

### 第四层：方案权衡

**Q："先微调定制风格 + 再 RAG 注入事实"你说是黄金组合。为什么不只用 RAG（用 prompt 指定风格 + 检索事实）？**

RAG 的 prompt 指定风格不稳定。在 prompt 里写"用幽默风格回答"+ 检索事实，模型"大概"能幽默，但风格控制不稳定（有时幽默、有时严肃，取决于 query 和检索内容），且"幽默"的风格在不同 query 上一致性差。微调定制风格是把"幽默风格"内化到权重（用大量幽默风格的样本微调），模型稳定输出幽默风格（无需 prompt 提示，所有 query 都幽默），一致性高。且微调能定制"复杂风格"（如模仿特定作家、特定品牌语调），prompt 难以精确描述（"像海明威一样简洁"这种风格，prompt 描述模糊，微调能学得更像）。所以风格定制用微调（稳定 + 精确），事实注入用 RAG（实时 + 可溯源），两者叠加是"稳定的风格 + 实时的事实"。生产场景（如品牌客服）先微调让模型说"品牌风格"，再 RAG 注入实时产品信息。

**Q：为什么不直接用最强的基础模型（如 GPT-4），它的风格和推理已经很好，省得微调？**

强模型好但贵 + 不可控 + 数据隐私。一是成本——GPT-4 的 token 单价是开源模型（自部署）的 10-100 倍，高频场景成本爆炸；二是可控性——GPT-4 是黑盒，无法精确定制风格（只能用 prompt 粗略控制），微调开源模型（Llama）能精确定制；三是数据隐私——业务数据（如客服对话、内部知识）不能发给第三方（GPT-4），必须用自部署的开源模型 + 微调；四是依赖风险——依赖 GPT-4 有"断供"风险（OpenAI 限流/停服/涨价），自部署开源模型自主可控。所以即使 GPT-4 更强，很多企业仍选择"开源模型 + 微调"（可控 + 经济 + 隐私）。微调的价值是"让开源模型逼近 GPT-4 在特定任务上的表现"，用领域数据弥补开源模型和 GPT-4 的差距。

### 第五层：验证与沉淀

**Q：你怎么决策用 RAG 还是微调，证明选择是对的？**

用"四维决策框架"。一是"知识 vs 能力"——需求是"注入事实知识"（如产品信息）用 RAG，是"改变生成能力"（如风格、格式、推理）用微调；二是"更新频率"——知识频繁更新（如每天）用 RAG（实时更新文档），稳定不变（如风格）用微调（一次训长期用）；三是"溯源需求"——需要溯源（如医疗、法律，答案要标注来源）用 RAG，不需要溯源（如创意写作）可微调；四是"数据隐私"——知识敏感（如内部文档）两者都可用（RAG 检索私有库，微调用私有数据训）。做 A/B 测试——对同一需求分别用 RAG 和微调实现，对比效果（质量/成本/延迟），选更优的。关键验证"是否选对"——如果 RAG 的答案有幻觉（检索不准导致），可能该换微调（内化知识更稳）；如果微调的答案过时（知识更新了但模型没重训），可能该换 RAG（实时检索）。

**Q：RAG vs 微调的决策怎么沉淀成团队规范？**

固化成"决策矩阵"：按"知识/能力 × 更新频率 × 溯源需求 × 隐私"四维分类，每类有推荐方案（如"频繁更新的知识 + 需溯源 → RAG"、"稳定的能力 + 不需溯源 → 微调"、"都要 → 微调 + RAG 组合"）。沉淀"各业务场景的案例"（如客服 = 微调风格 + RAG 知识、代码助手 = 微调代码能力 + RAG 文档、医疗问答 = RAG 为主 + 微调医学推理）。配套评估流程（新需求按决策矩阵选方案，用 A/B 验证，效果不达标调整）。把"先判断知识/能力，再按更新频率选"作为团队的标准化决策流程，避免"拍脑袋选方案"。定期复盘（回顾过去的选择是否合理，更新决策矩阵的经验）。

## 结构化回答

**30 秒电梯演讲：** RAG解决"知识注入"（动态、可更新），微调解决"能力获取"（风格、格式、推理方式）——RAG像开卷考试。

**展开框架：**
1. **RAG** — 知识更新快、需要溯源、数据量大 → 优先
2. **微调** — 需要特定输出格式/风格/推理模式 → 优先
3. **两者可以组合** — 微调提升基础能力 + RAG注入实时知识
