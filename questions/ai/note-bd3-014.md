# RAG和微调应该如何取舍？什么情况下优先用RAG？什么情况下考虑微调？如何结合？

> 来源：字节跳动大模型技术面试二面

## 决策框架

```
┌─────────────────────────────────────────────────────────┐
│                 RAG vs 微调决策树                        │
│                                                         │
│                  需求是什么？                             │
│                      │                                  │
│            ┌─────────┼─────────┐                        │
│            ▼                   ▼                        │
│       知识注入              能力获取                     │
│       (需要准确信息)        (需要特定行为)               │
│            │                   │                        │
│       知识更新频率?       什么能力?                      │
│        /         \        /        \                    │
│      高           低    格式/风格    推理能力             │
│       │           │       │           │                │
│     RAG ✅    微调 ✅   微调 ✅     微调 ✅              │
│                                                         │
│  组合场景: 微调学格式/风格 + RAG提供知识 = 最优          │
└─────────────────────────────────────────────────────────┘
```

## 对比矩阵

| 维度 | RAG | 微调 |
|------|-----|------|
| **知识更新** | ✅ 实时更新 | ❌ 需要重新训练 |
| **知识溯源** | ✅ 可引用来源 | ❌ 无法溯源 |
| **数据量需求** | ✅ 无限制 | ⚠️ 需要标注数据 |
| **计算成本** | 低（推理时检索） | 高（训练成本） |
| **输出风格** | ❌ 受限于prompt | ✅ 可深度定制 |
| **推理模式** | ❌ 基础模型行为 | ✅ 可改变推理方式 |
| **幻觉控制** | ✅ 有事实约束 | ❌ 可能强化幻觉 |
| **延迟** | ⚠️ 有检索开销 | ✅ 无额外延迟 |
| **隐私** | ✅ 可私有化部署DB | ✅ 可本地训练 |
| **领域适应** | ⚠️ 依赖prompt | ✅ 深度适配 |

## 何时优先RAG

```
✅ 适合RAG的场景:

1. 知识频繁更新
   - 企业FAQ/知识库（每天更新）
   - 新闻/政策文档（实时变化）
   - 产品文档（版本迭代频繁）

2. 需要事实准确性
   - 法律/医疗/金融知识查询
   - 客服系统（回答必须基于真实产品信息）
   - 技术文档问答

3. 需要可溯源
   - 研究报告（需要引用来源）
   - 审计/合规场景（回答必须有依据）

4. 知识量极大
   - 企业全量文档库（百万级文档）
   - 多语言知识库

5. 数据隐私敏感
   - 内部数据不出库（检索时实时查）
```

## 何时考虑微调

```
✅ 适合微调的场景:

1. 特定输出格式
   - JSON/XML结构化输出
   - 特定模板的报告生成
   - 代码风格适配

2. 特定语言风格
   - 品牌语气/口吻
   - 客服话术规范
   - 文学创作风格

3. 特定推理能力
   - 数学推理（CoT增强）
   - 代码生成（语言特定模式）
   - 多步规划（Agent能力）

4. 减少Token消耗
   - 微调后不需要冗长的prompt
   - 系统提示词内化到模型权重

5. 低延迟要求
   - 无需检索的实时场景
   - 边缘设备部署
```

## 工程结合方案

### 方案1: 先微调后RAG（最常见）

```
┌──────────────────────────────────────────────┐
│  Step 1: 微调（格式+风格）                    │
│                                              │
│  基础模型 → SFT → 领域模型                    │
│  (学习输出格式、回答风格、行业术语)             │
│                                              │
│  Step 2: RAG（知识注入）                      │
│                                              │
│  用户Query → 检索领域知识库                    │
│           → 组装Prompt → 领域模型 → 回答      │
│                                              │
│  效果: 模型"说话得体" + "内容准确"            │
└──────────────────────────────────────────────┘
```

### 方案2: RAFT (RAG Fine-Tuning)

```
RAFT训练策略:
  
  训练数据格式:
  {
    "question": "xxx是什么?",
    "context": [正确chunk, 干扰chunk, 干扰chunk],
    "answer": "正确答案（基于正确chunk推理）"
  }

  目标: 教模型在RAG场景下:
  - 正确从多个检索结果中筛选信息
  - 忽略不相关的干扰chunk
  - 学会引用来源

  效果: 比单独微调或单独RAG提升15-25%
```

### 方案3: 动态路由

```python
class HybridSystem:
    """根据查询类型动态选择RAG或微调模型"""
    
    def route(self, query):
        # 分类: 知识查询 vs 能力任务
        intent = self.classify(query)
        
        if intent == "knowledge_query":
            # RAG: 查知识库
            return self.rag_pipeline(query)
        elif intent == "formatting_task":
            # 微调模型: 特定格式输出
            return self.ft_model.generate(query)
        elif intent == "complex_reasoning":
            # 两者结合
            context = self.rag_retrieve(query)
            return self.ft_model.generate(query, context=context)
```

## 实际案例分析

```
案例: 企业智能客服系统

需求:
  1. 回答基于公司产品文档（知识查询）
  2. 按客服话术规范回答（风格控制）
  3. 能处理多轮对话（上下文理解）
  4. 无法回答时转人工（边界感知）

方案: 微调(LoRA) + RAG 组合

  微调数据: 5万条高质量客服对话（格式+话术+边界）
  RAG知识库: 产品文档 + FAQ + 历史工单

  流程:
    用户问题 
      → 检索知识库 (RAG获取相关文档)
      → 微调模型生成回答 (LoRA控制格式和风格)
      → 置信度判断 (低置信度→转人工)

  指标:
    准确率: 85% (纯RAG: 72%, 纯微调: 65%)
    风格一致性: 92% (纯RAG: 68%)
    转人工率: 15% (纯RAG: 30%)
```

**面试加分点**：提到微软的GraphRAG将知识图谱与RAG结合用于复杂推理；提到RAFT论文(2024)通过在训练数据中混入干扰文档来提升RAG鲁棒性；提到Tuning-playground模式（先RAG验证PMF，再微调优化体验）是产品迭代最佳实践；提到微调后的模型仍可以叠加RAG，两者是正交的增强手段而非互斥方案。
