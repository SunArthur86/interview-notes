# 【RAG混合检索】RRF是什么？为什么不用分数加权融合？

> 来源：小红书「前端 AI 项目必问：为啥不能只用向量检索？」（OCR图片内容）

## 一、为什么不能直接用分数加权

```
问题场景：

向量检索结果:                    BM25检索结果:
┌────────┬──────────┐          ┌────────┬──────────┐
│ 文档    │ 余弦分数  │          │ 文档    │ BM25分数 │
├────────┼──────────┤          ├────────┼──────────┤
│ 文档A  │ 0.95     │          │ 文档A  │ 28.5     │
│ 文档B  │ 0.87     │          │ 文档C  │ 22.3     │
│ 文档C  │ 0.82     │          │ 文档B  │ 15.1     │
└────────┴──────────┘          └────────┴──────────┘

直接加权 0.5×cos + 0.5×bm25:
  文档A: 0.5×0.95 + 0.5×28.5 = 14.73  ← BM25完全压制！
  文档B: 0.5×0.87 + 0.5×15.1 = 7.99
  文档C: 0.5×0.82 + 0.5×22.3 = 11.56

问题：BM25分数(0-30) >> 余弦分数(0-1)
      加权后被BM25完全主导，向量检索结果被淹没
```

## 二、RRF算法详解

```
RRF (Reciprocal Rank Fusion)

公式: RRF_score(d) = Σ_{i=1}^{M} 1 / (k + rank_i(d))

其中:
  M = 检索器数量（这里是2：向量+BM25）
  k = 平滑参数（通常取60）
  rank_i(d) = 文档d在第i个检索器中的排名（从1开始）

示例计算（k=60）:

文档A: 向量排第1, BM25排第1
  RRF = 1/(60+1) + 1/(60+1) = 1/61 + 1/61 = 0.0328

文档B: 向量排第2, BM25排第3
  RRF = 1/(60+2) + 1/(60+3) = 1/62 + 1/63 = 0.0322

文档C: 向量排第3, BM25排第2
  RRF = 1/(60+3) + 1/(60+2) = 1/63 + 1/62 = 0.0322

→ 文档A得分最高（两个检索器都排第1）
→ 文档B和C分数相近（一个向量靠前一个BM25靠前）
```

### k值的影响

```
k值对分数的影响:

           RRF分数
    0.035 ┤
    0.030 ┤  ●━━━ k=1 (对top排名极敏感)
    0.025 ┤      ╲
    0.020 ┤       ●━━━ k=10 (中等敏感)
    0.015 ┤            ╲
    0.010 ┤             ●━━━ k=60 (平缓，工业标配)
    0.005 ┤
    0.000 ┤──┬──┬──┬──┬──┬──
           1  5  10 15 20  排名

k=1: 第1名=0.5, 第2名=0.33 → 差距大，top-1主导
k=60: 第1名=0.0164, 第2名=0.0161 → 差距小，多路共识主导
```

## 三、代码实现

```python
def rrf_fusion(vector_results, bm25_results, k=60, top_n=20):
    """
    RRF排名融合
    vector_results: 向量检索结果列表（已按相似度排序）
    bm25_results: BM25检索结果列表（已按分数排序）
    """
    scores = {}
    
    # 向量检索排名
    for rank, doc in enumerate(vector_results, 1):
        doc_id = doc['id']
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    
    # BM25检索排名
    for rank, doc in enumerate(bm25_results, 1):
        doc_id = doc['id']
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    
    # 按RRF分数排序
    ranked = sorted(scores.items(), key=lambda x: -x[1])
    return ranked[:top_n]

# 使用示例
fused = rrf_fusion(vector_top10, bm25_top10, k=60, top_n=20)
```

## 四、方案对比

| 融合方法 | 原理 | 优点 | 缺点 | 适用场景 |
|---------|------|------|------|---------|
| RRF | 倒数排名求和 | 简单、无需调参、稳定 | 不区分检索器质量 | 通用混合检索 |
| 加权RRF | 带权重的RRF | 可给更准的检索器更高权重 | 需要调权重 | 有先验信息 |
| CombSUM | 归一化分数求和 | 利用分数信息 | 归一化方法敏感 | 分数可比时 |
| Learned Ranking | ML模型学习排序 | 效果最好 | 需要训练数据 | 大规模搜索 |

## 五、面试加分点

1. **工业界验证**：Elasticsearch 8.x原生支持RRF，说明其工业界认可度——提及ES的RRF retriever让面试官知道你关注实际工具
2. **多路扩展**：RRF天然支持>2路检索器——可以加第三路（如元数据过滤检索、知识图谱检索），公式不变
3. **k值调优经验**：k=60是论文推荐值，但在特定场景可以调：FAQ类（结果集小）用k=30，大文档库用k=80
4. **与Reranker配合**：RRF是「召回阶段」的融合（快但粗），Reranker是「精排阶段」（慢但准）——两者是串联关系不是替代关系
5. **分布式计算**：大规模场景下RRF可以分布式计算——每个检索节点返回TopK排名，中心节点做RRF聚合，延迟可控

## 结构化回答

**30 秒电梯演讲：** RRF不看分数只看排名——不同检索器的分数尺度不同（余弦0-1 vs BM25可达30+），直接比分数不公平，比排名才合理。

**展开框架：**
1. **核心问题** — 向量检索分数0-1，BM25分数0-30+，尺度完全不同
2. **RRF公式** — score(doc) = Σ 1/(k + rank_i(doc))，k通常取60
3. **不看分数只** — 不看分数只看排名，消除尺度差异

**收尾：** 您想深入聊：RRF的k值对不同场景有最优值吗？怎么调？
