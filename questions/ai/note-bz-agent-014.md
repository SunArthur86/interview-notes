# 主流认知框架（CoT/ReAct/Plan-Execute/Reflexion）各自适用什么场景？

## 一、四大框架速览

```
┌──────────────┬────────────────┬──────────────────────┐
│ 框架          │ 一句话           │ 核心机制               │
├──────────────┼────────────────┼──────────────────────┤
│ CoT          │ 先想清楚再答     │ 推理链(Thought→Answer) │
│ ReAct        │ 边想边做         │ Thought→Act→Obs循环   │
│ Plan-Execute │ 先规划全局再干   │ Plan→Execute→Replan   │
│ Reflexion    │ 失败了反思再试   │ Act→Eval→Reflect循环  │
└──────────────┴────────────────┴──────────────────────┘
```

## 二、各框架详解与适用场景

### 1. CoT（思维链）

```
机制：Thought → Thought → ... → Answer
      （纯推理，无外部行动）

适用：
  ✓ 数学计算（"鸡兔同笼"）
  ✓ 逻辑推理（"甲乙丙谁说谎"）
  ✓ 常识推理（"为什么天空是蓝的"）
  ✗ 需要实时信息的（"今天天气"）
  ✗ 需要执行动作的（"发邮件"）

特点：最简单，单次调用，适合"脑内可解"的问题
```

### 2. ReAct（推理+行动）

```
机制：Thought → Action → Observation → Thought → ...

适用：
  ✓ 需要查信息的（"最新新闻""库存查询"）
  ✓ 需要工具的（"计算""搜索""调用API"）
  ✓ 通用Agent任务（客服/助手）
  ✗ 步骤超多且明确的（用Plan更高效）
  ✗ 完全无工具的纯推理（用CoT即可）

特点：最通用，是现代Agent的基础范式
```

### 3. Plan-and-Execute（计划-执行）

```
机制：Plan(全局规划) → Execute(逐步) → Replan(必要时)

适用：
  ✓ 步骤明确可分解的（"写报告""部署系统"）
  ✓ 多步骤有依赖的（"做调研→分析→总结"）
  ✓ 需要全局视角的（避免ReAct的局部贪心）
  ✗ 高度不确定需探索的（计划容易失效）
  ✗ 简单任务（杀鸡用牛刀）

特点：全局规划，适合结构化复杂任务
```

### 4. Reflexion（反思）

```
机制：Act → Evaluate → Reflect → 重试（带反思记忆）

适用：
  ✓ 有明确成败反馈的（代码/数学/游戏）
  ✓ 难题（一次做不对，需迭代）
  ✓ 错误可诊断的（能说出"为什么错"）
  ✗ 主观任务（评估难，反思无依据）
  ✗ 实时性要求高的（多轮太慢）

特点：从错误学习，适合"可验证"的难题
```

## 三、按任务特征选型

### 决策树

```
任务来了
   │
   ├─ 需要外部信息/工具吗？
   │    ├─ 否 → CoT（纯推理即可）
   │    │
   │    └─ 是 → 步骤是否多且明确？
   │         ├─ 否（需探索）→ ReAct
   │         │
   │         └─ 是（可分解）→ Plan-Execute
   │
   └─ 失败后能判断对错吗？
        ├─ 能 → 加上 Reflexion（迭代优化）
        └─ 不能 → 一次性，不反思
```

### 场景对照表

| 场景 | 推荐框架 | 理由 |
|------|---------|------|
| "23×17=?" | CoT | 纯计算，脑内可解 |
| "今天新闻" | ReAct | 需搜索（工具） |
| "写市场调研报告" | Plan-Execute | 多步骤可分解 |
| "修这个bug" | Reflexion | 可测试验证，需迭代 |
| "客服对话" | ReAct | 灵活，需查信息 |
| "证明数学定理" | CoT+Reflexion | 推理+可验证 |

## 四、框架的组合使用

生产中常**组合**多个框架，而非单选：

```
混合架构示例：

┌──────────────────────────────────────────────┐
│  Plan-Execute 做骨架（全局规划）               │
│                                                │
│  Plan: [step1, step2, step3, step4]            │
│                                                │
│  每个step内部用 ReAct 执行（保持灵活）          │
│    step2: Thought→Act→Obs→Thought...          │
│                                                │
│  失败时触发 Reflexion（反思重试）               │
│    step3失败 → Reflect → 带经验重试            │
│                                                │
│  推理环节本质都是 CoT                          │
└──────────────────────────────────────────────┘
```

```python
def hybrid_agent(goal):
    # Plan-Execute骨架
    plan = planner.decompose(goal)
    
    reflections = []  # Reflexion记忆
    
    for step in plan.steps:
        # 每步用ReAct执行（内部Thought是CoT）
        for trial in range(MAX_TRIALS):
            result = react_agent.run(step, reflections)
            if result.success:
                break
            # Reflexion反思
            reflections.append(reflect(step, result))
        else:
            # 多次失败，上升到Replan
            plan = planner.replan(goal, plan)
    
    return aggregate(plan.results)
```

## 五、成本与延迟对比

```
┌──────────────┬────────┬──────────┬────────────┐
│ 框架          │ 调用次数 │ Token消耗 │ 延迟        │
├──────────────┼────────┼──────────┼────────────┤
│ CoT          │ 1       │ 低        │ 最低        │
│ ReAct        │ N步     │ 中        │ 中          │
│ Plan-Execute │ 1+N     │ 中        │ 中          │
│ Reflexion    │ K轮×N步 │ 高        │ 高          │
└──────────────┴────────┴──────────┴────────────┘

成本递增：CoT < ReAct ≈ Plan-Execute < Reflexion
```

## 六、面试加分点

1. **不要说"哪个最好"**：每个框架有适用场景，选型匹配任务特征才专业
2. **强调组合**：生产中几乎都是混合架构（Plan 骨架 + ReAct 节点 + Reflexion 兜底）
3. **从任务结构出发**：可分解？需工具？可验证？——这三个问题决定框架选择

## 记忆要点

- 选型决策树：需外部工具用ReAct，多步且明确用Plan-Execute，可验证难题加Reflexion
- 场景对比：纯脑内计算用CoT，查新闻用ReAct，写长篇报告用Plan-Execute
- 生产实践：常组合使用，如Plan做骨架，ReAct执行步骤，Reflexion处理失败


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：CoT、ReAct、Plan-Execute、Reflexion 四个框架，为什么不能做一个"万能框架"把它们全包了？非要分四个？**

因为它们解决的是不同维度的问题，强行合并会互相干扰。CoT 解决"单步推理深度"，ReAct 解决"推理与行动交织"，Plan-Execute 解决"全局规划"，Reflexion 解决"跨任务学习"。一个任务的"确定性强弱、是否需工具、步骤多少、是否可迭代"不同，最优框架就不同。如：简单推理任务用 CoT 够，套上 Plan-Execute 的规划层是多余开销；复杂多步任务用 Plan-Execute，CoT 单步推理撑不住。万能框架会"什么都支持但什么都不优"——为了通用性牺牲每个场景的最优性。所以工程上是"框架选型路由"而非"万能框架"，按任务特性选最合适的。

### 第二层：证据与定位

**Q：一个 Agent 任务效果差，你怎么判断是"选错框架"（该用 Plan-Execute 却用了 ReAct）还是"框架内参数没调好"？**

看失败模式。1）如果是"方向性错误"（如做了无关步骤、漏关键步骤、走弯路才发现错），通常是框架选错——ReAct 缺全局视野导致，换 Plan-Execute 改善；2）如果是"单步执行错"（如某步工具调用参数错、推理错），是框架内问题（prompt/工具/LLM 能力），换框架没用；3）如果是"重复犯错"（同类错误反复出现），缺 Reflexion 的经验积累层。判断方法：分析 trace 的失败点——失败在"步骤规划"层（该做什么做错了）→ 框架选型问题；失败在"单步执行"层（知道做什么但做错了）→ 框架内调优。先定位再对症下药，避免盲目换框架。

### 第三层：根因深挖

**Q：四大框架里，Plan-Execute 在实际项目里采用率最低（多数用 ReAct），为什么？是 Plan-Execute 不好吗？**

不是不好，是"适用场景窄 + 实现门槛高"。1）适用场景窄——Plan-Execute 适合"步骤多、有依赖、需全局优化"的复杂任务，但实际业务多数任务是"中等复杂度"（3-5 步），ReAct 的逐步决策够用，Plan-Execute 的全局规划优势不明显；2）实现门槛高——Plan-Execute 要写 Planner（生成计划）+ Executor（执行）+ 重规划机制（失败回溯），比 ReAct 的单循环复杂，工程成本高；3）规划质量依赖 LLM——Planner 的计划质量直接决定任务成败，弱模型规划差导致 Plan-Execute 反而不如 ReAct（逐步至少能适应）。所以 Plan-Execute 在"超高复杂度 + 强模型"场景才有优势，多数场景 ReAct 性价比更高。

**Q：既然 ReAct 最通用，为什么不所有任务都用 ReAct，把 CoT/Plan-Execute/Reflexion 都当 ReAct 的"模式"内嵌进去？**

可以内嵌，但"框架"的价值在于"显式化 + 可配置"。ReAct 可以在 Thought 里做 CoT（深度推理）、在循环外加 Reflexion（反思层），本质上四大框架可以组合。但把它们做成"独立框架"而非"全塞进 ReAct"的好处：1）显式化——每个框架的 prompt 模板、控制流、评测指标不同，独立框架让每种模式的最优实践清晰；2）可配置——开发者按任务选框架（配置驱动），而非在 ReAct 里调"要不要加规划/要不要加反思"的开关；3）可对比——独立框架方便 AB 测试（同任务用不同框架对比效果）。实务确实有"混合框架"（如 Plan-Execute + 每步 ReAct + 失败后 Reflexion），但这是"组合"不是"塞进一个万能框架"。

### 第四层：方案权衡

**Q：实际项目里，你按什么维度选框架？给一个决策树。**

三个维度决策：1）是否需要外部工具——不需要（纯推理）用 CoT；需要（要查/算/API）用 ReAct/Plan-Execute；2）任务复杂度（步骤数）——简单（1-3 步）用 ReAct（逐步够），复杂（5+ 步、有依赖）用 Plan-Execute（先规划）；3）是否需要跨任务学习——同类任务反复出现用 Reflexion（积累经验），一次性任务不用。决策树：纯推理→CoT；需工具+简单→ReAct；需工具+复杂→Plan-Execute；任何框架+重复类任务→+Reflexion 层。还要看模型能力：弱模型（7B）用简单的 CoT/ReAct（Plan-Execute 的规划质量撑不住），强模型（70B/R1）可用 Plan-Execute 发挥全局规划优势。

**Q：四大框架都不是完美的，有没有"第五种框架"解决了它们的共同短板？**

前沿研究方向有几个候选。1）Tree-of-Thoughts（ToT）——解决 CoT/ReAct 单链的"一步错全错"，用树形多分支探索+回溯，但成本高（前述）；2）Self-Consistency——生成多条 CoT 链投票，降低单链出错，但成本翻倍；3）Agentic RL（如 R1）——把"何时推理/何时行动/如何规划"的能力内化到模型权重，不再依赖固定框架，模型自主决定策略，这是终极方向（框架能力→模型能力）。目前看 Agentic RL 训练的模型（如 R1）能根据任务自主选择推理深度、工具调用、规划策略，相当于"框架内化"，是四大框架的进化方向。但训练成本高，多数团队仍用"框架+通用模型"的组合。

### 第五层：验证与沉淀

**Q：你怎么证明你选的框架是"这个任务的最优解"，而不是"换个框架更好"？**

框架 AB 测试。固定任务集，分别用 CoT/ReAct/Plan-Execute（+/- Reflexion）跑，对比：1）完成率/准确率；2）效率（步数/最优步数、延迟、成本）；3）稳定性（多次运行的方差）。最优框架是"完成率最高 + 成本可接受 + 稳定性好"的那个。如果 ReAct 和 Plan-Execute 完成率差 <3% 但 ReAct 成本低 30%，选 ReAct；如果 Plan-Execute 完成率高 15% 且任务价值高，选 Plan-Execute。这个 AB 测试要在任务立项时做一次（定框架），任务演进中定期重测（任务变了最优框架可能变）。

**Q：四大框架的选型经验怎么沉淀成团队 SOP？**

整理成"框架选型决策表"：任务维度（需工具/复杂度/重复性）× 推荐框架 × 适用模型 × 默认配置。配套：1）框架模板库——每个框架的标准 prompt 模板 + 实现代码，开发者按表选框架后套模板；2）AB 测试脚本——固定任务集，一键跑多框架对比，输出完成率/成本/稳定性报告；3）框架评测集——按任务类型（推理/多跳/规划/迭代）各建评测集，新框架/新模型自动跑回归。这套写入团队 Agent SOP，新人按决策表选框架 + 套模板，避免"凭感觉选框架"和"每次重写框架实现"。

## 结构化回答

**30 秒电梯演讲：** 四大认知框架各有所长——CoT擅长单步推理，ReAct擅长边想边做，Plan-Execute擅长全局规划，Reflexion擅长从错误学习。选型看任务的复杂度、确定性和反馈可得性。

**展开框架：**
1. **CoT** — 单步深度推理（数学/逻辑）
2. **ReAct** — 推理+行动交替（通用Agent）
3. **Plan-Execute** — 先规划后执行（多步确定任务）

**收尾：** 您想深入聊：这些框架能组合吗？——能，Plan-Execute的每步可用ReAct，ReAct失败可加Reflexion？
