# 主Agent和子Agent的模型该怎么分配，为什么不一定都用同一个大模型？

## 分配原则

```
多Agent系统中的模型分层：

┌──────────────────────────────────┐
│         主Agent (Router)          │
│         模型: 32B / GPT-4         │
│         职责: 意图理解+任务拆分     │
│               +结果整合+质量把关    │
│         要求: 强推理+全局视野       │
└──────────┬───────────┬───────────┘
           │           │
     ┌─────▼───┐  ┌───▼─────┐  ┌────────┐
     │子Agent A │  │子Agent B │  │子Agent C│
     │7B/14B    │  │7B/14B   │  │14B/32B  │
     │搜索执行   │  │数据处理  │  │质量验证  │
     │高频简单  │  │结构化    │  │需判断力  │
     └─────────┘  └─────────┘  └────────┘
```

## 分配决策框架

```python
class ModelAssignmentPolicy:
    """根据子任务特征分配模型"""

    def assign(self, subtask: dict) -> str:
        complexity = subtask['complexity']    # simple/medium/complex
        quality_req = subtask['quality_req']  # high/medium/low
        frequency = subtask['frequency']      # high/low

        # 决策矩阵
        if complexity == 'complex' or quality_req == 'high':
            return 'qwen-32b'   # 复杂推理/高质量要求
        elif complexity == 'medium':
            return 'qwen-14b'   # 中等任务
        else:
            return 'qwen-7b'    # 简单高频任务

    def get_assignment_table(self):
        return {
            # 子Agent角色 → 推荐模型 → 理由
            '搜索执行Agent': '7B',
              # 任务简单（调API+格式化），7B准确率>95%，成本最低
            '信息提取Agent': '7B',
              # 结构化提取（NER/分类），7B足够
            '翻译Agent': '14B',
              # 翻译需要理解语义，14B质量更好
            '摘要Agent': '14B',
              # 需要理解全文逻辑，14B更可靠
            '质量验证Agent': '32B',
              # 质量验证需要批判性思维，32B判断更准
            '安全审核Agent': '32B',
              # 安全相关不能出错，用最强模型
            '创意生成Agent': '32B',
              # 创意需要发散思维，32B更有多样性
            '主Agent(路由)': '32B',
              # 路由决策影响全局，必须用最强模型
        }
```

## 动态升级机制

```python
class AdaptiveModelAgent:
    """子Agent结果质量不达标时自动升级模型"""

    async def execute_with_escalation(self, task: dict):
        # Level 1: 先用小模型尝试
        result = await self.call_model('7b', task)

        # 质量自评
        quality_score = await self.assess_quality(result)

        if quality_score >= 0.85:
            return result  # 质量够用

        # Level 2: 升级到14B
        result = await self.call_model('14b', task)
        quality_score = await self.assess_quality(result)

        if quality_score >= 0.85:
            return result

        # Level 3: 最终用32B兜底
        return await self.call_model('32b', task)
```

## 成本对比

```
场景：日均10万次请求，5个子Agent

全用32B：  10万 × 5 × ¥8 = ¥4,000,000/天  ❌ 太贵
全用7B：   10万 × 5 × ¥0.5 = ¥250,000/天  ❌ 质量不够
差异化分配：
  主Agent(32B): 10万 × ¥8 = ¥800,000
  搜索Agent(7B): 10万 × ¥0.5 = ¥50,000
  提取Agent(7B): 10万 × ¥0.5 = ¥50,000
  摘要Agent(14B): 10万 × ¥2 = ¥200,000
  验证Agent(32B): 10万 × ¥8 = ¥800,000
  总计: ¥1,900,000/天  ✅ 节省52.5%
```

## 面试加分点

1. **不是越小越好**：关键路径上的Agent（路由、验证、安全）绝不能省模型，省下的成本不够一次事故的损失
2. **混合厂商**：主Agent用GPT-4（推理强），子Agent用开源模型（成本低），通过统一API抽象层管理
3. **监控指标**：每个子Agent独立监控成功率、延迟、成本，发现退化及时调整模型
4. **A/B测试**：换模型必须做A/B测试，确保质量不下降再全量切换

## 记忆要点

- 核心结论：为了兼顾成本与质量，主子 Agent 必须采用差异化模型分配
- 分配原则：主 Agent 及复杂安全审核用大模型(如 32B)
- 子 Agent 分配：高频简单执行用小(7B)，中等理解用中(14B)
- 动态升级：先用小模型探底，质量自评不达标时自动升级大模型兜底

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：主 Agent 用 32B、子 Agent 用 7B 的差异化分配听起来合理。但如果全用 32B，质量肯定最优（每个任务都用最强模型），成本高但用户满意。为什么不优先保证质量，成本是次要的？**

因为商业 Agent 的可持续性要求"成本 ≤ 用户价值"，无限堆成本不可持续。具体分析：一是单位经济性（unit economics）——全用 32B，单次请求成本 ¥32（5 子 Agent × ¥8 + 主 ¥8），如果单次请求给用户创造的价值（如节省时间）折合 ¥10，那每单亏 ¥22，不可持续。差异化分配成本 ¥9.5，若质量损失可接受（< 5%），单单接近盈亏平衡或微利，可持续。二是质量边际递减——7B 到 14B 质量提升显著（如准确率 85%→92%），14B 到 32B 提升递减（92%→95%）。简单任务（搜索、提取）7B 已 95%+，用 32B 的边际收益 < 1%，成本翻倍不划算。三是用户体验不只是质量——延迟也是体验。32B 推理慢（200-500ms），多 Agent 串行总延迟高；7B 快（50-100ms），整体响应更快，用户感知"流畅"比"质量极致"更重要。四是场景容忍度——简单任务（搜索、提取）即使偶有错误，用户可容忍（重试成本低）；关键任务（安全审核、支付）不能错，用大模型。所以质量优先是"关键路径保质量，非关键路径控成本"，而非"全用最大模型"。全用 32B 是质量最优但商业不可持续，差异化是"质量够用 + 成本可控 + 延迟优"的工程最优。

### 第二层：证据与定位

**Q：你说简单任务 7B 准确率 95%+。但怎么验证"7B 够用"而不只是假设？如果 7B 在某些边界情况（如复杂查询、歧义输入）失败，怎么发现？**

需数据驱动的验证，不能假设。一是离线评估：构建测试集（含简单、中等、复杂、边界情况），分别用 7B/14B/32B 跑，对比准确率。如果 7B 在简单子集 95%+、边界子集 80%（低于 14B/32B 的 90%），说明 7B 在边界不够，需升级或路由。二是线上 A/B：部分流量用 7B，部分用 14B/32B，对比用户行为指标（任务完成率、重试率、满意度）。如果 7B 组的完成率显著低（如 85% vs 14B 的 92%），说明 7B 不够，需调整。三是质量自评（self-assessment）：7B 执行后自评置信度（如让模型输出 confidence），低置信度的结果路由到 14B/32B 复核（动态升级机制）。通过监控自评置信度分布和复核率，发现 7B 的薄弱场景。四是错误案例库：收集 7B 失败的案例（用户反馈、人工审核），分类失败模式（复杂查询、歧义、长尾实体），针对性优化（补充 prompt、微调、或该类任务固定用 14B）。五是持续监控：线上 7B 的准确率监控（抽样标注），分布漂移（新查询类型）导致准确率下降时调整。验证逻辑是"离线测试集 + 线上 A/B + 质量自评 + 错误案例库 + 持续监控"，数据驱动确认"7B 够用"的范围，边界场景动态升级。

### 第三层：根因深挖

**Q：动态升级机制（7B 先试，质量不达标升 14B，再不达标升 32B）听起来优雅。但"质量自评"怎么实现？模型怎么判断自己的输出"够不够好"？模型能可靠自评吗？**

模型自评是难题，完全可靠不可能，但可用启发式 + 交叉验证提升可靠性。自评方法：一是置信度输出——让模型在输出结果时同时输出置信度（如 "confidence: 0.7"），基于 logit 或 prompt 要求。但模型置信度校准差（过度自信），不能直接用。二是结构化检查——针对任务类型设计检查规则：搜索任务的"结果数量 > 0 且相关"、提取任务的"输出符合 schema 且字段非空"、翻译的"语言正确且无未翻译段"。这些规则比模型自评更可靠（确定性）。三是交叉验证——7B 输出后，用另一个 7B（或同模型不同 prompt）复核，两者不一致则升级。成本是 2×7B，仍比直接 32B 便宜。四是参考答案对比——对有 ground truth 的任务（如分类），7B 输出与已知答案对比，不一致升级。五是启发式信号——输出长度异常（过短可能失败）、重复（模型卡住）、格式错误（JSON 解析失败）作为升级信号。模型自评的局限：模型不知道自己不知道（unknown unknowns），对错误输出可能高置信度。所以不能纯依赖模型自评，要结合结构化检查和交叉验证。实践配置：结构化检查（确定性，首选）+ 置信度阈值（辅助）+ 交叉验证（关键任务）+ 错误信号（格式/长度异常），多信号触发升级。动态升级不是"模型自评说了算"，是"多信号综合判断"。

**Q：那如果动态升级的触发阈值（如置信度 < 0.85 升级）设错，会导致大量不必要的升级（浪费成本）或漏升级（质量差）。阈值怎么调？升级率多少是合理的？**

阈值调优是成本和质量的权衡，需数据驱动。一是基线分析：离线测试集上，统计 7B 在不同置信度区间的准确率（如置信度 0.8-0.9 的准确率 88%，0.9-1.0 的准确率 96%），找到"准确率突降"的置信度点作为阈值（如 0.85 以下准确率 < 85%，设为升级阈值）。二是升级率监控：线上升级率 = 升级请求数 / 总请求数。升级率 10-30% 通常合理（多数 7B 够用，少数升级）；> 50% 说明 7B 对该任务普遍不够，应直接用 14B（避免频繁升级的延迟和复杂度）；< 5% 说明阈值过严（漏升级，质量差），应放宽。三是质量 vs 成本曲线：扫描不同阈值（0.7/0.8/0.85/0.9），记录每个阈值下的总成本和整体准确率，画帕累托曲线，选"成本可接受且准确率达标"的阈值。四是延迟考量：升级增加延迟（7B 失败后再跑 14B/32B，串行），升级率高的任务延迟显著增加。对延迟敏感场景，阈值宜宽（少升级）或直接用中模型（14B）避免升级。五是动态调整：根据线上反馈（用户重试率、满意度）动态调阈值，如重试率高则收紧阈值（多升级保质量）。实践：起步用经验阈值（0.85），离线分析校准，线上监控升级率和质量指标，A/B 调优。没有"通用最优阈值"，按任务和场景调，关键是监控和迭代。

### 第四层：方案权衡

**Q：主 Agent 用 32B 做路由决策。但路由决策（判断任务类型、分配子 Agent）真的需要 32B 吗？这个决策本质上像分类，用 7B 分类器不行吗？为什么路由要用最强模型？**

路由决策看似简单（分类），实际复杂，32B 的必要性在于"全局视野 + 推理 + 容错"。一是路由不只是分类——它包括意图理解（多意图拆分、隐式意图）、任务拆分（一个大任务拆成多个子任务，需理解任务结构）、依赖分析（子任务间的依赖关系）、结果整合（多子任务结果汇总成连贯回复）。这些是复杂推理，非简单分类，7B 容易拆错或漏依赖。二是路由错误代价高——路由是全局决策，错一次影响所有子任务。如用户"查天气并订机票"，路由错把"订机票"漏掉，整个任务失败；或依赖分析错（先订机票再查天气，但用户想根据天气决定行李），逻辑混乱。32B 的强推理降低路由错误，避免全局失败。三是容错与降级——主 Agent 要处理子任务失败（某子任务超时/错误），决定降级策略（跳过/重试/换模型）。这需理解任务语义（哪些可跳过、哪些必须重试），7B 判断力不足。四是少量但关键——主 Agent 只调一次（路由决策），即使 32B 成本 ¥8，占总成本比例小（¥9.5 中的 ¥8）；而子 Agent 调多次（5 次），用 7B 省 ¥7.5×5。所以主 Agent 用 32B 的"单次高成本"换"全局正确性"，性价比高。但简单场景（单意图、无拆分）可用 7B 分类器路由，32B 只用于复杂多意图——分级路由（7B 初判，复杂时升级 32B）进一步优化。

**Q：不同厂商模型混用（主 Agent GPT-4，子 Agent 开源 Qwen）听起来灵活。但混用会增加复杂度（多 API、多格式、多错误模式）。混用的收益真的大于复杂度成本吗？为什么不统一用一家？**

混用的收益在特定场景大于复杂度，但需权衡。混用的收益：一是能力互补——GPT-4/Claude 推理强、英文优（适合主 Agent 的复杂路由），开源模型（Qwen/GLM）中文优、成本低（适合子 Agent 的执行）、可本地部署（数据隐私）。单一厂商可能某维度弱（如 GPT-4 中文不如 Qwen，Qwen 推理不如 GPT-4）。二是成本优化——关键路径用强模型（GPT-4），非关键用便宜开源，总成本低于全用 GPT-4。三是避免锁定——混用降低对单一厂商的依赖（厂商涨价/停服/降质时有替代）。复杂度成本：一是 API 抽象层——需统一接口（如 OpenAI 兼容格式），封装不同厂商的差异（请求格式、错误码、限流），增加开发维护。二是 Prompt 适配——不同模型对 prompt 的响应不同（GPT-4 偏指令、Claude 偏 XML、Qwen 偏中文），需分别优化 prompt，工作量倍增。三是错误处理——各厂商的错误模式不同（超时、限流、格式错误），错误处理逻辑复杂。四是质量不一致——混用模型的能力差异可能导致整体行为不一致（某子任务质量波动）。实践权衡：如果团队有能力维护抽象层，混用收益大（能力 + 成本 + 灵活性）；如果团队小、追求简单，统一用一家（如全 Qwen 或全 GPT-4）降低复杂度。折中方案是"主用一家 + 关键任务备选另一家"（如主 Qwen，安全审核用 GPT-4），而非全混用。这是团队能力和场景驱动的决策。

### 第五层：验证与沉淀

**Q：模型分配策略（哪个 Agent 用哪个模型）上线后，怎么持续验证"分配合理"？如果某子 Agent 的 7B 实际不够用（准确率低），怎么发现并调整？**

多层监控和调整机制。一是分 Agent 指标监控：每个子 Agent 独立监控成功率、延迟、成本、升级率（动态升级的触发率）。如果某子 Agent 成功率低（< 90%）或升级率高（> 40%），说明 7B 不够，考虑升级到 14B。二是用户行为归因：用户投诉或任务失败时，追溯是哪个子 Agent 出错（通过 trace 日志），统计各子 Agent 的错误贡献。错误集中的子 Agent 是薄弱环节，需升级模型或优化 prompt。三是 A/B 测试调整：发现某子 Agent 可能不够，A/B 测试（7B vs 14B），对比成功率和成本，数据驱动决定是否升级。四是成本效益复盘：定期复盘总成本和各 Agent 成本占比，结合质量指标，评估分配是否最优。如某 32B 子 Agent 成本占比高但成功率与 14B 接近，降级到 14B 省 成本。五是模型迭代跟进：新模型发布（如 Qwen 2.5 vs 2.0）时，重新评估分配（新 7B 可能追上旧 14B），及时调整。证明逻辑是"分 Agent 监控 → 错误归因 → A/B 调整 → 成本复盘 → 模型迭代"，形成"监控-发现-调整-验证"的闭环。模型分配不是一次定终身，是根据线上数据和模型迭代持续优化的动态策略。

**Q：怎么让团队在做模型分配决策时不凭直觉（"感觉这个用 7B"），而是有数据支撑的系统方法？**

沉淀分配决策框架和数据驱动流程。一是决策框架：按任务维度（复杂度、质量要求、频率、关键性）评分，查推荐模型表（如复杂或关键 → 32B，中等 → 14B，简单高频 → 7B），框架标准化避免直觉。二是起步基线：新 Agent 起步用中等模型（14B），跑两周收集数据（成功率、错误模式），再决定降级（7B）或保持。避免起步就赌 7B（可能不够）或 32B（浪费）。三是离线评估规范：每个候选分配方案在测试集上评估（准确率、延迟、成本），数据对比而非直觉。四是线上 A/B 规范：模型调整必做 A/B（如 7B vs 14B），对比业务指标（完成率、满意度、成本），数据驱动切换。五是成本追踪：按 Agent 和模型追踪成本（API 费用），定期生成成本报告，发现高成本低收益的分配。六是踩坑库：常见错误（关键路径用小模型导致事故、动态升级阈值不当导致延迟高、混用模型 prompt 不适配）及案例。让模型分配是"框架评分 → 基线起步 → 离线评估 → A/B 验证 → 成本追踪"的数据驱动流程，不靠直觉。

## 结构化回答

**30 秒电梯演讲：** 主Agent和子Agent不一定用同一模型。主Agent负责任务路由和编排（需强推理→大模型），子Agent负责具体执行（简单任务→小模型），通过差异化分配优化成本和质量。

**展开框架：**

2. **执行类子Agent** — 搜索、提取、翻译→7B/14B
3. **验证类子Agent** — 质量检查、安全审核→14B
