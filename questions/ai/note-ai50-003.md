# 向量检索和关键词检索各适合什么场景？为什么做混合检索？

## 两种检索的本质区别

```
┌─────────────────────────────────────────────────┐
│              用户Query: "怎么提高模型准确率"        │
│                                                   │
│  ┌─── 向量检索 ───┐    ┌─── 关键词检索 ───┐     │
│  │ 匹配语义相似度   │    │ 匹配字面关键词     │     │
│  │                 │    │                   │     │
│  │ ✅ "提升LLM精度" │    │ ❌ (不含"准确率")  │     │
│  │ ✅ "优化模型效果" │    │ ❌ (不含关键词)    │     │
│  │ ❌ "准确率定义"   │    │ ✅ "准确率评估方法" │     │
│  └─────────────────┘    └───────────────────┘    │
│                                                   │
│  混合检索 = 两者结果合并 → 覆盖最全面               │
└─────────────────────────────────────────────────┘
```

## 适用场景对比

| 维度 | 向量检索 | 关键词检索(BM25) |
|------|---------|-----------------|
| 语义匹配 | ✅ 同义词、近义词都能命中 | ❌ 只能字面匹配 |
| 精确匹配 | ❌ 专有名词、ID容易漏 | ✅ 精确命中 |
| 数值/代码 | ❌ 不擅长 | ✅ 精确匹配 |
| 长尾词 | ❌ 训练数据少时效果差 | ✅ 不依赖训练 |
| 多语言 | ✅ 跨语言匹配 | ❌ 需要分词器支持 |
| 实时性 | ⚠️ 新文档需要重新embedding | ✅ 倒排索引即时生效 |
| 可解释性 | ❌ 黑盒相似度 | ✅ 可以看到匹配了哪些词 |

### 典型适用场景

**向量检索更适合:**
- "怎么做情感分析" → 召回"文本情感分类方法"
- "性能优化方案" → 召回"系统提速策略"
- 概念性、开放性查询

**关键词检索更适合:**
- "Elasticsearch 7.17" → 精确版本号
- "ERROR_CODE_4096" → 精确错误码
- "SELECT * FROM" → 代码片段
- 精确性、特定性查询

## 混合检索实现

### 标准流水线

```python
def hybrid_retrieval(query, top_k=10):
    """向量+关键词混合检索"""
    
    # Step 1: 双路独立召回
    vector_results = vector_store.search(
        embedding_model.encode(query),
        top_k=top_k * 3  # 扩大召回
    )
    
    keyword_results = es_client.search(
        index="documents",
        body={
            "query": {"match": {"content": query}},
            "size": top_k * 3
        }
    )['hits']['hits']
    
    # Step 2: RRF融合
    return rrf_fusion(vector_results, keyword_results, top_k)

def rrf_fusion(*result_lists, k=60, top_k=10):
    """
    Reciprocal Rank Fusion
    k: 平滑常数，通常60
    """
    scores = {}
    doc_sources = {}  # 记录每篇文档来自哪路检索
    
    for list_idx, results in enumerate(result_lists):
        source_name = ['vector', 'keyword'][list_idx]
        for rank, doc in enumerate(results):
            doc_id = doc['id']
            if doc_id not in scores:
                scores[doc_id] = 0
                doc_sources[doc_id] = []
            scores[doc_id] += 1.0 / (k + rank + 1)
            doc_sources[doc_id].append(source_name)
    
    # 按RRF分数排序
    ranked = sorted(scores.items(), key=lambda x: -x[1])[:top_k]
    return [(doc_id, score, doc_sources[doc_id]) for doc_id, score in ranked]
```

### 为什么选RRF而不是加权融合？

| 方法 | 公式 | 优点 | 缺点 |
|------|------|------|------|
| 加权融合 | `α·s_vec + (1-α)·s_kw` | 直观，可调权重 | 两路分数分布不同，不公平 |
| RRF | `Σ 1/(k+rank)` | 无需归一化，对分数分布不敏感 | 忽略了分数的绝对值 |
| CombSUM | `Σ normalize(s)` | 考虑了绝对分数 | 依赖归一化质量 |

**工程结论**: RRF是最稳健的选择，不需要调参，在大多数场景下效果优于或接近加权融合。

## 工程实践要点

1. **双写一致性**: 文档入库时同时写ES和向量库，用消息队列保证最终一致
2. **Embedding版本管理**: 换embedding模型时需要重新构建整个向量库
3. **分词器选择**: 中文用IK分词器，代码用标准分词器
4. **召回量**: 一般取top_k的3-5倍做初始召回，给RRF足够的候选池

## 记忆要点

- 向量检索靠语义相似度（懂同义词），关键词检索靠字面匹配（精准抓专有名词ID）。
- 向量检索擅长开放性概念提问，而关键词检索擅长特定数值、错误码和代码精确查询。
- 混合检索用RRF（倒数排名融合）而非加权融合，因为RRF无需分数归一化且对异常值更鲁棒。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：混合检索你说用 RRF 融合向量和关键词两路。为什么不能只用向量检索（语义强，覆盖同义词）？**

向量检索对"精确匹配"无能为力。向量检索靠语义相似度，对"开放性概念"（如"怎么做红烧肉"，能召回"猪肉烹饪方法"等同义内容）强，但对"精确字符串"（如"错误码 ERR_12345"、"订单号 ABC123"、"函数名 calculate_tax"）弱——这些精确字符串的 embedding 和通用 embedding 空间不对齐（"ERR_12345" 的 embedding 可能和"ERR_99999"几乎一样，向量检索分不清，召回错的）。关键词检索（BM25）靠字面匹配，精确抓"ERR_12345"（只召回含这个字符串的文档），精准。两路互补：向量管"语义模糊查询"，关键词管"精确匹配查询"，RRF 融合两路结果。只走向量会漏掉"精确匹配"场景（用户查错误码、订单号、代码符号时，向量召回不准），生产 RAG 必须混合。

### 第二层：证据与定位

**Q：混合检索的某些查询结果比单路（纯向量或纯关键词）还差。怎么定位是 RRF 融合问题、还是某一路质量差拖累？**

看两路各自的 Recall 和融合后的 Recall。一是两路 Recall——针对失败 query，看纯向量 Recall 和纯关键词 Recall，如果两路都低（如向量 0.5、关键词 0.4），是两路都差（embedding 和分词都需优化），RRF 融合也救不了（垃圾进垃圾出）；如果一路高一路低（如向量 0.85、关键词 0.5），融合后应接近高的那路（向量），如果融合反而低于向量（如 0.7），是 RRF 融合问题（关键词路的噪声拉低了融合）。二是 RRF 参数——k 值（如 60）是否合理，k 太小让某一路的 top-1 主导（winner-take-all），调 k 看是否改善。三是两路召回数——如果关键词路召回很少（如只有 2 个结果），对 RRF 的贡献小（可能被向量路淹没），需增加关键词路的召回量。治法：两路都差优化各自（embedding/分词）；RRF 融合差调 k 或给两路不同权重（加权 RRF）；召回少增加 top_k。

### 第三层：根因深挖

**Q：RRF 你说"用排名融合，不用分数"。为什么不用加权分数融合（看起来更精确，能调权重）？**

分数融合有"分数不可比"的根因问题。向量的分数是 cosine 相似度（0-1），关键词的分数是 BM25（可能是 5-30），两者尺度不同。即使归一化（min-max 或 z-score），两路的分数分布形态不同（向量可能正态分布、BM25 可能长尾），归一化后仍不可比（同一归一化值在两路代表不同的"相关程度"）。加权融合（如 0.7 × 向量分数 + 0.3 × 关键词分数）依赖分数可比，不可比则融合结果无意义。RRF 用"排名"（序数，第 1、第 2），排名语义一致（两路都是"越靠前越相关"），可比。RRF 回避了"分数不可比"的难题，只看排名，简单有效。加权融合在"分数可对齐"时（如两路都用概率）更精确，但向量和 BM25 的分数不可对齐，RRF 是务实选择。

**Q：那为什么不直接用 LambdaMART（学习排序模型，训练融合权重），精度比 RRF 高？**

LambdaMART 精度高但需要标注数据且复杂。LambdaMART 是"学习排序"模型，用标注数据（query-doc 对的相关性标签）训练一个融合模型（特征是两路的分数/排名等，输出是融合后的排序），精度比 RRF 高 2-5%。但代价：一是需标注数据——要标几千到几万的 query-doc 对的相关性（人工标注成本高）；二是训练管线——要维护 LambdaMART 的训练/评估/部署，工程复杂；三是冷启动——新业务没标注数据时无法用 LambdaMART（RRF 零参数立即可用）。工程优先级：先用 RRF 跑起来（80% 精度，20% 成本），有标注数据后升级 LambdaMART 榨取剩余精度。RRF 是"务实起步"，LambdaMART 是"精细优化"。大多数场景 RRF 够用（精度提升 2-5% 不值得标注成本），只有对精度极致要求的场景（如搜索排序）才上 LambdaMART。

### 第四层：方案权衡

**Q：关键词检索你用 BM25（Elasticsearch）。为什么用 BM25 而非 TF-IDF 或更新的算法？**

BM25 是 TF-IDF 的改进且工程成熟。BM25 相比 TF-IDF 加了"饱和函数"（词频达到一定程度后边际效益递减，防止高频词主导）和"文档长度归一化"（长文档天然词频高，归一化防长文档占优），检索质量优于 TF-IDF。且 BM25 是 Elasticsearch/Lucene 的默认评分算法，工程成熟（优化好、部署简单、生态完善）。更新的算法（如 BM25F、DFRee）在某些场景略优，但提升小（1-2%），生态不如 BM25（ES 原生支持）。所以 BM25 是"经典且够用"的选择。中文场景的关键是"分词"（BM25 基于词，中文要用 IK 分词器或 jieba），分词质量比评分算法更重要（分词错则 BM25 失效）。所以关键词检索的核心是"选好分词器"（中文 IK、代码标准分词器），BM25 作为评分算法足够。

**Q：为什么不直接用 Elasticsearch 的 KNN 检索（ES 也支持向量检索），把向量和关键词放在一个系统，省得维护两套（ES + Milvus）？**

ES 的 KNN 在大规模时性能不如专业向量库。ES 的 KNN（基于 HNSW 或 LSH）在百万级向量时性能可接受（延迟几十 ms），但千万级以上性能下降（ES 不是专门的向量库，ANN 优化不如 Milvus/Qdrant）。且 ES 的 KNN 功能较新（8.x 才成熟），某些高级特性（如 IVF-PQ、量化）支持不如专业库。生产大规模 RAG（千万级文档）用"ES（关键词）+ Milvus（向量）"两套，各自最优（ES 的 BM25 最强、Milvus 的 ANN 最快）。小规模（<百万）用 ES KNN（一套系统省运维），够用。选型看规模——小规模用 ES 统一（省事），大规模用两套（性能优）。趋势是 ES 的 KNN 在进步（未来可能统一），但当前大规模仍专业分开。

### 第五层：验证与沉淀

**Q：你怎么衡量混合检索（向量 + 关键词 + RRF）的效果，证明比单路好？**

定义指标：一是 Recall@K（混合 vs 纯向量 vs 纯关键词，混合应 >任一单路，因互补）；二是"查询类型覆盖"（语义查询和精确查询都能召回，混合应覆盖两类）；三是 nDCG（排序质量）；四是延迟（两路并行 + RRF 融合的总延迟，应 <100ms）。做对比实验：纯向量 vs 纯关键词 vs RRF 混合，在"语义查询"和"精确查询"两类 golden set 上分别对比。预期：语义查询上向量/混合优于关键词，精确查询上关键词/混合优于向量，混合在两类上都不少于单路最优。关键验证"互补性"——如果混合后没超过单路，说明两路结果高度重叠（没互补价值），要调一路（换 embedding 或分词）。A/B 测试线上效果——单路 vs 混合，看用户满意度和"查不到"的反馈率。

**Q：混合检索方案怎么沉淀成 RAG 系统标配？**

固化成"混合检索 pipeline"：向量路（bge-m3 + Milvus）+ 关键词路（BM25 + ES）并行召回、RRF 融合（k=60）、可选 Reranker 精排。沉淀"分词器选型表"（中文 IK、英文标准、代码专用）、"RRF 参数经验"（k=60）、"两路召回数配比"（向量 top-50、关键词 top-50）。配套监控（Recall@K、两路各自的贡献率、延迟），异常告警。把"向量 + 关键词 + RRF"作为 RAG 检索的默认架构（覆盖语义和精确两类查询），新业务接入即获得混合检索能力。积累"各领域的 embedding 和分词器推荐"（如医疗用领域 embedding + 专业词典分词），帮助新业务快速配置。

## 结构化回答

**30 秒电梯演讲：** 向量检索擅长语义匹配，关键词检索擅长精确匹配，两者互补才能覆盖所有查询场景——就像找文件。

**展开框架：**
1. **向量检索** — 语义相似，适合模糊查询、概念查询
2. **关键词检索(BM25)** — 精确匹配，适合专有名词、ID、代码、数值
3. **混合检索** — 混合检索 = 向量召回 + BM25召回 + RRF/加权融合
