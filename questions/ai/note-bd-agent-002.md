# AI Agent项目实际用到了哪些Agent技术能力？

## 四大核心能力全景

```
┌─────────────────────────────────────────┐
│              Agent 核心能力              │
├─────────────┬───────────┬───────┬───────┤
│  Workflow   │  Memory   │  RAG  │ Tool  │
│   编排      │   管理     │ 检索  │ 调用  │
├─────────────┼───────────┼───────┼───────┤
│ 任务拆分     │ 短期记忆   │ 向量  │ 查询  │
│ 流程控制     │ 长期记忆   │ 检索  │ 保存  │
│ 失败恢复     │ 摘要压缩   │ 混合  │ 更新  │
│ 状态管理     │ 过期清理   │ 检索  │ 接口  │
└─────────────┴───────────┴───────┴───────┘
```

## 1. Workflow编排

**职责**：控制生成流程，决定节点执行顺序和条件分支

```python
# 典型工作流编排
workflow = [
    ("planner", "任务拆分"),      # 接收用户输入，生成章节计划
    ("rag", "知识检索"),          # 检索历史章节和角色卡
    ("writer", "内容生成"),       # 基于计划+上下文生成
    ("reviewer", "质量检查"),     # 检查连贯性和一致性
    ("repair", "问题修复"),       # 定向修复问题段落
    ("save", "结果存储"),         # 保存到数据库
]
```

**关键设计**：
- 每个节点有明确的输入/输出Schema
- 节点间通过共享State传递数据
- 支持条件路由（审稿通过→保存，未通过→修复）

## 2. Memory管理

| 类型 | 内容 | 存储方案 | 生命周期 |
|------|------|---------|---------|
| 短期记忆 | 最近N章摘要 | Redis | 会话级 |
| 长期记忆 | 角色设定、世界观 | 向量数据库 | 永久 |
| 工作记忆 | 当前任务计划 | 内存State | 单次生成 |

**核心问题与解决**：
- **设定遗忘**：长期记忆注入每次Prompt → 人设稳定
- **上下文溢出**：摘要压缩 + 滑动窗口 → 控制Token
- **检索精度**：Embedding + 关键词混合 → 召回率提升

## 3. RAG检索

```
用户请求 → Query改写 → 向量化 → 向量检索(Top-K) 
    → 重排序 → 上下文组装 → 注入Prompt
```

**实际用途**：
- 召回历史章节内容（保证剧情连续）
- 检索角色设定卡（保证人设一致）
- 查询世界观设定（避免设定冲突）

## 4. Tool Calling

```python
# 注册的工具能力
tools = [
    {"name": "query_chapter", "desc": "查询指定章节内容"},
    {"name": "save_content", "desc": "保存生成内容到数据库"},
    {"name": "search_character", "desc": "搜索角色设定信息"},
    {"name": "update_status", "desc": "更新任务执行状态"},
    {"name": "call_business_api", "desc": "调用业务系统接口"},
]
```

**设计要点**：
- 每个Tool有明确的Schema定义（输入/输出/错误处理）
- 执行时设置超时和重试机制
- 敏感操作（写入/删除）需要权限校验

## 四大能力的协同关系

```
用户意图
  │
  ▼
Workflow ──→ 调用RAG（检索知识）
  │              │
  │              ▼
  │         注入Memory（组合上下文）
  │              │
  ▼              ▼
Tool ←─── LLM推理决策 ───→ 更新Memory
  │
  ▼
执行结果 → 存入Memory → 反馈给Workflow
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 的四大能力（Workflow/Memory/RAG/Tool）你说是标配，为什么不能只用其中一两个？比如只要 RAG 不行吗？**

因为单能力覆盖不了 Agent 的完整定义。只有 RAG 是"问答系统"（被动检索回答），不是 Agent（主动执行任务）。Agent 的核心是"感知→规划→行动"闭环：Workflow 编排这个闭环的流程，Memory 维持闭环的状态（跨轮次记忆），RAG 提供闭环的知识（外部信息），Tool 实现"行动"（与外部系统交互）。缺任何一个，闭环都断：没 Tool 只能说不能做，没 Memory 每轮从零开始，没 RAG 只靠参数知识。四大能力是 Agent 的必要条件，不是可选项。

### 第二层：证据与定位

**Q：你的 Agent 在多轮对话里"忘了"用户第一轮说的关键信息，怎么定位是 Memory 没存、存了没检索到、还是检索到了没被用？**

三段排查。一是看 Memory 写入日志——第一轮的关键信息是否被写入长期记忆（向量化存储），如果没写是存储逻辑问题（如只存了最后一句没存关键事实）；二是看检索日志——当前轮 query 检索 Memory 是否召回了那条记忆，如果召回分数低是 embedding/检索问题（query 和记忆表述差异大）；三是看生成 Trace——检索到的记忆是否进了 prompt 且模型是否引用，如果进 prompt 了但模型没用是生成层问题（prompt 没强调"参考历史记忆"）。

### 第三层：根因深挖

**Q：Memory 你用向量库存，但发现"用户上周说他偏好中文回答"这条记忆没被正确召回，根因是什么？**

根因可能是 embedding 不匹配或记忆污染。embedding 不匹配：用户当时说"请用中文"，当前轮 query 是"帮我查订单"，两者 embedding 相似度低，召回不到——偏好信息是"全局设定"而非"与当前 query 相关"，向量检索（按 query 相似度召回）天然召回不到全局设定。治本是把 Memory 分两类：episodic（事件记忆，按 query 相似度召回）和 semantic/global（用户偏好等全局设定，每轮都注入 prompt，不依赖检索）。记忆污染：向量库里存了太多低价值记忆，噪声淹没了有用记忆，要做记忆筛选和价值评分。

**Q：那为什么不直接把所有历史对话全塞进 prompt，省得搞 Memory 检索？**

上下文窗口有限且 token 成本高。多轮对话历史可能上万 token，全塞进 prompt 超过模型窗口（如 8k/32k），且长 prompt 导致模型注意力衰减（lost in the middle）和成本翻倍。Memory 的价值是"压缩历史为可检索的关键事实"——把 10 轮对话压成 20 条结构化记忆（如"用户偏好中文""用户订单号是 XXX"），每轮只检索相关的 3-5 条注入 prompt，既不丢关键信息又不爆 token。全塞是"暴力"，Memory 是"智能压缩 + 按需召回"。

### 第四层：方案权衡

**Q：短期记忆存 Redis、长期记忆走向量库，为什么分开？不能都用向量库？**

访问模式和生命周期不同。短期记忆（当前会话的上下文）是高频读写、会话级生命周期（会话结束可清），用 Redis（内存、低延迟、自动 TTL 过期）最合适；长期记忆（用户偏好、历史事实）是低频写、按需读、永久保存，用向量库（支持语义检索、持久化）最合适。都用向量库的话，短期记忆的每次轮次都写向量库（embedding 计算开销大），且会话级数据混入永久库造成噪声。分开是"按访问模式选存储"，Redis 保速度，向量库存知识。

**Q：为什么不直接用大模型的长上下文（如 Gemini 的 1M token）装所有历史，省得搞 Memory 分层？**

长上下文是"装得下"但不等于"用得好"。研究表明，即使 1M 上下文，模型对中间位置的信息召回率显著下降（needle-in-haystack 测试），且长 prompt 的推理质量和指令遵循都下降。更现实的是成本——1M token 的单次推理费用极高，每轮对话都带 1M 历史不可持续。Memory 分层是"主动筛选关键信息"而非"全量塞入"，既保证模型 focus 在相关信息上，又控成本。长上下文适合"单次处理超长文档"的场景，不适合"多轮对话累积历史"。

### 第五层：验证与沉淀

**Q：你怎么衡量 Agent 的四大能力各自是否达标，而不是整体看"能用"？**

定义各能力的独立指标。Workflow：任务完成率、平均节点数、流程偏离率；Memory：记忆召回率（该用的历史信息是否被召回）、记忆污染率（无用记忆占比）；RAG：Context Recall/Precision/Faithfulness；Tool：tool_call_success_rate（工具调用成功率）、参数错误率。每个能力有独立看板，哪个指标掉就优化哪个能力，而不是等整体效果差才模糊排查。

**Q：Agent 的四大能力怎么沉淀成可复用的工程框架？**

封装成 Agent SDK：`Agent(workflow, memory, rag, tools)` 四个组件可插拔。Memory 提供 Redis（短期）+ 向量库（长期）双实现，业务侧配置切换；RAG 封装统一的检索接口（支持向量/关键词/混合）；Tools 提供 Schema 定义 + 权限校验 + 超时重试的标准封装。沉淀"各行业的 Workflow 模板""Memory 的存储 schema 标准""常用 Tool 的实现库"，新 Agent 开发时组合配置，不从头造。
