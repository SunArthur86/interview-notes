# 【月之暗面面经】如果产品要支持命令式输入和自然语言输入并存，前端会怎么做？

## 一、两种输入模式的定位

| 维度 | 命令式输入 | 自然语言输入 |
|------|-----------|------------|
| 典型用户 | 高频老用户、技术用户 | 新用户、非技术用户 |
| 典型场景 | "快速做一件事" | "探索性请求" |
| 效率 | 极高（精确、无歧义） | 中等（需要AI理解意图） |
| 示例 | `/gen-site --template=portfolio --src=./assets` | "用我assets文件夹里的内容做一个个人作品集网站" |
| 参数补全 | 有参数提示和自动补全 | AI自动推断参数 |
| 确定性 | 高（固定语法） | 中（AI可能误解） |

## 三、输入路由器实现

```typescript
class InputRouter {
  parse(rawInput: string): ParsedInput {
    const trimmed = rawInput.trim();
    
    // 检测是否为命令式输入（以 / 开头）
    if (trimmed.startsWith('/')) {
      return this.parseCommand(trimmed);
    }
    
    // 检测是否为快捷命令（:开头，如 :site, :ppt）
    if (trimmed.startsWith(':')) {
      return this.parseQuickCommand(trimmed);
    }
    
    // 否则走自然语言解析
    return this.parseNaturalLanguage(trimmed);
  }
  
  // 命令解析
  private parseCommand(input: string): CommandInput {
    const tokens = this.tokenize(input);
    const cmd = tokens[0].slice(1);  // 去掉 /
    const args = this.parseArgs(tokens.slice(1));
    
    return {
      type: 'command',
      command: cmd,         // 'gen-site'
      args: args,           // { template: 'portfolio', src: './assets' }
      raw: input,
    };
  }
  
  // 自然语言解析
  private async parseNaturalLanguage(input: string): Promise<NLInput> {
    // 使用LLM将自然语言转为结构化任务
    const intent = await llm.parseIntent(input);
    
    return {
      type: 'natural',
      intent: intent.action,      // 'gen-site'
      params: intent.params,      // { template: 'portfolio' }
      confidence: intent.score,   // 0.85
      raw: input,
    };
  }
}

// 统一转为 Task 对象
function toTask(parsed: ParsedInput): Task {
  return {
    id: generateId(),
    action: parsed.command || parsed.intent,
    params: parsed.args || parsed.params,
    inputMode: parsed.type,   // 记录来源模式
    raw: parsed.raw,
    status: 'queued',
  };
}
```

## 五、就地参数补全

```typescript
// 当命令缺少必要参数时，就地补全而非报错
async function validateAndFill(command: CommandInput): Promise<Task | null> {
  const spec = commandRegistry.get(command.command);
  const missingParams = spec.required.filter(p => !(p in command.args));
  
  if (missingParams.length === 0) {
    return toTask(command);  // 参数完整，直接执行
  }
  
  // 参数缺失——就地补全
  for (const param of missingParams) {
    if (param === 'src') {
      // 文件类参数：弹出文件选择器
      const path = await showFilePicker(param.description);
      if (path) command.args[param.name] = path;
    } else if (param === 'template') {
      // 枚举类参数：弹出选择器
      const value = await showOptionPicker(param.options);
      if (value) command.args[param.name] = value;
    } else {
      // 文本类参数：在输入框就地提示
      promptInInput(`请输入 ${param.name}: ${param.description}`);
      return null;  // 等待用户补全
    }
  }
  
  return toTask(command);
}
```

## 七、常见坑

- **两套输入走两套系统**：命令式和自然语言各自独立处理，导致功能不一致
- **命令参数报错不引导**：缺少参数直接报错退出，不提供补全路径
- **历史记录只存一种格式**：老用户看不到命令格式，新用户看不懂命令
- **没有命令发现机制**：用户不知道有哪些命令可用，没有 `/help` 或补全

## 记忆要点

- 双轨定位：命令式主打高确定性与极速，自然语言主打低门槛与意图探索
- 统一入口路由：输入框接收指令，由路由器根据前缀分流至不同解析引擎
- 殊途同归：无论是正则提取还是AI推断，最终都需归一化为标准的Task对象

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么要同时支持命令式和自然语言两种输入，只留自然语言不行吗？现在的 LLM 意图理解已经很准了。**

因为两类用户的效率需求和确定性需求完全不同。自然语言的优势是低门槛（新用户不用学命令）和灵活（能表达复杂意图如"用 assets 文件夹做一个偏简约风的 portfolio 网站"），但劣势是"慢"（要调 LLM 解析意图，增加 300-800ms 延迟）和"不确定"（LLM 可能误解，confidence 0.6 时用户不知道参数对不对）。命令式的优势是"极速"（本地正则解析，<10ms）和"精确无歧义"（`/gen-site --template=portfolio --src=./assets` 不会有误解），劣势是"学习成本高"。对于高频老用户（一天发起 20+ 个任务），每次多等 500ms 的 LLM 解析、还要核对参数是否被正确推断，体验远不如直接敲命令。所以两种模式服务不同场景：命令式给"知道自己要什么"的高频用户，自然语言给"探索可能性"的新用户。只留自然语言会让高频用户流失（效率太低），只留命令式会让新用户流失（门槛太高）。

### 第二层：证据与定位

**Q：你怎么定位"自然语言输入被误解"是路由器分流逻辑问题，还是 LLM 意图解析能力问题？**

用解析置信度 + 用户修正率双指标定位。在自然语言解析后记录 confidence 分数（LLM 返回的 intent score），任务执行后追踪用户是否做了修正（如手动改了参数、取消了任务重发）。如果 confidence 高（>0.85）但用户仍频繁修正，说明 LLM 过度自信——解析逻辑没问题但模型能力不足，要换更强的模型或加 few-shot 示例；如果 confidence 低（<0.6）且路由器仍直接执行了任务（没走二次确认），说明路由器的置信度阈值设置有问题——应该低于阈值时弹出"我理解你要做 X，参数是 Y，对吗？"的确认框；如果命令式输入也被误解（如 `/gen-site` 被路由到了 NLU 解析器），说明路由器的分流逻辑有 bug（前缀检测失效）。三种情况的解法完全不同：模型能力问题要调 prompt 或换模型，阈值问题要调路由策略，分流 bug 要修路由器代码。

### 第三层：根因深挖

**Q：为什么命令式和自然语言要统一映射到同一个 Task 对象，而不是各自走独立的执行路径？**

因为"两套执行路径"必然导致功能分叉和维护灾难。如果命令式走 CommandExecutor、自然语言走 NLExecutor，会出现：(1) 功能不一致——命令式支持的参数自然语言不支持（或反之），用户反馈"为什么命令能做自然语言不能做"；(2) 修复重复——修一个 bug 要改两处，而且两处的行为可能微妙不同；(3) 数据割裂——历史记录里命令式任务和自然语言任务无法统一搜索和回放。统一到 Task 对象的根因是"输入方式是表达层差异，任务执行是业务层共性"——无论用户用 `/gen-site` 还是"帮我做一个网站"，底层要做的事是一样的（读文件、选模板、生成、导出），差异只在"如何解析出 action 和 params"。所以输入路由器负责把两种输入归一化为 Task（action + params + inputMode），下游的执行引擎只认 Task 对象，完全不关心输入来自哪条路径。这样新增第三种输入方式（如语音、拖拽手势）也只需加一个 parser，执行层零改动。

**Q：那如果命令式需要支持自然语言没有的高级参数（如 --dry-run 预览模式），为什么不给命令式单独开一条执行路径？**

因为"高级参数"不是"独立执行路径"的理由，而是"Task 对象字段扩展"的问题。--dry-run 本质是 Task 的一个 flag（dryRun: true），执行引擎读到这个 flag 时走预览分支而非正式执行分支。这个 flag 对自然语言输入同样有价值——用户说"帮我做一个网站看看效果"，NLU 解析器可以推断出 dryRun=true。所以高级参数应该作为 Task 对象的可选字段，由两种输入方式共享，而不是给命令式开独立路径。真正需要"命令式专属"的是"开发调试类"操作（如 --verbose 输出详细日志、--no-cache 跳过缓存），这些对自然语言用户无意义，可以在 Task 里标记 mode: 'debug'，执行引擎在 debug 模式下输出额外信息。这样即使是最复杂的参数差异，也能通过 Task 对象的字段扩展解决，不需要割裂执行路径。根因是"执行逻辑的差异要建模为数据（Task 字段），而非控制流（独立路径）"。

### 第四层：方案权衡

**Q：命令参数缺失时你选"就地补全"（弹文件选择器/选项器）还是"报错让用户重新输入"，怎么选？**

选就地补全，根因是"用户输入成本"和"任务发起成功率"的权衡。报错让用户重输的成本：用户看到"缺少 --src 参数"→ 要重新打完整命令 `/gen-site --template=portfolio --src=` → 容易打错 → 再次报错 → 挫败感累积 → 放弃使用。就地补全的成本：检测到 --src 缺失 → 自动弹出文件选择器 → 用户选文件 → 自动填入 → 继续执行，整个过程用户只需点两下。就地补全的任务发起成功率接近 100%，报错重输的成功率随错误次数指数下降。选就地补全的实现要点：(1) 按参数类型选补全 UI（文件类→文件选择器，枚举类→下拉选项，文本类→输入框聚焦提示）；(2) 补全后实时校验（文件是否存在、格式是否支持）；(3) 提供"转自然语言"的降级路径（命令式补全太麻烦时，提示用户"也可以直接说'用 assets 做一个 portfolio 网站'"）。

**Q：那如果就地补全的弹窗太多（一个命令 5 个参数弹 5 次窗），为什么不直接报错让用户一次性补全？**

因为"弹 5 次窗"是补全 UI 设计的问题，不是"就地补全"这个策略的问题。弹 5 次窗的解法是"批量补全面板"——检测到多个参数缺失时，一次性弹出一个结构化表单（类似 IDE 的重构对话框），把 5 个缺失参数列为 5 个输入项，用户一次性填完点确认。这样既保留了"就地补全"的低门槛（用户不需要重输命令），又避免了"逐个弹窗"的打断感。另一个优化是"智能默认值"——对于可推断的参数（如 --src 缺失但用户最近选择了 ./assets 目录，默认填入），只对真正无法推断的参数才要求用户补全。所以根因是"补全的 UX 设计要优化"，而非"放弃补全改报错"。报错让用户重输的体验永远劣于引导式补全，因为前者把认知负担全推给用户，后者由系统承担引导责任。

### 第五层：验证与沉淀

**Q：你怎么证明"双轨输入"比"纯自然语言"更有效，而不是只是增加了系统的复杂度？**

用分群指标验证。把用户按使用频率分群（高频：日均 10+ 任务 / 中频：日均 3-10 / 低频：日均 <3），对比两组（纯自然语言 vs 双轨）的关键指标：(1) 高频用户的任务发起耗时——双轨组的高频用户应大量使用命令式，单任务发起耗时应从 5 秒（打自然语言 + 等 LLM 解析）降到 1 秒（敲命令 + 本地解析）；(2) 中低频用户的留存率——双轨组不应低于纯自然语言组（说明命令式的存在没有吓跑新用户）；(3) 命令使用率分布——如果高频用户中命令式占比超过 50%，而中低频用户中自然语言占比超过 80%，说明双轨设计精准服务了两类人群。如果高频用户的命令使用率低于 20%，说明命令系统不够好用（补全差、命令难记），需要优化而非放弃双轨。

**Q：怎么让团队在新增 Agent 能力时，自觉同时注册命令式和自然语言两种入口，而不是只做一个？**

把"双入口注册"做成 Agent SDK 的强制约束。第一，定义 AgentManifest 接口，每个 Agent 必须声明 commandSpec（命令名、参数、补全规则）和 intentSpec（意图关键词、NLU 示例），缺一个不允许注册；第二，Agent 注册时自动生成命令补全提示和 NLU few-shot 示例——开发者只需写一份参数定义，SDK 自动派生出命令式的补全 UI 和自然语言的训练样本；第三，测试覆盖率要求——每个 Agent 必须有"命令式发起"和"自然语言发起"两组测试用例，都通过才能合并；第四，在 CI 里扫描 Agent 注册表，如果一个 Agent 只有 commandSpec 没有 intentSpec（或反之），CI 失败。这样双入口就从"产品规范"变成了"注册接口的必填字段"，不遵循就无法接入 Agent 体系。

## 结构化回答

**30 秒电梯演讲：** 就像使用Photoshop——专业用户用快捷键（命令式）秒操作，新手用菜单和搜索（自然语言）慢慢找。两种方式最终都触发同一个功能，但入口不同。

**展开框架：**
1. **命令式输入适合高频老用户** — 自然语言适合探索和复杂请求
2. **两类输入都映射到** — 同一任务模型
3. **命令模式的参数缺** — 失要就地补全

**收尾：** 如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
