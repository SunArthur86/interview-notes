# 【AI Infra推理优化】稀疏注意力方案怎么选？StreamingLLM vs H2O vs PyramidKV

> 来源：小红书「ai infra面试：kv cache夺命追问破局指南下」

## 一、为什么需要稀疏注意力

```
传统Full Attention KV Cache增长:

Token:    1K    4K    16K    64K    128K
KV Cache: 2GB → 8GB → 32GB → 128GB → 256GB
                                    ↑ GPU显存爆炸!

稀疏注意力策略:
                    ┌──────────────────────────┐
  Full Attention:  │██████████████████████████│ 全保留
                    └──────────────────────────┘
                    ┌──┐              ┌────────┐
  StreamingLLM:    │SI│..............│ Window │ │ Sink+窗口
                    └──┘              └────────┘
                    ┌──┐  ┌─┐ ┌──┐  ┌─┐ ┌────┐
  H2O:             │★★│  │★│ │★★│  │★│ │★★★★│ │ Top-K重要
                    └──┘  └─┘ └──┘  └─┘ └────┘
                    ┌──────────────┐
  PyramidKV:       │██████████████│ 浅层：大缓存（全局）
                    └──────────┐
                  (中层)       │██████████│ 中等
                                       └────┐
                  (深层)                │████│ 深层：小缓存（局部）
```

## 二、三种方案详解

### StreamingLLM — 恒定显存的无限长对话

```
原理：Attention Sink + 滑动窗口

KV Cache保留策略:
┌───┬─────────────────────────────────┬───────────┐
│Sink│         (丢弃)                  │  Sliding  │
│1-4 │                                │  Window   │
│tok │                                │  最近2048  │
└───┘                                └───────────┘

为什么开头token重要？Attention Sink现象：
→ 开头token在训练时几乎出现在每个序列中
→ 模型学会把"无处安放"的注意力分配给开头token
→ 如果丢弃开头token，模型生成质量急剧下降

显存：O(window_size) 恒定，与对话长度无关
适合：无限长对话、流式输出（ChatGPT式应用）
```

### H2O (Heavy-Hitter Oracle) — 动态保留高注意力token

```
原理：统计每个token的累计注意力分数，保留Top-K

步骤:
1. 每生成一个token，记录它对所有历史token的注意力分数
2. 累加每个历史token被关注的总分数
3. 保留分数最高的K个token（Heavy Hitters）
4. 丢弃低分token的KV Cache

┌─────────────────────────────────────┐
│ Token:    A   B   C   D   E   F     │
│ 累计分数: 8.2 0.3 5.1 0.1 7.8 2.0  │
│                          ↑          │
│ 保留:     ✓       ✓       ✓   ✓    │ ← Top-4
│ 丢弃:          ✗       ✗           │
└─────────────────────────────────────┘

适合：摘要、RAG检索增强（信息密集型文本）
```

### PyramidKV — 金字塔式分层缓存

```
原理：不同层保留不同比例的KV Cache

Layer 0  (浅层): ████████████████████ 保留 80%
Layer 10 (中层): ███████████████      保留 60%
Layer 20 (中层): ██████████           保留 40%
Layer 30 (深层): ██████               保留 20%
Layer 40 (深层): ███                  保留 10%

洞察：
→ 浅层注意力分散（捕获全局语法结构）→ 需要大缓存
→ 深层注意力集中（聚焦关键语义）→ 小缓存足够
→ 金字塔形状匹配了注意力从分散到集中的演化

适合：多任务混合负载（通用大模型场景）
```

## 三、方案对比

| 方案 | 显存模式 | 优势 | 劣势 | 最佳场景 |
|------|---------|------|------|---------|
| StreamingLLM | O(1)恒定 | 无限长对话 | 丢失中间信息 | 流式对话、客服Bot |
| H2O | O(k)可控 | 保留关键信息 | 需要注意力统计开销 | 摘要、RAG |
| PyramidKV | O(n×ratio) | 层级自适应 | 实现复杂 | 多任务通用 |
| Full（不压缩） | O(n)线性 | 无信息损失 | 显存爆炸 | 短上下文 |

## 五、面试加分点

1. **Attention Sink原理**：能解释为什么开头token获得异常高注意力（训练偏差——开头token出现在所有序列中，模型把它当"垃圾注意力"的垃圾桶），这个洞察会让面试官印象深刻
2. **组合方案**：StreamingLLM和H2O可以组合——用attention sink保底+H2O选择中间重要token+滑动窗口覆盖最近上下文
3. **评估方法**：不能只看PPL，要在下游任务（长文档QA、代码生成、多轮对话）上评估，因为PPL可能掩盖特定位置的生成退化
4. **工程实现**：vLLM 0.5+开始支持部分稀疏注意力方案，提及与PagedAttention的兼容性
5. **与KV Cache量化的叠加**：稀疏注意力（减少KV block数量）+量化（减少每个block的字节数）可以叠加使用，双重压缩

## 结构化回答

**30 秒电梯演讲：** 稀疏注意力是在长上下文中只保留重要的KV Cache，丢弃不重要的，以恒定显存支持无限长对话——你在听一场3小时的会议。。

**展开框架：**
1. **没有万能方案** — 没有万能方案，必须结合token重要性分布实测选择
2. **StreamingLLM** — attention sink（开头）+滑动窗口，恒定显存，适合无限长对话
3. **H2O** — 动态保留高注意力分数token，适合信息密集文本

**收尾：** 您想深入聊：Attention Sink现象的原因是什么？为什么开头token总获得高注意力？

## 苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 稀疏注意力方案要同时优化什么？ | 长上下文推理性能（KV Cache显存和attention计算）和质量（不丢失关键信息）——不同方案在两者间取舍不同 |
| 证据追问 | StreamingLLM、H2O、PyramidKV各自的'稀疏'是怎么实现的？ | StreamingLLM保留attention sink前几个token+滑动窗口；H2O动态丢弃低注意力分数的KV；PyramidKV按层金字塔结构分配KV预算 |
| 边界追问 | 什么场景用StreamingLLM？什么场景用H2O？ | StreamingLLM适合超长固定窗口（流式对话）；H2O适合问答等需要全局信息的场景；PyramidKV适合层级特征明显的场景 |
| 反例追问 | 稀疏注意力会丢信息，什么任务一定不能用？ | 需要精确长程依赖的任务（代码补全、长文档精确检索、数学证明）会因丢弃关键KV出错，这类任务慎用稀疏 |
| 风险追问 | 稀疏注意力的质量风险有哪些？ | 丢失关键历史信息导致幻觉、不同输入稀疏模式不稳定、长尾case质量塌陷、与RAG/Function Calling交互可能出错 |
| 验证追问 | 怎么评估稀疏注意力的质量损失？ | 长文档benchmark（LongBench/RULER）、needle-in-haystack测试、业务badcase分析、对比全注意力基线 |
| 沉淀追问 | 稀疏注意力选型怎么沉淀？ | 规范：按场景选型矩阵、必跑长文档benchmark、badcase监控、保留全注意力兜底开关 |

### 现场对话示例
**面试官**：稀疏注意力方案StreamingLLM、H2O、PyramidKV怎么选？
**候选人**：StreamingLLM用attention sink+滑窗适合流式长对话；H2O动态丢弃低分KV适合问答；PyramidKV按层金字塔分配预算适合层级特征明显的任务。
**面试官**：稀疏注意力有什么风险？
**候选人**：丢失关键历史信息导致幻觉、长尾case质量塌陷、与RAG/Function Calling交互可能出错，需要精确长程依赖的任务不能用。
**面试官**：怎么评估质量损失？
**候选人**：用LongBench/RULER长文档benchmark、needle-in-haystack测试、业务badcase分析，对比全注意力基线判断可接受度。
