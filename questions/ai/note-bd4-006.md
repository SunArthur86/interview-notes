# GraphRAG 和普通向量 RAG 的区别？适合本地生活知识库吗？

## 核心区别

```
┌──────────────── 向量 RAG ────────────────┐
│                                          │
│  Query: "老房子川菜附近有什么景点？"       │
│          │                               │
│          ▼ Embed                         │
│  [向量] ←→ 向量库中最相似的chunks         │
│  找到: "老房子川菜位于朝阳区..."           │
│  找到: "朝阳公园是北京著名景点..."         │
│                                          │
│  问题: 可能找到不相关的"老房子"或"川菜"    │
│  缺陷: 无法理解"附近"的空间关系           │
└──────────────────────────────────────────┘

┌──────────────── GraphRAG ────────────────┐
│                                          │
│  知识图谱:                               │
│  [老房子川菜]──位于──→[朝阳区三里屯]      │
│       │                  │               │
│     属于               附近               │
│       │                  │               │
│       ▼                  ▼               │
│  [川菜餐厅]         [三里屯太古里]        │
│                       │                  │
│                     附近                  │
│                       │                  │
│                       ▼                  │
│                  [工体北路]               │
│                  [团结湖公园]             │
│                                          │
│  查询: 从"老房子川菜"出发, 2跳找到景点     │
│  精确: 空间关系清晰, 无幻觉               │
└──────────────────────────────────────────┘
```

## 详细对比

| 维度 | 向量 RAG | GraphRAG |
|------|---------|----------|
| **索引结构** | 向量(Embedding) | 图(实体+关系+向量) |
| **检索方式** | 相似度匹配(Top-K) | 图遍历 + 向量召回 |
| **多跳推理** | ❌ 不支持 | ✅ 支持(2-3跳) |
| **关系查询** | ❌ 弱 | ✅ 强 |
| **全局摘要** | ❌ 差(只看局部) | ✅ 好(社区检测+摘要) |
| **建图成本** | 低(切块+Embedding) | 高(NER+关系抽取+建图) |
| **查询延迟** | 低(10-50ms) | 中(50-200ms) |
| **维护成本** | 低(增量索引) | 高(图谱更新) |
| **幻觉率** | 中 | 低(有实体约束) |

## 本地生活知识库是否适合 GraphRAG

### ✅ 适合的理由

```
本地生活数据天然是图结构:

餐厅 ──位于──→ 商圈
  │              │
  ├──属于──→ 菜系
  │              │
  ├──提供──→ 团购
  │
  └──靠近──→ 地铁站/景点/停车场

用户问: "三里屯附近人均100的川菜，要有停车场"
→ GraphRAG: 商圈=三里屯 → 菜系=川菜 → 价位≤100 → 有停车场
→ 精确过滤，比向量检索准确

用户问: "老房子川菜和巴蜀风哪个离地铁近？"
→ 向量RAG: 很可能答非所问(找不到空间关系)
→ GraphRAG: 餐厅→地铁站(距离关系) → 直接比较
```

### ❌ 不适合的部分

```
1. 用户评论/体验类内容 → 适合向量RAG
   "老房子川菜好吃吗？" → 向量检索评论chunks

2. 实时数据(排队、库存) → 不需要图结构
   → 直接查Redis/DB

3. 建图成本: 本地生活数百万商家
   → 实体抽取+关系标注成本高
   → 建议: 核心数据(商圈/商家关系)用图，评论用向量
```

## 生产级混合方案

```python
class HybridRAG:
    """向量RAG + GraphRAG 混合检索"""

    def search(self, query):
        # Step 1: 意图判断 — 需要关系推理吗？
        if self.needs_reasoning(query):
            # Step 2a: GraphRAG — 关系查询
            entities = self.ner.extract(query)
            graph_results = self.graph.traverse(
                entities, max_hops=2
            )
        else:
            # Step 2b: 向量RAG — 语义检索
            graph_results = []

        # Step 3: 向量RAG补充语义内容
        vector_results = self.vector_db.search(
            embed(query), top_k=5
        )

        # Step 4: 融合排序
        fused = self.rerank(
            graph_results + vector_results,
            strategy="reciprocal_rank_fusion"
        )
        return fused[:5]
```

## GraphRAG 建图流程

```
Step 1: 实体抽取 (NER)
  文档 → "老房子川菜" (Restaurant), "朝阳区" (Location)

Step 2: 关系抽取 (RE)
  (老房子川菜) --位于--> (朝阳区)
  (老房子川菜) --属于--> (川菜)

Step 3: 实体消歧
  "老房子" → 统一为 "老房子川菜馆(三里屯店)"

Step 4: 图谱存储 (Neo4j / NebulaGraph)
  节点: 实体 + 属性
  边: 关系 + 权重

Step 5: 社区检测 + 摘要 (Leiden算法)
  将图分成社区 → 每个社区生成摘要 → 支持全局问答
```

## 记忆要点

- 向量RAG靠相似度匹配找局部Chunks，GraphRAG靠实体关系图遍历支持多跳推理
- 因为GraphRAG建图成本高，所以纯文本评论仍用向量RAG，关系查询用图
- 本地生活天然是图结构：餐厅到商圈/地铁站的关系极其适合GraphRAG精确过滤
- 生产级最佳实践：混合检索(Hybrid)，先用意图判断决定走Graph还是走向量

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：GraphRAG 在向量检索基础上叠加知识图谱。为什么需要图？向量检索（找相似 chunk）不够吗？**

向量检索解决不了"多跳关系查询"。场景："中关村附近的川菜馆"——向量检索用 query embedding 找相似 chunk，可能召回"中关村餐厅列表"的 chunk，但这个 chunk 不一定按"菜系"分类（可能混了川菜、粤菜、日料），召回精度差。GraphRAG 把"餐厅 → 菜系"、"餐厅 → 商圈"、"商圈 → 地理位置"建成图，查询时先按"商圈=中关村"过滤节点，再遍历"菜系=川菜"的边，精确找到"中关村 + 川菜"的餐厅。向量检索是"语义相似度匹配"（近似，可能召回不相关的），图遍历是"精确关系查询"（准确，基于结构化关系）。对于"有明确关系结构"的查询（如多跳、条件过滤），图远优于向量。对于"语义模糊查询"（如"适合约会的餐厅"，无明确关系，靠语义），向量优于图。两者互补，混合检索（Hybrid）按查询类型路由。

### 第二层：证据与定位

**Q：GraphRAG 的某些查询比向量 RAG 还慢（建图遍历耗时长）。怎么定位是图太大（节点多）、遍历策略差、还是查询本身不适合图？**

看查询的图遍历路径。一是图规模——节点和边的数量，如果图有千万节点、亿条边，遍历成本高（即使有索引，多跳遍历的复杂度随跳数增长）；二是遍历策略——是否用了图索引（如 Neo4j 的节点标签索引、边的类型索引），如果全图扫描（无索引），慢；三是查询适配性——查询是否真的需要图遍历（如"中关村川菜馆"需要图的关系过滤，但"川菜的特点"只需向量检索文本），如果"不需要图的查询也走图"，是无谓开销。定位方法：看查询的执行计划（如 Neo4j 的 PROFILE），分析遍历了多少节点/边、用了哪些索引、耗时在哪。如果遍历节点多（如扫了 10 万节点），是图太大或索引差；如果遍历少但仍慢（如锁等待），是系统问题。关键判断"该查询是否适合图"——关系查询走图，语义查询走向量，混合检索按意图路由。

### 第三层：根因深挖

**Q：GraphRAG 你说"建图成本高"。具体高在哪？为什么不直接自动从文档抽取实体关系建图？**

建图成本在"实体抽取 + 关系抽取 + 图维护"。一是实体抽取——从非结构化文档（如餐厅评论）抽取实体（餐厅名、菜系、商圈）需要 NER 或 LLM，精度不完美（漏抽或错抽，如把"海底捞"错抽成"海底"）；二是关系抽取——抽取实体间关系（餐厅 -[属于]-> 菜系），需要关系分类模型或 LLM，精度更低（关系类型多，标注数据少）；三是图维护——知识更新（新餐厅开业）要增量更新图，删除/修改节点和边，保持一致性，复杂。自动抽取（用 LLM）可行但精度差——LLM 抽取实体关系的 F1 约 70-80%（错误率高，图质量差影响查询精度），人工校验成本高。所以 GraphRAG 适合"关系结构明确 + 可结构化建图"的场景（如本地生活的餐厅-商圈-菜系关系，可从结构化数据库导入图，无需从非结构化文本抽取），而非"纯非结构化文本"（抽取成本高、精度差，不如向量检索）。

**Q：那为什么不直接用 SQL（结构化数据库）做关系查询（如 WHERE 商圈='中关村' AND 菜系='川菜'），省得建图？**

SQL 适合"单跳或简单多表 join"，图适合"多跳遍历"。场景一（简单过滤）："中关村川菜馆"——SQL 用 `SELECT * FROM restaurants WHERE area='中关村' AND cuisine='川菜'`，精确高效，不需要图。场景二（多跳）："和中关村川菜馆同商圈的日料店"——SQL 要 join（餐厅表 join 自己，条件是同商圈且不同菜系），多跳 join 的 SQL 复杂且慢（数据库优化器对多跳 join 不友好）。图遍历是 `MATCH (r:Restaurant {cuisine:'川菜', area:'中关村'})-[:SAME_AREA]->(r2:Restaurant {cuisine:'日料'})`，直观且图数据库（Neo4j）对多跳遍历优化好（邻接表存储，遍历边 O(1)）。场景三（路径查询）："从餐厅 A 到餐厅 B 经过哪些商圈"——SQL 几乎无法表达（递归 CTE 复杂），图遍历天然支持（最短路径算法）。所以简单关系用 SQL，多跳/路径用图。本地生活场景如果是简单过滤（商圈 + 菜系），SQL 够用；如果涉及复杂关系推理（如"推荐的关联餐厅"），图更优。

### 第四层：方案权衡

**Q：你说"本地生活天然是图结构"（餐厅-商圈-地铁的关系）。但实际生产你用混合检索（意图判断决定走图还是向量）。为什么不全部走图（关系明确时图最准）？**

全部走图不适合"语义模糊查询"。虽然餐厅-商圈-菜系是结构化关系（图查询准），但用户的查询不全是"结构化"的——如"适合情侣约会的餐厅"、"有氛围感的餐厅"、"性价比高的川菜"。这些查询是"语义模糊"的（"氛围感"、"性价比"无法用图的关系精确表达），需要向量检索（用 embedding 匹配语义相似的餐厅描述/评论）。如果这类查询也走图，要么无法表达（图没有"氛围感"的边），要么强行加"氛围感"标签（人工标注成本高）。所以混合检索——意图判断识别查询类型：结构化查询（"中关村川菜"）走图（精确），语义查询（"氛围感餐厅"）走向量（模糊匹配），综合两者结果。全走图会损失语义查询能力，全走向量会损失关系精确性，混合是平衡。

**Q：为什么不直接把图和向量融合到一个系统（如向量库支持图关系，或图库支持向量检索），省得维护两套？**

融合系统是趋势但不成熟。当前有些系统尝试融合（如 Neo4j 的向量索引、Weaviate 的图网络），但各自偏向一侧——Neo4j 的向量检索性能不如专业向量库（Milvus），Weaviate 的图遍历能力不如专业图库（Neo4j）。专业系统在各自领域优化深（Milvus 的 ANN 极快、Neo4j 的图遍历极快），融合系统两头不精。生产实践：分开维护图库（Neo4j）和向量库（Milvus），应用层做混合检索（按意图路由），结果融合（RRF 或简单的结果合并）。两套系统的运维成本高（两个数据库），但性能最优。如果规模小（<百万节点/向量），用融合系统（如 Weaviate）省运维；规模大用专业系统分开。未来融合系统成熟后可能替代，当前专业分开 + 应用层融合是主流。

### 第五层：验证与沉淀

**Q：你怎么衡量 GraphRAG（或混合检索）相比纯向量 RAG 的提升？**

定义指标：一是查询准确率（在 golden set 上，混合检索应 >纯向量，特别是关系查询场景）；二是召回精度（关系查询的精确匹配率，GraphRAG 应 >向量）；三是延迟（GraphRAG 的图遍历延迟 vs 向量的 ANN 延迟，应可接受）；四是意图路由准确率（意图判断决定走图还是向量，路由错会损害效果）。做对比实验：纯向量 vs 纯图 vs 混合（意图路由），在"关系查询"和"语义查询"两类 golden set 上分别对比。预期：关系查询上图/混合显著优于纯向量，语义查询上向量/混合优于纯图。关键验证"意图路由的价值"——路由准确率应 >90%（路由错的效果差）。验证"混合的延迟"——图遍历 + 向量检索的总延迟应 <500ms（可接受）。A/B 测试线上效果——纯向量 vs 混合，看查询准确率和用户满意度。

**Q：GraphRAG 方案怎么沉淀成 RAG 系统标配？**

固化成"混合检索框架"：意图分类器（结构化查询走图、语义查询走向量、混合查询两者都走 + 融合）、图库（Neo4j，存结构化关系）、向量库（Milvus，存非结构化 embedding）、结果融合（RRF 或加权）。沉淀"建图规范"（从结构化数据导入图，避免从非结构化文本抽取）、"意图分类的 prompt 模板"、"图查询和向量查询的融合策略"。配套监控（查询准确率、路由准确率、延迟、图和向量的使用比例），准确率降告警。把"混合检索"作为 RAG 系统的默认架构（有结构化关系的场景），新业务接入时按数据特点建图 + 向量。积累"各场景的建图经验"（如本地生活的餐厅-商圈-菜系图、电商的商品-品类-属性图），复用建图模式。
