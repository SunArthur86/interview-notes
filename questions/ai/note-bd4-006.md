# GraphRAG 和普通向量 RAG 的区别？适合本地生活知识库吗？

## 核心区别

```
┌──────────────── 向量 RAG ────────────────┐
│                                          │
│  Query: "老房子川菜附近有什么景点？"       │
│          │                               │
│          ▼ Embed                         │
│  [向量] ←→ 向量库中最相似的chunks         │
│  找到: "老房子川菜位于朝阳区..."           │
│  找到: "朝阳公园是北京著名景点..."         │
│                                          │
│  问题: 可能找到不相关的"老房子"或"川菜"    │
│  缺陷: 无法理解"附近"的空间关系           │
└──────────────────────────────────────────┘

┌──────────────── GraphRAG ────────────────┐
│                                          │
│  知识图谱:                               │
│  [老房子川菜]──位于──→[朝阳区三里屯]      │
│       │                  │               │
│     属于               附近               │
│       │                  │               │
│       ▼                  ▼               │
│  [川菜餐厅]         [三里屯太古里]        │
│                       │                  │
│                     附近                  │
│                       │                  │
│                       ▼                  │
│                  [工体北路]               │
│                  [团结湖公园]             │
│                                          │
│  查询: 从"老房子川菜"出发, 2跳找到景点     │
│  精确: 空间关系清晰, 无幻觉               │
└──────────────────────────────────────────┘
```

## 详细对比

| 维度 | 向量 RAG | GraphRAG |
|------|---------|----------|
| **索引结构** | 向量(Embedding) | 图(实体+关系+向量) |
| **检索方式** | 相似度匹配(Top-K) | 图遍历 + 向量召回 |
| **多跳推理** | ❌ 不支持 | ✅ 支持(2-3跳) |
| **关系查询** | ❌ 弱 | ✅ 强 |
| **全局摘要** | ❌ 差(只看局部) | ✅ 好(社区检测+摘要) |
| **建图成本** | 低(切块+Embedding) | 高(NER+关系抽取+建图) |
| **查询延迟** | 低(10-50ms) | 中(50-200ms) |
| **维护成本** | 低(增量索引) | 高(图谱更新) |
| **幻觉率** | 中 | 低(有实体约束) |

## 本地生活知识库是否适合 GraphRAG

### ✅ 适合的理由

```
本地生活数据天然是图结构:

餐厅 ──位于──→ 商圈
  │              │
  ├──属于──→ 菜系
  │              │
  ├──提供──→ 团购
  │
  └──靠近──→ 地铁站/景点/停车场

用户问: "三里屯附近人均100的川菜，要有停车场"
→ GraphRAG: 商圈=三里屯 → 菜系=川菜 → 价位≤100 → 有停车场
→ 精确过滤，比向量检索准确

用户问: "老房子川菜和巴蜀风哪个离地铁近？"
→ 向量RAG: 很可能答非所问(找不到空间关系)
→ GraphRAG: 餐厅→地铁站(距离关系) → 直接比较
```

### ❌ 不适合的部分

```
1. 用户评论/体验类内容 → 适合向量RAG
   "老房子川菜好吃吗？" → 向量检索评论chunks

2. 实时数据(排队、库存) → 不需要图结构
   → 直接查Redis/DB

3. 建图成本: 本地生活数百万商家
   → 实体抽取+关系标注成本高
   → 建议: 核心数据(商圈/商家关系)用图，评论用向量
```

## 生产级混合方案

```python
class HybridRAG:
    """向量RAG + GraphRAG 混合检索"""

    def search(self, query):
        # Step 1: 意图判断 — 需要关系推理吗？
        if self.needs_reasoning(query):
            # Step 2a: GraphRAG — 关系查询
            entities = self.ner.extract(query)
            graph_results = self.graph.traverse(
                entities, max_hops=2
            )
        else:
            # Step 2b: 向量RAG — 语义检索
            graph_results = []

        # Step 3: 向量RAG补充语义内容
        vector_results = self.vector_db.search(
            embed(query), top_k=5
        )

        # Step 4: 融合排序
        fused = self.rerank(
            graph_results + vector_results,
            strategy="reciprocal_rank_fusion"
        )
        return fused[:5]
```

## GraphRAG 建图流程

```
Step 1: 实体抽取 (NER)
  文档 → "老房子川菜" (Restaurant), "朝阳区" (Location)

Step 2: 关系抽取 (RE)
  (老房子川菜) --位于--> (朝阳区)
  (老房子川菜) --属于--> (川菜)

Step 3: 实体消歧
  "老房子" → 统一为 "老房子川菜馆(三里屯店)"

Step 4: 图谱存储 (Neo4j / NebulaGraph)
  节点: 实体 + 属性
  边: 关系 + 权重

Step 5: 社区检测 + 摘要 (Leiden算法)
  将图分成社区 → 每个社区生成摘要 → 支持全局问答
```

## 记忆要点

- 向量RAG靠相似度匹配找局部Chunks，GraphRAG靠实体关系图遍历支持多跳推理
- 因为GraphRAG建图成本高，所以纯文本评论仍用向量RAG，关系查询用图
- 本地生活天然是图结构：餐厅到商圈/地铁站的关系极其适合GraphRAG精确过滤
- 生产级最佳实践：混合检索(Hybrid)，先用意图判断决定走Graph还是走向量
