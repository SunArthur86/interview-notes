# 向量间相似度如何计算？

## 一、三种常用相似度

```
┌──────────────────────────────────────────────────┐
│              三种向量相似度度量                      │
├──────────────────────────────────────────────────┤
│                                                    │
│  1. 余弦相似度（Cosine Similarity）— 最常用        │
│     衡量"方向"的接近程度                           │
│     范围：[-1, 1]，越大越相似                      │
│                                                    │
│  2. 点积（Dot Product）                            │
│     方向 × 模长                                    │
│     受向量长度影响                                 │
│                                                    │
│  3. 欧氏距离（Euclidean Distance）                 │
│     绝对距离                                       │
│     范围：[0, +∞)，越小越相似                      │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、余弦相似度（最常用）

```python
import numpy as np

def cosine_similarity(vec_a, vec_b):
    """余弦相似度：衡量方向夹角"""
    dot = np.dot(vec_a, vec_b)              # 点积
    norm_a = np.linalg.norm(vec_a)          # 向量A的模长
    norm_b = np.linalg.norm(vec_b)          # 向量B的模长
    return dot / (norm_a * norm_b)

# 例：
# A = "我喜欢猫" → [0.3, 0.5, 0.1]
# B = "我爱小猫" → [0.4, 0.6, 0.2]
# cos(A,B) ≈ 0.98 （方向几乎一致，语义相似）

# C = "今天天气好" → [0.1, 0.0, 0.9]
# cos(A,C) ≈ 0.15 （方向差异大，语义不相似）

# 特点：
# - 只看方向，不看长度（模长归一化后算）
# - 适合语义相似度（语义相近=方向相近）
# - 不受文本长度影响
```

## 三、点积（Dot Product）

```python
def dot_product(vec_a, vec_b):
    """点积：方向 × 模长"""
    return np.dot(vec_a, vec_b)

# 与余弦的关系：
# dot(A,B) = |A| × |B| × cos(A,B)
# 
# 如果向量已归一化（|A|=|B|=1）：
# dot(A,B) = cos(A,B)  ← 两者相同

# 点积受模长影响：
# 长文本的Embedding模长可能更大
# 点积会偏好长文档（不公平）
# 所以RAG通常用余弦（先归一化）
```

## 四、欧氏距离

```python
def euclidean_distance(vec_a, vec_b):
    """欧氏距离：绝对距离"""
    return np.sqrt(np.sum((vec_a - vec_b) ** 2))

# 范围：[0, +∞)，越小越相似
# 特点：
# - 衡量绝对位置差异
# - 受向量模长和方向双重影响
# - 适合需要考虑"绝对差异"的场景
```

## 五、三种度量的对比

```
┌──────────────┬──────────────────┬──────────────────────┐
│ 度量          │ 关注点              │ RAG适用性              │
├──────────────┼──────────────────┼──────────────────────┤
│ 余弦相似度    │ 方向（角度）        │ ★★★ 最常用            │
│              │ 不受模长影响        │ 不受文本长度影响        │
├──────────────┼──────────────────┼──────────────────────┤
│ 点积          │ 方向 × 模长        │ ★★ 向量归一化后=余弦  │
│              │ 受模长影响          │ 计算更快(不用归一化)   │
├──────────────┼──────────────────┼──────────────────────┤
│ 欧氏距离      │ 绝对位置            │ ★ 较少用              │
│              │ 受模长和方向影响    │ 适合聚类/可视化        │
└──────────────┴──────────────────┴──────────────────────┘
```

## 六、为什么 RAG 通常用余弦相似度

```python
# RAG场景的特点：
# 1. 文档长度不一（一句话 vs 一段落）
#    → Embedding模长不同
#    → 点积会偏好长文档
#    → 余弦归一化后公平比较

# 2. 我们关心"语义相似"（方向）
#    而非"语义强度"（模长）
#    → 余弦最合适

# 实践：Embedding后先归一化，然后用点积（=余弦但更快）
def normalized_embedding(vec):
    """归一化：模长变为1"""
    return vec / np.linalg.norm(vec)

# 归一化后：dot(normalized_a, normalized_b) == cosine(a, b)
# 但点积计算比余弦少两次开方，更快
```

```python
# 实践中的阈值参考（余弦相似度）
thresholds = {

```

## 八、面试加分点

1. **余弦最常用的原因**：不受模长影响，公平比较不同长度文本——讲清"为什么"
2. **归一化技巧**：归一化后点积=余弦但更快——体现工程优化
3. **给阈值参考**：0.7 以上算相关，体现实战经验

## 记忆要点

- 余弦相似度：只看向量方向不看模长，最常用，不受文本长度影响。
- 点积：方向乘模长，受向量长度影响，若向量已归一化则等同于余弦。
- 欧氏距离：衡量绝对位置差，受模长和方向双重影响，RAG中较少用。
- RAG首选：因长文本模长大易作弊，通常用余弦或提前归一化后用点积加速。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：相似度计算有余弦/点积/欧氏三种，为什么不统一用一种（如都用余弦）？不同场景为什么要选不同的？**

因为三者度量的"相似性维度"不同。1）余弦相似度——只看方向（夹角），不看模长，适合"语义相似"（方向相同即语义相近，模长是向量长度不代表语义强度），文本 embedding 标准化后多用余弦；2）点积——方向×模长，既看方向又看模长，适合"模长有意义的场景"（如 embedding 模长反映信息量/重要性，点积综合方向和重要性），但 embedding 通常标准化后点积=余弦；3）欧氏距离——绝对距离（坐标差），看"绝对位置接近度"，适合"空间聚类/异常检测"（如离群点检测用欧氏），但高维下欧氏距离区分度下降（维度灾难）。所以场景：文本语义检索用余弦（方向代表语义），模长有意义的用点积，空间聚类/异常用欧氏。RAG 场景默认余弦（embedding 标准化，方向即语义）。

### 第二层：证据与定位

**Q：向量检索结果"相似度都很高"（如 top-10 的余弦都 >0.9）但很多不相关，怎么定位是相似度度量选错还是 embedding 模型问题？**

分两方面排查。1）embedding 模型——如果 embedding 把"不相关的"编码成"方向相近的"（如"苹果手机"和"苹果水果"余弦 0.95），是 embedding 模型区分度差（没区分语义），换/微调 embedding；2）度量——如果 embedding 本身合理（相关和不相关的方向有差异），但余弦相似度都高（区分不开），可能是度量问题（余弦对某些方向的差异不敏感），试其他度量（如欧氏，看是否能区分）；3）维度灾难——高维（如 1536 维）下所有向量对都"较近"（余弦集中在 0.8-0.95，区分度低），是高维的固有问题，降维（如 PCA 到 256 维）或调阈值（不只看相似度值，结合 ranking）。定位方法：抽样相关/不相关文档对，算三种相似度，看哪个度量能区分（相关 >不相关），用区分度好的度量。常见根因：embedding 区分度差（换模型），高维区分度低（降维/调阈值）。

### 第三层：根因深挖

**Q：余弦相似度只看方向，但 embedding 向量的模长是否完全没用？什么情况下模长有意义？**

模长在特定 embedding 训练方式下有意义。1）标准化 embedding——大多数 embedding 模型（如 BERT/句向量）训练时标准化（模长=1），此时余弦=点积，模长无意义，用余弦即可；2）非标准化 embedding——某些 embedding（如对比学习训练但未强标准化），模长可能反映"信息量/确信度"（如训练充分的文档模长大，模糊文档模长小），此时点积（方向×模长）可能优于余弦（综合确信度）；3）实践中——多数 RAG 用标准化 embedding（余弦够），但如果发现余弦效果一般，试试点积（看模长是否携带有用信号）；4）归一化控制——如果想用余弦但 embedding 非标准化，显式归一化（向量除以模长）再用余弦，等价于控制模长。原则：默认余弦（标准化场景），非标准化 embedding 试点积，模长有意义则点积更优。

**Q：欧氏距离在高维下"区分度下降"（维度灾难），为什么？这对选度量有什么影响？**

维度灾难的原理：高维下随机向量对 的欧氏距离趋于一致（都接近某均值），区分度下降。1）原理——高维空间体积巨大，随机向量分布稀疏，任意两点距离趋于相同（如 1000 维下，最近和最远邻距离差异小）；2）影响——欧氏距离在高维下"远近不分"（最近邻和第 100 近邻距离差不多），检索质量差；3）余弦的鲁棒性——余弦只看方向（夹角），不受模长/绝对位置影响，在高维下仍保持区分度（方向差异比绝对距离敏感）；4）实践——高维 embedding（如 768/1536 维）用余弦（鲁棒），低维（如 64 维聚类特征）可用欧氏（低维区分度好）。所以 RAG 高维 embedding 默认余弦（规避维度灾难），低维特征空间才用欧氏。如果要降维（如 PCA 到 128 维）也可，但可能损失信息，权衡。

### 第四层：方案权衡

**Q：向量库（如 Milvus）支持多种度量（COSINE/IP/L2），存储时选了一个度量后能换吗？怎么选？**

选度量要和 embedding 训练时的"度量假设"一致。1）一致性——embedding 模型训练时用某个度量（如对比学习常用余弦），检索时用相同度量才符合训练分布（如训练用余弦，检索用欧氏可能效果差）；2）不可随意换——向量库建索引时按某度量建（如 HNSW 按 L2 建的图，用余弦查询结果不对），换度量要重建索引（reindex，成本高）；3）选法——查 embedding 模型文档（如 bge 用余弦、OpenAI text-embedding 用点积/余弦），按模型推荐选；4）实务——默认余弦（主流 embedding 用），特殊模型按文档。如果用错度量（如余弦 embedding 用 L2 检索），召回会差（检索度量和训练不匹配），要 reindex 改度量。原则：检索度量=embedding 训练度量，建库前确认，避免 reindex。

**Q：混合相似度（如 0.7*余弦+0.3*BM25）要融合向量和关键词检索，但分数尺度不同怎么融合？**

归一化或用排名融合。1）归一化——余弦（0-1）和 BM25（无界）尺度不同，先归一化（如 min-max 把 BM25 归一到 0-1），再加权融合（0.7*余弦+0.3*BM25_norm），但归一化方式影响结果（min-max 对异常值敏感）；2）RRF（倒数排名融合）——不看分数只看排名（余弦 top-1 贡献 1/2，BM25 top-1 贡献 1/2，总和排序），尺度无关，简单鲁棒，业界默认；3）加权 RRF——给不同检索器不同权重（如向量权重高），加权 RRF 分数（如向量 rank 的贡献 *1.5），兼顾权重和鲁棒；4）学习融合——用小模型（如 LTR）学习"给定向量分+BM25 分，输出相关性"，最准但要训练数据。实务：默认 RRF（免调），想加权用加权 RRF，有充足数据上 LTR。原则：避免直接加权分数（尺度问题），用排名融合或学习融合。

### 第五层：验证与沉淀

**Q：你怎么衡量相似度度量选择是否正确（对检索效果的影响）？**

AB 对比。固定评估集（query-文档对），对比不同度量的 Recall@K/MRR：1）余弦 vs 点积 vs 欧氏——跑三种度量，看哪个 Recall/MRR 高（高的是更适合的度量）；2）匹配 embedding——确保度量和 embedding 训练一致（如 bge 余弦训练，用余弦应最好），如果训练度量的反而差，说明 embedding 或索引有问题；3）端到端——不同度量的 RAG 答案准确率对比（度量好答案好）。最优：Recall/MRR 高 + 端到端好 = 度量选对。还要看不同度量在不同 query 类型上的表现（如精确 query 欧氏可能好，语义 query 余弦好），按主要 query 类型选。

**Q：相似度计算的选型和调优怎么沉淀成检索平台的最佳实践？**

建最佳实践库：1）度量推荐——按 embedding 模型推荐度量（如 bge→余弦，OpenAI→点积），内置映射，开发者选模型自动推荐度量；2）融合策略——内置 RRF/加权 RRF/LTR 等融合算法，可配置（如选哪些检索器、权重）；3）诊断工具——分析检索结果（相似度分布/区分度），辅助定位度量/embedding 问题；4）评估集成——度量/融合策略变更后自动评估（Recall/MRR），退化告警。这套写入团队检索平台 SOP，让"选相似度度量"从"拍脑袋"变成"按模型推荐+评估验证"，标准化最佳实践。
