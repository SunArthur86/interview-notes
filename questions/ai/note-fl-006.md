# 【字节飞连面经】Agentic RAG 和传统 RAG 区别？Query Rewrite 算 Agentic 吗？

## 一、核心区别

| 维度 | 传统 RAG | Agentic RAG |
|------|---------|-------------|
| 检索次数 | 1 | N |
| 决策权 | 没有，固定流程 | LLM 决定要不要再查、查什么 |
| 工具数 | 1（一个检索器） | 多（多个数据源 + 计算工具） |
| 适用 | 简单事实查询 | 多跳推理、跨文档对比、深度研究 |
| 成本 | 低 | 3-10x |
| 稳定性 | 高 | 低（多了决策点就多了失败点） |

## 二、Query Rewrite 算 Agentic 吗？严格说不算

- **Query Rewrite**：固定一步预处理，无决策回路 → 仍是传统 RAG
- **算 Agentic 的标准**：至少要有"根据上一轮结果决定要不要再检索"这个决策点

```
传统 RAG（含 Query Rewrite）：
  query → rewrite → retrieve(1次) → generate

Agentic RAG：
  query → retrieve → 看结果 → 不够？→ rewrite → retrieve(再) → ... → generate
                       ↑_________决策回路________↓
```

## 三、多轮检索设计

| 范式 | 做法 | 适用 |
|------|------|------|
| **Self-Ask** | LLM 自问"我还需要知道什么"，生成子问题 → 检索 → 答 → 拼装 | 多跳推理 |
| **ReAct-RAG** | Thought → Search → Observation → Thought → ... 直到 Answer | 信息不全需边问边查 |
| **Plan-and-Execute** | 先出完整 plan（5 步），再执行 | 步骤明确的研究 |

飞连 IT 工单场景适合 **ReAct-RAG**（用户描述往往不完整，需要边问边查）。

## 四、检索失败时 Agent 怎么调整策略

```
检索失败 → 
  [1] 改写 query（同义词、去停用词、英文化）
  [2] 切数据源（wiki 没查到 → 工单历史 → 代码库）
  [3] 降难度（找不到完整答案就找相关上下文，让 LLM 推理）
  [4] 兜底：透明告知"未找到相关文档" + 转人工
```

## 五、成本和稳定性控制

```
硬上限：每个 query 最多 N 轮（典型 3-5）
  ↓
早停：每轮算一次置信度，超过 τ 就停
  ↓
缓存：同一个 sub-query 命中过就不重复
  ↓
降级：Agentic RAG 失败/超时 → fallback 到传统 RAG
  ↓
小模型决策 + 大模型总结：决策环节用 Haiku / 豆包 lite，便宜稳定
```

## 六、加分点

提到 **Anthropic 的 deep-research / multi-agent research 论文**：把研究任务拆给多个 sub-agent 并行做，每个 sub-agent 负责一个子问题，最后 orchestrator 汇总。比单 agent 串行检索快 3-5 倍。

## 七、Agentic RAG 失败 fallback

- 超时 / 步数耗尽 → 用最后一轮检索结果做传统 RAG 生成（保证有答案）
- 全部检索失败 → 透明告知 + 转人工（不要硬编）

## 八、扩展

- **置信度评估**：让 LLM 自评"我对这个答案有多确定"（0-1），但 LLM 自评有过度自信偏差 → 用 Retrieval confidence（检索分数）+ LLM 自评 + 多次采样一致性 综合判断
- **成本预算**：给每个 query 设 token budget，超了就强制降级
- **评测指标**：除了答案准确率，还要看"平均检索轮数""平均成本""早停命中率"

## 记忆要点

- 核心区别：传统RAG无决策权单次检索，Agentic靠LLM决策多跳检索
- Query Rewrite不算Agentic，缺乏“看结果决定是否再查”的决策回路
- 多轮检索三范式：Self-Ask拆子问题、ReAct边想边查、Plan-and-Execute先规划
- 控制高成本与不稳定性：设最大轮数上限、置信度早停、失败则降级传统RAG

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你强调 Agentic RAG 的核心是"决策回路"。但传统 RAG 加个"如果结果不够就再查一次"的 if 判断不也是决策吗？你说的决策回路到底比这个 if 强在哪？**

if 判断是"规则决策"——基于确定性条件（如检索结果 < N 条就再查），覆盖不了"结果够了但质量差"或"结果多但答非所问"这种语义判断。Agentic RAG 的决策回路是"LLM 决策"——模型读完检索结果后判断"这些信息够不够回答、缺什么维度、要不要换 query 再查"，这是基于语义理解的决策，不是规则能覆盖的。本质区别是决策能力：规则决策是离散触发条件，LLM 决策是连续的语义评估。代价是 LLM 决策引入不确定性（可能误判"够了"或"不够"），所以必须加上下界（最大轮数、早停阈值）兜底。

### 第二层：证据与定位

**Q：Agentic RAG 成本是传统 RAG 的 3-10x。你怎么知道多花的成本是花在"有效检索"上，而不是 Agent 在兜圈子重复查同一个东西？**

靠 trace 里每轮检索的 query 和结果去重率分析。每次检索落库 `sub_query`、`retrieved_doc_ids`、`is_cached`。事后统计：如果某次会话的 5 轮检索里，有 3 轮的 `sub_query` 语义高度相似（用 embedding 相似度 > 0.9 判断）且召回文档重叠率 > 70%，说明 Agent 在兜圈子——这是无效成本。正常的多轮检索应该是每轮 `sub_query` 指向不同子问题（如"A 公司 CEO"→"该 CEO 母校"），召回文档重叠率低。监控指标是"有效检索轮数占比" = 去重后有效轮数 / 总轮数，低于 60% 说明决策回路有问题，要优化 Agent 的"还要不要查"判断。

### 第三层：根因深挖

**Q：你用置信度早停（超过 τ 就停）。LLM 自评置信度有过度自信偏差——它总觉得自己查够了。你怎么定 τ，又怎么防止它过度自信提前停？**

LLM 单一自评确实不可靠（研究显示 LLM 自评与真实正确率的相关性弱）。所以置信度不能只看 LLM 自评，要三路融合：一是检索侧置信度（Rerank 分数，分数高说明证据强）；二是 LLM 自评（让模型输出 0-1 的 confidence 并要求给出理由）；三是多次采样一致性（同一问题采样 3 次答案，一致说明稳定）。三者加权综合判断，任一路过低都不早停。τ 的设定要在标注集上画"早停率 vs 准确率"曲线——τ 太高早停几乎不触发（成本高），τ 太低早停过度（准确率掉），选准确率开始明显下降前的拐点（典型 0.7-0.8）。

**Q：那如果 Agent 的决策回路本身判断不准（比如该再查时它说够了，或不该查时它一直查），为什么不用固定的 Plan-and-Execute 把步骤定死，反而用 ReAct 这种动态决策？**

因为 Plan-and-Execute 的前提是"能预先知道所有步骤"，但飞连 IT 工单场景的用户描述往往不完整（"我登不上"——是密码错、网络问题还是账号被封？），必须边查边根据结果调整方向，预先 plan 不出来。ReAct 的动态决策虽然有不稳定性，但能处理信息逐步展开的场景。正确做法是两者结合而非二选一：用 Plan-and-Execute 做粗框架（先查账号状态、再查网络、再查权限），在每个 plan 步骤内部用 ReAct 做细粒度调整（查账号状态时发现异常，追问具体错误码）。这样既有框架的稳定性，又有 ReAct 的灵活性。纯 ReAct 的问题是没有全局规划容易跑偏，纯 Plan 的问题是太死板。

### 第四层：方案权衡

**Q：Agentic RAG 失败时你降级到传统 RAG。但传统 RAG 效果本来就差，降级后用户体验也差。为什么不直接转人工，反而要给一个质量差的答案？**

因为"质量差但有答案"比"没答案转人工"对用户更好——大部分用户的问题其实是简单事实查询，传统 RAG 的单次检索就能答对 60-70%，只有复杂多跳问题才需要 Agentic。降级到传统 RAG 是"保底有答案"，不是"给错误答案"。关键是降级时要标注置信度：如果传统 RAG 的检索结果置信度也低（Rerank 分数 < 阈值），这时才转人工而不是硬答。所以链路是：Agentic RAG（首选）→ 传统 RAG + 高置信度（降级但能用）→ 转人工（彻底兜底）。三层兜底覆盖不同场景，不是非此即彼。

**Q：你说小模型决策 + 大模型总结（决策用 Haiku/豆包 lite，总结用大模型）。为什么不全程用大模型，决策环节难道不需要强推理能力？**

决策环节需要的是"判断够不够"这种轻量分类，不是深度推理——强推理留给最后的总结生成。小模型做决策有三个好处：一是快（Haiku 延迟是 Sonnet 的 1/3），多轮决策累计的延迟差异显著；二是便宜（成本 1/10），而决策在一个 query 里要跑 N 次；三是足够准——"要不要再查"是二分类，小模型的判断准确率和 大模型差距小（可能 85% vs 92%），但成本差 10 倍。只有当下游验证发现小模型决策误判率高（如过早停导致准确率掉 5 个点）时，才考虑把决策也升级到大模型。这是性价比工程，不是能力工程。

### 第五层：验证与沉淀

**Q：你怎么证明 Agentic RAG 比传统 RAG 值得多花 3-10x 成本？业务上这笔账怎么算？**

算 ROI 而不是单看准确率。一边是收益：Agentic RAG 把多跳问题的准确率从 50% 提到 80%（举例），多答对的 30% 问题原本要转人工，人工成本是 X 元/单；一边是成本：Agentic RAG 每次多花 Y 元 token 成本。如果 X × 30% 单量 > Y × 总单量，就值得。具体到飞连 IT 工单：如果月 10 万单，多跳问题占 20%（2 万单），Agentic 让其中 30%（6000 单）不再转人工，每单人工成本 20 元，省 12 万/月；Agentic 增加的 token 成本假设 3 万/月，净收益 9 万。还要看用户满意度提升的间接收益（NPS、续费率）。证明逻辑是"省下的人工成本 > 增加的算力成本"，不是单看技术指标。

**Q：怎么让团队后续迭代 Agentic RAG 时不破坏稳定性（比如有人把最大轮数从 5 调到 10 导致成本爆炸）？**

把成本和稳定性的边界做进系统而不是靠人。一是硬限：最大轮数、token budget、单 query 成本上限写在配置中心，变更要 review 且有自动告警（成本超日均 2 倍触发）；二是回归：维护一份 Agentic RAG 评测集（含简单/多跳/失败兜底场景），任何 Prompt/参数/模型变更触发 CI 跑评测，看"准确率、平均轮数、平均成本、早停命中率"四指标，任一指标退化超阈值 block；三是灰度：新版本先灰度 5% 流量，观察成本曲线和准确率（用 LLM-as-a-Judge 抽样），稳定再全量。让稳定性靠系统保障，不靠"下次注意"。
