# 【阶跃星辰/字节面经】用 RL 训练 MoE 架构容易遇到哪些问题？GSPO 相比 GRPO 做了哪些优化？

## 一、MoE 架构回顾

```
MoE（Mixture of Experts）：
  每个 token 经过路由器（router）选择 top-k 个专家处理
  
  token → router → 选择 [专家1, 专家3] → 加权输出
              ↑
         路由是离散选择（哪个专家）

关键特性：
  - 稀疏激活（每个 token 只用 k 个专家，总参数大但计算量小）
  - 路由可学习（router 是可训练的）
  - 负载均衡是关键（避免某些专家过载某些闲置）
```

## 二、RL 训练 MoE 的四大问题

### 问题1：专家负载不均（最严重）
```
RL 优化会让"表现好的专家"获得更多优势
  → 高优势 token 都涌向某个专家
  → 该专家过载，其他闲置
  → 模型容量浪费 + 训练效率低

例如：GRPO 算出某 token 优势高 → 增大该 token 选某专家的概率
     → 越来越多 token 涌向该专家 → 负载崩溃
```

### 问题2：路由不稳定（震荡）
```
token 级 RL（GRPO）每个 token 都有梯度影响路由
  → 不同 token 的优势信号可能矛盾
  → router 参数反复震荡
  → 同一 token 在不同训练步选不同专家
  → 训练不收敛
```

### 问题3：训练效率低
```
MoE 的稀疏激活让每个 token 只更新 k 个专家的参数
  → RL 梯度只反传到被选中的专家
  → 其他专家这一步没更新
  → 相比 dense 模型，参数更新更稀疏
  → 需要更多步数才能收敛
```

### 问题4：expert collapse（专家坍缩）
```
某些专家长期不被选中
  → 参数不更新
  → 永远学不好
  → 永远不被选中（死循环）
  → 该专家"死亡"，浪费模型容量
```

## 三、GRPO 直接用在 MoE 上的问题

```
GRPO 的优势 A_i 是 token 级（实际是序列级，但梯度反传到每个 token）
  → 每个 token 的 log π 都被 A_i 加权更新
  → 影响 router 对该 token 的路由决策
  → 三个 MoE 问题都加剧：
     1. 高优势 token 涌向同一专家 → 负载更不均
     2. token 级梯度噪声 → 路由更震荡
     3. 稀疏更新 → 效率更低
```

## 四、GSPO 的三大优化

### 优化1：序列级优势（核心）
```
GRPO：一个序列内所有 token 共享同一个 A_i（序列级优势）
       但梯度反传时每个 token 都贡献
       
GSPO 进一步：
  优势计算和梯度传播都从"序列整体"出发
  减少单 token 对路由的扰动
  → 路由更稳定（一整条序列的优化方向一致）
```

**为什么序列级更适合 MoE**：序列级优化让 router 看到的是"整条序列该走哪些专家"，而非"每个 token 单独抢专家"，减少路由震荡。

### 优化2：显式负载均衡损失
```
GSPO 在 RL 损失中加入负载均衡项：

L_total = L_RL + α · L_balance

L_balance = 专家使用频率的方差/熵
  → 强制各专家被使用频率接近均匀
  → 防止负载倾斜和 expert collapse

具体（类似 Switch Transformer 的辅助损失）：
  L_balance = N · Σ_i (f_i · P_i)
  f_i：专家 i 实际被选中的 token 比例
  P_i：router 给专家 i 的平均概率
  → f_i · P_i 大时惩罚（某专家既被选中概率高又被实际选）
```

### 优化3：重要性采样修正
```
MoE 的稀疏激活让"行为策略"和"目标策略"差异大
  → off-policy 程度更高
  → 需要更准确的重要性采样修正

GSPO 用 token 级重要性权重修正：
  对每个 token 乘 importance weight
  适应 MoE 的稀疏激活特性
```

## 五、GSPO vs GRPO 总结

| 维度 | GRPO | GSPO |
|------|------|------|
| 优势粒度 | 序列级（但梯度 token 级） | **序列级（梯度也平滑）** |
| 负载均衡 | 无 | **显式负载均衡损失** |
| 重要性采样 | 基础 | **token 级修正** |
| MoE 适配 | 差（加剧问题） | **好（专门优化）** |
| Dense 模型 | 主流 | 也可用 |

## 六、其他 MoE-RL 优化思路

```
1. 路由正则化：限制 router 输出分布的熵，防止过于尖锐
2. 专家 dropout：训练时随机让某些专家不可用，强制负载分散
3. capacity factor：硬性限制每个专家处理的 token 数上限
4. expert choice：反过来让专家选 token（而非 token 选专家）
5. 共享专家：保留一个 dense 专家处理通用信息，MoE 专家处理专长
```

## 七、加分点

- 说出 **MoE RL 的核心矛盾是"RL 梯度扰动离散路由"**，这是所有问题的根源
- 说出 **GSPO 的负载均衡损失**借鉴 Switch Transformer 的辅助损失
- 说出 **expert collapse** 是 MoE 训练的经典失败模式

## 八、雷区

- ❌ "GRPO 不能用在 MoE" → 能用，但效果差，需 GSPO 优化
- ❌ "负载均衡只需 RL 处理" → 预训练阶段也要（Switch Transformer 的 aux loss）
- ❌ 混淆 MoE 的"参数稀疏"和"激活稀疏" → 激活稀疏（每 token 只用 k 个专家）是核心

## 九、扩展

- **DeepSeek-V3/R1 的 MoE**：用共享专家 + 路由专家，缓解负载均衡
- **DAPO**（字节）：另一款 GRPO 变体，针对 RLHF 优化（动态采样 + 长度归一化）
- **Mixtral / Qwen-MoE**：开源 MoE 模型，训练细节公开可参考

## 记忆要点

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RL 训 MoE 听起来就是把 GRPO 跑在 MoE 模型上，为什么会产生"专家负载不均"？dense 模型 RL 不会有这问题，是因为 dense 没有路由层吗？**

核心是 MoE 的路由（router）是"可学习的离散选择"。dense 模型所有参数对每个 token 都激活，RL 梯度均匀反传到所有参数，不存在"谁多用谁少用"的问题。MoE 的 router 根据输入决定每个 token 去哪些专家，这个路由决策是可微的（softmax 概率 + top-k 选择），RL 梯度会反向影响 router 的参数，进而改变路由分布。问题是：RL 优化目标是"最大化回报"，如果某些专家处理高 reward 的 token 表现好，RL 会增强 router 把更多 token 路由到这些"好专家"——正反馈循环导致专家负载倾斜（热门专家过载，冷门专家闲置甚至 collapse 死亡）。dense 模型没有 router，没有这个反馈循环，所以不会有负载不均。根因是"RL 梯度 + 可学习路由"的耦合：RL 不知道"负载要均衡"，它只优化 reward，reward 信号驱使 router 集中到高效专家，破坏了负载均衡。GSPO 的负载均衡损失就是显式给 RL 加约束，防止这个正反馈失控。

### 第二层：证据与定位

**Q：你说 RL 训 MoE 会"专家负载倾斜"。但训练中怎么观测？监控哪些指标能确认是"负载不均"导致的训练失败，而不是其他原因（如学习率问题）？**

监控三类指标。一是专家负载分布（最直接）：每步记录每个专家被选中的 token 数（或 token 比例），算负载方差或基尼系数。健康训练的负载应接近均匀（N 个专家每个约 1/N），如果某几个专家占 > 30% 而其他 < 5%，就是负载倾斜。可以画"专家负载热力图"（x 轴训练 step，y 轴专家编号，颜色是负载），看倾斜趋势。二是路由熵：router 输出的 softmax 分布的熵，高熵（接近均匀）健康，低熵（集中于少数专家）危险。熵持续下降是 expert collapse 的前兆。三是训练失败的具体症状：loss/reward 震荡（路由不稳定导致梯度方向变）、某些专家的梯度范数长期为 0（专家死亡，不更新参数）、effective expert count 下降（实际在用的专家数减少）。区分是否负载不均导致：如果学习率问题，loss 会发散或 NaN；负载不均则是 loss 震荡 + 负载分布明显倾斜 + 专家死亡率上升。定位方法是"看负载分布 + 路由熵 + 专家梯度"，三者同时异常基本确诊为负载问题。

### 第三层：根因深挖

**Q：GSPO 用"序列级优势"减少对路由的扰动。但序列内每个 token 的 log π 仍要被梯度反传，序列级只是 A 是标量。为什么"序列级 A"能减少路由震荡？梯度的 token 级噪声不还是存在吗？**

序列级 A 减少"路由梯度的方向冲突"。关键区分"A 的粒度"和"梯度反传的粒度"。GRPO 的 A 是序列级（一个 prompt 的 G 个回答各一个 A_i，序列内 token 共享），梯度确实反传到每个 token 的 log π。问题不是"A 的粒度"而是"梯度方向的冲突"。token 级 RL（假设每个 token 一个 A_t）下，不同 token 的 A_t 可能正负矛盾（有的 token 该增概率有的该降），router 收到混乱信号——同一 token 的路由梯度在不同 step 方向相反，震荡。序列级 A 下，一条序列的所有 token 共享同一个 A_i（正或负），梯度方向一致——要么整体增强这条序列的概率（A_i > 0），要么整体降低（A_i < 0），router 收到的信号是"这条序列的路由模式整体好/差"，方向一致不矛盾，路由更新平滑。但你说得对，token 级梯度噪声（不同 token 对 router 参数的梯度值不同）仍存在，序列级 A 只是让"符号一致"（同序列同号），数值噪声未完全消除。GSPO 的效果是"减少方向冲突"，不是"消除数值噪声"，配合负载均衡损失才能完全稳定。另外，序列级 A 让 router 看到"序列模式"而非"单 token"，更适合 RLHF 的序列级 reward 语义（reward 是整段回答的，序列级 A 与之对齐）。

**Q：那如果序列级 A 减少 router 震荡，为什么不在 dense 模型 RL 里也用序列级？dense 模型用 token 级（如 PPO 的 GAE 给每个 token 一个 A_t）不也会有"方向冲突"吗？为什么不统一用序列级？**

dense 模型 RL 其实也有方向冲突问题，但 dense 没有路由层，冲突的后果是"参数更新方向震荡"（训练慢但不崩），不像 MoE 会导致"路由震荡 + 负载崩溃"（致命）。所以 dense 模型容忍 token 级 A 的噪声，且 token 级 A 有信息优势——GAE 的 token 级 A_t 提供信用分配（哪些 token 贡献了 reward），比序列级 A（整序列一个值）更精细，dense 模型的所有参数都参与每个 token 的计算，能利用这个细粒度信号。MoE 的情况不同：每个 token 只激活 k 个专家，token 级 A 的梯度只更新 k 个专家，其他专家不更新——token 级的细粒度信号在 MoE 下导致"更新稀疏 + 路由震荡"双重问题，得不偿失。所以选择取决于架构：dense 用 token 级（信息丰富，tolerance 高），MoE 用序列级（信息粗但稳定，tolerance 低）。这是架构驱动的选择，不是"dense 也有问题但忽略了"——dense 的问题不致命所以用 token 级换信息，MoE 的问题致命所以用序列级换稳定。

### 第四层：方案权衡

**Q：GSPO 用显式负载均衡损失 L_balance = N·Σ(f_i·P_i)。这个损失借鉴 Switch Transformer。但 Switch 是预训练阶段用的，GSPO 在 RL 阶段加这个损失，会不会和 RL 目标冲突？怎么权衡 L_RL 和 L_balance 的权重 α？**

会有冲突，需要小心权衡 α。冲突的本质：L_RL 优化"reward 最大化"，倾向把 token 路由到高效专家（负载倾斜）；L_balance 优化"负载均匀"，强制 token 分散到所有专家。两者方向相反，α 太大则 RL 目标被削弱（reward 上不去），α 太小则负载均衡失效（回到倾斜）。权衡 α 的方法：一是从 Switch Transformer 的经验值起步（α=0.01），然后调。二是看两个指标的平衡：reward 曲线应稳定上升（L_RL 有效），负载方差/路由熵应稳定在健康范围（L_balance 有效）。如果 reward 上不去，降 α；如果负载倾斜，升 α。三是动态 α：训练初期负载容易倾斜（router 未稳定），用大 α 强制均衡；训练后期 router 稳定后，降 α 让 RL 目标主导。四是替代设计：用硬约束（capacity factor，每专家处理 token 数上限）代替软损失，避免 α 调参——但硬约束可能导致 token 被丢弃（溢出），损失信息。GSPO 的做法是软损失 + 调 α，工程上灵活但需调参。DeepSeek-V3 用"共享专家 + 路由专家"架构缓解：共享专家处理通用信息（always on），路由专家处理专长，即使路由倾斜，共享专家兜底，降低了对 L_balance 的依赖。

**Q：GSPO 的负载均衡损失用 f_i·P_i，为什么是这个形式？不用简单的"负载方差" Σ(f_i - 1/N)²？f_i·P_i 的设计巧妙在哪？**

f_i·P_i 的设计巧妙在"同时惩罚概率和实际选择的双重集中"。f_i 是专家 i 实际被选中的 token 比例（硬统计），P_i 是 router 给专家 i 的平均概率（软输出）。f_i·P_i 大意味着"专家 i 既被高概率选中又被实际选中"（双重热门），应惩罚。只用负载方差 Σ(f_i - 1/N)² 的问题：它只看 f_i（实际选择），router 可以"输出高概率给某专家但实际选择被 top-k 截断"——f_i 均匀但 P_i 倾斜，router 的软输出仍集中，下一步可能突然倾斜。f_i·P_i 同时约束 P_i（软输出）和 f_i（硬选择），防止"软输出倾斜但硬选择暂未倾斜"的隐患。具体来说，L_balance = N·Σ(f_i·P_i)，当所有专家 f_i = P_i = 1/N 时取最小值 N·N·(1/N²) = 1（均匀），任何倾斜都让 L > 1。f_i 和 P_i 的乘积还提供了"梯度信号"——梯度对 P_i（router 参数可微）有效，反传时鼓励 router 降低过度热门专家的概率。这是 Switch Transformer 的设计，GSPO 直接沿用。简单方差只约束 f_i 但 f_i 对 router 不可微（top-k 是离散的），梯度传不回 router，所以必须用 f_i·P_i 让 P_i 参与计算，才能有效更新 router。

### 第五层：验证与沉淀

**Q：怎么证明 GSPO 比 GRPO 更适合 MoE RL？不只是理论上说"负载更均衡"，而是实验数据上看到的差异？**

对照实验设计。一是固定 setup：同 MoE 架构（如 8 专家 top-2）、同 base 模型、同 RLHF 数据，只改优化器（GRPO vs GSPO），对比训练。二是核心对比维度：专家负载分布（每步的负载方差/基尼系数，GSPO 应更接近均匀）、路由熵（GSPO 应更高更稳定）、expert collapse 频率（GSPO 应 0 死亡专家，GRPO 可能多个）、final reward（GSPO 应持平或略高，因为容量利用率高）、训练稳定性（reward 曲线方差，GSPO 应更平滑）。三是 ablation：分别去掉 GSPO 的三个优化（序列级 A、负载损失、重要性采样），验证每个的贡献——预期是负载损失贡献最大（防 collapse），序列级 A 次之（减震荡），重要性采样最小（off-policy 修正）。四是诊断可视化：画训练过程中的"专家负载热力图"和"路由决策一致性"（同 token 在不同 step 是否选同专家），GSPO 应显示稳定的负载分布和一致的路由。五是论文证据：Switch Transformer 的 aux loss 实验、DeepSeek-MoE 的训练细节、GSPO 原始论文（如有）的对比，都是公开参考。证明逻辑是"同条件只改方法，多维度（负载/reward/稳定性）对比 + ablation 分解贡献"，用数据证明 GSPO 的 MoE 适配性。

**Q：怎么让团队在训 MoE 时，不踩"直接用 GRPO 导致专家死亡"的坑？沉淀一套 MoE RL 的训练 checklist。**

沉淀 MoE RL 训练 checklist 和诊断规范。一是 checklist（训前检查）：启用负载均衡损失（α=0.01 起步）；考虑 GSPO 或加序列级 A 选项；设置 capacity factor（如 1.5，防止单专家溢出崩溃）；配置 expert dropout（训练时随机禁用部分专家，强制负载分散）；共享专家架构（DeepSeek 风格，兜底通用信息）。二是监控规范（训中监控）：专家负载分布（每步记日志，画热力图）；路由熵（实时监控，低于阈值告警）；专家死亡率（梯度范数为 0 的专家数，> 0 立即告警）；effective expert count（实际在用专家数）。三是应急处理：负载倾斜超阈值时，临时升 α 或启用 capacity factor；专家死亡时，考虑 reset 死亡专家参数或重新初始化 router；路由震荡时，降 learning rate 或加 router 参数的 EMA 平滑。四是回归测试：每次 MoE RL 跑完，检查"所有专家都被合理使用"（负载方差 < 阈值、0 死亡专家），不达标不交付。让 MoE RL 是"checklist 预防 + 监控发现 + 应急处理 + 回归验证"的闭环，不靠"跑起来再说"。
