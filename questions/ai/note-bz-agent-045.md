# 什么是 RAG？整体流程是怎样的？

## 一、RAG 核心定义

**RAG** = **R**etrieval **A**ugmented **G**eneration（检索增强生成）

**核心思想：** 在 LLM 生成答案前，先从外部知识库**检索**相关信息，把检索结果作为上下文**增强**Prompt，让 LLM 基于检索结果**生成**答案。

```
传统LLM（闭卷考试）：
  用户问题 → LLM（凭训练记忆）→ 答案
  问题：知识过时/幻觉/不知道私有数据

RAG（开卷考试）：
  用户问题 → 检索知识库 → 相关文档 → 增强Prompt → LLM → 答案
  优势：知识最新/可溯源/可访问私有数据
```

## 二、RAG 完整流程

```
┌──────────────────────────────────────────────────────┐
│              RAG 两阶段流程                             │
├──────────────────────────────────────────────────────┤
│                                                        │
│  阶段1: 数据准备（离线，一次性或定期更新）              │
│                                                        │
│  原始文档 → 加载 → 分块(Chunking) → 向量化(Embedding) │
│              ↓                              ↓          │
│         PDF/Word/网页              每块转成向量         │
│         DB/API数据                         ↓          │
│                                   存入向量数据库       │
│                                                        │
├──────────────────────────────────────────────────────┤
│                                                        │
│  阶段2: 查询回答（在线，每次提问）                     │
│                                                        │
│  用户问题 → 向量化 → 向量检索(top-k) → Rerank        │
│                                ↓                       │
│                         最相关的文档块                  │
│                                ↓                       │
│  组装Prompt: [系统指令 + 检索到的文档 + 用户问题]      │
│                                ↓                       │
│                    LLM生成答案                         │
│                                ↓                       │
│                    返回答案+来源引用                    │
│                                                        │
└──────────────────────────────────────────────────────┘
```

## 三、各环节详解

### 1. 文档加载（Loading）

```python
from langchain.document_loaders import (
    PyPDFLoader,        # PDF
    WebBaseLoader,      # 网页
    NotionLoader,       # Notion
    CSVLoader,          # 表格
)

docs = PyPDFLoader("manual.pdf").load()
# 每个文档: {page_content: "...", metadata: {source, page}}
```

### 2. 文档分块（Chunking）

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每块500字符
    chunk_overlap=50,    # 块间重叠50字符（防切断语义）
)
chunks = splitter.split_documents(docs)
# 一个100页PDF → 拆成几百个小块
```

### 3. 向量化（Embedding）

```python
from langchain.embeddings import OpenAIEmbeddings

embedder = OpenAIEmbeddings()
vector = embedder.embed_query("什么是Agent")
# → [0.1, -0.3, 0.5, ...]  # 1536维向量
# 语义相近的文本，向量也相近
```

### 4. 存储（Vector Store）

```python
from langchain.vectorstores import Chroma

db = Chroma.from_documents(
    chunks, 
    embedder,
    persist_directory="./vector_db"
)
# 文档块+向量存入向量数据库，支持相似度检索
```

### 5. 检索（Retrieval）

```python
# 用户提问
query = "Agent怎么选工具"

# 向量化查询 → 检索最相似的文档块
relevant = db.similarity_search(query, k=5)
# 返回5个最相关的文档块
```

### 6. 生成（Generation）

```python
prompt = f"""
基于以下参考文档回答问题。如果文档中没有答案，说"我不知道"。

参考文档:
{format_docs(relevant)}

问题: {query}
"""
answer = llm(prompt)
```

## 四、RAG 解决的核心问题

```
┌──────────────────┬──────────────────────────────┐
│ 问题              │ RAG如何解决                    │
├──────────────────┼──────────────────────────────┤
│ 知识过时          │ 知识库可随时更新，无需重训模型  │
│ 幻觉              │ 基于检索文档回答，可溯源       │
│ 私有数据无法访问  │ 把私有文档建索引，检索注入     │
│ 微调成本高        │ RAG无需训练，接入即用          │
│ 知识更新频繁      │ 只更新知识库，模型不变         │
│ 需要引用来源      │ 检索结果自带来源              │
└──────────────────┴──────────────────────────────┘
```

## 五、RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| **适合** | 事实知识/动态数据 | 能力/风格/格式 |
| **更新成本** | 低（更新知识库） | 高（重新训练） |
| **时效性** | 实时 | 训练时固定 |
| **可溯源** | 是 | 否 |
| **计算成本** | 推理时检索 | 训练时高 |
| **知识量** | 无限（看库大小） | 有限（参数容量） |

```
最佳实践：RAG + 微调结合
  - 微调：让模型学会"怎么回答"（风格/格式/领域推理）
  - RAG：提供"回答什么"（最新事实/私有知识）
```

## 六、RAG 的演进

```
Naive RAG（基础）→ Advanced RAG（优化）→ Agentic RAG（智能）

Naive RAG:  查询→检索→生成（固定流程）
Advanced RAG: + 查询改写/重排序/混合检索/分块优化
Agentic RAG: Agent自主决策检索策略（何时/查什么/查几次）
```

## 七、面试加分点

1. **"开卷考试"类比**：最直观解释 RAG 的价值
2. **强调知识与能力解耦**：RAG 的本质是把知识外部化，这是区别于微调的核心
3. **提可溯源**：RAG 能给出来处，这是比纯 LLM 重要的优势（尤其企业场景）

## 记忆要点

- 一句话定义：检索增强生成，相当于让大模型从闭卷变开卷考试。
- 离线阶段(数据准备)：文档加载→分块→向量化→存入向量库。
- 在线阶段(查询回答)：问题向量化→检索Top-K文档→组装增强Prompt→LLM生成带引用的答案。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：RAG 解决 LLM 知识过时和幻觉，但为什么不直接把新知识 fine-tune 进模型，而是用 RAG 外挂？**

因为 RAG 和 fine-tune 解决的问题不同。1）知识时效性——RAG 检索最新文档（实时更新知识库即可），fine-tune 要重训才能更新知识（成本高、周期长），时效性 RAG 完胜；2）知识可追溯——RAG 的答案基于检索到的具体文档（可引用来源），fine-tune 的知识内化在参数里（无法追溯来源，易幻觉），可解释性 RAG 好；3）成本——RAG 只维护外部知识库（增删改文档），fine-tune 每次知识变更要重训（算力+数据成本），RAG 低；4）适用场景——频繁更新/海量/需引用的知识用 RAG（如产品文档/法规），稳定的能力/风格用 fine-tune（如让模型学会某领域推理方式）。所以 RAG 和 fine-tune 互补，不是二选一——RAG 管"知识"，fine-tune 管"能力"。

### 第二层：证据与定位

**Q：RAG 系统生成的答案不对（如答非所问/信息错），怎么定位是检索没检索到（召回问题）还是 LLM 没用好检索结果（生成问题）？**

检查检索结果。1）看检索到的文档——把 RAG 检索 top-K 的文档打印出来，人工判断"这些文档是否包含回答问题所需的信息"，如果不包含，是召回问题（检索没找到对的文档）；2）看 LLM 输入——如果检索文档包含答案但 LLM 输出错，是生成问题（LLM 没正确理解/使用检索结果）。定位方法：在 RAG pipeline 中加 trace（query→检索文档→LLM prompt→输出），逐环节检查。常见根因：召回问题（query 和文档表述不一致/分块不合理/embedding 不准）、生成问题（prompt 没强调"基于检索结果回答"/检索结果太多 LLM 没全用/LLM 幻觉忽略检索）。

### 第三层：根因深挖

**Q：RAG 检索用 embedding 相似度，但 embedding 模型不懂业务术语（如"飞书"匹配不到"Feishu"），导致召回差，怎么优化？**

三招对齐语义。1）术语字典——预处理时把业务术语标准化（如建立"飞书↔Feishu"映射，query/文档统一替换），消除术语不一致；2）query 改写——用 LLM 改写 query 扩展表述（如"飞书"改写为"飞书/Feishu/字节跳动办公软件"），多表述都去检索；3）微调 embedding——用业务数据（query-文档对）微调 embedding 模型（如对比学习让"飞书"和"Feishu 文档"embedding 靠近），让 embedding 学会业务语义。进阶：混合检索（向量+关键词 BM25），关键词检索不受术语影响（"飞书"关键词能匹配到含"飞书"的文档，即使 embedding 不匹配）。选型按成本：术语字典最快（规则），query 改写中等（LLM），微调 embedding 最重但效果最好。

**Q：RAG 要把文档切块（chunk）再检索，但块切大了（检索不准，块内信息杂）切小了（信息碎片，不完整），怎么确定块大小？**

按"语义完整性和检索精度"平衡。1）语义完整——每块应包含相对完整的信息（如一个段落的主题/一个知识点），切太小（如一句话）信息碎片，LLM 难理解；2）检索精度——每块应聚焦单一主题，切太大（如整页）块内信息杂，检索时相似度被稀释（query 和块的部分相关但整体相似度不高）。确定方法：1）经验值——一般 200-500 token/块（兼顾完整性和精度）；2）语义切分——按文档结构切（段落/标题/句子），而非固定字符数，保证语义边界；3）重叠切分——相邻块重叠一部分（如 50 token），避免切在关键信息中间导致两块都不完整；4）AB 测试——不同块大小跑评估集（Recall@K/答案准确率），选最优。

### 第四层：方案权衡

**Q：RAG 检索可以用向量检索（语义）或关键词检索（BM25），为什么实际多用混合检索（两者结合）？**

因为两者互补。1）向量检索——擅长语义匹配（query 和文档表述不同但意思相同，如"如何省钱"匹配"节约成本的方法"），但不擅长精确匹配（如产品型号"X-100"向量检索可能匹配不到）；2）BM25——擅长精确匹配（关键词命中，如"X-100"能匹配到含"X-100"的文档），但不擅长语义（同义表述匹配不到）。混合检索——两者并行检索，结果融合（如 RRF——倒数排名融合），兼顾语义和精确。选型：纯语义场景（如开放问答）向量够，含专有名词/代码/型号的场景必须加 BM25。实务：默认混合（覆盖更广），除非确认场景纯语义才纯向量。

**Q：RAG 检索 top-K 文档给 LLM，K 大了 context 长（成本高+attention 稀释），K 小了可能漏关键文档，怎么定 K？**

按"召回率和 context 成本"权衡。1）召回率——K 大召回全（关键文档更可能在 top-K），K 小可能漏；2）context 成本——K 大 context 长（token 贵+LLM attention 在长 context 下稀释，可能忽略关键文档即 lost-in-the-middle）。定 K 的方法：1）测 Recall@K——用评估集（query+golden 文档），统计 golden 在 top-K 的比例，K 增大到 Recall 不显著提升为止（如 Recall@5=90%、@10=92%，K=5 够了）；2）加 rerank——先检索 top-20（保召回），再 rerank 选 top-5（精排，质量高），这样 K 小但质量好；3）动态 K——根据 query 难度动态调（简单 query K=3，复杂 K=10）。原则：用 Recall@K 定基准，用 rerank 提质量，不盲目增大 K。

### 第五层：验证与沉淀

**Q：你怎么衡量 RAG 系统的效果（检索准不准、答案好不好）？**

两层指标。1）检索层——Recall@K（golden 文档在 top-K 的比例，衡量召回）、MRR（golden 的平均排名，衡量排序质量）、NDCG（考虑相关性的排名质量）；2）生成层——答案准确率（答案是否正确，人工或 LLM 评判）、答案完整性（是否覆盖要点）、引用准确率（引用的来源是否支持答案）、幻觉率（答案是否包含检索文档外的不实信息）。综合指标用 RAGAS（RAG 专项评估框架，含 faithfulness/answer relevancy/context relevancy 等）。还要看端到端指标（用户满意度/任务完成率）。监控：线上 case 持续收集，定期评估，退化告警。

**Q：RAG 系统怎么沉淀成团队的知识问答能力？**

建 RAG 平台：1）知识库管理——文档接入（多源：wiki/数据库/文件）、自动解析/分块/索引（增量更新，文档变更自动 reindex）；2）检索服务——统一检索接口（向量+BM25+rerank），可配置（K/权重/模型）；3）生成服务——RAG pipeline 封装（检索+prompt+LLM），支持引用/溯源；4）评估闭环——评估集管理+自动评估（RAGAS）+退化告警；5）质量治理——bad case 收集→归因（召回/生成）→优化（调分块/换模型/prompt）→验证。这套写入团队知识平台 SOP，让"知识问答"从"每个应用自建 RAG"变成"统一平台能力"，应用按需调用。
