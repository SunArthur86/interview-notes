# 【神州专车面经】某城市某天订单量突然下降15%，怎么排查？给分析框架

## 一、排查框架（先确认数据，再内外归因）

```
[0] 先确认数据准确性（最重要！）
    │  排除数据问题，避免"虚惊一场"
    ▼
[1] 维度拆解
    │  普降还是局部？局部在哪个维度？
    ▼
[2] 内归因
    │  供给/产品/技术
    ▼
[3] 外归因
    │  天气/竞品/节假日/事件
    ▼
[4] 漏斗定位
    │  曝光→下单→完成，哪环掉了
    ▼
[5] 结论 + 止损
```

## 二、第一步：排除数据问题（最常被忽略）

| 排查项 | 怎么查 |
|--------|--------|
| **埋点丢失** | 今日数据量 vs 昨日同时段，看是不是采集断了 |
| **链路延迟** | 数据还在 Kafka/Flink 链路里没跑完，看任务状态 |
| **口径变化** | 最近改过指标定义吗？（如"完成订单"改成了"支付订单"） |
| **时区/日期** | 跨天/夏令时导致时间窗口错位 |
| **去重逻辑** | 去重规则变了导致计数下降 |

**关键**：很多"业务下降"其实是数据链路问题。先确认数据准再分析业务。

## 三、维度拆解：普降还是局部

```sql
-- 按城市拆（确认是单城市还是全国）
SELECT city, COUNT(*) today, 
       LAG(COUNT(*)) OVER(ORDER BY dt) yesterday,
       (COUNT(*) - LAG(COUNT(*)) OVER(ORDER BY dt)) / LAG(COUNT(*)) OVER(ORDER BY dt) AS change_rate
FROM orders WHERE dt IN ('${today}', '${yesterday}')
GROUP BY city, dt;

-- 按时段拆（确认是全天还是某个时段）
SELECT HOUR(create_time) AS hr, COUNT(*), ...
GROUP BY hr;

-- 按车型拆（确认是某车型还是全车型）
SELECT car_type, COUNT(*), ...;

-- 按用户分层（新老用户/会员等级）
SELECT is_new_user, vip_level, COUNT(*), ...;
```

**判断**：
- **普降**（所有城市/时段/车型都降）→ 系统性问题（技术故障/全局事件）
- **局部降**（某城市/某时段/某车型）→ 定向问题，缩小范围

## 四、内归因：供给 / 产品 / 技术

### 供给端
```sql
-- 司机在线数是否下降？
SELECT dt, COUNT(DISTINCT driver_id) AS active_drivers
FROM driver_online_log GROUP BY dt;

-- 司机接单率是否下降？（有单但不接）
SELECT dt, SUM(CASE WHEN accepted THEN 1 ELSE 0 END) / COUNT(*) AS accept_rate
FROM order_dispatch_log GROUP BY dt;
```
- 司机少 → 供给不足，订单派不出去
- 接单率降 → 司机拒单（可能派单策略变了，或竞品补贴高）

### 产品端
- 最近发版了吗？（新版本有 bug）
- 有功能下线/改版吗？（如改了下单流程导致转化率降）

### 技术端
- 系统故障？查错误日志、告警
- 派单算法调整？查算法上线记录
- 接口响应慢？查 APM 链路

## 五、外归因：天气 / 竞品 / 节假日 / 事件

| 因素 | 怎么查 |
|------|--------|
| **天气** | 该城市今天天气（暴雨/大雪/台风），极端天气出行需求反常 |
| **竞品** | 竞品（滴滴）今天有补贴活动吗？司机/乘客被分流 |
| **节假日** | 是节假日/调休？出行模式与工作日不同 |
| **事件** | 大型活动（演唱会/展会）、交通管制、政策变化（如限号） |

**注意**：出行场景天气影响极大——暴雨时需求暴涨（但供给不足导致成交率降），大雪时需求暴跌。

## 六、漏斗定位：曝光→下单→完成

```
曝光（APP打开看到叫车入口）
  → 点击叫车          转化率 = 点击/曝光
    → 输入目的地下单    转化率 = 下单/点击
      → 司机接单       接单率 = 接单/下单
        → 完成         完成率 = 完成/接单
```

**定位哪环掉了**：
- 曝光降 → APP 流量问题（推送/ASO/竞品分流）
- 点击降 → 入口/界面问题
- 下单降 → 价格/预估不准/无车提示
- 接单降 → 供给不足/派单策略
- 完成降 → 取消率高（等待久/司机取消）

## 七、结论 + 止损

```
排查结论：
  数据准确性：✅ 确认无异常
  维度拆解：仅X城市下降，集中在早高峰时段
  内归因：该城市早高峰司机在线数下降20%（供给不足）
  外归因：X城市今天暴雨，部分司机不出车
  漏斗定位：接单率从80%降到55%（有单但无人接）

止损措施：
  - 短期：给该城市司机发雨天补贴激励出车
  - 中期：优化雨天派单策略（提高接单奖励）
  - 长期：建立雨天供给预警模型，提前激励
```

## 八、加分点

- 说出 **"先确认数据准确性"是第一步**，避免虚惊一场
- 说出 **维度拆解是定位的关键**——普降和局部的解法完全不同
- 说出 **漏斗分析定位环节**——不是笼统说"订单降了"，而是"哪一环的转化率降了"

## 九、雷区

- ❌ 上来就猜原因（"可能是模型挂了"）→ 不结构化，容易漏
- ❌ 只看总量不拆维度 → 局部问题被平均掩盖
- ❌ 忽略数据准确性 → 折腾半天发现是埋点断了

## 十、扩展

- **统计显著性**：下降 15% 是显著还是噪声？做假设检验（t-test），看 p-value
- **异常检测**：用同环比 + 移动平均 + 控制图（如 3-sigma）自动检测异常，设告警
- **归因模型**：用 Shapley value 或因果推断量化各因素贡献（天气贡献多少、供给贡献多少）

## 记忆要点

- 排查零步：必须先确认数据准确性（埋点丢失或延迟），避免数据故障造成的虚惊一场
- 维度拆解：多维度下钻看是全局普降还是局部骤降，据此缩小业务排查范围
- 漏斗定位：顺藤摸瓜找转化率掉落的环节（曝光-点击-下单-接单-完成），精准定位内外因


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：订单量下降 15%，排查的第一步为什么是"确认数据准确性"而不是"找原因"？**

因为如果是数据问题（埋点丢失、链路延迟、口径变化），所有后续分析都是错的。在脏数据上找原因会得出错误结论，浪费时间和误导决策。先确认数据准确（订单表 count 和业务系统对账、埋点上报率正常、统计口径没变），再分析原因。这是"先验证假设前提"的工程思维——数据是分析的基础，基础错了结论必错。

### 第二层：证据与定位

**Q：怎么快速判断是"普降"（所有城市都降）还是"局部"（某几个城市降）？**

维度拆解。按城市分组，算每个城市的订单量同环比变化：1) 如果所有城市都降 15% 左右，是普降，根因可能是全局因素（产品改版、系统故障、节假日）；2) 如果只有 2-3 个城市降 50%+，其他正常，是局部，根因在城市特定因素（如某城市暴雨、竞品补贴、政策变化）。拆解维度包括：城市、时段、车型、用户分层（新老用户）。定位"降在哪"是找"为什么降"的前提。

### 第三层：根因深挖

**Q：拆解后发现"新用户订单降 40%，老用户正常"，根因是获客渠道问题还是产品问题？**

继续拆解新用户漏斗：曝光 → 点击 → 注册 → 首单。1) 如果曝光就降了（投放量减少），是获客渠道问题（市场预算削减、广告效果下降）；2) 如果曝光正常但点击降（CTR 下降），是广告素材或落地页问题；3) 如果注册正常但首单降（转化率下降），是产品体验或补贴问题。根因判断靠漏斗分析——看哪一环的转化率掉了，定位到具体环节再深挖。

**Q：那为什么不直接看"新用户首单转化率"这个指标，而要拆整个漏斗？**

因为只看首单转化率无法区分"前端问题"还是"后端问题"。如果曝光降了 50%、首单也降 40%，但漏斗各环节转化率正常，根因是"流量少了"（前端获客问题），不是产品问题。如果曝光正常但首单降，才是产品转化问题。漏斗拆解能区分"量的问题"和"率的问题"——量的问题找前端（投放、入口），率的问题找后端（产品、体验）。

### 第四层：方案权衡

**Q：内外归因（内部产品/技术 vs 外部天气/竞品/节假日）怎么权衡，先查哪个？**

按"可干预性"排序。先查外部不可控因素（天气、节假日、政策）——如果是这些，业务侧只能接受，不用花精力排查。再查竞品动态（对手补贴、价格战）——如果是竞品，要快速响应。最后查内部因素（产品改版、系统故障、数据异常）——这是可干预的，也是排查的重点。顺序：外部确认（快速排除）→ 竞品监控 → 内部深挖。这样能把精力集中在可解决的问题上。

**Q：为什么不直接做 A/B 测试定位原因，而要靠分析框架？**

A/B 测试适合"验证假设"（如"我猜是新版本导致的，回滚验证"），但前提是已有假设。订单下降 15% 的初期是"没有假设"的探索阶段，需要分析框架（维度拆解 + 漏斗 + 同环比）定位到可能原因，再设计 A/B 验证。而且有些因素无法 A/B（如天气、竞品动作）。所以分析框架是"找方向"，A/B 是"验证方向"，两者配合。

### 第五层：验证与沉淀

**Q：怎么确认找到的根因是对的，而不是"看起来对"？**

三个验证：1) 时间相关性——根因变量和订单下降的时间点是否吻合（如某城市暴雨是 10:00 开始，订单下降也是 10:00 开始）；2) 空间相关性——根因影响的范围和订单下降的范围是否一致（如暴雨只在 A 城，订单下降也主要在 A 城）；3) 反事实验证——根因消除后订单是否恢复（雨停后 A 城订单回升）。三个相关性都满足才能确认根因。沉淀为异常排查 SOP：数据校验 → 维度拆解 → 漏斗分析 → 内外归因 → 根因验证。

## 结构化回答

**30 秒电梯演讲：** 订单量突然下降15%的排查要用"先确认数据准确性，再内外归因"的框架。第一步先排除数据问题（埋点丢失/链路延迟/口径变化），再按"内部因素（供给/产品/技术）"和"外部因素（天气/竞品/节假日/事件）"分层排查。

**展开框架：**
1. **第一步** — 排除数据问题（埋点丢失/链路延迟/口径变化/时区）
2. **维度拆解** — 按城市/时段/车型/用户分层，看是普降还是局部
3. **内归因** — 供给（司机少）/产品（功能bug）/技术（系统故障）

**收尾：** 您想深入聊：怎么区分"短期波动"和"趋势性下降"？
