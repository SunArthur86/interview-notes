# 提示词工程能解决什么问题？

## 一、提示词工程能解决的问题

```
┌──────────────────────────────────────────────────┐
│            Prompt工程能解决的四大类问题             │
├──────────────────────────────────────────────────┤
│                                                    │
│  1. 输出控制（让输出符合预期）                      │
│     - 格式：JSON/表格/Markdown                     │
│     - 风格：正式/口语/专业                          │
│     - 长度：简短/详细                              │
│     - 内容：包含/排除特定信息                       │
│                                                    │
│  2. 能力激发（让模型发挥已有能力）                  │
│     - 推理：CoT激发链式推理                        │
│     - 角色：设定专家角色激活领域知识                │
│     - 创意：开放性引导激发创造力                    │
│                                                    │
│  3. 行为约束（让模型守规矩）                        │
│     - 安全：拒绝有害请求                           │
│     - 边界：不回答超出范围的问题                    │
│     - 一致：多次回答保持一致                        │
│                                                    │
│  4. 质量提升（让输出更好）                          │
│     - 示例：Few-shot规范输出                       │
│     - 自检：要求模型检查自己的答案                  │
│     - 分解：复杂任务分步完成                        │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、具体问题与Prompt对策

### 问题1：输出格式不可控

```python
# 问题：LLM输出格式随意，难以程序解析
# 解决：严格格式约束

PROMPT = """
按以下JSON格式输出，不要任何其他文字：
{
  "summary": "一句话总结",
  "key_points": ["要点1", "要点2"],
  "sentiment": "positive/negative/neutral"
}
"""
```

### 问题2：回答不够专业

```python
# 问题：回答泛泛而谈，不够深入
# 解决：角色设定+领域约束

PROMPT = """
你是拥有10年经验的[领域]专家。
回答要求：
- 使用专业术语（附简单解释）
- 给出具体案例而非泛泛而谈
- 标注信息的可信度
- 涉及最佳实践时引用行业标准
"""
```

### 问题3：推理错误

```python
# 问题：数学/逻辑推理出错
# 解决：CoT思维链

PROMPT = """
请一步一步推理：
1. 首先，[第一步分析]
2. 然后，[基于第一步的推论]
3. 因此，[最终结论]

让我们一步步思考。
"""
```

### 问题4：幻觉

```python
# 问题：编造不存在的信息
# 解决：基于上下文+诚实约束

PROMPT = """
只基于以下参考信息回答。如果信息中没有答案，
明确说"根据提供的信息，我无法回答"。

参考信息: {context}
"""
```

### 问题5：回答不一致

```python
# 问题：同样的问题，不同时间答案不同
# 解决：Few-shot + 明确规则

PROMPT = """
规则（必须严格遵守）：
- 价格一律用人民币，不用美元
- 日期格式统一用YYYY-MM-DD
- 不确定的标注"待确认"

示例：
输入: "价格$100，date 2024/1/1"
输出: "价格: 100人民币，日期: 2024-01-01"
"""
```

## 三、提示词工程解决不了的问题

```
┌──────────────────┬──────────────────────────────┐
│ 解决不了的问题     │ 原因                            │
├──────────────────┼──────────────────────────────┤
│ 模型不知道的知识   │ 训练数据没有（需RAG/微调）      │
│ 超出模型推理能力   │ 如复杂数学（需更强模型）        │
│ 固有偏见          │ 训练数据偏差（需微调对齐）      │
│ 完全消除幻觉      │ 概率模型本质（只能减少）        │
│ 上下文窗口限制    │ 硬约束（需RAG/压缩）            │
│ 实时信息          │ 训练截止后的事（需工具调用）    │
└──────────────────┴──────────────────────────────┘

当Prompt解决不了时：
  知识问题 → RAG（检索增强）
  能力问题 → 换更强模型
  风格问题 → 微调
  实时性   → 工具调用
```

## 四、Prompt 工程的投入产出

```
投入少收益大（优先做）：
  - 格式约束（一句话搞定）
  - Few-shot示例（2-3个）
  - CoT指令（加一句话）
  - 角色设定

投入中收益中：
  - 复杂Prompt模板设计
  - 多版本A/B测试
  - 边界case覆盖

投入大收益递减：
  - 超长Prompt（效果反而下降）
  - 过度优化单个case
  → 此时考虑换模型/微调/RAG
```

## 五、面试加分点

1. **能解决vs不能解决**：讲清边界，Prompt 不是万能的——体现客观
2. **四大类问题**：输出控制/能力激发/行为约束/质量提升，体系化
3. **知道何时升级**：Prompt 解决不了时用 RAG/微调/换模型——体现全局视野

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说 Prompt 工程能解决"输出控制"，那你怎么定义"解决"？是把格式正确率做到多少算解决？**

解决的标准是程序可解析，不是一个绝对百分比。我的口径是：JSON 场景下 `json.loads` 成功率 ≥ 99.9%，也就是千分之一以下需要走兜底修复。为什么定 99.9% 而不是 100%——因为概率模型不可能保证 100%，强求会让 Prompt 越写越长反而触发"中间遗忘"。所以"解决"= 主路径靠 Prompt 命中，剩下 0.1% 靠 JSON Repair 后处理兜底。

### 第二层：证据与定位

**Q：你线上一段时间 JSON 解析失败率从 0.05% 涨到了 2%，怎么定位是 Prompt 退化还是别的？**

先看两个证据交叉验证。第一看失败样本的原始输出——如果失败样本集中在某类输入（比如超长文本、含特殊字符），大概率是输入分布漂移触发 Prompt 没覆盖的 case；如果失败样本随机分布且原始输出里多了口语化前缀（"好的，结果是："），那就是模型版本更新后指令跟随能力下降。第二看模型版本灰度日志——如果失败率飙升的时间点和模型灰度切换时间吻合，就是模型行为漂移，和 Prompt 无关，得在 Prompt 里加更强的格式锚点（如"第一个字符必须是 { "）。

### 第三层：根因深挖

**Q：CoT 你说能解决推理错误，但 GPT-4 加了 CoT 之后某些数学题反而错了，为什么？**

CoT 不是万能的，它会引入两类问题。一是中间步骤一旦出错会"带歪"最终结论——链式推理的错误是放大的，前面错后面全错。二是把简单问题复杂化——本来模型一步直觉能对的题，强行拆步骤反而把概率高的路径走偏。这类题通常是数值近似题或常识题，正确做法不是无脑加 CoT，而是按题型分流：需要严密多步推导的（行程问题、概率题）加 CoT，一步直觉题不加。

**Q：那为什么不直接换更强的模型，还要花精力调 CoT？**

因为成本和延迟。换 GPT-4 级别模型推理成本是 GPT-3.5 的 10-15 倍，单次延迟从 1s 涨到 5s+。如果 CoT 能让 3.5 在 80% 的中等难度题上达到 4 的水平，那这 80% 的请求就不该走 4。正确架构是模型路由——简单题走小模型不加 CoT，中等题走小模型加 CoT，难题才上大模型。

### 第四层：方案权衡

**Q：你说 Prompt 解决不了知识盲区要上 RAG。但 RAG 也可能召回不到，这种场景你选 RAG 还是微调？**

看知识的更新频率和量级。如果知识是高频更新的（比如产品文档、政策），必须 RAG——微调一次成本高且无法实时更新，召回不到是召回侧的问题（该上 Rerank、Query Rewrite），不是要不要 RAG 的问题。如果知识是低频稳定的领域常识（比如医学解剖术语），微调更合适——这类知识不需要实时性，微调后模型内化它，比每次 RAG 召回稳定。我的判断标准：更新周期 < 1 周选 RAG，> 1 个月且量级 < 10 万条选微调，两者之间走 RAG + 微调结合。

**Q：为什么不把所有不能解决的都交给"换更大模型"？**

因为换模型解决不了根本问题，还带来副作用。换模型解决不了知识盲区（GPT-4 也不知道你公司内部数据），解决不了上下文窗口硬上限（换更大的模型窗口也有限），而且大模型在小任务上更贵更慢，ROI 为负。换模型只在一种情况下成立：当前模型的能力天花板确实被触到了（比如 3.5 做不了多步逻辑推理，准确率卡在 60%），这时才升级。

### 第五层：验证与沉淀

**Q：你改了 Prompt 之后怎么证明是改对了，而不是那几天输入简单？**

用固定的评测集而非线上流量验证。我会维护一个 200-500 条的标注评测集，覆盖 easy/medium/hard 三档，每次改 Prompt 前后都在这个集合上跑全量对比，看准确率/格式合规率/人工评分三个指标。如果新 Prompt 在评测集上提升 5% 但线上没动，说明评测集和线上分布有偏差，要回去补评测集，而不是说线上没用。

**Q：这套经验怎么沉淀让团队复用？**

沉淀成 Prompt 资产库 + 评测基线两个东西。Prompt 资产库按场景归类（摘要/抽取/对话），每个 Prompt 带适用模型、评测准确率、已知失败 case。评测基线是 CI 化的——每次 Prompt 变更自动在评测集上跑，准确率回退 > 2% 就拦截发布。这样团队改 Prompt 不是凭感觉，是有数据护栏。
