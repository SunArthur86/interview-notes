# 提示词工程能解决什么问题？

## 一、提示词工程能解决的问题

```
┌──────────────────────────────────────────────────┐
│            Prompt工程能解决的四大类问题             │
├──────────────────────────────────────────────────┤
│                                                    │
│  1. 输出控制（让输出符合预期）                      │
│     - 格式：JSON/表格/Markdown                     │
│     - 风格：正式/口语/专业                          │
│     - 长度：简短/详细                              │
│     - 内容：包含/排除特定信息                       │
│                                                    │
│  2. 能力激发（让模型发挥已有能力）                  │
│     - 推理：CoT激发链式推理                        │
│     - 角色：设定专家角色激活领域知识                │
│     - 创意：开放性引导激发创造力                    │
│                                                    │
│  3. 行为约束（让模型守规矩）                        │
│     - 安全：拒绝有害请求                           │
│     - 边界：不回答超出范围的问题                    │
│     - 一致：多次回答保持一致                        │
│                                                    │
│  4. 质量提升（让输出更好）                          │
│     - 示例：Few-shot规范输出                       │
│     - 自检：要求模型检查自己的答案                  │
│     - 分解：复杂任务分步完成                        │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 二、具体问题与Prompt对策

### 问题1：输出格式不可控

```python
# 问题：LLM输出格式随意，难以程序解析
# 解决：严格格式约束

PROMPT = """
按以下JSON格式输出，不要任何其他文字：
{
  "summary": "一句话总结",
  "key_points": ["要点1", "要点2"],
  "sentiment": "positive/negative/neutral"
}
"""
```

### 问题2：回答不够专业

```python
# 问题：回答泛泛而谈，不够深入
# 解决：角色设定+领域约束

PROMPT = """
你是拥有10年经验的[领域]专家。
回答要求：
- 使用专业术语（附简单解释）
- 给出具体案例而非泛泛而谈
- 标注信息的可信度
- 涉及最佳实践时引用行业标准
"""
```

### 问题3：推理错误

```python
# 问题：数学/逻辑推理出错
# 解决：CoT思维链

PROMPT = """
请一步一步推理：
1. 首先，[第一步分析]
2. 然后，[基于第一步的推论]
3. 因此，[最终结论]

让我们一步步思考。
"""
```

### 问题4：幻觉

```python
# 问题：编造不存在的信息
# 解决：基于上下文+诚实约束

PROMPT = """
只基于以下参考信息回答。如果信息中没有答案，
明确说"根据提供的信息，我无法回答"。

参考信息: {context}
"""
```

### 问题5：回答不一致

```python
# 问题：同样的问题，不同时间答案不同
# 解决：Few-shot + 明确规则

PROMPT = """
规则（必须严格遵守）：
- 价格一律用人民币，不用美元
- 日期格式统一用YYYY-MM-DD
- 不确定的标注"待确认"

示例：
输入: "价格$100，date 2024/1/1"
输出: "价格: 100人民币，日期: 2024-01-01"
"""
```

## 三、提示词工程解决不了的问题

```
┌──────────────────┬──────────────────────────────┐
│ 解决不了的问题     │ 原因                            │
├──────────────────┼──────────────────────────────┤
│ 模型不知道的知识   │ 训练数据没有（需RAG/微调）      │
│ 超出模型推理能力   │ 如复杂数学（需更强模型）        │
│ 固有偏见          │ 训练数据偏差（需微调对齐）      │
│ 完全消除幻觉      │ 概率模型本质（只能减少）        │
│ 上下文窗口限制    │ 硬约束（需RAG/压缩）            │
│ 实时信息          │ 训练截止后的事（需工具调用）    │
└──────────────────┴──────────────────────────────┘

当Prompt解决不了时：
  知识问题 → RAG（检索增强）
  能力问题 → 换更强模型
  风格问题 → 微调
  实时性   → 工具调用
```

## 四、Prompt 工程的投入产出

```
投入少收益大（优先做）：
  - 格式约束（一句话搞定）
  - Few-shot示例（2-3个）
  - CoT指令（加一句话）
  - 角色设定

投入中收益中：
  - 复杂Prompt模板设计
  - 多版本A/B测试
  - 边界case覆盖

投入大收益递减：
  - 超长Prompt（效果反而下降）
  - 过度优化单个case
  → 此时考虑换模型/微调/RAG
```

## 五、面试加分点

1. **能解决vs不能解决**：讲清边界，Prompt 不是万能的——体现客观
2. **四大类问题**：输出控制/能力激发/行为约束/质量提升，体系化
3. **知道何时升级**：Prompt 解决不了时用 RAG/微调/换模型——体现全局视野
