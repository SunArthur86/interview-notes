# 【月之暗面面经】如果一个 Agent 会生成站点、表格和 PPT，前端怎样组织产物面板？

## 一、问题本质：产物 ≠ 消息

传统 Web 聊天的输出是「文本消息」，所有内容都在消息流里线性排列。但 AI-Native 桌面产品的输出是「结构化产物」：

```
一个"帮我做竞品分析"的任务可能产出：
  ├── 📊 竞品对比表格（Excel格式）
  ├── 🌐 竞品分析网站（HTML站点）
  ├── 📑 分析报告PPT（可演示）
  ├── 📈 市场趋势图（图片）
  └── 📝 文字摘要（Markdown）
```

**如果把这些产物塞进消息流，用户很难找到和操作它们**。产物面板的核心目标是：把产物从消息流中拆出来，做成可独立操作的对象。

## 三、产物对象模型

```typescript
// 产物是独立于消息流的一等对象
interface Artifact {
  id: string;
  taskId: string;
  
  // 产物类型
  kind: 'site' | 'sheet' | 'ppt' | 'chart' | 'document' | 'code' | 'image';
  
  // 内容
  title: string;
  content: string;          // 原始内容（HTML/JSON/Markdown）
  previewUrl?: string;      // 预览URL（如果是站点）
  
  // 版本管理
  version: number;
  parentVersion?: number;   // 基于哪个版本重生成
  changelog?: string;       // 本次变更说明
  
  // 状态
  status: 'generating' | 'draft' | 'reviewing' | 'confirmed' | 'exported';
  
  // 来源
  inputRefs: string[];      // 引用了哪些输入
  promptUsed: string;       // 生成时使用的prompt
  
  // 可执行的动作
  actions: ArtifactAction[];
}

interface ArtifactAction {
  type: 'preview' | 'edit' | 'regenerate' | 'export' | 'publish' | 'diff';
  label: string;
  icon: string;
  available: boolean;
}
```

## 四、按产物类型的差异化处理

| 产物类型 | 预览方式 | 编辑方式 | 导出格式 | 特殊处理 |
|---------|---------|---------|---------|---------|
| 站点 | iframe内嵌预览 | 代码编辑器/可视化编辑 | HTML压缩包 | 支持实时预览 |
| 表格 | 表格视图 | 在线编辑单元格 | xlsx/csv | 支持公式高亮 |
| PPT | 幻灯片缩略图+大图 | 逐页编辑 | pptx/pdf | 支持演讲模式 |
| 图表 | 图片/交互图 | 参数调整 | png/svg | 支持数据联动 |
| 文档 | Markdown渲染 | 富文本编辑 | md/docx | 支持评论批注 |

## 五、局部重生成

**整单重跑是最浪费的**——用户只想改一个表格的某一行，不应该重新生成整个站点。

```typescript
// 局部重生成协议
interface PartialRegenerate {
  artifactId: string;
  scope: {
    type: 'cell' | 'row' | 'page' | 'section' | 'full';
    target: string;     // 具体目标（行号/页码/段落ID）
  };
  instruction: string;  // 修改指令
  preserveContext: boolean;  // 是否保留原上下文
}

// 示例：只重新生成PPT的第3页
{
  artifactId: 'ppt-001',
  scope: { type: 'page', target: '3' },
  instruction: '将第3页改为强调移动端数据',
  preserveContext: true
}
```

**局部重生成后，新版本只替换指定部分，其他部分保持不变**。用户可以对比新旧版本差异。

## 六、产物面板与对话流的解耦

```
┌─────────────────────────────────────────────────┐
│                  全局状态管理                      │
│                                                 │
│  ┌──────────────┐      ┌──────────────────┐     │
│  │  TaskStore    │      │  MessageStore     │     │
│  │  (任务+产物)   │      │  (对话流)         │     │
│  │              │      │                  │     │
│  │  Task[]      │      │  Message[]       │     │
│  │  └─Artifact[]│      │                  │     │
│  └──────┬───────┘      └────────┬─────────┘     │
│         │                       │               │
│         └───────┬───────────────┘               │
│                 │                               │
│          ┌──────┴──────┐                        │
│          │  View Layer  │                        │
│          │             │                        │
│          │ 产物面板视图  │                        │
│          │ 对话流视图    │                        │
│          │ 混合视图     │                        │
│          └─────────────┘                        │
└─────────────────────────────────────────────────┘
```

**解耦的好处**：
- 产物面板可以独立滚动、筛选、搜索
- 对话流保持轻量，不被大量产物内容撑爆
- 用户可以只看产物不看对话，或只看对话不看产物
- 新增产物类型时不需要修改对话流组件

## 七、常见坑

- **把产物塞在消息流里**：消息越来越长，用户找不到产物，也无法独立操作
- **整单重跑**：用户只想改一个细节，却要重新生成所有产物，浪费时间
- **产物没有版本管理**：用户修改后发现不如之前的好，无法回退
- **不同产物类型用统一渲染**：站点用文字描述、PPT用JSON展示，用户体验差

## 记忆要点

- 设计初衷：因为结构化产物塞入消息流难检索，所以要拆成独立可操作对象
- 核心数据模型：定义Artifact对象，包含专属类型、多版本历史和生命周期状态
- UI组织方式：以任务面板为主体，网格化平铺各类产物，提供预览导出及Diff比对
