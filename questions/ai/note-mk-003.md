# 【月之暗面面经】如果一个 Agent 会生成站点、表格和 PPT，前端怎样组织产物面板？

## 一、问题本质：产物 ≠ 消息

传统 Web 聊天的输出是「文本消息」，所有内容都在消息流里线性排列。但 AI-Native 桌面产品的输出是「结构化产物」：

```
一个"帮我做竞品分析"的任务可能产出：
  ├── 📊 竞品对比表格（Excel格式）
  ├── 🌐 竞品分析网站（HTML站点）
  ├── 📑 分析报告PPT（可演示）
  ├── 📈 市场趋势图（图片）
  └── 📝 文字摘要（Markdown）
```

**如果把这些产物塞进消息流，用户很难找到和操作它们**。产物面板的核心目标是：把产物从消息流中拆出来，做成可独立操作的对象。

## 三、产物对象模型

```typescript
// 产物是独立于消息流的一等对象
interface Artifact {
  id: string;
  taskId: string;
  
  // 产物类型
  kind: 'site' | 'sheet' | 'ppt' | 'chart' | 'document' | 'code' | 'image';
  
  // 内容
  title: string;
  content: string;          // 原始内容（HTML/JSON/Markdown）
  previewUrl?: string;      // 预览URL（如果是站点）
  
  // 版本管理
  version: number;
  parentVersion?: number;   // 基于哪个版本重生成
  changelog?: string;       // 本次变更说明
  
  // 状态
  status: 'generating' | 'draft' | 'reviewing' | 'confirmed' | 'exported';
  
  // 来源
  inputRefs: string[];      // 引用了哪些输入
  promptUsed: string;       // 生成时使用的prompt
  
  // 可执行的动作
  actions: ArtifactAction[];
}

interface ArtifactAction {
  type: 'preview' | 'edit' | 'regenerate' | 'export' | 'publish' | 'diff';
  label: string;
  icon: string;
  available: boolean;
}
```

## 四、按产物类型的差异化处理

| 产物类型 | 预览方式 | 编辑方式 | 导出格式 | 特殊处理 |
|---------|---------|---------|---------|---------|
| 站点 | iframe内嵌预览 | 代码编辑器/可视化编辑 | HTML压缩包 | 支持实时预览 |
| 表格 | 表格视图 | 在线编辑单元格 | xlsx/csv | 支持公式高亮 |
| PPT | 幻灯片缩略图+大图 | 逐页编辑 | pptx/pdf | 支持演讲模式 |
| 图表 | 图片/交互图 | 参数调整 | png/svg | 支持数据联动 |
| 文档 | Markdown渲染 | 富文本编辑 | md/docx | 支持评论批注 |

## 五、局部重生成

**整单重跑是最浪费的**——用户只想改一个表格的某一行，不应该重新生成整个站点。

```typescript
// 局部重生成协议
interface PartialRegenerate {
  artifactId: string;
  scope: {
    type: 'cell' | 'row' | 'page' | 'section' | 'full';
    target: string;     // 具体目标（行号/页码/段落ID）
  };
  instruction: string;  // 修改指令
  preserveContext: boolean;  // 是否保留原上下文
}

// 示例：只重新生成PPT的第3页
{
  artifactId: 'ppt-001',
  scope: { type: 'page', target: '3' },
  instruction: '将第3页改为强调移动端数据',
  preserveContext: true
}
```

**局部重生成后，新版本只替换指定部分，其他部分保持不变**。用户可以对比新旧版本差异。

## 六、产物面板与对话流的解耦

```
┌─────────────────────────────────────────────────┐
│                  全局状态管理                      │
│                                                 │
│  ┌──────────────┐      ┌──────────────────┐     │
│  │  TaskStore    │      │  MessageStore     │     │
│  │  (任务+产物)   │      │  (对话流)         │     │
│  │              │      │                  │     │
│  │  Task[]      │      │  Message[]       │     │
│  │  └─Artifact[]│      │                  │     │
│  └──────┬───────┘      └────────┬─────────┘     │
│         │                       │               │
│         └───────┬───────────────┘               │
│                 │                               │
│          ┌──────┴──────┐                        │
│          │  View Layer  │                        │
│          │             │                        │
│          │ 产物面板视图  │                        │
│          │ 对话流视图    │                        │
│          │ 混合视图     │                        │
│          └─────────────┘                        │
└─────────────────────────────────────────────────┘
```

**解耦的好处**：
- 产物面板可以独立滚动、筛选、搜索
- 对话流保持轻量，不被大量产物内容撑爆
- 用户可以只看产物不看对话，或只看对话不看产物
- 新增产物类型时不需要修改对话流组件

## 七、常见坑

- **把产物塞在消息流里**：消息越来越长，用户找不到产物，也无法独立操作
- **整单重跑**：用户只想改一个细节，却要重新生成所有产物，浪费时间
- **产物没有版本管理**：用户修改后发现不如之前的好，无法回退
- **不同产物类型用统一渲染**：站点用文字描述、PPT用JSON展示，用户体验差

## 记忆要点

- 设计初衷：因为结构化产物塞入消息流难检索，所以要拆成独立可操作对象
- 核心数据模型：定义Artifact对象，包含专属类型、多版本历史和生命周期状态
- UI组织方式：以任务面板为主体，网格化平铺各类产物，提供预览导出及Diff比对

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么不把站点、表格、PPT 这些产物直接渲染在对话消息里，非要拆成独立的产物面板？消息流加个折叠区不行吗？**

因为产物和消息的本质不同：消息是"一次性的线性文本"，读完就过了；产物是"可迭代的工作成果"，用户会反复预览、编辑、导出、重生成、版本对比。如果产物塞在消息流里，会立刻遇到三个问题：(1) 检索困难——一个长任务产出 5 个产物，混在 50 条消息里，用户要 Ctrl+F 找；(2) 操作困难——消息流里无法独立滚动产物网格、无法并排 Diff 两个版本；(3) 上下文膨胀——把 HTML 站点的完整源码塞进消息会让对话区卡顿。拆成独立面板后，Artifact 成为有 id、version、status 的一等对象，可以独立挂载预览器、编辑器、导出器，用户在产物面板操作产物时完全不影响对话流。折叠区只是"视觉压缩"，没有解决"产物是结构化对象而非文本"的根本问题。

### 第二层：证据与定位

**Q：你怎么定位"用户找不到产物"是产物面板设计问题，而不是任务本身太复杂、产物太多？**

用产物操作埋点定位。在每个产物卡片上埋：曝光数（产物出现在用户视口的次数）、点击预览率、点击编辑率、点击导出率。如果一个任务的产物曝光率高（说明用户看到了）但点击率低（说明用户没操作），问题在产物本身没价值或预览不够吸引人；如果曝光率低（产物在面板下方用户要滚动才看到），问题在面板布局——这时要看"首个产物到用户视口的距离"，如果需要滚动超过 2 屏，说明网格太密集或缺少置顶逻辑。还有一个关键指标是"产物找回耗时"——用户完成任务后回头找一个特定产物的平均耗时，如果超过 10 秒，说明缺乏搜索/筛选/分类能力。这些数据能精确定位是布局问题、检索问题还是产物质量问题。

### 第三层：根因深挖

**Q：为什么局部重生成比整单重跑重要？根因是性能问题还是用户心理问题？**

两者都是，但根因是"用户对产物的控制粒度需求"和"AI 重生成的不确定性"之间的矛盾。性能层面：整单重跑一个含 5 个产物的竞品分析任务要 5 分钟，但用户只想改表格第 3 行的数据，局部重生成只要 15 秒，性能差 20 倍。但更深层是心理层面：整单重跑意味着用户之前手动编辑过的其他产物（如 PPT 改了标题、站点调了配色）全部被覆盖丢失，用户会有强烈的"失控感"——"我不敢再手动改了，因为一重生成就没了"。这种失控感会直接导致用户不敢深度参与产物的迭代，把 AI 产物当作"一次性输出"而非"可协作的工作成果"。局部重生成通过 scope（cell/row/page/section）精确定位重生成范围，保留其他部分，让用户的手动编辑和 AI 的重生成可以共存，本质上是在解决"人机协作的信任"问题。

**Q：那如果 AI 的局部重生成质量比整单重跑差（因为上下文不完整），为什么不为了保证质量放弃局部重生成？**

因为"质量略降但可控"远优于"质量满分但不可控"。局部重生成质量差的根因是上下文不完整——只传了"改第 3 行"的指令，没传整个表格的全局结构。解法不是退回整单重跑，而是优化局部重生成的上下文注入：传给 AI 的不仅是局部修改指令，还包括产物全局结构摘要（如"这是一个 10 行 5 列的竞品对比表，第 3 行是 Notion 的数据"）+ 相邻行作为上下文锚点 + preserveContext=true 的开关（让 AI 知道要保留其他行不动）。这样局部重生成的质量可以逼近整单重跑的 90%，但成本只有 1/20。另一个折中是"局部重生成 + 全局一致性校验"——局部改完后自动检查整体结构（表格行列数、PPT 页数、站点链接）是否一致，不一致才触发全局重跑。所以解法是"让局部更聪明"而非"放弃局部"。

### 第四层：方案权衡

**Q：产物版本管理你用线性版本链（v1→v2→v3）还是树状版本（支持分叉）？怎么选？**

桌面 AI 产物场景选线性版本链，不选树状分叉。根因是"用户的心智模型"：桌面端用户把产物当作"工作文档"，迭代路径是线性的（越改越好），极少出现"我要从 v2 分叉出两个平行版本同时演进"的需求——那是 Git 的场景（代码分支），不是文档协作的场景。树状版本会引入"合并冲突"的复杂性，对非技术用户是认知灾难。线性版本链的实现也更简单：每次重生成 append 一个新版本，parentVersion 指向上一个，用户在 v1/v2/v3 之间线性切换和 Diff。唯一需要树状的场景是"多个用户协同编辑同一产物"，但桌面端是单机产品，不存在这个场景。所以线性版本链在覆盖 95% 需求的同时，把认知复杂度从 O(分支数) 降到 O(版本数)。

**Q：那如果用户说"我想比较 v1 和 v3 哪个更好，但不想丢掉 v2"，为什么不支持非线性比较来满足这个需求？**

因为"非线性比较"和"树状版本"是两回事——前者是只读的 Diff，后者是可写的分叉。用户的需求（比较 v1 和 v3）用线性版本链完全满足：线性链的任意两个版本都可以做 Diff，只是不产生新的分叉分支。具体实现是 Diff 工具接受任意两个 version 参数，用户选 v1 和 v3 做对比，看到差异后决定"回退到 v1"或"保持 v3"或"基于 v3 重生成 v4"。这整个过程不需要树状结构。真正需要树状的是"从 v2 同时演进两个版本到 v2.1 和 v2.2 再合并"——这个需求在桌面文档场景几乎不存在，强行支持只会让 UI 复杂化（用户要理解"当前在哪个分支""合并方向是什么"）。所以不是"不支持非线性比较"，而是"用线性链 + 任意版本 Diff 就能满足比较需求，不需要引入树状的复杂度"。

### 第五层：验证与沉淀

**Q：你怎么证明"产物面板与对话流解耦"比"产物嵌在消息里"更有效，有数据支撑吗？**

用三个关键指标对比验证：(1) 产物操作完成率 = 用户成功预览/编辑/导出产物的次数 / 产物生成总数，解耦后应显著提升，因为产物不再被消息淹没；(2) 任务回看时长 = 用户回到历史任务找到目标产物的平均耗时，解耦后应从"滚动消息流找"降到"直接看产物网格"；(3) 局部重生成使用率 = 局部重生成次数 / 总重生成次数，如果解耦后这个率从 0% 提升到 30% 以上，说明用户因为能独立操作产物才敢于局部迭代。A/B 测试时，对照组用"产物嵌消息"方案，实验组用"产物面板解耦"方案，跑两周看这三个指标的差异。如果实验组的产物采纳率（被导出/保存的产物占比）高出对照组 20% 以上，就证明解耦设计有效。

**Q：怎么让团队在新增产物类型（如思维导图、流程图）时，自觉遵循统一的 Artifact 对象模型，而不是各自定义一套数据结构？**

把 Artifact 模型做成"注册表驱动"的强制约束。第一，定义 ArtifactTypeSpec 接口，每种产物类型必须实现 renderer（预览组件）、editor（编辑组件）、exporter（导出逻辑）、differ（版本对比）四个插槽，缺一个编译不过；第二，产物对象在运行时必须通过 ArtifactTypeRegistry 注册，直接 new 一个没注册的产物类型会抛异常；第三，产物内容必须用统一的 ArtifactContent 容器（format + data + encoding），不允许产物类型自定义 content 字段结构——自定义逻辑放在 content.data 内部，外层容器统一；第四，在脚手架 CLI 里固化模板——`npm run new-artifact-type mindmap` 自动生成符合规范的四个组件文件和注册代码，开发者照着填业务逻辑，想偏离规范反而更费劲。这样"遵循模型"就从 Code Review 时的口头要求，变成了"不遵循就跑不起来"的工程约束。

## 结构化回答

**30 秒电梯演讲：** 就像一个项目经理的办公桌——桌上不是堆满聊天记录，而是按项目分区摆放成果物：左边是设计稿、中间是Excel表格、右边是PPT演示，每样东西都能单独修改和重新生成。

**展开框架：**
1. **按任务分组** — 再按产物类型展示预览和下一步动作
2. **每个产物都带来源** — 输入、版本和可编辑入口
3. **支持局部重生成** — 而不是整单重跑

**收尾：** 如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
