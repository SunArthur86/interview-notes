# 模型版本管理系统实现要点？

## 一、为什么需要模型版本管理

```
不管理的混乱：
  - "线上用的是哪个版本的模型？" → 没人知道
  - 效果变差了 → 不知是模型问题还是数据问题
  - 想回滚 → 旧版权重已删
  - A/B测试 → 无法对比版本
  - 合规审计 → 无法追溯

管理的价值：
  ✓ 可追溯：知道每个时刻用的什么
  ✓ 可回滚：出问题快速恢复
  ✓ 可对比：A/B测试数据驱动
  ✓ 可审计：满足合规要求
  ✓ 可协作：团队协同迭代
```

## 三、版本管理系统设计

```python
class ModelVersionManager:
    """模型版本管理系统"""
    
    def register_version(self, model_config):
        """注册新版本"""
        version = ModelVersion(
            id=generate_id(),
            model_name=model_config["model"],
            prompt_version=model_config["prompt_version"],
            data_version=model_config["data_version"],
            config=model_config["params"],
            created_at=now(),
            created_by=current_user(),
            status="registered",
        )
        
        # 关联评估结果
        version.metrics = self.evaluate(version)
        
        self.registry.save(version)
        return version
    
    def deploy(self, version_id, strategy="canary"):
        """部署版本"""
        version = self.registry.get(version_id)
        
        if strategy == "canary":
            # 灰度发布：先5%流量
            self.router.set_traffic(version_id, percentage=5)
        elif strategy == "blue_green":
            # 蓝绿部署：新版本就绪后切流量
            self.prepare_green(version_id)
            self.switch_traffic(version_id)
        elif strategy == "full":
            # 全量
            self.router.set_traffic(version_id, percentage=100)
        
        # 记录上线
        self.audit_log.record("deploy", version_id, strategy)
```

## 四、回滚机制

```python
class RollbackManager:
    """快速回滚"""
    
    def rollback(self, reason):
        """紧急回滚到上一稳定版"""
        current = self.router.get_current_version()
        previous = self.registry.get_previous_stable(current)
        
        # 立即切流量
        self.router.set_traffic(previous.id, percentage=100)
        self.router.set_traffic(current.id, percentage=0)
        
        # 告警
        alert(f"已回滚: {current.id} → {previous.id}, 原因: {reason}")
        
        # 记录
        self.audit_log.record("rollback", current.id, reason)
    
    def auto_rollback(self):
        """自动回滚（监控触发）"""
        metrics = self.monitor.get_current_metrics()
        if metrics.error_rate > 10% or metrics.satisfaction < 3.0:
            self.rollback("自动回滚: 指标恶化")
```

## 五、A/B 测试

```python
class ModelABTest:
    """模型版本A/B测试"""
    
    def assign(self, user_id):
        """按用户分流"""
        # 同一用户始终同一版本（避免体验跳变）
        hash_val = hash(user_id) % 100
        if hash_val < 50:
            return self.version_a  # 50%流量
        return self.version_b      # 50%流量
    
    def compare(self):
        """对比两个版本的指标"""
        return {
            "version_a": self.get_metrics(self.version_a),
            "version_b": self.get_metrics(self.version_b),
            "significance": self.t_test(
                self.version_a, self.version_b
            ),
        }
    
    def promote_winner(self):
        """优胜版本全量"""
        comparison = self.compare()
        if comparison["significance"] < 0.05:  # 统计显著
            winner = comparison["better_version"]
            self.deploy(winner, strategy="full")
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：LLM 应用的版本你说是"模型+Prompt+数据+配置"的组合，那我只改了一个错别字（Prompt 里的），也要升版本走灰度吗？不累吗？**

要分变更的风险等级走不同流程。改错别字、调措辞这种 PATCH 级变更（不影响输出 schema 和行为逻辑），走轻量流程——注册新版本但在评测集上快速回归（5 分钟跑完），通过后直接替换不走灰度，因为风险极低。改输出格式、加新工具、换模型这种 MAJOR 变更（影响下游或行为），必须走完整灰度（5%→50%→100%）。所以不是每次都走重流程，是按变更分级——PATCH 走快通道，MAJOR 走灰度。版本号上区分（PATCH 升末位、MAJOR 升主位），让人一眼看出变更级别。

### 第二层：证据与定位

**Q：灰度发布时，5% 流量的新版本指标看起来正常，但放量到 50% 时突然指标崩了，为什么小流量时没发现？**

根因是"小流量的样本量不足以暴露长尾问题"。5% 流量一天可能只有几千条，某些长尾 case（特定输入、特定用户群体）在小流量里根本没出现，到 50% 才被触发。典型情况：新 Prompt 对某类输入（如含方言、含错别字）处理崩了，但小流量里这类输入少没暴露。定位方法：把 50% 流量时的失败 case 拉出来，看是否集中在某类之前没见过的输入。对策是灰度时不能只看聚合指标，要分群体看（按用户类型/输入特征分桶），任一分桶指标恶化就暂停放量。另外评测集要覆盖长尾，小流量阶段跑评测集能提前发现。

### 第三层：根因深挖

**Q：自动回滚你设了"错误率 > 10% 或满意度 < 3.0"的阈值，但满意度是滞后指标（用户用完才评分），等满意度掉下来时已经影响一批用户了，怎么更早发现？**

满意度确实是滞后的，所以要用"先行指标"做提前回滚。先行指标是那些在用户感知前就异常的技术信号：LLM 调用的错误率/延迟（技术侧先崩）、输出格式的解析失败率（schema 不匹配先暴露）、拒答率/转人工率飙升（行为异常先于用户不满）。我的自动回滚是多指标 OR 触发——任一先行指标超阈值（如格式解析失败率 > 5%）就回滚，不等滞后的满意度。满意度作为"确认指标"事后验证。这样能在用户大规模不满前就回滚，把影响面控制在灰度的小流量里。

**Q：那为什么不直接把回滚阈值调到极敏感（错误率 > 1% 就回滚），更早止损？**

因为会频繁误回滚，导致系统抖动。LLM 本身有偶发错误（模型偶发幻觉、网络抖动），正常情况错误率也有 1-3% 的基线波动。阈值设 1% 会在正常波动时误触发回滚，系统在"新版本-回滚-再上-再回滚"之间反复跳，用户体验比偶发错误更差。所以阈值要在"基线波动 + 标准差"之上——看历史基线错误率是 2%±0.5%，阈值设在 4%（基线 + 4 倍标准差），只有真正的异常（不是波动）才触发。这是统计上的控制图原理，避免误报。

### 第四层：方案权衡

**Q：A/B 测试你按 user_id hash 分流，但 LLM 输出有随机性（temperature），同一用户同一问题两次答案可能不同，这会污染 A/B 对比吗？**

会，所以要控制变量。A/B 测试对比的是"版本的系统效果"，要排除随机性干扰。做法是：每个版本对同一 query 采样多次（如 3 次）取均值或众数，减少单次随机波动。另外评测指标要选"对随机性不敏感"的——比如"答案是否包含正确事实"（多次稳定）而非"答案的具体措辞"（每次不同）。如果是离线评测集对比，固定随机种子让两个版本在相同 seed 下生成，消除随机性。线上 A/B 因为无法固定 seed，靠大样本量（每个版本 > 1 万次）让随机性在统计上平均掉。所以随机性可控，但要在评测设计时就考虑，不能事后补救。

**Q：为什么不直接全量切新版本，出了问题再回滚，比灰度快多了？**

因为"出问题"的影响面不同。全量切一旦出问题，100% 用户受影响，回滚虽然有但要分钟级生效（流量切换 + 缓存失效 + 在途请求处理），这几分钟里所有用户体验受损。灰度把影响面控制在 5%，即使出问题也只有 5% 用户受影响，且能快速回滚不影响其他 95%。对于 LLM 这种"行为不可 100% 预测"的系统，灰度是必要护栏。全量切只适合"紧急修复"（线上有 bug 必须立即换）或"经过充分灰度验证的版本"。

### 第五层：验证与沉淀

**Q：你怎么证明版本管理系统本身是可靠的，回滚真的能在事故时秒级生效？**

做"回滚演练"。定期（每月）在准生产环境模拟事故：人为让当前版本指标恶化（注入故障让错误率飙升），触发自动回滚，测量从"指标超阈值"到"流量切回旧版"的端到端耗时。通过标准是 < 1 分钟（含检测 + 决策 + 流量切换 + 缓存失效）。演练还能发现回滚链路的隐藏问题（如旧版本的镜像已被清理无法拉起、配置不兼容等），这些问题不演练到真事故才发现就晚了。有演练记录和耗时数据，回滚能力才是被验证的。

**Q：这套版本管理怎么沉淀成 MLOps 平台？**

沉淀成"模型/应用发布平台"，封装版本注册/评估/灰度/回滚/监控的全流程，业务方只需提交新版本（权重+Prompt+配置），平台自动跑评测、按策略灰度、监控指标、异常自动回滚。平台提供标准化的"版本 manifest"（声明模型/Prompt/数据/配置的版本绑定），防止人漏绑某个要素。配套发布审批流（高风险变更需 reviewer 批准）。这样版本管理是平台能力，不是每个团队自己写脚本，也保证全公司的发布流程一致、可审计。
