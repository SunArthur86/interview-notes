# 【阿里淘天AI二面】Agent服务在电商大促时，流量大且有流量尖刺，如何解决？

> 来源：阿里巴巴淘天淘工厂 AI应用开发 二面面经（小红书）

## 一、问题分析

```
正常日：QPS = 100          大促日：QPS = 10,000~50,000

Agent服务特点：
  - LLM推理是计算密集型 → 单GPU QPS有限（10-50）
  - 延迟高（1-5秒/请求）→ 积压风险大
  - 成本高（每次推理消耗token）→ 流量尖刺直接打爆预算

如果不做防护：
  瞬时尖刺 → GPU排队 → 延迟飙升 → 请求超时 → 雪崩
```

## 三、第一层：入口侧防护

### 1. 限流

```
令牌桶限流（推荐Agent场景）：

  ┌─────────┐    按固定速率生成令牌
  │ 令牌桶   │ ────→ 🪙 🪙 🪙 🪙
  │(容量=500)│
  └─────────┘
       │
  请求来了 → 拿一个令牌 → 通过
  没令牌了  → 拒绝(返回429) / 排队等待

  为什么选令牌桶而非漏桶？
  - 令牌桶允许突发流量（桶里有积累的令牌）
  - 漏桶是严格匀速，对LLM场景可能过于保守
  - 但Agent场景建议用令牌桶+较小容量，控制突发

  分布式限流：Redis + Lua实现
```


### 2. 负载均衡

```
                    Nginx / API Gateway
                   ┌───┬───┬───┬───┐
                   │   │   │   │   │
                   ▼   ▼   ▼   ▼   ▼
                 GPU1 GPU2 GPU3 GPU4 GPU5

  策略：
  - 加权轮询：按GPU算力分配（A100权重 > T4权重）
  - 最少连接数：优先分配给当前负载最低的GPU
  - 一致性哈希：相同用户路由到同一GPU（保持上下文缓存）

  Agent特殊考虑：
  - 上下文亲和性：同一会话尽量路由到同一实例（利用KV Cache）
  - 健康检查：GPU推理服务的心跳检测（不只是HTTP 200）
```

### 3. 削峰

```
同步 → 异步化改造：

  Before（同步）：
  用户 → API → Agent推理 → 返回
        ↑ 阻塞等待3-5秒，连接被占用

  After（异步削峰）：
  用户 → API → Kafka → 消费者按能力取消息 → Agent推理
    ↓                                    ↓
  立即返回"排队中"                     WebSocket/SSE推送结果

  削峰效果：
  - Kafka积压消息 → 消费者按GPU能力消费
  - 瞬时10000请求 → GPU只处理100/秒
  - 其余排队等待，不会压垮系统
```

## 五、第三层：资源侧优化

### 1. 热门问题缓存

```
多级缓存策略：

  用户问题 → Caffeine本地缓存（L1，1-5秒TTL）
           → Redis分布式缓存（L2，5-30分钟TTL）
           → Agent实时推理

  大促前预热：
  - 分析历史热门问题Top 1000
  - 提前生成答案并存入缓存
  - 大促时80%的请求命中缓存，只有20%走实时推理

  缓存Key设计：
  hash(规范化后的用户问题) → 缓存答案
  问题标准化：去空格、同义词替换、错别字纠正
```

### 2. 服务降级

```
Agent服务降级策略（从轻到重）：

  Level 1（轻度）：减少工具调用次数
  ┌────────────────────────────────────────┐
  │ 正常：Agent最多调用10次工具             │
  │ 降级：限制为3次                          │
  │ → 减少每次请求的计算量                   │
  └────────────────────────────────────────┘

  Level 2（中度）：关闭多轮对话
  ┌────────────────────────────────────────┐
  │ 正常：支持多轮对话（保持历史上下文）      │
  │ 降级：只支持单轮问答（每次独立）          │
  │ → 大幅减少token消耗和延迟               │
  └────────────────────────────────────────┘

  Level 3（重度）：回退到FAQ检索模式
  ┌────────────────────────────────────────┐
  │ 正常：完整Agent（RAG + LLM推理 + 工具） │
  │ 降级：纯FAQ向量检索 + 模板填充          │
  │ → 绕过LLM推理，延迟 < 200ms            │
  │ → 牺牲灵活性，保证基本可用              │
  └────────────────────────────────────────┘

  Level 4（极端）：只读模式
  ┌────────────────────────────────────────┐
  │ 所有写操作暂停                          │
  │ 只返回预置的"系统繁忙"页面               │
  │ → 完全保护后端                          │
  └────────────────────────────────────────┘

  降级触发条件：
  - GPU利用率 > 85% → Level 1
  - 请求队列积压 > 500 → Level 2
  - P99延迟 > 10s → Level 3
  - 系统接近崩溃 → Level 4

  恢复机制：
  - 压力下降后逐步恢复（Level 3→2→1→正常）
  - 避免震荡（恢复后有观察期才允许继续降级）
```

## 六、大促前准备

```
大促前容量规划：

  1. 预估QPS：
     去年同期QPS × 增长率 × 安全系数(1.5)

  2. 计算所需资源：
     所需GPU数 = 预估QPS / 单GPU QPS × 冗余系数

  3. 压力测试：
     模拟大促流量，验证各层防护是否有效

  4. 预热缓存：
     提前生成热门问题答案

  5. 降级预案：
     制定各级降级触发条件和恢复流程

  6. 监控告警：
     实时监控QPS、延迟、GPU利用率、缓存命中率
```

## 七、面试加分点

1. **提到LLM推理的特殊性**：与传统API不同，LLM推理是计算密集型，不能简单靠加机器解决，GPU调度是瓶颈
2. **提到量化/蒸馏降本**：大促期间可临时切换到量化模型(INT8/INT4)或蒸馏小模型，牺牲少量精度换取2-4倍吞吐
3. **提到请求合并**：多个相似请求可以做batch推理（vLLM的continuous batching），提高GPU利用率
4. **提到熔断机制**：类似Hystrix/Sentinel，当下游LLM服务异常时自动熔断，避免级联故障
5. **提到混沌工程**：大促前做混沌测试（模拟GPU故障、网络分区），验证降级策略是否有效

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Agent 服务大促扛流量，为什么不直接多备 GPU 算力，而要搞限流+降级+削峰三板斧？**

因为 LLM 推理是 GPU 计算密集型，扩容有瓶颈。传统 Web 服务加机器即可（CPU 弹性），但 LLM 推理受限于 GPU 显存（KV Cache 占用）和算力，GPU 供货周期长（提前备货数月）、成本极高（A100 单卡数十万）。盲目扩容会导致大促后资源闲置浪费。三板斧的动机是"在有限 GPU 下最大化有效吞吐"——限流挡住超量流量不压垮系统，削峰（MQ 异步化）把尖刺流量拉平，降级保核心牺牲非核心，是在资源硬约束下的最优策略。

### 第二层：证据与定位

**Q：大促时 Agent 服务的 P99 延迟从 2 秒飙到 15 秒，你怎么定位是 GPU 排队、还是某个工具调用慢、还是流量没削峰？**

看 Trace 的分段耗时。如果 LLM 推理环节的 queue time（请求在 GPU 队列等待时间）占大头，是 GPU 排队（算力不足）；如果某个 tool_call（如查订单的 RPC）耗时从 100ms 涨到 5s，是下游服务瓶颈；如果整体并发量没被 MQ 削峰（同步处理导致瞬时堆积），是削峰失效。关键指标是 GPU 利用率（>90% 说明算力瓶颈）和队列深度（持续上涨说明处理跟不上到达）。

### 第三层：根因深挖

**Q：GPU 利用率才 60% 但延迟已经炸了，根因是什么？不是算力不够？**

根因可能是 KV Cache 显存瓶颈或 batch 调度不优。GPU 利用率看的是计算单元，但 LLM 推理的瓶颈常在显存——每个并发请求的 KV Cache 占显存（如 7B 模型每请求 1-2GB），显存满了就无法接新请求，即使计算单元空闲。表现为"GPU 计算利用率不高但吞吐上不去"。治本是上 vLLM 的 PagedAttention（显存碎片化管理，提升并发数）或限制 max_concurrent_requests。另一个可能是 batch 没攒满（请求到达率低时单请求独占 GPU），用 continuous batching 解决。

**Q：那为什么不直接用更小的模型（如 1.5B 替代 7B）省显存提并发，省得搞 PagedAttention？**

小模型质量不够。客服/Agent 场景对推理质量有要求，1.5B 模型的工具调用准确率和复杂推理能力远不如 7B，错误率上升导致重试增多，反而更耗算力。PagedAttention 的价值是在不降质量的前提下（还是 7B）通过显存管理提升并发数 2-5 倍，是"免费"的吞吐提升。正确策略是：先用 PagedAttention + continuous batching 把 7B 的吞吐榨干，仍不够时再考虑量化（INT8 减半显存）或蒸馏小模型，质量降幅可控才换。

### 第四层：方案权衡

**Q：降级策略你设的是"关闭多轮对话/限制工具调用"，为什么降级这些而不是直接返回固定话术？**

分级降级，固定话术是最后兜底。关闭多轮对话（改为单轮 RAG 直答）能保住 80% 的简单问题秒回，复杂问题排队，体验降级可控；限制工具调用（如禁用耗时的查订单工具）减少单请求耗时，提升吞吐。这两者牺牲的是"复杂问题的解决能力"但不完全瘫痪服务。直接返回固定话术（"系统繁忙"）是全量放弃，用户完全无服务，是最后的熔断兜底。降级要渐进，每级保留尽可能多的服务能力。

**Q：为什么不直接对非 VIP 用户返回固定话术（全量降级），只保 VIP，省得搞复杂的分级？**

全量降级非 VIP 的体验损失太大，可能引发舆情。电商大促时非 VIP 是绝大多数（90%+），全降级等于服务下线。且"非 VIP"不等于"低价值"——一个大额订单可能来自非 VIP。正确做法是按"请求价值"而非"用户等级"分级：核心业务（下单/支付/退款）保 SLA，非核心（闲聊/推荐）降级。分级降级的核心是"识别哪些可降哪些不可降"，按业务关键度而非用户身份。

### 第五层：验证与沉淀

**Q：你怎么验证大促的限流降级配置真的扛得住，而不是上线了才发现配置错？**

大促前做全链路压测 + 混沌测试。压测用历史峰值流量的 1.5-2 倍打，验证限流阈值、MQ 削峰能力、GPU 弹性扩容是否生效，观察 P99 延迟和错误率。混沌测试主动注入故障（模拟 GPU 宕机、下游 RPC 超时、网络分区），验证降级开关是否正确触发、熔断是否生效。压测发现瓶颈就调配置，直到能扛住目标流量。大促时配实时看板（QPS、P99、GPU 利用率、限流拒绝数、降级触发数），SRE 盯盘随时手动干预。

**Q：大促的高可用经验怎么沉淀成常态化的容灾能力？**

把大促配置沉淀成"高可用预案模板"：限流阈值（按日常 QPS 的 N 倍）、降级开关清单（哪些可降）、熔断配置（阈值和恢复策略）、扩容预案（GPU 预留池）。日常做小流量演练（每周模拟一个故障，验证降级生效），避免大促时配置失效或人不会操作。沉淀"历次大促的容量规划数据""故障 case 库""降级决策树"，形成 SOP，新大促按模板执行而非临时拍脑袋。

## 结构化回答

**30 秒电梯演讲：** Agent服务面对大促流量尖刺，需要从三个层面解决：入口侧（限流+负载均衡+削峰）、请求侧（请求分级+优先处理高价值用户）、资源侧（热门缓存+服务降级）。核心理念是"保核心、降非核心、挡超量流量"。

**展开框架：**
1. **入口侧三板斧** — 限流(令牌桶/漏桶)、负载均衡(Nginx/LB)、削峰(消息队列异步化)
2. **请求侧分级** — VIP用户优先、核心业务优先、非核心降级
3. **资源侧优化** — 热门问题缓存(Caffeine+Redis)、服务降级(关闭多轮对话/限制工具调用)

**收尾：** 您想深入聊：令牌桶和漏桶限流的区别？Agent服务适合哪种？
