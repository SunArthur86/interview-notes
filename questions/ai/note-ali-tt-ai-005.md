# 【阿里淘天AI二面】Agent服务在电商大促时，流量大且有流量尖刺，如何解决？

> 来源：阿里巴巴淘天淘工厂 AI应用开发 二面面经（小红书）

## 一、问题分析

```
正常日：QPS = 100          大促日：QPS = 10,000~50,000

Agent服务特点：
  - LLM推理是计算密集型 → 单GPU QPS有限（10-50）
  - 延迟高（1-5秒/请求）→ 积压风险大
  - 成本高（每次推理消耗token）→ 流量尖刺直接打爆预算

如果不做防护：
  瞬时尖刺 → GPU排队 → 延迟飙升 → 请求超时 → 雪崩
```

## 三、第一层：入口侧防护

### 1. 限流

```
令牌桶限流（推荐Agent场景）：

  ┌─────────┐    按固定速率生成令牌
  │ 令牌桶   │ ────→ 🪙 🪙 🪙 🪙
  │(容量=500)│
  └─────────┘
       │
  请求来了 → 拿一个令牌 → 通过
  没令牌了  → 拒绝(返回429) / 排队等待

  为什么选令牌桶而非漏桶？
  - 令牌桶允许突发流量（桶里有积累的令牌）
  - 漏桶是严格匀速，对LLM场景可能过于保守
  - 但Agent场景建议用令牌桶+较小容量，控制突发

  分布式限流：Redis + Lua实现
```

```java
// Guava RateLimiter（单机限流）
RateLimiter limiter = RateLimiter.create(100); // 100 QPS
if (!limiter.tryAcquire(1, 500, TimeUnit.MILLISECONDS)) {
    return Response.tooManyRequests("系统繁忙，请稍后重试");
}
// 继续处理Agent请求
```

### 2. 负载均衡

```
                    Nginx / API Gateway
                   ┌───┬───┬───┬───┐
                   │   │   │   │   │
                   ▼   ▼   ▼   ▼   ▼
                 GPU1 GPU2 GPU3 GPU4 GPU5

  策略：
  - 加权轮询：按GPU算力分配（A100权重 > T4权重）
  - 最少连接数：优先分配给当前负载最低的GPU
  - 一致性哈希：相同用户路由到同一GPU（保持上下文缓存）

  Agent特殊考虑：
  - 上下文亲和性：同一会话尽量路由到同一实例（利用KV Cache）
  - 健康检查：GPU推理服务的心跳检测（不只是HTTP 200）
```

### 3. 削峰

```
同步 → 异步化改造：

  Before（同步）：
  用户 → API → Agent推理 → 返回
        ↑ 阻塞等待3-5秒，连接被占用

  After（异步削峰）：
  用户 → API → Kafka → 消费者按能力取消息 → Agent推理
    ↓                                    ↓
  立即返回"排队中"                     WebSocket/SSE推送结果

  削峰效果：
  - Kafka积压消息 → 消费者按GPU能力消费
  - 瞬时10000请求 → GPU只处理100/秒
  - 其余排队等待，不会压垮系统
```

## 四、第二层：请求侧分级

```
请求分级体系：

┌─────────────────────────────────────────┐
│ P0级：VIP用户 / 核心业务                  │
│ → 独立资源池，快速通道                    │
│ → 示例：黑卡会员咨询、高价值订单问题       │
├─────────────────────────────────────────┤
│ P1级：普通用户 / 核心业务                  │
│ → 正常处理队列                            │
│ → 示例：普通用户订单查询                   │
├─────────────────────────────────────────┤
│ P2级：普通用户 / 非核心业务                │
│ → 低优先级队列，可能延迟                   │
│ → 示例：闲聊、一般咨询                     │
├─────────────────────────────────────────┤
│ P3级：异常/恶意请求                        │
│ → 直接限流/拒绝                           │
│ → 示例：刷接口、超高频请求                 │
└─────────────────────────────────────────┘
```

## 五、第三层：资源侧优化

### 1. 热门问题缓存

```
多级缓存策略：

  用户问题 → Caffeine本地缓存（L1，1-5秒TTL）
           → Redis分布式缓存（L2，5-30分钟TTL）
           → Agent实时推理

  大促前预热：
  - 分析历史热门问题Top 1000
  - 提前生成答案并存入缓存
  - 大促时80%的请求命中缓存，只有20%走实时推理

  缓存Key设计：
  hash(规范化后的用户问题) → 缓存答案
  问题标准化：去空格、同义词替换、错别字纠正
```

### 2. 服务降级

```
Agent服务降级策略（从轻到重）：

  Level 1（轻度）：减少工具调用次数
  ┌────────────────────────────────────────┐
  │ 正常：Agent最多调用10次工具             │
  │ 降级：限制为3次                          │
  │ → 减少每次请求的计算量                   │
  └────────────────────────────────────────┘

  Level 2（中度）：关闭多轮对话
  ┌────────────────────────────────────────┐
  │ 正常：支持多轮对话（保持历史上下文）      │
  │ 降级：只支持单轮问答（每次独立）          │
  │ → 大幅减少token消耗和延迟               │
  └────────────────────────────────────────┘

  Level 3（重度）：回退到FAQ检索模式
  ┌────────────────────────────────────────┐
  │ 正常：完整Agent（RAG + LLM推理 + 工具） │
  │ 降级：纯FAQ向量检索 + 模板填充          │
  │ → 绕过LLM推理，延迟 < 200ms            │
  │ → 牺牲灵活性，保证基本可用              │
  └────────────────────────────────────────┘

  Level 4（极端）：只读模式
  ┌────────────────────────────────────────┐
  │ 所有写操作暂停                          │
  │ 只返回预置的"系统繁忙"页面               │
  │ → 完全保护后端                          │
  └────────────────────────────────────────┘

  降级触发条件：
  - GPU利用率 > 85% → Level 1
  - 请求队列积压 > 500 → Level 2
  - P99延迟 > 10s → Level 3
  - 系统接近崩溃 → Level 4

  恢复机制：
  - 压力下降后逐步恢复（Level 3→2→1→正常）
  - 避免震荡（恢复后有观察期才允许继续降级）
```

## 六、大促前准备

```
大促前容量规划：

  1. 预估QPS：
     去年同期QPS × 增长率 × 安全系数(1.5)

  2. 计算所需资源：
     所需GPU数 = 预估QPS / 单GPU QPS × 冗余系数

  3. 压力测试：
     模拟大促流量，验证各层防护是否有效

  4. 预热缓存：
     提前生成热门问题答案

  5. 降级预案：
     制定各级降级触发条件和恢复流程

  6. 监控告警：
     实时监控QPS、延迟、GPU利用率、缓存命中率
```

## 七、面试加分点

1. **提到LLM推理的特殊性**：与传统API不同，LLM推理是计算密集型，不能简单靠加机器解决，GPU调度是瓶颈
2. **提到量化/蒸馏降本**：大促期间可临时切换到量化模型(INT8/INT4)或蒸馏小模型，牺牲少量精度换取2-4倍吞吐
3. **提到请求合并**：多个相似请求可以做batch推理（vLLM的continuous batching），提高GPU利用率
4. **提到熔断机制**：类似Hystrix/Sentinel，当下游LLM服务异常时自动熔断，避免级联故障
5. **提到混沌工程**：大促前做混沌测试（模拟GPU故障、网络分区），验证降级策略是否有效
