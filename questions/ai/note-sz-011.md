# 【神州专车面经】模型离线 AUC 高但线上效果差，查哪些问题？

## 一、五类根因排查框架

```
[1] 特征不一致（最常见，占 60%）
    │  线上特征 ≠ 离线特征
    ▼
[2] 数据分布不一致
    │  训练历史数据 vs 线上当下数据，分布漂移
    ▼
[3] 样本不一致
    │  训练正负采样 vs 线上真实分布
    ▼
[4] 评估目标不一致
    │  AUC 是排序指标 vs 业务看转化/收入
    ▼
[5] 工程链路问题
       特征延迟/模型版本错/缓存脏
```

## 二、特征不一致（最常见，重点查）

### 离线特征 vs 线上特征的差异源

| 差异类型 | 例子 |
|---------|------|
| **线上缺特征** | 离线用了"用户最近30天订单数"，线上实时算不出来 |
| **计算逻辑不同** | 离线用 SQL 算"7日活跃"，线上用 Flink 算，口径不一致 |
| **实时性不同** | 离线是 T+1 全量，线上是实时增量，同一特征值不同 |
| **特征工程不同** | 离线做了归一化/分箱，线上忘了做（或反过来） |
| **默认值不同** | 离线空值填 0，线上填 -1 |

### 怎么排查
```python
# 1. 线上打日志：把请求时的特征向量落表
# 2. 离线复算：用同一批请求的原始数据重新算特征
# 3. 对比：线上特征 vs 离线复算特征，找差异大的字段
SELECT request_id, feature_name, 
       online_value, offline_value,
       ABS(online_value - offline_value) AS diff
FROM feature_consistency_log
WHERE ABS(online_value - offline_value) > 0.01
ORDER BY diff DESC;
```

## 三、数据分布不一致（分布漂移）

```
训练数据：2025年6月的订单（历史）
线上数据：2026年6月的订单（当下）

可能漂移：
  - 季节性：6月 vs 12月，出行模式不同
  - 业务变化：新开了城市/新上了车型/改了定价
  - 外部环境：竞品补贴/政策变化/疫情
```

### 怎么排查
```python
# PSI（Population Stability Index）衡量分布稳定性
# PSI < 0.1 稳定，0.1-0.25 轻微漂移，> 0.25 显著漂移
def psi(train_dist, online_dist):
    return sum((online - train) * ln(online / train))

# 对每个特征算 PSI，找漂移大的
```

### 解法
- 定期重训（周/月级）
- 用近期数据做 fine-tune
- 在线学习（FTRL）实时适应

## 四、样本不一致

```
训练样本：正负采样后 1:10（负样本下采样）
线上样本：真实分布 1:1000（负样本远多）

问题：
  - 模型学到的概率是采样后的，不是真实概率
  - 校准没做（Platt Scaling / Isotonic Regression）
```

### 解法
- 采样后做概率校准
- 或用负样本权重（`sample_weight`）而非下采样

## 五、评估目标不一致

```
AUC：排序指标（把正样本排在负样本前面的概率）
业务指标：转化率/GMV/点击率/留存

不一致场景：
  - AUC 高但排序头部不准（业务只看 Top-K，AUC 看全局）
  - AUC 高但校准差（概率值偏离真实点击率）
  - AUC 高但业务目标不同（AUC 优化排序，业务要 GMV）
```

### 解法
- 看业务相关指标（Top-K Precision / NDCG / GAUC）
- 看校准指标（LogLoss / Brier Score）
- 用业务目标做评估（线上 A/B）

## 六、工程链路问题

| 问题 | 现象 | 怎么查 |
|------|------|--------|
| **特征延迟** | 实时特征算慢了，用旧值或默认值 | 看特征计算延迟 P99 |
| **模型版本错** | 线上加载的是旧模型 | 看模型版本日志 |
| **缓存脏** | 特征缓存没更新，用了过期值 | 看缓存 TTL 和更新逻辑 |
| **请求路由错** | A/B 实验分桶错，流量没到新模型 | 看实验配置和分桶日志 |

## 七、排查流程（结构化）

```
[1] 先查工程链路（最快排除）
    │  模型版本对吗？特征延迟正常吗？A/B 分桶对吗？
    ▼
[2] 查特征一致性
    │  线上特征落表，离线复算对比，找差异
    ▼
[3] 查分布漂移
    │  算特征 PSI，看是否漂移
    ▼
[4] 查样本一致性
    │  采样比例对吗？概率校准了吗？
    ▼
[5] 查评估目标
    │  看业务相关指标（不只 AUC）
```

## 八、加分点

- 说出 **特征一致性是最常见原因**，占 60% 以上
- 说出 **GAUC（Group AUC）**：按用户分组算 AUC，比全局 AUC 更贴合推荐/排序场景
- 说出 **离线在线一致性管道**：每天跑一致性校验任务，特征 diff 超阈值告警

## 九、雷区

- ❌ "AUC 高就上线" → 必须配线上 A/B + 一致性监控
- ❌ 只查模型不查特征 → 80% 问题在特征
- ❌ 忽略工程链路 → 明明模型没问题，是链路 bug

## 十、扩展

- **特征监控平台**：监控每个特征的分布（均值/方差/缺失率）+ PSI，漂移告警
- **影子模式（Shadow Mode）**：新模型线上不真正服务，只打日志预测，对比老模型效果
- **Multi-task Learning**：同时优化 AUC 和业务目标，避免目标不一致

## 记忆要点

- 口诀法：排查工程链路、特征一致性、分布漂移、样本采样、评估目标五大维度
- 因果句：因为线上线下代码隔离，所以特征不一致最常见，占比超60%需重点打日志排查
- 对比句：离线历史数据易受时间或业务变动产生分布漂移，线上实时数据则会暴露采样偏差
- 因果句：因为AUC仅评估全局排序能力，所以必须辅以GAUC和真实业务转化指标做评估


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：离线 AUC 高但线上效果差，这个问题为什么普遍存在？是 AUC 这个指标错了还是离线评估方法错了？**

两者都不是完全错，是"离线评估和线上目标的错配"。AUC 衡量的是"排序能力"（正样本得分是否高于负样本），但线上业务目标可能是"转化率"或"收入"，两者不是线性相关。再加上训练数据是历史分布、线上是实时分布，存在分布漂移。所以离线高 AUC 是必要不充分条件——离线差肯定不能上，离线好不保证线上好。要补齐"离线在线一致性"的验证。

### 第二层：证据与定位

**Q：离线 AUC 0.85，线上转化率反而比旧模型低，怎么定位是哪一类不一致？**

按五类排查：1) 特征不一致——线上推理用的特征和离线训练用的特征是否一致（字段、计算逻辑、时间戳），用线上日志的 feature 和离线特征对账；2) 数据分布不一致——线上流量分布和训练集是否一致（如训练集是历史数据、线上是最新用户行为）；3) 样本不一致——训练集的正负样本采样比例和线上真实比例是否一致；4) 评估目标不一致——AUC vs 业务目标（如点击率 vs 转化率）；5) 工程链路——线上特征延迟、模型版本错配、缓存脏数据。每类用不同的诊断方法。

### 第三层：根因深挖

**Q：最常见的根因是"特征不一致"，为什么线上特征和离线特征会对不上？**

根因是"特征工程的两个环境割裂"。离线训练用 Spark/Hive 批量算特征（如"用户过去 30 天订单数"），线上推理用实时服务算特征（如 Flink + Redis）。两套实现：1) 计算逻辑差异——离线用 SQL、线上用代码，边界条件（如时区、去重逻辑）处理不同；2) 数据源差异——离线用数仓历史表、线上用实时流，延迟和完整性不同；3) 时间戳口径——离线特征是 T+1 快照、线上是实时累计，定义不一致。根因是"同一特征名，两套实现，没有契约约束"。

**Q：那为什么不直接把特征计算逻辑写成一份代码，离线线上共用，避免不一致？**

这正是特征平台（Feature Store）的思路——定义一次特征，离线训练和线上推理都从同一个 Feature Store 取。但实现有挑战：1) 离线要批量回填历史特征（point-in-time correctness，保证训练时只用当时可见的特征），线上要低延迟实时读取，技术栈不同；2) 特征版本管理——特征定义变更要同时影响离线和线上，需要强治理。所以 Feature Store 是"理想"，落地需要工程投入。短期解法是"特征对账"——定期抽样线上特征和离线特征做 diff，发现不一致立即修复。

### 第四层：方案权衡

**Q：AUC 是排序指标，业务看转化率，为什么不直接用业务指标（转化率）做离线评估？**

可以但有两个限制：1) 离线数据没有"真实转化率"——离线只有历史日志（用户是否点击/转化），但无法知道"如果用新模型推荐，转化率是多少"（反事实问题）；2) 转化率受多种因素影响（推荐质量、产品体验、用户状态），不只是模型。所以离线用 AUC（排序能力）做筛选，线上 A/B 测真实转化率（业务效果）。两者配合：AUC 过滤掉差的模型，A/B 验证好的模型。纯用转化率离线评估做不到（反事实），纯用 AUC 不足以反映业务效果。

**Q：那为什么不直接相信线上 A/B，跳过离线评估？**

成本和时间。线上 A/B 要：1) 流量分流（影响真实用户体验，有风险）；2) 周期长（统计学显著要跑 1-2 周）；3) 成本高（每次实验占流量、占资源）。离线评估是"低成本快速筛选"，10 个候选模型离线筛到 2 个，再线上 A/B。如果跳过离线直接 A/B，10 个模型每个跑 2 周，要 20 周，且大量差的模型浪费线上流量。所以离线 + 线上是"漏斗式筛选"，离线负责效率，线上负责准确性。

### 第五层：验证与沉淀

**Q：怎么建立"离线在线一致性"的持续监控，避免上线后才发现效果差？**

三道防线：1) 特征一致性监控——线上推理时记录特征值，离线重新算该时刻的特征值，做 diff，不一致率 > 1% 告警；2) 预测分布监控——线上预测分数的分布（如正样本预测概率的直方图）和离线评估集的分布对比，分布漂移（KL 散度 > 阈值）告警；3) 离线在线相关性——积累多个模型的"离线 AUC vs 线上业务指标"数据点，算相关性，如果相关性低，说明离线评估体系需要改进。沉淀为模型上线门禁：特征一致性、预测分布稳定性、离线 AUC 三项达标才能上线。
