# Rerank之后的截断策略是怎么设计的？为什么选这个K值？

## 三种截断策略对比

```
Rerank得分排序后：

文档A: 0.95 ████████████████████ 
文档B: 0.88 ██████████████████ 
文档C: 0.82 ████████████████ 
文档D: 0.71 ██████████████ 
文档E: 0.45 █████████        ← 分数骤降点
文档F: 0.32 ██████ 
文档G: 0.15 ███ 

策略1 - 固定K=5:          策略2 - 阈值0.5:         策略3 - 混合(K=10,τ=0.5):
[A,B,C,D,E]              [A,B,C,D]               [A,B,C,D]
↑ 引入了低分E            ↑ 精准但可能漏召回        ✅ 最优平衡
```

## 策略一：得分阈值法

```python
def threshold_truncate(reranked: list, threshold: float = 0.5) -> list:
    """低于阈值的文档全部丢弃"""
    return [d for d in reranked if d['score'] >= threshold]

# 优势：自适应——好时候多保留，差时候少保留
# 劣势：极端情况下可能返回0条（无召回兜底）
```

## 策略二：固定K值法

```python
def fixed_k_truncate(reranked: list, k: int = 5) -> list:
    """总是取Top-K"""
    return reranked[:k]

# 优势：简单可控，上下文Token消耗可预测
# 劣势：低质文档也会被强行保留
```

## 策略三：混合策略（推荐）

```python
def hybrid_truncate(
    reranked: list,
    max_k: int = 10,
    min_score: float = 0.5,
    min_return: int = 1,
) -> list:
    """同时设阈值和K上限，保证至少返回min_return条"""
    # 先按阈值过滤
    filtered = [d for d in reranked if d['score'] >= min_score]
    # 再按K上限截断
    result = filtered[:max_k]
    # 兜底：如果过滤后为空，至少返回得分最高的min_return条
    if len(result) < min_return:
        result = reranked[:min_return]
    return result
```

## K值设定的四个维度

| 维度 | 考量 | 建议K值 |
|------|------|--------|
| **上下文窗口** | 每条约500Token，K=10→5000Token | K≤上下文的20% |
| **回答质量** | K=5-10时RAG准确率最高 | K=5~10 |
| **成本控制** | 每多1条→增加~500Token成本 | 按预算反推 |
| **业务场景** | 客服(精确) vs 研究(广泛) | 客服K=3~5, 研究K=8~10 |

## 得分骤降检测（动态截断）

```python
def elbow_truncate(reranked: list) -> list:
    """检测得分曲线的"拐点"（elbow），在骤降处截断"""
    scores = [d['score'] for d in reranked]
    # 计算相邻文档的得分差
    diffs = [scores[i] - scores[i+1] for i in range(len(scores)-1)]
    # 找到最大跌幅的位置
    if not diffs:
        return reranked
    elbow_idx = diffs.index(max(diffs))
    # 在拐点处截断（保留拐点之前的文档）
    return reranked[:elbow_idx + 1]

# 示例：
# scores = [0.95, 0.88, 0.82, 0.71, 0.45, 0.32, 0.15]
# diffs   = [0.07, 0.06, 0.11, 0.26, 0.13, 0.17]
# max diff at index 3 (0.26) → 保留前4条
```

## 面试加分点

1. **Lost in the Middle**：截断后的文档顺序也有讲究——把最相关的放最前面和最后面（首因/近因效应），避免中间位置被忽略
2. **零召回兜底**：Rerank后全部低分时，应返回兜底回答（"暂无相关信息"）而不是强行用低质文档生成
3. **AB测试**：截断策略没有万能公式，必须用线上AB测试对比不同K值/阈值的准确率、延迟、成本
4. **动态调整**：根据Query难度动态调整K——简单事实题K=3，复杂推理题K=8

## 记忆要点

- 三大策略：固定K截断(可控易引入噪声)、得分阈值法(精准易零召回)、混合策略(双限兜底)
- 混合策略最优：同时设置K上限和最低分阈值，并保证至少返回1条避免业务中断
- K值四考量：受限于上下文Token(建议占20%)，RAG质量最佳为5~10条，兼顾业务与成本
- 高级截断：计算相邻文档得分差，在得分骤降拐点动态截断以规避尾部噪声

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Rerank 后截断你推荐"混合策略"（K 上限 + 分数阈值），为什么不直接用分数阈值（低于 0.7 丢弃），更精准？**

因为纯分数阈值有"零召回"风险。不同 query 的 Rerank 分数分布不同——明确 query 可能 top-5 都 >0.9（该返回 5 条），模糊 query 可能 top-1 只有 0.5（阈值 0.7 会全部丢弃，返回空）。业务上"返回空"比"返回低质量"更糟糕（用户觉得系统坏了）。混合策略是"阈值过滤质量 + K 上限控量 + 至少 1 条兜底"，既保证质量（阈值拦噪声）又保证可用（至少返回）。动机是"质量与可用性的双保险"。

### 第二层：证据与定位

**Q：你用混合策略截断后，某些 query 返回了 1 条（兜底），但用户反馈"这 1 条不对"。怎么定位是 Reranker 给错分还是截断逻辑问题？**

看那条返回的文档的 Rerank 分数。如果分数 >0.7（质量达标）但内容不对，是 Reranker 误判（给了高分但语义不相关），要优化 Reranker 模型或训练数据；如果分数 <0.7 但因为"至少返回 1 条"兜底才出现，是 query 本身难（没有高相关文档），要补知识库或返回"未找到相关信息"。关键区分"Reranker 打分错"和"知识库里就没有正确答案"——前者是排序问题，后者是召回/知识覆盖问题，解法不同。

### 第三层：根因深挖

**Q：你设的 Rerank 分数阈值是 0.7，但不同 Reranker 模型（bge-reranker vs Cohere）的分数尺度不同，0.7 在不同模型上含义不同。根因怎么处理？**

根因是 Reranker 的输出分数没有统一语义（不像概率有明确含义）。bge-reranker 输出的是 logit（范围不定），Cohere 输出的是 0-1 的相关性分数，两者的 0.7 含义完全不同。治本有三：一是用每个 Reranker 各自的分数分布（在验证集上算 mean/std）做 z-score 归一化，用归一化后的分数设阈值（如 >均值+1std）；二是用排名而非分数（如取 top-K，不看绝对分数），回避尺度问题；三是用校准集（标注的相关/不相关对）把分数映射成概率，按概率设阈值（如 P(相关)>0.5）。换 Reranker 时必须重新校准阈值，不能沿用。

**Q：那为什么不直接用固定 K 截断（如 top-5），不看分数，省得处理分数尺度问题？**

固定 K 有"噪声固定"问题。如果某 query 只有 2 条高相关文档，固定 top-5 会多塞 3 条噪声，稀释信号甚至误导模型（lost in the middle）。如果某 query 有 15 条高相关文档，固定 top-5 会漏掉 10 条有用信息。固定 K 假设"每个 query 的相关文档数相同"，实际不同 query 的相关文档数差异大。分数阈值能自适应（相关多就多返回、相关少就少返回），但需处理尺度。混合策略（K 上限 + 阈值 + 至少 1）兼具自适应和可控性，是两者的折中。

### 第四层：方案权衡

**Q：高级截断你提到"按相邻文档得分差在拐点动态截断"，这比固定 K/阈值好在哪？为什么不默认用它？**

拐点截断自适应每个 query 的分数分布。Rerank 分数通常呈"前几条高、后面骤降"的模式，拐点（分数差最大的位置）是"相关→不相关"的自然分界。比固定 K（不看分布）和固定阈值（不看排名）更贴合数据。但不默认用的原因是：拐点检测对噪声敏感（某条偶发低分可能被误判为拐点），需要平滑处理（如移动窗口看趋势）；且分数平缓下降的 query（没有明显拐点）会退化为"全返回"或"全丢弃"。工程上是"拐点检测 + K/阈值兜底"——优先用拐点，检测不到拐点时回退到固定 K。

**Q：为什么不直接让 LLM 在生成时自己判断"哪些 context 相关用哪些"，省得截断？**

LLM 对长 context 的利用效率不高（lost in the middle 研究）。即使给了 20 条 context，LLM 倾向于关注开头和结尾，中间的被忽略。且更多 context 意味着更多 token 成本和更高幻觉风险（LLM 可能引用不相关 context 的片段）。截断是"在喂给 LLM 前预先筛选"，保证 LLM 看到的都是高相关内容，提升利用率和准确率。把筛选权完全交给 LLM 是"偷懒"，工程上应该在检索层把好关，让 LLM 专注生成而非筛选。

### 第五层：验证与沉淀

**Q：你怎么证明混合策略截断比固定 K 或纯阈值更好，而不是碰巧在某些 query 上有效？**

构建评测集，每个 query 标注"正确相关文档集合"，对比三种截断策略的 nDCG@10 和端到端 RAG 准确率。混合策略应在 nDCG（检索质量）和准确率（生成质量）上都不低于另两者，且在"极端 query"（相关文档特别多或特别少）上显著更优。按 query 类型分层看，证明混合策略的"自适应"价值。同时监控线上"返回 1 条兜底"的触发率，过高（如 >10%）说明阈值太严，要调低。

**Q：截断策略怎么沉淀成 RAG 框架的可配置能力？**

封装成 `truncate(reranked_docs, strategy, params)` 函数：strategy 支持 fixed_k / score_threshold / hybrid / elbow_detection，params（K、阈值、最低返回数）可配。沉淀"各业务的截断参数基线"（如客服 K=5 阈值 0.7、法律检索 K=10 阈值 0.5）、"拐点检测算法实现""Reranker 换型时的阈值重校准流程"。配套截断效果看板（平均返回文档数、兜底触发率、nDCG），新业务按基线起步再调优。

## 结构化回答

**30 秒电梯演讲：** Rerank后需要截断保留高质量文档，三种策略：得分阈值（低于0.7丢弃）、固定K值（取Top-5）、混合策略（阈值+K上限）。K值需平衡上下文窗口、回答质量和成本。

**展开框架：**
1. **得分阈值法** — 低于固定分数的不保留，灵活但需调参
2. **固定K值法** — 总是取Top-N，简单但可能引入低质文档
3. **混合策略** — 同时设阈值和K上限，最稳健
