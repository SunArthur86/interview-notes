# 能不能直接给 Agent 30 个以上的工具？有什么风险？

## 一、给 30+ 工具的四大风险

```
┌──────────────────────────────────────────────┐
│  风险1：选择准确率下降                          │
│    30个工具，LLM"选择困难症"                    │
│    相似工具（query_order vs query_log）易混淆   │
│    研究：工具>10个，准确率开始显著下降          │
├──────────────────────────────────────────────┤
│  风险2：Token成本激增                          │
│    每个工具描述~100token                       │
│    30个工具=3000token/轮                       │
│    每轮多花3000token，长对话成本翻倍            │
├──────────────────────────────────────────────┤
│  风险3：上下文稀释                             │
│    工具描述占满上下文，挤占用户对话空间          │
│    重要信息被工具描述"淹没"                      │
├──────────────────────────────────────────────┤
│  风险4：误调用高风险工具                        │
│    工具多时LLM可能误选delete而非query           │
│    高危工具（删除/支付）被误触发                 │
└──────────────────────────────────────────────┘
```

## 二、解决方案：工具检索（Tool RAG）

```python
class ToolRetriever:
    """按需检索工具，只给LLM相关的子集"""
    
    def __init__(self, all_tools):
        self.tools = all_tools
        # 预计算所有工具描述的embedding
        self.tool_embeddings = {
            t.name: embed(t.description) for t in all_tools
        }
    
    def select(self, user_query, top_k=5):
        """根据用户查询检索最相关的工具"""
        query_emb = embed(user_query)
        
        # 计算相似度
        scores = {
            name: cosine_sim(query_emb, emb)
            for name, emb in self.tool_embeddings.items()
        }
        
        # 返回top-k最相关的工具
        relevant = sorted(scores, key=scores.get, reverse=True)[:top_k]
        return [self.tools[name] for name in relevant]

# 使用：从30个工具中选出5个最相关的
relevant_tools = tool_retriever.select("我要查订单物流", top_k=5)
# 只把这5个工具的描述给LLM
```

## 三、解决方案：工具分层分组

```python
# 按业务域分组，先选组再选工具
TOOL_CATEGORIES = {
    "订单管理": {
        "description": "订单查询、修改、退款相关",
        "tools": ["query_order", "cancel_order", "refund", "track_logistics"]
    },
    "用户管理": {
        "description": "用户信息、地址、偏好相关",
        "tools": ["get_profile", "update_address", "set_preference"]
    },
    "支付": {
        "description": "支付、充值、提现相关",
        "tools": ["pay", "recharge", "withdraw"]
    },
    # ... 每组4-6个工具
}

class HierarchicalToolSelector:
    def select(self, query):
        # 第一层：选类别（4个类别描述，token少）
        category = self.llm.select_category(query, TOOL_CATEGORIES)
        
        # 第二层：在选中类别内选工具（4-6个，准确率高）
        tools = TOOL_CATEGORIES[category]["tools"]
        selected = self.llm.select_tool(query, tools)
        
        return selected
```

## 四、解决方案：动态加载

```python
class DynamicToolManager:
    """根据对话上下文动态加载工具"""
    
    def get_tools_for_context(self, session_state):
        """基于会话状态判断需要哪些工具"""
        
        # 用户在讨论订单 → 加载订单相关工具
        if session_state.current_topic == "order":
            return self.load_tools(["query_order", "refund", ...])
        
        # 用户在闲聊 → 不需要工具
        if session_state.current_topic == "chat":
            return []
        
        # 默认：加载最常用的基础工具
        return self.load_tools(["search", "get_time"])
```

├──────────────────────────────────────────────┤
│                                                │

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：给 Agent 30+ 工具可以但有风险，那为什么一定要给这么多工具？10 个以内不就够了吗？**

取决于业务复杂度。简单场景（如天气查询）3-5 个工具够；复杂场景（如企业助手要查订单/改地址/退款/查物流/查积分/兑换/咨询/转人工等）轻松 20-30 个工具。强行限制 10 个会导致功能不全（用户的需求覆盖不了），用户体验差。所以"工具多"是业务驱动的现实需求，不能为"选得准"而砍功能。正确的方向是"接受工具多，优化选择机制"（工具检索/分层管理/路由），让 LLM 在多工具下仍能选准，而非限制工具数。

### 第二层：证据与定位

**Q：30 个工具时 LLM 选错率飙升，你怎么量化"工具数 vs 选错率"的关系？**

做工具数消融实验。固定任务集，分别用 5/10/20/30 个工具跑 Agent，统计工具选择准确率。典型结果：5 个工具准确率 95%、10 个 90%、20 个 75%、30 个 60%。准确率随工具数下降的原因：1）context 膨胀——30 个工具描述塞进 prompt 占几千 token，LLM 注意力分散；2）相似工具混淆——工具多必有功能相似的，LLM 分不清。量化这个曲线后，确定"准确率可接受的工具数阈值"（如要求 >85% 则不超过 10 个常驻）。超过阈值的工具走"按需检索"（不全塞 prompt，按意图检索 top-5 塞进去）。

### 第三层：根因深挖

**Q：工具检索（按意图检索 top-K 工具，而非全塞 prompt）能解决多工具问题，但检索本身可能漏掉正确的工具，怎么保证不漏？**

三层保障。1）召回率优先——检索时先广撒网（top-20 候选），保证正确的工具在候选里（召回率 >95%）；再用 rerank 精选 top-5 给 LLM；2）多路召回——不只 embedding 检索（语义匹配），加关键词检索（精确匹配工具名）和规则路由（如"涉及订单的优先查 order 类工具"），多路并集不漏；3）兜底机制——如果检索的 top-5 都不匹配（LLM 判断"这些工具都不合适"），回退到"全量工具让 LLM 再选"（虽慢但不漏）或"转人工"。漏工具比选错更严重（漏了直接做不了），所以召回率要保，宁可多召回（rerank 精选）也不能漏。

**Q：相似工具混淆（如"查订单"和"查订单详情"）是 30+ 工具的典型问题，怎么从工具设计上避免？**

工具设计要"功能正交、边界清晰"。1）功能正交——每个工具有明确独特的功能，不重叠（如"查订单列表"和"查单个订单详情"合并成一个"查订单"工具，用参数控制返回列表还是详情）；2）命名+描述区分——相似工具的命名和描述要强调差异（如"search_orders（按条件搜索订单列表）"vs"get_order_detail（按订单号查单个订单详情）"，描述明确"列表 vs 单个"）；3）合并优于拆分——能合并的相似工具尽量合并（用参数区分），减少工具数。如不是 30 个独立工具，而是 15 个工具（部分有参数控制子功能），选错率显著降。

### 第四层：方案权衡

**Q：多工具问题的解法有"工具检索/分层管理/工具路由"，三者什么区别？怎么选？**

三者层次不同。1）工具检索（RAG 选工具）——按意图从工具库检索 top-K 塞 prompt，动态、按需；适合"工具多但每次只用几个"的场景。2）分层管理——工具按类别分层（核心层常驻 prompt，扩展层按需加载），静态+动态结合；适合"有明确核心工具+长尾工具"的场景。3）工具路由——先用分类器判断"这个问题归哪类工具域"（如订单域/物流域），再在域内选，分层路由；适合"工具可清晰分域"的场景。选型：工具数中等（10-20）用分层管理（核心+扩展）；工具多且无明确分层（20+）用工具检索（动态召回）；工具可清晰分域（如电商的多业务线）用工具路由。实务常组合：路由分域+域内检索+核心常驻。

**Q：既然工具多了选不准，为什么不把这些工具的"功能"内化到模型权重（微调让模型记住），而非要靠 prompt 描述？**

微调内化工具有前述的"动态性/泛化性"问题（工具变了要重训、新工具不会用）。但对"超高频核心工具"（如某业务的 3-5 个核心工具），微调是值得的——SFT 让模型对这些工具的调用准确率从 80% 提到 95%+，且 prompt 里不用塞这 5 个工具的描述（省 token）。所以最优是"核心工具微调内化 + 长尾工具 prompt 描述 + 按需检索"三层：核心（高频）走权重（准且省 token），长尾（低频）走 prompt（灵活），检索负责"何时用哪层"。这样既解决多工具的准确率，又保持灵活性。

### 第五层：验证与沉淀

**Q：你怎么证明"工具检索+分层管理"在 30 工具场景下确实比"全塞 prompt"选得准？**

AB 测试。固定 30 工具 + 标注任务集（每个 query 标注正确工具），对比：1）全塞 prompt（30 个描述都给 LLM）；2）工具检索（召回 top-5）；3）分层管理（5 核心+25 扩展按需）。指标：工具选择准确率、token 成本、延迟。典型结果：全塞准确率 60%（token 多）、检索 85%（token 少）、分层 88%（token 中）。检索和分层在准确率和成本上都优于全塞，证明"动态选择"优于"全量展示"。还要测"召回率"——检索是否漏掉正确工具（漏了即使精选也选不对），召回率应 >95%。

**Q：多工具管理机制怎么沉淀成框架能力？**

封装成 ToolManager 组件：1）工具注册——工具注册时分类（核心/扩展）+ 标签（域/功能）+ 完整描述（功能+场景+参数示例）；2）分层加载——核心工具常驻 prompt，扩展工具按需检索；3）工具检索——embedding 召回 + rerank，返回 top-K（开发者配 K）；4）工具路由——按意图分类路由到工具域，域内选；5）准确率监控——统计选择准确率/召回率/误调率，异常告警。开发者注册工具+标分类，框架自动管理加载/检索/路由。这套写入团队 Agent 框架 SOP，新 Agent 支持任意数量工具且选得准。
