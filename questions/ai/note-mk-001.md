# 【月之暗面面经】AI-Native 桌面产品为什么不能只照搬 Web 聊天页？

## 一、核心差异：Web 聊天 vs AI-Native 桌面

传统 Web 聊天页的设计假设是：**用户输入文字 → AI 返回文字 → 用户继续输入**。这是一个线性的、单线程的、纯文本的交互循环。而 AI-Native 桌面产品的设计假设完全不同：

| 维度 | Web 聊天页 | AI-Native 桌面产品 |
|------|-----------|-------------------|
| 输入来源 | 文本输入框 | 文件拖拽 + 目录选择 + 截图 + 命令行 + 自然语言 |
| 输出产物 | 文本消息流 | 站点、表格、PPT、代码文件、配置文件 |
| 任务生命周期 | 一次性请求-响应 | 长任务队列 + 多任务并发 + 产物版本管理 |
| 上下文管理 | 当前会话 | 跨会话的任务上下文 + 文件索引 + 产物引用 |
| 系统能力 | 受限于浏览器沙箱 | 文件系统 + 进程管理 + 系统通知 + 全局快捷键 |

## 二、为什么不能照搬？四个根本原因

### 1. 桌面端有真实文件和目录

Web 聊天页无法访问用户的本地文件系统（除非通过 `<input type="file">` 上传）。而桌面 AI 产品可以：

```
用户选择一个目录
  → AI 自动索引目录下所有文件
  → AI 理解项目结构
  → AI 生成分析报告/修改文件/创建新文件
  → 产物直接写入用户指定的本地路径
```

这个流程在 Web 上根本无法实现。**桌面 AI 的核心价值之一就是让 AI 直接操作本地文件**。

### 2. 任务和产物是第一公民

Web 聊天页的信息组织单位是「消息」。但在桌面 AI 产品中，信息组织单位应该是：

```
任务（Task）
  ├── 输入引用（InputRefs: 文件、网页、截图）
  ├── 执行计划（Plan: 步骤分解）
  ├── 产物列表（Outputs: 站点、表格、PPT）
  ├── 版本历史（Versions: 每次重生成的快照）
  └── 状态机（Status: queued → running → review → done）
```

**用户关心的不是"AI 说了什么"，而是"AI 给我产出了什么、产物在哪、能不能继续编辑"**。

### 3. 长任务和多任务并行

Web 聊天是同步的——发一条消息等一个回复。桌面 AI 产品经常需要：

- 同时执行多个 Agent 任务（生成站点 + 生成报告 + 生成图表）
- 任务可能运行几分钟到几十分钟
- 用户需要在任务运行时切换到其他工作
- 任务完成后需要系统通知 + 一键回到任务现场

```typescript
// 桌面端任务状态机
type TaskStatus = 'queued' | 'running' | 'paused' | 'review' | 'done' | 'failed';

interface Task {
  id: string;
  status: TaskStatus;
  inputRefs: FileRef[];      // 引用的本地文件/目录
  outputs: Artifact[];       // 产物对象（站点/PPT/表格）
  createdAt: number;
  updatedAt: number;
  traceId: string;           // 链路追踪 ID
}
```

### 4. 桌面端交互节奏不同

Web 聊天的节奏是「即时响应」（< 3 秒）。桌面 AI 产品的节奏是分层的：

```
即时层（< 1s）：输入框联想、命令补全、拖拽响应
快速层（1-5s）：单文件分析、格式转换
长任务层（5s-10min）：站点生成、批量文件处理、报告撰写
后台层（> 10min）：大型索引构建、模型微调预热
```

**前端需要为每个层级设计不同的 UI 反馈模式**——不能用一个转圈 loading 覆盖所有情况。

## 三、正确的 UI 组织方式

```
┌──────────────────────────────────────────────────────┐
│  顶部导航：全局任务状态 + 通知中心 + 设置              │
├──────────┬───────────────────────────┬───────────────┤
│          │                           │               │
│  左侧栏   │     主工作区               │   右侧面板    │
│  任务列表  │  ┌─────────────────────┐ │   产物预览    │
│  +分类    │  │ 任务输入区            │ │   +Diff视图   │
│  +搜索    │  │ (文件/命令/自然语言)   │ │   +版本历史   │
│          │  ├─────────────────────┤ │               │
│          │  │ 执行计划 & 进度       │ │               │
│          │  ├─────────────────────┤ │               │
│          │  │ 产物面板              │ │               │
│          │  │ (站点/表格/PPT预览)    │ │               │
│          │  └─────────────────────┘ │               │
├──────────┴───────────────────────────┴───────────────┤
│  底部状态栏：当前任务/文件索引状态/云端连接状态         │
└──────────────────────────────────────────────────────┘
```

### 关键设计原则

1. **任务优先于对话**：主界面以任务卡片/任务列表为组织单位，而非消息流
2. **产物可落地**：每个 AI 产物都能导出到本地文件、复制到剪贴板或发布到指定目标
3. **长任务有回看**：任务执行过程被完整记录，支持随时回看和局部重跑
4. **多入口并存**：命令式输入（`/generate-site`）和自然语言输入（"帮我做一个个人主页"）共存
5. **权限透明**：文件读写、目录访问、系统调用都有明确的权限提示和撤销路径

## 四、常见坑

- **把桌面 AI 产品做成 Web 对话页**：没有把文件、窗口和任务系统接起来，用户体验和网页版完全一样
- **只考一次问答**：不考长任务、产物落地和回看路径，面试时讲不出深度
- **忽略本地文件权限**：没有设计权限提示和撤销路径，用户不敢用

## 五、面试加分点

面试中主动补充以下工程约束，能显著拉开与普通前端候选人的差距：

| 约束维度 | 具体内容 |
|---------|---------|
| 权限管理 | 文件读写授权、目录访问范围、系统调用审批 |
| 灰度发布 | Agent 能力分批上线、A/B 测试不同产物模板 |
| 埋点设计 | 任务完成率、产物采纳率、重跑率、上下文补充率 |
| 成本控制 | 本地索引 vs 云端推理的取舍、Token 预算管理 |
| 链路追踪 | traceId 贯穿本地→云端→产物全链路 |
| 失败恢复 | 长任务中断后的断点续传、部分产物保留 |
| 多人协作 | 多窗口任务共享、产物版本冲突处理 |
| 效果验证 | 产物质量评分、用户反馈闭环、Bad case 收集 |

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：桌面 AI 产品为什么不直接复用 Web 聊天页的交互范式，非要把"任务"和"产物"提为一等公民？**

因为 Web 聊天页的核心交互单位是「消息」，用户的心智模型是"我问一句、AI 答一句"，产物就是一段可读文本。但桌面端用户的心智模型是"我交给 AI 一个工作"——他要的是落到本地目录的 PPT、可继续编辑的站点、可导出的表格，而不是聊天记录里的一串 Markdown。所以信息组织单位必须从 message 升级为 Task：一个 Task 挂着 inputRefs（引用的本地文件）、outputs（Artifact 列表）、versions（每次重生成的快照）、status（queued/running/review/done 的状态机）。用户关心的不是"AI 说了什么"，而是"产物在哪、能不能改、能不能接着跑"。

### 第二层：证据与定位

**Q：你凭什么判断桌面端"长任务并发"是个真需求，而不是 Demo 阶段被过度设计？**

证据来自桌面端与 Web 端的任务时长分布差异：Web 聊天 95% 的请求在 3 秒内返回（同步即可），但桌面端生成站点、批量改文件、写报告的任务耗时通常在 30 秒到 10 分钟之间，而且用户在等待时不会盯着一个转圈——他会切窗口去做别的事。如果在 Demo 阶段不做长任务队列，商用后必然遇到三个场景翻车：任务跑了一半用户关了主窗口、两个任务并发写同一份文件、任务失败但用户不知道在哪一步失败。验证方法是把任务耗时分桶（<1s / 1-5s / 5s-10min / >10min），如果 5s-10min 这桶占比超过 20%，长任务队列就是 P0 需求而不是过度设计。

### 第三层：根因深挖

**Q：为什么"交互节奏分层"必须做，而不能用一个统一的 Loading 组件覆盖所有耗时？**

因为不同耗时的任务，用户的心理预期和容忍阈值完全不同：<1s 的即时层（命令补全、拖拽响应）用户不会等任何 Loading，出现转圈反而会被认为卡顿；1-5s 的快速层需要 Skeleton 占位 + 流式输出；5s-10min 的长任务层必须有进度条 + 阶段提示（"索引文件中"/"生成站点中"/"保存产物中"）+ 可后台运行 + 系统通知召回；>10min 的后台层则要支持断点续跑。根因是"用户的等待焦虑和任务耗时是指数关系"——3 秒无反馈用户开始焦虑，30 秒无反馈用户认为崩溃，3 分钟无反馈用户直接关闭。统一的 Loading 无法表达"任务还活着、卡在哪一步、还要多久"，必然导致用户频繁中断和重跑。

**Q：那如果产品初期任务都很短（都在 10 秒内），为什么不先省掉分层，等任务变长了再补？**

因为"等任务变长再补"是一个典型的架构债陷阱。交互节奏分层不是 UI 层的样式调整，而是从 Task 状态机、事件总线、通知系统到底层任务队列的整体设计。如果初期把 Task 状态机写死成同步的 running→done 两态，后期要改成 queued/running/paused/review/failed 五态时，所有依赖 Task 状态的组件（产物面板、回看页、通知中心）都要重写。更关键的是，长任务回看的 traceId 链路追踪必须从第一天就埋好——后期补埋点会导致历史任务无法回放，商用后的第一周就会因为"任务失败了无法定位"被投诉。所以初期任务短也要预留分层接口，分层 UI 可以后做，但状态机和链路追踪必须前置。

### 第四层：方案权衡

**Q：任务并发时，多个 Agent 同时写本地文件，你用编辑锁（悲观）还是版本合并（乐观）？怎么选？**

桌面端产物文件通常是结构化文档（站点 HTML、PPT JSON、表格），多人/多任务同时编辑的概率低，但同一用户多窗口并发编辑同一产物的概率高（窗口 A 在生成站点，窗口 B 在编辑该站点的旧版本）。我选悲观编辑锁 + 超时自动释放：任务开始时对产物文件加锁（写一个 .lock 文件，记录持锁 taskId 和时间戳），其他任务/窗口尝试写入时检测到锁就排队或提示用户"该产物正在被任务 A 使用"。选悲观而非乐观的根因是：桌面端文件回滚成本极高（没有 Git），一旦乐观合并出错，用户很难恢复；而悲观锁的代价只是"用户多等几秒"，远小于数据丢失。乐观合并只适用于纯文本场景（如多人协同文档），且需要强大的 CRDT 算法支撑，成本和收益不匹配。

**Q：那如果用户强烈抱怨"文件被锁了改不了"，为什么不直接去掉编辑锁？**

因为去掉编辑锁的代价是"静默数据覆盖"——窗口 A 的长任务跑了 5 分钟生成的新站点，被窗口 B 用户的手动编辑覆盖，用户根本不知道发生了什么。这个问题的严重性远高于"文件被锁"。正确做法不是去掉锁，而是优化锁的体验：把锁的粒度从"文件级"细化到"产物字段级"（站点的内容和样式分开加锁）、把锁的提示从"报错弹窗"改成"侧边栏轻提示 + 一键接管"、把锁的超时从固定值改成"基于任务预估耗时动态调整"。还可以引入"影子副本"机制——用户编辑时先写到副本，任务完成后用三路合并（base/用户副本/AI 产物）自动合并，冲突字段才提示用户选择。这样既保留了并发安全，又降低了用户被锁的体感。

### 第五层：验证与沉淀

**Q：你怎么证明桌面端的"任务/产物优先于消息"这个设计判断是对的，而不是主观偏好？**

用三个北极星指标验证：(1) 任务完成率 = 进入 done 状态的 Task 数 / 发起的 Task 数，如果以消息为单位统计则无法区分"用户发了 10 条消息其实只是完成了一个任务"和"用户完成了 10 个任务"；(2) 产物采纳率 = 被用户导出/保存/继续编辑的 Artifact 数 / 生成的 Artifact 总数，这个指标只有在产物是一等公民（有独立 id、版本、状态）时才能精确统计；(3) 任务回看率 = 用户点击历史任务进入回看页的次数 / 总任务数，如果这个率 >15% 说明用户真的把任务当作可追溯的工作单元而非一次性对话。如果上线后这三个指标都显著优于"消息流"对照组，就证明设计判断成立。

**Q：怎么让团队在新接 Agent 能力时，自觉遵循"任务/产物优先"的设计范式，而不是又退回消息流写法？**

把范式沉淀成工程约束而不是文档说教。第一，在产物对象模型层做强校验——Artifact 必须挂载在某个 Task 下，没有 taskId 的产物在类型系统层就编译不过；第二，做一个脚手架 CLI（如 `npm run new-agent`），自动生成符合范式的 Agent 模板（包含 Task 状态机、产物注册、traceId 埋点），开发者想偷懒写消息流反而更费劲；第三，在 Code Review 检查清单里固化三条硬规则（产物必须有 version 字段、长任务必须有 traceId、写文件必须走权限层），违反就阻塞合并；第四，每周复盘 Bad case 时用"产物采纳率"倒查——如果某个 Agent 的产物采纳率低于 60%，大概率是它偷偷退化成了消息流写法，直接定位到代码整改。
