# 【月之暗面面经】多模态输入同时有网页、文档和本地截图时，桌面端该怎样做上下文管理？

## 一、问题本质：上下文爆炸

一个典型的桌面 AI 任务可能有这样的输入：

```
输入素材：
  ├── 网页URL × 3（每个约 5000 字 = 15000 字）
  ├── PDF文档 × 2（每个约 20 页 = 约 40000 字）
  ├── 本地截图 × 5（每张需OCR + 描述 = 约 5000 字）
  └── 历史对话 × 20条（约 8000 字）
  
总计：约 68000 字 ≈ 90000 tokens
```

如果把这些全部内联到 prompt 中，不仅消耗大量 token，还会导致 AI 注意力分散、关键信息被淹没。

## 三、上下文对象模型

```typescript
// 来源对象——原始素材
interface SourceObject {
  id: string;
  type: 'webpage' | 'document' | 'screenshot' | 'file' | 'task-history';
  
  // 原始信息
  uri: string;              // URL / 文件路径 / 数据引用
  rawContent?: string;      // 原始内容（可能很大）
  
  // 提取结果
  extracted: {
    summary: string;        // 一句话摘要
    keyPoints: string[];    // 关键要点
    fullText?: string;      // 完整文本（按需展开）
    metadata: {
      title?: string;
      author?: string;
      createdAt?: number;
      wordCount?: number;
      language?: string;
    };
  };
  
  // 消耗状态
  consumed: boolean;        // 是否已被当前任务引用
}

// 上下文槽位——当前任务激活的上下文
interface ContextSlot {
  id: string;
  sourceId: string;         // 引用的来源对象
  mode: 'summary' | 'keypoints' | 'fulltext' | 'reference-only';
  priority: 'high' | 'medium' | 'low';
  note?: string;            // 用户备注："这个网页是竞品定价页"
}
```

- 默认只展示摘要（省 token），用户点击「全文」才展开

## 五、多模态来源的提取策略

```typescript
class ContextExtractor {
  // 网页：提取正文 + 关键段落
  async extractWebpage(url: string): Promise<ExtractResult> {
    const html = await fetch(url);
    const article = readabiliy(html);  // 正文提取
    
    return {
      summary: await llm.summarize(article.text, { maxWords: 100 }),
      keyPoints: await llm.extract(article.text, { type: 'key-points' }),
      fullText: article.text,
      metadata: { title: article.title, wordCount: article.length }
    };
  }
  
  // 文档：按页解析 + 关键页提取
  async extractDocument(filePath: string): Promise<ExtractResult> {
    const pages = await pdfParser(filePath);
    
    return {
      summary: await llm.summarize(pages.map(p => p.text).join('\n')),
      keyPoints: await llm.extract(pages, { type: 'key-pages' }),
      fullText: pages,  // 分页存储，按需加载
      metadata: { pageCount: pages.length }
    };
  }
  
  // 截图：OCR + 视觉理解
  async extractScreenshot(imagePath: string): Promise<ExtractResult> {
    const ocrText = await ocr(imagePath);
    const visualDesc = await visionModel.describe(imagePath);
    
    return {
      summary: visualDesc,
      keyPoints: ocrText.split('\n').filter(Boolean),
      fullText: ocrText,
      metadata: { type: 'image', ocrConfidence: 0.92 }
    };
  }
}
```

## 六、上下文切换与版本保留

当用户切换到另一个任务时，当前上下文不应丢失：

```typescript
// 任务级上下文快照
interface ContextSnapshot {
  taskId: string;
  slots: ContextSlot[];      // 激活的槽位
  timestamp: number;
  tokenCount: number;
}

// 切换任务时自动保存当前上下文快照
function switchTask(newTaskId: string) {
  // 1. 保存当前任务的上下文快照
  saveSnapshot(currentTask);
  
  // 2. 加载目标任务的上下文快照
  const snapshot = loadSnapshot(newTaskId);
  
  // 3. 检查素材是否过期
  for (const slot of snapshot.slots) {
    if (isStale(slot.sourceId)) {
      notify(`素材"${slot.sourceId}"可能已更新，是否重新加载？`);
    }
  }
}
```

## 七、常见坑

- **把所有素材全量塞进prompt**：token爆炸，AI注意力分散，成本飙升
- **来源对象不可见**：用户不知道AI用了哪些素材，无法调试结果不好的原因
- **上下文切换丢失**：切到另一个任务回来，之前的素材引用全没了
- **大对象重复拷贝**：同一个20MB的PDF在多个任务中各存一份，浪费内存

## 记忆要点

- 痛点：多模态原始素材全塞入会引发Token爆炸和注意力分散
- 三层管理架构：底层Source原始素材库，中层按需提取摘要，顶层动态引用槽
- 按需加载机制：Prompt只注入引用层和摘要，完整内容随用随取不重复拷贝

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么多模态素材不能直接全量塞进 Prompt，非要搞三层架构（Source/Extraction/Reference）？AI 的上下文窗口不是已经到 1M token 了吗？**

因为"窗口能装下"不等于"AI 能用好"。全量塞入有三个问题：(1) Token 成本爆炸——3 个网页 + 2 个 PDF + 5 张截图 + 20 条历史对话约 9 万 token，每次调用都付费，一个复杂任务跑 10 轮就是 90 万 token 的成本；(2) 注意力稀释——Transformer 的注意力机制在长上下文下存在"lost in the middle"现象，关键信息放在中间位置时召回率下降 30% 以上，AI 反而答得更差；(3) 不可调试——用户不知道 AI 用了哪些素材、忽略了哪些，结果不好时无法定位是哪个素材出了问题。三层架构的价值是把"存什么（Source）""提取什么（Extraction）""当前用什么（Reference）"解耦：Source 层存完整原始内容（本地索引不重复拷贝），Extraction 层做摘要和关键点提取，Reference 层只把当前任务需要的槽位注入 Prompt。这样 Token 消耗从 9 万降到 1.2 万，AI 注意力集中在高优先级素材上。

### 第二层：证据与定位

**Q：你怎么定位"AI 回答质量差"是上下文管理问题，而不是模型能力问题？**

用"上下文消融实验"定位。当 AI 回答质量差时，做对照测试：(1) 只把当前激活的 Reference 槽位（摘要模式）发给 AI——这是默认模式；(2) 把某个高优先级素材切换为 fulltext 全文模式重新发——如果质量提升，说明摘要丢了关键信息；(3) 移除某个低优先级素材重新发——如果质量不变或提升，说明该素材是噪音。如果三种调整后质量都不变，才是模型能力问题。另一个诊断信号是看"素材消费日志"——记录 AI 生成时实际引用了哪些素材（通过 attention 权重或 prompt 中出现的 sourceId 追踪），如果 AI 的回答完全没引用任何激活素材，说明上下文注入逻辑有 bug 或素材优先级设置失效。这样能精确定位是"素材没进来""素材进来了但被忽略"还是"模型本身不行"。

### 第三层：根因深挖

**Q：为什么用"引用 + 摘要"而不是"引用 + 全文按需加载"？摘要必然丢信息，根因是什么？**

根因是"Token 预算"和"注意力效率"的双重约束。即使按需加载全文，把一个 4 万字的 PDF 全文注入 Prompt，单这一项就消耗 5 万 token，占满 128K 窗口的 40%，AI 会把注意力过度分配给这份文档而忽略其他素材——这是注意力机制的固有缺陷（softmax 会让长文档"霸占"注意力权重）。摘要的价值不是"省 token"（虽然也省），而是"压缩信噪比"——把 4 万字里有价值的 500 字提取出来，让 AI 的注意力集中在这 500 字上，回答质量反而比给全文更高。关键信息丢失的风险通过"分层摘要"缓解：第一层是一句话摘要（10 token），第二层是关键要点列表（200 token），第三层是关键段落（2000 token），用户按需展开深度。AI 生成时默认用第二层，只有在明确需要细节时（如用户问"第 3 页的具体数据是什么"）才动态升级到第三层。

**Q：那如果摘要阶段 LLM 本身就提取错了关键信息（幻觉），为什么不直接给全文让 AI 自己判断？**

因为"摘要幻觉"是可检测可重试的，而"全文注意力稀释"是系统性且不可检测的。摘要提取错了，用户能在上下文面板里看到摘要内容（"Free/Plus/Business/Enterprise 四档"），一眼就知道对不对，错了可以手动编辑摘要或切换全文模式重试——这是一个可见的、可控的失败。但全文注入导致注意力稀释是隐性的——AI 看了全文但"选择性地忽略了关键段落"，用户根本不知道 AI 跳过了什么，无法调试。所以解法不是"放弃摘要给全文"，而是"降低摘要幻觉概率"：(1) 摘要时附带原文片段作为 grounding（让 LLM 基于原文提取而非自由生成）；(2) 关键数字和专有名词用正则强制保留（摘要必须包含原文中的价格、百分比、产品名）；(3) 对高风险素材（如合同条款）自动切换全文模式，不对摘要准确度做赌注。根因是"可控的小概率错误"优于"不可控的系统性衰减"。

### 第四层：方案权衡

**Q：上下文槽位（ContextSlot）你用固定数量（如最多 12 个槽位）还是动态无上限？怎么选？**

选固定上限（如 12 个），原因有三：(1) Token 预算约束——每个槽位即使只注入摘要也消耗约 500-1000 token，12 个槽位约 1.2 万 token，留出足够窗口给 AI 生成，无上限会导致 Token 不可控；(2) 注意力效率——研究表明 LLM 能有效关注的信息源数量有上限（约 7±2 个），超过这个数量注意力开始分散，固定 12 个槽位里激活的通常不超过 7 个（其余是未激活的备用素材），保证 AI 注意力集中；(3) 用户认知负荷——上下文面板里 12 个槽位用户还能管理（调整优先级、切换摘要/全文），无上限会让面板变成"素材垃圾桶"，用户反而懒得整理。固定上限的代价是"素材超限时要做取舍"，解法是自动降级机制：当用户尝试激活第 13 个素材时，系统建议把最低优先级的素材移出槽位（而非直接拒绝），让用户做有意识的取舍而非无意识的堆积。

**Q：那如果用户坚持要同时引用 20 个素材（如做一份综合 20 份报告的分析），为什么不放开上限？**

因为"放开上限"解决不了"AI 处理不了 20 份报告"这个根本问题——即使窗口装得下 20 份摘要（约 2 万 token），AI 的注意力也无法在 20 份素材间有效分配，生成质量会显著下降。正确的解法是"分层聚合"而非"放开上限"：第一阶段让 AI 对每 5 份报告分别生成子摘要（4 组 × 5 份），第二阶段把 4 个子摘要聚合为一个总摘要注入主任务的上下文槽位。这样用户引用了 20 份报告，但主任务的实际槽位只有 1 个"综合摘要"，Token 消耗和注意力集中度都可控。实现上用一个"素材组（SourceGroup）"概念——用户可以把多个素材打包成一个组，系统自动生成组的聚合摘要，组作为一个槽位参与主任务。这比放开上限更符合 AI 的认知特性，也保持了 Token 预算可控。

### 第五层：验证与沉淀

**Q：你怎么证明"三层架构 + 按需加载"比"全量塞入"更优，用什么指标验证？**

用三组对偶指标验证：(1) 成本指标——单任务平均 Token 消耗，三层架构应从 9 万降到 1.5 万以下，降幅 80%+；(2) 质量指标——用一组标准测试题（有标准答案的任务），对比全量注入 vs 三层架构的回答准确率，关键看"关键信息召回率"（AI 回答中包含原文关键事实的比例），三层架构应持平或略高（因为注意力更集中）；(3) 用户调试效率——当 AI 回答不好时，用户定位"哪个素材出了问题"的平均耗时，三层架构因为有可见的上下文面板和素材消费日志，应从"无法定位"降到 30 秒内。A/B 测试时还要监控一个反向指标："用户手动切换全文模式的频率"——如果超过 20% 的素材被用户手动切全文，说明摘要质量不达标，需要优化提取策略而非放弃三层架构。

**Q：怎么让团队在新接入素材类型（如视频、音频、3D 模型）时，自觉走三层架构（Source/Extraction/Reference），而不是直接把原始数据塞进 Prompt？**

把三层架构做成"素材接入 SDK"的强制约束。第一，定义 SourceObject 接口，新素材类型必须实现 extract() 方法返回 summary + keyPoints + fullText 三层数据，没有 extract 方法的素材无法注册到 Source 层；第二，Prompt 构造器（PromptBuilder）只接受 ContextSlot 引用，不接受原始内容——即使开发者想直接塞全文，PromptBuilder 也会强制把它包装成一个 fulltext 模式的 ContextSlot，走统一的 Token 预算计算和优先级排序；第三，在素材注册时做 Token 预算校验——如果新素材的 fullText 模式超过单素材 Token 上限（如 2 万），强制要求实现分页加载或分层摘要，否则注册失败；第四，定期审计——扫描所有任务的 Prompt 构造日志，统计各素材类型的平均 Token 消耗，对异常高消耗的素材（如某个素材平均占 50% Token）触发优化工单。这样三层架构就从"设计规范"变成了"SDK 层的强制约束"，不遵循就无法接入。

## 结构化回答

**30 秒电梯演讲：** 多模态上下文管理的核心是把来源对象、提取结果和当前任务引用拆开管理——用户能清楚看到哪些素材正在被当前任务消费，大对象优先存引用和摘要而非重复拷贝，上下文切换时保留历史版本方便回退。

2. **用户能看到** — 用户能看到哪些素材正在被当前任务消费
3. **大对象优先** — 大对象优先存引用和摘要，不重复拷贝
