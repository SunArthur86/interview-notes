# 【神州专车面经】数仓怎么分层？为什么这么设计？

## 一、经典四层架构

```
┌──────────────────────────────────────────────────┐
│  ADS（应用层）                                     │
│  直接面向 BI/报表/API                              │
│  例：司机日效率榜、城市订单看板                     │
├──────────────────────────────────────────────────┤
│  DWS（汇总层 / 轻度汇总）                          │
│  按主题 + 时间粒度聚合                              │
│  例：dws_driver_daily（司机日汇总）                │
├──────────────────────────────────────────────────┤
│  DWD（明细层）                                     │
│  清洗 + 维度退化成明细宽表（一次颗粒度）            │
│  例：dwd_order_di（订单明细，退化司机/乘客信息）    │
├──────────────────────────────────────────────────┤
│  ODS（贴源层）                                     │{
  "name": "finalize_structure", "params": {"a": "b"}
}                                                 │
│  原始数据原样存，不加工业逻辑                       │
│  例：ods_order（和业务库 binlog 一致）              │
├──────────────────────────────────────────────────┤
│  业务库 / 日志 / 第三方数据                         │
└──────────────────────────────────────────────────┘
```

可选加 **DIM（维度层）**：存维度表（司机维度、城市维度），供 DWD 退化用。

## 二、各层职责详解

### ODS（贴源层）
- **原样存**业务库 binlog / 日志 / 第三方数据
- 不加任何工业逻辑（最多做格式转换）
- 保留历史快照（按天分区）
- 作用：**数据备份 + 贴源供下游加工**

### DWD（明细层）
- 清洗：去重、空值处理、异常值过滤
- **维度退化**：把维度信息 join 进事实表（如订单表退化司机姓名/城市）
- 颗粒度 = **业务最细颗粒度**（一条订单一行）
- 作用：**统一明细口径，被多个汇总层复用**

### DWS（汇总层）
- 按**主题**（司机/订单/营收）+ **时间粒度**（日/周/月）聚合
- 例：`dws_driver_daily`（司机日汇总：日单量/日流水/在线时长）
- 作用：**预计算加速，避免每次从明细算**

### ADS（应用层）
- 直接面向 BI/报表/API
- 高度定制化，按需求建
- 作用：**应用直接查，不用碰底层**

## 三、为什么分层（核心价值）

| 价值 | 说明 |
|------|------|
| **解耦** | 每层职责单一，改一层不影响其他层 |
| **复用** | DWD 被多个 DWS 复用，DWS 被多个 ADS 复用 |
| **可追溯** | 出问题逐层查：ADS 异常 → 查 DWS → 查 DWD → 查 ODS |
| **性能** | 汇总层预计算，应用层查询不用从明细实时算 |
| **口径统一** | 同一指标在 DWD/DWS 统一口径，避免"各算各的" |

## 四、出行场景的分层实例

```
ODS:  ods_order（订单原始，含 status 变更链）
       ods_driver（司机原始）
       ods_gps（GPS 轨迹原始）

DWD:  dwd_order_di（订单明细，退化司机/城市/时段维度）
       dwd_driver_online_di（司机上线明细）

DWS:  dws_driver_daily（司机日汇总：单量/流水/时长/效率分）
       dws_city_hourly（城市小时汇总：订单量/供需比）

ADS:  ads_driver_efficiency_rank（司机效率榜）
       ads_city_hotzone（城市热点区域）
```

## 五、加分点

- 说出 **维度退化**：把维度表的信息 join 进事实表，避免每次查询都 join（星型/雪花模型的选择）
- 说出 **DWD 的颗粒度原则**：一次颗粒度（一条业务事件一行），不要在 DWD 就聚合
- 说出 **DWS 的主题划分**：按业务实体（司机/订单/车辆）而非按部门划分

## 六、雷区

- ❌ "ODS 直接给应用查" → 性能差且口径乱
- ❌ "DWD 就聚合" → 失去明细复用价值
- ❌ "不分层，一张大宽表搞定" → 改需求就推倒重来

## 七、扩展

- **Kimball 维度建模 vs Inmon 范式建模**：维度建模（星型/雪花）适合数仓分层，范式建模适合 3NF 企业仓
- **实时数仓（Lambda/Kappa）**：实时层用 Flink 写 DWD/DWS，离线层用 Spark 批，两者口径需对齐
- **Data Mesh**：去中心化数仓，每个业务域自治，适合超大型组织

## 记忆要点

- 经典四层：ODS贴原样存、DWD清洗退维保明细、DWS主题聚合预计算、ADS高度定制出应用
- DWD必须保持业务最细粒度，DWS按主题和时间聚合，禁跨层加粗计算
- 分层核心价值：解耦上下游、共用复用防重复计算、逐层可追溯、统一口径


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：数仓为什么要分层？直接一张大宽表查询不是更简单吗？**

分层解决四个问题：1) 解耦——每层职责单一，改一层不影响其他层；2) 复用——DWS 汇总层被多个 ADS 报表复用，避免重复计算；3) 可追溯——出问题逐层排查，定位是 ODS 数据问题还是 DWD 清洗问题；4) 性能——预计算 DWS 层加速查询，避免 ADS 层每次扫全量明细。直接一张大宽表的问题是"每次需求变更都要改这张表"，且无法复用中间结果，重复计算成本高。

### 第二层：证据与定位

**Q：ADS 报表数据对不上业务系统，怎么定位是哪一层出了问题？**

逐层对账，自下而上：1) ODS vs 业务系统——ODS 数据是否和源系统一致（如果不一致是采集/同步问题）；2) DWD vs ODS——清洗逻辑是否正确（如过滤条件、字段映射）；3) DWS vs DWD——聚合逻辑是否正确（如 group by 字段、时间窗口）；4) ADS vs DWS——是否有多层 join 或复杂逻辑引入错误。每层用 count、sum、distinct 校验，定位到具体层后看 SQL 逻辑。

### 第三层：根因深挖

**Q：DWD 层和 DWS 层的数据量都很大，查询慢，根因是分区没设计好还是数据倾斜？**

两个原因都有，要看具体现象。1) 分区问题——如果查询没命中分区（如按 dt 分区但 query 没带 dt 条件），会全表扫描，慢是必然；2) 数据倾斜——如果某分区数据量是其他的 10x（如大促日的订单分区），reduce 阶段单点慢。区分方法：看执行计划的 scan 行数和 reduce 阶段的 task 耗时分布。分区问题是"扫得多"，倾斜问题是"扫得不均匀"。解法不同：分区问题加分区裁剪，倾斜问题加盐打散。

**Q：那为什么不直接在 DWS 层做物化视图，而要靠 ADS 层查询？**

DWS 层做物化视图（如 Kylin、ClickHouse 的物化表）适合"预聚合模式固定"的场景。但 ADS 层的需求多变（不同 BI 报表的维度组合不同），物化所有组合会导致存储爆炸（维度组合的笛卡尔积）。所以 DWS 层只做通用粒度的汇总（如按天、按城市、按品类的宽表），ADS 层在此基础上做定制查询。物化视图是"以空间换查询时间"，适合高频固定的查询，不适合多变需求。

### 第四层：方案权衡

**Q：星型模型（事实居中维度辐射）和雪花模型（维度再规范化）怎么选？**

权衡点是"查询性能 vs 存储和一致性"。星型模型——事实表直接关联维度表，join 少，查询快，但维度表冗余（如城市信息在多张维度表重复）；雪花模型——维度表再拆细（城市表拆成城市-省份-国家），规范化好、冗余少，但 join 多、查询慢。数仓场景查询频率高、写频率低，倾向星型模型（牺牲冗余换查询性能）。3NF 雪花模型适合 OLTP，星型适合 OLAP。

**Q：为什么不直接把所有数据打成一张超级宽表（事实 + 所有维度），避免 join？**

超级宽表的问题：1) 维度变更难——改一个维度字段要重刷整张宽表；2) 冗余膨胀——高基数维度（如 user_id）会让宽表行数等于事实表，但每行都带一堆低频维度字段，存储浪费；3) 灵活性差——新维度需求要改宽表 schema。宽表适合"特定分析场景"（如用户行为分析宽表），但不适合作为通用数仓中间层。星型模型在灵活性和性能间平衡，是通用解。

### 第五层：验证与沉淀

**Q：怎么衡量数仓分层设计是否合理，而不是"为了分层而分层"？**

三个指标：1) 复用率——DWS 层被 ADS 引用的次数，复用率高说明分层有价值；2) 产出时效——从 ODS 到 ADS 的全链路产出时间，分层后应该比"直算"更快（因为有预计算）；3) 故障恢复时间——某层出问题后修复的时间，分层应该比"单层"更快定位。沉淀为数仓规范文档：分层职责定义、命名规范（ods_/dwd_/dws_/ads_ 前缀）、字段类型规范，新人按规范开发。
