## 五、接手策略的核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| 主链路优先 | 先保证核心功能可用 | 先改UI/动效 |
| 用真实链路验证 | 用真实任务而非mock验证 | 只做技术验证不做业务验证 |
| 不破坏现有功能 | 渐进式改进而非推翻重建 | 第一个月就大重构 |
| 数据驱动决策 | 用指标发现真实问题 | 凭感觉决定优先级 |
| 建立反馈闭环 | 让用户反馈能到达开发 | 改完不验证效果 |

## 六、常见坑

- **先改视觉层**：UI再好看，任务跑不通也没用——先理顺主链路
- **不做权限审计**：接手后不检查现有权限设计，留下安全隐患
- **没有回放和指标**：改完不知道好没好，因为没有度量手段
- **推翻重建**：不尊重现有代码和设计，全部推翻重来——风险极高

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：为什么接手前三个月要先补"任务中心和产物协议"，而不是先改视觉层（UI/动效）让用户立刻感受到变化？**

因为视觉层是"锦上添花"，主链路是"雪中送炭"——优先级应该基于"对用户价值的影响"而非"可见度"。接手一个系统时，如果主链路（任务能不能跑通、产物能不能落地）有问题，用户连基本工作都完不成，这时候改 UI 动效（让按钮的 hover 效果更丝滑）对用户毫无价值——用户不会因为"按钮动画好看"而原谅"任务跑不通"。视觉层的问题（UI 不够好看）用户能忍受，主链路的问题（任务频繁失败、产物丢失、多窗口状态错乱）用户无法忍受——前者影响"满意度"，后者影响"可用性"。三个月的时间有限，必须把资源投向"影响可用性"的主链路修复，而非"影响满意度"的视觉优化。另一个原因：视觉层的改动（如换 UI 框架、调配色）不依赖主链路的健康状况，随时可以做；但主链路的修复（如统一产物对象模型、理顺状态机）如果不先做，后续的视觉层改动可能因为数据结构不稳定而反复返工（如产物模型改了三次，UI 组件跟着改三次）。所以先理顺主链路是"为后续所有工作打地基"。

### 第二层：证据与定位

**Q：你怎么定位接手的产品"哪块最该补"，而不是凭感觉排优先级？**

用"用户流失漏斗 + 投诉热力图"定位。第一步，看用户流失漏斗——从"发起任务"到"产物落地"每个环节的流失率，流失率最高的环节就是最该补的。如果流失集中在"任务执行失败"（40% 的任务报错），最该补任务中心的可靠性（状态机、断点续跑）；如果流失集中在"产物不被采纳"（用户生成后不看/不导出），最该补产物协议（质量、预览、编辑）；如果流失集中在"文件操作被拒绝"（用户不敢授权），最该补权限设计。第二步，看投诉热力图——收集用户工单和反馈，按类型分类统计，top 3 投诉类型就是最该补的方向。如果 top 3 是"任务跑着跑着就没了""多窗口数据不一致""文件被误覆盖"，分别对应任务状态管理、多窗口同步、权限/写入校验。第三步，看技术债评估——代码审查时标记"高风险高复杂度"的模块（如没有测试的权限系统、hardcode 的状态转换），这些是"随时可能爆炸的定时炸弹"，即使当前没出问题也要优先补。三个数据源交叉验证后，优先级就清晰了，而非凭感觉。

### 第三层：根因深挖

**Q：为什么 Month 1 理顺主链路的关键是"统一产物对象模型"，而不是先修具体的 bug？**

因为"产物对象模型不统一"是大量具体 bug 的根因——修 bug 是治标，统一模型是治本。如果 5 个 Agent 各自定义产物格式（Agent A 的产物是 { type, html }，Agent B 的是 { kind, content }，Agent C 的是 { artifactType, data }），会导致：(1) 产物面板的渲染逻辑充满 if-else（按 agentType 判断用哪种字段），每加一个 Agent 就要改产物面板，bug 频发；(2) 版本管理无法统一——不同格式的产物做 Diff 要写不同的算法，某个 Agent 的版本回退有 bug；(3) 产物落地（导出/发布）逻辑各自为政——有的 Agent 支持导出有的不支持，用户体验不一致。统一为 Artifact 对象模型（id/taskId/kind/content/version/status）后，产物面板、版本管理、Diff、导出全部基于统一接口实现，一次修复惠及所有 Agent，bug 数量指数级下降。所以 Month 1 先做模型统一（治本），Month 1 后期和 Month 2 再修具体 bug（治标），效率最高——如果不先统一模型，修 bug 时会发现"这个 bug 的根因是模型不统一，改了模型 bug 自动消失"，白费修 bug 的功夫。

**Q：那如果团队急着修一个线上 P0 bug（如"任务状态丢失导致用户数据没了"），为什么不先修 bug 再做模型统一？**

因为"P0 bug 修复"和"模型统一"不冲突，可以并行——但顺序上 P0 bug 用 hotfix 快速止血，模型统一作为月度工程并行推进。具体策略：P0 bug（任务状态丢失）的 hotfix 是"把任务状态从组件提升到全局 store"（1-2 天的紧急修复），这个 hotfix 本身就是"模型统一"的一部分（任务状态上浮到 TaskStore 是主链路梳理的核心动作）。所以修 P0 bug 的过程就是在做模型统一——不是两件事，是同一件事的不同阶段。真正的风险是"借口修 bug 推迟模型统一"——每次都说"先修 bug 再重构"，结果 bug 永远修不完（因为模型不统一导致新 bug 不断产生），模型统一永远没时间做。正确做法是"在 Month 1 就把模型统一作为目标，P0 bug 作为模型统一过程中的验证场景"——修任务状态丢失 bug 时顺便统一 TaskStore，修产物渲染 bug 时顺便统一 Artifact 模型，一举两得。

### 第四层：方案权衡

**Q：接手策略你主张"渐进式改进"而非"推翻重建"，为什么不直接重构（如果现有代码质量很差）？**

因为"推翻重建"的风险远大于收益，尤其是对已经上线的复杂系统。推翻重建的三个致命风险：(1) 时间黑洞——重写整个系统通常比预期长 3-5 倍（因为要重新理解所有业务逻辑和边界情况），3 个月变成 1 年，期间用户还在用旧系统（bug 没修），体验持续恶化；(2) 回归风险——重写时必然遗漏旧系统的隐性逻辑（如某个特殊场景的兼容处理），上线后出现大量"以前能用的功能现在不能用了"的回归 bug；(3) 机会成本——团队花 1 年重写，这 1 年里没有任何新功能交付，竞品在前进，产品竞争力下降。渐进式改进的优势：(1) 持续交付价值——每个月都有可见的改进（Month 1 主链路顺了、Month 2 权限安全了、Month 3 质量可度量了），用户持续感受到变好；(2) 风险可控——每次改动范围小（只改一个模块），出问题容易回滚；(3) 保留有效逻辑——现有代码中合理的部分（如某 Agent 的业务逻辑）保留，只改架构层（状态管理、对象模型），不重复劳动。除非现有系统的架构完全无法支撑未来发展（如硬编码到改一个功能要动 50 个文件），否则渐进式改进几乎总是优于推翻重建。

**Q：那如果现有代码质量差到"改一个 bug 引发三个新 bug"（牵一发而动全身），为什么还不推翻重建？**

因为"改一个 bug 引发三个新 bug"的根因通常是"缺乏测试覆盖"而非"架构不行"——补测试比推翻重建更能解决这个问题。没有测试的代码，任何改动（无论渐进式还是重建）都会引入回归 bug，因为开发者无法知道改动影响了哪些功能。推翻重建后如果仍然不写测试，新系统很快又会变成"改一个 bug 引发三个新 bug"——只是把旧的技术债换成了新的技术债。正确做法是"渐进式改进 + 补测试同步推进"：Month 1 统一产物模型时，先为核心链路（任务状态机、产物 CRUD）写集成测试，有了测试保护后再做重构，改动引发的新 bug 能立刻被测试捕获。具体策略：" characterize then refactor"——先用测试固化现有行为（characterization test，记录当前系统的输入输出，即使当前逻辑有 bug 也先固化），然后重构时确保测试仍然通过，这样重构不会引入回归。所以根因是"测试缺失导致改动不安全"，解法是"补测试"而非"重建"——重建只是逃避了补测试的苦工，问题依然存在。

### 第五层：验证与沉淀

**Q：你怎么证明"三个月路线图"的执行是有效的，产品真的变好了？**

用 Month 1-3 的验证标准 + 核心指标趋势联合验证。每月有明确的验收标准（Month 1：全链路任务跑通 + 组件卸载后任务继续；Month 2：权限分层 + 写入 Diff 预览；Month 3：回放系统 + 质量指标看板），这些是"功能交付验证"——完成了说明工程目标达成。但功能完成不等于产品变好，还要看核心指标趋势：任务完成率（Month 1 后应从接手时的未知/混乱变为可度量且 >85%）、产物采纳率（Month 2 后权限和上下文改善应提升采纳率）、重跑率（Month 3 后回放和 Diff 改善应降低重跑率）。如果三个月后这三个指标都显著改善（如完成率从 60%→85%、采纳率从 45%→70%、重跑率从 35%→15%），就证明路线图有效。如果功能都交付了但指标没改善，说明"做了功能但没解决真实问题"——可能是优先级判断错误（补了不重要的模块），需要根据指标重新调整后续方向。

**Q：怎么让接手团队在执行三个月路线图时，自觉按"主链路优先"的顺序，而不是被各种临时需求打乱节奏？**

把三个月路线图做成"团队 OKR + 需求过滤框架"的强制约束。第一，把路线图转化为季度 OKR（Month 1：任务状态机统一 + 产物模型统一；Month 2：权限分层 + 上下文校验；Month 3：回放系统 + 质量指标），OKR 是团队承诺，临时需求不能随意挤占 OKR 资源；第二，建立"需求过滤框架"——所有临时需求（如产品经理加的新功能、销售承诺客户的定制）必须回答"它是否服务于当前月的路线图目标"，不服务的需求推迟到三个月后再评估，避免路线图被临时需求稀释；第三，每周同步会检查路线图进度——如果某周路线图工作被临时需求挤占超过 30%，负责人有权拒绝低优先级的临时需求，保护路线图执行；第四，建立"路线图可见性"——把三个月路线图画在团队可见的地方（看板/周报），让所有干系人（产品/销售/管理层）知道当前在做什么、为什么不能随便加需求，形成共识。这样路线图就从"计划"变成了"受保护的承诺"。

## 结构化回答




**30 秒电梯演讲：** 就像接手一家餐厅——第一件事不是重新装修门面（视觉层），而是先搞清楚厨房出菜流程（任务和产物协议）、食材供应链（文件和上下文管理）、品质反馈机制（回放和复盘）。主链路顺畅了，装修才有意义。

**展开框架：**
1. **先补任务中心** — 先补任务中心和产物协议
2. **再补文件授权** — 再补文件授权和上下文管理
3. **最后补回放** — 最后补回放、diff和问题复盘

**收尾：** 如果桌面端要接文件、网页和本地目录，你先画哪套权限边界？
