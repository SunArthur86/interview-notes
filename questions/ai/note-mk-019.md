## 二、Month 1: 任务中心 + 产物协议

### 目标：理顺主链路

// Week 1-2: 梳理现有任务状态机
// 问题诊断清单：

☑ 任务和产物的关系是否清晰？

☑ 任务中心能否跨窗口复用？

  id: string;
  taskId: string;
  kind: ArtifactKind;
  content: string;
  version: number;
  status: ArtifactStatus;
  // ...
}

// Week 4: 三步发布链路
// 确保草稿→确认→发布的完整流程可用
```

### 验证标准

```
Month 1 验收：
  ✅ 一个完整的用户任务可以走通全链路
     发起 → 索引 → 推理 → 产物 → 编辑 → 确认 → 发布
  ✅ 任务在组件卸载后继续执行
  ✅ 多窗口可以看到一致的任务状态
  ✅ 产物有版本历史，支持回退
```

```

## 四、Month 3: 回放 + 质量

### 目标：建立质量保障体系

```typescript
// Week 9-10: 回放系统
// 如果发现用户无法理解AI为什么给出某个结果：
// → 实现五段回放（输入→计划→执行→产物→修改）

// → 摘要Diff（新手）
// → 结构化Diff（高级用户）
// → 原始Diff（开发者）
// → 大产物的页面级Diff

// Week 12: 质量指标体系
// → 任务完成率、产物采纳率、重跑率
// → 按Agent/产物类型拆分
// → Bad case收集和改进闭环
```

### 验证标准

```
Month 3 验收：
  ✅ 用户可以回放任意已完成任务的执行过程
  ✅ 产物diff支持三级切换
  ✅ 有核心指标的看板
  ✅ Bad case有收集渠道和改进流程
```

## 五、接手策略的核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| 主链路优先 | 先保证核心功能可用 | 先改UI/动效 |
| 用真实链路验证 | 用真实任务而非mock验证 | 只做技术验证不做业务验证 |
| 不破坏现有功能 | 渐进式改进而非推翻重建 | 第一个月就大重构 |
| 数据驱动决策 | 用指标发现真实问题 | 凭感觉决定优先级 |
| 建立反馈闭环 | 让用户反馈能到达开发 | 改完不验证效果 |

## 六、常见坑

- **先改视觉层**：UI再好看，任务跑不通也没用——先理顺主链路
- **不做权限审计**：接手后不检查现有权限设计，留下安全隐患
- **没有回放和指标**：改完不知道好没好，因为没有度量手段
- **推翻重建**：不尊重现有代码和设计，全部推翻重来——风险极高
