# Agent 简历项目怎么找到可以深挖的点？

## 一、什么是"可深挖的点"

```
不可深挖（CRUD，面试官不感兴趣）：
  ✗ "用LangChain搭了个Agent"
  ✗ "集成了OpenAI API"
  ✗ "实现了对话功能"
  ✗ "用了向量数据库"

可深挖（有难点，面试官想问）：
  ✓ "检索召回率从60%优化到85%"
  ✓ "工具调用准确率从70%提升到92%"
  ✓ "单任务Token成本降低70%"
  ✓ "P99延迟从5s优化到1s"
  ✓ "搭建了评估闭环，持续优化"
  ✓ "解决了多用户记忆隔离问题"
```

## 三、深挖点的表达结构（STAR + 数据）

```markdown
# 好的简历条目示例

## ❌ 不好（太笼统）
"开发智能客服Agent，支持自动回答用户问题"

## ✅ 好（深挖+量化）
"智能客服Agent（日活1000+）
- **问题**：FAQ匹配准确率仅60%，大量转人工
- **方案**：设计向量+BM25+Rerank三路混合检索，
  HyDE查询改写解决语义鸿沟
- **结果**：准确率提升至88%，转人工率从40%降至18%
- **技术**：LangChain + Milvus + bge-reranker"
```

## 四、找深挖点的提问法

```
问自己这些问题，答案就是深挖点：

1. "这个项目最难的部分是什么？"
   → 难点就是深挖点

2. "哪些地方花了最多时间调优？"
   → 调优的过程是深挖点

3. "上线后遇到什么问题？怎么解决的？"
   → 线上问题是好素材

4. "和同类产品比，你的优势在哪？"
   → 差异化优势

5. "如果重做，你会怎么改进？"
   → 反思体现深度

6. "有哪些trade-off决策？"
   → "为什么用A不用B"是面试官爱问的
```

```

─────────────────────────────────────────────

```

## 六、简历深挖的"避坑"

```
❌ 避免的写法：
  - "负责Agent开发"（太泛）
  - "使用LLM技术"（没说做了什么）
  - "提升了效率"（没数据）
  - 堆砌技术名词（没说怎么用的）

✅ 推荐的写法：
  - 动词开头（设计/优化/实现/解决）
  - 有具体问题（"召回率低""延迟高"）
  - 有量化结果（"从X到Y""提升Z%"）
  - 有技术细节（具体用什么方案）
  - 有反思（"如果重做会..."）
```

## 七、面试加分点

1. **找"非 CRUD"的点**：面试官不在乎你用了什么框架，在乎你解决了什么难问题
2. **量化一切**：有数字的简历比没数字的强 10 倍
3. **准备反思**：面试官爱问"如果重做你会怎么改"——有反思说明真理解了

## 记忆要点

- 可深挖的本质是难点与量化，因为纯CRUD（如搭LangChain）面试官不感兴趣，而召回率升至85%才是亮点
- 挖掘五字诀：准（准确率）、快（延迟/并发）、省（Token/路由）、稳（降级/熔断）、架（多智能体设计）
- 牢记STAR+数据结构：因为交代了痛点与方案，所以必须用量化结果证明业务价值
- 灵魂拷问出素材：因为项目的Trade-off选型（为何用A不用B）和最耗时调优点必被深问，所以重点准备

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说"召回率从 60% 提到 85%"是可深挖的点，但如果我项目里召回率本来就没那么低、也没做大幅优化，是不是就没亮点可讲了？**

不是只有"从差到好"才是亮点。亮点的本质是"有难度的决策 + 可量化的结果"，有三类都算。第一类是优化类（从 60 到 85）。第二类是选型类——"为什么选 Milvus 不选 Pinecone，考虑了成本/可控性/延迟三个维度，最终选 Milvus 让运维成本降 X"，选型 trade-off 同样有深度。第三类是踩坑类——"上线后遇到 Lost in Middle 导致中段召回率掉到 40%，通过重要信息前置 + 分块调整修复到 85%"，从踩坑到修复的过程也是亮点。所以不是非要"优化数字大"，是有没有遇到真实难点并解决，难点本身就是深挖点。

### 第二层：证据与定位

**Q：简历上写了"召回率提升到 85%"，面试官问"你怎么算的 85%"，结果我答不上来评测集怎么构造的，怎么办？**

这是最容易被问倒的点，必须提前准备。85% 的算法是 recall@k = 检索到的相关文档数 / 应检索到的相关文档总数，关键在"相关文档怎么定义"。你要能说清楚三件事：评测集是 200 条 query，每条的 relevant doc 是人工标注的（标注标准是"包含答案的段落"）；recall@5 是看前 5 个检索结果里有没有 relevant doc；85% 是这 200 条的平均召回。如果你说不出"评测集多少条、怎么标注、k 是几"，面试官会认为数字是编的。所以写简历前必须真的搭过评测集、跑过数字，能把评测方法讲清楚比数字本身更重要。

### 第三层：根因深挖

**Q：面试官追问"召回率从 60 到 85，具体做了哪几个动作，每个动作贡献了多少？"如果我只说"加了 Rerank 和改了分块"，会被深问到答不上，怎么准备？**

必须做消融实验记数据。准备时把优化路径拆成可独立验证的步骤：基线（纯向量检索 recall@5=60%）→ +增大 top_k（65%）→ +BM25 混合（76%）→ +查询改写（80%）→ +Rerank 精排（83%）→ +父子分块（85%）。每步的边际贡献能说清楚，说明你不是"一顿操作碰巧好了"，而是有方法论地系统优化。面试官深问任何一个动作（"BM25 怎么融合的，RRF 的 alpha 调多少"），你都有具体答案。准备方法是在优化过程中记录每步的评测结果，而不是事后回忆——事后回忆一定说不清细节。

**Q：那为什么面试官非得问这么细，知道思路不就够了，又不是写论文？**

因为 LLM 应用是"概率系统"，同样的方法在不同业务上效果差异巨大，面试官无法靠"你用了 Rerank"判断你真不真懂。只有问到细节（BM25 和向量的分数怎么对齐、RRF 还是线性加权、alpha 怎么调的），才能区分"真的调过"还是"看了博客照着写"。这是 LLM 领域面试的特点——方法名人人会说，细节决定真假。所以准备时不是背方法名，是把每个方法的关键参数和踩过的坑记清楚，这是区分度所在。

### 第四层：方案权衡

**Q：项目里我同时做了召回率优化、成本优化、延迟优化，简历上全写还是挑一个重点写？全写显得丰富，挑一个显得有深度。**

挑 1-2 个深写，不要全列。原因：简历的目的是"勾引面试官问你"，不是"展示你做了多少"。全列变成流水线（每个一笔带过），面试官不知道往哪问，可能跳过。深写 1-2 个（每个带数据 + 选型 + 踩坑），面试官会顺着深问，你正好展示深度。我的建议：选"最能体现你思考深度"的那个（通常是有明确 trade-off 的选型决策，而非纯调参），其余在"其他优化"里一笔带过作为补充。深度 > 广度，面试考察的是深度。

**Q：为什么不把所有技术名词（LangGraph/Milvus/Rerank/HyDE）都堆上，显得技术栈全？**

因为堆名词会被反向深问击穿。你写 HyDE，面试官问"HyDE 的假设答案如果错了怎么处理"，你答不上反而减分。技术名词是"承诺"，写了就要能答到细节。正确做法是只写你真的懂细节的（能答 3 层追问的），不懂的别写。少而精的名词 + 每个都能深答，比一堆名词但都答不深强得多。面试官见过太多"简历写一堆实际答不上"的，堆名词反而是减分信号。

### 第五层：验证与沉淀

**Q：我怎么知道简历上的深挖点经得起面试官追问，而不是自己觉得好但一问就露馅？**

用"模拟面试"验证。找一个懂行的朋友（或自己分饰两角），对着简历每一条，连续追问 5 层（是什么→怎么做→为什么这么做→有什么替代方案→怎么验证效果），凡是答到第 3 层就卡壳的，要么补准备要么从简历删。这个自测能暴露哪些点是"真懂"哪些是"虚的"。标准是每条简历内容都能扛住 5 层追问不露怯，扛不住的别写。这是最有效的简历打磨方法——以"能否扛住追问"作为留删标准。

**Q：这套"找深挖点"的方法怎么复用到下个项目？**

沉淀一个"项目复盘模板"。每个项目结束时按五类深挖点（准/快/省/稳/架）自查：有没有准确率提升的数据、有没有性能优化的对比、有没有成本降低的测算、有没有稳定性方案、有没有架构决策。每类有的话记下"痛点→方案→数据→选型 trade-off"四要素。这个模板强制你在项目过程中就记录深挖素材（而不是事后回忆），下个项目的简历自然有料。模板还能帮你发现"这个项目好像五类都没亮点"——那就是项目太平庸，要么主动找难点要么不写。
