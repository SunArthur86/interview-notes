# 【拼多多 AI大模型开发】设计拼多多电商智能客服系统

> 来源：拼多多AI大模型开发面经（小红书）二面核心架构题

## 一、费曼类比

```
智能客服系统 = 7层智能大厦:

         用户咨询
            │
  ┌─────────┼──────────────────────────────────────────┐
  │  L1     │  接入层: 限流/熔断/会话隔离/恶意拦截         │
  │  L2     │  意图层: BERT意图识别 + 规则兜底             │
  │  L3     │  检索层: RAG售后政策 + 商品知识库            │
  │  L4     │  决策层: 规则优先(90%) + 大模型兜底(10%)     │
  │  L5     │  风控层: 赔付校验/资损拦截/话术过滤          │
  │  L6     │  推理层: CPU小模型集群 + GPU大模型按需       │
  │  L7     │  兜底层: 静态话术/人工转接/超时降级          │
  └────────────────────────────────────────────────────┘
            │
         用户回复
```

## 二、第一性原理分析

```
核心约束:
  - 日均亿级咨询 → 高并发
  - 资损风险(赔付/退款) → 高安全
  - 成本敏感(拼多多极致降本) → 低成本
  - 政策频繁变更 → 高时效

设计方案:
  分层处理(降本) + 风控拦截(安全) + 降级链路(稳定)
```

## 三、详细答案

### 3.1 七层架构详解

```
┌─────────────────────────────────────────────────────────────────┐
│ L1 接入层                                                        │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • 限流: 令牌桶/滑动窗口 → 洪峰时自动排队                    │  │
│ │ • 熔断: Hystrix/Sentinel → 下游故障时快速失败               │  │
│ │ • 会话隔离: 不同用户独立上下文，互不干扰                     │  │
│ │ • 恶意请求拦截: 黑名单/频率限制/验证码                       │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L2 意图层                                                        │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • 轻量BERT意图分类: 物流/退款/赔付/投诉/商品咨询/其他       │  │
│ │ • CPU推理 INT8量化 → 延迟<20ms                              │  │
│ │ • 关键词规则兜底: BERT置信度低时用正则匹配                   │  │
│ │ • 短句模糊处理: 口语化表达 → 归一化                          │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L3 检索层                                                        │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • RAG售后政策库: 向量检索+关键词混合 → 时效过滤             │  │
│ │ • 商品规格知识库: SKU/规格/库存/价格                        │  │
│ │ • 冲突政策过滤: 新旧版本政策 → 只返回最新版本                │  │
│ │ • 版本隔离: A/B测试时不同用户看到不同策略                    │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L4 决策层（核心！）                                               │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • 规则引擎优先: FAQ/政策匹配 → 90%请求CPU秒解               │  │
│ │ • 大模型兜底: 规则未命中 → LLM+RAG生成回复                  │  │
│ │ • 分流策略: 简单→规则 | 中等→小模型 | 复杂→大模型           │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L5 风控层（拼多多最看重！）                                       │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • 赔付规则强校验: 金额→查规则→拦截超额                      │  │
│ │ • 资损拦截: 禁止模型私自给出赔付承诺                         │  │
│ │ • 敏感话术过滤: 政治/色情/辱骂/竞品                          │  │
│ │ • 高风险转人工: 投诉/大额赔付/媒体来电                      │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L6 推理层                                                        │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • CPU集群: BERT INT8/小模型 → 高QPS低成本                   │  │
│ │ • GPU集群: LLM按需扩缩容 → 处理复杂生成                     │  │
│ │ • 长短任务分离: 简短回复(<100ms) vs 长文本生成(>1s)         │  │
│ └───────────────────────────────────────────────────────────┘  │
│                             ↓                                   │
│ L7 兜底层                                                        │
│ ┌───────────────────────────────────────────────────────────┐  │
│ │ • 静态话术模板: "您的咨询已记录，将在24h内回复"             │  │
│ │ • 人工转接: 高风险/模型置信度低 → 转人工坐席                │  │
│ │ • 超时降级: 模型超时3s → 返回默认话术                       │  │
│ └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 大促洪峰方案

```
洪峰策略: "不崩、不卡、不贵"

不崩:
  • 队列削峰: Kafka缓冲 → 消费者按处理能力拉取
  • 限流熔断: 超过阈值自动排队/拒绝低优先级请求
  • 弹性扩容: K8s HPA → CPU/GPU按需扩缩

不卡:
  • 长短请求分离: 简单查询走快通道(规则) 复杂走慢通道(LLM)
  • 热点缓存: 高频问题答案缓存(Redis)
  • 静态话术兜底: 降级时秒回模板消息

不贵:
  • 降级策略: 大促期间关闭大模型层 → 全走规则+小模型
  • batch合并: 请求批量推理提高GPU利用率
  • 优先级调度: VIP用户优先，普通用户排队
```

### 3.3 多轮对话上下文管理

```
问题: 多轮对话上下文无限膨胀 → Token爆炸 → 成本爆炸

解决方案:
┌───────────────────────────────────────────────────┐
│ 1. 滑动窗口: 只保留最近N轮对话                      │
│    [用户:退款] [AI:好的] [用户:退多少] [AI:全额]    │
│    ←─── 保留窗口 ───→                              │
│                                                   │
│ 2. 关键信息常驻: 提取关键实体独立存储                │
│    {订单号: 12345, 退款金额: 50元, 状态: 待审核}    │
│    → 每轮注入，不受窗口限制                         │
│                                                   │
│ 3. 冗余摘要压缩: 早期对话用LLM压缩成摘要            │
│    10轮对话 → "用户咨询了物流和退款问题，已退款50元" │
│    → 摘要替代原始对话                               │
└───────────────────────────────────────────────────┘
```

### 3.4 BadCase闭环

```
BadCase自动回流:
  用户反馈不满意
       ↓
  自动收集: Query + Context + Response + 用户操作
       ↓
  人工标注: 标注正确答案 + 错误类型
       ↓
  分类:
    - 规则未覆盖 → 新增规则
    - RAG召回差 → 优化知识库/Prompt
    - 模型生成错 → 加入微调数据集
    - 风控漏判 → 补充风控规则
       ↓
  模型迭代: 定期微调 + 规则更新 + A/B测试
```

## 四、技术选型

| 组件 | 技术选型 | 原因 |
|------|---------|------|
| 意图识别 | BERT INT8 (CPU) | 延迟<20ms，成本低 |
| RAG检索 | Milvus + BM25混合 | 向量+关键词双路召回 |
| 大模型 | 内部模型/API | 按需扩缩 |
| 消息队列 | Kafka | 削峰填谷 |
| 缓存 | Redis | 热点FAQ缓存 |
| 限流 | Sentinel | 淘系成熟方案 |
| 监控 | Prometheus+Grafana | 实时监控各层指标 |

## 五、苏格拉底式面试提问

1. **"你说规则引擎处理90%的请求，但如果规则引擎误判了呢？比如把投诉分类成了咨询？"** — 引出规则引擎的准确率监控、纠错机制、用户反馈闭环
2. **"RAG检索的售后政策有时效性，你怎么保证大模型不会用过期政策回答？"** — 引出版本过滤、时间戳校验、冲突检测
3. **"如果大模型给出的回复和规则引擎的回复矛盾，以哪个为准？"** — 以规则为准（确定性优先），大模型回复必须过风控校验
4. **"大促期间GPU资源不够，关掉大模型层后用户体验怎么保证？"** — 引出降级文案设计、人工坐席扩容、事后跟进
5. **"你的系统如何评估整体效果？有哪些北极星指标？"** — 引出自助解决率、转人工率、CSAT满意度、资损率

## 六、面试加分点

1. **分层架构清晰** — 7层每层职责明确，不是简单"大模型+RAG"
2. **强调风控层** — 拼多多最看重的是资损控制，赔付规则强校验
3. **提到大促降级** — 大促关大模型层，全走规则+小模型，体现成本意识
4. **多轮对话方案** — 滑动窗口+关键信息常驻+摘要压缩，不只是简单截断
5. **BadCase闭环** — 展示系统不是一次性的，有持续迭代能力
6. **量化指标** — 90%规则 + 10%大模型，成本降低90%+

## 结构化回答

**30 秒电梯演讲：** 设计一个电商智能客服系统，核心是7层架构：接入层→意图层→检索层→决策层→风控层→推理层→兜底层。拼多多的标准答案强调稳定性和成本控制。

**展开框架：**
1. **7层架构** — 接入层→意图层→检索层→决策层→风控层→推理层→兜底层
2. **决策层** — 规则引擎优先(90%简单问题)，大模型兜底(10%复杂问题)
3. **风控层** — 赔付规则强校验、资损拦截、禁止模型私自承诺

**收尾：** 您想深入聊：如何保证RAG检索的时效性？售后政策频繁变更怎么处理？
