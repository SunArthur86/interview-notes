# 【字节飞连面经】AI 编程工具的使用流程？怎么保证 AI 生成代码质量？

## 一、标准使用流程

```
[1] 先在 CLAUDE.md 写项目规则
    │  架构约定、命名规范、禁忌操作
    │  （"不准 mock 数据库""不准跳过 lint"）
    │  这是 AI 编程的"宪法"
    ▼
[2] 用 plan 模式让它先出方案
    │  复杂改动先出 plan，人 review plan
    │  比让它一上来狂改文件靠谱
    ▼
[3] 小步实施 + 每步跑测试
    │  每完成一个 Task 立刻跑测试 / type check
    │  绝不积压一堆改动后才验证
    ▼
[4] 完成后人工 review diff
    │  + 跑完整测试套件
    │  + type check + lint
```

## 二、质量三层兜底

| 层 | 作用 | 成本 |
|----|------|------|
| **项目规则文件**（CLAUDE.md） | 约束生成方向（写什么、不写什么） | 低（写一次） |
| **单测/集成测试** | 防回归（改了不能坏） | 中（要维护） |
| **人工 Code Review** | catch 设计问题（AI 看不到的） | 高（每 PR 都要） |

**为什么必须三层**：单层都有盲区——规则文件管不住细节、测试管不住设计、Code Review 管不住大规模回归。

## 三、测试驱动

- 让 AI **先写测试再写实现**（TDD）
- 或改完立刻让它跑现有测试
- 测试覆盖率不够的地方，让 AI 先补测试再改

## 四、Code Review：留意 AI 三个典型问题

| 问题 | 表现 | 怎么 catch |
|------|------|-----------|
| **过度抽象** | 简单逻辑套一堆 interface/factory | 看是否有"用不到的抽象层" |
| **无意义的兜底 try-catch** | `try: ... except: pass` 吞掉异常 | 搜 `except Exception` / `except: pass` |
| **自作主张加新依赖** | 为了一个小功能引入整个库 | 看 `package.json` / `requirements.txt` diff |

## 五、加分点

- 提到自己有一套 **keyboard shortcut / hook 配置 / custom skill**，说明是深度用户而不是偶尔玩玩
- **多模型交叉验证**：复杂决策让另一个模型（Codex / Gemini）独立 review 一次，catch 单模型盲点

## 六、雷区

- ❌ "让 AI 一次性改完所有再 review" → 出问题定位困难
- ❌ "AI 写的代码跑通就 merge" → 没人工 review 容易埋雷
- ❌ "不写 CLAUDE.md，每次重新解释项目规则" → 重复劳动 + 不一致

## 七、扩展

- **CLAUDE.md 必写内容**：架构分层（哪层依赖哪层）、命名规范、禁忌操作（不准 mock DB / 不准跳过 lint）、测试约定（怎么跑）、性能红线
- **hook 配置**：pre-commit hook 自动跑 lint+test，AI 改完自动验证
- **AI 编程的边界**：架构决策、安全敏感（鉴权/加密）、业务核心逻辑，这些 AI 可以辅助但人来拍板

## 记忆要点

- 标准流程：先写CLAUDE.md定规矩，复杂改动先出plan，小步实施每步跑测试
- 质量三层兜底：规则文件(约束方向)、单测/集成(防回归)、人工Review(抓设计)
- TDD驱动：让AI先写测试再写实现，或改完立刻跑测试验证，绝不积压改动
- Review抓AI典型问题：过度抽象、无意义兜底(try-except pass)、自作主张加新依赖

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说复杂改动先让 AI 出 plan 再 review。但出 plan 本身要花时间，而且 plan 和最终代码可能有出入。为什么不让 AI 直接写代码，反正后面有人工 review 兜底？**

因为 plan 阶段 catch 问题的成本远低于代码阶段。plan 是"设计方向"的快速对齐——如果方向错了（比如 AI 打算用方案 A 但业务要方案 B），在 plan 阶段一句话就能推翻重来；如果让它直接写代码，写完几百行你才发现方向错，回滚和重写的成本是 plan 的 10 倍。而且 AI 写完代码后有"沉没成本"心理（人 review 时觉得"都写完了改方向太可惜"），容易将就。plan 的本质是"在小代价阶段暴露大方向问题"，这是软件工程 review 设计文档的同类实践，只是对象从人变成了 AI。出入问题确实存在，但 plan 对齐大方向后，细节出入在代码 review 阶段 catch，两阶段各自负责不同粒度。

### 第二层：证据与定位

**Q：AI 写的代码上线后出了 bug（比如某个边界条件没处理）。你怎么定位是 AI 写错了，还是 review 时没看出来？靠什么改进？**

靠 PR 的 review 评论和 AI 的生成 trace 对比定位。如果 review 评论里有人提过"这里要处理空值"但 AI 没改就 merge 了，是 review 流程执行问题（评论没落实就合）；如果 review 完全没提到那个边界，是 review 盲区（人或 AI 都没意识到）。改进分两路：盲区问题补测试——这个边界没覆盖说明缺测试用例，让 AI 补一个针对空值的单测防回归；流程问题加 gate——review 评论必须 resolved 才能 merge（GitHub 的 PR review 机制）。数据上统计"线上 bug 里有多少是 review 提过但没落实的"vs"review 完全没覆盖的"，前者是流程问题（加 gate），后者是认知问题（补测试 + review checklist）。

### 第三层：根因深挖

**Q：你重点抓 AI 三个典型问题：过度抽象、无意义 try-catch、自作主张加依赖。为什么是这三个，而不是别的（比如命名不规范、注释缺失）？**

因为这三个是"隐蔽且危害大"的，而命名/注释是"明显且易改"的。命名不规范一看就知道，lint 工具能自动 catch，成本极低。但这三个是设计层面的坏味道：过度抽象让代码难以理解（为了 3 个调用点搞个 factory），未来维护成本高；无意义 try-catch（`except: pass`）吞掉异常导致线上问题难定位，是 bug 的温床；自作主张加依赖（为一个小功能引入整个库）增加供应链风险和包体积。这三个的共同点是"AI 看起来很合理但实际有害"——AI 训练语料里有大量这类坏味道代码，它学会了模仿。catch 它们不能靠工具（lint 抓不住设计层面的过度抽象），必须靠人工 review 时带着"AI 易犯这三个错"的警觉去读 diff。

**Q：那如果 AI 写的代码有过度抽象，但功能是对的、测试也过了。为什么不在 review 阶段直接让它重构掉抽象，而是要人指出？**

因为 AI 自己意识不到"这是过度抽象"——在它的训练分布里，interface/factory/strategy 模式是"好代码"的标志（优秀开源项目里常见），它不会主动判定"这个场景不需要"。所以人指出后让 AI 重构是正确流程：人说"这里只有一种实现，去掉 factory 直接调"，AI 就能改。关键是在 review 时主动识别"用不到的抽象层"——判断标准是"这个抽象是否有第二个实现/调用点"，如果没有就是过度。让 AI 重构而非人手改，因为重构是机械活（AI 擅长），人的价值在于"判断该不该抽象"这个决策。这是人机分工：人做判断，AI 做执行。

### 第四层：方案权衡

**Q：你用三层兜底（规则文件 + 测试 + review）。但如果项目测试覆盖率只有 30%，测试这层形同虚设。为什么不要求先补到 80% 覆盖率再用 AI 编程？**

因为"先补到 80% 再用 AI"是理想但不可行的——老项目补测试的工程量极大（可能几个人月），业务等不起。务实做法是分层补：核心逻辑（鉴权、支付、风控）必须先补到 80%+ 再让 AI 碰；非核心（工具函数、UI）允许低覆盖率，但 AI 改动时必须"为改动的部分补测试"（增量覆盖）。规则写在 CLAUDE.md："AI 每次改动必须附带覆盖该改动的测试，否则 review 不通过。"这样覆盖率随 AI 编程自然提升，而非专门停下来补。整体覆盖率作为长期指标监控（每季度看趋势），但不作为"能否用 AI"的前置硬门槛——那是阻止业务迭代的教条。

**Q：CLAUDE.md 你说是"AI 编程的宪法"。但宪法太长 AI 可能忽略细节，太短又约束不住。怎么平衡 CLAUDE.md 的详略？**

CLAUDE.md 的详略原则是"写约束和禁忌，不写实现细节"。约束和禁忌是"方向性"的（如"不准 mock 数据库""鉴权必须走 AuthService"），AI 记得住且能遵守；实现细节（如"AuthService 怎么实现"）不该写，那是代码和测试的事。判断标准：CLAUDE.md 里每一条都应该是"AI 容易做错且靠测试抓不住的"——比如"不准跳过 lint"是 AI 易犯且 lint 本身抓不住（AI 会注释掉 lint 规则）；而"变量用驼峰命名"是 lint 能抓的，不用写进 CLAUDE.md。长度控制在 100-200 行，超出说明混入了实现细节。定期 review 删过时条款（架构变了旧约束要删），保持 CLAUDE.md 是"活的宪法"。

### 第五层：验证与沉淀

**Q：你怎么证明用 AI 编程工具后，团队的开发效率真的提升了，而不是"感觉快了但 bug 也多了"抵消掉？**

对比引入 AI 编程前后的量化指标。一是吞吐：每人每月 merge 的 PR 数 / 代码行数，引入后应显著提升。二是质量：线上 bug 率（bug 数 / 千行代码）、线上 bug 的平均修复时长，如果 bug 率上升但修复时长下降（AI 写的 bug 好定位），净影响可能仍正向。三是周期：从需求提得到 merge 的平均时长，应缩短。关键是要看"净效率"= 吞吐提升 - 质量回退成本。如果 bug 率涨了 20% 但吞吐涨了 100%，且 bug 多是易修的小问题（review 遗漏的边界），净效率仍是正。证明逻辑是"算总账"，不是单看"AI 写的代码 bug 多不多"。

**Q：怎么让团队新成员也按这套流程（CLAUDE.md + plan + 小步 + review）用 AI，而不是各自随意用导致质量参差？**

把流程做进工具而非靠口头宣导。一是 CLAUDE.md 进 git，新人 clone 就有，不用记忆；二是 pre-commit hook 强制跑 lint+test，AI 改完不过 hook 无法 commit，倒逼小步实施；三是 PR 模板里加"是否经过 plan review"checkbox，复杂改动没勾 plan 的 PR 直接 block；四是 review checklist 里固定列出"AI 三个典型问题"（过度抽象/无意义 try-catch/新依赖），reviewer 逐项打勾。把流程固化成工具 gate，新人的默认行为就是合规的——"要费力才能违规"比"要费力才能合规"更有效。再配一份"AI 编程 do/don't"的简短文档（一页纸），新人入职必读。
