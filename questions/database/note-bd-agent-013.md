# B树和B+树在数据库索引里的核心区别是什么？

## 结构对比

```
B树 (B-Tree):
              [30]
             /      \
        [10, 20]    [40, 50]
        /  |  \     /  |  \
     [5] [15] [DATA] [DATA] [DATA]
      ↑              ↑
      叶子有DATA     非叶子也有DATA
      
特点：每个节点都可能存key+data
查找可能在非叶子节点就命中

B+树 (B+Tree):
              [30]              ← 非叶子只存key
             /      \
        [10, 20]    [40, 50]    ← 非叶子只存key（索引）
        /  |  \     /  |  \
     [5,10][15,20][30,40][45,50]  ← 所有DATA在叶子
       ↔      ↔      ↔      ↔    ← 叶子间链表连接
       
特点：非叶子只存key，数据全在叶子
查找总是到叶子才结束，叶子间有链表
```

## 核心区别

| 维度 | B树 | B+树 |
|------|-----|------|
| **数据存储位置** | 所有节点都可能存data | 只有叶子节点存data |
| **查询稳定性** | 不稳定（可能在中间命中） | 稳定（总是到叶子） |
| **范围查询** | 需要中序遍历回溯 | 顺着叶子链表扫一遍 |
| **非叶子fanout** | 小（每个key带data） | 大（只存key，每页放更多） |
| **树高** | 较高 | 较低（fanout大） |
| **磁盘IO** | 较多 | 较少 |
| **适用场景** | 文件系统、MongoDB | MySQL、PostgreSQL |

## 为什么B+树更适合数据库？

### 1. 更高的fanout → 更矮的树 → 更少的IO

```
假设一个磁盘页16KB：

B树每个节点：
  key(8B) + data(假设1KB) ≈ 1KB/条
  → 每页放 ~16 条
  → 100万条数据的树高 ≈ 4层 → 4次IO

B+树非叶子节点：
  key(8B) + 指针(6B) ≈ 14B/条
  → 每页放 ~1170 条
  → 100万条数据的树高 ≈ 2-3层 → 2-3次IO

  B+树第1层: 1170 keys
  B+树第2层: 1170 × 1170 = 1,368,900 keys > 100万 ✅
  → 只需3次IO（根节点常驻内存→实际2次IO）
```

### 2. 范围查询效率

```sql
-- 查询 age BETWEEN 20 AND 30 的所有记录

B树的范围查询：
  找到age=20的节点 → 中序遍历 → 回到父节点 → 下一个子节点
  → 大量随机IO，效率差

B+树的范围查询：
  找到age=20的叶子节点 → 顺着链表往右扫 → 直到age>30
  → 顺序IO，效率极高
  [20] → [22] → [25] → [28] → [30] → stop
```

### 3. 查询稳定性

```
B树：查key=10可能在第1层就找到，查key=50要到第3层
     → 响应时间不稳定

B+树：无论查什么key，都要走到叶子节点
     → 响应时间稳定，便于性能预估
```

## 面试加分点

1. **fanout计算**：能估算B+树每页可放的key数量（约1170个）
2. **范围查询对比**：B+树叶子链表做顺序IO，B树要中序遍历做随机IO
3. **MongoDB选B树的原因**：MongoDB单文档查询为主，不需要频繁范围扫描
4. **聚簇索引**：InnoDB的主键索引就是聚簇索引——叶子节点存完整行数据

## 记忆要点

- 数据分布：B树所有节点存数据，而B+树数据全在叶子节点，非叶子只存索引
- 性能对比：B树查询速度不稳定（中途可能命中），而B+树查询必须到叶子，性能极其稳定
- 范围查询：B树靠中序遍历需大量随机IO，而B+树靠叶子节点双向链表顺序IO极快
- 磁盘IO：因为B+树非叶子只存key（Fanout大），所以树更矮（通常2-3层），磁盘IO次数远少于B树


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：B+树的非叶子节点只存 key 不存 data，你说这让"fanout 更大、树更矮"，但 data 存哪里了？不会因为存在叶子导致整体更费空间吗？**

data 存在叶子节点，整体空间没省（甚至叶子节点多了双向链表指针，稍多）。"fanout 大"的价值不是省空间，是"省磁盘 IO"。B+树一个节点对应一个磁盘页（InnoDB 默认 16KB），非叶子节点只存 key+指针（假设 key 8 字节、指针 6 字节，一页能放约 1170 个 key），意味着每个非叶子节点能分叉 1170 路。三层 B+树（根+中间+叶子）能索引 1170×1170≈137 万个叶子页，每页存若干行，总索引能力达千万到亿级数据，而 IO 只有 3 次（根和中间页常驻 buffer pool，实际可能 1-2 次）。B 树非叶子也存 data，单页放的 key 少（几十个），三层只能索引几千行，要索引千万数据要 5-6 层，IO 翻倍。所以 B+树是用"同样的页大小放更多 key"换取"树更矮、IO 更少"。

### 第二层：证据与定位

**Q：你说 B+树查询"必须到叶子所以稳定"，B 树"中途可能命中所以不稳定"，稳定是好事吗？不稳定不也意味着有时更快？**

不稳定在某些场景是缺点——预测性差。B 树查 root 可能直接命中（1 次 IO，极快），也可能查到最深的叶子（h 次 IO）。对延迟敏感的场景（如 OLTP 交易），你希望每次查询都是确定的 2-3 次 IO（P99 可控），而不是"有时 1 次有时 5 次"。B+树的"稳定"指每次都到叶子，查询时间方差小，P99 可预测。B 树的"不稳定"是均值可能更低但方差大，P99 可能很高。数据库场景要的是"P99 可控"（慢查询会拖垮连接池），所以选 B+树。但 B 树在"点查为主、数据随机分布"的内存数据库场景反而合适（无磁盘 IO 顾虑），所以 MongoDB 早期用 B 树（后来也转 B+树）。

### 第三层：根因深挖

**Q：B+树叶子节点用双向链表连起来支持范围查询，为什么不直接在非叶子节点做范围遍历？**

非叶子节点不存 data，做范围遍历也要回到叶子。更关键的是，叶子层有序+链表使得"找到范围起点后，顺着 next 指针顺序读"，这是磁盘友好的顺序 IO。如果在非叶子层遍历，要在树的不同分支间跳转（中序遍历），每次跳转是随机 IO（不同页）。磁盘的顺序 IO 速度是随机 IO 的 10-100 倍（SSD 也是几倍差距），所以"叶子链表顺序扫"远快于"非叶子树形遍历"。这就是为什么 `SELECT * FROM t WHERE id BETWEEN 100 AND 200` 在 B+树上极快（找到 id=100 的叶子，顺序扫 100 个叶子页），而在 B 树上要反复回溯父节点，慢且 IO 多。双向链表（不是单向）支持反序范围查询（ORDER BY id DESC）。

**Q：那为什么不用哈希索引，等值查询 O(1) 不是比 B+树的 O(logN) 更快吗？**

哈希索引确实等值查询 O(1)，但有三个致命局限：一、不支持范围查询——哈希后无序，`WHERE id > 100` 退化成全表扫描；二、不支持排序——`ORDER BY` 要重新排序，B+树叶子天然有序可省排序；三、不支持前缀匹配——`WHERE name LIKE 'abc%'` 在 B+树上能用索引（前缀有序），哈希索引不行。数据库 80% 的查询涉及范围、排序、JOIN，这些都需要有序结构。哈希索引只适合"纯 KV 点查"场景（如 Redis、Memcached）。MySQL 的 InnoDB 主索引必须支持范围查询（主键、唯一键常用范围扫），所以选 B+树。MySQL Memory 引擎用哈希索引（适合临时表点查），但不是通用存储引擎。

### 第四层：方案权衡

**Q：B+树三层能索引千万数据，那亿万级数据要分库分表吗？还是 B+树层数增加也能撑？**

B+树层数增加理论能撑更多数据（四层能索引 1170³≈16 亿），但每多一层多一次磁盘 IO，查询变慢。实践上，单表超过 5000 万-1 亿行（取决于行大小）时，B+树高度增加、索引维护成本（插入时页分裂）上升、buffer pool 命中率下降，性能开始劣化。这时分库分表是更优解——把数据水平切分到多个库/表，每个分片保持小规模（如 1000 万），B+树维持 3 层。所以"B+树层数增加"和"分库分表"都能扩容，但前者牺牲单查询性能（IO 增加），后者牺牲架构复杂度（要处理分片路由、跨分片查询）。权衡：数据量 < 1 亿优化 B+树（调页大小、索引、SQL），> 1 亿考虑分库分表。

**Q：为什么 InnoDB 页大小默认 16KB，不是 4KB 或 64KB？**

16KB 是"IO 效率 vs 内存利用"的平衡点。一、IO 效率——操作系统磁盘 IO 单位通常是 4KB，16KB 是 4 个连续页，一次 IO 读 16KB 比读 4 次 4KB 快（减少 IO 次数）；二、内存利用——页太大（64KB）会让 buffer pool 的页数变少（同样内存能缓存的页少），命中率下降；三、页分裂代价——页越大，页分裂时数据移动越多，写性能下降。16KB 是 MySQL 经过大量测试的经验值，SSD 普及后也有人调到 32KB/64KB（SSD 随机 IO 快，大页更省 IO 次数），但 InnoDB 的很多优化（如 change buffer、adaptive hash）都是基于 16KB 调优的，改大可能引入未知问题。生产环境一般保持 16KB，除非有明确压测数据支持调整。

### 第五层：验证与沉淀

**Q：你怎么验证 B+树索引在实际查询中真的被用上、且 IO 次数符合预期？**

三步验证：一、EXPLAIN——`EXPLAIN SELECT ...` 看 type 列（const/eq_ref/ref/range/index/ALL），ALL 是全表扫描（索引失效），其他说明用了索引；二、EXPLAIN ANALYZE（MySQL 8.0+）——看实际执行计划，包括"rows examined per loop"（实际扫描行数），如果远大于"filtered rows"说明索引选择性差；三、`SHOW STATUS LIKE 'Innodb_buffer_pool_read_requests'` 对比 `Innodb_buffer_pool_reads`——前者是总读请求，后者是未命中 buffer pool 落磁盘的次数，比值反映缓存命中率。线上监控慢查询日志（`long_query_time=0.1` 抓所有 >100ms 查询），分析是否因索引失效导致全表扫描。这些手段组合能确认 B+树索引健康。

**Q：这道题做完，你沉淀出了什么可复用的索引设计原则？**

四条原则：一、索引选择性——区分度高的列建索引（如 user_id），低区分度（如 gender 只有男女）建索引意义不大；二、覆盖索引——把查询需要的列都放进索引，避免回表（如 `SELECT name FROM t WHERE id=X`，建 (id, name) 联合索引，直接从索引返回）；三、最左前缀——联合索引 (a, b, c) 支持 a、(a,b)、(a,b,c) 查询，不支持 (b,c) 查询，建索引要按查询频率排序；四、避免索引失效——`WHERE DATE(create_time)` 会在函数列上失效索引，改用 `WHERE create_time >= '...' AND create_time < '...'`。这套原则用于评审所有 DDL，避免乱建索引（写多读少场景索引是负担）或建错索引。


## 结构化回答

**30 秒电梯演讲：** B树非叶子节点也存数据，B+树只在叶子存数据+链表——查询更稳定、范围扫描更高效。打个比方，B树像字典正文直接在每个目录项后写内容(中间也可能命中)，B+树像目录只放页码所有内容在叶子页(稳定+连贯)。

**展开框架：**
1. **数据分布** — B树所有节点存数据，而B+树数据全在叶子节点，非叶子只存索引
2. **性能对比** — B树查询速度不稳定（中途可能命中），而B+树查询必须到叶子，性能极其稳定
3. **范围查询** — B树靠中序遍历需大量随机IO，而B+树靠叶子节点双向链表顺序IO极快
