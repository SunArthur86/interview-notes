# 【拼多多一面】Redis 持久化方式(RDB/AOF)及优缺点

## 🎯 一句话本质

Redis提供两种持久化机制：**RDB**（Redis Database，定时内存快照）和 **AOF**（Append Only File，命令追加日志），以及4.0引入的**混合持久化**。

## 🧒 费曼类比

```
RDB（拍照片）：
  9:00 拍第一张 → 9:30 拍第二张 → 10:00 拍第三张
  如果9:45断电 → 恢复到9:30的状态（丢了15分钟数据）

AOF（记账本）：
  SET name "Alice"  ← 记一条
  SET age 25         ← 记一条
  DEL name           ← 记一条
  断电了？从头到尾重放一遍，一条不丢！

混合持久化：
  先拍一张9:00的全量快照 → 再从9:00开始记账
  断电恢复 = 加载快照（秒级） + 重放增量日志（少）
```

## 📊 RDB vs AOF 全面对比

```
┌──────────────────────────────────────────────────────────────────┐
│                        Redis 持久化架构                           │
├────────────────────────┬─────────────────────────────────────────┤
│         RDB            │              AOF                        │
│   ┌──────────────┐    │    ┌──────────────────────────────┐     │
│   │ dump.rdb     │    │    │ appendonly.aof               │     │
│   │ (二进制快照)  │    │    │ (文本命令日志)                │     │
│   └──────┬───────┘    │    └──────────┬───────────────────┘     │
│          │             │               │                         │
│   save/bgsave          │   appendfsync                           │
│   ↓ fork子进程         │   ↓ always/everysec/no                  │
│   ↓ COW机制            │   ↓ 写入策略                             │
│   ↓ 生成.rdb文件       │   ↓ 追加.aof文件                         │
└────────────────────────┴─────────────────────────────────────────┘
              ↕ Redis 4.0+ 混合持久化 ↕
    ┌─────────────────────────────────────────┐
    │  aof文件 = RDB全量数据(前半) + AOF增量(后半)  │
    └─────────────────────────────────────────┘
```

| 对比维度 | RDB | AOF |
|---------|-----|-----|
| 文件格式 | 二进制压缩 | 文本命令 |
| 文件体积 | 小（压缩） | 大（未重写时持续增长） |
| 恢复速度 | 快（直接加载二进制） | 慢（需重放所有命令） |
| 数据安全性 | 低（可能丢失快照间隔数据） | 高（最多丢1秒数据） |
| 对性能影响 | fork时短暂阻塞 | 持续IO写（取决于策略） |
| 可读性 | 不可读 | 可读（文本命令） |

## 🔧 RDB详解

### 触发方式

```bash
# 1. 手动触发
save      # 阻塞主线程，生产禁用！
bgsave    # fork子进程，推荐

# 2. 配置自动触发
save 900 1     # 900秒内1次修改 → bgsave
save 300 10    # 300秒内10次修改 → bgsave
save 60 10000  # 60秒内10000次修改 → bgsave

# 3. shutdown/主从全量同步时自动触发
```

### bgsave原理（Copy-On-Write）

```
主进程 ──fork()──→ 子进程
  │                   │
  │ 写时复制(COW)      │ 遍历所有key
  │ 主进程继续处理      │ 序列化到dump.rdb
  │ 客户端请求          │ 完成后发信号给主进程
  │                   │
  └───────────────────┘
  
  如果主进程修改了某个key → OS复制该页 → 子进程仍然读旧值 → 一致性保证
```

## 🔧 AOF详解

### 三种刷盘策略

| 策略 | 配置 | 含义 | 性能 | 安全性 |
|------|------|------|------|--------|
| always | `appendfsync always` | 每条命令都fsync到磁盘 | 最差 | 最高（不丢数据） |
| everysec | `appendfsync everysec` | 每秒fsync一次 | 好 | 高（最多丢1秒） |
| no | `appendfsync no` | 由OS决定何时fsync | 最好 | 最低（可能丢较多） |

> **生产推荐**：`everysec`，平衡性能与安全。

### AOF重写（Rewrite）

```
原始AOF文件：                    重写后：
SET k1 v1                        SET k1 v3
SET k2 v2                        SET k2 v2
DEL k1                           SET k4 v4
SET k1 v3
INCR counter   → 合并 →          counter值直接写入，去掉历史命令
INCR counter
SET k4 v4

结果：文件从7条命令 → 3条，体积大幅缩小
```

## 💻 混合持久化配置（Redis 4.0+）

```bash
# 开启AOF
appendonly yes
appendfsync everysec

# 开启混合持久化（4.0+默认开启）
aof-use-rdb-preamble yes

# 触发AOF重写时：
# 1. 子进程将当前内存生成RDB格式写入新aof文件前半部分
# 2. 重写期间的增量命令以AOF格式追加到后半部分
# 3. 替换旧aof文件
```

```
混合持久化后的AOF文件结构：
┌─────────────────────────┬──────────────────────────────┐
│   RDB二进制数据（全量）    │   AOF文本命令（增量部分）       │
│   恢复速度快              │   数据不丢失                  │
└─────────────────────────┴──────────────────────────────┘
```

## 📋 面试加分点

1. **fork阻塞**：fork子进程时如果内存占用很大（如20GB），fork本身可能阻塞数百毫秒。这就是为什么大实例不适合频繁bgsave。

2. **AOF损坏修复**：`redis-check-aof --fix appendonly.aof` 可以修复末尾被截断的AOF文件。

3. **主从架构中的持久化策略**：
   - 主节点：关闭持久化（纯内存，性能最优），或只开AOF everysec
   - 从节点：开启RDB + AOF，作为数据备份节点

4. **Redis 7.0 multi-part AOF**：将单个AOF文件拆分为base（RDB格式）+ incr（增量）+ manifest（清单），解决了大AOF重写时磁盘IO暴增的问题。

## ❓ 苏格拉底式面试追问

1. **"你说bgsave用COW机制，那如果fork之后主进程大量写入新数据，会发生什么？"**
   → 引导分析COW的内存翻倍风险——大量页被复制，内存可能OOM

2. **"AOF重写过程中又有新命令写入，Redis怎么保证不丢这些命令？"**
   → 重写期间的新命令同时写入旧AOF和aof_rewrite_buffer，重写完成后追加到新文件

3. **"生产环境你选RDB还是AOF？如果数据完全允许丢失呢？如果一秒都不能丢呢？"**
   → 分场景讨论：缓存场景可关闭持久化；数据库场景用混合持久化

4. **"Redis主从切换时，如果新主的AOF文件比旧主少了几秒数据，怎么处理这种不一致？"**
   → Sentinel/Cluster会标记数据不一致，需要业务层做幂等处理

5. **"如果Redis内存30GB，RDB恢复需要多久？AOF恢复需要多久？有没有办法优化？"**
   → RDB约30秒-1分钟，AOF可能10-30分钟。优化：混合持久化、分片减小单实例体积


## 结构化回答

**30 秒电梯演讲：** Redis持久化有RDB（内存快照）和AOF（命令日志）两种方式，4.0+支持混合持久化，兼顾恢复速度和数据安全。

**展开框架：**
1. **RDB口诀** — 快照小恢复快，但两次快照间数据可能丢
2. **AOF口诀** — 日志全不丢数据，但文件大恢复慢
3. **混合持久化** — 混合持久化 = RDB全量快照 + AOF增量命令，Redis 4.0+默认推荐
