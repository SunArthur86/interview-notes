# 【拼多多 Java服务端】MySQL索引底层用的什么数据结构？和其他数据结构相比有什么优势？

> 来源：拼多多复活赛一面面经（小红书）

## 一、费曼类比

```
B+树 vs 其他结构的图书馆类比:

B+树 = 多层目录卡片系统:
┌─────────────────────────────────────┐
│ 1F: 大类目录 (计算机|文学|历史...)    │  ← 非叶节点(只存方向)
│   ↓                                 │
│ 2F: 小类目录 (AI|数据库|网络...)      │  ← 非叶节点
│   ↓                                 │
│ 3F: 书架 (按序排列，书架间有走道)     │  ← 叶子节点(存实际数据)
│   ←──── 双向链表连接 ────→          │
└─────────────────────────────────────┘
找书: 从1F→2F→3F，3次"IO"就能找到任何书
范围: 找到第一本后，沿走道走到相邻书架即可
```

## 二、第一性原理分析

**核心约束：磁盘IO是瓶颈**

```
磁盘读取以"页"为单位(InnoDB默认16KB/页):
  - 随机IO: ~10ms (SSD ~0.1ms)
  - 顺序IO: ~1ms (SSD ~0.01ms)
  - 内存访问: ~100ns

目标: 最小化IO次数

B+树如何实现:
  每个节点 = 一个数据页(16KB)
  非叶子节点只存: 键值(假设8B) + 子节点指针(6B) = 14B/条
  一个页可存: 16384 / 14 ≈ 1170个键

  3层B+树容量:
    根节点:    1170个键
    第二层:    1170 × 1170 = 1,368,900个键  
    第三层(叶子): 1,368,900 × 每页约16行 ≈ 2000万行!
    
  → 2000万行数据只需3次IO!
```

## 三、详细答案

### 3.1 B+树结构

```
                    ┌───────────────────────┐
                    │  根节点(非叶子)         │  ← IO #1
                    │  [10|20|30|40|50]      │
                    └───┬───┬───┬───┬───┬───┘
            ┌──────────┘   │   │   │   └──────────┐
            ↓              ↓   │   ↓              ↓
    ┌──────────────┐ ┌────────┐│┌────────┐ ┌──────────────┐
    │ [1|5|8]      │ │[11|15]│││[21|25] │ │[31|35|38]    │ ← IO #2
    │ (非叶子)      │ │       │││        │ │              │
    └──┬──┬──┬─────┘ └─┬──┬──┘│└─┬──┬───┘ └──┬──┬──┬─────┘
       ↓  ↓  ↓         ↓  ↓   ↓   ↓  ↓        ↓  ↓  ↓
     叶子 叶子 叶子   叶子 叶子 叶子 叶子 叶子  叶子 叶子 叶子
     [1] [5] [8]    [11] [15] [20] [25] [30] [31] [35] [40] ← IO #3
     数据 数据 数据  数据 数据 数据 数据 数据 数据 数据 数据
     
     ←──┘←──┘←──┘←──┘←──┘←──┘←──┘←──┘←──┘←──┘←──┘
          叶子节点双向链表（←→ 顺序遍历）
```

### 3.2 B+树 vs B树 vs 其他数据结构

| 数据结构 | 查找复杂度 | 范围查询 | 磁盘IO | 树高(百万数据) | 适合数据库? |
|---------|-----------|---------|--------|-------------|------------|
| **B+树** | O(log N) | 极快(链表遍历) | 3-4次 | 3层 | **最优** |
| **B树** | O(log N) | 慢(需中序遍历) | 3-4次 | 3-4层 | 可以但范围查询差 |
| **红黑树** | O(log N) | 慢 | **20+次** | ~20层 | 差(树太高) |
| **Hash** | O(1) | **不支持** | 1次 | N/A | 仅等值查询 |
| **跳表** | O(log N) | 快 | 未知 | ~20层 | Redis用 |

### 3.3 B+树 vs B树 核心区别

```
B树: 数据分布在所有节点
         ┌───────────────────┐
         │ [10 DATA|20 DATA] │  ← 非叶节点也存数据
         └───┬───────────┬───┘
             ↓           ↓
     [5 DATA]          [15 DATA]
     
     问题: 非叶节点存数据 → 每页存键少 → 树更高 → IO更多
           范围查询: 需要中序遍历整棵树 → 大量随机IO

B+树: 数据只在叶子节点
         ┌───────────────────┐
         │   [10 | 20]        │  ← 非叶节点只存键+指针
         └───┬───────────┬───┘
             ↓           ↓
     [5 DATA]←→[10 DATA]←→[15 DATA]←→[20 DATA]  ← 叶子链表
     
     优势: 非叶节点存键多 → 树矮 → IO少
           范围查询: 顺着链表走 → 顺序IO
```

### 3.4 聚簇索引 vs 二级索引

```
聚簇索引 (Clustered Index):
  → 数据行按主键B+树组织，叶子节点 = 完整数据行
  → 一张表只有一个聚簇索引
  → InnoDB的"表"本质上就是聚簇索引

  ┌─────────────────────────────────────────┐
  │      聚簇索引 B+树                        │
  │                                         │
  │  非叶节点: [主键值 + 页指针]              │
  │                                         │
  │  叶子节点: [主键 | 列1 | 列2 | ... | 列N] │ ← 完整数据行
  └─────────────────────────────────────────┘

二级索引 (Secondary Index):
  → 叶子节点存: 索引列值 + 主键值(不是数据行!)
  → 查到主键后需要"回表"到聚簇索引找完整数据

  ┌─────────────────────────────────────────┐
  │      二级索引 B+树 (如idx_name)           │
  │                                         │
  │  非叶节点: [name值 + 页指针]              │
  │                                         │
  │  叶子节点: [name值 | 主键ID]             │ ← 只有索引列+主键
  │           ↓                              │
  │      回表到聚簇索引 → 找到完整数据行       │
  └─────────────────────────────────────────┘

覆盖索引 (Covering Index):
  → SELECT的列都在索引中 → 不需要回表!
  → 如: SELECT id, name FROM users WHERE name='Alice'
  → idx_name的叶子节点[name|id]已经包含所需列
```

### 3.5 为什么不用Hash索引？

```
Hash索引:
  优点: 等值查询O(1)，极快
  缺点:
    ✗ 不支持范围查询 (WHERE age > 18)
    ✗ 不支持排序 (ORDER BY)
    ✗ 不支持模糊匹配 (LIKE 'abc%')
    ✗ 不支持最左匹配 (联合索引)

  → Memory引擎用Hash索引
  → InnoDB不支持Hash索引(但有自适应Hash索引AHI)
```

## 四、实际例子

```sql
-- 创建索引
CREATE INDEX idx_name_age ON users(name, age);  -- 联合索引

-- 索引选择分析:
EXPLAIN SELECT * FROM users WHERE name = 'Alice' AND age > 20;
-- type=range, key=idx_name_age (走联合索引)

EXPLAIN SELECT * FROM users WHERE age = 20;
-- type=ALL (索引失效! 违反最左匹配)

-- 覆盖索引(避免回表)
EXPLAIN SELECT id, name FROM users WHERE name = 'Alice';
-- Extra: Using index (覆盖索引, 无需回表)
```

## 五、索引失效场景

| 场景 | 例子 | 原因 |
|------|------|------|
| 函数操作 | `WHERE YEAR(create_time) = 2024` | 索引列被函数包裹 |
| 隐式类型转换 | `WHERE phone = 13800138000` (phone是varchar) | 类型不匹配 |
| 最左匹配违反 | 联合索引(a,b,c)，查询只有b | 跳过了a |
| LIKE左模糊 | `WHERE name LIKE '%abc'` | 前缀不确定 |
| OR条件 | `WHERE a=1 OR b=2` (b无索引) | 需要全表扫描b |
| != / NOT IN | `WHERE status != 1` | 优化器认为扫描行数太多 |

## 六、苏格拉底式面试提问

1. **"为什么InnoDB要求有主键，不设会怎样？"** — 会选择唯一非空索引，没有则自动生成隐藏的ROW_ID
2. **"为什么推荐自增主键而不是UUID？"** — 自增主键顺序插入不会导致页分裂，UUID随机插入导致大量页分裂和碎片
3. **"联合索引(a,b,c)，WHERE b=1 AND c=2能走索引吗？"** — 不能，违反最左匹配，但WHERE a=1 AND c=2可以部分走索引(只用a)
4. **"B+树高度为什么通常只有3-4层？"** — 高扇出(每页存上千个键)使树高极低
5. **"如果一个表有10亿行，B+树会有几层？"** — 4层(第4层叶子节点: 1170^3 × 16 ≈ 260亿，4次IO足够)

## 七、面试加分点

1. **算出3层B+树存2000万行** — 展示对存储容量的定量理解
2. **画出聚簇索引vs二级索引** — 明确回表概念
3. **提到覆盖索引优化** — 真实项目中减少回表的常见手段
4. **知道Hash索引的局限** — 不支持范围/排序/模糊匹配
5. **提到自增主键的优势** — 避免页分裂、提高写入性能
