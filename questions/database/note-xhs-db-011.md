# 场景题：笔记类型数据（直播/图文/视频，有类型关联关系）应该用 MySQL 还是 Redis 存？什么场景必须用 MySQL？（小红书Java一面）

## 一、场景分析：笔记类型数据

```
数据特征：
  - 笔记有类型（直播、图文、视频）
  - 类型之间有关联关系
  - 需要查询某类型的所有笔记
  - 可能需要范围查询、聚合统计
```

### 推荐方案：MySQL 主存储 + Redis 缓存

```
用户请求
    │
    ▼
Redis 缓存命中？ ──是──→ 直接返回
    │ 否
    ▼
MySQL 查询（带类型条件、JOIN、聚合）
    │
    ▼
结果写入Redis缓存（TTL过期）
    │
    ▼
返回结果
```

### 为什么选 MySQL 做主存储？

| 需求 | MySQL能力 | Redis局限 |
|------|----------|-----------|
| 类型关联关系 | 外键、JOIN查询 | 无关联查询能力 |
| 复杂条件查询 | WHERE type='video' AND status=1 | 需要额外维护索引结构 |
| 范围查询 | WHERE create_time BETWEEN ... | ZSet只能按score范围 |
| 事务一致性 | ACID事务 | 仅Lua脚本保证原子性 |
| 数据持久可靠 | redo log + binlog 双保险 | RDB/AOF有窗口期 |
| 聚合统计 | GROUP BY、COUNT、SUM | 需要SCAN全量遍历 |

## 二、什么场景必须用 MySQL 不能用 Redis？

### 场景1：范围查询（面试官追问点）

```sql
-- 这类查询 Redis 基本无法实现
SELECT * FROM notes 
WHERE type = 'video' 
  AND create_time BETWEEN '2026-06-01' AND '2026-06-30'
  AND author_id IN (1001, 1002, 1003)
ORDER BY like_count DESC
LIMIT 20 OFFSET 0;
```

Redis 的 ZSet 只能按一个 score 排序，多维度范围+排序+分页几乎无法实现。

### 场景2：复杂关联查询

```sql
-- 三表JOIN：笔记←笔记类型←类型属性
SELECT n.*, t.type_name, t.icon_url, a.author_name
FROM notes n
JOIN note_types t ON n.type_id = t.id
JOIN authors a ON n.author_id = a.id
WHERE t.category = 'live_streaming'
  AND n.status = 1;
```

### 场景3：事务一致性

```sql
-- 发布笔记同时更新计数，必须事务保证
BEGIN;
INSERT INTO notes (title, content, type_id) VALUES (...);
UPDATE note_stats SET total_count = total_count + 1 WHERE type_id = ?;
COMMIT;
```

### 场景4：精确计数的财务/订单场景

```sql
-- 订单金额计算、对账等不允许任何误差
SELECT SUM(amount) FROM orders 
WHERE merchant_id = 1001 
  AND DATE(create_time) = '2026-07-01';
```

Redis 的 INCR 虽然原子但无法做复杂聚合，且数据可靠性不如MySQL。

## 三、Redis 的优势场景（反面对比）

```
✅ Redis 适合：
  - 热点笔记的列表缓存（ZSet排行榜）
  - 点赞计数（INCR原子操作）
  - 用户最近浏览历史（List）
  - 全局唯一ID生成（INCR）
  - 实时在线状态（Bitmap/Set）

❌ Redis 不适合做主存储：
  - 无复杂查询能力
  - 内存成本远高于磁盘
  - 持久化不如关系数据库可靠
  - 数据一致性保障弱
```

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：笔记类型数据你说"MySQL 主存、Redis 缓存"，但为什么不反过来 Redis 主存？笔记内容是半结构化（图文/视频/直播），Redis 不是更灵活吗？**

反过来不行，因为 MySQL 是"唯一可信数据源"，Redis 是易失缓存。理由：一、持久化——MySQL 的 ACID 和事务保证数据可靠（写入成功必不丢），Redis 即使持久化（RDB/AOF）也有窗口丢失风险（AOF everysec 丢 1 秒、主从异步复制丢更多），主存储不能容忍丢失；二、复杂查询——笔记类型关联（如查询某用户某类型的所有笔记）涉及 JOIN、索引、范围查询，MySQL 的 B+树擅长，Redis 的 KV/Sorted Set 模型难以表达复杂关联；三、事务——笔记的创建要更新多个表（笔记表、用户表计数、类型关联表），MySQL 事务保证原子，Redis 的 MULTI/EXEC 弱于真正事务。所以 Redis 适合"读加速"（缓存热点笔记列表），不适合"主存储"。半结构化数据可以用 MySQL 的 JSON 字段或关联表表达，灵活性不是问题。

### 第二层：证据与定位

**Q：线上一条笔记在 Redis 缓存里有，但 MySQL 查不到，你怎么定位这种"缓存幽灵数据"？**

典型原因：缓存与 DB 不一致。排查链路：一、确认缓存值——`GET note:123` 拿到缓存内容，看 update_time 或 version；二、查 MySQL——`SELECT * FROM notes WHERE id = 123`，如果返回空，说明 MySQL 里这条笔记已删除但 Redis 缓存未清；三、追溯历史——查 MySQL 的 binlog（用 canal 或 mysqlbinlog）看这条笔记是否被 DELETE 过、删除时是否有同步删 Redis 缓存的逻辑；四、查应用日志——搜 note:123，看最近的删除操作有没有触发缓存清理。常见根因：删除笔记的代码路径漏了"删 Redis 缓存"（如异步删除任务失败未重试），或缓存 TTL 太长导致幽灵数据长期存在。解决：补全删除时的缓存清理逻辑 + 设置合理 TTL 兜底。

### 第三层：根因深挖

**Q：你说"类型关联关系"要存 MySQL，具体怎么设计表结构？为什么不在笔记表里直接存 type 字段？**

要看关联的复杂度。如果"类型"是简单枚举（直播/图文/视频，一对一），笔记表里加 `type TINYINT` 字段够用，简单查询 `WHERE type = 1` 即可。如果"一个笔记可属于多个类型"或"类型有层级/属性"，要用关联表——如 `note_type_relations(note_id, type_id, extra)`，多对多关联。小红书的笔记可能同时是"视频 + 教程 + 美食"（多标签），单字段不够。设计原则：一对一用字段、一对多用逗号分隔或 JSON 字段（小数据量）、多对多用关联表。选关联表的好处是支持索引（`SELECT * FROM note_type_relations WHERE type_id = X` 走索引快）和扩展（加关联属性如"该类型的权重"）。所以"字段 vs 关联表"取决于关联复杂度，简单场景别过度设计关联表。

**Q：那为什么不直接全用 MongoDB 这类文档数据库存笔记？它天然支持半结构化和多类型。**

MongoDB 确实适合半结构化文档存储，但要看整体架构。如果系统已经用 MySQL 做主存储（团队熟悉、运维成熟、事务可靠），引入 MongoDB 会增加运维成本（新中间件、备份方案、监控）。MySQL 的 JSON 字段（5.7+）已经能存半结构化数据，配合虚拟列+索引可以查询 JSON 内字段，够用。MongoDB 的优势在"超大规模文档 + 灵活 schema + 水平扩展"，但小红书的笔记数据用 MySQL 关联表 + JSON 字段能扛（单表亿级靠分库分表）。所以选型看团队栈和规模，不是"哪个技术更新"。如果团队已有 MongoDB 经验且文档 schema 频繁变化，选 MongoDB；如果是 MySQL 为主的团队，用 MySQL JSON 字段。技术选型不是追新，是匹配团队能力和业务需求。

### 第四层：方案权衡

**Q：笔记数据 MySQL 主存 + Redis 缓存，缓存策略你选"Cache Aside"（先读缓存未命中再读 DB 回填），为什么不用"Write Through"或"Write Behind"？**

三种策略的权衡：一、Cache Aside（旁路缓存）——应用层管缓存，读未命中回填、写时删缓存，简单灵活，是互联网最常用的；二、Write Through（直写）——写时同时更新缓存和 DB，缓存与 DB 强一致，但写延迟高（等 DB）且要缓存层支持事务；三、Write Behind（回写）——写时只更新缓存、异步刷 DB，写延迟低但崩溃丢数据。选 Cache Aside 的理由：读写都够快（写只删缓存不等 DB）、一致性可接受（短窗口不一致 + TTL 兜底）、实现简单（应用层控制）。Write Through 的代价是写性能差（同步等 DB），Write Behind 的代价是数据可能丢失。对笔记这类"读多写少 + 可容忍秒级不一致"的场景，Cache Aside 是最佳平衡。强一致场景（如资金）才考虑 Write Through。

**Q：如果笔记更新非常频繁（每秒千次），Cache Aside 的"写时删缓存"会让缓存命中率暴跌，怎么破？**

高频更新场景缓存命中率确实会降。应对策略：一、合并更新——用消息队列或批量异步刷 DB + 缓存，减少写次数（如收集 1 秒内的更新批量应用）；二、短 TTL + 异步刷新——缓存设短 TTL（如 5 秒），命中过期时触发异步刷新（不阻塞读），读返回旧值（可接受）；三、读写分离——写走主库、读走从库 + 缓存，写不直接删缓存而是标记失效，读时检查标记；四、放弃缓存——如果命中率本来就低（< 50%），缓存收益小于维护成本，直接读 DB（加索引优化）。我的实践：超高频更新字段（如实时点赞数）单独用 Redis 计数器（INCR），不进 MySQL 缓存层；低频字段（如笔记正文）用 Cache Aside。按字段更新频率分级缓存策略。

### 第五层：验证与沉淀

**Q：你怎么验证"MySQL 主存 + Redis 缓存"方案在故障下（缓存宕机、DB 宕机）都可用？**

故障注入测试：一、Redis 宕机——kill Redis，应用应降级直读 DB（不报错），响应时间可能变长但功能正常；二、MySQL 宕机——kill MySQL，读应仍能从 Redis 拿到缓存数据（降级读），但写入失败（要返回明确错误或排队）；三、缓存与 DB 不一致——手动制造（删 MySQL 但留缓存），验证监控能否发现（定时抽样对比脚本）。线上监控：Redis 的命中率（`INFO stats` 的 keyspace_hits/misses）、MySQL 的 QPS 和慢查询、缓存不一致检测任务（抽样对比）。这些验证确保方案在故障下"降级而非崩溃"，是生产级缓存方案的必备测试。

**Q：这道题做完，你沉淀出了什么可复用的"主存+缓存"设计方法论？**

四步设计法：一、定主存——按数据可靠性、查询复杂度、事务需求选 MySQL/PostgreSQL（关系型）或 MongoDB（文档型），主存必须可靠；二、定缓存——按读 QPS 和数据特点选 Redis 类型（String 缓存对象、ZSet 缓存排行、Hash 缓存字段）；三、定一致性策略——默认 Cache Aside + TTL 兜底，强一致场景加 binlog 订阅删缓存或 Write Through；四、定降级方案——缓存宕机降级读 DB、DB 宕机降级读缓存，保证可用性。这套方法论适用于所有"缓存层"设计，核心是"主存保可靠、缓存保性能、降级保可用"。面试时遇到"Redis + MySQL 怎么配合"，按这四步答，逻辑清晰。
