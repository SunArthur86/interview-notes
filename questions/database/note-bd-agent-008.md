# Redis和MySQL同时存业务状态时，怎么保证最终一致性？

## 核心原则：MySQL是唯一数据源

```
┌──────────────┐         ┌──────────────┐
│    Redis     │         │    MySQL     │
│  (过程态)    │←───同步───│  (最终态)    │
│  库存预占     │         │  库存实际     │
│  锁单状态     │         │  订单数据     │
│  会话缓存     │         │  支付记录     │
└──────────────┘         └──────────────┘
     ↑                         ↑
   临时数据                   永久数据
   可丢失                     不可丢失
```

## 读策略：Cache Aside（旁路缓存）

```python
def read_data(key):
    # 1. 先读Redis
    data = redis.get(key)
    if data:
        return json.loads(data)
    
    # 2. 未命中，读MySQL
    data = mysql.query("SELECT * FROM t WHERE id = %s", key)
    
    # 3. 回填Redis（设置TTL）
    if data:
        redis.setex(key, 300, json.dumps(data))
    
    return data
```

## 写策略：先写MySQL后删Redis ⭐

```python
def write_data(key, new_value):
    """推荐方案：先DB后删Cache"""
    try:
        # 1. 先写MySQL（事务）
        mysql.execute("UPDATE t SET value=%s WHERE id=%s", 
                      new_value, key)
        mysql.commit()
        
        # 2. 删除Redis缓存（不是更新）
        redis.delete(key)
        
    except Exception as e:
        mysql.rollback()
        raise e
```

### 为什么"先DB后删Cache"而不是"先删Cache后DB"？

```
❌ 先删Cache后写DB的问题：

线程A: 删除Cache ──────────── 写DB ────────
线程B:            读Cache(未命中) ── 读DB(旧值) ── 写Cache(旧值)
                                          ↑
                                    缓存被旧值污染！

✅ 先DB后删Cache的流程：

线程A: 写DB ──── 删除Cache ────
线程B:     读Cache(命中旧值) ──────  (下次读会从DB拿新值)
                                         ↑
                                    最多脏读一个TTL周期
```

## 异常补偿机制

### 1. MQ重试（实时补偿）

```python
def write_with_mq_compensation(key, value):
    """先DB后删Cache + MQ补偿"""
    # 1. 写MySQL
    mysql.execute("UPDATE ...")
    mysql.commit()
    
    # 2. 发MQ消息（异步删Cache）
    mq.send("cache_invalidate", {"key": key})
    
    # 消费者：删除失败重试3次
    # → 3次失败进入DLQ → 定时任务兜底
```

### 2. 定时任务对账（兜底补偿）

```python
def reconciliation_task():
    """每隔5分钟对比Redis和MySQL数据"""
    # 找出差异记录
    hot_keys = redis.scan("stock:*")
    for key in hot_keys:
        redis_val = redis.get(key)
        mysql_val = mysql.query("SELECT stock FROM ...")
        
        if redis_val != mysql_val:
            # 以MySQL为准，修正Redis
            redis.set(key, mysql_val)
            log.warning(f"数据不一致已修正: {key}")
```

### 3. 库存预扣场景的特殊处理

```python
def pre_deduct_with_fallback(item_id, qty):
    """库存预扣 + 落库失败回滚"""
    # 1. Redis预扣（原子操作）
    remaining = redis.eval(DEDUCT_SCRIPT, item_id, qty)
    if remaining < 0:
        return False  # 库存不足
    
    try:
        # 2. 异步落库MySQL
        mysql.execute("UPDATE stock SET qty=qty-%s WHERE id=%s", 
                      qty, item_id)
        mysql.commit()
    except Exception:
        # 3. 落库失败→回滚Redis预扣
        redis.incrby(f"stock:{item_id}", qty)
        return False
    
    return True
```

## 完整一致性保障体系

```
                    写入请求
                       │
                       ▼
              ┌──── MySQL写入 ────┐
              │                    │
              ▼                    ▼
         删除Redis缓存          发MQ消息(异步)
              │                    │
              ▼                    ▼
         成功？              MQ消费者重试
         ├ 是 → 完成         ├ 成功 → 完成
         └ 否 → MQ补偿       └ 失败 → DLQ → 定时对账
                                    │
                                    ▼
                               以MySQL为准修正
```

## 面试回答要点

> "我们MySQL是最终数据源，Redis更多存过程态，不做强一致双写。

> **读**用Cache Aside——先查Redis，未命中查MySQL再回填。

> **写**用先写MySQL后删Redis——为什么不是先删Cache？因为先删后写DB有并发脏读风险，后删最多脏读一个TTL。

> **库存预扣**场景如果落库失败，就释放Redis占用；**支付和结算**链路再通过RabbitMQ重试和定时任务补偿。"

## 面试加分点

1. **明确分工**：MySQL=最终态，Redis=过程态，不做强一致双写
2. **知道为什么后删**：能解释"先删Cache后写DB"的并发脏读问题
3. **补偿体系**：MQ实时补偿 + 定时任务兜底对账
4. **预扣回滚**：落库失败时释放Redis预扣，保证不超卖

## 记忆要点

- 核心主从：MySQL是唯一数据源，而Redis仅作临时过程态。
- 写策略口诀：先写DB后删Cache。
- 反面论证：因为先删Cache时并发读会把旧数据写回，所以极易造成脏读污染。
- 兜底补偿机制：延迟双删或利用MQ重试机制保证缓存最终必定删除。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：MySQL 和 Redis 双写，你选"先写 DB 后删缓存"而不是"先删缓存后写 DB"，为什么？**

两种顺序各有失败场景，但"先删缓存后写 DB"有个致命的并发脏读：线程 A 删缓存 → 线程 B 读缓存未命中、读 DB 拿到旧值、把旧值写回缓存 → 线程 A 写 DB 新值。结果缓存里是旧值、DB 是新值，缓存与 DB 不一致，且一直脏到缓存过期。"先写 DB 后删缓存"的失败场景是：删缓存失败 → 缓存仍是旧值，但这个可以通过"重试删 + 延迟双删"兜底修复。两者对比，"先删后写"的脏读是"读请求把旧值写回"导致的、且修复困难（要等缓存自然过期），"先写后删"的失败是"删操作失败"导致的、可用 MQ 重试修复。所以选"先写后删"，因为它的失败模式更可控、可修复。

### 第二层：证据与定位

**Q：线上发现某条数据缓存和 DB 不一致持续了 10 分钟，你怎么定位是哪一步出问题？**

三步排查：一、看应用日志——搜这条数据的 key，找到最近的"写 DB + 删缓存"日志，确认删除是否执行、是否成功；二、看 Redis——`TTL key` 看剩余过期时间，`GET key` 看当前值，对比 DB 当前值，确认是否真的不一致；三、看时间线——把"写 DB 时间、删缓存时间、缓存值的时间戳（如果存了）、DB 值的 update_time"对齐看。常见根因：删缓存时 Redis 短暂网络抖动导致删除失败但没重试、或者并发读请求在删缓存后又把旧值写回（即"先删后写"的脏读，但你以为是"先写后删"）。定位方法是在删缓存操作加详细日志（key、value、结果、耗时），出问题时能回溯。

### 第三层：根因深挖

**Q：你用了"延迟双删"——写 DB 后删缓存、sleep N 毫秒、再删一次。这个 sleep N 怎么定？为什么需要双删？**

双删是为了对抗"并发读把旧值写回"。场景：线程 A 写 DB（新值）→ 线程 A 删缓存 → 线程 B 在 A 写 DB 之前已经读到 DB 旧值、此时把旧值 set 回缓存 → 缓存是旧值、DB 是新值。单删删不掉"线程 B 后续 set 的旧值"，因为 B 的 set 发生在 A 的删除之后。解法：A 删完后再 sleep 一段时间（覆盖 B 完成 set 的时间窗口），再删一次，把 B 写回的旧值清掉。sleep 时长的依据：一次读请求的完整耗时（读 DB + 写缓存），通常 500ms-1s。太短覆盖不到 B 的 set，太长占用线程资源。这是工程上的"经验值"，不是严格证明——更稳妥的方案是"订阅 binlog 异步删缓存"（如 Canal），彻底避免并发 set 问题。

**Q：那为什么不直接用"订阅 binlog 异步删缓存"，而要先在应用层删一次？**

binlog 订阅删缓存（Canal 监听 MySQL binlog，收到变更消息后删 Redis）有毫秒级延迟——写 DB 后到 binlog 投递到 Canal、Canal 消费删缓存，中间这几十到几百毫秒内缓存仍是旧值。如果在这窗口内有读请求，会读到旧值。所以应用层"先删一次"是为了"立即让缓存失效"，binlog 订阅是"兜底保证最终删成功"。两者是"即时性 + 可靠性"的组合：应用层删保证读请求立刻看到新值（虽然可能因失败而不彻底），binlog 订阅保证删除最终一定成功（即使应用层删失败）。这种"双保险"在金融、电商等强一致场景常用，简单场景用其中一种即可。

### 第四层：方案权衡

**Q：如果业务说"缓存不一致导致用户看到旧价格，投诉了"，你会牺牲一致性还是性能？**

先量化再决策。一、不一致窗口多长？如果是 1 秒内（删缓存延迟），大多数用户感知不到，可接受；如果是 10 分钟（删失败未重试），是 bug 必须修。二、影响面？是单条数据还是批量？单条可能是个例故障，批量是系统性问题。三、业务容忍度？价格这类敏感数据要求强一致，宁可慢不能错——方案是"读 DB 为主、Redis 仅做热点加速 + 短 TTL（如 5 秒）"，用短 TTL 控制不一致窗口上限。描述文案这类非敏感数据可以"长 TTL + 最终一致"，用户看到旧文案影响小。所以不是"二选一"，是"按数据敏感度分级"——敏感数据走强一致（DB 直读或短 TTL）、非敏感走最终一致（长 TTL）。

**Q：为什么不直接放弃缓存，所有读都走 DB，这样就没有一致性问题了？**

能，但 DB 撑不住。读 QPS 上万时，MySQL 的连接数、CPU、IO 都会成为瓶颈。缓存的核心价值是"挡住 90%+ 的读请求"，让 DB 只处理缓存未命中的少量请求和所有写请求。放弃缓存等于把所有读压力直接打到 DB，要么 DB 撑不住（接口超时），要么成本爆炸（要堆很多 DB 机器）。所以缓存的本质是"用一致性换性能和成本"——接受秒级不一致，换取 10 倍以上的吞吐和更低的机器成本。只有数据量小、QPS 低、一致性要求极高的场景才"不缓存直读 DB"。这是架构上的经典权衡，不是技术问题，是业务取舍。

### 第五层：验证与沉淀

**Q：你怎么验证缓存一致性方案在各种故障下都 work？**

四类故障注入测试：一、删缓存失败——mock Redis 删除返回 0（key 不存在）或抛异常，验证 MQ 重试是否触发、最终缓存被删；二、并发读写——10 线程并发写、10 线程并发读同一 key，跑 1 万次后对比缓存值与 DB 值，统计不一致比例（应 < 阈值）；三、Redis 宕机——kill Redis，验证应用是否降级直读 DB（不应报错）；四、DB 主从延迟——模拟从库延迟 5 秒，验证读请求是否走主库（强一致场景）。线上监控：缓存不一致检测定时任务，抽样对比缓存与 DB，不一致率超 0.1% 告警。这些测试和监控是缓存方案的"安全网"，不做的化出问题就是线上事故。

**Q：这道题做完，你沉淀出了什么可复用的缓存一致性方法论？**

三条方法论：一、"写策略默认先写 DB 后删缓存"——这是最稳的默认选项，失败模式可控；二、"兜底机制分层"——MQ 重试删（应对删除失败）+ 延迟双删（应对并发脏读）+ 订阅 binlog（应对应用崩溃），层层兜底；三、"按数据敏感度选一致级别"——敏感数据强一致（短 TTL 或直读 DB）、非敏感最终一致（长 TTL + 异步修复）。这套方法论我已经固化成团队的"缓存使用规范"，新增缓存场景必须填一个 checklist（写策略、兜底机制、TTL、敏感度），review 通过才能上线，避免每个开发者踩一遍同样的坑。
