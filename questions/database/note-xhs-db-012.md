# 分库分表设计、分片键如何选择？（华为od Java二面）

## 一、为什么要分库分表？

```
单表性能瓶颈：
  - 数据量 > 1000万行 → B+树层数从3层→4层，IO增加
  - 写入QPS > 5000 → 单库连接数/IO打满
  - 单表文件 > 50GB → DDL变更锁表时间长

垂直拆分：                     水平拆分：
┌─────────┐                  ┌──────┐ ┌──────┐ ┌──────┐
│  Order   │                  │DB_0   │ │DB_1   │ │DB_2   │
│  Table   │                  │order_0│ │order_1│ │order_2│
│(50+字段)  │  ──→             │       │ │       │ │       │
└────┬────┘                  │user_0 │ │user_1 │ │user_2 │
     │                        └──────┘ └──────┘ └──────┘
     ▼
┌─────────┐  ┌──────────┐
│ Order    │  │OrderItem │   每个库表结构相同
│ (核心)   │  │(明细)     │   数据按分片键分散
└─────────┘  └──────────┘
```

## 二、分片键选择（核心考点）

### 选择原则

| 原则 | 说明 |
|------|------|
| **高基数** | 分片键取值范围大，数据分布均匀 |
| **查询频率** | 大部分查询条件包含此字段 |
| **不可变** | 值不随时间变化，避免数据迁移 |
| **单调性** | 避免热点（如自增ID可能集中在最后一个分片） |

### 常见分片键对比

```
1. user_id（推荐⭐）
   ✅ 大部分查询带userId（我的订单、我的文章）
   ✅ 高基数，分布均匀
   ❌ 不带userId的查询需要广播所有分片

2. order_id（含userId编码）
   ✅ orderId = timestamp + userId_hash + seq
   ✅ 订单查询直接定位
   ❌ 设计复杂

3. create_time（不推荐）
   ❌ 热点问题：最新数据集中在最后一个分片
   ❌ 范围查询仍需跨分片

4. hash(user_id) % N
   ✅ 数据均匀分布
   ❌ 扩容时需要重新hash迁移数据（一致性Hash可缓解）
```

### 分片策略

```
┌─────────────────────────────────────────┐
│           分片算法                        │
├──────────────┬──────────────────────────┤
│  Range分片    │ id: 1-1000→DB0          │
│  (范围)       │ id: 1001-2000→DB1       │
│               │ ✅范围查询友好            │
│               │ ❌可能热点               │
├──────────────┼──────────────────────────┤
│  Hash分片     │ hash(userId) % 4 → DB0-3│
│  (哈希)       │ ✅分布均匀               │
│               │ ❌扩容需迁移              │
├──────────────┼──────────────────────────┤
│  一致性Hash   │ 虚拟节点环               │
│  (Consistent)│ ✅扩容只影响相邻节点      │
│               │ ❌实现复杂               │
└──────────────┴──────────────────────────┘
```

## 三、分库分表带来的问题

### 1. 跨库 JOIN 不可用

```java
// ❌ 无法跨库JOIN
SELECT o.*, u.name FROM order_0 o JOIN user_1 u ON o.uid = u.id;

// ✅ 解决方案：
// 方案A：应用层组装（分别查询再合并）
List<Order> orders = orderMapper.selectByUid(uid);
List<Long> uids = orders.stream().map(Order::getUid).collect();
List<User> users = userMapper.selectByIds(uids);
// 手动组装

// 方案B：宽表冗余（订单表冗余存储用户名）
// 方案C：Elasticsearch做联合查询
```

### 2. 分布式事务

```java
// ❌ 单库事务失效
// DB0: INSERT INTO order_0 ...
// DB1: UPDATE stock_1 ... 

// ✅ 解决方案：
// 1. Seata AT模式（最终一致性，自动补偿）
// 2. TCC（Try-Confirm-Cancel）
// 3. 本地消息表 + MQ重试（最终一致性）
// 4. Saga模式（长事务）
```

### 3. 全局唯一ID

```
分库后自增ID会冲突，需要全局ID方案：

1. Snowflake雪花算法（推荐⭐）
   │ 1bit符号位 │ 41bit时间戳 │ 10bit机器ID │ 12bit序列号 │
   ✅ 趋势递增、不依赖DB、高性能
   ❌ 时钟回拨问题

2. 号段模式（Leaf-Segment）
   ✅ 简单可靠、ID递增
   ❌ 依赖DB、有波峰风险

3. Redis INCR
   ✅ 简单
   ❌ 依赖Redis可用性、ID不连续
```

### 4. 分页查询

```
-- ❌ OFFSET分页需要跨库合并
SELECT * FROM order LIMIT 10000, 20;
-- 需要每个分片都查10020条再全局排序取20条

-- ✅ 解决方案：
-- 1. 禁止深度分页（限制最大页数）
-- 2. 游标分页（WHERE id > last_id LIMIT 20）
-- 3. ES辅助查询
```

## 四、主流中间件对比

| 中间件 | 类型 | 特点 |
|--------|------|------|
| **ShardingSphere-JDBC** | Client端 | 无Proxy开销，Java原生，支持事务 |
| **ShardingSphere-Proxy** | Proxy端 | 多语言支持，透明代理 |
| **MyCat** | Proxy端 | 老牌方案，社区维护减少 |
| **Vitess** | Proxy端 | YouTube出品，云原生 |

## 五、分库分表 vs 其他方案

```
问题：单表数据量过大
     │
     ├── 先尝试：读写分离 + 索引优化（< 1000万行）
     │
     ├── 再尝试：冷热分离（历史表归档）
     │
     ├── 再尝试：分区表（PARTITION BY RANGE）
     │
     └── 最后：分库分表（> 5000万行，QPS > 10000）
```
