# 分库分表设计、分片键如何选择？（华为od Java二面）

## 一、为什么要分库分表？

```
单表性能瓶颈：
  - 数据量 > 1000万行 → B+树层数从3层→4层，IO增加
  - 写入QPS > 5000 → 单库连接数/IO打满
  - 单表文件 > 50GB → DDL变更锁表时间长

垂直拆分：                     水平拆分：
┌─────────┐                  ┌──────┐ ┌──────┐ ┌──────┐
│  Order   │                  │DB_0   │ │DB_1   │ │DB_2   │
│  Table   │                  │order_0│ │order_1│ │order_2│
│(50+字段)  │  ──→             │       │ │       │ │       │
└────┬────┘                  │user_0 │ │user_1 │ │user_2 │
     │                        └──────┘ └──────┘ └──────┘
     ▼
┌─────────┐  ┌──────────┐
│ Order    │  │OrderItem │   每个库表结构相同
│ (核心)   │  │(明细)     │   数据按分片键分散
└─────────┘  └──────────┘
```

## 二、分片键选择（核心考点）

### 选择原则

| 原则 | 说明 |
|------|------|
| **高基数** | 分片键取值范围大，数据分布均匀 |
| **查询频率** | 大部分查询条件包含此字段 |
| **不可变** | 值不随时间变化，避免数据迁移 |
| **单调性** | 避免热点（如自增ID可能集中在最后一个分片） |

### 常见分片键对比

```
1. user_id（推荐⭐）
   ✅ 大部分查询带userId（我的订单、我的文章）
   ✅ 高基数，分布均匀
   ❌ 不带userId的查询需要广播所有分片

2. order_id（含userId编码）
   ✅ orderId = timestamp + userId_hash + seq
   ✅ 订单查询直接定位
   ❌ 设计复杂

3. create_time（不推荐）
   ❌ 热点问题：最新数据集中在最后一个分片
   ❌ 范围查询仍需跨分片

4. hash(user_id) % N
   ✅ 数据均匀分布
   ❌ 扩容时需要重新hash迁移数据（一致性Hash可缓解）
```

### 分片策略

```
┌─────────────────────────────────────────┐
│           分片算法                        │
├──────────────┬──────────────────────────┤
│  Range分片    │ id: 1-1000→DB0          │
│  (范围)       │ id: 1001-2000→DB1       │
│               │ ✅范围查询友好            │
│               │ ❌可能热点               │
├──────────────┼──────────────────────────┤
│  Hash分片     │ hash(userId) % 4 → DB0-3│
│  (哈希)       │ ✅分布均匀               │
│               │ ❌扩容需迁移              │
├──────────────┼──────────────────────────┤
│  一致性Hash   │ 虚拟节点环               │
│  (Consistent)│ ✅扩容只影响相邻节点      │
│               │ ❌实现复杂               │
└──────────────┴──────────────────────────┘
```

## 三、分库分表带来的问题

### 1. 跨库 JOIN 不可用

```java
// ❌ 无法跨库JOIN
SELECT o.*, u.name FROM order_0 o JOIN user_1 u ON o.uid = u.id;

// ✅ 解决方案：
// 方案A：应用层组装（分别查询再合并）
List<Order> orders = orderMapper.selectByUid(uid);
List<Long> uids = orders.stream().map(Order::getUid).collect();
List<User> users = userMapper.selectByIds(uids);
// 手动组装

// 方案B：宽表冗余（订单表冗余存储用户名）
// 方案C：Elasticsearch做联合查询
```

### 2. 分布式事务

```java
// ❌ 单库事务失效
// DB0: INSERT INTO order_0 ...
// DB1: UPDATE stock_1 ... 

// ✅ 解决方案：
// 1. Seata AT模式（最终一致性，自动补偿）
// 2. TCC（Try-Confirm-Cancel）
// 3. 本地消息表 + MQ重试（最终一致性）
// 4. Saga模式（长事务）
```

### 3. 全局唯一ID

```
分库后自增ID会冲突，需要全局ID方案：

1. Snowflake雪花算法（推荐⭐）
   │ 1bit符号位 │ 41bit时间戳 │ 10bit机器ID │ 12bit序列号 │
   ✅ 趋势递增、不依赖DB、高性能
   ❌ 时钟回拨问题

2. 号段模式（Leaf-Segment）
   ✅ 简单可靠、ID递增
   ❌ 依赖DB、有波峰风险

3. Redis INCR
   ✅ 简单
   ❌ 依赖Redis可用性、ID不连续
```

### 4. 分页查询

```
-- ❌ OFFSET分页需要跨库合并
SELECT * FROM order LIMIT 10000, 20;
-- 需要每个分片都查10020条再全局排序取20条

-- ✅ 解决方案：
-- 1. 禁止深度分页（限制最大页数）
-- 2. 游标分页（WHERE id > last_id LIMIT 20）
-- 3. ES辅助查询
```

```
问题：单表数据量过大
     │
     ├── 先尝试：读写分离 + 索引优化（< 1000万行）
     │
     ├── 再尝试：冷热分离（历史表归档）
     │

```
## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：分库分表你说"5000 万行以上考虑"，这个 5000 万怎么来的？不是 B+树三层能存千万级吗？**

B+树三层确实能存千万级（约 2000 万行），但"能存"和"性能好"是两回事。5000 万这个阈值的依据：一、B+树高度——5000 万行（行平均 200 字节）约 1000 万叶子页，fanout=1170 时树高 3-4 层（边缘），查询 IO 偶尔到 4 次，性能开始波动；二、buffer pool 命中率——单表数据量超过 buffer pool 大小（如 16GB）时，命中率下降，磁盘读增多，查询变慢；三、索引维护成本——INSERT 触发页分裂的概率随数据量上升，写入性能下降；四、DDL 耗时——加列、建索引等 DDL 在大表上耗时分钟级，影响线上。所以 5000 万是经验阈值，不是硬性规则——行小（100 字节）可以到 1 亿，行大（1KB）2000 万就吃力。判断标准是"性能是否开始劣化"（P99 升高、慢查询增多），而非绝对行数。

### 第二层：证据与定位

**Q：线上库出现慢查询增多，你怎么判断是"该分库分表了"还是"索引/SQL 问题"？**

排除法。先排查索引/SQL：一、EXPLAIN 慢查询——如果 type=ALL（全表扫描）或 rows 巨大，是索引/SQL 问题，加索引或重写 SQL 能解决；二、检查 buffer pool 命中率——`SHOW STATUS LIKE 'Innodb_buffer_pool_read_requests'` 和 `Innodb_buffer_pool_reads`，命中率（1 - reads/requests）应 > 99%，低于 95% 说明数据量大到内存装不下；三、看表行数和数据大小——`SHOW TABLE STATUS` 的 Rows 和 Data_length，如果 Data_length 接近或超过 buffer pool 大小，且慢查询持续增多，是要分表的信号；四、DDL 耗时——加索引耗时分钟级，说明表太大。综合判断：索引/SQL 问题是"个别查询慢"，分表信号是"整体性能下降 + 多数查询变慢 + DDL 慢"。先优化索引/SQL（成本低），无效再分表（成本高）。

### 第三层：根因深挖

**Q：分片键你选"用户 ID"，但查询经常要"按笔记 ID 查"，跨分片怎么办？根因是什么？**

根因是"一个分片键无法同时满足多个查询维度"。按用户 ID 分片后，同一用户的笔记在同一分片（按用户查快），但"按笔记 ID 查"不知道笔记在哪个分片，要广播到所有分片（scatter-gather），N 个分片查 N 次合并，慢。解决方案：一、建立"映射表"——单独一个不分片的表存 `note_id → user_id` 映射，查笔记时先查映射拿 user_id，再路由到正确分片（两次查询但都走索引）；二、基因法——把 user_id 的某些 bit 嵌入 note_id 生成规则（如 note_id = user_id 后 4 位 + 时间戳 + 序列），从 note_id 能直接算出分片，无需映射表；三、双写索引——把笔记索引冗余到 ES 或独立分片（按 note_id 分），查询走 ES 定位再回主库取详情。这三种各有取舍，映射表简单但多一次查询、基因法巧妙但 note_id 生成有约束、双写索引强大但增加系统复杂度。按查询频率和延迟要求选。

**Q：那为什么不直接不分片，用"读写分离 + 索引优化"扛住？**

读写分离和索引优化能扛"读 QPS 高"的场景（加从库分担读），但扛不住"单表数据量大"的场景。读写分离的局限：一、写仍走主库——写 QPS 高时主库撑不住；二、单表数据量不变——即使读分散到从库，每个从库的表还是那么大，慢查询依旧；三、主从延迟——写后立即读可能读到从库旧数据。索引优化的局限：一、加索引有上限——不能给所有列建索引（写入变慢）；二、大表索引维护慢——INSERT 触发的索引页分裂随表增大变频繁。所以读写分离 + 索引优化解决"读压力大"，分库分表解决"单表数据量大"。两者是递进关系——先读写分离扛读 QPS，单表仍大再分表扛数据量。我的演进路径：单库→读写分离→分区表→分库分表，按瓶颈逐级升级。

### 第四层：方案权衡

**Q：分片键选用户 ID，但热门用户（如大 V）的数据量远超普通用户，单个分片会倾斜，怎么破？**

热点用户倾斜问题。方案：一、热点单独分片——识别大 V（如粉丝数 > 100 万），把他们的数据单独放到独立分片（甚至独立集群），不参与普通分片路由；二、子分片——单用户数据超大时，按二级维度（如时间）再切分，user_id + create_month 联合分片；三、冷热分离——大 V 的历史数据归档到冷存储（如 HBase），热数据留在主分片；四、读侧聚合——写仍按 user_id 分片，读用 ES 或缓存做二级索引聚合。我的实践：用"粉丝数阈值"识别大 V（动态调整），大 V 数据走专用分片 + 缓存层（Redis 缓存热门笔记列表），普通用户走常规分片。这是分片设计的"长尾处理"——不能假设数据均匀，要为热点预留方案。

**Q：为什么不用一致性哈希做分片，那样加减节点时数据迁移少？**

一致性哈希适合"节点数频繁变化 + KV 存储"（如 Memcached、Cassandra），但对关系数据库分库分表有几个不匹配：一、范围查询困难——一致性哈希的分布是无序的，范围查询要 scatter-gather 所有分片；二、事务跨分片——一致性哈希的数据位置不确定，跨分片事务难做；三、运维习惯——DBA 习惯"指定分片"做运维（如某分片单独备份），一致性哈希的动态分布让运维复杂。所以 MySQL 分库分表通常用"取模分片"（user_id % N）或"范围分片"（id 1-1000 万在分片 1），分片规则明确、运维简单。一致性哈希更适合"无中心 + 自动均衡"的分布式存储，不是关系数据库分片的主流。选型看场景，不是哪个更"先进"。

### 第五层：验证与沉淀

**Q：你怎么验证分库分表方案在数据增长、节点扩容场景下都 work？**

两类验证：一、数据分布——灌入测试数据（模拟千万用户、亿级笔记），检查各分片的数据量是否均衡（最大/最小分片数据比 < 1.5），倾斜说明分片键选错；二、查询正确性——跨分片查询（如"按笔记 ID 查"）应通过映射表或广播返回正确结果，单分片查询（如"按用户 ID 查")应只路由到一个分片（不广播）；三、扩容验证——从 4 分片扩到 8 分片，验证数据迁移脚本（按新分片规则 rehash）正确迁移且不丢不重，迁移期间双写（新旧分片都写）保证切换无损。线上监控：各分片的 QPS、数据量、慢查询数，任一分片 QPS 远超均值说明热点倾斜。扩容演练：每季度做一次"模拟扩容"，确保迁移流程 ready，不要等真扩容时才发现脚本有 bug。

**Q：这道题做完，你沉淀出了什么可复用的分库分表设计方法论？**

五步设计法：一、判断必要性——先读写分离 + 索引优化 + 分区表，无效再分库分表（成本高）；二、选分片键——选查询频率最高的维度（通常 user_id），避免跨分片查询；三、解决跨维度查询——映射表、基因法、双写索引三选一；四、处理热点——大 V 单独分片、子分片、冷热分离；五、预留扩容方案——分片数选 2 的幂（2/4/8/16），扩容时翻倍迁移量减半。这套方法论也适用于其他分片场景（ES 分片、Kafka 分区），核心是"按业务查询模式选分片键 + 为长尾预留方案"。面试时遇到"如何分库分表"，按这五步答，逻辑完整。


## 结构化回答

**30 秒电梯演讲：** 分库分表就是把一张大表按某个字段（分片键）拆到多个数据库的多张表中。分片键选得好，大部分查询都能定位到单个分片；选不好，查询要广播到所有分片。

**展开框架：**
1. **垂直拆分按业务/字段拆** — 垂直拆分按业务/字段拆，水平拆分按数据行拆
2. **分片键选user_id** — 分片键选user_id（高基数+查询频率高+不可变）
3. **分片算法** — Range(范围友好但可能热点)、Hash(均匀但扩容难)
