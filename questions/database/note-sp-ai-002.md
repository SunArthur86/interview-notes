# B树和B+树的区别？

## 结构对比图

```
B树 (B-Tree):
           [30|DATA]              ← 非叶子也有DATA
          /         \
    [10|D, 20|D]    [40|D, 50|D]  ← 每个key都带DATA
    /    |    \     /    |    \
  [5|D] [15|D] [25|D] [35|D] [45|D]
  
  特点：查找可能在非叶子命中返回


B+树 (B+Tree):
           [30]                    ← 非叶子只有KEY
          /     \
    [10, 20]     [40, 50]          ← 非叶子只有KEY(索引)
    /  |  \       /  |  \
  [5][10][15] ←→ [20][25] ←→ [30][35] ←→ [40][45][50]
                   ↑                     ↑
              叶子存DATA             叶子间链表连接
  
  特点：查找总是到叶子；叶子链表支持范围扫描
```

## 四大核心区别

| 区别 | B树 | B+树 |
|------|-----|------|
| **数据位置** | 所有节点都存key+data | 只有叶子存data |
| **查询稳定性** | 不稳定(可能中间命中) | 稳定(总是到叶子) |
| **范围查询** | 中序遍历(随机IO) | 叶子链表(顺序IO) |
| **fanout** | 小(key+data占空间) | 大(只key，每页更多) |

## 详细分析

### 1. 数据存储位置

```
B树：找key=10
  → 根节点[30]比较 → 左子节点[10,20] → 命中key=10 → 直接返回DATA
  → 可能只需2层就拿到数据

B+树：找key=10  
  → 根节点[30] → 左子节点[10,20] → 继续到叶子[5,10] → 返回DATA
  → 总是要走到叶子
```

### 2. 查询稳定性

```
B树：
  查key=10 → 第2层命中(快)
  查key=50 → 第3层命中(慢)
  → 响应时间不稳定

B+树：
  无论查什么key → 都到叶子节点
  → 响应时间稳定一致
  → 便于系统性能预估和SLA保证
```

### 3. 范围查询（关键优势）

```sql
-- 查询 age BETWEEN 20 AND 40

B树的范围查询：
  1. 找到age=20 → 获取data
  2. 中序遍历 → 回到父节点 → 下一个子节点
  3. 每次跳转 = 磁盘随机IO
  → 效率差(大量随机IO)

B+树的范围查询：
  1. 找到age=20的叶子节点
  2. 顺着链表向右扫描
  [20]→[22]→[25]→[28]→[30]→[35]→[40]→stop
  3. 顺序IO → 预读优化 → 极快
```

### 4. fanout与树高

```
假设磁盘页16KB，key=8B，指针=6B，data=1KB

B树每页：
  (key + data + 指针) = (8 + 1024 + 6) ≈ 1KB/条
  → 每页放 ~16 条
  → 100万数据树高 ≈ 4层 → 4次IO

B+树非叶子每页：
  (key + 指针) = (8 + 6) = 14B/条
  → 每页放 ~1170 条
  → 100万数据树高 ≈ 2层 → 2次IO(根节点常驻内存)
```

## 适用场景

| 数据结构 | 适用场景 | 代表产品 |
|---------|---------|---------|
| **B树** | 单点查询多、随机访问多 | MongoDB、文件系统 |
| **B+树** | 范围查询多、OLTP场景 | MySQL、PostgreSQL |

> MongoDB选B树的原因：MongoDB以单文档查询为主(WHERE _id=xxx)，很少做范围扫描。

## 面试加分点

1. **四点概括**：数据位置、查询稳定性、范围查询、fanout
2. **fanout计算**：能估算16KB页可放1170个key(只存key)
3. **范围查询是关键**：B+树叶子链表让范围扫描变顺序IO，这是数据库选B+树的核心原因
4. **MongoDB对比**：B树适合KV查询，B+树适合范围查询

## 记忆要点

- 数据分布：B树所有节点均存Key+Data，而B+树仅叶子节点存Data，非叶子只存索引Key
- 查询稳定性：B树查询可能在非叶子提前命中故不稳定，而B+树必须查到叶子节点故极其稳定
- 范围查询优势：B树中序遍历回溯产生随机IO，而B+树顺着叶子链表直接做顺序IO极速扫描
- Fanout差异：因为B+树非叶子不存Data，单页能容纳更多Key（约1170个），树高更矮，磁盘IO更少


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：B+树非叶子节点不存 data 你说是为了"fanout 大"，但 fanout 大为什么就查询快？链条关系是什么？**

链条：非叶子不存 data → 单页（16KB）放的 key 多 → fanout 大（约 1170）→ 树高降低（log_1170 N）→ 磁盘 IO 少。关键环节是"树高降低 → IO 少"。B+树每多一层，查询多一次磁盘 IO（根页常驻内存，中间页和叶子页可能要读盘）。N=千万数据：fanout=1170 时树高 3（IO≤3 次）；如果 fanout=100（像 B 树），树高 5（IO≤5 次）。SSD 单次 IO 约 100μs，3 次 IO=300μs，5 次 IO=500μs，差 40%。HDD 上差距更大（10ms/次，3 次 vs 5 次差 20ms）。所以 fanout 大 → 树矮 → IO 少 → 查询快，这是 B+树为磁盘优化的核心。内存数据库（Redis）不在乎 IO，所以用跳表（fanout=2）也行。

### 第二层：证据与定位

**Q：你怎么验证一个具体的 B+树索引的树高和 fanout？**

MySQL 没有直接"查看树高"的命令，但可推断：一、`SHOW INDEX FROM t`——看 `Cardinality`（基数，即索引中不同值的数量，估算值）；二、`EXPLAIN`——看 `rows` 列（估算扫描行数），结合 `key_len` 推断索引使用；三、`Innodb_buffer_pool_pages_*` 状态变量——看 buffer pool 里的页分布，`pages_index` 是索引页数。fanout 估算：页大小 16KB / (key 大小 + 指针 6 字节)。主键 bigint（8 字节）时，fanout ≈ 16KB / 14 ≈ 1170。树高估算：log_1170(N)，N=千万时约 3。精确测量要靠 `innodb_ruby` 或 `ibd2sdi` 工具解析 InnoDB 内部结构，工程上不常用，估算够用。

### 第三层：根因深挖

**Q：B+树叶子节点双向链表，为什么是"双向"不是"单向"？单向链表不能范围查询吗？**

单向链表支持正向范围查询（`ORDER BY id ASC`，从起点顺 next 指针扫）。双向链表额外支持反向范围查询（`ORDER BY id DESC`，从起点逆 prev 指针扫）。如果没有 prev 指针，反向查询要"先找到范围终点、再反向扫"，但单向链表无法从终点回溯，只能从 head 正向遍历到终点再反向——O(N) 退化。双向链表让正反向范围查询都 O(logN + K)（K 是结果数）。这是数据库支持 `ORDER BY ... ASC/DESC` 的基础。另外双向链表在节点删除时更高效（O(1) 找前驱），单向链表删除要 O(N) 找前驱。所以双向链表是为"双向范围 + 高效删除"设计的。

**Q：那为什么 B 树（非 B+树）也有叶子链表的说法？它和B+树的叶子链表有什么区别？**

严格定义里 B 树的叶子节点不连链表（B 树每个节点都存 data，叶子无特殊连接）。但有些教材的"B 树变种"（如 B+树的过渡形态）给叶子加了链表。标准 B 树（如 CLRS 教材定义）叶子无链表，范围查询要中序遍历回溯父节点。B+树叶子有链表，范围查询顺序扫。这是 B 树和 B+树的核心区别之一。实际工程中，B 树很少用于数据库（MongoDB 早期 WiredTiger 用 B 树变种，但也加了链表优化），主流数据库都用 B+树。所以面试时说"B 树叶子无链表、B+树叶子有链表"是标准答案，不要混淆。

### 第四层：方案权衡

**Q：B+树的非叶子节点只存 key，这意味着查询"必须到叶子"。为什么不学 B 树在非叶子也存 data，提前命中更快？**

B 树"非叶子存 data 提前命中"看似快，实际有两个问题：一、查询不稳定——非叶子命中是"运气"（数据恰好在根或中间节点），有时 1 次 IO 有时要到叶子，方差大；二、fanout 小——非叶子存 data 占空间，单页放的 key 少（如 fanout=100），树更高，平均 IO 反而多。B+树"必须到叶子"虽然单次查询都是 3 次 IO（到叶子），但稳定可预测（P99 可控），且 fanout 大（树矮）。数据库场景要的是"稳定的 P99"而非"偶尔快"，且磁盘 IO 是主要成本，B+树用"稳定的 3 次 IO"换 B 树的"不稳定的 1-5 次 IO"，整体更优。这是"均值 vs 方差"的工程取舍，数据库选低方差。

**Q：为什么不直接用 LSM 树（如 RocksDB、HBase 用）替代 B+树？LSM 写入不是更快吗？**

LSM 树（Log-Structured Merge Tree）写入是顺序追加（append-only），确实比 B+树的"页内插入+分裂"快。但 LSM 的代价：一、读放大——要查多层 SSTable + MemTable + BloomFilter，点查可能要多次 IO；二、空间放大——多版本数据 + compaction 前的冗余，空间占用大；三、compaction 开销——后台合并 SSTable 占用 CPU/IO，可能影响前台性能。B+树的读是稳定的 O(logN)（无读放大），写是原地更新（无 compaction）。所以 LSM 适合"写多读少"（如日志、时序数据），B+树适合"读多写少 + 通用查询"（如 OLTP）。MySQL 选 B+树是通用场景的稳妥选择，RocksDB/HBase 选 LSM 是写密集场景的针对性优化。没有"谁更好"，按读写比例选。

### 第五层：验证与沉淀

**Q：你怎么验证 B+树索引在范围查询时真的用叶子链表顺序扫，而非多次点查？**

EXPLAIN ANALYZE 看执行计划。`SELECT * FROM t WHERE id BETWEEN 1000 AND 2000`，EXPLAIN 应显示 type=range，rows≈1000（范围内行数），Extra 无 `Using index` 可能回表。关键是"rows"接近范围大小（而非全表行数），说明走了索引范围扫（叶子链表）。对比：如果索引失效（如 `WHERE id + 1 BETWEEN 1001 AND 2001`），type 变 ALL，rows=全表行数。性能对比：B+树范围扫 1000 行约 1-5ms（顺序 IO），失效后全表扫百万行秒级。用 `SHOW STATUS LIKE 'Handler_read_next'` 看顺序读次数（叶子链表扫会增加这个计数器），`Handler_read_rnd_next` 是随机读（回表会增加）。这些指标验证了"顺序扫 vs 随机扫"。

**Q：这道题做完，你沉淀出了什么可复用的索引优化经验？**

四条经验：一、建索引优先建"高选择性 + 覆盖查询列"的索引，避免回表；二、范围查询依赖 B+树叶子链表，所以范围列建索引有效（`WHERE create_time BETWEEN ...` 走索引）；三、排序（ORDER BY）走索引的有序性可避免 filesort，让排序列在索引里且顺序匹配；四、联合索引按"等值查询列→范围查询列→排序列"顺序建，最大化最左前缀利用。这套经验用于所有 SQL 优化，核心是"理解 B+树结构 → 推导索引行为 → 写匹配索引的 SQL"。不理解 B+树，索引优化就是瞎猜；理解了，每条索引的建与不建都有据可循。
