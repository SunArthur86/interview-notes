# MySQL里脏读和幻读分别是什么？数据库怎么避免？

## 四种并发问题与隔离级别

```
隔离级别安全性递增、性能递减 →

读未提交    读已提交    可重复读    串行化
(RU)       (RC)       (RR)       (Serializable)
  │          │          │           │
  │          │          │           │
脏读 ✅      脏读 ❌    脏读 ❌     脏读 ❌
不可重复读✅  不可重复读✅ 不可重复读❌ 不可重复读❌
幻读 ✅      幻读 ✅    幻读 ❌*    幻读 ❌

* InnoDB的RR级别通过MVCC+Next-Key Lock在大多数场景下也能避免幻读

InnoDB默认隔离级别: RR (可重复读)
```

## 脏读（Dirty Read）

```
事务A                        事务B
  │                           │
  ├── UPDATE balance=500 ──→  │  (未提交)
  │                    │      │
  │                    └── 读取balance=500  ← 脏读！
  │                           │
  ├── ROLLBACK ───────────────┘  (回滚)
  │
  ▼
balance恢复为1000，但事务B已经读到了500这个不存在的数据
```

**避免方式**：将隔离级别提升到至少RC（读已提交）
- RC级别下，事务只能读到其他事务已提交的数据
- InnoDB在RC级别使用MVCC，每次SELECT生成新的Read View

## 幻读（Phantom Read）

```
事务A                        事务B
  │                           │
  ├── SELECT WHERE age>20 ──  │  返回5行
  │                    │      │
  │                    │      ├── INSERT(age=25) ──→ COMMIT
  │                    │      │
  ├── SELECT WHERE age>20 ──  │  返回6行 ← 幻读！
  │                           │    （多了一行，像"幻影"一样出现）
  ▼
同一条件两次查询结果集行数不同，新出现的行就是"幻行"
```

**幻读 vs 不可重复读**：
- 不可重复读：同一行数据被修改（UPDATE）
- 幻读：结果集中出现了新行或少了行（INSERT/DELETE）

## InnoDB如何避免幻读

### 方案一：MVCC（快照读）

```sql
-- 普通SELECT是快照读，不会幻读
SELECT * FROM users WHERE age > 20;
-- → 读取事务开始时的快照数据
-- → 即使其他事务INSERT了新行，也看不到
```

**MVCC原理**：
```
事务A开始 → 创建Read View（记录当前活跃事务列表）
           │
           ├── SELECT → 从Undo Log中读取版本链
           │             找到对当前Read View可见的版本
           │
           └── 无论其他事务怎么INSERT/UPDATE
               事务A始终看到一致的数据快照
```

### 方案二：Next-Key Lock（当前读）

```sql
-- 当前读需要加锁防止幻读
SELECT * FROM users WHERE age > 20 FOR UPDATE;
-- → 锁定 age > 20 的所有现有行 + 间隙
-- → 其他事务无法在这个范围INSERT新行

-- UPDATE/DELETE也是当前读
UPDATE users SET status = 1 WHERE age > 20;
-- → 同样加Next-Key Lock
```

**Next-Key Lock = Record Lock + Gap Lock**：

```
已有数据: age = [18, 20, 25, 30]

SELECT WHERE age > 20 FOR UPDATE 加锁：

Record Lock: 锁定 age=25 和 age=30 的行
Gap Lock:    锁定 (20,25), (25,30), (30, +∞) 的间隙

合在一起 = Next-Key Lock:
(20, 25], (25, 30], (30, +∞]

效果：其他事务无法在 age>20 范围INSERT任何新行
```

## 隔离级别对比表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|---------|------|----------|------|------|
| Read Uncommitted | ✅发生 | ✅发生 | ✅发生 | 最高 |
| Read Committed | ❌避免 | ✅发生 | ✅发生 | 高 |
| Repeatable Read | ❌避免 | ❌避免 | ❌避免* | 中 |
| Serializable | ❌避免 | ❌避免 | ❌避免 | 最低 |

> *InnoDB的RR级别通过MVCC+Next-Key Lock在大多数场景避免了幻读，但不是SQL标准的严格要求。

## 面试加分点

1. **区分脏读和幻读**：脏读是读到未提交数据，幻读是结果集行数变化
2. **MVCC原理**：知道快照读通过Read View + Undo Log实现
3. **Next-Key Lock**：能解释Record Lock + Gap Lock的组合
4. **MySQL选RR的原因**：历史原因（binlog复制需要RR保证一致性），其他数据库大多默认RC

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：脏读和幻读你用"读到的是 UPDATE 还是 INSERT"区分，但本质上都是"读到不该读的数据"，为什么数据库要分两个概念？**

因为防御机制不同。脏读读到的是"未提交的修改"（其他事务 UPDATE 后还没 COMMIT），防御靠 MVCC（只读已提交版本）。幻读读到的是"新插入的行"（其他事务 INSERT 并 COMMIT），在同一事务内两次范围查询结果集行数不同（像幻影一样出现）。防御靠间隙锁（锁住"间隙"防止 INSERT）。两者本质区别：脏读是"同一行的数据被改"，幻读是"结果集的行数变了"。数据库分这两个概念，是因为 MVCC 解决不了幻读——MVCC 只能保证"读到的行内容一致"，但不能阻止新行被插入到查询范围内。所以 RR 隔离级别下，快照读靠 MVCC 防脏读+防不可重复读，当前读（FOR UPDATE）靠 Next-Key Lock 防幻读，两套机制分工。

### 第二层：证据与定位

**Q：你说 InnoDB 默认 RR，但很多人说"互联网公司用 RC（读已提交）"，你怎么决策？**

RC vs RR 的核心差异是"防不防幻读"和"binlog 格式"。RR 防幻读（间隙锁）但间隙锁会降低并发（锁住范围，INSERT 阻塞）；RC 不防幻读但并发更好。binlog 层面：RR 必须 ROW 格式（避免主从复制不一致），RC 可 STATEMENT 格式。互联网选 RC 的理由：一、间隙锁在并发场景易导致锁等待死锁，RC 无间隙锁更稳；二、多数业务不强依赖"可重复读"语义。选 RR 的理由：一、数据一致性要求高（如金融）；二、binlog ROW 格式更可靠。我的决策：默认 RC（并发优先），只有业务明确要求"事务内可重复读"或"防幻读"时才升 RR。但要注意 RC 下要业务层处理"不可重复读"（用乐观锁版本号）。

### 第三层：根因深挖

**Q：MVCC 怎么实现"快照读"？为什么 RR 下同一事务多次 SELECT 结果一致，而 RC 下每次 SELECT 结果可能变？**

MVCC 靠 Read View（读视图）实现。Read View 记录"当前活跃事务 ID 列表"，读取时根据行的 `trx_id`（最后修改该行的事务 ID）判断可见性：如果 trx_id 在活跃列表里（未提交）则不可见，顺着 `undo log` 链找上一个版本，直到找到可见版本。RR 和 RC 的差异在"Read View 生成时机"：RR 在事务第一次 SELECT 时生成 Read View，整个事务复用同一个（所以多次 SELECT 看到同一快照）；RC 在每次 SELECT 时都生成新的 Read View（所以能看到其他事务新提交的数据）。这是"快照保留时机"的设计选择——RR 牺牲实时性换一致性，RC 牺牲一致性换实时性。

**Q：那为什么不直接用"串行化"隔离级别，彻底解决所有并发问题？**

串行化（Serializable）把所有读加共享锁、写加排他锁，事务完全串行执行，并发度趋近于 1（一次只能一个事务）。在多核 CPU、高并发场景下，这是灾难性的性能退化。串行化的价值是"理论上的绝对正确"，但工程上几乎不用——它的性能代价（10-100 倍慢）远超它解决的一致性问题（其他隔离级别 + 业务层乐观锁能解决 99% 场景）。所以 Serializable 主要用于教学和极端严格的场景（如银行核心账务），互联网业务用 RR 或 RC + 业务幂等性。这是"理论正确 vs 工程可行"的经典权衡，不要为了"绝对正确"牺牲可用性。

### 第四层：方案权衡

**Q：RR 下当前读用 Next-Key Lock 防幻读，但间隙锁容易导致死锁，你怎么权衡？**

间隙锁死锁场景：事务 A 锁住 (10, 20) 间隙，事务 B 锁住 (15, 25) 间隙，A 要 INSERT 15（等 B 的间隙锁）、B 要 INSERT 12（等 A 的间隙锁），死锁。降低风险的策略：一、缩小查询范围——WHERE 条件越精确，间隙锁范围越小（用等值查询代替范围查询）；二、缩小事务——事务越短，持锁时间越短，死锁窗口越小；三、统一加锁顺序——所有事务按同一顺序访问表/行，避免交叉等待；四、监控 `innodb_row_lock_waits` 和 `innodb_deadlock_detect`，死锁频发时考虑降级到 RC（无间隙锁）。权衡：金融级一致性用 RR + 间隙锁 + 死锁检测重试，互联网高并发用 RC + 乐观锁。没有银弹，按业务选。

**Q：为什么不用应用层加锁（如 Redis 分布式锁）替代数据库的间隙锁？**

能，但有两个劣势：一、一致性弱——Redis 锁和 MySQL 数据不在一个系统，Redis 锁失效（过期、主从切换）时 MySQL 的并发保护就没了；二、复杂度高——要在每个查询前加锁、查询后释放，代码侵入大。数据库的间隙锁是"数据自带的并发控制"，跟事务绑定，ACID 保证强。所以"防幻读"首选数据库间隙锁（RR），只有跨数据源的并发（如 Redis + MySQL 双写）才用 Redis 分布式锁。但间隙锁的代价是并发降低，所以高并发场景倾向"RC + 应用层乐观锁（版本号）"——读取时拿版本号、写入时校验版本号，避免了数据库间隙锁的锁竞争。这是"数据库锁 vs 应用层锁"的典型选型。

### 第五层：验证与沉淀

**Q：你怎么验证隔离级别设置生效、防幻读真的 work？**

构造可复现的测试场景：开两个 MySQL 会话 A、B，隔离级别设 RR。会话 A 开事务 `BEGIN; SELECT * FROM t WHERE id BETWEEN 1 AND 10;`（假设返回 5 行）；会话 B `INSERT INTO t VALUES (5); COMMIT;`；会话 A 再次 `SELECT * FROM t WHERE id BETWEEN 1 AND 10;`——RR 下应仍返回 5 行（快照读防幻读），RC 下返回 6 行。当前读测试：会话 A `SELECT * FROM t WHERE id BETWEEN 1 AND 10 FOR UPDATE;`——RR 下会锁住间隙，会话 B 的 INSERT 应阻塞（等待间隙锁）。线上验证：`SHOW VARIABLES LIKE 'transaction_isolation'` 确认配置，监控 `innodb_row_lock_waits` 看间隙锁竞争情况，过高说明可能要降级 RC。

**Q：这道题做完，你沉淀出了什么可复用的事务隔离选型方法论？**

三步选型法：一、看一致性要求——金融/账务用 RR 或 Serializable，一般业务用 RC；二、看并发压力——高并发优先 RC（无间隙锁），低并发可用 RR；三、看业务幂等设计——无论哪个隔离级别，业务层都要有幂等性（版本号、唯一键）兜底，不依赖数据库锁做唯一防线。这套方法论我用于评审所有的事务设计——新人常犯的错是"无脑用 RR"或"无脑加 FOR UPDATE"，其实 80% 的场景 RC + 乐观锁就够了。隔离级别是工具，不是越严越好，要按场景匹配。
