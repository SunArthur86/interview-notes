# 【月之暗面面经】如果产品要扩到更多桌面能力，哪层前端架构最该先做成平台？

<!-- ANSWER_BODY_HERE -->

## 记忆要点

- 核心抽离：能力扩展本质是跨进程通信（IPC），所以最该先把 RPC/IPC 通信层平台化
- 统一标准：因原生与Web协议异构，所以需定义统一的消息协议与安全鉴权机制
- 能力注册：需做成插件化架构，新能力只需实现标准接口即可动态注册接入
- 资源调度：前端作为调度中枢，需平台化管理权限分配与多窗口的生命周期事件分发


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：平台化你优先做"任务编排层"（Task/Context/Product 统一模型），但为什么不优先做"UI 组件库"（更直观、团队都在用）？**

UI 组件库是"复用 UI"，任务编排层是"复用业务逻辑"。UI 组件库（如按钮、表单、表格）是"通用 UI"，各家组件库（如 Ant Design、Element）已经成熟，自研收益低（除非有特殊设计需求）。任务编排层是"AI 桌面产品的业务核心"——所有 Agent 能力（写作、分析、生成）都依赖任务编排（Task 状态机、Context 管理、Product 生命周期），如果不平台化，每个 Agent 各自实现任务管理，代码重复且不一致（如 A Agent 的任务管理有断点续传，B Agent 没有）。平台化任务编排层后，新 Agent 只需实现"业务逻辑"（如怎么生成 PPT），任务管理（状态机、断点续传、产物落盘）由平台提供，一致且可靠。所以"任务编排层是被依赖最广的公共底座"，优先平台化收益最大，UI 组件库用现成的即可。

### 第二层：证据与定位

**Q：团队说"平台化后开发变慢了"（要学平台 API、受平台约束），你怎么定位是平台设计问题还是团队适应问题？**

看两类证据：一、平台 API 设计——如果 API 复杂（如要配 10 个参数才能跑一个任务）、文档不全（团队不知道怎么用）、约束死板（如必须用平台的 Task 模型不能扩展），是平台设计问题；二、团队适应——如果 API 简洁清晰、文档齐全，但团队不用（如习惯老方式、嫌学习成本高），是适应问题。量化指标：一、上手成本——新 Agent 接入平台的平均耗时（如 1 天 vs 1 周），耗时长说明 API 复杂；二、平台使用率——新开发的 Agent 是否都用平台（如 80% 用平台 vs 20% 用平台），使用率低说明平台没吸引力。常见根因：平台设计过度抽象（为了通用牺牲易用）、约束太死（不能扩展）、文档差。修复：简化 API（提供合理默认值，减少必填参数）、补充文档和示例、支持扩展点（允许 Agent 定制）。

### 第三层：根因深挖

**Q：任务编排层平台化你抽象了 Task/Context/Product 模型，但"过度抽象"导致"什么都能做但什么都不好用"，怎么平衡通用性和易用性？**

过度抽象是平台化的常见陷阱。通用性（支持所有场景）和易用性（简单场景好用）是矛盾的：为支持所有场景，API 要灵活（很多配置项），简单场景反而繁琐（要填一堆配置）。平衡手段：一、合理默认值——API 提供默认值（如简单任务只需传 `taskType` 和 `input`，其他字段默认），复杂场景再覆盖配置（如自定义断点续传策略）；二、分层 API——底层 API（灵活但复杂）+ 高层 API（简单但受限），简单场景用高层 API（如 `runSimpleTask(input)`），复杂场景用底层 API（如 `runTask(config)`）；三、约定优于配置——常见场景有约定（如"写作 Agent 的 Task 模型是 ..."），按约定就少配置。核心："默认好用（简单场景）+ 可配置（复杂场景）"，而非"一切可配置"（过度抽象）。借鉴 Rails 的"约定优于配置"和 Spring Boot 的"自动配置"。

**Q：那为什么不直接为每个 Agent 定制任务编排（不过度抽象），简单直接？**

定制的问题：代码重复和不一致。每个 Agent 定制任务编排，意味着：一、代码重复——5 个 Agent 各自实现任务状态机、断点续传、产物落盘，重复代码 5 份，维护成本 5 倍；二、不一致——A Agent 的任务管理有断点续传，B Agent 没有（开发者忘了或实现不同），用户体验不一致（A 失败可续跑，B 失败从头跑）；三、难演进——想给所有 Agent 加"任务监控"功能（如日志采集），要改 5 个 Agent 的代码，而非改 1 个平台。平台化的价值：一、代码复用——任务管理逻辑只写一份，所有 Agent 复用；二、一致性——所有 Agent 的任务行为一致（都有断点续传、都有日志）；三、易演进——新功能（如监控）加在平台，所有 Agent 自动获得。所以"适度抽象的平台"优于"各自定制"，关键是平衡通用性和易用性（合理默认 + 分层 API）。

### 第四层：方案权衡

**Q：平台化你用"插件化架构"（Agent 作为插件接入），但为什么不直接用"单体架构"（所有 Agent 代码在一个项目），简单？**

单体架构在 Agent 数量少时简单（如 2-3 个 Agent），但 Agent 增多后变臃肿：一、代码耦合——所有 Agent 代码在一起，改一个可能影响其他（如共用全局状态、代码冲突）；二、编译慢——所有 Agent 一起编译，加一个 Agent 编译时间变长，开发体验差；三、无法独立扩展——第三方无法开发 Agent（所有代码在主项目），失去生态扩展能力。插件化架构解决：一、解耦——每个 Agent 是独立插件，代码隔离，改一个不影响其他；二、按需编译/加载——插件独立编译、按需加载，主项目轻量；三、可扩展——第三方开发插件（如 VS Code 插件生态），产品能力无限扩展。所以"Agent 数量少时单体简单，增多后插件化必要"，AI 桌面产品要扩更多 Agent，插件化是必然选择。

**Q：为什么不直接用现成的插件框架（如 VS Code Extension API、Figma Plugin API），而要自研平台？**

现成框架的问题：一、领域不匹配——VS Code Extension API 是为"代码编辑器"设计的（如语法高亮、代码补全），AI 桌面产品的 Agent 需求不同（如多步任务编排、产物管理、AI 调用），套用 VS Code API 要大量改造；二、技术栈绑定——VS Code Extension 用特定技术栈（如其 UI 框架），与团队技术栈（如 Vue/React）可能不符；三、控制权——用别人的框架受其演进影响（如 API 变更、停止维护），自研可控。自研平台的优势：一、领域匹配——专为 AI 桌面 Agent 设计（Task/Context/Product 模型）；二、技术栈自由——用团队熟悉的 Vue/React；三、可控——演进自主。代价是开发成本（自研要投入）。所以"借鉴现成框架的设计理念（插件化、注册表、权限），但自研实现匹配领域"，而非直接套用。

### 第五层：验证与沉淀

**Q：你怎么验证平台化真的提升了开发效率（而非只是"看起来先进"）？**

量化指标：一、新 Agent 接入耗时——平台化前后，接入一个新 Agent 的平均耗时（如平台化前 2 周，平台化后 3 天），显著下降说明效率提升；二、代码复用率——平台化后，Agent 代码中"平台提供"的比例（如 70% 是平台逻辑，30% 是 Agent 业务），高复用说明平台有效；三、一致性指标——所有 Agent 是否有一致的能力（如都有断点续传、日志），平台化前可能 50% Agent 有断点续传，平台化后 100%，一致性提升；四、第三方插件——是否有第三方（或团队外成员）能独立开发 Agent（如实习生 1 周开发一个 Agent），说明平台标准化达标。

**Q：这道题沉淀出什么可复用的平台化设计经验？**

四条原则：一、优先平台化"公共底座"——任务编排层（Task/Context/Product）被依赖最广，优先平台化，UI 组件库用现成的；二、平衡通用性和易用性——合理默认值 + 分层 API + 约定优于配置，避免过度抽象（什么都能做但都不好用）；三、插件化架构——Agent 作为插件（解耦、按需加载、可扩展），借鉴 VS Code 但自研匹配领域；四、效率验证——新 Agent 接入耗时下降、代码复用率高、一致性强，才是真效率提升。核心洞察："平台化本质是'提取公共不变量'——借鉴操作系统内核（稳定底座 + 应用生态），AI 桌面产品的'内核'是任务编排层，平台化后新增能力只需实现业务逻辑，核心是'稳定底座 × 插件化扩展'。"
