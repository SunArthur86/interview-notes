# 【月之暗面面经】本地索引、云端推理和产物导出是三段链路，前端怎样串起来？

<!-- ANSWER_BODY_HERE -->

## 记忆要点

- 分层解耦架构：前端通过统一任务调度层串联三段链路，不相互阻塞
- 因为异步链路耗时久，所以必须有全生命周期的状态机（执行/等待/完成）
- 产物导出需防丢失，前端应拦截退出并做断点续传机制


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：三段链路（本地索引→云端推理→产物导出）在不同环境执行，为什么前端要用统一状态机串联，而不是每段各自管理状态？**

各自管理状态会导致"状态断层"。如本地索引完成自己的状态机（done），但云端推理不知道（还在等本地索引完成的信号），链路卡住。统一状态机的价值：一、全局视角——一个状态机管理整个链路（如 `indexing → indexed → inferring → inferred → exporting → done`），前端随时知道"任务到哪一步"；二、流转驱动——状态转换触发下一段（如 indexed 触发云端推理调用），自动流转不依赖人工；三、错误恢复——失败时状态机记录"在哪一步失败"，从该步骤重试（如 inferring 失败，重试 inferring 而非从头 indexing）。所以统一状态机是"链路的编排器"，借鉴工作流引擎（如 Airflow）的 DAG + 状态机思路。

### 第二层：证据与定位

**Q：用户说"任务卡在云端推理半天没响应"，你怎么定位是网络问题、云端 API 问题还是前端状态机 bug？**

分段定位：一、网络——前端 DevTools 的 Network 面板看推理 API 的请求状态（pending 说明网络不通或服务端未响应，timeout 说明超时）；二、云端 API——如果有响应，看响应状态码和 body（如 429 限流、500 服务端错、200 但 body 错）；三、状态机——看前端状态机的当前状态（如停在 inferring），检查是否有状态转换逻辑 bug（如收到 API 响应但没触发 inferring → inferred 转换）。常见根因：API 调用没设 timeout（网络断开时一直 pending）、流式响应中断未处理（SSE 断开前端没检测到）、状态机转换条件写错（如等错的字段）。

### 第三层：根因深挖

**Q：云端推理你用流式返回（SSE），但流式中途断了（如 30 秒后连接断开），前端怎么处理才能不丢已有结果？**

流式中断处理：一、检测断开——SSE 的 onerror 事件或 fetch 的 ReadableStream 中断检测，断开时记录已收到的部分结果（如已收到 70% 的推理结果）；二、断点续传——向云端发请求"从断点继续"（如传已收到的最后一个 chunk id），云端从该位置继续推（需要云端支持）；三、降级——如果云端不支持断点续传，用已收到的部分结果（如 70% 推理结果）展示给用户，提示"结果不完整，可重新生成"；四、自动重试——短暂断开（如网络抖动）自动重连（指数退避，如 1s/2s/4s），超过阈值（如 3 次）才降级。关键：已收到的结果要落盘（不能只在内存），断开后前端能恢复已收到的部分。

**Q：那为什么不直接用一次性返回（非流式），等云端推理完再返回，避免流式中断的复杂处理？**

一次性返回的问题：一、延迟感知差——长任务（如 30 秒推理）用户看到的是空白等待，不知道进度，体验差；二、超时风险——长请求易触发网关超时（如 nginx 默认 60s）、浏览器超时，请求失败；三、资源占用——服务端要持有连接 30 秒，并发高时连接数爆炸。流式返回解决：一、进度感知——逐 chunk 返回，前端实时展示（如"正在分析..."），用户知道在工作；二、避免超时——SSE 是持续推送，不触发请求超时；三、断点续传——流式天然支持"收到哪算哪"。所以流式是长推理任务的标配，中断处理是必要代价。短推理（< 5 秒）可用一次性返回，简单。

### 第四层：方案权衡

**Q：本地索引你用全量扫描（首次扫描所有文件建索引），但用户文件多（如 10 万文件）首次扫描几分钟，为什么不增量扫描（只扫变化的文件）？**

首次必须全量（建基线），后续增量。首次全量扫描慢（10 万文件几分钟），但只发生一次，之后用增量扫描：一、文件监听——用 OS 的文件变更通知（如 macOS 的 FSEvents、Windows 的 ReadDirectoryChangesW）或定期扫描（如每 5 分钟扫一次 mtime 变化的文件）；二、增量更新——只对变化的文件重新索引（如新文件添加、修改文件重新向量化、删除文件移除索引），扫描量从 10 万降到几十；三、惰性索引——首次扫描只建文件列表（快），向量化在用户首次查询该文件时才做（按需），避免首次全量向量化。所以"首次全量建基线，后续增量更新，按需向量化"是平衡首次速度和后续效率的方案。

**Q：为什么不把索引完全放云端（用户文件上传云端建索引），前端不用管本地索引？**

云端索引的问题：一、隐私——用户文件（尤其敏感文档）上传云端有隐私顾虑，AI 桌面产品强调"本地优先"（如 Kimi、Notion AI 都强调本地数据处理）；二、带宽——10 万文件上传（可能几十 GB），首次同步耗时且耗流量；三、离线——无网络时无法索引（用户离线场景不能用）；四、成本——云端存储和索引计算要付费。本地索引的优势：隐私（数据不出本地）、离线可用、无带宽成本。云端推理是"必要"（大模型在云端），但索引应本地（数据隐私 + 离线）。所以"本地索引 + 云端推理"是隐私和能力的平衡，而非全云端。

### 第五层：验证与沉淀

**Q：你怎么验证三段链路的串联设计可靠（不丢任务、不卡住、错误可恢复）？**

故障注入测试：一、网络中断——在 indexing/inferring/exporting 各阶段模拟网络断开（如用 Charles 断网），验证链路是否能检测断开、自动重试、降级提示；二、云端 API 错误——mock 云端返回 500/429/超时，验证重试逻辑和用户提示；三、本地崩溃——任务执行中杀进程，重启后验证是否能从 Checkpoint 恢复；四、大文件压力——上传大文件（如 1GB）测试索引和导出的稳定性。线上监控：任务成功率（应 > 99%）、各阶段耗时分布（P50/P99）、失败任务的重试成功率。核心："链路可靠性靠故障注入测试 + 线上监控验证，不是'理论上应该可靠'"。

**Q：这道题沉淀出什么可复用的三段链路编排经验？**

四条原则：一、统一状态机——一个状态机管理整个链路（indexing→inferring→exporting→done），驱动流转和错误恢复；二、流式 + 断点续传——长推理用流式返回，检测中断、已收结果落盘、支持续传或降级；三、首次全量 + 后续增量——本地索引首次全量建基线，后续增量更新（文件监听 + mtime），按需向量化；四、本地优先——索引本地（隐私、离线、省带宽），推理云端（大模型能力），不全云端。核心洞察："三段链路本质是'分布式工作流'——借鉴 Airflow 的 DAG + 状态机，前端是编排中心，处理数据流向、状态流转、错误恢复，让用户感知到的是一个流畅任务而非三个割裂步骤。"

## 结构化回答

**30 秒电梯演讲：** 就像工厂流水线——原材料入库(本地索引)→车间加工(云端推理)→成品出库(产物导出)，前端是调度中心管理整个流程。

**展开框架：**
1. **本地索引** — 文件扫描/向量化/缓存
2. **云端推理** — API调用/流式返回/重试
3. **产物导出** — 格式转换/保存/通知
