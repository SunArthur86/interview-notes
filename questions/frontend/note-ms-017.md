# 【月之暗面面经】你如何判断候选人真的做过 AI-Native 桌面产品，而不是普通 Web 前端？

## 一、核心判断框架

判断候选人是否真正做过 AI-Native 桌面产品，核心不是看他用了什么技术栈，而是看他是否具备**系统级思维 × AI 交互设计能力**的交叉经验。

普通 Web 前端和 AI-Native 桌面前端的本质差异：

| 维度 | 普通 Web 前端 | AI-Native 桌面前端 |
|------|-------------|-------------------|
| **运行环境** | 浏览器沙箱 | 操作系统级（文件系统/进程/权限） |
| **交互模式** | 请求-响应 | 流式输出/多轮对话/Agent自主执行 |
| **产出物** | 页面渲染 | 代码/文档/文件等可编辑的真实产物 |
| **失败处理** | HTTP 错误码 | AI 幻觉/超时/部分正确/中途失败 |
| **状态管理** | UI 状态 | UI状态 + AI执行状态 + 文件系统状态 |
| **性能关注** | 页面加载/渲染 | 流式渲染 + 长任务内存 + 原生交互 |
| **安全边界** | XSS/CSRF | AI 注入/文件越权/敏感数据泄露 |

**一句话总结**：做过 AI-Native 桌面的人，会自然地谈论文件路径、系统权限、AI 产物的编辑确认流程、Agent 执行链路的失败恢复——这些是 Web 聊天框经验完全覆盖不到的。

## 二、六维评估体系

### 评估维度总表

| 评估维度 | 权重 | 核心考察点 | 区分信号 |
|---------|------|-----------|---------|
| **① 桌面工程能力** | 20% | 系统集成/原生交互/性能 | 能否谈论 Electron/Tauri 底层 |
| **② AI 交互设计能力** | 25% | 流式/中断/确认/产物管理 | 能否设计非聊天框的 AI 交互 |
| **③ Agent 执行链路** | 20% | 多步骤编排/失败恢复 | 是否做过非即时的 AI 任务流 |
| **④ 产物管理能力** | 15% | 编辑-确认-发布-版本 | 是否处理过 AI 产出物的生命周期 |
| **⑤ 系统级安全思维** | 10% | 权限/隔离/敏感数据 | 能否识别 AI 场景特有的安全风险 |
| **⑥ 工具链与可观测性** | 10% | 链路追踪/指标/调优 | 是否做过 AI 相关的前端可观测 |

### 各维度详细评估标准

#### ① 桌面工程能力（权重 20%）

| 信号级别 | 表现 |
|---------|------|
| 🟢 强信号 | 讨论过 IPC 通信优化、多窗口管理、原生菜单/快捷键集成、自动更新；知道 Electron 主进程/渲染进程的性能瓶颈 |
| 🟡 中信号 | 用过 Electron/Tauri 但主要做 UI 封装，很少触及系统级 API |
| 🔴 弱信号 | 只做过浏览器内应用，对桌面端进程模型无概念 |

#### ② AI 交互设计能力（权重 25%）

| 信号级别 | 表现 |
|---------|------|
| 🟢 强信号 | 能讨论流式输出的前端渲染策略、中断/恢复 UX、AI 置信度可视化、渐进式确认、降级体验设计 |
| 🟡 中信号 | 做过聊天框式 AI 对话 UI，但没深入想过流式渲染性能和交互问题 |
| 🔴 弱信号 | 认为做 AI 前端就是对接 API + 渲染 Markdown |

#### ③ Agent 执行链路经验（权重 20%）

| 信号级别 | 表现 |
|---------|------|
| 🟢 强信号 | 做过多步骤 AI 任务（代码生成+测试运行+修错循环）、讨论过执行链路的暂停/恢复/回滚、分支管理 |
| 🟡 中信号 | 做过简单的 AI 对话，但对多步骤 Agent 执行无经验 |
| 🔴 弱信号 | AI 经验限于一问一答的 API 调用 |

#### ④ 产物管理能力（权重 15%）

| 信号级别 | 表现 |
|---------|------|
| 🟢 强信号 | 设计过 AI 产物的编辑-确认-保存-版本管理-导出完整流程；处理过产物与本地文件系统的同步 |
| 🟡 中信号 | 展示过 AI 输出但没有后续管理流程 |
| 🔴 弱信号 | 未思考过 AI 产物生成后的编辑和持久化问题 |

## 三、深度问题示例库

以下问题是区分"做过"和"没做过"的利器。每个问题背后考察的是候选人是否遇到过真实工程问题。

### 第一组：桌面工程 × AI 交互（系统级思维）

**Q1: "你的 AI 桌面产品中，AI 生成的代码/文件是如何写入用户本地文件系统的？这个过程涉及哪些用户确认和安全考虑？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论了：①写文件前预览 diff → 用户确认 → 写入；②文件写入的事务性（写入失败回滚）；③路径安全（防止路径穿越攻击）；④多文件修改时的原子性；⑤增量写入 vs 全量覆盖的策略 |
| 🟡 一般 | 提到了文件保存功能，但没讨论确认流程和安全问题 |
| 🔴 不足 | 只说"调用 fs 模块写文件"，没想过用户确认和安全 |

**Q2: "Electron/Tauri 中，AI 流式输出大量文本时，主进程和渲染进程之间的 IPC 通信怎么优化？遇到过什么瓶颈？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论：①IPC 消息频率限制（throttle/batch）；②大文本分段传输；③Web Worker 处理渲染减轻主线程压力；④内存管理——长时间对话的消息清理策略；⑤虚拟列表渲染长对话 |
| 🔴 不足 | 没考虑过 IPC 频率问题，或以为流式输出就是 WebSocket |

**Q3: "用户在 AI 执行长任务（如分析整个代码库）时关闭了应用窗口，重新打开后怎么恢复？"**

| 回答质量 | 典型回答 |
|---------|------|
| 🟢 优秀 | 讨论了：①任务状态持久化到本地（IndexedDB/SQLite）；②重新打开时检测未完成任务；③断点续传 vs 从头重跑的选择；④后台进程保活策略（Electron 的 tray 后台运行）；⑤用户感知——恢复时的 UI 状态还原 |
| 🔴 不足 | 只说"重新发一次请求"，没有状态恢复概念 |

### 第二组：AI 交互设计能力

**Q4: "AI 回答到一半，用户想中断重新提问，你的中断 UX 是怎么设计的？技术上怎么实现？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论了：①中断按钮的时机和位置（不能太显眼影响阅读，也不能太隐蔽）；②中断后已生成内容的保留/丢弃策略；③中断的技术实现——AbortController/fetch cancel；④中断后快速重新发送的 UX；⑤服务端 SSE 连接清理 |
| 🟡 一般 | 知道需要中断功能，但 UX 设计和技术实现都比较粗糙 |
| 🔴 不足 | 没想过这个问题，或只说"刷新页面" |

**Q5: "AI 输出了一段代码，但其中有 2 处错误，你的产品怎么让用户发现和修正这些错误？"**

| 回答质量 | 典型回答 |
|---------|------|
| 🟢 优秀 | 讨论了：①逐行/逐块的 inline diff 视图；②用户可选择性采纳部分代码（不需要全盘接受或全盘拒绝）；③在线编辑后实时 lint；④错误高亮和自动修复建议；⑤多版本对比（原始 vs 用户修改后） |
| 🔴 不足 | 只想到"展示代码 + 用户自己复制去改" |

**Q6: "你怎么处理 AI 输出的不确定性？比如同一个问题，AI 有时候返回很好，有时候很差。"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论：①置信度展示和低置信度降级；②"再试一次"的快捷操作；③多轮自校验（AI 自检 → 发现问题 → 自动修正）；④用户反馈闭环（👎 → 收集 → 模型优化）；⑤不同温度参数的 A/B 实验 |
| 🔴 不足 | 认为不确定性是后端问题，前端只负责展示 |

### 第三组：Agent 执行链路

**Q7: "描述一下你做过的最复杂的 AI Agent 执行链路。它有多少步？中间某步失败了怎么处理？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 能具体描述：①多步骤链路（如"读取文件→分析结构→生成代码→运行测试→根据测试结果修错→输出"）；②每步的输入/输出格式；③失败重试策略（哪些步骤可重试，哪些需要用户介入）；④超时处理；⑤执行过程的可视化（进度条/步骤展示）；⑥回滚机制 |
| 🔴 不足 | AI 经验限于一问一答，没有多步骤链路概念 |

**Q8: "Agent 自主执行时修改了用户的文件，但用户发现修改是错的，怎么办？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论了：①修改前的自动备份/git stash；②一键回滚功能；③修改 diff 的可视化预览（修改前确认 vs 先修改后回滚的选择）；④部分回滚（只撤销部分修改）；⑤学习机制——用户拒绝的修改模式分析 |
| 🔴 不足 | 只说"用户自己手动改回来" |

### 第四组：产物管理与版本

**Q9: "用户用 AI 生成了一个文档，然后手动改了一些内容，再让 AI 补充，怎么处理 AI 改动和用户改动的冲突？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论了：①版本追踪（AI 版本 vs 用户版本 vs 混合版本）；②冲突检测和解决策略；③基于 CRDT/OT 的协同编辑思路；④AI 续写时的上下文感知（知道哪些是用户改的）；⑤Lock 机制——锁定用户正在编辑的区域 |
| 🔴 不足 | 没想过 AI 和用户编辑的冲突问题 |

### 第五组：系统级安全思维

**Q10: "AI Agent 有权限读写用户的文件系统，你怎么防止 AI 被 prompt injection 后执行危险操作？"**

| 回答质量 | 典型回答 |
|---------|---------|
| 🟢 优秀 | 讨论了：①操作白名单（AI 只能操作工作区目录内文件）；②危险操作二次确认（删除/覆盖/移动）；③敏感文件检测（.env、密钥文件）；④AI 操作审计日志；⑤沙箱隔离——在容器/虚拟环境中执行 AI 的文件操作 |
| 🔴 不足 | 没意识到 prompt injection 可以导致文件系统风险 |

## 四、面试流程设计

### 4.1 推荐面试节奏（60 分钟）

```
Phase 1: 简历深挖（15分钟）
├── 聚焦桌面产品项目的系统级细节
├── 问"你做了什么"→ 追问"为什么这样做"
└── 信号：能否说出具体的技术决策和 trade-off

Phase 2: 开放式设计题（25分钟）
├── 抛出真实场景（如"设计一个AI编程助手的产物管理流程"）
├── 观察候选人自发讨论的维度
└── 信号：是否主动涉及文件/权限/确认/版本/恢复

Phase 3: 深度追问（15分钟）
├── 从上面问题库中挑 2-3 个匹配候选人经验的问题
├── 层层追问直到触及候选人能力边界
└── 信号：在边界处是坦诚说"没做过"还是泛泛而谈

Phase 4: 反向提问（5分钟）
├── "你对 AI 桌面产品有什么看法？"
└── 信号：是否有自己的思考和观点
```

### 4.2 评分标准

```yaml
评分维度:
  桌面工程:
    1分: 无桌面开发经验
    3分: 用过Electron/Tauri，做过基本桌面功能
    5分: 深入理解桌面架构，能优化IPC/内存/启动性能

  AI交互设计:
    1分: 只做过API对接+渲染
    3分: 做过聊天式AI，了解流式渲染
    5分: 设计过非聊天式的AI交互，有完整的确认/中断/降级/产物管理设计

  系统级思维:
    1分: 只关注UI层
    3分: 考虑过文件/权限等系统交互
    5分: 能自发讨论系统级约束与AI能力的交叉问题

  Agent链路:
    1分: 无多步骤AI经验
    3分: 做过简单的多轮对话
    5分: 设计过复杂Agent执行链路，有失败恢复/分支管理经验

综合判断:
  ≥ 16分 (4×4+): 强烈推荐——真正做过AI-Native桌面
  12-15分: 值得考虑——有潜力但某维度有gap
  < 12分: 不推荐——AI桌面经验不足
```

## 五、常见误区与红旗信号

### 候选人的红旗信号（🚩）

| 红旗信号 | 含义 |
|---------|------|
| 把 AI 前端等同于"对接 LLM API + 渲染 Markdown" | 没做过深度 AI 交互 |
| 说不清 AI 产物的编辑/保存/版本管理流程 | 没做过产物管理 |
| 没想过 AI 任务中途失败的处理 | 没做过 Agent 链路 |
| 认为流式输出只是"打字机效果" | 没深入想过流式渲染工程 |
| 不知道 prompt injection 对前端的影响 | 缺乏 AI 安全思维 |
| 只谈模型选型和 prompt 工程，不谈前端架构 | 可能是 AI 后端而非 AI 前端 |

### 面试官的常见误判

| 误判 | 纠正 |
|------|------|
| "做过聊天机器人就是做过 AI 前端" | 聊天机器人 ≠ AI-Native 桌面产品 |
| "用了 LangChain/OpenAI SDK 就够了" | 关键是交互设计和系统级思维，不是 SDK 使用 |
| "会用 React 就是前端能力达标" | AI 桌面前端的难点在 AI 交互和系统集成，不在 UI 框架 |

## 六、总结

判断候选人是否真正做过 AI-Native 桌面产品，**核心标志是：候选人能否自发地讨论系统级约束（文件/权限/进程/安全）与 AI 能力（流式/Agent/产物/不确定性）交叉产生的工程问题**。

这些交叉问题——文件写入的事务性、AI 修改的回滚机制、长任务的状态恢复、流式渲染的内存管理、产物的版本冲突——只有真正做过的人才会遇到，也只有在实践中才能形成深度理解。面试的终极区分方法就是问这些问题：**做过的人会两眼发光地讲细节，没做过的人只能泛泛而谈或者沉默**。

面试的本质不是考知识面，而是通过深度问题把候选人带到他的能力边界，然后看边界在哪里。AI-Native 桌面产品是一个新兴领域，真正有深度经验的人不多——**找到那些在交叉地带有真实思考的人，他们才是团队最需要的。**
