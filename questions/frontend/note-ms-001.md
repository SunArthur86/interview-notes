# 【月之暗面面经】AI-Native 桌面产品为什么不能只照搬 Web 聊天页？

## 一、问题的本质：Web 聊天页的假设在桌面端全部失效

Web 聊天页的设计隐含了四条假设，而桌面端把它们全部打破了：

| 隐含假设 | Web 聊天页 | 桌面 AI 产品 |
|---|---|---|
| 上下文来源 | 只有用户打字的 prompt | 本地文件、目录、剪贴板、系统窗口、其他 App |
| 会话生命周期 | 一次对话 = 一个 tab | 多任务并行、后台长任务、跨窗口拖拽 |
| 产物形态 | Markdown 文本 / 代码块 | 落地为本地文件、网站、表格、PPT、工程目录 |
| 系统能力 | 浏览器沙箱隔离 | 原生文件读写、进程间通信、系统托盘、全局快捷键 |

照搬 Web 聊天页意味着：用户每次都要手动把文件内容粘贴进对话框，产物只能复制到剪贴板再另存，长任务没有窗口承载——这是把"桌面"降级成"浏览器壳"，白白浪费了桌面端最值钱的能力。

## 二、架构对比：消息流 vs 任务—上下文—产物

### 2.1 Web 聊天页的架构（消息流为中心）

```
┌──────────────────────────────────────────┐
│              消息流（单一时间线）            │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  │
│  │ User │→ │  AI  │→ │ User │→ │  AI  │  │
│  └──────┘  └──────┘  └──────┘  └──────┘  │
│                                            │
│  状态：conversation[]  渲染：单列滚动        │
│  上下文：仅对话历史                         │
│  产物：嵌在 AI 消息里的代码块/文本            │
└──────────────────────────────────────────┘
```

### 2.2 AI-Native 桌面产品的架构（任务—上下文—产物为中心）

```
┌────────────────────────────────────────────────────────────────────┐
│                        桌面 AI 工作台                                │
│                                                                     │
│  ┌──────────┐   ┌──────────────────────┐   ┌──────────────────┐   │
│  │ 任务侧栏  │   │     主工作区           │   │   产物面板        │   │
│  │          │   │                      │   │                  │   │
│  │ Task #1  │   │  ┌──────────────────┐ │   │  📊 表格 v2      │   │
│  │ Task #2  │   │  │ 上下文区          │ │   │  🌐 站点预览     │   │
│  │ Task #3  │   │  │ · 本地文件 / 目录  │ │   │  📄 PPT v1      │   │
│  │ ─────── │   │  │ · 剪贴板 / 网页    │ │   │  📁 工程目录     │   │
│  │ + 新任务 │   │  └──────────────────┘ │   │                  │   │
│  │          │   │  ┌──────────────────┐ │   │  每个产物：       │   │
│  │ ⚙️ 设置  │   │  │ 对话/指令区       │ │   │  · 预览          │   │
│  │ 🔔 通知  │   │  │ (消息流降级为       │ │   │  · 局部重生成    │   │
│  │          │   │  │  任务内的子模块)    │ │   │  · 版本管理      │   │
│  │          │   │  └──────────────────┘ │   │  · 导出到本地    │   │
│  └──────────┘   └──────────────────────┘   └──────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────── │
│  │ 桌面系统能力层：文件读写 · 窗口管理 · 全局快捷键 · 系统托盘 · IPC │
│  └─────────────────────────────────────────────────────────────── │
└────────────────────────────────────────────────────────────────────┘
```

**关键差异**：消息流从"主角"降级为任务内的一个子模块，任务侧栏和产物面板才是桌面端的组织骨架。

## 三、具体 UX 差异逐项拆解

### 3.1 上下文输入：从"粘贴文本"到"拖拽接入"

| 维度 | Web 聊天 | 桌面 AI |
|---|---|---|
| 文件输入 | 手动上传，有大小/类型限制 | 拖拽文件/目录直接接入，支持整个工程目录 |
| 上下文持久化 | 每次重新粘贴 | 文件链接持久挂载，变更自动感知（file watcher） |
| 多源混合 | 只能文本拼接 | 文件 + 剪贴板 + 截图 + 当前窗口上下文同时存在 |

**UX 设计要点**：桌面端需要一个持久的"上下文区"（Context Panel），让用户始终看到 AI 当前能访问哪些文件/目录，而不是每次对话前重新贴。

### 3.2 窗口与多任务：从"单 tab"到"多任务并行"

Web 聊天的会话是线性的，开多个 tab 也是互相独立的孤岛。桌面端天然支持：

- **多窗口**：不同任务可拆窗，比如左边写代码、右边让 AI 跑数据分析
- **后台长任务**：Agent 跑一个 10 分钟的站点生成，用户可以切到其他任务，完成后通知
- **任务切换**：任务侧栏 + Cmd+Tab 风格的快速切换，每个任务独立保留上下文

### 3.3 产物：从"消息内文本块"到"可操作的文件对象"

这是最核心的差异。Web 聊天里 AI 生成的代码只能复制，桌面端则要：

- **预览**：站点直接渲染、表格内联展示、PPT 缩略图翻页
- **落地**：一键导出到本地文件系统，保留目录结构
- **继续编辑**：产物不是终态，用户可以选中局部（比如某一页 PPT、某一列数据）让 AI 重生成
- **版本管理**：每次重生成保留版本历史，可回退

### 3.4 系统级入口：从"打开浏览器"到"无处不在"

桌面端可以利用的系统入口包括：

- **全局快捷键**：任意 App 中划词 → 快捷键 → 唤起 AI 处理选中内容
- **系统托盘 / 菜单栏**：常驻后台，随时唤起
- **文件右键集成**：右键本地文件 → "用 AI 分析"
- **拖拽到 Dock 图标**：拖文件到 App 图标直接进入任务

这些入口意味着 AI 的触发点是"碎片化 + 系统级"的，不能强制用户先进一个聊天页面。

## 四、前端架构层面的核心调整

### 4.1 状态模型：从 `conversation[]` 到 `tasks[] → context + messages + artifacts`

```typescript
// Web 聊天的状态
interface ChatState {
  messages: Message[]
}

// 桌面 AI 产品的状态
interface DesktopAIState {
  tasks: Task[]          // 任务是顶层组织单元
  activeTaskId: string
}

interface Task {
  id: string
  title: string
  context: {             // 上下文是独立模块
    files: FileRef[]
    directories: DirRef[]
    clipboard?: ClipboardRef
  }
  messages: Message[]    // 对话只是子模块
  artifacts: Artifact[]  // 产物是独立模块
  status: 'running' | 'idle' | 'error'
  versions: Version[]    // 版本管理
}
```

### 4.2 渲染布局：从"单列滚动"到"三栏工作台"

桌面端建议三栏布局（可折叠）：

- **左栏：任务侧栏** —— 任务列表、新建任务、后台任务状态
- **中栏：主工作区** —— 上下文区 + 对话区（对话区可折叠/最小化）
- **右栏：产物面板** —— 当前任务的产物列表，点击进入预览/编辑

### 4.3 进程模型（Electron 场景）

```
Main Process（主进程）
  ├── 文件系统操作（读/写/监听）
  ├── 窗口管理（多窗口、托盘、快捷键）
  └── IPC 桥接
  
Renderer Process（渲染进程）
  ├── Vue/React UI
  ├── 任务状态管理
  └── 产物预览渲染

Worker Thread
  └── AI 流式响应处理 / 长任务编排
```

文件操作必须走主进程，渲染进程通过 IPC 请求——这条边界决定了权限提示必须在前端拦截层设计（详见 note-ms-002）。

## 五、为什么"照搬"会失败——三个典型陷阱

1. **把桌面当浏览器壳**：产物只支持复制粘贴，用户用完还得手动存文件 → 丧失桌面端的"落地"价值
2. **单会话线性流**：用户开了三个任务，全挤在一个聊天流里翻找 → 丧失桌面端的"多任务"价值
3. **忽略系统入口**：只做了一个聊天窗口，没有全局快捷键/右键集成/拖拽 → 丧失桌面端的"无处不在"价值

## 六、面试加分点

- **对标产品**：可提 Cursor（代码场景的任务+文件+产物组织）、Kimi Desktop、Claude Desktop 的差异化设计
- **架构原则一句话**：消息流是 AI 的"输入输出管道"，但桌面端的产品骨架是**任务（Task）— 上下文（Context）— 产物（Artifact）** 三层，管道只是任务内的子模块
- **能力边界意识**：桌面端做 AI 产品不是"加壳"，而是要重新设计状态模型、窗口模型、产物模型，这三层是架构核心

## 记忆要点

- Web是单线消息流，桌面端是任务-上下文-产物三区架构
- 因为桌面端能调用系统能力（文件/进程/全局快捷键），所以上下文极大丰富
- Web产物是纯文本，而桌面端产物是结构化文件（可直接落地保存）


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说桌面 AI 不能照搬 Web 聊天页，核心是"任务-上下文-产物"三区架构。但 Web 版 ChatGPT 也有 Artifact（产物面板），这算不算已经做了三区？为什么桌面端还不够？**

Web 版 Artifact 是"产物区"的雏形，但它是"消息流里的浮窗"，不是独立架构层。差异：一、产物生命周期——Web 的 Artifact 绑定在某条消息上（删消息丢产物），桌面端产物是独立实体（可脱离原任务存在、可保存到文件系统）；二、上下文来源——Web 只能拖文件进对话框，桌面端能引用本地目录、监听剪贴板、跨窗口拖拽，上下文是结构化的（文件树而非粘贴文本）；三、产物落地——Web 产物只能复制或下载，桌面端能直接写入本地工程目录、调用本地工具链（如 npm install、git commit）。所以"Artifact 是产物区的视图层"，桌面端要的是"产物作为一等公民的完整架构"——产物有自己的状态、权限、生命周期，不被消息流绑定。这才是"AI-Native 桌面"的区别。

### 第二层：证据与定位

**Q：你怎么判断一个 AI 桌面产品"做对了架构"而不是"Web 加壳"？从用户行为能看出哪些信号？**

三个用户行为信号：一、多任务并行——用户能否同时跑多个长任务、在任务间切换不打断（Web 加壳通常单线程对话，切走就丢）；二、产物可落地——用户能否把 AI 产物直接保存为本地文件、编辑后回流到 AI（Web 加壳只能复制粘贴）；三、上下文持续——用户能否"引用上次任务的产物"作为新任务的输入（Web 加壳每次对话从零开始）。如果用户总是"手动复制粘贴 + 单线对话"，说明产品架构是 Web 思维，没利用桌面能力。所以判断不是看代码，是看"用户用得像桌面软件还是像网页"。

### 第三层：根因深挖

**Q：你提到"长任务不绑组件生命周期"，这是桌面端的核心痛点。根因是什么？Vue/React 的响应式系统在这里有什么局限？**

根因是"Web 前端的状态模型是为短交互设计的"。Vue/React 的组件状态（ref/reactive、useState）绑定在组件实例上，组件卸载时状态销毁。Web 场景下这合理——用户切走页面就是"结束这次交互"。但桌面端长任务（如 5 分钟的文档分析）的生命周期远超组件——用户发起任务后切到别的页面，发起页组件卸载，如果任务状态在组件里，状态丢失、任务无法继续。Vue/React 的响应式系统本身没问题，问题是"状态放错了层"——长任务状态应该放在组件外的全局 store（Pinia/Redux），store 不随组件卸载。所以不是框架局限，是"状态分层"的架构原则——UI 临时状态在组件、长生命状态在 store。这是所有桌面端前端都要遵守的原则，不限于 AI 产品。

**Q：那为什么不所有状态都放全局 store，省得分层？**

全局 store 过载有两个问题：一、性能——Vue 的响应式依赖追踪（reactive 包装）有开销，store 里大量字段任何变更都触发依赖重算，全局 store 放临时 UI 状态（如某弹窗开关）会引发不必要的重渲染；二、可维护性——store 变成"什么都往里塞"的垃圾场，状态来源混乱、调试困难。所以分层是"按生命周期归类"——短命状态（弹窗、草稿）留组件、长命状态（任务、产物）上 store。这跟代码里"局部变量 vs 全局变量"的取舍一样，不是越全局越好。我的判据："用户切走页面后还要回来看到这个状态吗？"——要，则 store；不要，则组件。这个判据比"按状态类型分"更可操作。

### 第四层：方案权衡

**Q：任务-上下文-产物三区架构，你用 Vue + Pinia 实现，但产物可能是文件（在文件系统）、可能是内存对象、可能是远端站点，怎么统一管理？**

抽象一个"产物模型（Artifact Model）"统一管理。产物对象字段：id（唯一）、type（file/memory/site/...）、uri（本地路径 / 远端 URL / 内存引用）、meta（创建时间、关联任务、版本）、content（懒加载，按 type 取）。前端操作产物通过 ArtifactStore（Pinia），store 内部按 type 分发：file 类型调 Electron IPC 读写本地、memory 类型直接访问、site 类型 HTTP 请求。这样上层（UI、任务系统）统一面对 Artifact 对象，不感知底层存储差异。这是"统一抽象 + 分层实现"——上层一致接口、底层按类型适配。代价是抽象有泄漏风险（如 file 类型的进度条 vs site 类型的进度条语义不同），要在模型里加 type-specific 字段（如 file 有本地路径、site 有部署状态）。

**Q：为什么不用单纯的"文件路径"表示所有产物（一切落地为文件）？**

不是所有产物都适合落地为文件。如"AI 生成的图表"在编辑过程中是内存对象（频繁改、预览），用户确认后才落地为 PNG。如果强制"一切是文件"，每次编辑都要读写文件系统（慢、IO 开销），且"未确认的中间态"不该污染文件系统。所以"内存对象 + 落地时机"分离——产物在内存里是对象（快速编辑），用户点"保存"才落地为文件。这是"工作内存 vs 持久化"的分离，类似 IDE 的"未保存 buffer"——编辑在内存、Ctrl+S 才写盘。所以统一模型是"产物对象（内存）+ 可选的持久化（文件）"，不是"一切皆文件"。

### 第五层：验证与沉淀

**Q：你怎么验证三区架构真的比 Web 加壳好（用户体验提升）？**

指标对比：一、多任务并行率——用户同时进行的任务数（应 > 1，Web 加壳通常 = 1）；二、产物落地率——AI 产物被保存到本地/导出的比例（应高，说明产物有价值且能落地）；三、任务切换无损率——用户切走再切回，任务状态完整恢复的比例（应 100%，Web 加壳常丢失）；四、上下文复用率——新任务引用旧任务产物的比例（应 > 0，说明上下文持续）。这些指标通过埋点采集（任务事件、产物事件、用户操作），定期分析。如果"多任务并行率低 + 产物落地率低"，说明用户仍把它当 Web 用，产品没体现桌面价值，要回归架构反思。上线后 A/B 对比（旧版 Web 加壳 vs 新版三区架构）量化收益。

**Q：这道题做完，你沉淀出了什么可复用的 AI 桌面前端设计原则？**

四条原则：一、任务-上下文-产物三区——不堆消息流，按三区组织状态和 UI；二、状态分层——UI 临时状态在组件、长命状态在 store（Pinia），按"切走页面后是否需要"判据；三、产物是一等公民——产物有独立模型、生命周期、持久化，不被消息绑定；四、用桌面能力——文件系统、多窗口、全局快捷键、进程间通信，这些是桌面相对 Web 的核心价值，要用起来。这套原则用于所有 AI 桌面产品设计，核心是"利用桌面环境 + 状态分层 + 产物独立"。


## 结构化回答

**30 秒电梯演讲：** 桌面端有文件、窗口、目录和系统级入口，核心是围绕任务、上下文和产物组织，而不是消息流。

**展开框架：**
