# 【月之暗面面经】AI-Native 桌面产品从 Demo 到商用，前端最容易在哪些点翻车？

<!-- ANSWER_BODY_HERE -->

## 记忆要点

- 核心矛盾：Demo重演示而商用重稳定，必须在内存泄漏与全局异常处做防御
- 边界安全：因AI产物不可控，所以必须对生成的DOM/脚本做严格XSS消毒与沙箱隔离
- 状态管控：因为流式渲染极易掉帧，所以需做DOM差量更新与渲染节流（如绝对分块）
- 极限健壮：大段输出易崩溃，必须做指令截断、撤销栈隔离及复杂文档的降级渲染


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Demo 到商用你说翻车点在"稳定性/安全/性能/可运维"，但为什么不把"功能更多"作为商用重点（Demo 功能少，商用要加更多功能）？**

商用 success 不是"功能多"，是"功能可用且可靠"。Demo 的目标是"验证可行性"（如演示 AI 能生成站点），功能少但理想环境下能用。商用的目标是"真实用户长期使用"，用户环境复杂（大文件、弱网、长时间运行）、用户行为多样（误操作、极端用法），如果稳定性差（长任务崩溃）、安全漏洞（文件误删）、性能差（大文件卡顿），功能再多用户也弃用。所以商用优先"加固现有功能"（稳定性/安全/性能），而非"加新功能"。类比：Demo 是"概念车"（能跑就行），商用是"量产车"（要耐久、安全、省油、可维修），量产车的核心是"可靠性工程"而非"更多功能"。

### 第二层：证据与定位

**Q：商用后用户反馈"应用用着用着变卡"，你怎么定位是内存泄漏还是性能退化？**

Chrome DevTools 分两步定位：一、Memory 面板——用堆快照（Heap Snapshot）对比，如果堆内存持续增长（如每次操作涨 10MB 且不释放），是内存泄漏（对象未释放）；如果堆内存稳定但 CPU 占用高，是性能退化（计算开销大）。二、Performance 面板——录制卡顿时段的火焰图，看主线程耗时：如果 scripting 占比高（如 GC 频繁，是内存泄漏触发频繁垃圾回收）、rendering 占比高（如 DOM 节点多，是渲染性能问题）。常见根因：一、内存泄漏——AI 任务的大对象（如完整文档数据）未释放、事件监听器未清理、闭包持有大对象；二、性能退化——大文件渲染（如上万行表格无虚拟滚动）、频繁 reflow（如流式输出每字触发重排）。

### 第三层：根因深挖

**Q：长任务内存泄漏你定位到是"AI 流式输出的事件监听器未清理"，但为什么不直接说"代码 bug，改了就行"，还要系统性解决？**

单次 bug 修复是"打地鼠"（修一个冒一个），系统性解决是"防止再发生"。流式输出的事件监听器泄漏是"资源生命周期管理"问题的表现——AI 桌面产品有大量异步资源（事件监听器、定时器、WebSocket/SSE 连接、Web Worker），每个都有生命周期（创建 → 使用 → 销毁），如果销毁逻辑不完整（如组件卸载时没清理监听器），泄漏。系统性解决：一、资源管理框架——封装资源的创建和销毁（如 `createResource()` 返回对象带 `dispose()` 方法），组件卸载时自动调用 `dispose()`，而非手动清理（易漏）；二、lint 规则——静态检查"addEventListener 是否有对应 removeEventListener"、"setInterval 是否有 clearInterval"，未配对则报错；三、自动化检测——CI 跑内存泄漏测试（如 Puppeteer 操作后检测堆内存）。核心："单次修复 + 防护机制（框架 + lint + 自动检测），系统性防止同类问题"。

**Q：那为什么不直接用 GC（垃圾回收）自动清理，还要手动管理资源生命周期？**

GC 只管"内存对象的释放"，不管"外部资源的释放"。事件监听器是"外部引用"——`element.addEventListener('click', handler)` 中，element 持有 handler 的引用，即使组件销毁，只要 element 还在（如 DOM 未移除），handler 不会被 GC（element 强引用 handler）。同理，setInterval 的回调、SSE 的连接、Web Worker 的引用，都是外部资源，GC 不自动清理。必须手动调用 `removeEventListener`、`clearInterval`、`EventSource.close()`、`worker.terminate()` 释放。所以"外部资源手动管理 + 内存对象 GC 自动管理"，不能依赖 GC 解决所有泄漏。Rust 的所有权机制（编译时保证资源释放）更彻底，但 JS 没有所有权机制，靠工程手段（框架 + lint + 检测）弥补。

### 第四层：方案权衡

**Q：AI 产物安全你说要"XSS 消毒和沙箱隔离"，但为什么不只做 XSS 消毒（足够防 XSS 攻击）？**

XSS 消毒只防"脚本注入"（如 `<script>` 标签），防不了"其他攻击"。AI 产物可能含多种恶意内容：一、HTML 注入——如 `<img onerror="...">` 触发 JS，XSS 消毒（如 DOMPurify）能处理；二、CSS 注入——如 `<style>body{display:none}</style>` 隐藏页面或窃取数据（CSS 能读取属性），XSS 消毒可能漏（取决于配置）；三、iframe 钓鱼——如 `<iframe src="phishing.com">`，XSS 消毒能处理但配置复杂；四、JS 执行——AI 产物含 JS 代码（如生成的前端页面），如果直接执行，即使无 XSS，也可能有恶意逻辑（如发请求、读数据）。沙箱隔离解决"即使有恶意内容也限制影响"——如 iframe sandbox 属性（禁止脚本执行、表单提交、跨域访问）、CSP（内容安全策略，限制资源加载）。所以"XSS 消毒（防已知攻击）+ 沙箱隔离（兜底，即使漏过也限制影响）"，纵深防御。

**Q：为什么不直接禁止 AI 产物含 HTML/JS（纯文本输出），彻底消除安全风险？**

纯文本输出的代价：失去 AI 产物的丰富性。AI 桌面产品的产物是多样的（站点、PPT、富文本文档），这些产物天然含 HTML/CSS（用于排版和样式），禁止 HTML 则 AI 只能输出纯文本，产品能力大幅下降（如不能生成可视化站点、不能生成带样式的报告）。安全风险不能靠"禁止功能"解决，要靠"安全地渲染功能"：一、XSS 消毒——清理危险标签和属性（如 script、onerror）；二、沙箱隔离——iframe sandbox 渲染 AI 产物的 HTML，限制脚本执行和跨域访问；三、CSP——内容安全策略，限制资源加载源（如只允许加载白名单域名的资源）。所以"允许丰富产物 + 多层安全防护"，而非"禁止产物简化问题"。类比：浏览器不禁止网页含 JS（否则 Web 没用），而是用沙箱和同源策略隔离风险。

### 第五层：验证与沉淀

**Q：你怎么验证商用版本的稳定性达到标准（如能支撑真实用户长期使用）？**

压力测试和长期运行测试：一、压力测试——模拟真实用户的极端用法（如同时发起 10 个长任务、上传超大文件、频繁切换窗口），验证应用不崩溃、不卡顿；二、长期运行测试——应用连续运行 24-48 小时，模拟用户长期使用，检测内存泄漏（堆内存不应持续增长）和性能退化（响应时间不应逐渐变长）；三、故障注入——模拟异常场景（如网络断开、API 超时、磁盘满），验证错误处理和降级；四、灰度发布——先给小比例用户（如 1%）使用，监控崩溃率和性能指标，稳定后逐步扩大。商用标准：崩溃率 < 0.1%、内存泄漏 < 10MB/小时、P99 响应时间 < 500ms。

**Q：这道题沉淀出什么可复用的 Demo 到商用经验？**

四条原则：一、可靠性优先于功能——商用优先加固现有功能（稳定/安全/性能），而非加新功能，核心是"能用且可靠"而非"功能多"；二、系统性防护——单次 bug 修复 + 防护机制（资源管理框架 + lint + 自动检测），系统性防止同类问题，而非打地鼠；三、纵深防御——XSS 消毒 + 沙箱隔离 + CSP 多层安全，而非依赖单一防护；四、压力验证——压力测试（极端用法）+ 长期运行测试（内存泄漏）+ 灰度发布（真实用户），验证商用稳定性。核心洞察："Demo 到商用本质是'可靠性工程'——借鉴汽车从概念车到量产车（耐久测试 + 安全碰撞 + 流水线），AI 产品的核心是'稳定性 + 安全性 + 可运维性'的工程化跨越，而非功能堆砌。"
