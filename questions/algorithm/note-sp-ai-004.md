# 怎么判断链表相交？

## 双指针法（最优解）

### 核心思路

```
链表A:  a1 → a2 → c1 → c2 → c3
                     ↑
链表B:  b1 → b2 ────┘

a = A到交点的距离 = 2
b = B到交点的距离 = 2  
c = 交点到末尾的距离 = 3

双指针路径：
pA: a1 → a2 → c1 → c2 → c3 → null → b1 → b2 → c1 ← 相遇！
    |___a___|___c____|___b___|
    总长 = a + c + b = 2+3+2 = 7

pB: b1 → b2 → c1 → c2 → c3 → null → a1 → a2 → c1 ← 相遇！
    |___b___|___c____|___a___|
    总长 = b + c + a = 2+3+2 = 7

a+c+b = b+c+a → 保证两个指针同时到达交点！
```

### 代码实现

```python
class ListNode:
    def __init__(self, x):
        self.val = x
        self.next = None

def getIntersectionNode(headA: ListNode, headB: ListNode) -> ListNode:
    """
    双指针法：时间O(n+m)，空间O(1)
    """
    if not headA or not headB:
        return None
    
    pA, pB = headA, headB
    
    # pA走完A链表后走B链表，pB走完B链表后走A链表
    # 如果相交，必然在交点相遇
    # 如果不相交，同时走到None
    while pA != pB:
        pA = pA.next if pA else headB
        pB = pB.next if pB else headA
    
    return pA  # 相交则返回交点，不相交则返回None
```

### 执行过程演示

```
A: 4 → 1 → 8 → 4 → 5
B: 5 → 6 → 1 → 8 → 4 → 5
              ↑ 交点(8)

Step 1: pA=4, pB=5
Step 2: pA=1, pB=6
Step 3: pA=8, pB=1
Step 4: pA=4, pB=8
Step 5: pA=5, pB=4
Step 6: pA=null→跳到B头=5, pB=5
Step 7: pA=6, pB=null→跳到A头=4
Step 8: pA=1, pB=1
Step 9: pA=8, pB=8 ← 相遇！返回节点8
```

## 其他解法对比

### 方法二：哈希集合

```python
def getIntersectionNode_hash(headA, headB):
    """时间O(n+m)，空间O(n)"""
    visited = set()
    
    # 遍历A链表，存入集合
    cur = headA
    while cur:
        visited.add(cur)
        cur = cur.next
    
    # 遍历B链表，查集合
    cur = headB
    while cur:
        if cur in visited:
            return cur  # 第一个在A中出现的是交点
        cur = cur.next
    
    return None
```

### 方法三：长度差法

```python
def getIntersectionNode_len(headA, headB):
    """先算长度差，长的先走差值，再同时走"""
    def get_length(head):
        length = 0
        while head:
            length += 1
            head = head.next
        return length
    
    lenA, lenB = get_length(headA), get_length(headB)
    diff = abs(lenA - lenB)
    
    # 长的先走diff步
    curA, curB = headA, headB
    if lenA > lenB:
        for _ in range(diff):
            curA = curA.next
    else:
        for _ in range(diff):
            curB = curB.next
    
    # 同时走，找交点
    while curA and curB:
        if curA == curB:
            return curA
        curA = curA.next
        curB = curB.next
    
    return None
```

## 三种方法对比

| 方法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|----------|----------|------|------|
| **双指针** | O(n+m) | O(1) | 最优 | 思路不直观 |
| **哈希集合** | O(n+m) | O(n) | 简单直观 | 空间开销大 |
| **长度差法** | O(n+m) | O(1) | 直观易理解 | 需要两次遍历 |

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：判断链表相交你用双指针（pA 走完 A 走 B，pB 走完 B 走 A），为什么不直接用 HashSet 存 A 的所有节点？**

HashSet 方案直观——遍历 A 存入 Set，遍历 B 查 Set，时间 O(n+m) 空间 O(n)。双指针方案时间一样 O(n+m)，但空间 O(1)，更优。双指针的精妙在于利用了数学等式：设 A 独有部分长 a、B 独有部分长 b、公共部分长 c，则 pA 走 a+c+b 步、pB 走 b+c+a 步，两者路径长度相等（a+c+b = b+c+a），必然在交点相遇。HashSet 是"空间换思维简单"，双指针是"零空间 + 数学洞察"，面试官要的就是后者。选双指针是因为它在空间上有质的提升（O(n)→O(1)），不是常数优化。

### 第二层：证据与定位

**Q：你说 a+c+b = b+c+a 保证相遇，但如果两链表不相交，双指针会死循环吗？**

不会死循环。不相交时 c=0，pA 走 a+b 步到达 null（走完 A 的 a 步，再走 B 的 b 步），pB 走 b+a 步到达 null，两者同时到达 null，循环退出，返回 null。关键证据：pA 和 pB 各自最多走 a+b（或 b+a）步就会同时到 null，不会无限切换。具体实现里，pA 走到 null 后切到 B 的 head，再走到 null 就是终点，不会"切回 A"形成循环。所以"切换链表"只发生一次（从 A 尾切到 B 头），不是反复横跳。代码里用 `pA = pA == null ? headB : pA.next` 这个三元保证只切一次。

### 第三层：根因深挖

**Q：双指针解法的一个变种是"长度差法"——先算两链表长度差 d，长链表先走 d 步再同步走。这跟你的切换法哪个更好？**

两者本质相同（都是消除长度差后同步），但切换法代码更简洁（5 行核心），长度差法要两次遍历算长度 + 一次对齐 + 一次同步（代码长且边界多）。切换法的优势是"不需要显式算长度"——通过走完自己再走对方的天然对齐，把长度差消解掉。但长度差法在面试中有个价值：它更"可解释"，面试官追问"为什么这样走能相遇"时，长度差法能直接说"消除长度差后两指针到交点距离相等"，比切换法的数学等式更直观。所以答题策略：先讲长度差法（好解释），再说"还能用切换法更简洁"，展示多角度。

**Q：那为什么不能用"值相等"判断相交，而非要"节点引用相等"？**

这是链表相交的定义问题。LeetCode 160 明确："相交指节点引用相同，不是值相同"。如果两链表有相同值的节点但不是同一个对象，不算相交。原因：实际场景里链表节点可能存复杂对象，"值相等"定义模糊（深相等还是浅相等？），而"引用相等"是确定的无歧义语义。代码上 `pA == pB` 比较的是 Java 对象引用（内存地址），不是 `.val` 比较。如果错用 `.val` 比较，遇到 [1,2,3] 和 [4,2,5]（都有值为 2 的节点但不相交）会误报相交。这是这题最易踩的坑。

### 第四层：方案权衡

**Q：如果有 3 个链表判断共同交点，你的双指针法还能扩展吗？**

双指针法依赖"两路对齐"的数学性质，3 个链表时双指针扩展不自然（三条路长度对齐复杂）。3 个或多个链表求共同交点，标准解是 HashSet——把第一个链表存入 Set，遍历第二个链表把命中节点存入第二个 Set，遍历第三个链表查第二个 Set... 逐级过滤，最终剩下的就是共同交点。时间 O(总节点数)，空间 O(最长链表)。权衡：双指针适合 2 个链表（空间 O(1)），HashSet 适合 ≥3 个链表（实现简单但空间 O(n)）。这是"问题规模决定方法选型"的典型例子，不要硬套单一方法。

**Q：为什么不用"长度差法 + 取最小值对齐"的暴力法，让 3 个指针都从各自链表等距位置出发？**

理论上可以，但实现复杂——要先算 3 个链表长度，取最短的作为基准，长链表各跳过 (自己长度 - 最短长度) 步，再同步前进直到引用相等。代码量和边界 case 比 HashSet 多一倍，且可读性差（三个指针同步的循环条件难写）。工程取舍：当方法数 ≥3 时，"显式数据结构（HashSet/HashMap）"通常比"多指针技巧"更易实现和维护。这跟"两个链表用双指针、多个用 HashSet"的选型逻辑一致——规模小用技巧、规模大用结构。

### 第五层：验证与沉淀

**Q：你怎么验证双指针解法在各种相交/不相交场景下都对？**

五类用例：① 两链表不相交 [1,2,3] 和 [4,5,6]→null；② 在头节点相交（一个链表是另一个的前缀）；③ 在中间相交（Y 型）；④ 一个链表是空；⑤ 两链表都只有一个节点且相同/不同。重点测边界——pA 走到 null 切到 headB 的时机、两链表同时走完都到 null 的退出条件。对拍：随机生成两链表（可能相交可能不相交），跟 HashSet 版逐 case 对比。不相交 case 是高频 bug 区——很多错误实现会死循环或返回错误节点。

**Q：这道题沉淀出了什么可复用的链表技巧？**

两条可复用：一、"双指针消除长度差"——不只用于相交判断，回文链表（找中点）、合并有序链表（双指针同步）都用到；二、"引用相等 vs 值相等"的区分——这是链表题的通用陷阱，LeetCode 上凡是涉及"相同节点"的题（相交、环形、克隆）都要用引用比较。模板库里我存了"双指针链表技巧"专项：快慢指针（找中点、判环）、前后指针（删除倒数第 N 个）、交叉指针（相交判断），三个技巧覆盖了大部分中等链表题。
