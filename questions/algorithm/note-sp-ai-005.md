# 冒泡排序的时间复杂度？

## 算法核心

```
原始数组: [5, 3, 8, 1, 9, 2]

第1轮：相邻比较交换，最大的冒到最后
[5,3,8,1,9,2] → [3,5,8,1,9,2] → [3,5,1,8,9,2] → [3,5,1,8,9,2] → [3,5,1,8,2,9]
                                                                              ↑ 9到位

第2轮：
[3,5,1,8,2,9] → [3,1,5,8,2,9] → [3,1,5,2,8,9]
                                              ↑ 8到位

第3轮：
[3,1,5,2,8,9] → [1,3,5,2,8,9] → [1,3,2,5,8,9]
                                    ↑ 5到位

第4轮：
[1,3,2,5,8,9] → [1,2,3,5,8,9]
                ↑ 3到位

第5轮：
[1,2,3,5,8,9] → 无交换 → 已排序 ✅
```

## 代码实现

### 基础版本

```python
def bubble_sort(arr):
    n = len(arr)
    for i in range(n - 1):           # n-1轮
        for j in range(n - 1 - i):   # 每轮比较n-1-i次
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr
```

### 优化版本（提前终止）

```python
def bubble_sort_optimized(arr):
    """加入flag，已排序时提前终止 → 最好O(n)"""
    n = len(arr)
    for i in range(n - 1):
        swapped = False  # 本轮是否发生交换
        for j in range(n - 1 - i):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
                swapped = True
        
        if not swapped:  # 本轮无交换 → 已排序
            break
    return arr
```

## 时间复杂度分析

| 情况 | 复杂度 | 说明 |
|------|--------|------|
| **最好** | O(n) | 数组已排序，优化版第一轮就终止 |
| **平均** | O(n²) | 比较次数 = n(n-1)/2 |
| **最坏** | O(n²) | 逆序数组，每次都要交换 |

### 推导过程

```
比较次数 = (n-1) + (n-2) + ... + 1 = n(n-1)/2

n=10:  45次比较
n=100: 4950次比较  
n=1000: 499500次比较

O(n²)
```

## 空间复杂度

```
O(1) — 原地排序，只用常数级额外空间
```

## 稳定性

```
稳定排序 ✅

相等元素不交换：
[3a, 3b, 1] → [3a, 1, 3b] → [1, 3a, 3b]
               ↑              ↑
          3a和3b不交换    3a仍在3b前面
```

## 冒泡排序 vs 其他排序

| 排序 | 最好 | 平均 | 最坏 | 空间 | 稳定 |
|------|------|------|------|------|------|
| **冒泡** | O(n) | O(n²) | O(n²) | O(1) | ✅ |
| 选择 | O(n²) | O(n²) | O(n²) | O(1) | ❌ |
| 插入 | O(n) | O(n²) | O(n²) | O(1) | ✅ |
| 快排 | O(nlogn) | O(nlogn) | O(n²) | O(logn) | ❌ |
| 归并 | O(nlogn) | O(nlogn) | O(nlogn) | O(n) | ✅ |

## 面试加分点

1. **最好O(n)的条件**：加了swapped flag优化后，已排序数组只需一轮
2. **稳定排序**：相等元素相对位置不变，这在多关键字排序时很重要
3. **与选择排序区别**：冒泡每轮可能多次交换，选择每轮只交换一次
4. **实用价值低**：O(n²)排序中插入排序实际表现通常优于冒泡

## 记忆要点

- 时间复杂度记口诀：最好O(n)，最坏平均皆O(n²)。因为未优化全遍历为n²，优化后一轮无交换提前终止为n。
- 空间复杂度O(1)，因为是原地交换只用常数级额外空间。
- 核心特性：冒泡是稳定排序，因为相等元素不交换相对位置。
- 对比选择排序：冒泡每轮可多次交换且稳定，而选择每轮只交换一次且不稳定。
- 加分提醒：实际开发中，同为O(n²)级别，插入排序的性能通常优于冒泡排序。


## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：冒泡排序你说最好 O(n)，但教科书都说冒泡是 O(n²)，这个 O(n) 是怎么来的？**

教科书说的 O(n²) 是平均和最坏情况。最好 O(n) 的前提是加了"本轮是否发生交换"的优化标志（swapped flag）：每轮开始 swapped=false，发生交换就置 true；一轮结束如果 swapped 仍是 false，说明数组已有序，提前 break。对已排好序的输入（如 [1,2,3,4,5]），第一轮遍历不发生任何交换，swapped=false，直接退出，只走了 n-1 次比较，所以 O(n)。不加这个优化的朴素冒泡，无论输入是否有序都要跑满 n-1 轮，最好情况也是 O(n²)。所以"冒泡最好 O(n)"是"加了优化的冒泡"的特性，不是冒泡算法本身的固有性质。

### 第二层：证据与定位

**Q：你说冒泡是稳定排序，怎么证明？什么操作会破坏它的稳定性？**

稳定性定义：值相等的元素排序后相对位置不变。冒泡的比较条件是 `if (a[j] > a[j+1]) swap`，严格大于——相等时不交换。所以两个值相等的元素永远不会被 swap 操作改变相对位置，稳定性得证。会破坏稳定性的写法：把比较改成 `if (a[j] >= a[j+1]) swap`（大于等于），此时相等元素也会交换，稳定性丢失。这是冒泡和选择的本质区别——选择排序每轮找最小值时，会把最小值与当前位置交换，可能跨越多个相等元素，所以选择排序天然不稳定（即使比较用严格小于）。稳定性不是"算法名字决定的"，是"具体实现里的比较+交换逻辑决定的"。

### 第三层：根因深挖

**Q：同样是 O(n²) 的排序，插入排序实测通常比冒泡快，根因是什么？**

根因是"交换次数"和"常数操作"。冒泡每轮可能多次交换（每次相邻交换），插入排序是把当前元素"插入"到已排序部分的正确位置——实现上可以用"整体后移"代替"逐次交换"，减少赋值次数。具体数据：冒泡对逆序数组要 n(n-1)/2 次交换，每次交换是 3 次赋值；插入用"后移+单次插入"是 n(n-1)/2 次比较 + n(n-1)/2 次后移 + n 次插入，赋值次数约为冒泡的 1/3。另外插入排序对"近乎有序"的输入有自适应——每个元素只需后移 1-2 步，接近 O(n)。所以实际工程里 O(n²) 首选插入排序（如 Python 的 Timsort 在小数组退化时用插入），冒泡基本只在教学中出现。

**Q：那为什么快速排序最坏是 O(n²)，却比 O(n log n) 的归并排序更常用？**

快排的 O(n²) 只在"输入已有序且每次选第一个做 pivot"这种极端 case 触发，随机化 pivot 或三数取中后，最坏情况概率极低，平均 O(n log n) 且常数比归并小（原地排序不需要额外 O(n) 空间、缓存友好）。归并虽然稳定且最坏也是 O(n log n)，但需要 O(n) 额外空间，对大数据集不友好。工程取舍：快排"平均快 + 原地 + 缓存友好"是默认选择，归并用于"需要稳定性"或"链表排序"（链表归并不需要额外空间）的场景。Java 的 Arrays.sort 对基本类型用优化快排（双轴 QuickSort）、对对象用 TimSort（归并+插入的混合），就是这个取舍的体现。

### 第四层：方案权衡

**Q：如果面试官问"什么时候该用冒泡排序"，你怎么回答？**

诚实回答：几乎没有生产场景该用冒泡。它的价值是教学——最简单的排序，便于讲清"比较+交换"的基本范式，以及作为"稳定性"和"优化标志"的教学载体。唯一可能用冒泡的场景是"n 极小（≤8）且代码体积受限"（嵌入式、单片机），但即使这种场景插入排序也更优。如果面试官追问"为什么不用快排"，答：快排对极小数组有递归开销，O(n²) 的简单排序在小 n 下常数更小——但首选插入不是冒泡。所以这道题的"正确回答"不是为冒泡辩护，而是展示对排序家族的全面理解，知道每个算法的定位和适用边界。

**Q：为什么大多数标准库用 Timsort（归并+插入混合）而不是单一快排？**

Timsort 解决了两个工程问题：一、稳定性——对象排序要求稳定（如按多关键字排序），快排不稳定，归并稳定，Timsort 保留归并的稳定性；二、现实数据常是"局部有序"的（runs），Timsort 识别这些 runs 直接合并，对近乎有序的数据接近 O(n)，比纯归并快。快排在"基本类型排序"里仍是首选（不需要稳定性，求极致性能）。所以标准库的选型是"按数据类型分流"——基本类型快排（快）、对象 Timsort（稳+现实数据友好），这是工程上"对场景精细化匹配"的典范，比"一个算法打天下"更合理。

### 第五层：验证与沉淀

**Q：你怎么验证自己实现的冒泡排序真的稳定、真的最好 O(n)？**

两类验证：一、稳定性测试——构造 `[(3,'a'),(3,'b'),(1,'c'),(3,'d')]`，按数字排序后应保持 'a','b','d' 的相对顺序（即 `[(1,'c'),(3,'a'),(3,'b'),(3,'d')]`），如果变成 `[(1,'c'),(3,'b'),(3,'a'),(3,'d')]` 说明不稳定（比较条件写错）；二、最好 O(n) 测量——对已排序的 [1..1000] 计数比较次数，应约为 999 次（一轮 break），而不是约 50 万次（朴素版）。再加随机数据测试与 `Arrays.sort` 对拍。这些测试不只验证正确性，还验证算法的"理论性质"是否在实现中保留，是排序题的标准验证套路。

**Q：这道题沉淀出了什么可复用的排序选型经验？**

排序选型决策树：① n ≤ 16 → 插入排序（常数小）；② 基本类型大数据 → 双轴快排（Arrays.sort 默认）；③ 对象/需稳定 → Timsort（Collections.sort 默认）；④ 链表 → 归并排序（不需要额外空间）；⑤ 内存受限 → 堆排序（原地 + 最坏 O(n log n)）；⑥ 海量数据外部排序 → 多路归并（分块+合并）。这个决策树覆盖了 95% 的工程场景，剩下 5%（如并行排序、GPU 排序）是专门领域。面试时遇到排序题，先问清"数据规模 + 是否要稳定 + 数据特征（是否近乎有序）"，再从决策树里选，而不是上来就写快排。
