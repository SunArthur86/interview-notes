# 【滴滴面经】压测时最多能存多少数据？不同数据量下，短链系统的 QPS 表现怎么样？

## 一、回答框架

这道题考察的是**容量规划与性能量化的能力**。核心逻辑：短链系统本质上是一个 `短码 → 长URL` 的 KV 查询系统，QPS 的天花板不是存储量本身，而是**缓存命中率**。数据量越大，单节点内存放不下全部热点数据，缓存命中率下降，请求穿透到数据库，QPS 就会断崖式下降。

---

## 二、存储量估算

先量化"能存多少数据"，再谈 QPS。

### 2.1 单条短链存储开销

| 字段 | 大小 | 说明 |
|------|------|------|
| 短码（如 6~7 位 Base62） | 7B | `aB3xK9q` |
| 长URL | 平均 ~100B | 电商/新闻类链接较长 |
| 元数据（创建时间、过期时间、用户ID等） | ~50B | TTL、统计用 |
| Redis overhead（dictEntry + SDS + robj） | ~70B | Redis 内部数据结构开销 |
| **单条总计** | **~230B** | 含 Redis 内部开销 |

### 2.2 各数据量级存储需求

| 数据量 | 纯数据大小 | Redis 占用内存（含开销+碎片） | 说明 |
|--------|-----------|----------------------------|------|
| 百万级（10⁶） | ~100 MB | ~300 MB | 单机轻松容纳 |
| 千万级（10⁷） | ~1 GB | ~3 GB | 单机仍可容纳（16GB/32GB 内存） |
| 亿级（10⁸） | ~10 GB | ~30 GB | 需大内存机器或 Redis Cluster 分片 |
| 十亿级（10⁹） | ~100 GB | ~300 GB | 必须 Redis Cluster + 多分片 |

> **面试加分点**：提到 Redis 的 `maxmemory-policy` 配置。如果内存不够，会触发淘汰策略（LRU/LFU），导致缓存命中率进一步下降，形成恶性循环。

## 三、不同数据量下的 QPS 表现

### 3.1 核心性能对比表

以下基于典型短链系统架构（Nginx + Go/Java 应用 + Redis 缓存 + MySQL 持久化），**单节点应用服务器（4C8G）**：

| 数据量 | 缓存命中率 | Redis QPS | DB QPS | 系统整体 QPS | 平均延迟 | 说明 |
|--------|-----------|-----------|--------|-------------|---------|------|
| **百万级** | 99.5%+ | ~10万 | <500 | **~10万** | 2~5ms | 热数据全在内存，瓶颈在应用层 |
| **千万级** | 97%~98% | ~8万 | ~2,000 | **~5万** | 5~10ms | 部分冷数据穿透，DB 开始有压力 |
| **亿级** | 90%~93% | ~5万 | ~5,000 | **~2万** | 10~30ms | 缓存空间不足，穿透严重，DB 成为瓶颈 |
| **十亿级** | <85% | ~3万 | ~10,000+ | **~5,000** | 30~100ms | 单机 Redis 无法承载，必须集群化 |

### 3.2 为什么数据量增大后 QPS 断崖式下降？

```
数据量↑ → 热数据集 > 可用内存 → 缓存淘汰 → 命中率↓
    → 更多请求穿透到 DB（MySQL 查询 ~1-5ms vs Redis ~0.1ms）
        → DB 连接池耗尽 → 请求排队 → 延迟飙升 → QPS 下降
```

**关键转折点**：当缓存命中率从 99% 降到 90%，看起来只降了 9 个百分点，但**穿透到 DB 的请求量增加了 10 倍**（从 1% 到 10%），这就是非线性衰减的本质。

## 四、缓存命中率对 QPS 的影响（深度分析）

### 4.1 量化命中率与 QPS 的关系

假设：
- Redis 单次查询：0.1ms（~1万 QPS/连接，多连接复用）
- MySQL 单次查询：2ms（索引命中）
- 应用层处理：0.5ms

| 缓存命中率 | 等效平均延迟 | 理论 QPS | 下降幅度 |
|-----------|-------------|---------|---------|
| 99.9% | 0.1 × 0.999 + 2 × 0.001 = **0.1ms** | ~10万 | 基准 |
| 99% | 0.1 × 0.99 + 2 × 0.01 = **0.12ms** | ~8万 | -20% |
| 95% | 0.1 × 0.95 + 2 × 0.05 = **0.2ms** | ~5万 | -50% |
| 90% | 0.1 × 0.9 + 2 × 0.1 = **0.3ms** | ~3万 | -70% |
| 80% | 0.1 × 0.8 + 2 × 0.2 = **0.5ms** | ~1.5万 | -85% |
| 50% | 0.1 × 0.5 + 2 × 0.5 = **1.1ms** | ~5,000 | -95% |

> **面试核心结论**：缓存命中率低于 **95%** 时 QPS 开始明显下降，低于 **90%** 时出现断崖。

### 4.2 如何维持高缓存命中率

| 策略 | 具体做法 | 效果 |
|------|---------|------|
| **本地缓存（L1）+ Redis（L2）** | Caffeine/Guava 做本地缓存，TTL 60s | 顶配热点，命中率可达 99.9% |
| **缓存预热** | 新数据写入时同步写缓存 | 避免冷启动穿透 |
| **热点 Key 检测** | 实时统计高频访问的短码 | 主动加载到本地缓存 |
| **布隆过滤器** | 对不存在的短码做拦截 | 防止缓存穿透 |
| **合理设置 TTL** | 根据访问频率差异化设置 TTL | 避免冷数据挤占热数据空间 |

## 五、压测工具与方法

### 5.1 压测工具选型

| 工具 | 适用场景 | 特点 |
|------|---------|------|
| **wrk** | HTTP 接口压测 | 多线程、Lua 脚本支持、轻量 |
| **JMeter** | 复杂场景压测 | GUI、分布式、丰富插件 |
| **vegeta** | Go 编写的压测工具 | 常率/突峰模式、JSON 输出 |
| **Redis-benchmark** | 直接压 Redis | 官方工具、快速验证 Redis 性能 |

### 5.2 wrk 压测示例

```bash
# 压测短链跳转接口，50线程 × 200连接，持续60秒
wrk -t50 -c200 -d60s --latency \
    -s lua_script.lua \
    http://localhost:8080/redirect/

# lua_script.lua: 随机访问不同短码
wrk.method = "GET"
local codes = {"aB3xK9", "xY7pL2", ...}  -- 预生成短码列表
request = function()
    local url = "/redirect/" .. codes[math.random(#codes)]
    return wrk.format(nil, url)
end
```

## 六、面试加分点

1. **量化思维**：不笼统说"性能会下降"，而是给出具体数字——百万级 10万QPS、千万级 5万QPS、亿级 2万QPS
2. **二八定律**：80% 的访问集中在 20% 的短链上，利用这个特性优化缓存
3. **冷热分离**：近期生成的短链访问频率高，可按时间分片存储
4. **多级缓存**：L1（本地缓存）→ L2（Redis）→ DB，逐层兜底
5. **监控指标**：`redis-cli --stat`、`INFO stats`、Prometheus + Grafana 实时监控缓存命中率

---

## 七、总结一句话

> 短链系统的 QPS 不是由存储量决定的，而是由**缓存命中率**决定的。百万级数据全在内存中，QPS 可达 10 万；亿级数据缓存放不下，命中率降到 90% 以下，QPS 断崖到 2 万。容量规划的核心是：**用多级缓存 + 合理淘汰策略，把命中率维持在 95% 以上**。
