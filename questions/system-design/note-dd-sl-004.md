# 【滴滴面经】压测时最多能存多少数据？不同数据量下，短链系统的 QPS 表现怎么样？

## 一、回答框架

这道题考察的是**容量规划与性能量化的能力**。核心逻辑：短链系统本质上是一个 `短码 → 长URL` 的 KV 查询系统，QPS 的天花板不是存储量本身，而是**缓存命中率**。数据量越大，单节点内存放不下全部热点数据，缓存命中率下降，请求穿透到数据库，QPS 就会断崖式下降。

---

## 二、存储量估算

先量化"能存多少数据"，再谈 QPS。

### 2.1 单条短链存储开销

| 字段 | 大小 | 说明 |
|------|------|------|
| 短码（如 6~7 位 Base62） | 7B | `aB3xK9q` |
| 长URL | 平均 ~100B | 电商/新闻类链接较长 |
| 元数据（创建时间、过期时间、用户ID等） | ~50B | TTL、统计用 |
| Redis overhead（dictEntry + SDS + robj） | ~70B | Redis 内部数据结构开销 |
| **单条总计** | **~230B** | 含 Redis 内部开销 |

### 2.2 各数据量级存储需求

| 数据量 | 纯数据大小 | Redis 占用内存（含开销+碎片） | 说明 |
|--------|-----------|----------------------------|------|
| 百万级（10⁶） | ~100 MB | ~300 MB | 单机轻松容纳 |
| 千万级（10⁷） | ~1 GB | ~3 GB | 单机仍可容纳（16GB/32GB 内存） |
| 亿级（10⁸） | ~10 GB | ~30 GB | 需大内存机器或 Redis Cluster 分片 |
| 十亿级（10⁹） | ~100 GB | ~300 GB | 必须 Redis Cluster + 多分片 |

> **面试加分点**：提到 Redis 的 `maxmemory-policy` 配置。如果内存不够，会触发淘汰策略（LRU/LFU），导致缓存命中率进一步下降，形成恶性循环。

---

## 三、不同数据量下的 QPS 表现

### 3.1 核心性能对比表

以下基于典型短链系统架构（Nginx + Go/Java 应用 + Redis 缓存 + MySQL 持久化），**单节点应用服务器（4C8G）**：

| 数据量 | 缓存命中率 | Redis QPS | DB QPS | 系统整体 QPS | 平均延迟 | 说明 |
|--------|-----------|-----------|--------|-------------|---------|------|
| **百万级** | 99.5%+ | ~10万 | <500 | **~10万** | 2~5ms | 热数据全在内存，瓶颈在应用层 |
| **千万级** | 97%~98% | ~8万 | ~2,000 | **~5万** | 5~10ms | 部分冷数据穿透，DB 开始有压力 |
| **亿级** | 90%~93% | ~5万 | ~5,000 | **~2万** | 10~30ms | 缓存空间不足，穿透严重，DB 成为瓶颈 |
| **十亿级** | <85% | ~3万 | ~10,000+ | **~5,000** | 30~100ms | 单机 Redis 无法承载，必须集群化 |

### 3.2 为什么数据量增大后 QPS 断崖式下降？

```
数据量↑ → 热数据集 > 可用内存 → 缓存淘汰 → 命中率↓
    → 更多请求穿透到 DB（MySQL 查询 ~1-5ms vs Redis ~0.1ms）
        → DB 连接池耗尽 → 请求排队 → 延迟飙升 → QPS 下降
```

**关键转折点**：当缓存命中率从 99% 降到 90%，看起来只降了 9 个百分点，但**穿透到 DB 的请求量增加了 10 倍**（从 1% 到 10%），这就是非线性衰减的本质。

## 四、缓存命中率对 QPS 的影响（深度分析）

### 4.1 量化命中率与 QPS 的关系

假设：
- Redis 单次查询：0.1ms（~1万 QPS/连接，多连接复用）
- MySQL 单次查询：2ms（索引命中）
- 应用层处理：0.5ms

| 缓存命中率 | 等效平均延迟 | 理论 QPS | 下降幅度 |
|-----------|-------------|---------|---------|
| 99.9% | 0.1 × 0.999 + 2 × 0.001 = **0.1ms** | ~10万 | 基准 |
| 99% | 0.1 × 0.99 + 2 × 0.01 = **0.12ms** | ~8万 | -20% |
| 95% | 0.1 × 0.95 + 2 × 0.05 = **0.2ms** | ~5万 | -50% |
| 90% | 0.1 × 0.9 + 2 × 0.1 = **0.3ms** | ~3万 | -70% |
| 80% | 0.1 × 0.8 + 2 × 0.2 = **0.5ms** | ~1.5万 | -85% |
| 50% | 0.1 × 0.5 + 2 × 0.5 = **1.1ms** | ~5,000 | -95% |

> **面试核心结论**：缓存命中率低于 **95%** 时 QPS 开始明显下降，低于 **90%** 时出现断崖。

### 4.2 如何维持高缓存命中率

| 策略 | 具体做法 | 效果 |
|------|---------|------|
| **本地缓存（L1）+ Redis（L2）** | Caffeine/Guava 做本地缓存，TTL 60s | 顶配热点，命中率可达 99.9% |
| **缓存预热** | 新数据写入时同步写缓存 | 避免冷启动穿透 |
| **热点 Key 检测** | 实时统计高频访问的短码 | 主动加载到本地缓存 |
| **布隆过滤器** | 对不存在的短码做拦截 | 防止缓存穿透 |
| **合理设置 TTL** | 根据访问频率差异化设置 TTL | 避免冷数据挤占热数据空间 |

## 五、压测工具与方法

### 5.1 压测工具选型

| 工具 | 适用场景 | 特点 |
|------|---------|------|
| **wrk** | HTTP 接口压测 | 多线程、Lua 脚本支持、轻量 |
| **JMeter** | 复杂场景压测 | GUI、分布式、丰富插件 |
| **vegeta** | Go 编写的压测工具 | 常率/突峰模式、JSON 输出 |
| **Redis-benchmark** | 直接压 Redis | 官方工具、快速验证 Redis 性能 |

### 5.2 wrk 压测示例

```bash
# 压测短链跳转接口，50线程 × 200连接，持续60秒
wrk -t50 -c200 -d60s --latency \
    -s lua_script.lua \
    http://localhost:8080/redirect/

# lua_script.lua: 随机访问不同短码
wrk.method = "GET"
local codes = {"aB3xK9", "xY7pL2", ...}  -- 预生成短码列表
request = function()
    local url = "/redirect/" .. codes[math.random(#codes)]
    return wrk.format(nil, url)
end
```

## 六、面试加分点

1. **量化思维**：不笼统说"性能会下降"，而是给出具体数字——百万级 10万QPS、千万级 5万QPS、亿级 2万QPS
2. **二八定律**：80% 的访问集中在 20% 的短链上，利用这个特性优化缓存
3. **冷热分离**：近期生成的短链访问频率高，可按时间分片存储
4. **多级缓存**：L1（本地缓存）→ L2（Redis）→ DB，逐层兜底
5. **监控指标**：`redis-cli --stat`、`INFO stats`、Prometheus + Grafana 实时监控缓存命中率

---

## 七、总结一句话

> 短链系统的 QPS 不是由存储量决定的，而是由**缓存命中率**决定的。百万级数据全在内存中，QPS 可达 10 万；亿级数据缓存放不下，命中率降到 90% 以下，QPS 断崖到 2 万。容量规划的核心是：**用多级缓存 + 合理淘汰策略，把命中率维持在 95% 以上**。

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：你说数据量增大 QPS 会下降，为什么？短链查询是 KV 查找，O(1) 复杂度不该和数据量相关吗？**

因为瓶颈不在计算复杂度，在缓存命中率。Redis 是内存数据库，单实例内存有限（如 32GB），亿级短链（每条 200 字节 × 1 亿 = 20GB）刚好接近上限。数据量小时全部装进 Redis，命中率 99%+；数据量大时只能装热点部分，长尾短链（不常访问的）不在缓存，访问时 miss → 穿透到 DB → DB 查询慢（5-10ms）→ QPS 断崖式下降。QPS 衰减的根因是"缓存命中率随数据量增大而下降"，不是算法变慢。

### 第二层：证据与定位

**Q：压测时数据量从百万涨到亿级，QPS 从 10 万降到 2 万，你怎么确认是缓存命中率问题？**

直接看缓存命中率指标：
1. Redis hit rate——`INFO stats` 里的 `keyspace_hits / (keyspace_hits + keyspace_misses)`。如果命中率从 99% 降到 85%，确认是命中率下降。
2. DB QPS——看 DB 的查询 QPS 是否飙升。缓存命中时 DB QPS 应该很低（只处理 miss），如果 DB QPS 从 1000 涨到 2 万，说明大量请求穿透。
3. 慢查询——DB 慢查询日志是否有大量短链查询，确认穿透确实打到了 DB。

### 第三层：根因深挖

**Q：缓存命中率从 99% 降到 85%，根因是数据量大还是热点集中？**

看访问分布。如果短链访问符合"二八定律"（20% 的短链占 80% 流量），那即使亿级数据，热点短链都能进缓存，命中率应该高。命中率低说明长尾访问多——很多不常访问的短链被偶尔请求，这些不在缓存。可以用 Redis 的 `OBJECT FREQ`（LFU 模式）或采样分析访问分布，确认是"数据量超出内存"还是"访问太分散缓存装不下热点"。根因不同解法不同：前者扩 Redis 内存，后者引入本地缓存或多级缓存。

**Q：为什么不直接把 Redis 内存加到几百 GB，把所有数据装进去，命中率不就 100% 了？**

成本和性能都不划算。① 成本——单实例 Redis 内存超过 64GB 后，使用的 RDB 持久化、主从同步、故障恢复都会变慢（fork 大内存进程耗时几秒），不建议单实例太大。要扩到几百 GB 必须用集群（分片），成本线性增长。② 性能——Redis 是单线程，大内存意味着 key 数量多，某些命令（如 KEYS、大范围 SCAN）会阻塞，且内存大时 RDB/AOF 重写更慢。③ 边际收益——把命中率从 95% 提到 100% 要翻倍内存，但 QPS 只提升 5-10%，ROI 低。正确做法是用有限内存装热点 + 本地缓存补充，而不是追求 100% 命中。

### 第四层：方案权衡

**Q：亿级数据 QPS 只剩 2 万，业务要 5 万，你怎么优化？**

多级缓存 + 冷热分离：
1. 本地缓存（Caffeine）——在应用进程内缓存最热的几千个短链（容量受 JVM 堆限制），命中率 60-70%，查询 0.01ms。
2. Redis 分布式缓存——缓存更多短链（GB 级），命中率 95%，查询 0.5ms。
3. DB 冷数据——穿透到 DB 的请求走读写分离的从库，减轻主库压力。
4. 布隆过滤器——DB 里不存在的短链（恶意请求、已删除）用布隆过滤器拦截，防穿透。
分层后，本地缓存挡 70%，Redis 挡 25% 剩余，DB 只扛 5%，QPS 能从 2 万提到 10 万+。权衡：每加一层缓存增加复杂度（一致性、缓存预热），但性能收益显著。

**Q：为什么不直接用 SSD 替代 Redis（RocksDB 等），几百 GB 数据低成本存储，不就不用多级缓存了？**

因为 SSD 的查询延迟远高于内存。RocksDB（LSM-Tree）查询约 0.1-1ms，比 Redis（0.5ms 但纯内存）没有优势甚至更慢（SSD 随机读延迟）。而且 RocksDB 是嵌入式（与应用同进程），高 QPS 下与应用争抢 CPU。Redis 的优势是"纯内存 + 网络服务"，延迟稳定在 0.5ms。SSD 适合"冷数据存储 + 低频访问"（如日志、归档），不适合"热路径 KV 查询"。多级缓存的本质是"用不同介质服务不同热度"，比单一介质（全 SSD 或全内存）更高效。

### 第五层：验证与沉淀

**Q：你怎么证明容量规划模型是准确的（数据量 vs QPS 的关系）？**

压测验证 + 线上对照：
1. 分档压测——分别用百万、千万、亿级数据量压测，记录 QPS、命中率、TP99，拟合"数据量-QPS"曲线。
2. 线上验证——监控系统记录实际数据量增长和 QPS 变化，与压测模型对比，校准模型参数。
3. 容量预警——基于模型预测"数据量达 X 亿时 QPS 降到 Y"，提前扩容，避免被动。

**Q：容量规划怎么沉淀？**

1. 容量评估模板——"数据量 + 缓存容量 + 预期命中率 → 所需 Redis 节点数 + 应用实例数"，新业务按模板填参数出预估。
2. 压测平台化——把分档压测做成自动化工具，新业务上线前自动跑容量压测，出报告。
3. 多级缓存 SDK——封装"Caffeine + Redis + DB"的三级缓存组件，业务接入只配策略，不用自己搭。
