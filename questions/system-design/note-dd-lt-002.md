# 【滴滴面经】Redis 的 incr 和 Redisson 锁你都用了，它们分别解决什么问题？为什么两个都要上？

## 一、问题本质：并发控制的不同粒度

这个问题的核心在于理解：**incr 和 Redisson 锁不是竞争关系，而是互补关系——它们工作在不同的抽象层级，解决不同粒度的并发问题。**

用一句话概括：

> **incr 解决的是「单步原子计数」问题——快、轻、无锁开销。**
> **Redisson 锁解决的是「多步临界区一致性」问题——稳、全、可重入可续期。**

---

## 二、Redis `INCR` —— 轻量级原子计数器

### 2.1 `INCR` 的工作原理

Redis 的 `INCR` 命令对指定 key 的值执行 **原子递增** 操作。由于 Redis 是单线程模型，`INCR` 天然保证原子性——不存在两个客户端同时读到同一个旧值的问题。

```
# 假设库存 key 当前值为 10
客户端A:  INCR prize:stock   → 返回 11（扣减视角: 用DECR返回9）
客户端B:  INCR prize:stock   → 返回 12
# 不可能两个客户端都返回 11
```

在防超卖场景中，实际用的是 `DECR`（递减）或 `DECRBY`，本质相同：

```
DECR prize:stock:1001   → 返回当前值
# 如果返回值 >= 0 → 还有库存，扣减成功
# 如果返回值 < 0  → 库存超卖，需要归还 +1
```

### 2.2 `INCR`/`DECR` 代码示例

```java
@Service
public class StockCounterService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    /**
     * 用 DECR 原子扣减库存
     * @return 扣减后剩余库存值；< 0 表示库存不足（已超卖边界）
     */
    public long deductStock(Long prizeId) {
        String key = "prize:stock:" + prizeId;
        Long remaining = redisTemplate.opsForValue().decrement(key);

        if (remaining != null && remaining >= 0) {
            return remaining;  // 扣减成功
        }

        // remaining < 0：库存不足，归还
        redisTemplate.opsForValue().increment(key);
        return -1;  // 表示库存不足
    }

    /**
     * 用 INCR 统计参与人数（天然原子计数器）
     */
    public long incrementParticipantCount(Long activityId) {
        String key = "lottery:participants:" + activityId;
        return redisTemplate.opsForValue().increment(key);
    }
}
```

### 2.3 `INCR` 的优势与局限

**优势：**
- **极高性能**：O(1) 时间复杂度，单次 RTT，QPS 可达 10 万+
- **天然原子**：Redis 单线程保证，无需额外加锁
- **代码简洁**：一行命令解决问题

**局限：**
- **只能保护单个操作**：check 是读 stock，deduct 是 DECR——虽然 DECR 本身原子，但如果要在扣减前检查「stock > 0」，check + deduct 就不原子了
- **无法保护多步业务**：扣库存 + 校验资格 + 写记录，这些步骤之间的间隙没有保护
- **超卖边界问题**：DECR 返回负值才知道超卖，需要额外归还逻辑

> **重要澄清**：`DECR` 能保证递减本身是原子的，但「判断库存是否够 → 再扣减」这两步不是原子的。`DECR` 的做法是「先扣再判断」——如果扣完发现变成负数，再 INCR 归还。这是一个乐观策略，在高并发下可能有大量无效 DECR + INCR 操作。相比之下，**Lua 脚本可以在内部先判断再扣减，避免无效操作**。

## 三、Redisson 分布式锁 —— 复杂临界区的守护者

### 3.1 Redisson 锁的工作原理

Redisson 是基于 Redis 的高级 Java 客户端，它的分布式锁（`RLock`）实现远比简单的 `SETNX` 复杂：

1. **加锁**：基于 Lua 脚本执行 `HASH` 结构写入（key=锁名，field=线程ID，value=重入次数）
2. **可重入**：同一线程可多次获取同一把锁，内部维护重入计数
3. **看门狗续期**：默认锁超时 30 秒，后台线程每 10 秒自动续期，避免业务未执行完锁就过期
4. **自动释放**：持锁线程宕机后，锁会在超时后自动释放，避免死锁

### 3.2 Redisson 锁代码示例

```java
@Service
public class LotteryLockService {

    @Autowired
    private RedissonClient redissonClient;

    /**
     * 用 Redisson 分布式锁保护完整抽奖流程
     * 锁粒度: 用户级（防同一用户并发抽奖）
     */
    public LotteryResult doLotteryWithLock(Long userId, Long activityId) {
        String lockKey = "lottery:user:" + userId;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 尝试加锁：等待最多3秒，不指定leaseTime以启用看门狗
            boolean locked = lock.tryLock(3, TimeUnit.SECONDS);

            if (!locked) {
                return LotteryResult.fail("操作太频繁，请稍后再试");
            }

            // ========= 临界区开始 =========
            // 以下多步操作在同一时刻只有一个线程能执行

            // Step 1: 检查用户资格（是否已抽过）
            if (hasUserPlayed(userId, activityId)) {
                return LotteryResult.fail("您已参与过");
            }

            // Step 2: 扣减库存（此处仍可用DECR/Lua）
            Long remaining = deductStock(activityId);
            if (remaining < 0) {
                return LotteryResult.fail("奖品已抢完");
            }

            // Step 3: 写中奖记录
            saveLotteryRecord(userId, activityId, remaining);

            // Step 4: 异步发奖
            sendPrizeAsync(userId, activityId);

            // ========= 临界区结束 =========

            return LotteryResult.success();

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return LotteryResult.fail("系统繁忙");
        } finally {
            // 只有持有锁的线程才能释放
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 3.3 Redisson 锁的优势与局限

**优势：**
- **保护整个临界区**：多步操作全部在锁的保护下，一致性有保障
- **可重入**：同一线程可多次获取，避免自死锁
- **看门狗续期**：业务执行时间长也不会因锁过期而出问题
- **公平锁支持**：`redissonClient.getFairLock(key)` 可实现 FIFO 公平锁

**局限：**
- **性能开销**：每次加锁/解锁都有网络 RTT + Lua 脚本执行，约 2-5ms
- **串行化瓶颈**：同一把锁下的请求是串行的，QPS 受限于持锁时间
- **不适合纯计数场景**：如果只需要原子计数，用锁是杀鸡用牛刀

## 四、incr vs Redisson 锁：全景对比

### 4.1 维度对比表

| 对比维度 | Redis `INCR`/`DECR` | Redisson 分布式锁 |
|----------|---------------------|-------------------|
| **本质** | 原子计数器 | 互斥锁 |
| **解决问题** | 单个数值的原子增减 | 多步操作的临界区一致性 |
| **并发模型** | 无锁（单线程天然原子） | 悲观锁（互斥排他） |
| **保护范围** | 单条命令 | 一段代码（临界区） |
| **性能** | 极高，O(1)，单次RTT，10万+ QPS | 中等，加锁/解锁各一次RTT，数千 QPS |
| **持锁开销** | 无 | 看门狗续期线程 + 内存结构维护 |
| **可重入** | 不适用 | ✅ 支持同线程多次获取 |
| **自动续期** | 不适用 | ✅ 看门狗自动续期 |
| **超时释放** | 不适用 | ✅ 避免死锁 |
| **公平性** | 不适用 | 支持公平锁/非公平锁 |
| **代码复杂度** | 极低（一行命令） | 中等（try-finally，异常处理） |
| **适用场景** | 库存扣减、计数、限流、序列号 | 业务事务、防重复、资源独占 |

### 4.2 使用场景决策表

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 纯库存扣减（无业务逻辑） | `DECR` / Lua 脚本 | 单步操作，不需要锁的开销 |
| 统计在线人数 / 参与人数 | `INCR` | 纯计数，无竞态风险 |
| 接口限流（N次/分钟） | `INCR` + `EXPIRE` | 计数判断，轻量高效 |
| 生成唯一序列号 | `INCR` | 原子递增，天然唯一 |
| 抽奖（校验资格 + 扣库存 + 写记录） | **Redisson 锁** | 多步操作需临界区保护 |
| 防同一用户重复提交 | **Redisson 锁**（用户级锁） | 需要检查 + 操作的一致性 |
| 定时任务防重复执行 | **Redisson 锁** | 多节点部署，需跨JVM互斥 |
| 秒杀扣库存（超高并发） | Lua 脚本（不用锁） | QPS要求极高，锁成为瓶颈 |
| 库存扣减 + 订单创建（强一致） | **Redisson 锁** | 两个资源的一致性保证 |
| 分布式ID生成 | `INCR`（或 `INCRBY`） | 原子递增即可 |
| 转账（扣A加B） | **Redisson 锁** | 两步操作需原子 |

## 五、为什么两个都要上？—— 互补关系详解

### 5.1 类比理解

> **`INCR` 是高速公路 ETC 通道**——车不停车直接通过，极快，但只能处理「一辆车」。
>
> **Redisson 锁是停车场闸机**——需要完整的进出流程（取卡、验证、抬杆），慢一些，但能处理「一整套流程」。

### 5.2 在抽奖系统中的协作

在实际的抽奖系统中，两者协同工作，各自负责自己最擅长的部分：

```
用户抽奖请求
    │
    ▼
┌─────────────────────────────────────────────────┐
│  阶段1: Redis DECR 扣减库存（INCR 的逆向操作）     │
│  ───────────────────────────────────────────── │
│  • 目标: 快速拦截无库存请求                        │
│  • 特点: 单步原子，无锁，10万+ QPS                 │
│  • 逻辑: DECR → 返回值<0 ? 归还+拒绝 : 继续        │
│  • 作用: 承受99%+的流量冲击                       │
└───────────────────────┬─────────────────────────┘
                        │ 库存充足，继续
                        ▼
┌─────────────────────────────────────────────────┐
│  阶段2: Redisson 锁保护业务临界区                  │
│  ───────────────────────────────────────────── │
│  • 目标: 保证用户级业务一致性                      │
│  • 特点: 多步操作在锁内串行执行                     │
│  • 逻辑:                                          │
│    ├─ 检查用户是否已抽奖（防重复）                   │
│    ├─ 写中奖记录                                   │
│    ├─ 发送奖品MQ                                   │
│    └─ DB乐观锁扣减（兜底）                          │
│  • 作用: 保证复杂业务逻辑的完整性                   │
└───────────────────────┬─────────────────────────┘
                        │
                        ▼
                    抽奖成功
```

**为什么不能只用 `INCR`？**

如果只用 `DECR` 扣减库存，虽然库存数字不会超卖，但会出现：
- 同一用户并发请求两次，库存扣了2次（用户重复中奖）
- 库存扣减成功但写记录失败，库存丢失（数据不一致）
- 扣库存和发奖之间出现间隙，其他请求读到不一致状态

**为什么不能只用 Redisson 锁？**

如果只用 Redisson 锁保护整个流程：
- 所有请求串行化，QPS 从 10 万骤降到数千
- 高并发下大量线程在锁前排队，响应时间飙升
- 锁本身成为性能瓶颈，系统吞吐急剧下降

### 5.3 分层互补的本质

```
┌──────────────────────────────────────────────────┐
│              流量漏斗模型                          │
│                                                  │
│  请求量: 100,000 QPS                              │
│  ┌──────────────────────────┐                    │
│  │ INCR/DECR 层 (无锁)       │  拦截 95%+ 无效请求  │
│  │ 10万 QPS                 │  (无库存/黑名单等)    │
│  └────────────┬─────────────┘                    │
│               │ 约 5,000 QPS 通过                  │
│  ┌────────────▼─────────────┐                    │
│  │ Redisson 锁层            │  保护有效请求的      │
│  │ 数千 QPS                 │  多步业务逻辑        │
│  └────────────┬─────────────┘                    │
│               │ 约 5,000 QPS                      │
│  ┌────────────▼─────────────┐                    │
│  │ DB 乐观锁层              │  最终一致性保证      │
│  └──────────────────────────┘                    │
└──────────────────────────────────────────────────┘
```

`INCR` 在最上层扛住海量并发（性能极高），过滤掉绝大多数无效请求；Redisson 锁只处理通过第一层的有效请求，量级已经大幅降低，锁的串行化开销在可接受范围内。

## 六、Redisson 看门狗机制详解

面试中追问「看门狗」时的标准回答：

```java
// 不指定 leaseTime → 启用看门狗
lock.tryLock(3, TimeUnit.SECONDS);

// 指定 leaseTime → 禁用看门狗
lock.tryLock(3, 10, TimeUnit.SECONDS);  // 10秒后自动释放，不续期
```

**看门狗工作流程：**

1. 加锁成功后，锁的超时时间设为 `lockWatchdogTimeout`（默认 30 秒）
2. 启动一个定时任务，每隔 `lockWatchdogTimeout / 3`（默认 10 秒）检查一次
3. 如果当前线程仍持有锁，将超时时间重置为 30 秒
4. 如果当前线程已释放锁或已死亡，定时任务停止

**这个机制解决了两个核心问题：**
- **业务执行时间长**：不用担心锁提前过期导致其他线程误获锁
- **线程崩溃/宕机**：看门狗也跟着停止，锁会在 30 秒后自动释放，不会死锁

## 七、什么场景下只用 `INCR` 就够了？

当满足以下**全部条件**时，可以只用 `INCR`，不需要分布式锁：

1. ✅ **单步操作**——只需要对单个数值做原子增减
2. ✅ **无前置条件检查**——不需要「先检查再操作」
3. ✅ **无多资源一致性需求**——只涉及一个 key
4. ✅ **允许最终一致**——不要求强一致

典型场景：
- **API 限流**：`INCR` + `EXPIRE` 实现「每分钟N次」
- **统计计数**：页面 PV、UV、在线人数
- **分布式序列号**：`INCR` 生成自增 ID
- **简单库存**：优惠券领取，扣减后不需要复杂后续逻辑

## 八、总结

| 问题 | `INCR`/`DECR` | Redisson 锁 |
|------|---------------|-------------|
| **解决什么** | 原子计数 | 临界区互斥 |
| **粒度** | 单条命令 | 一段代码 |
| **性能** | 10万+ QPS | 数千 QPS |
| **在系统中的角色** | 流量第一道防线 | 业务一致性保障 |
| **类比** | ETC 快速通道 | 停车场闸机 |

两者**不是二选一，而是协同配合**：`INCR`/`DECR` 在流量入口扛住高并发、快速过滤无效请求；Redisson 锁在业务层保证多步操作的一致性和完整性。理解「**不同粒度的并发问题用不同工具解决**」是高并发系统设计的核心思维。

在实际抽奖系统中，典型的分层是：**Lua 原子扣减（比纯 DECR 更安全）→ Redisson 锁（业务保护）→ DB 乐观锁（兜底）**。`INCR` 的思想是基础，Lua 脚本是其增强版（加了前置判断），Redisson 锁是更大粒度的保护——三者构成完整的并发安全体系。

## 记忆要点

- 场景对比：incr解决单步原子计数（极快），而Redisson锁保护多步业务临界区。
- incr局限：单命令虽原子，但无法保护扣减后的多步逻辑，故需配合分布式锁。
- Redisson特性：支持可重入，且自带看门狗自动续期，防止业务超时引发死锁。
- 互补关系：incr作高性能前置过滤，而Redisson锁兜底复杂业务逻辑一致性。
