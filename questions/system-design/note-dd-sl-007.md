# 【滴滴面经】你这个架构比较简单，理论上做到 5 万 QPS 应该不难，你觉得还有哪些优化空间？

## 一、题目背景与分析

面试官的潜台词有两层：

1. **肯定你现有方案**：基础架构（LB + 应用集群 + Redis 缓存 + MySQL 分库分表）确实能支撑 5 万 QPS 量级的短链跳转。
2. **考察架构纵深**：看你能不能从"能用"到"好用"再到"极致"，展示分层优化的系统性思维。

回答框架应该沿着请求生命周期逐层展开——**从用户浏览器到数据库存储**，每一层都有明确的优化手段和可量化的收益。

> **核心原则**：性能优化不是单点优化，而是全链路的分层压榨。面试时要给出"优化前 → 优化后"的量化对比，展示工程思维。

## 三、逐层优化方案

### 第一层：CDN 预缓存（拦截在源站之前）

**现状问题**：短链 302 跳转每次都要回源到应用服务器，即使命中 Redis，也消耗网关和应用资源。

**优化方案**：

- 短链跳转的核心场景是 **301/302 重定向**，响应体极小（仅 Location Header）。
- 将高频短链的 302 响应推送到 CDN 边缘节点，用户请求直接在 CDN 完成跳转，**根本不打到源站**。
- 使用 CDN 的 **主动预热（Prefetch）**：短链创建时即推送 302 响应到 CDN。

**收益量化**：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 源站 QPS | 50,000 | **~1,000**（仅未命中部分） |
| 用户延迟 | 30-80ms | **5-15ms**（边缘节点就近） |
| 源站带宽 | 高 | **降低 98%** |

> **面试加分点**：301（永久重定向）可以被浏览器本地缓存，对于不变的短链映射，应优先使用 301 而非 302，让浏览器直接缓存，连 CDN 都省了。

---

### 第二层：多级缓存 Local Cache + Redis（应用层核心优化）

**现状问题**：所有请求都打到 Redis，Redis 集群 QPS 成为瓶颈（单 Redis 实例约 10 万 QPS）。

**优化方案 — L1（本地缓存）+ L2（Redis 集群）**：

```
请求 → L1 Caffeine/Guava Cache (JVM内存)
         ↓ miss
       L2 Redis Cluster
         ↓ miss
       MySQL (回源并回填缓存)
```

**L1 本地缓存关键设计**：

- 使用 **Caffeine**（比 Guava Cache 性能高 30%+），采用 **W-TinyLFU** 淘汰策略——在突发流量下命中率显著优于传统 LRU。
- 设置短 TTL（如 10-60 秒），防止数据过期不一致。
- 容量预估：热点短链 10 万条 × 平均 Key 80B ≈ 8MB，JVM 内存完全无压力。

**收益量化**：

| 层级 | 延迟 | 单节点 QPS 上限 | 5万QPS所需节点 |
|------|------|----------------|---------------|
| L1 Caffeine | **~0.01ms** | **200万+** | 1 个应用节点即可 |
| L2 Redis | ~0.5ms | 10万/实例 | 1-2 个 Redis 实例 |
| MySQL | ~5ms | 2万/实例（读） | 多从库 |

> **核心收益**：L1 命中率如果达到 80%，意味着只有 **20% 的请求（约 1万 QPS）** 才需要查 Redis，Redis 压力直接降一个数量级。

**L1 / L2 一致性保障**：

- **Redis Pub/Sub 广播失效**：主表数据变更时，发布消息到 Channel，各应用节点订阅后主动 evict 本地缓存。
- **Redisson Cache + 事件驱动**：基于 Redis 的 Cache 实现自动同步。
- **兜底 TTL**：即使消息丢失，短 TTL（如 30s）保证最终一致。

---

### 第三层：连接池调优（消除隐性瓶颈）

**现状问题**：默认连接池配置（如 HikariCP `maximumPoolSize=10`）在高并发下成为瓶颈。

**优化方案**：

**数据库连接池（HikariCP）**：

```java
HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(50);        // 根据 (核心数 × 2 + 有效磁盘数) 调优
config.setMinimumIdle(20);
config.setConnectionTimeout(2000);     // 快速失败
config.setMaxLifetime(1800000);        // 30分钟回收
config.addDataSourceProperty("cachePrepStmts", "true");
config.addDataSourceProperty("prepStmtCacheSize", 250);
config.addDataSourceProperty("prepStmtCacheSqlLimit", 2048);
```

**Redis 连接池（Lettuce 池化）**：

- Lettuce 原生支持 **多线程共享连接**（基于 Netty 的非阻塞设计），比 Jedis 池化效率更高。
- 连接复用 + Pipeline 批量，大幅减少 TCP 握手开销。

**收益量化**：

| 优化项 | 优化前 | 优化后 |
|--------|--------|--------|
| DB 连接等待 | 高峰 20-50ms 等待 | **<1ms** |
| 线程阻塞 | Tomcat 线程池打满 | **线程利用率稳定 60%** |
| 系统整体 RT | P99 ~50ms | **P99 ~10ms** |

> **面试加分点**：提到连接池大小不是越大越好——数据库的 `innodb_thread_concurrency` 有限制，连接过多反而上下文切换开销大。**HikariCP 官方建议 `maximumPoolSize = (core_count × 2 + effective_spindle_count)`**。

---

### 第四层：Redis Pipeline 批量操作

**现状问题**：批量查询场景（如短链批量解析、统计批量 PV 更新）每条命令一次 RTT，N 条命令 = N × 0.5ms。

**优化方案**：

```java
// Pipeline 批量执行，N 条命令只需 1 次 RTT
redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
    for (String shortCode : codeList) {
        connection.get(("url:" + shortCode).getBytes());
    }
    return null;
});
```

**收益量化**：

| 场景 | 逐条执行 | Pipeline 批量 |
|------|---------|--------------|
| 100 个 Key 查询 | 50ms（100 × 0.5ms） | **~1.5ms**（1 次 RTT + 序列化） |
| 网络 RTT 占比 | 99% | **<10%** |

> **面试加分点**：补充 **MGET / MSET** 原子批量命令 vs Pipeline 的区别——MGET 是单命令多 Key，Pipeline 是多命令打包，两者可组合使用。

---

### 第五层：读写分离 + 数据库优化

**现状问题**：写主库和读从库混在一起，写操作（短链创建、PV 更新）影响读性能。

**优化方案**：

```
                    ┌─→ Master (写入: 短链创建、过期更新)
Application Layer ──┤
                    ├─→ Slave-1 (读: 短链跳转解析)
                    ├─→ Slave-2
                    └─→ Slave-3
```

- **写主库**：短链创建、PV 计数（先写 Redis INCR，定时批量同步到 DB）。
- **读从库**：短链解析查询走从库，负载均衡到多个 Slave。
- **ShardingSphere** / **MyCat** 中间件实现透明读写分离。

**收益量化**：

| 配置 | 读 QPS 上限 | 适用场景 |
|------|-----------|---------|
| 1 主 0 从 | ~2 万 | 初期 |
| 1 主 3 从 | ~8 万 | 当前需求 |
| 1 主 9 从 | ~20 万 | 未来扩容 |

> **注意点**：主从延迟（通常 <1s）可能导致"刚创建的短链立即查不到"——解决方案是写后强制读主（`@Master` 注解路由）或利用 **半同步复制** 降低延迟。

---

### 第六层：布隆过滤器（缓存穿透防线）

**现状问题**：恶意请求大量不存在的短码，绕过缓存直接打到数据库（缓存穿透），压垮 DB。

**优化方案**：

- 在 Redis 前加一层 **Bloom Filter**，存储所有已存在的短码。
- 查询时先过布隆过滤器：不存在 → 直接返回 404，不走 Redis 和 DB。

```
请求 → Bloom Filter → [不存在] → 直接 404（O(1)，~0.1ms）
                     → [可能存在] → Redis → DB
```

**收益量化**：

- 拦截 99%+ 的无效请求
- DB 负载在恶意攻击场景下**降低 95%**
- Bloom Filter 1000 万 Key 仅需 ~12MB 内存（1% 误判率）

> **面试加分点**：提到 **布谷鸟过滤器（Cuckoo Filter）**——支持删除操作，比传统布隆过滤器更适合短链过期删除场景。

## 四、全链路优化收益汇总

| 优化层 | 措施 | QPS 提升 | RT 降低 | 系统稳定性 |
|--------|------|---------|---------|-----------|
| CDN 预缓存 | 边缘节点拦截 | 源站降低 98% | 80→15ms | 抗 DDoS |
| 多级缓存 | L1+L2 | +5-10 倍 | 0.5→0.01ms | Redis 降载 80% |
| 连接池调优 | HikariCP 参数优化 | +30-50% | 50→10ms (P99) | 消除线程阻塞 |
| Pipeline 批量 | 批量查询 | 减少 N×RTT | 50→1.5ms | 网络效率提升 |
| 读写分离 | 1主多从 | 读容量线性扩展 | 主库负载降低 | 写读隔离 |
| 布隆过滤器 | 防穿透 | DB 降载 95% | 无效请求 O(1) | 容灾 |

**综合效果**：从 5 万 QPS 可推进到 **20-50 万 QPS**，P99 延迟从 50ms 降到 **<10ms**。

## 五、面试加分项

### 5.1 异步化与削峰

- **PV 统计异步化**：短链 PV 计数先写 Redis INCR，定时（每 10 秒）批量 Flush 到 DB，将高频写转为低频批量写。
- **消息队列削峰**：短链创建场景写入 MQ（Kafka/RocketMQ），消费者异步落库，削峰填谷。

### 5.2 JVM 层优化

- **G1GC 调优**：`-XX:MaxGCPauseMillis=50`，减少 GC 停顿对 P99 的影响。
- **堆外内存**：Netty 直接内存 + Caffeine off-heap，减少 GC 压力。

### 5.3 监控体系

- **全链路追踪**：SkyWalking / Jaeger 追踪每个短链跳转的完整调用链。
- **实时指标**：Prometheus + Grafana 监控缓存命中率、RT 分位线、DB 慢查询。

## 六、总结回答模板

> "从 5 万 QPS 往上优化，我认为核心思路是 **分层压榨每一层的性能剩余**：
>
> 1. **最外层**用 CDN 预缓存把 98% 的跳转拦截在边缘节点，源站几乎不需要承担流量；
> 2. **应用层**加 Caffeine 本地缓存作为 L1，配合 Redis 作为 L2，形成多级缓存，将缓存延迟从毫秒级压到微秒级；
> 3. **中间件层**做连接池调优和 Pipeline 批量操作，减少网络 RTT 和线程阻塞；
> 4. **存储层**做读写分离和分库分表，读容量线性扩展；
> 5. **防护层**用布隆过滤器防止缓存穿透。
>
> 综合下来，系统的瓶颈会不断后移，最终理论极限取决于 Redis 集群规模和分库分表数量，做到 **几十万 QPS** 是完全可行的。"
