# 【大厂二面】RocketMQ 消息积压怎么排查？

> 来源：小红书 Java大厂二面 RocketMQ踩坑复盘

## 二、排查思路——先定位瓶颈在哪一端

```
消息积压告警
        │
        ▼
┌───────────────────┐
│ Step1: 看消费TPS   │
│ 和堆积量趋势        │
└────────┬──────────┘
         │
    ┌────┴────┐
    ▼         ▼
 TPS正常    TPS骤降/为0
 但堆积涨    │
    │       ├── 消费者宕机？
    │       ├── 消费者卡死(死锁/GC)？
    │       ├── 消费逻辑异常(持续报错)？
    ▼       │
 Broker端   ▼
 瓶颈      消费端瓶颈
```

## 二、消费端排查与优化

### 常见消费端瓶颈

```
┌─────────────────────────────────────────────────┐
│              消费端瓶颈排查清单                     │
├─────────────────────────────────────────────────┤
│                                                  │
│  1. 消费者实例数够不够？                           │
│     → consumer实例数 < topic队列数 → 可以扩容      │
│                                                  │
│  2. 消费线程数配置多少？                           │
│     → consumeThreadMax 默认64，是否被改小？        │
│                                                  │
│  3. 单条消息处理耗时多少？                         │
│     → jstack看消费线程在做什么                     │
│     → 是否有慢SQL/外部调用超时                     │
│                                                  │
│  4. 消费者是否频繁报错重试？                       │
│     → 看消费者日志，是否大量异常                   │
│     → 毒消息导致无限重试                           │
│                                                  │
│  5. 消费者GC正常吗？                              │
│     → jstat -gcutil 看FGC频率和耗时              │
│                                                  │
└─────────────────────────────────────────────────┘
```

### 优化方案

```java
// 优化1：增加并发消费者
consumer.setConsumeThreadMin(20);
consumer.setConsumeThreadMax(64);

// 优化2：批量消费（拉取多条批量处理）
consumer.setPullBatchSize(32); // 每次拉取32条

// 优化3：异步消费——将IO操作扔进线程池
public void onMessage(MessageExt msg) {
    CompletableFuture.runAsync(() -> {
        processMessage(msg);  // 异步处理，释放消费线程
    }, ioThreadPool);
}

// 优化4：扩容消费者实例数
// ⚠️ 不能超过Topic队列数！RocketMQ的队列是消费并行度的上限
// Topic有16个队列 → 最多16个消费者实例并行
```

```
扩容消费者注意事项

Topic队列数 = 16

  消费者实例1 ◄── 分配队列 0-3
  消费者实例2 ◄── 分配队列 4-7
  消费者实例3 ◄── 分配队列 8-11
  消费者实例4 ◄── 分配队列 12-15
  消费者实例5 ◄── 空闲！没有队列分配 ❌

  → 消费者实例数不能超过队列数
  → 如果要更多消费者，先增加Topic队列数
```

### 顺序消费场景的扩容难题

```
顺序消费：同一业务Key的消息必须按顺序消费
  → 同一Key的消息只能在同一队列
  → 同一队列只能被一个消费者消费

  扩容方法：按业务Key做哈希分区
  consumer.setMessageQueueListener → 
    按userId % queueCount 分配到不同队列
  → 最多 queueCount 个消费者并行
  → 不同userId的消息可以并行消费
```

## 三、Broker端排查与优化

> **面试教训**：被问"Broker扛不住怎么办"时，要答刷盘策略+磁盘IO+参数调优，不要扯到分区数量（那是消费者并发的问题，不是Broker问题）。

```
┌─────────────────────────────────────────────────┐
│              Broker 端排查                        │
├─────────────────────────────────────────────────┤
│                                                  │
│  1. 磁盘IO瓶颈                                   │
│     → iostat -x 1 看磁盘利用率                    │
│     → %util > 80% → IO瓶颈                       │
│     → 解决：ASYNC_FLUSH换SSD / 增加磁盘           │
│                                                  │
│  2. 刷盘策略                                      │
│     → SYNC_FLUSH性能远低于ASYNC_FLUSH             │
│     → 非金融场景考虑改ASYNC_FLUSH                 │
│                                                  │
│  3. Broker CPU/内存                              │
│     → top看CPU使用率                              │
│     → 是否在大量做消息过滤/序列化                  │
│                                                  │
│  4. 网络带宽                                      │
│     → iftop/sar -n DEV看网卡流量                  │
│     → 万兆网卡打满→增加网卡或分流                  │
│                                                  │
│  5. Broker参数调优                                │
│     → sendMessageThreadPoolNums (发送线程池)      │
│     → pullMessageThreadPoolNums (拉取线程池)      │
│     → transientStorePoolEnabled (堆外内存池)      │
│                                                  │
└─────────────────────────────────────────────────┘
```

```bash
# 快速排查命令
iostat -x 1 5        # 磁盘IO情况
top -H -p <broker_pid>  # Broker线程CPU使用
sar -n DEV 1 5       # 网络流量
df -h                # 磁盘空间（满了会导致写入失败）
```

### Broker关键参数调优

```properties
# broker.conf 关键参数
# 发送线程池大小（默认1，生产建议增大）
sendMessageThreadPoolNums = 16

# 使用堆外内存池减少GC（默认关闭）
transientStorePoolEnabled = true

# 刷盘方式（高并发场景用异步）
flushDiskType = ASYNC_FLUSH

# PageCache缓存大小
maxTransferCountOnMessageInMemory = 1000
```

## 四、应急处理——积压严重时的临时方案

```
┌───────────────────────────────────────────┐
│           应急三板斧                        │
├───────────────────────────────────────────┤
│                                            │
│  1. 临时扩容（最快见效）                    │
│     → 增加消费者实例（不超过队列数）        │
│     → 增加消费线程数                        │
│                                            │
│  2. 降级跳过（积压太久消息已过期）           │
│     → 消息有TTL → 直接丢弃过期消息          │
│     → 消息无TTL → 转储到另一个Topic离线消费  │
│                                            │
│  3. 源头限流（治本）                        │
│     → 生产端限流降低TPS                     │
│     → 临时关闭非核心消息的生产               │
│                                            │
└───────────────────────────────────────────┘
```

## 五、面试加分点

1. **排查体系化**：先看趋势（突发vs缓慢），再定位瓶颈（消费端vs Broker），最后对症下药
2. **消费者不能无限扩容**：能说出消费者实例数受限于Topic队列数
3. **顺序消费场景**：能说出顺序消费时扩容需按业务Key分区
4. **Broker问题答到点**：刷盘策略、磁盘IO、参数调优，不要扯分区数量
5. **预防意识**：提到监控告警（堆积量/消费延迟）、容量规划、压测


## 结构化回答

**30 秒电梯演讲：** 消息积压本质是"生产速度>消费速度"。排查路径：先看消费速度是否正常（消费者没问题就看Broker），再区分是消费端瓶颈还是Broker瓶颈，最后对症下药——消费端扩容/优化，Broker侧调刷盘/调IO。

**展开框架：**
1. **排查顺序** — 先看消费TPS→正常则查Broker→异常则查消费端
2. **消费端优化** — 扩容(不超过队列数)、异步消费(批量拉取+线程池)、减少单条处理时间
3. **Broker瓶颈答** — 刷盘策略(ASYNC→更高效)、磁盘IO排查(iostat)、参数调优

**收尾：** 这块我踩过坑——要不要深入聊：消费者扩容到多少合适？有没有上限？（不能超过Topic队列数）？

## 苏格拉底式面试追问

| 追问层级 | 面试官可能这样问 | 高分回答方向 |
|----------|------------------|--------------|
| 目标追问 | 消息积压的核心矛盾是什么？ | 生产速度>消费速度，消息在Broker堆积导致延迟增大、可能OOM——要么提速消费、要么降速生产 |
| 证据追问 | 怎么定位积压根因？分几种情况？ | 分情况：消费者故障（宕机/bug）、消费慢（单条耗时/并发不够）、生产突增、Broker问题；用监控区分 |
| 边界追问 | 临时扩消费者一定有效吗？ | 不一定。如果是单条消费耗时高（如慢SQL），扩消费者无效；要分清是并发不够还是单条慢 |
| 反例追问 | 直接加机器扩消费者好吗？ | 不一定。消息顺序消费场景不能随意扩（破坏顺序）、下游DB扛不住、成本高；要按场景 |
| 风险追问 | 积压处置的风险？ | 扩消费者破坏顺序、下游被压垮、丢消息（跳过堆积）、降级影响业务 |
| 验证追问 | 怎么验证积压解决？ | 积压量监控下降、消费TPS提升、延迟恢复正常、无新增积压 |
| 沉淀追问 | 积压处置怎么沉淀？ | 规范：监控告警、分级预案（扩消费者/降级/限流）、顺序消费方案、容量规划 |

### 现场对话示例
**面试官**：RocketMQ消息积压怎么排查？
**候选人**：先定位根因：消费者故障/消费慢/生产突增/Broker问题，看监控区分；再对症——扩消费者、优化单条耗时、限流降速、临时降级。
**面试官**：直接扩消费者一定有效吗？
**候选人**：不一定，单条消费耗时高（慢SQL）扩并发无效；顺序消费场景不能随意扩破坏顺序；要分清是并发不够还是单条慢。
**面试官**：积压处置有什么风险？
**候选人**：扩消费者破坏顺序、下游被压垮、跳过堆积丢消息、降级影响业务，要按场景分级处置并监控验证。
