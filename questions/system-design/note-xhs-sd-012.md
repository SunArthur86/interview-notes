## 三、第二阶段：快速扩容（5-15分钟）

| 扩容对象 | 操作 | 工具 |
|---------|------|------|
| 应用服务 | K8s一键扩Pod,翻倍实例数 | kubectl scale / HPA |
| Redis | 扩容集群分片数 | Redis Cluster reshard |
| MQ | 增加Broker节点,扩分区数 | Kafka reassignment |
| MySQL | 开启读写分离,新增从库 | ShardingSphere / ProxySQL |
| CDN | 静态资源加急预热 | CDN控制台 |

```bash
# K8s快速扩容
kubectl scale deployment order-service --replicas=20
# 或开启HPA自动扩缩容
kubectl autoscale deployment order-service --min=5 --max=50 --cpu-percent=70
```

## 六、第五阶段：长效架构优化

| 措施 | 具体内容 | 目标 |
|------|---------|------|
| 全链路压测 | 模拟真实流量，摸清极限 | 预知系统容量 |
| 架构升级 | 多级缓存、分库分表、服务隔离 | 提升容量上限 |
| 自适应扩容 | K8s HPA自动扩缩容 | 应对突发流量 |
| 阈值预调 | 合理限流阈值，预留冗余 | 留出缓冲空间 |
| 复盘优化 | 梳理突增原因，优化薄弱环节 | 杜绝复发 |

## 七、面试加分点

1. **先限流后排查的原因**：线上突发流量最致命的是系统雪崩——如果先排查问题，几分钟内超高流量就会打垮服务→数据库→中间件→全线瘫痪。先限流降级可以快速止损，给排查留出时间
2. **雪崩的三层传播**：服务超时→线程池打满→上游等待超时→上游线程池打满→整条链路瘫痪。限流熔断在最早环节切断传播链
3. **容量规划公式**：系统极限QPS ≈ 单实例QPS × 实例数 × 0.7(安全系数)。平时应预留3-5倍流量冗余
4. **混沌工程**：通过Chaos Mesh等工具定期注入故障（杀Pod、延迟网络、磁盘满），验证系统的自动恢复能力
5. **SLO/SLA意识**：定义99.99%可用性 = 每月停机不超过4.3分钟，突发流量处理速度直接关系到SLO达标率

| 验证追问 | 怎么验证止血有效？ | 监控QPS/错误率/延迟恢复、用户投诉、下游压力、复盘确认根因 |
| 沉淀追问 | 应急预案怎么沉淀？ | 规范：分级预案（限流→降级→扩容）、值班SOP、容量水位告警、定期演练 |

### 现场对话示例
**面试官**：业务QPS突然提升100倍，你会怎么做？
**候选人**：先保可用：秒级限流/熔断/降级止血；分钟级弹性扩容、读写分离；小时级缓存预热SQL优化；架构重构是事后，止血优先。
**面试官**：直接重构架构对吗？
**候选人**：不对，线上紧急重构风险大见效慢，必须先限流降级保可用，架构优化是事后复盘后做的事。
**面试官**：扩容一定有效吗？
**候选人**：不一定，依赖下游（DB/第三方）扩容无效、启动慢的服务来不及，要先限流止血再分析瓶颈。
