# 【滴滴面经】单 Redis 的 QPS 大概是什么量级？

## 一、直接回答

单实例 Redis 的 QPS 基准量级是 **8万~12万**（读操作，小 Value，同机房）。

这个数字来自 Redis 官方基准测试（`redis-benchmark`），也是业界公认的参考值。但实际场景中，QPS 受到多种因素影响：

| 因素 | 影响方向 | 说明 |
|------|---------|------|
| Value 大小 | QPS↓ | Value 越大，网络传输和序列化耗时越长 |
| 命令复杂度 | QPS↓ | `GET/SET` 最快；`SORT/HGETALL/LRANGE` 慢得多 |
| 网络环境 | QPS↓ | 同机房 > 跨机房；万兆 > 千兆 |
| Pipeline | QPS↑ | 批量执行，减少网络 RTT，可提升 **3~10 倍** |
| 6.0 多线程IO | QPS↑ | 网络读写并行化，可提升 1.5~2 倍 |

## 二、Redis 单线程模型详解

### 2.1 为什么 Redis 是单线程？

Redis 的核心命令处理是单线程的（6.0 之前网络IO也是单线程），设计原因：

1. **纯内存操作**：内存读写速度极快（~100ns），单线程处理已能达到 10万+ QPS
2. **避免锁竞争**：多线程操作共享数据需要加锁，锁竞争的开销可能比单线程更大
3. **无上下文切换**：单线程避免了线程切换的 CPU 开销（每次切换 ~5μs）
4. **实现简单**：单线程模型天然保证命令的原子性和顺序执行

### 2.2 epoll 事件循环

Redis 使用 **Reactor 模式 + epoll** 实现事件驱动：

```
┌─────────────────────────────────────────┐
│           Redis 主事件循环               │
│                                         │
│  ┌─────────┐    ┌──────────────────┐   │
│  | epoll_wait │→| 处理就绪事件      │   │
│  └─────────┘    │  ├─ 读事件 → 解析命令 │
│       ↑         │  ├─ 执行命令         │
│       │         │  └─ 写事件 → 返回结果 │
│  ┌─────────┐    └──────────────────┘   │
│  | IO多路复用 |←                        │
│  └─────────┘                           │
└─────────────────────────────────────────┘
```

**核心流程**：
```
1. epoll_wait() 阻塞等待，返回就绪的 FD（文件描述符）列表
2. 遍历就绪 FD：
   a. 读事件 → 从 Socket 读取客户端命令（read()）
   b. 解析命令协议（RESP 协议）
   c. 执行命令（查内存数据结构）
   d. 写事件 → 将结果写入 Socket 缓冲区（write()）
3. 处理定时任务（过期Key清理、统计等）
4. 回到 Step 1
```

> **关键理解**：epoll 的优势是 **O(1)** 的事件通知复杂度。即使有 10 万个连接，`epoll_wait` 也只返回有事件的 FD，不会遍历所有连接。但**命令执行是串行的**——一次只执行一个命令。

### 2.3 单线程为什么能达到 10万 QPS？

```
单条命令耗时 = 命令解析 + 内存操作 + 结果写入
            ≈ 1μs（解析） + 0.1μs（内存操作） + 1μs（写入）
            ≈ 2~3μs

理论 QPS = 1,000,000μs / 2~3μs ≈ 30万~50万 QPS
```

理论值远高于实际值，差距来自：
- **网络 RTT**：客户端到 Redis 的往返延迟（同机房 ~0.5ms = 500μs）
- **系统调用开销**：`read()/write()` 的系统调用开销
- **事件循环开销**：epoll_wait 本身有微小延迟
- **实际 QPS ≈ 10万**，瓶颈通常在网络 IO 而非 CPU

## 三、影响 QPS 的关键因素

### 3.1 网络带宽限制

| 网络环境 | 带宽 | 理论 QPS 上限（Value=1KB） |
|---------|------|--------------------------|
| 千兆网卡 | 1Gbps ≈ 125MB/s | ~12万 QPS |
| 万兆网卡 | 10Gbps ≈ 1.25GB/s | ~120万 QPS |

> 千兆网卡下，如果 Value 平均 1KB，带宽在 ~12万 QPS 时就会打满。所以**单实例 QPS 天花板往往被网卡带宽卡住**。

### 3.2 命令类型对 QPS 的影响

| 命令 | 复杂度 | 典型 QPS | 说明 |
|------|--------|---------|------|
| `GET/SET/INCR` | O(1) | **10万+** | 最快 |
| `HGET/HSET` | O(1) | **10万** | 接近 GET/SET |
| `LRANGE 0 10` | O(N) | **8万** | N 小时影响不大 |
| `SORT` | O(N+M*logM) | **1万~3万** | 集合排序，CPU密集 |
| `KEYS *` | O(N) | **全库阻塞** | 生产禁用！ |
| `FLUSHALL` | O(N) | **阻塞数秒** | 摧毁性操作 |

### 3.3 Pipeline 优化（面试重点）

**原理**：客户端批量发送多条命令，不需要等待每条命令的响应，减少网络 RTT。

```python
# 无 Pipeline：每条命令一次 RTT
for code in short_codes:
    r.get(f"url:{code}")    # 10次 RTT = 10 × 0.5ms = 5ms

# 有 Pipeline：批量发送，一次 RTT
pipe = r.pipeline()
for code in short_codes:
    pipe.get(f"url:{code}")
results = pipe.execute()      # 1次 RTT = 0.5ms
```

| 方式 | RTT 次数 | 10条命令耗时 | QPS 提升倍数 |
|------|---------|-------------|-------------|
| 逐条执行 | 10次 | 5ms | 1x |
| Pipeline | 1次 | 0.6ms | **~8x** |

> **面试加分点**：Pipeline 不是原子操作（中间可能插入其他客户端的命令），如果需要原子性，用 `MULTI/EXEC` 事务或 Lua 脚本。

### 3.4 BigKey 对 QPS 的影响

```bash
# 一个包含 100万元素的 List
LPUSH huge_list val1 val2 ... val1000000

# LRANGE huge_list 0 -1 → 传输 ~10MB 数据
# 单线程模型下，这条命令执行期间所有其他命令都被阻塞！
```

**BigKey 的危害**：
- 阻塞事件循环：单个命令耗时 >10ms，影响所有客户端
- 网络带宽：大量数据传输占满网卡
- 内存倾斜：集群场景下导致某个分片内存远超其他分片

**发现与解决**：
```bash
# 发现 BigKey
redis-cli --bigkeys

# 解决方案
# 1. 拆分：将大集合拆成多个小集合
# 2. 压缩：Value 使用压缩格式
# 3. 异步删除：UNLINK 替代 DEL（4.0+）
```

## 四、Redis 6.0 多线程 IO 模型

### 4.1 为什么引入多线程 IO？

单线程模型的瓶颈不在 CPU 计算（内存操作极快），而在**网络读写**：
- 大量连接的 `read()/write()` 系统调用成为瓶颈
- 千兆/万兆网卡下，单线程处理 Socket IO 效率不足

### 4.2 多线程 IO 架构

```
┌──────────────────────────────────────────────┐
│             Redis 6.0 多线程 IO               │
│                                              │
│  ┌─────────────┐                             │
│  | 主线程（命令执行）|  ← 仍然单线程执行命令      │
│  └──────┬──────┘                              │
│         │                                     │
│    ┌────┴────┐                                │
│    │ 任务分发  │                                │
│    ┌────┬────┐                                │
│    ↓    ↓    ↓                                │
│  IO线程1 IO线程2 IO线程3                       │
│  (read) (read) (read)  ← 并行读取Socket数据    │
│  (write)(write)(write) ← 并行写入响应          │
└──────────────────────────────────────────────┘
```

**核心变化**：
- **网络读写（read/write）并行化**：使用多个 IO 线程处理
- **命令执行仍然单线程**：保证原子性和无锁

### 4.3 配置与性能提升

```conf
# redis.conf 开启多线程 IO
io-threads 4          # IO线程数（建议 CPU核数的一半）
io-threads-do-reads yes  # 读操作也使用多线程
```

| 配置 | QPS 提升对比 | 说明 |
|------|------------|------|
| 单线程（5.x） | 10万（基准） | 所有操作串行 |
| 多线程IO（6.0），2线程 | ~15万 | +50% |
| 多线程IO（6.0），4线程 | ~18~20万 | +80%~100% |

> **注意**：多线程 IO 对**小 Value + 高 QPS** 场景提升明显，对大 Value 场景提升有限（瓶颈在带宽而非 IO 线程）。

## 五、超过单机 QPS 怎么办？

### 5.1 水平扩展方案

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **主从复制** | 读写分离，读分担到从节点 | 读多写少 |
| **Redis Cluster** | 16384 个槽位，自动分片 | 数据量大、QPS高 |
| **Codis / Twemproxy** | 代理层分片 | 需要平滑迁移 |
| **客户端分片** | 应用层直接分片 | 追求极致性能 |

### 5.2 Redis Cluster 分片原理

```
# 16384 个槽位分配到多个节点
Node A: slots 0-5460        → key1, key4, key7...
Node B: slots 5461-10922    → key2, key5, key8...
Node C: slots 10923-16383   → key3, key6, key9...

# Key 到 Slot 的映射
slot = CRC16(key) % 16384
```

## 六、面试加分点

1. **精准数字**：单实例基准 8~12万 QPS，Pipeline 可提升到 50~100万命令/秒
2. **底层原理**：能讲清楚 epoll 事件循环的工作流程
3. **6.0 多线程**：区分"IO多线程"和"命令执行多线程"，强调命令执行仍单线程
4. **BigKey 危害**：用具体例子说明单线程模型下 BigKey 的阻塞效应
5. **容量规划**：知道什么时候单实例不够，需要集群——通常 QPS > 8万 或数据 > 10GB
6. **短链场景**：短链系统天然适合 Redis，`GET short_code → long_url` 是 O(1) 操作，单实例可支撑 10万 QPS 的短链跳转

---

## 七、总结一句话

> 单实例 Redis QPS 基准是 **8~12万**，瓶颈不在 CPU 计算而在**网络 IO**（单线程处理 Socket 读写）。Pipeline 批量执行可提升 3~10 倍；Redis 6.0 引入多线程 IO 可提升约 1 倍；如果仍不够，使用 Redis Cluster 水平扩展，每增加一个分片，整体 QPS 近似线性增长。短链系统这种 `短码→URL` 的纯 KV 查询场景，单 Redis 10万 QPS 完全够用，亿级数据再用集群。

## 记忆要点

- 基准量级：单机Redis读写QPS约8万至10万
- 高性能因：纯内存操作加epoll多路复用，单线程免锁无切换
- 瓶颈限制：千兆网卡约12万QPS即达上限，禁用O(N)复杂命令
