# 【快手Java一面】热点Key突然失效，数据库被打到限流，除了互斥锁还有什么方案？

> 来源：快手Java开发一面场景题复盘（小红书）

## 一、缓存击穿 vs 穿透 vs 雪崩

```
区分三种缓存异常：

  缓存击穿（本题）          缓存穿透             缓存雪崩
  ┌────────────┐         ┌────────────┐      ┌────────────┐
  │   1个热点Key │         │ 不存在的Key │      │ 大量Key同时 │
  │   突然失效   │         │ 每次都查不到 │      │   失效      │
  │      ↓      │         │      ↓      │      │      ↓      │
  │  大量请求    │         │  每次请求都  │      │  请求洪流   │
  │  打到DB     │         │  打到DB     │      │  打到DB     │
  └────────────┘         └────────────┘      └────────────┘
  热点Key TTL到期          恶意攻击/BUG        批量设置相同TTL
```

### 一级防御：Caffeine 本地缓存

```java
// Caffeine本地缓存，作为Redis的前置防线
Cache<String, Product> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(3, TimeUnit.SECONDS)  // 短TTL：3秒
    .build();

public Product getProduct(String key) {
    // 1. 先查本地缓存（纳秒级响应）
    Product product = localCache.getIfPresent(key);
    if (product != null) {
        return product;  // 80%请求在这里返回
    }

    // 2. 本地缓存Miss → 查Redis
    product = getFromRedis(key);
    if (product != null) {
        localCache.put(key, product);
        return product;
    }

    // 3. Redis也Miss → 查DB（加互斥锁）
    return getFromDBWithLock(key);
}

// 为什么Caffeine能防击穿？
// 假设10000个请求同时来：
// - 没有Caffeine：10000个请求同时打到Redis→Redis Miss→打到DB
// - 有Caffeine：第1个请求miss→查Redis→填充Caffeine
//   剩余9999个请求在3秒TTL内全部命中Caffeine→不打DB！
```

### 二级防御：Redis 逻辑过期

```
传统方案（物理过期）：
  Redis Key设TTL=30分钟 → 到期自动删除 → 下一个请求发现Miss → 查DB

逻辑过期方案（永不过期）：
  Redis Key不设TTL（永久存在），但在Value中存一个逻辑过期时间

  ┌─────────────────────────────────────────┐
  │ Redis Value (JSON):                     │
  │ {                                       │
  │   "data": {商品数据},                    │
  │   "expireTime": 1700000000000  ← 逻辑过期│
  │ }                                       │
  └─────────────────────────────────────────┘
```

```java
// 逻辑过期方案实现
public Product getProductWithLogicalExpire(String key) {
    String json = redis.get(key);  // Redis Key不设TTL，永不过期
    if (json == null) {
        // 正常情况不会走到这里（Key永不物理过期）
        return getFromDBWithLock(key);
    }

    CacheData cacheData = JSON.parseObject(json, CacheData.class);

    // 判断是否逻辑过期
    if (cacheData.getExpireTime().after(new Date())) {
        // 未过期 → 直接返回
        return cacheData.getData();
    }

    // 已过期 → 返回旧数据 + 异步刷新
    CACHE_REFRESH_POOL.submit(() -> {
        // 只让一个线程刷新（用Redis SETNX做分布式锁）
        if (redis.setnx(key + ":lock", "1", 10)) {
            try {
                Product fresh = getFromDB(key);
                CacheData newData = new CacheData(fresh, nextExpireTime());
                redis.set(key, JSON.toJSONString(newData));
            } finally {
                redis.del(key + ":lock");
            }
        }
    });

    // 返回旧数据（用户无感知！）
    return cacheData.getData();
}

// 逻辑过期的优势：
// 1. 用户永远不会等待DB查询（总是返回缓存数据）
// 2. 没有锁等待（旧数据直接返回）
// 3. 数据短暂不一致（几秒延迟），但用户体验最好
// 4. 永远不会缓存击穿（Key不会物理失效）
```

### 核弹方案：热点Key探测 + 主动预热

```
基于Flink的热点Key实时探测：

  Redis访问日志 → Flink实时统计 → 识别热点Key → 主动刷新

  ┌────────────────────────────────────────────┐
  │ Flink 实时热点探测                           │
  │                                            │
  │ 窗口：每10秒统计一次                          │
  │ 阈值：QPS > 1000 的Key标记为热点              │
  │                                            │
  │ 热点Key列表：                                │
  │   product:10086    QPS=5000  ← 热点         │
  │   product:10087    QPS=3000  ← 热点         │
  │   product:10088    QPS=50    ← 普通         │
  └──────────────────┬─────────────────────────┘
                     │
                     ▼
  ┌────────────────────────────────────────────┐
  │ 主动预热                                     │
  │                                            │
  │ 对热点Key：                                  │
  │ 1. 提前刷新（在TTL到期前刷新）                │
  │ 2. 延长TTL（从30分钟延长到2小时）             │
  │ 3. 推送到所有节点的本地缓存                   │
  └────────────────────────────────────────────┘
```

## 三、方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **互斥锁** | 只让一个线程查DB | 数据强一致 | 有锁等待延迟 | 低并发场景 |
| **Caffeine本地缓存** | 短TTL本地拦截 | 极快(纳秒级) | 各节点数据不一致 | 第一道防线 |
| **逻辑过期** | 永不物理过期，异步刷新 | 用户无感知 | 短暂数据不一致 | 高并发热点Key |
| **热点探测** | 提前预知热Key并刷新 | 从根本解决 | 实现复杂 | 超大流量场景 |

## 四、面试加分点

1. **提到缓存预热**：系统启动/大促前，主动将热点数据加载到缓存，避免冷启动击穿
2. **提到Redis集群热点Key**：单Key只能存在一个分片上，超热Key可以做"Key分片"（key:1, key:2...分散到多个分片）
3. **提到布隆过滤器**：对于缓存穿透（查不存在的Key），用布隆过滤器挡住
4. **提到数据一致性**：逻辑过期方案牺牲了短暂一致性换取可用性，面试中能讨论CAP权衡是加分项
5. **提到监控告警**：Redis的bigkey/hotkey监控，提前发现潜在热点Key

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：缓存击穿你说用"逻辑过期 + 异步刷新"替代互斥锁，为什么互斥锁不够好？**

因为互斥锁有等待开销。热点 key 失效时，互斥锁让一个线程查 DB 重建缓存，其他线程等待——等待的线程阻塞，RT 飙高（等待时间 = 重建时间，可能几百毫秒）。逻辑过期的思路是"永不真正过期"——缓存值里存 expireTime 字段，查询时逻辑判断过期就返回旧值 + 触发异步刷新（新线程重建），用户拿到的虽然是旧值但 RT 正常（无等待）。决策依据：热点 key 的场景下，"快速返回旧值"比"等待拿新值"体验好，业务容忍秒级旧数据（如商品库存差几件可接受）。

### 第二层：证据与定位

**Q：缓存击穿发生了（DB 被打到限流），你怎么确认是哪个 key 失效导致的？**

查 DB 慢查询和 Redis 失效日志：
1. DB 慢查询——看哪条 SQL 被高频查询（如 `SELECT * FROM product WHERE id = 123` 被 QPS 万级查询），反推这个 key 失效了。
2. Redis 监控——看 Redis 的 evict（key 被淘汰）或 expire（key 自然过期）事件，时间点与 DB 压力飙升吻合的 key 就是击穿源。
3. 应用日志——看缓存 miss 日志，哪个 key 的 miss 量突增。

### 第三层：根因深挖

**Q：热点 key 配了 TTL 1 小时，但不是整点失效（随机 TTL），为什么还会击穿？**

最可能是这个 key 是"全局唯一热点"——所有请求都查同一个 key（如首页推荐位的配置），即使随机 TTL，失效那一瞬间还是有万级请求同时发现 miss，全部穿透到 DB。随机 TTL 只能避免"多个 key 同时失效"（缓存雪崩），不能避免"单 key 失效时的并发重建"（缓存击穿）。根因是单 key 的并发重建问题，解法是逻辑过期（异步刷新，不阻塞）或互斥锁（只让一个线程重建）。随机 TTL 是防雪崩的手段，对击穿无效。

**Q：为什么不直接不给热点 key 设 TTL（永久有效），不就不会失效击穿了吗？**

因为数据会变更，不设 TTL 用户永远看到旧值。如商品库存 key 永不过期，扣库存后缓存还是旧值，用户看到"有货"但实际已售罄，下单失败。必须设 TTL 或主动更新保证数据新鲜。逻辑过期是折中——不设 Redis 层面的 TTL（key 一直在），但在 value 里存 expireTime，逻辑判断过期后异步刷新。这样 key 永远在（不击穿），数据也能刷新（异步更新）。

### 第四层：方案权衡

**Q：逻辑过期方案中，异步刷新的线程怎么保证只有一个线程去刷新（避免多个线程同时刷）？**

用分布式锁或标记位：
1. 标记位——缓存值里加 `refreshing: true/false`，第一个发现过期的线程 CAS 设置 `refreshing: true`，成功则异步刷新，失败的线程直接返回旧值。
2. 分布式锁——第一个线程抢锁（Redis SETNX），成功则刷新，失败返回旧值。
权衡：标记位是单实例内的（多实例各自可能刷一次，但 DB 能承受少量重复刷新），分布式锁是跨实例的（全局只有一个刷）。如果"刷新成本高"（如查大表聚合）用分布式锁，如果"刷新成本低"用标记位即可。热点 key 场景刷新成本低（单 key 查 DB），标记位够用。

**Q：为什么不直接用 Redis 的 Pub/Sub 通知所有实例刷新缓存，而要各自逻辑过期？**

因为 Pub/Sub 不可靠。Redis Pub/Sub 是"发即弃"——订阅者不在线时消息丢失。如果某实例在刷新通知发送时网络抖动没收到，它的缓存一直是旧值直到自然过期。而且 Pub/Sub 是广播，所有实例都刷新（N 个实例 N 次 DB 查询），不是"一个刷新其他复用"。逻辑过期 + 标记位是"各自判断 + 抢占式刷新"，简单且无依赖。Pub/Sub 适合"配置变更通知"（丢失可接受，定期对账兜底），不适合"缓存刷新协调"。

### 第五层：验证与沉淀

**Q：你怎么证明缓存击穿防御真的有效（DB 不被打穿）？**

混沌演练：
1. 主动制造击穿——在压测时手动删除热点 key（模拟失效），观察 DB QPS 是否飙升。
2. 对比方案——分别用"无防御""互斥锁""逻辑过期"三种方案跑同样的击穿场景，对比 DB QPS 峰值和用户 RT。
3. 线上验证——监控 DB QPS 曲线，正常应该是平稳的（即使有 key 失效），没有毛刺。

**Q：缓存击穿防御怎么沉淀？**

1. 逻辑过期 SDK——封装"缓存值带 expireTime + 逻辑判断 + 异步刷新"成通用组件，业务接入只配 refresh 逻辑。
2. 热点探测——用 Flink 实时统计 key 访问频率，自动识别热点 key 并提升防御级别（主动预热 + 逻辑过期）。
3. 应急预案——"DB 被打穿 → 自动触发限流 + 人工介入重建缓存"写入 runbook。


## 结构化回答

**30 秒电梯演讲：** 缓存击穿是热点Key突然失效，大量请求直接打到数据库。除了互斥锁外，三级防御：(1)本地缓存(Caffeine)设短TTL做第一层拦截。

**展开框架：**
1. **缓存击穿=单Key失效** — 缓存击穿=单Key失效→DB被打穿；缓存雪崩=大量Key同时失效；缓存穿透=查不存在的Key
