# 【面试题】Nacos AP 和 CP 实现原理

> 来源：面试随手记 — 分布式一致性

## 一、CAP 理论基础

```
CAP 定理：分布式系统最多同时满足三个中的两个

         Consistency (一致性)
            ╱ ╲
           ╱   ╲
          ╱     ╲
         ╱  只能  ╲
        ╱  选两个  ╲
       ╱ ──────── ╲
      ╱            ╲
 Availability    Partition
 (可用性)        Tolerance
                (分区容错)

  现实：网络分区(P)必然发生 → 实际是 C 和 A 二选一

  AP：分区时选择可用性 → 返回可能不一致的数据
      → Nacos默认、Eureka

  CP：分区时选择一致性 → 拒绝服务直到数据一致
      → Zookeeper、Consul(强一致模式)

  Nacos 同时支持两种！通过实例类型自动选择协议。
```

## 二、Nacos AP 模式 —— Distro 协议

### 工作原理

```
Distro 协议（Nacos自研的AP协议）：

  节点A        节点B        节点C        节点D
    │            │            │            │
    │  每个节点负责一部分数据的写入              │
    │  hash(service) % N → 决定哪个节点处理      │
    │            │            │            │

  Client 注册 "order-service"
       │
       ▼
  连接到 节点B
       │
       ├──> 节点B 检查：order-service 是否归我管？
       │    hash("order-service") % 4 = 2 → 归节点C管
       │
       ├──> 转发给 节点C 处理
       │
       └──> 节点C 写入本地 + 异步同步给 A、B、D

  核心特征：
  ✅ 每个节点都能接收请求（但只处理自己负责的部分）
  ✅ 异步同步给其他节点（最终一致）
  ✅ 节点宕机不影响其他节点（高可用）
  ❌ 短暂数据不一致（同步有延迟）
```

### Distro 数据同步流程

```
  正常同步：
  ┌─────────────────────────────────────────┐
  │ 节点C (order-service的负责节点)           │
  │   │                                     │
  │   ├── 每隔5秒 → 全量同步                  │
  │   │   → 把自己负责的所有数据同步给其他节点  │
  │   │                                     │
  │   ├── 有新数据 → 增量同步                 │
  │   │   → 立即发送变更的数据给其他节点       │
  │   │                                     │
  │   └── 接收方确认后 → 同步完成             │
  └─────────────────────────────────────────┘

  节点宕机恢复后：
  ┌─────────────────────────────────────────┐
  │ 节点C 恢复                               │
  │   │                                     │
  │   ├── 向其他节点请求全量数据              │
  │   │   → 获取自己负责的数据                │
  │   │                                     │
  │   ├── 开始正常提供注册服务                │
  │   │                                     │
  │   └── 开始正常同步                        │
  └─────────────────────────────────────────┘
```

### Distro vs Gossip

```
  Distro（Nacos）：
    - 每个节点负责特定数据子集
    - 负责节点 → 主动推给其他节点
    - 收敛速度快（主动推送）

  Gossip（Consul）：
    - 每个节点随机选择另一个节点交换数据
    - 像病毒传播一样扩散
    - 收敛速度较慢（随机传播）
    - 但去中心化程度更高
```

## 三、Nacos CP 模式 —— Raft 协议

### 工作原理

```
Raft 协议（强一致性共识算法）：

  ┌─────────────────────────────────────────────────┐
  │                Raft 集群                         │
  │                                                 │
  │    ┌──────────┐                                 │
  │    │  Leader   │ ← 所有写请求必须经过Leader       │
  │  ┌─┤ (节点C)   ├─┐                               │
  │  │ └──────────┘ │                               │
  │  │     │        │                               │
  │  │  写入日志    │                               │
  │  │     │        │                               │
  │  │     ▼        │                               │
  │  │ 复制给Follower│                               │
  │  │     │        │                               │
  │  ▼     ▼        ▼                               │
  │ ┌──────────┐ ┌──────────┐ ┌──────────┐         │
  │ │Follower  │ │Follower  │ │Follower  │         │
  │ │ (节点A)   │ │ (节点B)  │ │ (节点D)   │         │
  │ └──────────┘ └──────────┘ └──────────┘         │
  │                                                 │
  │ 多数派(N/2+1)确认 → 提交                         │
  │ 4个节点中3个确认 → 提交成功                       │
  └─────────────────────────────────────────────────┘

  核心特征：
  ✅ 强一致性（所有客户端看到相同数据）
  ✅ Leader故障 → 重新选举 → 新Leader接管
  ❌ 选举期间不可用（几秒到几十秒）
  ❌ 少数派节点不可用 → 无法达成多数 → 写入阻塞
```

### Raft Leader 选举过程

```
  初始状态：所有节点都是 Follower

  Step 1: 超时触发选举
  ┌────────────────────────────────────────┐
  │ Follower 在 electionTimeout(150-300ms) │
  │ 内没收到 Leader 心跳                    │
  │ → 变成 Candidate                       │
  │ → 发起选举                              │
  └────────────────────────────────────────┘

  Step 2: 请求投票
  ┌────────────────────────────────────────┐
  │ Candidate 向所有节点发送 RequestVote    │
  │ "我的term=2, log index=100, 请投我"    │
  └────────────────────────────────────────┘

  Step 3: 投票
  ┌────────────────────────────────────────┐
  │ 其他节点收到投票请求：                    │
  │ → Candidate的term比我高？               │
  │ → Candidate的log至少和我一样新？         │
  │ → 我还没投过票？                         │
  │ 全是 → 投赞成票                          │
  └────────────────────────────────────────┘

  Step 4: 获得多数派 → 成为Leader
  ┌────────────────────────────────────────┐
  │ Candidate 获得多数派投票 → 成为 Leader   │
  │ → 开始发送心跳维持权威                   │
  │ → 所有写请求经过 Leader                  │
  └────────────────────────────────────────┘

  脑裂处理：
  │   │          │   │
  │ A │←─分区──→│ C │
  │ B │          │ D │
  │(少数派)      │(多数派)
  │             │Leader C
  │             │正常工作 ✅
  │ A、B选举    │
  │ 但不够多数  │
  │ → 不能写   │
  │ → 不影响C  │
```

## 四、Nacos 如何选择 AP / CP

```
Nacos 通过实例类型自动选择协议：

  临时实例 (ephemeral = true)  ← 默认！
    → 使用 Distro 协议 (AP模式)
    → 客户端通过心跳维持注册
    → 心跳停止 → Nacos主动删除实例
    → 适合：微服务注册发现

  永久实例 (ephemeral = false)
    → 使用 Raft 协议 (CP模式)
    → Nacos主动探测实例健康
    → 实例宕机 → 标记不健康但不删除
    → 适合：数据库、消息队列等基础服务

  代码配置：
  # 注册时指定
  naming.registerInstance("order-service", "1.2.3.4", 8080,
      "DEFAULT", "1.2.3.4:8080", true);  // ephemeral=true → AP
```

```
Nacos 架构中两套协议并行：

  ┌─────────────────────────────────────────────┐
  │              Nacos Cluster                   │
  │                                              │
  │  ┌─────────────┐    ┌──────────────────┐   │
  │  │ Distro (AP)  │    │  Raft (CP)       │   │
  │  │             │    │                  │   │
  │  │ 临时实例数据 │    │ 永久实例数据      │   │
  │  │ 服务发现     │    │ 配置数据(部分)    │   │
  │  │ 异步同步     │    │ 强一致同步       │   │
  │  │ 最终一致     │    │ 多数派写入       │   │
  │  └─────────────┘    └──────────────────┘   │
  │                                              │
  │  两种协议独立运行，互不干扰                    │
  └─────────────────────────────────────────────┘
```

## 五、各注册中心 CAP 对比

| 注册中心 | CAP模式 | 一致性协议 | 特点 |
|---------|--------|-----------|------|
| **Nacos** | AP+CP(双模式) | Distro+Raft | 默认AP，可切CP |
| **Eureka** | AP | Peer-to-Peer复制 | 纯AP，已停止维护 |
| **Zookeeper** | CP | ZAB(类Paxos) | 强一致，不适合大规模注册 |
| **Consul** | CP(+AP可选) | Raft+Gossip | 默认CP，可切AP |
| **Etcd** | CP | Raft | 强一致，K8s使用 |

## 六、面试加分点

1. **提到为什么注册中心选AP**：服务发现场景下，宁可返回少量过期的实例列表（消费者重试即可），也不能让注册中心不可用（导致整个系统雪崩）
2. **提到Eureka的自我保护机制**：Eureka在AP模式下，如果15分钟内心跳失败率>85%，会进入自我保护模式——不再删除过期实例（宁可保留脏数据也不删除）
3. **提到Nacos的健康检查差异**：临时实例靠客户端心跳（push），永久实例靠服务端探测（pull），两种模式互补
4. **提到CAP的误解**：CAP不是非此即彼，而是在分区发生时的选择。没有分区时，系统可以同时是CA
5. **提到BASE理论**：AP模式的最终一致性在BASE理论中对应"Basically Available + Soft state + Eventually consistent"

## 苏格拉底式面试追问

> 这组追问模拟面试官层层逼问，每一问先回答"为什么"，再回答"怎么做"，最后回答"如何证明"。

### 第一层：目标与动机

**Q：Nacos 注册中心你为什么默认选 AP 模式而不是 CP？**

因为注册中心最怕不可用。服务发现场景下，如果注册中心选 CP（强一致），在网络分区或 Leader 选举时，注册中心不可用 → 新服务无法注册 → 消费者发现不了新实例 → 调用失败。选 AP（高可用 + 最终一致），即使分区时各节点数据短暂不一致，注册中心仍可读写——消费者可能拿到"稍过时的实例列表"（某实例刚下线但列表还没更新），调用失败后重试即可。决策依据：注册中心故障的影响（整个系统瘫痪）远大于"拿到过期数据"的影响（单次调用失败重试）。CAP 取舍上，注册中心选 A。

### 第二层：证据与定位

**Q：消费者调用某服务实例失败，你怎么定位是 Nacos 数据过期（实例已下线但 Nacos 没更新）还是实例真的故障？**

查三段：
1. Nacos 的实例列表——查 Nacos 控制台或 API（`/nacos/v1/ns/instance/list?serviceName=xxx`），看这个实例是否还在列表里。如果在但实际下线，是 Nacos 数据过期。
2. 实例健康检查——Nacos 的临时实例靠心跳保活，看这个实例的心跳是否超时（默认 15s 无心跳标记为不健康，30s 摘除）。如果心跳正常但实例实际下线，是"假活"（进程在但服务不可用）。
3. 实例健康端点——直接调实例的 `/actuator/health`，确认实例自身是否健康。

### 第三层：根因深挖

**Q：实例心跳正常（Nacos 认为健康）但实际服务不可用，根因是什么？**

最可能是"假活"——进程还在发心跳但业务逻辑挂了。常见：① 死锁——业务线程死锁，但心跳线程（独立线程）还在发心跳；② OOM 边缘——JVM 堆满，GC 频繁但进程没退出，心跳线程勉强运行；③ 端口被占——服务启动失败但旧的进程残留（或者运维误操作）。Nacos 的心跳只能检测"进程存活"，不能检测"业务健康"。解法是配合"健康端点"——实例除了发心跳，还要暴露 `/health` 端点（检查 DB 连接、关键依赖），Nacos 或负载均衡器定期探测端点，端点不健康也摘除实例。

**Q：为什么不直接用 CP 模式（Raft）保证强一致，这样消费者拿到的实例列表一定是最新的？**

因为 CP 在网络分区时牺牲可用性。Raft 协议要求"多数派确认"才能写入——如果 3 个节点有 2 个网络分区，少数派（1 个）无法达成多数，写入阻塞。注册中心不可写 = 新服务无法注册 = 系统瘫痪。而 AP 模式（Distro 协议）每个节点都能独立接受写入，分区时各分区独立运行，恢复后异步合并。注册中心的"强一致"收益（消费者拿到最新列表）不值得用"可用性"换——拿到旧列表最多重试一次，注册中心不可用是灾难。所以业界注册中心（Eureka、Nacos AP、Consul）都倾向 AP。

### 第四层：方案权衡

**Q：Nacos 的 AP 模式（Distro）和 Eureka 的 AP（Self-Preservation）有什么区别？**

两个实现 AP 的思路不同：
1. Nacos Distro——每个节点独立处理写请求（本地处理后异步同步给其他节点），读请求任何节点都能响应。分区时各节点独立读写，恢复后合并。是"多点写 + 异步同步"。
2. Eureka Self-Preservation——Eureka 也是 AP，但它的机制是"心跳摘除保护"。正常时实例心跳超时被摘除；如果大量实例同时心跳超时（可能是网络问题而非实例下线），Eureka 进入自我保护模式，不再摘除实例（宁可保留"可能已下线"的实例也不误摘）。是"保守摘除 + 保护机制"。

权衡：Nacos Distro 更激进（分区时各节点都能写，恢复后合并可能冲突），Eureka 更保守（宁可数据过时也不冒险）。Nacos 功能更全（同时支持 AP 和 CP，还能做配置中心），Eureka 已停止维护（2.x 闭源）。新项目选 Nacos。

**Q：为什么不用 Zookeeper（ZAB 协议，CP）做注册中心，它很成熟？**

因为 Zookeeper 是 CP，不适合注册中心。ZK 的 ZAB 协议要求多数派确认写入，Leader 选举期间（几秒到几十秒）整个集群不可写。注册中心不可写 = 新服务注册不了 = 系统瘫痪。而且 ZK 的强一致在注册中心是"过度一致"——注册中心的数据（实例列表）对一致性要求不高（秒级延迟可接受），对可用性要求极高。ZK 适合"强一致协调"场景（分布式锁、Leader 选举、配置），Dubbo 早期用 ZK 做注册中心是历史选择，现代实践（Dubbo 3.0、Spring Cloud）倾向 Nacos/Eureka 等 AP 方案。

### 第五层：验证与沉淀

**Q：你怎么证明 Nacos AP 模式的"最终一致"窗口可接受（数据不会长期不一致）？**

监控 + 演练：
1. 一致性监控——定时对比各 Nacos 节点的实例列表，差异应 < 1 秒（Distro 同步间隔）。长期不一致告警。
2. 分区演练——主动制造网络分区（断开某节点的网络），观察分区期间各节点的可用性（都能读写）和恢复后的数据合并（最终一致）。
3. 实际影响——监控"因数据不一致导致的调用失败率"，应该 < 0.1%（重试后成功）。

**Q：注册中心选型怎么沉淀？**

1. 选型决策框架——制定"什么场景选 AP/CP"的决策树：注册中心选 AP、配置中心看需求（强一致选 CP，高可用选 AP）、分布式锁选 CP。
2. 多注册中心容灾——关键服务同时注册到多个注册中心（Nacos + Eureka），消费者任一可用即可发现，单点故障不影响。
3. 健康检查规范——服务实例除了发心跳，必须暴露健康端点（检查业务依赖），避免"假活"实例被路由。


## 结构化回答

**30 秒电梯演讲：** Nacos同时支持AP和CP两种一致性模式。AP模式使用Distro协议（每个节点独立处理写请求，异步同步给其他节点），保证可用性和分区容错性，但允许短暂不一致。

**展开框架：**
1. **Nacos默认AP模式** — Nacos默认AP模式(Distro协议)：每节点独立可写→异步同步→最终一致→注册中心首选
2. **CP模式(Raft协议)** — Leader统一写→多数派确认→强一致→选举期间不可用(约几秒)
3. **临时实例** — 临时实例(ephemeral=true)走AP/Distro；永久实例(ephemeral=false)走CP/Raft

**收尾：** 这块我踩过坑——要不要深入聊：Distro协议和Gossip协议有什么区别？
