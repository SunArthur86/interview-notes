# 【面试题】Nacos AP 和 CP 实现原理

> 来源：面试随手记 — 分布式一致性

## 一、CAP 理论基础

```
CAP 定理：分布式系统最多同时满足三个中的两个

         Consistency (一致性)
            ╱ ╲
           ╱   ╲
          ╱     ╲
         ╱  只能  ╲
        ╱  选两个  ╲
       ╱ ──────── ╲
      ╱            ╲
 Availability    Partition
 (可用性)        Tolerance
                (分区容错)

  现实：网络分区(P)必然发生 → 实际是 C 和 A 二选一

  AP：分区时选择可用性 → 返回可能不一致的数据
      → Nacos默认、Eureka

  CP：分区时选择一致性 → 拒绝服务直到数据一致
      → Zookeeper、Consul(强一致模式)

  Nacos 同时支持两种！通过实例类型自动选择协议。
```

## 二、Nacos AP 模式 —— Distro 协议

### 工作原理

```
Distro 协议（Nacos自研的AP协议）：

  节点A        节点B        节点C        节点D
    │            │            │            │
    │  每个节点负责一部分数据的写入              │
    │  hash(service) % N → 决定哪个节点处理      │
    │            │            │            │

  Client 注册 "order-service"
       │
       ▼
  连接到 节点B
       │
       ├──> 节点B 检查：order-service 是否归我管？
       │    hash("order-service") % 4 = 2 → 归节点C管
       │
       ├──> 转发给 节点C 处理
       │
       └──> 节点C 写入本地 + 异步同步给 A、B、D

  核心特征：
  ✅ 每个节点都能接收请求（但只处理自己负责的部分）
  ✅ 异步同步给其他节点（最终一致）
  ✅ 节点宕机不影响其他节点（高可用）
  ❌ 短暂数据不一致（同步有延迟）
```

### Distro 数据同步流程

```
  正常同步：
  ┌─────────────────────────────────────────┐
  │ 节点C (order-service的负责节点)           │
  │   │                                     │
  │   ├── 每隔5秒 → 全量同步                  │
  │   │   → 把自己负责的所有数据同步给其他节点  │
  │   │                                     │
  │   ├── 有新数据 → 增量同步                 │
  │   │   → 立即发送变更的数据给其他节点       │
  │   │                                     │
  │   └── 接收方确认后 → 同步完成             │
  └─────────────────────────────────────────┘

  节点宕机恢复后：
  ┌─────────────────────────────────────────┐
  │ 节点C 恢复                               │
  │   │                                     │
  │   ├── 向其他节点请求全量数据              │
  │   │   → 获取自己负责的数据                │
  │   │                                     │
  │   ├── 开始正常提供注册服务                │
  │   │                                     │
  │   └── 开始正常同步                        │
  └─────────────────────────────────────────┘
```

### Distro vs Gossip

```
  Distro（Nacos）：
    - 每个节点负责特定数据子集
    - 负责节点 → 主动推给其他节点
    - 收敛速度快（主动推送）

  Gossip（Consul）：
    - 每个节点随机选择另一个节点交换数据
    - 像病毒传播一样扩散
    - 收敛速度较慢（随机传播）
    - 但去中心化程度更高
```

## 三、Nacos CP 模式 —— Raft 协议

### 工作原理

```
Raft 协议（强一致性共识算法）：

  ┌─────────────────────────────────────────────────┐
  │                Raft 集群                         │
  │                                                 │
  │    ┌──────────┐                                 │
  │    │  Leader   │ ← 所有写请求必须经过Leader       │
  │  ┌─┤ (节点C)   ├─┐                               │
  │  │ └──────────┘ │                               │
  │  │     │        │                               │
  │  │  写入日志    │                               │
  │  │     │        │                               │
  │  │     ▼        │                               │
  │  │ 复制给Follower│                               │
  │  │     │        │                               │
  │  ▼     ▼        ▼                               │
  │ ┌──────────┐ ┌──────────┐ ┌──────────┐         │
  │ │Follower  │ │Follower  │ │Follower  │         │
  │ │ (节点A)   │ │ (节点B)  │ │ (节点D)   │         │
  │ └──────────┘ └──────────┘ └──────────┘         │
  │                                                 │
  │ 多数派(N/2+1)确认 → 提交                         │
  │ 4个节点中3个确认 → 提交成功                       │
  └─────────────────────────────────────────────────┘

  核心特征：
  ✅ 强一致性（所有客户端看到相同数据）
  ✅ Leader故障 → 重新选举 → 新Leader接管
  ❌ 选举期间不可用（几秒到几十秒）
  ❌ 少数派节点不可用 → 无法达成多数 → 写入阻塞
```

### Raft Leader 选举过程

```
  初始状态：所有节点都是 Follower

  Step 1: 超时触发选举
  ┌────────────────────────────────────────┐
  │ Follower 在 electionTimeout(150-300ms) │
  │ 内没收到 Leader 心跳                    │
  │ → 变成 Candidate                       │
  │ → 发起选举                              │
  └────────────────────────────────────────┘

  Step 2: 请求投票
  ┌────────────────────────────────────────┐
  │ Candidate 向所有节点发送 RequestVote    │
  │ "我的term=2, log index=100, 请投我"    │
  └────────────────────────────────────────┘

  Step 3: 投票
  ┌────────────────────────────────────────┐
  │ 其他节点收到投票请求：                    │
  │ → Candidate的term比我高？               │
  │ → Candidate的log至少和我一样新？         │
  │ → 我还没投过票？                         │
  │ 全是 → 投赞成票                          │
  └────────────────────────────────────────┘

  Step 4: 获得多数派 → 成为Leader
  ┌────────────────────────────────────────┐
  │ Candidate 获得多数派投票 → 成为 Leader   │
  │ → 开始发送心跳维持权威                   │
  │ → 所有写请求经过 Leader                  │
  └────────────────────────────────────────┘

  脑裂处理：
  │   │          │   │
  │ A │←─分区──→│ C │
  │ B │          │ D │
  │(少数派)      │(多数派)
  │             │Leader C
  │             │正常工作 ✅
  │ A、B选举    │
  │ 但不够多数  │
  │ → 不能写   │
  │ → 不影响C  │
```

## 四、Nacos 如何选择 AP / CP

```
Nacos 通过实例类型自动选择协议：

  临时实例 (ephemeral = true)  ← 默认！
    → 使用 Distro 协议 (AP模式)
    → 客户端通过心跳维持注册
    → 心跳停止 → Nacos主动删除实例
    → 适合：微服务注册发现

  永久实例 (ephemeral = false)
    → 使用 Raft 协议 (CP模式)
    → Nacos主动探测实例健康
    → 实例宕机 → 标记不健康但不删除
    → 适合：数据库、消息队列等基础服务

  代码配置：
  # 注册时指定
  naming.registerInstance("order-service", "1.2.3.4", 8080,
      "DEFAULT", "1.2.3.4:8080", true);  // ephemeral=true → AP
```

```
Nacos 架构中两套协议并行：

  ┌─────────────────────────────────────────────┐
  │              Nacos Cluster                   │
  │                                              │
  │  ┌─────────────┐    ┌──────────────────┐   │
  │  │ Distro (AP)  │    │  Raft (CP)       │   │
  │  │             │    │                  │   │
  │  │ 临时实例数据 │    │ 永久实例数据      │   │
  │  │ 服务发现     │    │ 配置数据(部分)    │   │
  │  │ 异步同步     │    │ 强一致同步       │   │
  │  │ 最终一致     │    │ 多数派写入       │   │
  │  └─────────────┘    └──────────────────┘   │
  │                                              │
  │  两种协议独立运行，互不干扰                    │
  └─────────────────────────────────────────────┘
```

## 五、各注册中心 CAP 对比

| 注册中心 | CAP模式 | 一致性协议 | 特点 |
|---------|--------|-----------|------|
| **Nacos** | AP+CP(双模式) | Distro+Raft | 默认AP，可切CP |
| **Eureka** | AP | Peer-to-Peer复制 | 纯AP，已停止维护 |
| **Zookeeper** | CP | ZAB(类Paxos) | 强一致，不适合大规模注册 |
| **Consul** | CP(+AP可选) | Raft+Gossip | 默认CP，可切AP |
| **Etcd** | CP | Raft | 强一致，K8s使用 |

## 六、面试加分点

1. **提到为什么注册中心选AP**：服务发现场景下，宁可返回少量过期的实例列表（消费者重试即可），也不能让注册中心不可用（导致整个系统雪崩）
2. **提到Eureka的自我保护机制**：Eureka在AP模式下，如果15分钟内心跳失败率>85%，会进入自我保护模式——不再删除过期实例（宁可保留脏数据也不删除）
3. **提到Nacos的健康检查差异**：临时实例靠客户端心跳（push），永久实例靠服务端探测（pull），两种模式互补
4. **提到CAP的误解**：CAP不是非此即彼，而是在分区发生时的选择。没有分区时，系统可以同时是CA
5. **提到BASE理论**：AP模式的最终一致性在BASE理论中对应"Basically Available + Soft state + Eventually consistent"
