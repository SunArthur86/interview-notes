# 【拼多多 Java服务端】微服务有哪些负载均衡的方式？

> 来源：拼多多复活赛一面面经（小红书）— 原题：微服务有了解哪些负载均衡的方式？

## 一、费曼类比

```
用户请求 →  ┌──────────────────────────────────┐
            │     负载均衡器（分诊台）            │
            │                                    │
            │  轮询:  1→A, 2→B, 3→C, 4→A...     │
            │  随机:  随机挑一个                   │
            │  加权:  性能强的多分（A:5, B:3, C:2）│
            │  最少连接: 看谁最闲分给谁             │
            │  哈希:   同一用户总是去同一个实例     │
            └────┬──────┬──────┬─────────────────┘
                 ↓      ↓      ↓
              ┌─────┐┌─────┐┌─────┐
              │实例A ││实例B ││实例C │
              └─────┘└─────┘└─────┘
```

## 二、第一性原理分析

**为什么需要负载均衡？** 因为单机性能有上限。

```
单机瓶颈 → 水平扩展（多实例） → 需要流量分配 → 分配在哪？
                                    │
                    ┌───────────────┼───────────────┐
                    ↓               ↓               ↓
              服务端LB         客户端LB         DNS负载均衡
           (Nginx/LVS)      (Ribbon)         (DNS轮询)
```

## 三、详细答案

### 3.1 负载均衡的位置分类

```
                    ┌───────────────────────────────────────────┐
                    │              DNS 负载均衡                  │
                    │      (域名解析到不同IP)                     │
                    └───────────────────┬───────────────────────┘
                                        ↓
                    ┌───────────────────────────────────────────┐
                    │           服务端负载均衡 (L7)              │
                    │           Nginx / HAProxy                 │
                    │      (按HTTP Header/URL分发)              │
                    └───────────────────┬───────────────────────┘
                                        ↓
                    ┌───────────────────────────────────────────┐
                    │           服务端负载均衡 (L4)              │
                    │           LVS / F5硬件                    │
                    │      (按IP/Port分发，性能更高)             │
                    └───────────────────┬───────────────────────┘
                                        ↓
  ┌────────────────┐    ┌────────────────┐    ┌────────────────┐
  │  Service A     │←──→│  Service B     │←──→│  Service C     │
  │                │    │                │    │                │
  │ 客户端LB Ribbon │    │ 客户端LB Ribbon │    │ 客户端LB Ribbon │
  │ (从Nacos获取   │    │ (从Nacos获取   │    │ (从Nacos获取   │
  │  B的实例列表   │    │  C的实例列表   │    │  A的实例列表   │
  │  直接调用)     │    │  直接调用)     │    │  直接调用)     │
  └────────────────┘    └────────────────┘    └────────────────┘
```

### 3.2 核心策略详解

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **轮询** (Round Robin) | 按顺序逐一分配 | 简单公平 | 不考虑实例性能差异 | 实例配置相同 |
| **加权轮询** (Weighted RR) | 按权重比例分配 | 考虑性能差异 | 权重需手动配置 | 异构实例 |
| **最少连接** (Least Connections) | 分配给当前连接最少的 | 自适应负载 | 需要维护连接计数 | 长连接场景 |
| **一致性哈希** (Consistent Hash) | 同一Key固定到同一实例 | 会话保持 | 实例变更时部分重映射 | 缓存/会话 |
| **随机** (Random) | 随机选择 | 简单、无状态 | 短期可能不均匀 | 大量请求时趋近均匀 |

### 3.3 一致性哈希详解

```
传统哈希的问题:
  hash(key) % N → 当N变化时，几乎所有Key都要重新映射

一致性哈希:
  ┌──────────────────────────────────────────┐
  │         哈希环 (0 ~ 2^32-1)               │
  │                                          │
  │    Server A     Server B                 │
  │   /         \  /         \               │
  │  |    Key1    |    Key2    |             │
  │  |            |            |             │
  │   \         /  \         /               │
  │    Server C     Server D                 │
  │                                          │
  │  Key顺时针找到最近的Server               │
  │  Server增减只影响相邻段 → 最小迁移       │
  └──────────────────────────────────────────┘

  问题: 节点少时分布不均匀
  解决: 虚拟节点（每个物理节点映射到150-200个虚拟节点）
```

### 3.4 LVS vs Nginx 对比

| 维度 | LVS (四层) | Nginx (七层) |
|------|-----------|-------------|
| 工作层 | 传输层(IP+Port) | 应用层(HTTP Header/URL) |
| 性能 | 极高(内核态, 万兆线速) | 高(用户态, 数十万QPS) |
| 功能 | 转发为主 | 转发+静态资源+SSL+重写 |
| 模式 | DR/NAT/TUN/FNAT | 反向代理 |
| 适用 | 流量入口 | 业务入口 |

## 五、扩展知识

- **Spring Cloud LoadBalancer**: Ribbon的替代品，Spring Cloud 2020+默认
- **服务网格 (Istio/Linkerd)**: Sidecar代理模式，LB逻辑从应用中解耦
- **全局负载均衡 (GSLB)**: 跨机房/跨地域的流量调度

## 六、苏格拉底式面试提问

1. **"你说最少连接策略好，但短连接场景下连接数变化很快，怎么保证准确性？"** — 引出连接活跃度统计窗口、采样频率
2. **"一致性哈希如果某个节点宕机了，上面的流量怎么处理？"** — 引出虚拟节点重分布、副本机制
3. **"客户端LB和服务端LB能不能混用？混用有什么问题？"** — 引出双重LB导致负载不均、调试困难
4. **"大促时部分实例被限流了，负载均衡器怎么感知？"** — 引出健康检查、熔断器状态联动
5. **"负载均衡和Service Mesh的关系是什么？"** — 引出Sidecar模式、Envoy数据面、控制面

## 七、面试加分点

1. **能区分四层和七层LB** — LVS工作在IP+Port，Nginx工作在HTTP层
2. **理解客户端LB优势** — 去中心化、无额外跳转、与服务发现集成
3. **知道一致性哈希+虚拟节点** — 展示对数据分布的深入理解
4. **结合实际架构说** — 从DNS→LVS→Nginx→Ribbon分层描述
5. **提到服务网格趋势** — Istio/Envoy将LB从应用代码中解耦


## 结构化回答

**30 秒电梯演讲：** 负载均衡就是把请求分发到多个服务器，核心是'分什么'（策略）和'在哪分'（位置）。

**展开框架：**
1. **服务端LB** — Nginx(七层)/LVS(四层) — 集中式、有额外跳转开销
2. **客户端LB** — Ribbon/Spring Cloud LoadBalancer — 去中心化、直接调用、无额外跳转
3. **核心策略** — 轮询 > 加权轮询 > 最少连接 > 一致性哈希

**收尾：** 这块我踩过坑——要不要深入聊：一致性哈希的虚拟节点是什么？为什么需要？
